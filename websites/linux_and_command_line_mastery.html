The command line is the one skill every serious engineer shares. This course takes you from "I'm scared of the black window" to moving around a Linux system, wrangling files and text, controlling processes, and writing real shell scripts with confidence. Works on Linux, macOS, and Windows (via WSL). Everything is hands-on — open a terminal and type along.

How to use this course

  • Open a terminal and follow along. The command line is muscle memory. Reading about grep teaches you nothing; running it fifty times teaches you everything.
  • Each section follows the same shape: Concept → Commands → Real example → Exercises with solutions. Try every exercise before opening the solution.
  • Make a scratch directory to break things in: mkdir ~/cli-practice && cd ~/cli-practice. You can always delete it and start fresh.
  • You will not damage anything by exploring, as long as you stay out of system directories and avoid sudo and rm -rf until you understand them. This course flags every genuinely dangerous command with a ⚠️.
  • Each section ends with a cheat sheet — a scannable summary you'll come back to for months.

By the end you'll be comfortable doing the things developers do every day without thinking: navigating a project from the terminal, searching thousands of files in a second, chaining small tools into powerful one-liners, managing running programs, connecting to remote servers, and automating repetitive work with scripts.

SECTION 1

Getting Started

1.1 Introduction

Every graphical app you've ever used — your browser, your editor, your file explorer — is a friendly face on top of something more fundamental. Underneath, a computer is a machine that runs programs and manipulates files. The command line is the direct, text-based way to tell it what to do.

Why learn it when we have nice buttons and windows?

  • It's everywhere. Servers, cloud machines, containers, CI pipelines, embedded devices — almost none of them have a graphical interface. If you deploy software, you will work on a Linux machine through a terminal. There is no button.
  • It's precise and repeatable. A command is text. You can save it, share it, put it in a script, and run it a thousand times identically. You can't do that with a sequence of mouse clicks.
  • It's fast. Once it's in your fingers, renaming 500 files, finding every place a function is used, or extracting the error lines from a 2 GB log takes seconds — things that are painful or impossible with a mouse.
  • It composes. Small single-purpose tools snap together into pipelines that solve problems their authors never imagined. This is the single most powerful idea in this course, and you'll meet it in Section 6.

By the end of this course you will be able to:

  • Move around a Linux filesystem and manipulate files and directories fluently.
  • Read, search, filter, and transform text from the command line.
  • Understand and set file permissions.
  • Redirect and pipe data between programs.
  • Control running processes and jobs.
  • Write real, useful shell scripts.
  • Connect to remote machines over SSH and transfer files.
  • Build a personal toolkit of scripts and shortcuts that make you faster every day.

1.2 Terminal, shell, and Linux — what's what

Three words get used loosely and it's worth pinning them down, because confusing them causes a lot of early frustration.

  • The terminal (or "terminal emulator") is the window — the program that shows text and takes your keystrokes. Examples: GNOME Terminal, macOS Terminal, iTerm2, Windows Terminal. It's just a display.
  • The shell is the program running inside the terminal that reads what you type, interprets it, runs commands, and prints results. The most common shell is bash ("Bourne Again SHell"); macOS now ships zsh by default. They're 95% identical for everyday use; this course notes the rare differences.
  • Linux (or, on a Mac, the Unix-like macOS core) is the operating system underneath — the thing that actually manages files, memory, and hardware. The shell is how you talk to it.

Analogy: the terminal is the phone handset, the shell is the language you speak, and the operating system is the person on the other end who actually does things.

1.3 Opening a terminal

Linux: Press Ctrl+Alt+T, or search your applications for "Terminal."

macOS: Open Spotlight (⌘+Space), type "Terminal," press Enter. macOS is Unix under the hood, so nearly everything in this course works directly.

Windows: Windows is not Unix, and its old cmd.exe and PowerShell use different commands. The right move is WSL (Windows Subsystem for Linux), which gives you a real Linux environment inside Windows:

wsl --install

Restart when prompted, and you'll have Ubuntu with a real bash shell. Open it from the Start menu ("Ubuntu" or "WSL"). Everything in this course then works exactly as written. (Git Bash, which ships with Git for Windows, is a lighter alternative that covers most of the early sections.)

From here on, "the terminal" means a Linux/macOS shell prompt.

1.4 The prompt

When your shell is ready for a command, it shows a prompt. A typical one looks like:

arthur@laptop:~$

Reading it left to right: your username (arthur), an @, the hostname of the machine (laptop), a :, your current directory (~, which means your home folder — more in Section 2), and finally a $.

That trailing $ is the classic "I'm a normal user, type here" symbol. If you ever see a # instead, you're acting as the all-powerful root (administrator) user — be careful. In this course, examples that you type are shown after a $, and the machine's output is shown plainly beneath. You type only the part after the $.

1.5 Your first commands

Type each of these and press Enter:

whoami      # prints your username

date        # prints the current date and time

echo Hello from the command line

echo simply prints back whatever you give it. It looks trivial, but it's one of the most-used commands in all of shell scripting — you'll rely on it constantly.

pwd         # "print working directory" — where am I right now?

uname -a    # information about the operating system and kernel

Congratulations — you've now run commands. Nothing exploded. The terminal is just a place where you type the name of a program and it runs.

1.6 Anatomy of a command

Almost every command follows the same grammar:

command  [options]  [arguments]
  • The command is the program to run (ls, echo, cp).
  • Options (also called flags) change how it behaves. They usually start with a dash. Short options are a single letter (-l); long options are a word (--long). Short options can often be bundled: -l -a -h is the same as -lah.
  • Arguments are what it acts on — usually files, directories, or text.

For example:

ls -l -a /etc

Here ls is the command, -l and -a are options (long listing, show all files), and /etc is the argument (the directory to list). You could also write it ls -la /etc.

Getting this mental model early makes every future command readable: even a scary-looking line is just a program, some flags, and some targets.

1.7 Getting help

You will forget flags constantly. Every professional does. Knowing how to look them up fast is the actual skill.

The quick summary — most commands accept --help:

ls --help

The full manual — man (short for "manual") opens detailed documentation:

man ls

Inside man: scroll with the arrow keys or Space (page down), search by typing /word then Enter, and press q to quit. (That last one traps everyone once. If you're ever stuck in a full-screen text program, try q.)

A friendlier option — the community tool tldr shows practical examples instead of exhaustive prose. It's not installed by default, but it's worth it (you'll learn to install tools in Section 12):

tldr tar    # shows the handful of tar commands you'll actually use

Finally, to find out what a command even is:

type cd      # tells you cd is a shell builtin
type ls      # tells you where the ls program lives

1.8 A few survival keys

These work at the prompt and save enormous time. Learn them now; they'll feel natural within a day.

Key What it does
Tab Auto-complete a command or filename. Press it constantly.
↑ / ↓ Scroll through previous commands (history).
Ctrl+C Cancel the current command / stop a running program.
Ctrl+L Clear the screen (same as the clear command).
Ctrl+A / Ctrl+E Jump to the start / end of the line.
Ctrl+U / Ctrl+K Delete to start / end of the line.

Tab completion deserves special emphasis. Instead of typing Documents/, type Doc and press Tab — the shell finishes it. This isn't just faster; it also prevents typos and confirms the thing you're referring to actually exists.

SECTION 2

Navigating the Filesystem

2.1 The filesystem is a tree

Everything on a Linux system lives in a single upside-down tree of directories (folders). At the very top is the root directory, written / (just a forward slash). Every other directory branches off from it.

/
├── home/
│   └── arthur/          ← your home directory
│       ├── Documents/
│       ├── Downloads/
│       └── projects/
├── etc/                 ← system configuration files
├── usr/                 ← installed programs and their files
├── var/                 ← logs, caches, changing data
└── tmp/                 ← temporary files

There are no "drives" like C: and D: as on Windows. Everything — even a second disk or a USB stick — appears as a directory somewhere inside this one tree. This is cleaner than it sounds, and it's why Unix people say "everything is a file" (more on that later).

Your home directory is your personal space, usually /home/yourname on Linux or /Users/yourname on macOS. It's where your files live and where you have full control. The shell gives it a shortcut: the tilde ~ always means "my home directory."

2.2 Where am I? — pwd

Because the shell is always "sitting" in some directory (the working directory), the first navigation question is always "where am I?"

pwd

/home/arthur/projects

pwd = print working directory. When a command acts on a file without a full path, it acts relative to wherever pwd says you are.

2.3 What's here? — ls

ls lists the contents of a directory.

ls              # list the current directory
ls /etc         # list a specific directory

Its options are worth memorizing because you'll use them hourly:

ls -l           # long format: permissions, owner, size, date, name
ls -a           # all files, including hidden ones (names starting with .)
ls -h           # human-readable sizes (with -l): 4.0K, 16M, 2.1G
ls -t           # sort by modification time, newest first
ls -r           # reverse the sort order
ls -R           # recursive: list subdirectories too

They combine. The single most useful everyday invocation is:

ls -lah         # long, all files, human sizes — the "show me everything" command

A long-format line looks like this:

-rw-r--r--  1 arthur staff  2048 Jan  6 14:03 notes.txt

Reading it: the permissions (-rw-r--r--, decoded in Section 5), a link count, the owner (arthur), the group (staff), the size in bytes (2048), the last-modified date, and the name (notes.txt). Don't worry about every field yet — just know it's all here.

2.4 Hidden files

Any file or directory whose name starts with a dot (.) is hidden — ls skips it by default. This is a convention, not a security feature; it's used to keep configuration files out of your way. Your shell's settings live in hidden files like ~/.bashrc and ~/.gitconfig.

ls          # everyday files
ls -a       # also shows .bashrc, .gitconfig, ., .. and friends

You'll notice two odd entries in ls -a: . and ... They're special and crucial — the next topic.

2.5 Absolute vs relative paths

A path is the address of a file or directory. There are two kinds, and understanding the difference is the core skill of this section.

An absolute path starts from the root / and spells out the whole route. It means the same thing no matter where you currently are:

/home/arthur/projects/app/main.py

A relative path starts from your current directory. It's shorter, but its meaning depends on where you are:

app/main.py         # main.py inside app/ inside wherever I am now

Four building blocks show up in paths constantly:

Symbol Means
/ The root directory (at the start) or a separator (between names).
~ Your home directory. ~/Downloads is /home/arthur/Downloads.
. The current directory. ./run.sh means "the run.sh right here."
.. The parent directory, one level up. ../ goes up one; ../../ goes up two.

Those . and .. entries you saw in ls -a are what make . and .. work — every directory contains a link to itself and to its parent.

2.6 Moving around — cd

cd = change directory. It's how you walk the tree.

cd /etc                 # go to an absolute path
cd projects/app         # go down into projects/app (relative)
cd ..                   # go up one level to the parent
cd ../..                # go up two levels
cd ~                    # go home (also just: cd  with no argument)
cd -                    # go back to the previous directory you were in

That last one, cd -, is a hidden gem: it toggles between the current and previous directory, like the "back" button. Great when you're bouncing between two folders.

A worked walk through the tree:

pwd                     # /home/arthur
cd projects             # /home/arthur/projects
cd app                  # /home/arthur/projects/app
cd ..                   # /home/arthur/projects
cd ~/Downloads          # /home/arthur/Downloads
cd -                    # back to /home/arthur/projects

Use Tab completion the whole time. Type cd Doc + Tab and the shell offers Documents/. If nothing completes, the thing you typed doesn't exist where you are — an instant sanity check.

2.7 Seeing the shape of a directory — tree

ls shows one level. tree draws the whole nested structure at once:

tree            # the current directory and everything under it
tree -L 2       # limit depth to 2 levels
tree -d         # directories only

tree isn't always installed (Section 12 covers installing it), but it's the fastest way to understand an unfamiliar project's layout. If you don't have it, ls -R gives a rougher version.

2.8 A real example — exploring a new project

Suppose you just cloned a project and want to orient yourself. Here's the exact sequence an engineer runs, and why:

cd ~/projects/some-app     # go into it
pwd                        # confirm where I am
ls -lah                    # what's here? note README, src/, hidden config
tree -L 2                  # get the overall shape without drowning in detail
cd src                     # dive into the source
ls                         # what modules exist?
cd -                       # pop back to the top

In thirty seconds and six commands you now understand a codebase's layout better than clicking through a file explorer for two minutes. This orientation ritual — where am I, what's here, what's the shape — is something you'll do dozens of times a day.

SECTION 3

Creating & Managing Files and Directories

Now that you can move around, let's change things — make, copy, move, rename, and delete files and directories. These are the verbs you'll use constantly.

3.1 Making directories — mkdir

mkdir notes                 # make a directory called notes
mkdir a b c                 # make three at once

To create a whole nested path in one go, use -p ("parents") — it makes every directory in the path that doesn't exist yet, and doesn't complain if some already do:

mkdir -p projects/app/src   # creates projects, then app, then src as needed

Without -p, mkdir projects/app/src fails if projects or app don't exist yet. -p is what you almost always want when scaffolding a project.

3.2 Making empty files — touch

touch main.py               # create an empty file (if it doesn't exist)
touch a.txt b.txt c.txt     # several at once

touch has a second job: if the file already exists, it updates its "last modified" timestamp without changing the contents. That's occasionally useful for build tools, but day to day you'll use it to quickly create empty files.

To create a file with content, the quickest way is echo with redirection (you'll learn the mechanism properly in Section 6):

echo "print('hello')" > hello.py     # write that line into hello.py

3.3 Copying — cp

cp source.txt dest.txt          # copy a file
cp notes.txt ~/backup/          # copy into another directory (keeps the name)
cp a.txt b.txt c.txt dir/       # copy several files into a directory

To copy a directory and everything inside it, you need -r ("recursive"):

cp -r project/ project-backup/  # copy the whole tree

Useful flags:

cp -i source dest    # interactive: ask before overwriting an existing file
cp -v source dest    # verbose: print what it's doing
cp -r -p src/ dst/   # preserve timestamps and permissions

3.4 Moving and renaming — mv

There's no separate "rename" command in Unix, because moving and renaming are the same operation: you're changing a file's path. mv does both.

mv old.txt new.txt              # rename (same directory, new name)
mv report.txt ~/Documents/      # move into another directory (same name)
mv report.txt ~/Documents/final.txt   # move AND rename in one step
mv *.log logs/                  # move all .log files into logs/

mv has the same safety flags as cp:

mv -i old new    # ask before overwriting
mv -v old new    # print what happened

Unlike cp, mv doesn't need -r for directories — moving a directory just moves the whole thing by default.

3.5 Deleting — rm and rmdir ⚠️

Here is where you must slow down. The command line has no Recycle Bin. When you rm a file, it's gone — not "in the trash," gone. There's no undo.

rm file.txt             # delete a file
rm a.txt b.txt          # delete several
rmdir emptydir          # remove a directory — ONLY if it's empty

rmdir only removes empty directories, which is a nice safety net. To delete a directory with contents, you need recursive rm:

rm -r olddir/           # delete a directory and everything inside it

Two flags shape how careful rm is:

rm -i file.txt          # interactive: ask "are you sure?" for each file
rm -f file.txt          # force: never ask, ignore missing files

⚠️ The most dangerous command in Unix

rm -rf something/

rm -rf means "recursively force-delete, no questions asked." It is genuinely useful and genuinely dangerous. The infamous disaster is a stray space or variable:

rm -rf /              # ⚠️ tries to erase the ENTIRE system. Never run this.
rm -rf $DIR/          # ⚠️ if $DIR is empty/unset, this becomes rm -rf /

Rules that keep professionals safe:

  • Read the whole command before pressing Enter, especially the path.
  • Prefer rm -ri while learning, so it asks before each deletion.
  • Before an rm -rf, run the same path through ls first: ls something/ — see exactly what you're about to destroy.
  • Never rm -rf a path built from a variable without checking the variable isn't empty.

Respect rm, don't fear it. Used deliberately it's fine; used carelessly it ruins your day.

3.6 Wildcards (globbing)

The shell can match many filenames at once using glob patterns. This turns "operate on one file" into "operate on fifty" for free. The shell expands the pattern before the command runs, so ls *.txt becomes ls a.txt b.txt c.txt and ls never even sees the *.

Pattern Matches
* Any number of characters (including none). *.txt = every file ending in .txt.
? Exactly one character. file?.txt matches file1.txt, fileA.txt, not file10.txt.
[abc] Any one character in the set. file[12].txt matches file1.txt and file2.txt.
[a-z] Any one character in the range. [0-9] = any digit.
{a,b,c} Brace expansion — each listed item. file.{js,ts} = file.js file.ts.

Examples:

ls *.md                 # every Markdown file
rm *.tmp                # delete all .tmp files (⚠️ check with ls first!)
cp report.{txt,bak}     # shorthand — copies report.txt to report.bak
mv img_*.png images/    # move every file starting with img_ and ending .png
ls data[0-9].csv        # data0.csv through data9.csv

Habit worth building: before running a destructive command with a glob, run ls with the same pattern first. ls *.tmp shows you exactly what rm *.tmp would delete. This one habit prevents most glob-related accidents.

3.7 Links — one file, two names (ln)

Sometimes you want the same file to appear in two places. A symbolic link (symlink) is a small pointer — a signpost — to another file or directory.

ln -s /path/to/original linkname     # create a symbolic link

For example, projects often keep the "current" version behind a stable name:

ln -s app-v2.3.1 current             # `current` now points at app-v2.3.1

Now cd current takes you into app-v2.3.1. When you release v2.4.0, you repoint the link and everything referring to current follows along. In ls -l, a symlink shows an arrow: current -> app-v2.3.1.

There are also hard links (ln without -s), which are a second real name for the same underlying data rather than a pointer. Symbolic links are far more common in everyday work; reach for ln -s and you'll rarely need the other kind.

3.8 A real example — scaffolding a project

Watch how these commands combine to build a project skeleton in seconds:

mkdir -p myapp/src myapp/tests myapp/docs   # nested dirs in one shot
cd myapp
touch src/main.py src/utils.py tests/test_main.py
echo "# My App" > README.md                 # a starter README with content
cp README.md docs/index.md                   # seed the docs from the README
ls -R                                         # admire your handiwork

Six lines and you have a laid-out project. Do this a few times and you'll never scaffold with a mouse again.

SECTION 4

Viewing & Inspecting Files

You can navigate and create files; now let's look inside them and learn about them — without opening a heavyweight editor. These tools are fast, work over slow remote connections, and chain beautifully with everything in later sections.

4.1 Dumping a whole file — cat

cat (short for "concatenate") prints a file's entire contents to the screen:

cat notes.txt

It's perfect for short files. For long ones it flies past faster than you can read — use less (next) instead. cat earns its name by joining multiple files:

cat part1.txt part2.txt           # print both, in order
cat part1.txt part2.txt > whole.txt   # join them into one file (Section 6)

A couple of handy flags:

cat -n file.txt      # number every line
cat -A file.txt      # show invisible characters (tabs, line endings) — great for debugging

4.2 Reading long files — less

less opens a file in a scrollable viewer that loads instantly even for enormous files (it doesn't read the whole thing into memory). This is the right tool for logs, configs, and anything longer than a screen.

less /var/log/syslog

Navigation inside less (these are worth memorizing — they're the same keys man uses):

Key Action
Space / b Page down / page up
↓ / ↑ Line down / up
/word Search forward for "word" (then n = next, N = previous)
g / G Jump to the top / bottom of the file
q Quit

"How do I get out of this thing?" is q. The mnemonic: less is "the pager," and q quits it — just like man, which uses less under the hood.

4.3 The beginning and end — head and tail

Often you don't want the whole file — just a peek.

head file.txt         # first 10 lines
head -n 20 file.txt   # first 20 lines
tail file.txt         # last 10 lines
tail -n 50 file.txt   # last 50 lines

tail has a superpower that developers use daily: -f ("follow") keeps the file open and prints new lines as they're added. This is how you watch a live log while your app runs:

tail -f /var/log/app.log      # live-follow the log; Ctrl-C to stop
tail -n 100 -f app.log        # show the last 100 lines, then follow

Leave tail -f running in one terminal, trigger an action in your app, and watch the log react in real time. It's one of the most-used debugging tricks there is.

4.4 Counting — wc

wc = word count, though it counts more than words:

wc file.txt         # lines, words, bytes — all three
wc -l file.txt      # just the line count
wc -w file.txt      # just the word count
wc -c file.txt      # just the byte count

wc -l is surprisingly useful: "how many lines in this file?" answers questions like how many entries a list has or how many errors a log holds (once you combine it with grep in Section 7).

4.5 Learning about a file — file and stat

Filenames lie — a .txt extension doesn't guarantee text. file inspects the actual contents and tells you what something really is:

file photo.jpg        # photo.jpg: JPEG image data, 1920x1080
file script           # script: Bourne-Again shell script, ASCII text executable
file /bin/ls          # /bin/ls: ELF 64-bit executable

stat shows the full metadata — size, permissions, and the three timestamps every file carries (accessed, modified, changed):

stat notes.txt

You won't reach for stat daily, but when you need to know exactly when a file changed or its precise size in bytes, it's the authority.

4.6 A real example — triaging a log file

A server misbehaved and left a 500 MB log. Opening that in an editor would be painful. Here's the CLI triage:

ls -lh app.log            # how big is it? — 480M. Good thing we're not using an editor.
wc -l app.log             # how many lines? — 2,041,556
head -n 20 app.log        # what does the start look like? (format, timestamps)
tail -n 50 app.log        # what are the most recent events? (often where the crash is)
less app.log              # scroll and search interactively with /ERROR
tail -f app.log           # if the app is still running, watch it live

You've understood the shape and the latest state of a half-gigabyte file in seconds, touching only the parts you needed. In Section 7 you'll add grep and pull only the error lines out of it — but even with just these tools, you're already working the way professionals do.

SECTION 5

Permissions & Ownership

Linux is a multi-user system at its core, even on your personal laptop. Every file records who owns it and who is allowed to do what with it. Understanding this isn't optional trivia — it's why your script "won't run," why you get "Permission denied," and how servers stay secure. This section makes the cryptic -rwxr-xr-x string finally make sense.

5.1 Users and groups

Every file belongs to a user (its owner) and a group (a named set of users). Permissions are defined for three audiences:

  • Owner (u) — the user who owns the file.
  • Group (g) — members of the file's group.
  • Others (o) — everyone else on the system.

This lets you say things like "I can edit this, my team can read it, and no one else can touch it." On your own laptop you're usually all three, but the model is the same everywhere, and it matters enormously on shared servers.

Find out who you are and what groups you're in:

whoami       # your username
id           # your user id, group id, and all your groups
groups       # just the list of groups you belong to

5.2 Reading the permission string

Recall the long listing:

ls -l script.sh

-rwxr-xr-x  1 arthur staff  48 Jan  6 14:03 script.sh

That first block, -rwxr-xr-x, is ten characters. Break it apart:

 -   rwx   r-x   r-x
 │    │     │     │
type owner group others
  • Character 1 is the type: - for a regular file, d for a directory, l for a symbolic link.
  • Characters 2–4 are the owner's permissions.
  • Characters 5–7 are the group's permissions.
  • Characters 8–10 are others' permissions.

Within each trio, the three letters mean:

Letter On a file On a directory
r (read) View the contents List what's inside (ls)
w (write) Modify the contents Create/delete/rename files inside
x (execute) Run it as a program cd into it / access files inside

A dash - in any slot means that permission is absent. So -rwxr-xr-x reads as: a regular file; the owner can read, write, and execute; the group can read and execute but not write; others can read and execute but not write.

The directory column surprises people: x on a directory means "you can enter it." A directory you can read but not execute lets you see the names inside but not actually access them. Both usually go together.

5.3 Changing permissions — chmod

chmod ("change mode") sets permissions. It has two notations; learn both, because you'll see both.

Symbolic notation — readable, good for tweaking one thing:

chmod +x script.sh        # add execute for everyone
chmod u+x script.sh       # add execute for the owner only
chmod g-w file.txt        # remove write from the group
chmod o-r secret.txt      # remove read from others
chmod u+x,g-w,o-r file    # combine several changes
chmod a+r file            # a = all (owner+group+others)

The pattern is who (u/g/o/a) then +/-/= then which (r/w/x). = sets exactly those permissions, clearing the rest.

Octal (numeric) notation — compact, the professional default. Each permission is a number: read = 4, write = 2, execute = 1. Add them up per audience to get a single digit:

Digit Permissions Meaning
7 rwx 4+2+1 — full
6 rw- 4+2 — read/write
5 r-x 4+1 — read/execute
4 r-- read only
0 --- nothing

Three digits set owner, group, others in order:

chmod 755 script.sh   # rwxr-xr-x  — owner full, others read+run (typical script/program)
chmod 644 notes.txt   # rw-r--r--  — owner edits, everyone else reads (typical file)
chmod 600 secret.key  # rw-------  — owner only, no one else sees it (private keys, secrets)
chmod 700 ~/private   # rwx------  — a directory only you can enter (private folder)

You'll internalize a few of these by muscle memory fast: 644 for normal files, 755 for scripts and directories, 600 for secrets. Those three cover most of daily life.

5.4 The "won't run" moment

The most common beginner encounter with permissions is this:

./deploy.sh

bash: ./deploy.sh: Permission denied

The script is there, it's valid — but it isn't marked executable. The fix:

chmod +x deploy.sh
./deploy.sh              # now it runs

This trips up everyone the first time. When a script you just wrote or downloaded won't run, chmod +x is almost always the answer. Check with ls -l: if there's no x in the owner's trio, that's your culprit.

5.5 Changing ownership — chown and chgrp ⚠️

chown changes who owns a file; chgrp changes its group. Because handing a file to another user is a privileged act, these usually require sudo (administrator rights — Section 12):

sudo chown arthur file.txt          # make arthur the owner
sudo chown arthur:developers file   # set owner AND group at once
sudo chgrp developers file.txt      # change just the group
sudo chown -R arthur:arthur mydir/  # recursively fix a whole directory tree

You'll meet chown mostly when files created by one user (or by sudo, which makes files owned by root) need to belong to you so you can edit them. chown -R fixing an entire project's ownership is a common "why can't I write here?" rescue.

5.6 A note on sudo and root

The root user is the all-powerful administrator who bypasses permission checks entirely. You rarely log in as root; instead you borrow its power for one command with sudo ("superuser do"):

sudo apt update          # run this one command as root

Section 12 covers sudo properly. For now, three principles:

  • sudo exists so that dangerous actions require a deliberate extra step (and your password).
  • If a command fails with "Permission denied" on a system file or directory, sudo is often needed — but pause and make sure you actually should be changing that thing.
  • Never habitually prefix everything with sudo to make errors go away. That's how people accidentally break their systems. Permissions are usually telling you something true.

SECTION 6

Redirection & Pipes: the Unix Superpower

This is the section that changes how you think. Every tool you've met reads some input and produces some output. Redirection and pipes let you rewire where that input comes from and where that output goes — sending output to files, feeding files as input, and chaining programs so one's output becomes the next one's input. This is the single idea that makes the command line more powerful than any graphical app.

6.1 The three streams

Every running program has three standard channels, called streams:

  • stdin (standard input, stream 0) — where it reads input from. By default, your keyboard.
  • stdout (standard output, stream 1) — where it writes normal results. By default, your screen.
  • stderr (standard error, stream 2) — where it writes error messages. Also the screen by default, but a separate channel.

Why separate normal output from errors? So you can capture results while still seeing errors, or log errors without polluting your data. Keeping stdout and stderr distinct is a small design decision that pays off constantly.

6.2 Redirecting output to a file — > and >>

> sends a command's stdout into a file instead of the screen. It overwrites the file (creating it if needed):

echo "Hello" > greeting.txt      # greeting.txt now contains exactly "Hello"
ls -l > listing.txt              # save a directory listing to a file
date > timestamp.txt             # overwrite with the current date

>> appends instead of overwriting — it adds to the end, preserving what's there:

echo "first line" > log.txt      # create/overwrite
echo "second line" >> log.txt    # add below
echo "third line" >> log.txt     # add below
cat log.txt                      # first line / second line / third line

The difference matters enormously: > on an important file erases it. A classic mistake is command > file when you meant command >> file. When in doubt about clobbering something, use >>.

6.3 Redirecting errors — 2>

Because stderr is stream 2, you redirect it with 2>:

find / -name "*.conf" 2> errors.txt      # results to screen, errors to a file

That command searches the whole system; without 2>, "Permission denied" messages for folders you can't read would bury the real results. Sending them to errors.txt (or discarding them) keeps your output clean.

To send both stdout and stderr to the same place, redirect stdout to a file and then point stderr at "wherever stdout is going" with 2>&1:

command > everything.txt 2>&1    # both streams into one file

Order matters: 2>&1 must come after the > so stderr copies stdout's already-set destination.

6.4 The black hole — /dev/null

/dev/null is a special file that discards anything written to it — a bottomless trash can. Redirect to it when you want to run something for its effect, not its output:

command > /dev/null              # throw away normal output, keep errors visible
command 2> /dev/null             # throw away errors, keep normal output
command > /dev/null 2>&1         # silence everything

2> /dev/null is the polite way to run a command that spews harmless warnings you don't care about.

6.5 Redirecting input — <

< feeds a file into a command's stdin, as if you'd typed the file's contents:

sort < names.txt                 # sort takes its input from the file
wc -l < file.txt                 # count lines fed from the file

Many commands accept a filename as an argument directly (sort names.txt works too), so < is less common than >. But it's essential when a command only reads from stdin, and it's how here-documents and scripts feed data around.

6.6 Pipes — the main event |

Here is the idea that everything has been building toward. A pipe (|) connects the stdout of one command to the stdin of the next. Instead of saving to a temporary file and re-reading it, data flows directly from one program into another:

command1 | command2 | command3

Read left to right: command1's output becomes command2's input, whose output becomes command3's input. You're building an assembly line where each stage transforms the data a little.

A first taste:

ls -l | wc -l          # count how many entries ls printed (files in this directory)

ls -l lists the directory; instead of printing to the screen, its output flows into wc -l, which counts the lines. No temporary file, no manual step. Another:

cat access.log | grep "ERROR" | wc -l    # how many error lines are in the log?

cat streams the file → grep keeps only lines containing "ERROR" → wc -l counts them. Three tiny tools, each doing one job, combined into "count the errors in this log." You'll meet grep, sort, uniq, and friends properly in Section 7 — pipes are what make them extraordinary.

6.7 The Unix philosophy

There's a deep principle here, stated in the 1970s and still the reason the command line endures:

Write programs that do one thing and do it well. Write programs to work together. Write programs to handle text streams, because that is a universal interface.

That's why there's no single giant "process my log file" program. Instead there are dozens of small, sharp tools — grep, sort, cut, uniq, wc, head, tail, sed, awk — and pipes to snap them together however a problem demands. A pipeline you invent today might be a combination no one has ever typed before, and it just works. Master this and you can solve data-wrangling problems in one line that would take a page of code in most languages.

6.8 tee — split the stream

Sometimes you want output to go to a file and keep flowing down the pipe (so you can see it or process it further). tee does both — like a T-junction in plumbing:

command | tee output.txt                 # print to screen AND save to file
command | tee output.txt | grep ERROR    # save everything, but only show errors
ls -l | tee listing.txt | wc -l          # save the listing and count it

tee -a appends instead of overwriting, matching >>.

6.9 A real example — the pipeline mindset

You have a web server log and want the top 5 IP addresses by number of requests. No single tool does this; the pipeline does:

cut -d' ' -f1 access.log | sort | uniq -c | sort -rn | head -n 5

Trace the data through each stage:

  1. cut -d' ' -f1 access.log — split each line on spaces, keep field 1 (the IP address).
  2. sort — put identical IPs next to each other (required for the next step).
  3. uniq -c — collapse duplicates and prefix each with its count.
  4. sort -rn — sort by that count, numerically, highest first.
  5. head -n 5 — keep the top five.
   4213 203.0.113.7
   2988 198.51.100.42
   1104 203.0.113.19
    877 192.0.2.55
    431 198.51.100.9

That's the whole game: no field understands the others, yet together they answer a real question in one line. When you catch yourself thinking "first get X, then narrow to Y, then count, then rank" — that's a pipeline forming in your head. This mindset is the most valuable thing in the entire course.

🏋️ Exercise 6.4

Explain, stage by stage, what this pipeline does: cat names.txt | sort | uniq. Then explain why removing sort would break it.

Solution

Removing sort breaks it because uniq only collapses duplicates that are next to each other. If the same name appears on lines 1 and 50 without sorting, uniq won't notice they're duplicates. Sorting first guarantees identical lines are adjacent, which is exactly what uniq needs. (sort -u does both in one step.)

SECTION 7

Searching & Text Processing

Pipes gave you the plumbing; this section gives you the power tools that flow through it. grep, find, sort, uniq, cut, tr, and a taste of sed and awk are the workhorses of real command-line work. Learn to combine them and you can extract, filter, and reshape text faster than any GUI — and far faster than writing a script for it.

7.1 Searching inside files — grep

grep finds lines matching a pattern. It is possibly the most-used developer command after cd and ls.

grep "error" app.log            # print every line containing "error"
grep "TODO" main.py             # find your leftover TODO comments

The essential flags — memorize these, you'll use them daily:

grep -i "error" app.log     # case-insensitive (matches Error, ERROR, error)
grep -n "error" app.log     # show line numbers
grep -r "getUser" src/      # recursive: search every file under src/
grep -v "debug" app.log     # invert: lines that do NOT match
grep -c "error" app.log     # count matching lines instead of printing them
grep -w "cat" file          # whole word only (not "category" or "concatenate")
grep -l "error" *.log       # just list which files contain a match

The combination you'll type most is grep -rn — search recursively and show file:line for each hit. It's how you answer "where in this codebase is this function used?":

grep -rn "calculateTotal" .    # every use of calculateTotal, with locations

grep -E enables extended regular expressions — patterns with special meaning. A few you'll want:

grep -E "cat|dog" file      # lines with "cat" OR "dog"
grep -E "^Error" file       # lines that START with "Error" (^ = start of line)
grep -E "done$" file        # lines that END with "done" ($ = end of line)
grep -E "[0-9]+" file       # lines containing one or more digits

Regular expressions are a deep topic — ^ (start), $ (end), . (any char), * (zero or more), + (one or more), [...] (a set), and | (or) will carry you a long way. You don't need to master them now; recognize that grep speaks this language and reach for it as you grow.

7.2 Finding files — find

grep searches inside files; find searches for files by name, type, size, or age. It walks a directory tree and prints everything matching your criteria.

find . -name "*.py"             # every .py file under the current directory
find . -iname "readme*"         # case-insensitive name match
find /var/log -type f           # only regular files (not directories)
find . -type d -name "test*"    # only directories whose name starts with test
find . -size +100M              # files bigger than 100 MB
find . -mtime -7                # files modified in the last 7 days
find . -empty                   # empty files and directories

find can also act on what it finds, via -exec (the {} is replaced by each result; the \; ends the command):

find . -name "*.tmp" -delete                  # find and delete all .tmp files ⚠️
find . -name "*.sh" -exec chmod +x {} \;      # make every shell script executable
find . -name "*.log" -exec wc -l {} \;        # count lines in each log

find . -name "*.tmp" -delete is genuinely useful cleanup — but it's destructive, so run it without -delete first to see exactly what would be removed. This "look before you leap" habit applies to every destructive find.

7.3 Sorting and de-duplicating — sort and uniq

sort names.txt          # alphabetical
sort -r names.txt       # reverse
sort -n numbers.txt     # numeric (so 10 comes after 9, not before)
sort -rn numbers.txt    # numeric, largest first
sort -k2 data.txt       # sort by the 2nd column
sort -u names.txt       # sort AND remove duplicates in one step

uniq collapses adjacent duplicate lines — which is why it's almost always paired with sort first (as you saw in Section 6):

sort names.txt | uniq          # unique lines, sorted
sort names.txt | uniq -c       # prefix each with its count
sort names.txt | uniq -d       # show ONLY the lines that were duplicated

sort | uniq -c | sort -rn — "count each distinct value, then rank by frequency" — is one of the most useful three-command idioms in existence. It answers "what are the most common X?" for almost any X.

7.4 Extracting columns — cut

cut pulls out pieces of each line, either by delimiter-and-field or by character position:

cut -d',' -f1 data.csv        # field 1, splitting on commas (CSV)
cut -d',' -f1,3 data.csv      # fields 1 and 3
cut -d':' -f1 /etc/passwd     # usernames from the colon-separated passwd file
cut -c1-10 file.txt           # characters 1 through 10 of each line

-d sets the delimiter (default is Tab), and -f picks the fields. For neat, single-delimiter data it's perfect; for messier data, awk (below) is more flexible.

7.5 Transforming characters — tr

tr ("translate") swaps or deletes individual characters as text streams through it (it reads stdin, so pipe into it):

echo "hello" | tr 'a-z' 'A-Z'      # HELLO — uppercase
echo "hello world" | tr ' ' '_'    # hello_world — spaces to underscores
tr -d ' ' < file.txt               # delete all spaces
tr -s ' ' < file.txt               # squeeze runs of spaces into one
cat file | tr ',' '\n'             # turn a comma list into one-per-line

tr only works on single characters, not words — but for quick case changes and character swaps it's the lightest tool for the job.

7.6 A taste of sed and awk

Two heavyweight tools deserve a proper introduction, even though each could fill its own course. You don't need mastery — you need to recognize them and keep a few reliable one-liners.

sed ("stream editor") edits text as it flows past. Its famous trick is find-and-replace:

sed 's/old/new/' file.txt        # replace the FIRST "old" on each line with "new"
sed 's/old/new/g' file.txt       # replace ALL occurrences on each line (g = global)
sed 's/old/new/g' f > out.txt    # write the result to a new file
sed -i 's/old/new/g' file.txt    # edit the file IN PLACE ⚠️ (changes the original)
sed -n '10,20p' file.txt         # print only lines 10–20
sed '/^#/d' file.txt             # delete comment lines (those starting with #)

sed -i is powerful — it rewrites the file directly — so test without -i first and confirm the output looks right before committing to the in-place edit.

awk is a small programming language for column-oriented data. Its core model: for each line, it splits into fields $1, $2, … ($0 is the whole line), and runs your code:

awk '{print $1}' file            # print the first field of every line
awk '{print $1, $3}' file        # fields 1 and 3, space-separated
awk -F',' '{print $2}' data.csv  # use comma as the field separator
awk '{sum += $2} END {print sum}' data   # total up column 2
awk 'NR > 1' file                # skip the first line (a header row)
awk '$3 > 100' data              # print lines where column 3 exceeds 100

That sum += $2 ... END {print sum} pattern — accumulate a column and print the total at the end — is worth remembering; it turns a log or CSV into a single number instantly. awk rewards deeper study, but even these snippets handle a huge share of real tasks.

7.7 Feeding results into commands — xargs

Some commands read their targets from arguments, not stdin. xargs bridges the gap: it takes lines coming down a pipe and turns them into arguments for another command.

find . -name "*.log" | xargs wc -l          # count lines across all found logs
find . -name "*.tmp" | xargs rm             # delete every found .tmp file ⚠️
grep -rl "TODO" . | xargs wc -l             # line-count every file containing TODO

For filenames that might contain spaces, the safe form pairs find -print0 with xargs -0:

find . -name "*.log" -print0 | xargs -0 wc -l

find ... -exec and find ... | xargs overlap a lot; xargs is often faster for many files because it batches them into fewer command invocations.

7.8 A real example — analyzing a log the professional way

Back to that server log from Section 4 — now with the full toolkit. A cascade of real questions, each answered in one line:

# How many total errors?
grep -c "ERROR" app.log

# What are the 10 most common error messages?
grep "ERROR" app.log | cut -d']' -f2- | sort | uniq -c | sort -rn | head -n 10

# Which lines mention a specific user, with line numbers?
grep -n "user_id=4471" app.log

# How many distinct IP addresses hit us?
cut -d' ' -f1 app.log | sort -u | wc -l

# Show errors from today only, live as they arrive:
tail -f app.log | grep --line-buffered "ERROR"

None of these required writing a program, opening the file in an editor, or importing anything. That is the payoff of this entire course so far: a handful of sharp tools plus pipes let you interrogate data conversationally, one question at a time. This is exactly how engineers debug production incidents.

SECTION 8

The Shell Environment

You've been using the shell; now you'll configure it and understand how it thinks. This is where the terminal stops being a place you visit and becomes a place you live — with variables, history, aliases, and a startup file that makes it yours. Understanding the shell also demystifies a lot of "why did that happen?" moments.

8.1 Variables

The shell can store values in variables. Set one with name=value — crucially, no spaces around the =:

name="Arthur"
count=5

Read a variable back by prefixing its name with $:

echo "$name"          # Arthur
echo "Hello, $name"   # Hello, Arthur
echo "${name}s"       # Arthurs — braces separate the name from surrounding text

The ${...} brace form is safer when the variable name touches other characters. echo "$names" looks for a variable called names; echo "${name}s" correctly means "the value of name, then an s."

8.2 Environment variables and PATH

There are two kinds of variables. Shell variables exist only in your current shell. Environment variables are exported so that programs you launch inherit them:

export EDITOR="vim"        # now programs know your preferred editor
export API_URL="https://api.example.com"

See them all with env (or printenv NAME for one):

env | sort                 # every environment variable, sorted
echo "$HOME"               # your home directory
echo "$USER"               # your username
echo "$PATH"               # the command search path (below)

The most important environment variable is PATH — a colon-separated list of directories the shell searches, in order, whenever you type a command name:

echo "$PATH"

/usr/local/bin:/usr/bin:/bin:/home/arthur/.local/bin

When you type ls, the shell looks in each PATH directory until it finds an executable named ls, and runs the first one it finds. This is why "command not found" sometimes means "installed, but not in a directory on your PATH." You can see which one wins:

which ls          # /usr/bin/ls — the executable that will actually run
type ls           # similar, and also identifies builtins and aliases

To make your own scripts runnable from anywhere, you put them in a PATH directory (like ~/.local/bin) or add their directory to PATH — covered in Section 12 and used in the capstone.

8.3 Command substitution

You can capture the output of one command and use it as a value with $(...):

today=$(date +%Y-%m-%d)          # today = 2025-01-06
echo "Backup created on $today"

files=$(ls | wc -l)              # files = number of items in this directory
echo "There are $files files here"

echo "Kernel: $(uname -r)"       # embed a command's output right in a string

Command substitution is the glue of shell scripting — it's how a script asks the system a question and uses the answer. You'll lean on it heavily in Sections 10 and 11.

8.4 Quoting — the rules that prevent 90% of bugs

Quoting decides how the shell treats spaces and special characters. Getting it wrong is the #1 source of shell surprises, so learn the three cases clearly:

Double quotes "..." — group text but still expand $variables and $(commands):

name="Arthur Dev"
echo "Hello, $name"       # Hello, Arthur Dev   ← variable expands; space preserved

Single quotes '...' — completely literal; nothing expands:

echo 'Hello, $name'       # Hello, $name   ← printed exactly, no expansion
echo 'The cost is $5'     # The cost is $5 ← the $ is safe inside single quotes

No quotes — the shell splits on spaces and expands globs. This is why unquoted variables break on spaces:

file="my report.txt"
rm $file                  # ⚠️ becomes: rm my report.txt  → tries to delete TWO files!
rm "$file"                # correct: deletes the one file named "my report.txt"

The golden rule: quote your variables — write "$var", not $var — unless you have a specific reason not to. This single habit prevents a whole category of bugs, especially with filenames containing spaces. Use single quotes when you want a string left completely alone (like a $ price or a regex), double quotes when you want variables to expand inside.

8.5 Command history

Every command you run is remembered. This is a productivity goldmine.

history               # numbered list of past commands
history | tail -20    # the last 20
!!                    # re-run the previous command
!42                   # re-run command number 42 from history
!git                  # re-run the last command that started with "git"
sudo !!               # re-run the last command with sudo in front (a lifesaver)

The single best history trick is reverse search: press Ctrl+R and start typing part of a command you ran earlier. The shell finds the most recent match; press Ctrl+R again to step further back, then Enter to run it or → to edit it. Once this is in your fingers you'll stop retyping long commands forever.

$? holds the exit status of the last command — 0 means success, anything else means failure. $$ is the current shell's process ID, and $! is the PID of the last background job:

grep "needle" file; echo "$?"     # 0 if found, 1 if not found

8.6 Aliases — shortcuts for commands you type constantly

An alias is a short name that stands in for a longer command:

alias ll='ls -lah'          # now `ll` runs `ls -lah`
alias ..='cd ..'            # `..` goes up a directory
alias gs='git status'       # `gs` for git status
alias grep='grep --color=auto'   # always colorize grep output

Aliases you define at the prompt vanish when you close the terminal. To keep them, you put them in your shell's startup file — which is the next topic.

8.7 The startup file — .bashrc / .zshrc

When a shell starts, it runs a configuration script from your home directory: ~/.bashrc for bash, ~/.zshrc for zsh (check your shell with echo $SHELL). Anything in that file — aliases, exported variables, PATH additions, functions — takes effect in every new terminal. This is where you make the shell truly yours.

A typical addition looks like:

# ~/.bashrc

# My aliases
alias ll='ls -lah'
alias ..='cd ..'
alias gs='git status'

# Environment
export EDITOR="vim"
export PATH="$HOME/.local/bin:$PATH"    # put my own scripts on the PATH

# A friendlier prompt, history that remembers more
export HISTSIZE=10000

After editing the file, either open a new terminal or reload it in the current one with source, which runs the file in your existing shell:

source ~/.bashrc        # apply changes immediately, no need to reopen
# short form:
. ~/.bashrc

source (or its . shorthand) runs a script in the current shell rather than a new one, so its variables and aliases stick around. That distinction matters and comes up again in scripting.

8.8 A real example — a five-minute setup that pays off forever

The first thing many engineers do on a new machine is personalize the shell. Here's a small, sane starter set — add it to your ~/.bashrc and reload:

# --- Navigation ---
alias ll='ls -lah'
alias la='ls -A'
alias ..='cd ..'
alias ...='cd ../..'

# --- Safety nets ---
alias rm='rm -i'          # ask before deleting (remove once you're confident)
alias cp='cp -i'          # ask before overwriting
alias mv='mv -i'

# --- Git shortcuts ---
alias gs='git status'
alias ga='git add'
alias gc='git commit'
alias gp='git push'

# --- Quality of life ---
export EDITOR="nano"                       # or vim, or "code --wait"
export PATH="$HOME/.local/bin:$PATH"       # my scripts run from anywhere
export HISTSIZE=10000                      # remember more history

source ~/.bashrc     # live in the improved shell immediately

That's a few minutes of setup that saves you thousands of keystrokes over the following months. Your future self — the one typing ll and gs a hundred times a day — will thank you.

SECTION 9

Processes & Job Control

Every command you run becomes a process — a running program with its own ID, memory, and place in the system. Until now your commands finished in an instant. Real work involves programs that run for minutes or forever (servers, builds, downloads), and you need to see them, pause them, background them, and stop them. That's process and job control, and it's daily bread for anyone who runs software.

9.1 What a process is

When you launch a program, the operating system creates a process for it and assigns a PID (process ID) — a unique number. Processes can start other processes (their "children"), forming a tree. Each has an owner (a user), a state (running, sleeping, stopped), and resource usage (CPU, memory). Managing them means finding the right PID and sending it the right instruction.

9.2 Seeing what's running — ps, top, htop

ps takes a snapshot of processes:

ps                 # just your processes in this terminal
ps aux             # EVERY process on the system, detailed
ps aux | grep node # find processes related to "node"

ps aux | grep something is the everyday way to answer "is my server actually running, and what's its PID?" The columns you'll care about most are PID, %CPU, %MEM, and the COMMAND at the end.

For a live, continuously updating view — like Task Manager or Activity Monitor — use top:

top                # live process list, sorted by CPU. Press q to quit.

Inside top: it refreshes every few seconds; press q to quit, M to sort by memory, P to sort by CPU. Better still is htop (not always installed — Section 12), which adds color, easy scrolling, and click-to-kill:

htop               # friendlier top; F10 or q to quit

When a machine is slow or hot, top/htop instantly show you the culprit process eating the CPU.

9.3 Foreground, background, and jobs

A command normally runs in the foreground — it holds your terminal until it finishes. But you can detach long-running commands so you get your prompt back.

Add & to start a command in the background immediately:

sleep 60 &         # runs in the background; prompt returns at once

[1] 4823           # [job number] and PID

If a command is already running in the foreground and you want to move it aside, press Ctrl+Z to suspend (pause) it, then decide what to do:

jobs               # list background/suspended jobs in this shell
bg                 # resume the most recent suspended job IN the background
fg                 # bring a background job back to the FOREGROUND
fg %1              # bring job number 1 to the foreground

The full loop looks like this: you start a long download, realize you need the terminal, press Ctrl+Z to pause it, type bg to let it continue in the background, keep working, then fg to check on it later. jobs shows you what's parked.

Remember the distinction from earlier keys: Ctrl+C cancels a foreground program, while Ctrl+Z suspends it (you can resume it). Different tools for different intentions.

9.4 Stopping processes — kill and signals

To stop a process you send it a signal — a small message asking (or forcing) it to do something. kill sends signals by PID; despite the name, most of the time it's a polite request:

kill 4823          # send TERM (15): "please shut down cleanly"
kill -9 4823       # send KILL (9): "stop NOW" — the OS force-terminates it ⚠️
kill %1            # kill job number 1 (from jobs)

The two signals you'll actually use:

  • SIGTERM (15) — the default. It asks the program to terminate, giving it a chance to save state, close files, and clean up. Always try this first.
  • SIGKILL (9) — the sledgehammer. The kernel kills the process instantly; it gets no chance to clean up, which can leave corrupt files or orphaned resources. Use -9 only when a normal kill won't work.

By name rather than PID, pkill and killall match on the command:

pkill node         # signal every process whose name matches "node"
pkill -f "python server.py"    # match against the full command line

pkill -f is handy when several programs share a name and you need to target one by its arguments — but read your pattern carefully, because a broad match kills a lot at once.

9.5 Keeping a process alive — nohup

Normally, closing your terminal (or disconnecting from a server) kills the processes you started from it. When you need something to keep running after you log out — a long job on a remote machine, say — nohup detaches it from the terminal:

nohup ./long_job.sh &          # keeps running after you disconnect
nohup ./long_job.sh > job.log 2>&1 &   # ...and capture its output to a file

Output that would have gone to the terminal is saved to nohup.out (or the file you redirect to). On servers, this — or a terminal multiplexer like tmux, which you'll meet in Section 14 — is how long tasks survive a dropped connection.

9.6 Chaining commands by result — &&, ||, ;

You can run several commands on one line, and control whether later ones run based on earlier success. This rests on exit codes (Section 8): 0 means success, non-zero means failure.

cmd1 ; cmd2        # run cmd1, THEN cmd2 — always, regardless of success
cmd1 && cmd2       # run cmd2 ONLY IF cmd1 succeeded (exit 0)
cmd1 || cmd2       # run cmd2 ONLY IF cmd1 failed (non-zero exit)

These express real intent concisely:

mkdir build && cd build              # only cd if the mkdir worked
make && ./run                        # only run if the build succeeded
ping -c1 example.com && echo "up"    # only report "up" if the ping worked
command || echo "that failed"        # fallback message on failure
git pull && npm install && npm test  # a pipeline of steps, each gated on the last

&& is the workhorse — "do this, and only if it works, do the next thing" is exactly how you want multi-step operations to behave. You can inspect the last result manually too:

some_command
echo "$?"          # 0 = success, anything else = failure

9.7 A real example — managing a dev server

You're running a local server and it's misbehaving. The full cycle with these tools:

node server.js &                    # start it in the background
# ... it becomes unresponsive ...
ps aux | grep "node server.js"      # find its PID — say 5120
kill 5120                           # ask it to shut down cleanly
ps aux | grep "node server.js"      # confirm it's gone
# still there? escalate:
kill -9 5120                        # force it
# start fresh, keep it alive past this terminal, log its output:
nohup node server.js > server.log 2>&1 &
tail -f server.log                  # watch it come up (Ctrl-C stops watching, not the server)

Find it, ask it nicely, force it if you must, restart it detached, and watch the log. That sequence handles the vast majority of "my process is stuck" situations you'll ever face.

SECTION 10

Shell Scripting I: Fundamentals

Everything you've learned is about to compound. A shell script is just a file containing commands — the exact same commands you type at the prompt — that run one after another. Instead of typing a five-step process every time, you write it once and run it forever. This is where you stop being a command-line user and start being a command-line automator.

10.1 Your first script

A script is a plain text file. Create one:

# hello.sh
echo "Hello from a script!"
echo "Today is $(date)"
echo "You are $(whoami), in $(pwd)"

To run it, you have two options. The quick way is to pass it to the shell explicitly:

bash hello.sh          # run it with bash — works even without execute permission

The proper way — so you can run it like a real command — needs two things: a shebang line and execute permission (both below).

10.2 The shebang

The first line of a script should be a shebang — #! followed by the path to the interpreter that should run the file:

#!/usr/bin/env bash
echo "This script always runs with bash"

#!/usr/bin/env bash is the portable, recommended form: it asks the environment to find bash on the PATH, so the script works whether bash lives in /bin or /usr/local/bin. (You'll also see #!/bin/bash, which hard-codes the location — fine on most Linux, less portable.) The shebang is what lets the system run ./script.sh and know it's a bash script.

10.3 Making it executable and running it

Recall Section 5: a fresh file isn't executable. Give it the x bit, then run it with ./:

chmod +x hello.sh      # make it executable (once)
./hello.sh             # run it

Why ./? Because, for safety, the shell doesn't search your current directory for commands (that's not on your PATH). ./hello.sh explicitly says "run the hello.sh right here." Put the script in a PATH directory (Section 12) and you can drop the ./ and run it from anywhere by name.

10.4 Variables in scripts

Variables work exactly as they did at the prompt — and the quoting rules matter even more here:

#!/usr/bin/env bash
name="Arthur"
greeting="Hello"
echo "$greeting, $name!"           # Hello, Arthur!

# Capture command output into a variable
current_dir=$(pwd)
file_count=$(ls | wc -l)
echo "There are $file_count items in $current_dir"

Always quote your variables ("$name", not $name) — in a script that processes filenames or user input, an unquoted variable containing a space is a bug waiting to happen.

10.5 Reading input — arguments and read

Scripts become genuinely useful when they take input. There are two ways.

Command-line arguments — values passed when the script is launched — are available as numbered variables:

#!/usr/bin/env bash
echo "Script name: $0"        # the script itself
echo "First argument: $1"
echo "Second argument: $2"
echo "How many arguments: $#"
echo "All arguments: $@"

./greet.sh Arthur Dev
# Script name: ./greet.sh
# First argument: Arthur
# Second argument: Dev
# How many arguments: 2
# All arguments: Arthur Dev

$1, $2, … are the individual arguments; $0 is the script's own name; $# is the count; $@ is all of them. These are the standard way to make a script configurable.

Interactive input — read pauses and captures what the user types:

#!/usr/bin/env bash
read -p "What is your name? " name
echo "Hello, $name!"

read -p "Delete all logs? (y/n) " answer
echo "You chose: $answer"

read -p prints a prompt and stores the reply in a variable. Use arguments for scripts you'll run repeatedly or from other scripts, and read for interactive tools that ask questions.

10.6 Output — echo and printf

echo prints a line and is what you'll use most. When you need precise formatting — columns, numbers, no automatic newline — printf gives you control (it works like printf in C and many languages):

echo "Simple line"
echo -e "Tab\there\nNew line"       # -e interprets \t and \n
printf "%s is %d years old\n" "Ada" 36      # Ada is 36 years old
printf "%-10s %5d\n" "apples" 42            # left-align text, right-align number

printf needs an explicit \n for a newline (it doesn't add one automatically), which is exactly what makes it good for building neat tables and precise output.

10.7 Comments and safety settings

Comments start with # (everything after it on the line is ignored). Use them to explain why, not just what:

#!/usr/bin/env bash
# backup.sh — archive the project directory with a timestamp
# Usage: ./backup.sh <directory>

target="$1"     # the directory to back up

One line belongs near the top of almost every serious script — the safety trio:

set -euo pipefail

It changes bash from "keep going no matter what" to "fail fast and loudly," which is almost always what you want:

  • set -e — exit immediately if any command fails, instead of blindly continuing. Without this, a script whose first step fails will happily run the rest on broken assumptions.
  • set -u — treat use of an unset variable as an error, catching typos like $targett before they cause silent damage.
  • set -o pipefail — make a pipeline fail if any stage fails, not just the last one.

Adopt set -euo pipefail as a habit at the top of your scripts. It turns a whole class of silent, dangerous bugs into immediate, obvious errors.

10.8 A real example — a project backup script

Here's a small but genuinely useful script that ties the section together. It archives a directory into a timestamped .tar.gz and reports what it did:

#!/usr/bin/env bash
# backup.sh — create a timestamped compressed backup of a directory
# Usage: ./backup.sh <directory-to-back-up>
set -euo pipefail

# 1. Read and validate the argument
source_dir="$1"
timestamp=$(date +%Y%m%d_%H%M%S)
archive="backup_${timestamp}.tar.gz"

# 2. Create the archive (tar is covered in Section 13)
tar -czf "$archive" "$source_dir"

# 3. Report success with a size
size=$(du -h "$archive" | cut -f1)
echo "✅ Backed up '$source_dir' → $archive ($size)"

chmod +x backup.sh
./backup.sh my-project
# ✅ Backed up 'my-project' → backup_20250106_143022.tar.gz (2.4M)

Notice everything at work: a shebang, set -euo pipefail for safety, an argument ($1), command substitution ($(date …), $(du …)), a constructed filename with ${...}, quoted variables throughout, and a clear final message. This is a real script you could use today — and in the capstone you'll build several like it. In the next section you'll add the logic (conditionals and loops) that makes scripts truly smart.

SECTION 11

Shell Scripting II: Logic, Loops & Functions

A script that runs the same commands top to bottom is useful. A script that makes decisions and repeats work is powerful. This section adds the programming constructs — conditionals, loops, case, functions, arithmetic, and arrays — that turn a list of commands into a real tool. If you've programmed in any language, these ideas are familiar; bash's syntax is just quirkier, so pay attention to the spacing.

11.1 Conditionals — if and [[ ]]

An if statement runs a block only when a test succeeds. The bash syntax has punctuation you must get exactly right:

if [[ condition ]]; then
    # runs if condition is true
elif [[ other_condition ]]; then
    # runs if the first was false and this is true
else
    # runs if nothing above matched
fi

Two things trip up newcomers: the spaces inside [[ ]] are required ([[$x]] is an error; [[ $x ]] is correct), and the block closes with fi (if backwards). The modern [[ ]] is bash's test construct — prefer it over the older single-bracket [ ], which is more fragile with spaces and empty values.

Comparing numbers uses letter operators:

if [[ $age -ge 18 ]]; then echo "adult"; fi
Operator Meaning
-eq equal
-ne not equal
-gt / -lt greater than / less than
-ge / -le greater-or-equal / less-or-equal

Comparing strings uses symbols:

if [[ "$name" == "admin" ]]; then echo "welcome, admin"; fi
if [[ "$input" != "" ]]; then echo "you typed something"; fi
if [[ -z "$var" ]]; then echo "var is empty"; fi     # -z: empty?
if [[ -n "$var" ]]; then echo "var is set"; fi       # -n: non-empty?

Testing files is a bash superpower — checks that would take several lines in other languages:

if [[ -f config.txt ]]; then echo "the file exists"; fi     # -f: regular file exists
if [[ -d /var/log ]];   then echo "the directory exists"; fi # -d: directory exists
if [[ -e anything ]];   then echo "it exists (file or dir)"; fi
if [[ -x script.sh ]];  then echo "it's executable"; fi     # -x: executable
if [[ ! -f config.txt ]]; then echo "config is MISSING"; fi  # ! negates

That [[ -f ... ]] / [[ -d ... ]] family is how well-behaved scripts check their assumptions before acting — "does this file exist before I try to read it?"

11.2 Loops — for, while, until

A for loop repeats once for each item in a list. It shines at processing files:

for name in Alice Bob Carol; do
    echo "Hello, $name"
done

for file in *.txt; do              # loop over every .txt file
    echo "Processing $file..."
    wc -l "$file"
done

for i in {1..5}; do                # a numeric range
    echo "Iteration $i"
done

That middle example — loop over *.txt and do something to each — is the bread-and-butter of batch file processing. Quote the loop variable ("$file") so filenames with spaces survive.

A while loop repeats as long as a condition holds:

count=1
while [[ $count -le 3 ]]; do
    echo "count is $count"
    count=$((count + 1))           # arithmetic (below)
done

while also reads a file line by line — the standard idiom for processing input:

while read -r line; do
    echo "Line: $line"
done < input.txt

until is while's mirror image — it loops until a condition becomes true (useful for "wait for something"):

until ping -c1 server.example.com &> /dev/null; do
    echo "waiting for server..."
    sleep 2
done
echo "server is up!"

11.3 Choosing between many options — case

When you're checking one value against several possibilities, a chain of elifs gets ugly. case is cleaner:

case "$1" in
    start)
        echo "Starting the service..."
        ;;
    stop)
        echo "Stopping the service..."
        ;;
    restart)
        echo "Restarting..."
        ;;
    *)                              # * is the catch-all default
        echo "Usage: $0 {start|stop|restart}"
        ;;
esac

Each branch is a pattern, ends with ;;, and *) catches anything unmatched. case closes with esac (case backwards, keeping the theme). This is exactly how service scripts and CLI tools dispatch on a subcommand — you've likely typed something start and something stop at a case statement without knowing it.

11.4 Functions

A function is a named, reusable block — the same idea as in any language. Define it once, call it by name:

greet() {
    echo "Hello, $1!"              # $1 is the function's first argument
}

greet "Arthur"                     # Hello, Arthur!
greet "World"                      # Hello, World!

Functions get arguments the same way scripts do ($1, $2, $@, $#). Use local for variables that should stay inside the function, and return for a numeric status (0–255) or echo to "return" data:

file_count() {
    local dir="$1"                 # local: doesn't leak outside the function
    local count
    count=$(ls "$dir" | wc -l)
    echo "$count"                  # "return" a value by printing it
}

n=$(file_count /etc)               # capture it with command substitution
echo "There are $n things in /etc"

Functions keep scripts readable and DRY (don't repeat yourself): give a chunk of logic a clear name, and the main body of your script reads like a summary of what it does.

11.5 Arithmetic

Bash does integer math inside $(( )):

sum=$(( 3 + 4 ))              # 7
product=$(( 6 * 7 ))          # 42
remainder=$(( 17 % 5 ))       # 2  (modulo)
count=$(( count + 1 ))        # increment
echo "$(( (2 + 3) * 4 ))"     # 20  — parentheses group as usual

The operators are the familiar + - * / %. Note that bash arithmetic is integer only — $(( 7 / 2 )) is 3, not 3.5. For decimals you'd pipe to a tool like bc (echo "7 / 2" | bc -l), but most scripting needs only integers (counters, sizes, indices).

11.6 Arrays

An array holds multiple values under one name:

fruits=("apple" "banana" "cherry")

echo "${fruits[0]}"           # apple  — indexing starts at 0
echo "${fruits[@]}"           # apple banana cherry  — all elements
echo "${#fruits[@]}"          # 3  — how many elements
fruits+=("date")              # append an element

for fruit in "${fruits[@]}"; do    # iterate (quote it!)
    echo "- $fruit"
done

The syntax is dense — ${array[@]} for all elements, ${#array[@]} for the count — but you don't need arrays for simple scripts. Reach for them when you're collecting a list of things (filenames, results) to process together.

11.7 A real example — a smart backup script

Now combine Section 10's script with this section's logic into something that validates its input, handles errors, and reports clearly:

#!/usr/bin/env bash
# backup.sh — safely back up a directory to a timestamped archive
# Usage: ./backup.sh <directory>
set -euo pipefail

# --- validate input ---
if [[ $# -ne 1 ]]; then
    echo "Usage: $0 <directory-to-back-up>" >&2
    exit 1
fi

source_dir="$1"

if [[ ! -d "$source_dir" ]]; then
    echo "Error: '$source_dir' is not a directory." >&2
    exit 1
fi

# --- do the work ---
timestamp=$(date +%Y%m%d_%H%M%S)
archive="backup_${timestamp}.tar.gz"

echo "Backing up '$source_dir'..."
tar -czf "$archive" "$source_dir"

# --- report ---
size=$(du -h "$archive" | cut -f1)
echo "✅ Done: $archive ($size)"

./backup.sh                 # Usage: ./backup.sh <directory-to-back-up>   (exit 1)
./backup.sh nonexistent     # Error: 'nonexistent' is not a directory.    (exit 1)
./backup.sh my-project      # ✅ Done: backup_20250106_143022.tar.gz (2.4M)

The difference from the Section 10 version is all judgment: it checks the argument count, verifies the directory exists, writes errors to stderr (>&2), and exits with a non-zero code on failure so other scripts can tell it went wrong. That's the line between a snippet and a tool — and it's exactly what you'll build in the capstone.

SECTION 12

Package Management & Installing Software

So far you've used tools that come with the system. Real work means installing more — tree, htop, jq, language runtimes, databases. On Linux you almost never download installers from websites; instead a package manager fetches vetted software and its dependencies from official repositories with one command. This section shows you how, and clears up sudo, PATH, and where programs actually live.

12.1 What a package manager does

A package manager is the app store of the command line, but better: it knows about dependencies (if tool A needs library B, it installs B too), it keeps everything updatable from one place, and it removes software cleanly. You ask for a name; it handles the rest. Which manager you use depends on your operating system.

12.2 The major package managers

System Manager Install a package
Ubuntu / Debian / WSL apt sudo apt install NAME
Fedora / RHEL / CentOS dnf sudo dnf install NAME
Arch / Manjaro pacman sudo pacman -S NAME
macOS brew (Homebrew) brew install NAME

They differ in spelling but share the same verbs: update the list of available software, install, remove, search. Learn the shape once and you can work on any of them. This course uses apt (Ubuntu/Debian/WSL) in examples since it's the most common for newcomers; translate as needed.

12.3 Using apt (Debian/Ubuntu/WSL)

The everyday commands:

sudo apt update                 # refresh the list of available packages (do this first)
sudo apt install tree           # install a package
sudo apt install htop jq curl   # install several at once
sudo apt remove tree            # uninstall
sudo apt upgrade                # update all installed packages to newer versions
apt search markdown             # search for packages (no sudo needed to search)
apt show htop                   # details about a package

The one that catches beginners: run sudo apt update before installing. It refreshes the local catalog; without it, apt install may not find a package or may fetch a stale version. The rhythm is sudo apt update && sudo apt install NAME.

On macOS with Homebrew the equivalents are brew update, brew install NAME, brew uninstall NAME, brew upgrade, and brew search NAME — same ideas, no sudo needed because Homebrew installs into a directory you own.

12.4 sudo and root, properly

Installing system software modifies shared directories, so it requires administrator rights. sudo ("superuser do") runs a single command as the all-powerful root user, after checking you're allowed and asking for your password:

sudo apt install tree      # prompts for your password the first time

Three habits keep sudo safe:

  • Read the whole command before running it with sudo. Root bypasses the permission safety nets that normally protect you — a mistake as root can damage the system.
  • Don't sudo things that don't need it. If a command works without sudo, don't add it. Reflexively prefixing everything with sudo creates files owned by root that you then can't edit (and have to chown back — Section 5).
  • Never run random internet commands with sudo without understanding them. curl … | sudo bash hands root to a script you haven't read. Look first.

The prompt itself tells you your power level: $ means a normal user, # means you're root. Seeing # is a cue to be extra deliberate.

12.5 Where do installed programs go? — which, type, whereis

When you install something, the package manager drops an executable into a directory on your PATH (Section 8), which is why you can then just type its name. To see exactly where a command lives:

which htop         # /usr/bin/htop — the executable that runs when you type "htop"
type htop          # like which, but also spots aliases and shell builtins
whereis htop       # the binary, its source, and its man page locations
command -v htop     # portable "is this installed / where is it?" (great in scripts)

command -v NAME is the script-friendly way to check whether a tool is installed before using it:

if command -v jq &> /dev/null; then
    echo "jq is installed"
else
    echo "please install jq first"
fi

Recall that PATH is searched in order. If which shows an unexpected location, or a command "isn't found" though you installed it, the fix is usually a PATH issue — the install directory isn't listed, or another version earlier in PATH is winning.

12.6 Putting your own tools on the PATH

This is how your scripts graduate from "run with ./" to "run from anywhere by name" — the last step toward a real toolkit. Create a personal bin directory, add it to PATH in your startup file, and drop scripts there:

mkdir -p ~/.local/bin                       # a place for your scripts
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc                            # apply now

# now install one of your scripts:
cp backup.sh ~/.local/bin/backup            # copy (drop the .sh for a clean name)
chmod +x ~/.local/bin/backup
backup my-project                           # runs from ANY directory, no ./ needed

~/.local/bin is a conventional spot for personal executables and is already on the PATH on many systems. Putting it early in PATH ($HOME/.local/bin:$PATH) means your version wins if a name clashes. You'll do exactly this in the capstone to install your toolkit.

12.7 Dotfiles — your setup, portable

All this personalization — aliases, exported variables, PATH tweaks, functions — lives in dotfiles like ~/.bashrc, ~/.gitconfig, and ~/.ssh/config. Engineers keep these in a Git repository (a "dotfiles repo") so a brand-new machine becomes their machine in minutes: clone the repo, symlink the files, done. You don't need to build that today, but know the concept: your environment is just text files, which means it's versionable, shareable, and reproducible. It's a common early portfolio project, and a natural extension of the capstone.

12.8 A real example — setting up a fresh machine

The first ten minutes on a new Linux box or WSL install often look like this:

# 1. Refresh and update everything
sudo apt update && sudo apt upgrade -y

# 2. Install the tools this course loves
sudo apt install -y tree htop jq curl git tldr

# 3. Set up a personal scripts directory
mkdir -p ~/.local/bin
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc

# 4. Add a few aliases (Section 8)
cat >> ~/.bashrc << 'EOF'
alias ll='ls -lah'
alias ..='cd ..'
alias gs='git status'
EOF

# 5. Apply it all
source ~/.bashrc

# 6. Confirm a tool installed correctly
which tree && tree --version

In six steps the machine is updated, stocked with good tools, and personalized. Wrap steps 3–5 into a script and you've automated your own onboarding — which is, again, a great capstone extension.

SECTION 13

Networking & Remote Machines

The command line's real power shows up when the machine you're working on isn't in front of you. Servers, cloud instances, and deployment targets live somewhere else, and you reach them entirely through the terminal — no screen, no mouse, just a text connection. This section covers connecting to remote machines with SSH, moving files between them, fetching things from the web, and packaging directories into archives to ship around.

13.1 SSH — logging into a remote machine

SSH ("Secure Shell") opens an encrypted terminal session on another computer. Once connected, you're typing commands on that machine as if you were sitting at it:

ssh username@hostname            # connect (hostname can be a domain or IP)
ssh arthur@203.0.113.10          # by IP address
ssh arthur@server.example.com    # by domain name
ssh arthur@server.example.com -p 2222    # a non-default port with -p

The first time you connect to a new machine, SSH asks you to confirm its fingerprint (type yes) — this pins the server's identity so you'd be warned if it were ever impersonated. After connecting, your prompt changes to reflect the remote host; everything you type runs there. To return to your own machine, type exit or press Ctrl+D.

You can also run a single command remotely without starting an interactive session:

ssh arthur@server.example.com "df -h"        # run df -h there, print result here
ssh arthur@server.example.com "tail -n 50 /var/log/app.log"

13.2 SSH keys — logging in without a password

Typing a password every time is slow and less secure than the alternative: key-based authentication. You generate a key pair — a private key that stays secret on your machine and a public key you copy to the server. The server lets you in if your private key matches the public one it has, no password needed.

Generate a modern key pair once:

ssh-keygen -t ed25519 -C "you@example.com"

It asks where to save (accept the default, ~/.ssh/id_ed25519) and for an optional passphrase (recommended — it encrypts the private key on disk). This creates two files: ~/.ssh/id_ed25519 (private — never share or copy this) and ~/.ssh/id_ed25519.pub (public — safe to distribute).

Copy the public key to a server so it trusts you:

ssh-copy-id arthur@server.example.com        # the easy way — installs your public key

After that, ssh arthur@server.example.com logs you straight in. This is also exactly how you authenticate to GitHub and GitLab over SSH — you paste the public key into your account settings.

Guard your private key. SSH requires it to be readable only by you, or it refuses to use it:

chmod 600 ~/.ssh/id_ed25519      # rw for owner only (recall Section 5)

That 600 isn't arbitrary — a private key readable by others is a security hole, and SSH enforces the rule by rejecting loosely-permissioned keys.

13.3 The SSH config file — shortcuts for machines

Typing full user@host -p port strings gets old. ~/.ssh/config lets you define friendly aliases:

# ~/.ssh/config
Host prod
    HostName 203.0.113.10
    User arthur
    Port 2222
    IdentityFile ~/.ssh/id_ed25519

Now ssh prod expands to the full connection. You can define as many Host blocks as you like — one per server you touch — turning cryptic addresses into memorable names. It's one of those small setups that quietly saves time every day.

13.4 Copying files to and from remote machines — scp and rsync

scp ("secure copy") transfers files over SSH, using the same user@host:path syntax. The remote side just gains a colon and a path:

scp local.txt arthur@server:/home/arthur/        # local → remote
scp arthur@server:/var/log/app.log ./            # remote → local
scp -r myproject/ arthur@server:/home/arthur/    # a whole directory (-r)

rsync is scp's smarter cousin, and what professionals reach for on anything bigger than a file or two. It copies only what's changed, can resume interrupted transfers, and can mirror directories exactly:

rsync -av myproject/ arthur@server:/home/arthur/myproject/   # archive + verbose
rsync -avz --progress big/ arthur@server:/backup/            # -z compress, show progress
rsync -av --delete site/ arthur@server:/var/www/site/        # mirror (delete extras) ⚠️

The -a ("archive") flag preserves permissions, timestamps, and structure; -v is verbose; -z compresses during transfer. Because rsync skips unchanged files, re-running it after a small edit copies almost nothing — ideal for deploys and backups. The --delete flag makes the destination an exact mirror by removing files that no longer exist in the source, so double-check the paths before using it.

13.5 Fetching from the web — curl and wget

Two tools download from URLs. curl prints to stdout by default (so it pipes beautifully) and is the Swiss-army knife for talking to web services and APIs:

curl https://example.com                         # print the page to the terminal
curl -O https://example.com/file.zip             # save as the remote filename (-O)
curl -o myname.zip https://example.com/file.zip  # save under a name you choose (-o)
curl -L https://example.com                      # follow redirects (-L)
curl -s https://api.example.com/data | jq        # silent, pipe JSON into jq

wget is oriented toward downloading files (and whole sites) and saves to disk by default:

wget https://example.com/file.zip                # download to the current directory
wget -c https://example.com/big.iso              # -c: resume a partial download

Rule of thumb: curl for interacting with APIs and piping data; wget for grabbing files. Both are everywhere; curl especially shows up constantly in install instructions and scripts.

13.6 Archiving and compression — tar, gzip, zip

To move a directory across the network as one unit — or to back it up — you bundle it into an archive. On Unix, tar ("tape archive") packs many files into one, and gzip compression shrinks it. You'll see the same handful of flag combinations forever:

tar -czf project.tar.gz project/     # CREATE a gzipped archive (c=create z=gzip f=file)
tar -tzf project.tar.gz              # LIST contents without extracting (t=list)
tar -xzf project.tar.gz              # EXTRACT here (x=extract)
tar -xzf project.tar.gz -C /tmp/out  # extract into a specific directory (-C)

The letters decode simply: create / extract / list, z for gzip, f for "the filename follows." A memory hook: "create zezip file" and "extract zezip file." You can also exclude noise while packing:

tar -czf clean.tar.gz --exclude='*.log' --exclude='node_modules' project/

For single files, gzip/gunzip compress in place:

gzip bigfile.txt        # creates bigfile.txt.gz, removes the original
gunzip bigfile.txt.gz   # restores bigfile.txt

And when you need to exchange with Windows users, zip/unzip produce the cross-platform format:

zip -r project.zip project/    # create a .zip (recursive)
unzip project.zip              # extract it

13.7 A quick look at connectivity

A few commands help when "the network is weird." You don't need mastery — just enough to check basic reachability:

ping -c4 example.com     # send 4 packets — is the host reachable, and how fast?
ip addr                  # your machine's network interfaces and IP addresses
                         # (older systems: ifconfig)
curl -I https://example.com    # fetch just the HTTP headers — is the site responding?
ss -tulpn                # listening ports and what's using them (older: netstat)

ping answers "can I even reach it?", curl -I answers "is the web server responding?", and ss/netstat answers "what's listening on this machine?" — the first questions in most connectivity debugging.

13.8 A real example — deploying to a server

Watch the whole toolkit combine into a realistic deploy. You build locally, ship the files, and restart the service remotely:

# 1. Package the app locally, excluding junk
tar -czf release.tar.gz --exclude='node_modules' --exclude='.git' myapp/

# 2. Copy it to the server (thanks to ~/.ssh/config, just "prod")
scp release.tar.gz prod:/home/arthur/

# 3. Log in and deploy — a chain of commands run remotely
ssh prod "cd /home/arthur && tar -xzf release.tar.gz && sudo systemctl restart myapp"

# 4. Confirm it came back up by checking the log
ssh prod "tail -n 20 /var/log/myapp.log"

Package, transfer, extract, restart, verify — five lines, no graphical tools, fully repeatable (and one step away from being a deploy script, per Sections 10–11). This is genuinely close to how software reaches production every day.

Real-World Workflows & Productivity

You now have every core skill. This final teaching section is about fluency — the habits, tools, and glue that separate someone who can use the command line from someone who lives in it: editing files on a server, keeping sessions alive, scheduling work, reading logs under pressure, and debugging when a command misbehaves. 14.1 Editing files in the terminal — nano and vim survival

Sometimes there's no graphical editor — you're on a server and a config needs a one-line change. You need at least survival-level terminal editing.

nano is the friendly option; its shortcuts are listed right at the bottom of the screen (where ^ means Ctrl):

nano config.txt
  • Type normally to edit.
  • Ctrl+O then Enter — save ("write Out").
  • Ctrl+X — exit.
  • Ctrl+W — search.

If you have any choice, nano is the sane pick for quick edits.

vim is everywhere — it's on essentially every Unix machine ever — so you need enough to survive being dropped into it (it sometimes opens as the default editor without warning). Its trick is modes: you start in Normal mode (keys are commands), and press i to enter Insert mode (keys type text):

vim config.txt
  • Press i to start inserting text; type normally.
  • Press Esc to leave Insert mode and return to Normal mode.
  • In Normal mode, type :w Enter to save, :q Enter to quit.
  • :wq saves and quits; :q! quits without saving (the escape hatch when you've made a mess).

The famous "how do I exit vim?" answer: press Esc, then type :q! and Enter. Memorize just that and you'll never be trapped. Learning vim properly pays off enormously over a career, but survival-level is enough for now.

14.2 Staying alive on servers — tmux

When you SSH into a server and run something long, closing your laptop or dropping the connection normally kills it (recall nohup in Section 9). tmux (terminal multiplexer) solves this more powerfully: it runs a persistent session on the server that keeps going whether or not you're connected. Reconnect later and everything is exactly as you left it.

tmux                 # start a new session
tmux new -s work     # start a named session ("work")
# ... run things, then detach without stopping them:
#     press Ctrl-b, then d   (detach)
tmux ls              # list running sessions
tmux attach -t work  # reattach to "work" later, even from a different machine

The key idea: detach (Ctrl+b then d) leaves everything running; attach picks up where you left off. tmux also splits one terminal into multiple panes (Ctrl+b then % or "), so you can watch a log in one and work in another. On a remote server, starting your work inside tmux is the habit that saves you from losing a long job to a flaky connection.

14.3 Scheduling work — cron

cron runs commands automatically on a schedule — every night, every hour, every Monday. You edit your personal schedule (the "crontab") with:

crontab -e           # edit your scheduled jobs
crontab -l           # list them

Each line is five time fields followed by the command:

┌ minute (0–59)
│ ┌ hour (0–23)
│ │ ┌ day of month (1–31)
│ │ │ ┌ month (1–12)
│ │ │ │ ┌ day of week (0–6, Sun=0)
│ │ │ │ │
* * * * *  command-to-run

Real examples:

0 2 * * *      /home/arthur/.local/bin/backup /home/arthur/project   # 2:00 AM daily
*/15 * * * *   /home/arthur/check-health.sh                          # every 15 minutes
0 9 * * 1      /home/arthur/weekly-report.sh                         # 9:00 AM every Monday

The * means "every," and */15 means "every 15th." This is how the backup script you're about to build in the capstone becomes a backup system — write the script once, schedule it with cron, and it protects you forever without another thought.

14.4 Reading logs under pressure

When something breaks in production, the logs hold the answer, and everything from Sections 4, 6, and 7 converges here. The moves you'll make, roughly in order:

tail -n 100 /var/log/app.log            # what happened most recently?
tail -f /var/log/app.log                # watch it live as you reproduce the bug
grep -i "error" /var/log/app.log | tail -n 20      # the recent errors specifically
grep -c "error" /var/log/app.log        # how bad is it? (count)
grep "error" app.log | cut -d']' -f2- | sort | uniq -c | sort -rn | head   # top errors
less /var/log/app.log                   # scroll and /search interactively

The instinct to build is: start recent (tail), filter to what matters (grep), quantify (wc/uniq -c), then drill in (less, more specific grep). You're not reading the whole log — you're asking it sharper and sharper questions until it confesses.

14.5 Debugging a command that misbehaves

When a command doesn't do what you expect, a short checklist resolves most cases:

  • Read the error message. It usually names the problem — "No such file," "Permission denied," "command not found." Each points somewhere specific (wrong path, need chmod +x, not installed / not on PATH).
  • Check the exit code: echo "$?" — non-zero confirms it failed even if the output looked fine.
  • Look for the file/path: does it actually exist and are you where you think? ls -la and pwd.
  • Check permissions: ls -l — is the x bit there for a script? Do you own the file?
  • Echo your variables: if a script misbehaves, echo "$var" to see what they actually contain (empty? unexpected spaces?). Unquoted variables with spaces are a classic culprit.
  • Slow the pipeline down: build a long pipeline one stage at a time, checking the output after each |, so you can see exactly which stage goes wrong.
  • Trace a script: run it with bash -x script.sh to print each command as it executes, with variables expanded — you'll see precisely where it diverges from your intent.

That last one, bash -x, is the scripting equivalent of a debugger: it shows the script's actual behavior line by line. Most "why doesn't this work?" moments fall to read the error → check path/permissions → echo the variables → trace it.

15.3 Tool 1 — backup.sh

The requirements: take a directory, produce a timestamped .tar.gz, exclude noise like .git and node_modules, default to a sensible destination, validate input, and record each run in a log. Here's a complete, working implementation — read it and notice how every piece is something you've learned:

#!/usr/bin/env bash
# backup.sh — create a timestamped, compressed backup of a directory.
# Usage: backup.sh <directory> [destination]
set -euo pipefail

DEFAULT_DEST="$HOME/backups"
LOG_FILE="$HOME/backups/backup.log"
EXCLUDES=(".git" "node_modules" "__pycache__" "*.log" "*.tmp")

log() {
    local message="$1"
    echo "$message"
    mkdir -p "$(dirname "$LOG_FILE")"
    echo "$(date '+%Y-%m-%d %H:%M:%S')  $message" >> "$LOG_FILE"
}

die() { echo "Error: $1" >&2; exit 1; }

if [[ $# -lt 1 || $# -gt 2 ]]; then
    echo "Usage: $(basename "$0") <directory> [destination]" >&2
    exit 1
fi

source_dir="$1"
dest_dir="${2:-$DEFAULT_DEST}"
[[ -d "$source_dir" ]] || die "'$source_dir' is not a directory."

mkdir -p "$dest_dir"
name="$(basename "$(cd "$source_dir" && pwd)")"
timestamp="$(date +%Y%m%d_%H%M%S)"
archive="$dest_dir/${name}_${timestamp}.tar.gz"

exclude_args=()
for pattern in "${EXCLUDES[@]}"; do
    exclude_args+=(--exclude="$pattern")
done

log "Backing up '$source_dir' → $archive"
tar -czf "$archive" "${exclude_args[@]}" -C "$(dirname "$source_dir")" "$name"

size="$(du -h "$archive" | cut -f1)"
log "✅ Backup complete: $archive ($size)"

Make it executable and try it:

chmod +x backup.sh
./backup.sh ~/some-project
# Backing up '/home/arthur/some-project' → /home/arthur/backups/some-project_20250106_143022.tar.gz
# ✅ Backup complete: ... (2.4M)

The techniques on display: a log function that prints and appends to a file, a die helper for clean errors, ${2:-$DEFAULT_DEST} to supply a default argument, an array of excludes turned into --exclude flags with a loop, command substitution for the name and size, and set -euo pipefail guarding the whole thing. This is what a real script looks like.

15.4 Tool 2 — sysinfo.sh

A dashboard you can run any time to see how the machine is doing. The design choice worth noting: it uses set -uo pipefail without -e, and a have() helper, so that if one probe (like free or htop) isn't available, that section is skipped instead of crashing the whole report.

#!/usr/bin/env bash
# sysinfo.sh — a compact dashboard of the machine's current state.
set -uo pipefail

have() { command -v "$1" >/dev/null 2>&1; }
header() { printf '\n%s\n' "== $1 =="; }

header "SYSTEM"
printf "%-12s %s\n" "Hostname:" "$(hostname)"
printf "%-12s %s\n" "User:"     "$(whoami)"
printf "%-12s %s\n" "Date:"     "$(date '+%Y-%m-%d %H:%M:%S')"
have uname && printf "%-12s %s\n" "Kernel:" "$(uname -sr)"

have uptime && { header "UPTIME & LOAD"; uptime; }
have df      && { header "DISK USAGE"; df -h --output=source,size,used,avail,pcent 2>/dev/null || df -h; }
have free    && { header "MEMORY"; free -h; }

header "TOP PROCESSES (by CPU)"
ps -eo pid,comm,%cpu,%mem --sort=-%cpu 2>/dev/null | head -n 6 || ps aux | head -n 6

Run ./sysinfo.sh and you get a clean, aligned report — printf with %-12s does the column alignment (Section 10). This is the kind of thing you'll run first thing when a server feels slow.

15.5 Tool 3 — newproject.sh

Scaffolding a new project by hand is repetitive — exactly what scripts are for. This one takes a name and an optional language, builds a layout, writes a README and .gitignore, and makes the first commit. Its heart is a case statement (Section 11) that switches on the language:

#!/usr/bin/env bash
# newproject.sh — scaffold a new project and initialise git.
# Usage: newproject.sh <name> [python|node|bash|generic]
set -euo pipefail

if [[ $# -lt 1 || $# -gt 2 ]]; then
    echo "Usage: $(basename "$0") <name> [python|node|bash|generic]" >&2
    exit 1
fi

name="$1"
language="${2:-generic}"
[[ -e "$name" ]] && { echo "Error: '$name' already exists." >&2; exit 1; }

mkdir -p "$name"/{src,tests,docs}
cd "$name"

cat > README.md <<EOF
# $name

A new $language project, scaffolded on $(date +%Y-%m-%d).
EOF

case "$language" in
    python) echo 'print("Hello!")' > src/main.py
            printf '%s\n' "__pycache__/" "*.pyc" ".venv/" > .gitignore ;;
    node)   echo 'console.log("Hello!");' > src/index.js
            printf '%s\n' "node_modules/" "*.log" ".env" > .gitignore ;;
    bash)   printf '%s\n' '#!/usr/bin/env bash' 'echo "Hello!"' > src/main.sh
            chmod +x src/main.sh
            printf '%s\n' "*.log" "*.tmp" > .gitignore ;;
    generic) printf '%s\n' "*.log" ".DS_Store" > .gitignore ;;
    *) echo "Unknown language '$language'." >&2; exit 1 ;;
esac

if command -v git >/dev/null 2>&1; then
    git init -q && git add -A && git commit -q -m "Initial scaffold of $name"
fi

echo "✅ Created '$name' ($language)."

(The full version in the capstone folder handles the git-not-configured edge case and prints the resulting tree.) Notice the here-documents (cat > README.md <<EOF …) writing multi-line files, and how the case keeps each language's setup tidy and separate. Adding a new language is just another case branch — the script is built to grow.

15.6 Tool 4 — log-analyzer.sh

This one is pure Sections 6–7 — a pipeline factory. It reports total lines, counts by level, ranks the top error messages, and (if the log looks like a web log) finds the busiest IPs:

#!/usr/bin/env bash
# log-analyzer.sh — summarise a text log file.
# Usage: log-analyzer.sh <logfile>
set -euo pipefail

[[ $# -eq 1 ]] || { echo "Usage: $(basename "$0") <logfile>" >&2; exit 1; }
log="$1"
[[ -f "$log" ]] || { echo "Error: '$log' is not a file." >&2; exit 1; }

echo "Total lines: $(wc -l < "$log")"

echo "By level:"
for level in ERROR WARN INFO DEBUG; do
    count=$(grep -c -w "$level" "$log" || true)
    [[ "$count" -gt 0 ]] && printf "  %-8s %6d\n" "$level" "$count"
done

errors=$(grep -w "ERROR" "$log" || true)
if [[ -n "$errors" ]]; then
    echo "Top 5 error messages:"
    printf '%s\n' "$errors" | sed -E 's/^\[[^]]*\] *//' \
        | sort | uniq -c | sort -rn | head -n 5 | sed 's/^/  /'
fi

The centrepiece is the "count and rank" idiom you met in Section 7 — sort | uniq -c | sort -rn | head — aimed at error messages, with a sed step to strip timestamps so identical errors group together. The || true after grep -c is a small but important detail: grep exits non-zero when it finds nothing, which under set -e would kill the script, so we neutralise that.

15.7 Tool 5 — the shell setup

Finally, the aliases and functions that make your shell fast. Put these in toolkit_aliases.sh and source it from ~/.bashrc:

# toolkit_aliases.sh — source from ~/.bashrc
alias ll='ls -lah'
alias ..='cd ..'
alias gs='git status'

mkcd() { mkdir -p "$1" && cd "$1"; }          # make a dir and enter it

extract() {                                    # unpack any archive
    case "$1" in
        *.tar.gz|*.tgz) tar -xzf "$1" ;;
        *.tar)          tar -xf  "$1" ;;
        *.zip)          unzip    "$1" ;;
        *) echo "extract: don't know how to unpack '$1'" >&2; return 1 ;;
    esac
}

mkcd and extract are the kind of tiny functions that quietly save you dozens of keystrokes a day — a function because they need logic an alias can't express.

15.8 Install it — make your toolkit real

Now bring it all together with everything from Sections 8 and 12. Put the scripts on your PATH and load the aliases, so the tools work from any directory by name:

# 1. Put the scripts on your PATH
mkdir -p ~/.local/bin
cp ~/cli-toolkit/*.sh ~/.local/bin/
chmod +x ~/.local/bin/*.sh

# 2. Ensure ~/.local/bin is on PATH, and load the aliases
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc
cp ~/cli-toolkit/toolkit_aliases.sh ~/.toolkit_aliases
echo '[ -f ~/.toolkit_aliases ] && source ~/.toolkit_aliases' >> ~/.bashrc

# 3. Apply it
source ~/.bashrc

# 4. Use your tools from anywhere
sysinfo.sh
backup.sh ~/projects/app

Then schedule the backup so it runs itself every night (Section 14):

crontab -e
# add:
0 2 * * * $HOME/.local/bin/backup.sh $HOME/projects/app

You now have an automated backup system, a health dashboard, a project generator, and a log analyser — installed, on your PATH, and (for the backup) running on a schedule without you.

15.9 Ship it and extend it

Commit everything, write the README.md (what each tool does, how to install, how to use — see the capstone folder for a model), and push it to GitHub. That repository is a real portfolio piece: it demonstrates Bash, the Unix toolset, error handling, and good documentation, all at once.

When you want to go further, natural extensions include: a --help flag for each script; a config file the tools read; a deploy.sh that packages and rsyncs to a server (Section 13); colourised output; or turning your whole ~/.bashrc, aliases, and configs into a dotfiles repo you can clone onto any new machine. Each is a small, satisfying step from where you are now.

Where this fits

The command line isn't a topic you "finish" — it's the environment the rest of software engineering happens in. Everything you learn next will lean on it:

  • Git lives at the command line; you now have the fluency to use it without friction.
  • Docker and containers are configured and run through the terminal, using exactly the Linux concepts you just learned.
  • CI/CD pipelines are, at heart, shell scripts that run on someone else's machine.
  • Cloud servers (AWS, GCP, Azure) are reached over SSH and administered with these commands.
  • Debugging production means reading logs and inspecting processes — skills you now have.

Every one of those becomes dramatically easier because you're comfortable here.

What to learn next

A few natural directions, roughly in order of payoff:

  • Get properly good at one editor. Push past survival-level vim, or set up nano/VS Code the way you like. You edit constantly; fluency compounds.
  • Learn tmux well. Persistent sessions and split panes change how you work on servers.
  • Go deeper on regular expressions. They power grep, sed, awk, and nearly every language. A focused week pays off for a career.
  • Level up your scripting. Read other people's scripts, learn to use trap for cleanup, and study a style guide like Google's Shell Style Guide. When Bash starts to strain, that's your cue to reach for Python.
  • Build your dotfiles repo. Version your ~/.bashrc, aliases, and configs so any new machine becomes yours in minutes. It's a great habit and a nice portfolio piece.
  • Explore containers. With this Linux foundation, Docker will feel like a natural next step rather than a mysterious one.

A closing thought

The command line rewards use, not study. You don't get faster by reading about grep; you get faster by reaching for it a hundred times until your fingers know it. So keep the terminal open. Solve your next small annoyance with a pipeline instead of a mouse. Turn the three-command thing you did twice today into a script. Each small choice to stay on the command line makes the next one easier — until one day you realise the black window you were once scared of has become the most comfortable, powerful tool you own.

Welcome to the command line. You live here now.
