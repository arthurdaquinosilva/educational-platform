The Ultimate JavaScript Mastery — A Self-Study Course

JavaScript from the very beginning to the modern features used in production today. We cover the language itself, the DOM and browser APIs (where JS shines), and Node.js for backend work. Assumes you've done the Programming Foundations course — if loops, functions, and recursion already feel natural, you'll move fast.

How to use this course

  • You don't need to install anything. JavaScript runs in every browser. Your browser's developer console (right-click → Inspect → Console tab) is a REPL that's always available.
  • For larger scripts, use a code editor — VS Code is the de facto choice for JS development. You'll also want Node.js installed (from nodejs.org) for the final two sections. Node includes npm, which you'll meet later.
  • Type every example yourself. The muscle memory pays off, especially with JavaScript's many small operators and method names.
  • Open the dev console while reading sections that involve live demos — you can poke at the iframe's JavaScript directly.
  • Each section follows the same format: Concept → Syntax → Example → Exercise with solution.

This course doesn't repeat what you learned in Foundations. Loops, variables, conditionals, functions — those concepts transfer. What we cover here is JavaScript's particular way of doing each thing, and the features specific to JS and the browser.

SECTION 1

The JavaScript Story

JavaScript has the strangest history of any major programming language. Understanding it explains everything about the language — including the quirks that make people laugh, cry, and write 600-page books titled JavaScript: The Good Parts.

1.1 Ten Days in May, 1995

In 1995, Netscape was the dominant web browser. Web pages were static — HTML in, HTML out, no interactivity. Netscape wanted to add a scripting language so designers could make pages respond to clicks and form input without going through a server round-trip.

They hired Brendan Eich to design it. He was told two things:

  1. Make it look like Java. Java was the new hot language; partnerships with Sun Microsystems were valuable. The language should feel familiar to Java programmers.
  2. You have ten days.

Eich delivered. The language was originally called Mocha, then LiveScript, then renamed to JavaScript in December 1995 — purely a marketing decision to ride Java's coattails. JavaScript has roughly as much to do with Java as ham has to do with hamster. The naming has confused everyone ever since.

The result of those ten days was a language with brilliant ideas (first-class functions, closures, object literals, prototypal inheritance) and terrible ones (silent type coercion, == semantics, no module system, global scope by default). We're still living with all of it.

1.2 The Browser Wars

In 1996, Microsoft cloned the language for Internet Explorer, calling their version JScript. The two implementations diverged. Web developers had to write conditional code to handle browser differences. The pain became famous.

In 1997, ECMAScript was created — a standardized version of the language, named after the European Computer Manufacturers Association (Ecma) that maintained the spec. JavaScript is technically just the most popular implementation of ECMAScript. The terms are used interchangeably in practice.

The standard went through versions: - ES1 (1997), ES2 (1998), ES3 (1999) — minor refinements. - ES4 (2000s) — a hugely ambitious revision. Politically derailed. Never shipped. - ES5 (2009) — modest but important: Array.prototype.map, Object.create, strict mode, JSON. - ES6 / ES2015 — the massive modernization. let, const, arrow functions, classes, modules, promises, template literals, destructuring, default parameters, spread/rest operators. Everything before ES6 we'd now call "legacy JavaScript." Everything after is the language people learn today. - ES2016 onwards — yearly releases with smaller additions. async/await (2017), optional chaining ?. (2020), nullish coalescing ?? (2020), top-level await (2022).

When you see "modern JavaScript," it usually means ES6+. This course teaches modern JS, but we'll occasionally show legacy patterns to explain why things look the way they do.

1.3 JavaScript Escapes the Browser

For its first 14 years, JavaScript only ran in browsers. Then in 2009, Ryan Dahl built Node.js — a way to run JavaScript on a server using Google's V8 engine (the same engine Chrome uses). Suddenly JavaScript wasn't just a browser language; it was a general-purpose language for backends, scripts, build tools, and command-line apps.

Node's surprise success had two consequences:

  1. npm (Node Package Manager) became the largest package registry on Earth. Millions of packages. Whatever you want to do, someone has already published a library for it (and probably six competing ones).
  2. JavaScript developers became "full-stack." The same language now powered the frontend (browser), the backend (Node), and even build tooling. This is the modern web's whole story.

Today JavaScript runs in: - Every browser (the original target) - Node.js and its rivals (Deno, Bun) - Mobile apps via React Native - Desktop apps via Electron (VS Code, Slack, Discord, WhatsApp Desktop are all Electron) - Game engines (PlayCanvas, Babylon.js) - Servers, scripts, build tools, anything

JavaScript is, by usage volume, the most-deployed programming language in history. The ten-day language won.

If you're coming from Python (via the Foundations course), you'll find JS familiar in many ways but surprising in others:

SECTION 2

Setup & Your First Programs

JavaScript is everywhere. There's nothing to install for the first dozen sections of this course — every browser has JavaScript built in.

2.1 Three Places to Run JavaScript

You have three options, from easiest to most powerful:

1. The browser console — instant, zero setup.

Right-click anywhere on a webpage → Inspect → Console tab. (Or press F12 on Windows/Linux, Cmd+Option+I on Mac.) Type any JavaScript and press Enter. This is your REPL.

> 2 + 2
4
> let name = "Alice"
> `Hello, ${name}`
"Hello, Alice"

The console is brilliant for experimentation. It's also dangerous on real websites — if a site's developer was sloppy, console code runs with full access to whatever the page can do. Don't paste random code into the console of your bank's website. Browsers actually warn you about this if you open the console.

2. An HTML file with a <script> tag — for anything beyond a one-liner.

Create hello.html:

<!DOCTYPE html>
<html>
<head>
  <title>My First JS</title>
</head>
<body>
  <h1>Hello!</h1>
  <script>
    console.log("Hello from JavaScript!");
    document.querySelector("h1").textContent = "Hello from JS too!";
  </script>
</body>
</html>

Open the file in your browser. Open the dev console — you'll see "Hello from JavaScript!" printed. The <h1> will say "Hello from JS too!" because the script changed it.

3. Node.js — running JavaScript outside the browser.

Install Node from nodejs.org. Then in your terminal:

$ node
Welcome to Node.js v20.x.x.
> 2 + 2
4
> .exit

Or for scripts, create hello.js:

console.log("Hello from Node!");
const numbers = [1, 2, 3, 4, 5];
const total = numbers.reduce((a, b) => a + b, 0);
console.log("Sum:", total);

Run it: node hello.js. We'll come back to Node in Sections 18-19.

2.2 console.log — JS's print

console.log("Hello!");
console.log(42);
console.log("Pi is approximately", 3.14159);
console.log({ name: "Alice", age: 30 });    // objects show their structure

console.log is the JS equivalent of Python's print. Differences:

  • It accepts any number of arguments, joining them with spaces.
  • It pretty-prints objects and arrays — much more useful than Python's default string conversion.
  • There are siblings: console.warn, console.error, console.info, console.table, console.group. Try console.table([{a:1,b:2},{a:3,b:4}]) in the browser console.
  • In browsers, it writes to the dev console. In Node, it writes to stdout.

2.3 Where Does Your Script Run?

In the browser, the <script> tag runs as the browser encounters it while parsing HTML. Three common placements:

<!-- In <head> — runs BEFORE the body is parsed -->
<head>
  <script>
    // The DOM doesn't exist yet — this can't manipulate the body!
    document.querySelector("h1"); // returns null
  </script>
</head>

<!-- At the end of <body> — runs after the DOM is parsed -->
<body>
  <h1>Hello</h1>
  <script>
    document.querySelector("h1");  // works
  </script>
</body>

<!-- External file with defer — runs after DOM is parsed, in order -->
<head>
  <script src="app.js" defer></script>
</head>

The modern recommendation: external files with defer, in the <head>. The browser starts downloading the script early but waits to execute it until the page is parsed.

For inline scripts in tiny examples, putting them at the bottom of <body> works fine.

2.4 Comments

// Single-line comment

/*
  Multi-line comment.
  Spans as many lines as you want.
*/

/**
 * JSDoc-style comment.
 * Used to document functions formally.
 * @param {number} x - a number
 * @returns {number} the square
 */

  return x * x;
}

Standard slash-based comments. The JSDoc style is conventional for documenting library functions; editors use it for autocomplete and type hints.

2.5 Semicolons — Optional, But...

JavaScript inserts semicolons automatically at the end of most statements. This is called Automatic Semicolon Insertion (ASI):

let x = 5      // semicolon inserted automatically
let y = 10     // ditto
console.log(x + y)

Works fine. But ASI has weird edge cases that cause bugs:

return
{
  value: 42
}
// You expect this to return the object.
// Instead it returns undefined — ASI inserts a semicolon after `return`.
// The object becomes an unreachable block.

Fix:

return {
  value: 42
}
// or with explicit semicolons:
return {
  value: 42
};

Practical advice: there are two valid styles — "always use semicolons" and "never use semicolons (let ASI handle it)." Both work as long as you're consistent. The wider community is split, but always use semicolons is the safer default. We'll use them in this course.

If you choose no-semicolons style, learn the ASI gotchas — especially lines starting with (, [, or `, which need a leading semicolon to disambiguate.

Create greet.html:

<button id="myButton">Greet me</button>
<p id="output"></p>

  const button = document.getElementById("myButton");
  const output = document.getElementById("output");

  button.addEventListener("click", function() {
    output.textContent = "Hello, JavaScript developer!";
  });

Result

SECTION 3

Variables & Types

JavaScript looks similar to Python here, but with several distinctive features. We'll cover the three ways to declare variables (and why two of them are obsolete), the type system, and the type coercion that's caused 30 years of memes.

3.1 Three Ways to Declare a Variable

There are three keywords: let, const, and var. Modern JS uses let and const. var is legacy — you should know it exists, but don't use it.

let age = 30;            // can be reassigned
const name = "Alice";    // cannot be reassigned
var oldStyle = "legacy"; // works, but don't use

Use const by default. Most variables don't need to be reassigned. Constant-by-default is one of those subtly powerful disciplines that prevents bugs.

Use let when you genuinely need to reassign. Counters, accumulators, anything that updates.

Never use var. It has surprising scoping rules (function-scoped, not block-scoped) and gets hoisted. It only exists for backwards compatibility.

const PI = 3.14159;
const greeting = "Hello";

let count = 0;
count = count + 1;        // OK, let allows reassignment

const x = 5;
x = 10;                   // TypeError: Assignment to constant variable.

Important nuance: const prevents reassignment, not mutation. You can still modify the contents of an object or array declared with const:

const items = [1, 2, 3];
items.push(4);             // OK — modifying the array
console.log(items);        // [1, 2, 3, 4]

items = [];                // TypeError — reassigning the binding

const says "this name will always point to this particular object," not "this object will never change." To truly freeze an object: Object.freeze(items).

3.2 Block Scope

let and const are block-scoped — they exist only within the {...} block they're declared in:

if (true) {
  let secret = 42;
  console.log(secret);   // 42
}
console.log(secret);     // ReferenceError — secret doesn't exist out here

var, by contrast, was function-scoped — visible throughout the entire enclosing function, even before its declaration. This caused endless bugs. ES6's let/const fixed it.

3.3 The Basic Types

JavaScript has a small set of primitive types:

Type Examples Notes
number 42, 3.14, -7, NaN, Infinity All numbers are floats (IEEE 754 doubles)
string "hello", 'hi', `template` Three quote styles
boolean true, false
undefined undefined "Variable declared but no value assigned"
null null "Intentionally no value"
bigint 42n Arbitrary-precision integers (ES2020)
symbol Symbol("id") Unique identifiers (advanced)

Plus one non-primitive type:

Type Examples
object {name: "Alice"}, [1, 2, 3], function() {}, new Date()

Almost everything that isn't a primitive is an "object" in JavaScript's type system — arrays, functions, dates, regexes, instances of any class.

You can ask any value its type with typeof:

typeof 42;             // "number"
typeof "hello";        // "string"
typeof true;           // "boolean"
typeof undefined;      // "undefined"
typeof null;           // "object"   ← BUG. Yes really. Read on.
typeof {};             // "object"
typeof [];             // "object"
typeof function(){};   // "function" — special case

typeof null === "object" is a famous bug. It dates back to JavaScript v1 in 1995. Fixing it would break the web, so it stays. To check for null, use value === null directly.

3.4 null vs undefined

These are two distinct values that beginners confuse.

undefined — automatically assigned. Means "this hasn't been given a value."

let x;
console.log(x);                  // undefined — declared but not initialized

function greet(name) {
  console.log(name);
}
greet();                         // undefined — argument not passed

const obj = { a: 1 };
console.log(obj.b);              // undefined — property doesn't exist

null — manually assigned. Means "deliberately no value."

let user = null;                 // "we don't have a user yet"
function findUser(id) {
  return null;                   // "looked, but no user found"
}

Practical rule: let undefined happen naturally. Use null only when you explicitly want to signal "no value." Many codebases get away without null entirely.

3.5 Numbers — All Floats

Unlike Python (which has separate int and float), JavaScript has just one number type, and it's a float.

typeof 42;          // "number"
typeof 3.14;        // "number"
typeof 0;           // "number"
typeof -Infinity;   // "number"
typeof NaN;         // "number" ← NaN means "Not a Number" but is itself a number

This means JS inherits all of IEEE 754's quirks (see Foundations Section 17):

0.1 + 0.2 === 0.3   // false! Result is 0.30000000000000004

For large integers (over 2⁵³), you can lose precision:

9007199254740992 + 1 === 9007199254740993   // false!

For these cases, ES2020 added BigInt:

const big = 9007199254740992n + 1n;          // the `n` makes it a BigInt
console.log(big);                            // 9007199254740993n

BigInts can hold arbitrary integers but can't mix with regular numbers without explicit conversion. You rarely need them outside of cryptography or financial code.

Useful number tricks:

parseInt("42px")       // 42 — parses leading integer, ignores the rest
parseFloat("3.14e2")   // 314 — handles scientific notation
Number("42")           // 42 — strict conversion
Number("abc")          // NaN
+"42"                  // 42 — the unary plus does Number() conversion
isNaN(NaN)             // true
isFinite(Infinity)     // false
Math.round(3.7)        // 4
Math.floor(3.7)        // 3
Math.ceil(3.2)         // 4
Math.random()          // a float between 0 and 1

3.6 Strings

Three ways to write a string:

const a = "double-quoted";
const b = 'single-quoted';
const c = `template literal (backticks)`;

All work the same — but template literals (backticks) have superpowers:

const name = "Alice";
const age = 30;

// Old way:
const message1 = "Hello, " + name + "! You are " + age + ".";

// Template literal:
const message2 = `Hello, ${name}! You are ${age}.`;

// Multi-line:
const poem = `
  Roses are red,
  Violets are blue,
  Template literals
  Are useful too.
`;

${...} embeds any expression. Multi-line strings need no special syntax — newlines work directly.

String methods you'll use constantly:

"hello".length;                         // 5 — length is a property, not a method
"hello".toUpperCase();                  // "HELLO"
"hello".toLowerCase();                  // "hello"
"  hello  ".trim();                     // "hello"
"hello".includes("ell");                // true
"hello".startsWith("he");               // true
"hello".endsWith("lo");                 // true
"hello".indexOf("l");                   // 2 — first occurrence, or -1
"hello".replace("l", "L");              // "heLlo" — first only
"hello".replaceAll("l", "L");           // "heLLo" — all occurrences
"a,b,c".split(",");                     // ["a", "b", "c"]
"hello".repeat(3);                      // "hellohellohello"
"hello"[0];                             // "h" — indexing
"hello".charAt(0);                      // "h" — older equivalent
"hello".slice(1, 4);                    // "ell" — substring

Strings are immutable. Methods return new strings:

const s = "hello";
s.toUpperCase();
console.log(s);     // "hello" — still lowercase!
const upper = s.toUpperCase();  // assign the result

3.7 Type Coercion — The Famous Part

JavaScript has two equality operators and a system of implicit type conversion. This is the source of every "WTF JS" meme.

Strict equality (===) — compares value AND type, no conversion:

5 === 5            // true
5 === "5"          // false (number vs string)
true === 1         // false
null === undefined // false

Loose equality (==) — coerces types to "match" first:

5 == "5"           // true — string "5" gets converted to number 5
true == 1          // true — true coerces to 1
false == 0         // true
null == undefined  // true — but ONLY each other; null == false is false
"" == false        // true
" " == 0           // true — whitespace becomes 0
[] == false        // true — empty array becomes "" becomes 0

The rule: use === and !==. Always. The legacy == and != have intricate coercion rules that even experienced JS developers can't fully recite from memory. Modern style guides and linters all flag == as an error.

3.8 Truthy and Falsy

In a boolean context (an if, a while condition, the &&/|| operators), every value is either truthy or falsy.

Falsy values (only 7 of them — memorize these):

false
0           // and -0
0n          // BigInt zero
""          // empty string
null
undefined
NaN

Everything else is truthy — including some surprises:

"false"     // truthy (non-empty string)
" "         // truthy (string with one space)
[]          // truthy (empty array!)
{}          // truthy (empty object!)
"0"         // truthy (string with "0" in it)
new Boolean(false)  // truthy (the wrapper object, not the value)

This differs from Python, where empty [], {}, "", 0, and None are all falsy. In JS, empty arrays and objects are truthy.

To test if an array is empty, check its length:

const arr = [];
if (arr) { ... }          // ALWAYS true — empty array is truthy!
if (arr.length > 0) { }   // correct way

To convert any value to a real boolean explicitly:

Boolean("hello")   // true
Boolean("")        // false
!!"hello"          // true   — the double-negation trick
!!""               // false

The !!x idiom converts any value to its truthy/falsy boolean equivalent. You'll see it constantly.

3.9 Type Conversion

When you mix types, JS converts them. Sometimes that's nice. Sometimes it's the meme.

Number to String:

String(42)                // "42"
(42).toString()           // "42"
42 + ""                   // "42" — concatenation forces string

String to Number:

Number("42")              // 42
Number("42px")            // NaN — strict
parseInt("42px")          // 42 — lenient
parseFloat("3.14abc")     // 3.14
+"42"                     // 42 — unary plus

To Boolean:

Boolean(0)                // false
Boolean("hello")          // true
!!value                   // common idiom

The famous JS quirks (now you'll understand them):

"5" + 3                   // "53"   — + with string coerces both to strings
"5" - 3                   // 2      — - only works on numbers; string converts
1 + "1"                   // "11"
1 - "1"                   // 0

[] + []                   // ""     — arrays convert to strings, which is ""
[] + {}                   // "[object Object]"
true + true               // 2      — true is 1, 1+1=2

null + 1                  // 1      — null converts to 0
undefined + 1             // NaN    — undefined can't convert to number

These are all consistent, just not always intuitive. The simple defense: use ===, write explicit conversions, never mix types in arithmetic.

3.10 Variables — Naming Rules

Same conventions as most languages: - Letters, digits, underscores, dollar signs. - Cannot start with a digit. - Case-sensitive.

JavaScript convention is camelCase for variables and functions:

const userName = "Alice";              // ✓
const TotalScore = 100;                // ✗ — looks like a class
const total_score = 100;               // ✗ — Python style, not JS

PascalCase is reserved for classes and constructors:

class UserAccount { ... }

SCREAMING_SNAKE_CASE for top-level constants:

const MAX_RETRIES = 3;
const API_BASE_URL = "https://api.example.com";

$ and _ are allowed but conventionally have specific meanings: $ for DOM elements (jQuery legacy), _ for "unused" or for library-private variables (Lodash uses _ as its main identifier).

SECTION 4

Operators & Control Flow

The basics — arithmetic, comparisons, conditionals, loops — look similar to Python with a few notable twists. We'll cover everything quickly and dwell on the JS-specific operators (??, ?., ||=).

4.1 Arithmetic Operators

Standard:

2 + 3              // 5
10 - 4             // 6
3 * 4              // 12
10 / 3             // 3.3333... — always returns a float
10 % 3             // 1 — modulo
2 ** 10            // 1024 — exponentiation (ES2016)

JavaScript does NOT have floor division (// in Python). For integer division:

Math.floor(10 / 3)  // 3
Math.trunc(10 / 3)  // 3 (truncates toward zero, differs from floor for negatives)
~~(10 / 3)          // 3 — using bitwise NOT twice; clever but obscure

Increment / decrement:

let x = 5;
x++;      // x is now 6 (post-increment)
++x;      // x is now 7 (pre-increment)
x--;      // 6

The difference between x++ and ++x only matters when used inside an expression:

let a = 5;
console.log(a++);  // 5 — returns OLD value, then increments
console.log(a);    // 6

let b = 5;
console.log(++b);  // 6 — increments first, then returns
console.log(b);    // 6

In practice, prefer x += 1 for clarity. Reserve ++/-- for tight loop counters where it's idiomatic.

Augmented assignment:

x += 5;   // x = x + 5
x -= 2;
x *= 3;
x /= 4;
x %= 2;
x **= 2;

4.2 Comparison Operators

5 === 5        // true  — strict equality (always use this)
5 !== 4        // true  — strict inequality
5 == "5"       // true  — loose equality (avoid)
5 != "5"       // false — loose inequality (avoid)
5 > 3          // true
5 < 3          // false
5 >= 5         // true
5 <= 4         // false

Always use === and !==. The loose versions do type coercion you don't want.

4.3 Logical Operators

true && true       // true — AND
true && false      // false
true || false      // true — OR
!true              // false — NOT

Just like Python — except for one critical difference. && and || don't return booleans; they return one of the operands.

"hello" && "world"   // "world" — returns the LAST truthy if all truthy, else first falsy
"" && "world"        // ""      — short-circuits on first falsy
null || "default"    // "default" — returns first TRUTHY value, or last falsy if all falsy
"hello" || "world"   // "hello"
0 || "x" || null     // "x"

This is hugely useful as a default value idiom:

function greet(name) {
  name = name || "stranger";
  console.log(`Hello, ${name}!`);
}
greet();           // "Hello, stranger!"
greet("Alice");    // "Hello, Alice!"

But this has a subtle problem — what if you want 0 or "" to be valid values?

function setVolume(level) {
  level = level || 50;
  console.log(level);
}
setVolume(0);      // 50 — oops, 0 is falsy, so the default fires

That's where the nullish coalescing operator comes in.

4.4 Nullish Coalescing (??)

The ?? operator (ES2020) is like ||, but only treats null and undefined as "missing" — not 0, "", or false.

null ?? "default"        // "default"
undefined ?? "default"   // "default"
0 ?? "default"           // 0      (not nullish, so kept)
"" ?? "default"          // ""     (not nullish, kept)
false ?? "default"       // false  (not nullish, kept)

Fixed version of the volume example:

function setVolume(level) {
  level = level ?? 50;
  console.log(level);
}
setVolume(0);      // 0 — correct!
setVolume();       // 50 — default applies for actual missing value

Practical rule: use ?? when only null/undefined should trigger the default. Use || only when you actively want any falsy value to trigger the default (rare).

4.5 Optional Chaining (?.)

?. (ES2020) lets you safely access deeply-nested properties without crashing on null/undefined:

const user = { profile: { name: "Alice" } };
const otherUser = {};

// Old way:
const name = user && user.profile && user.profile.name;          // "Alice"
const missing = otherUser && otherUser.profile && otherUser.profile.name;  // undefined

// Optional chaining:
const name2 = user?.profile?.name;            // "Alice"
const missing2 = otherUser?.profile?.name;    // undefined — no crash

If anything before ?. is null or undefined, the whole expression short-circuits to undefined.

Works for function calls too:

obj.maybeMethod?.();          // calls only if it exists
arr?.[0];                     // safe array access

Combined with ??, you get the modern "safe lookup with default" pattern:

const username = user?.profile?.name ?? "Anonymous";

Beautiful, and reads exactly like English. Pre-2020 JS had to use three lines of && checks to do the same thing.

4.6 Logical Assignment Operators

ES2021 added three more shortcuts:

x ||= 5;    // x = x || 5    — assign only if x is falsy
x &&= 5;    // x = x && 5    — assign only if x is truthy
x ??= 5;    // x = x ?? 5    — assign only if x is null or undefined

Most useful is ??=:

const settings = { theme: "dark" };
settings.fontSize ??= 16;     // adds default for missing prop
settings.theme ??= "light";   // doesn't overwrite existing "dark"
console.log(settings);        // { theme: "dark", fontSize: 16 }

4.7 The Ternary Operator

JS's conditional expression looks like Python's but with different syntax:

// Python: x = "adult" if age >= 18 else "minor"
// JS:
const status = age >= 18 ? "adult" : "minor";

Form: condition ? valueIfTrue : valueIfFalse. Read it as "if condition then this else that."

Chainable, but readability suffers fast:

const grade = score >= 90 ? "A"
            : score >= 80 ? "B"
            : score >= 70 ? "C"
            : score >= 60 ? "D"
            : "F";

For long chains, an if/else if ladder is usually clearer.

4.8 if, else, else if

Same shape as most languages:

if (score >= 90) {
  grade = "A";
} else if (score >= 80) {
  grade = "B";
} else if (score >= 70) {
  grade = "C";
} else {
  grade = "F";
}

Conditions go in parentheses. Blocks go in braces. You can omit braces for single-statement blocks, but don't — they prevent a classic bug:

// Risky:
if (logged_in)
  doThing();
  doOtherThing();   // ← always runs, regardless of logged_in!

// Safe:
if (logged_in) {
  doThing();
  doOtherThing();
}

Always use braces. Linters universally enforce this.

4.9 The switch Statement

For comparing one value against many possibilities:

const day = "Tuesday";
switch (day) {
  case "Monday":
  case "Tuesday":
  case "Wednesday":
  case "Thursday":
  case "Friday":
    console.log("Weekday");
    break;
  case "Saturday":
  case "Sunday":
    console.log("Weekend");
    break;
  default:
    console.log("Unknown");
}

break is mandatory. Without it, execution "falls through" to the next case — sometimes intentional (as above, where multiple cases share code), often a bug. Always include break.

switch uses strict equality (===) for matching. No type coercion happens here.

In modern JS, switch is used less than you'd expect. A Map or object lookup is often cleaner:

const dayType = {
  Monday: "Weekday", Tuesday: "Weekday", Wednesday: "Weekday",
  Thursday: "Weekday", Friday: "Weekday",
  Saturday: "Weekend", Sunday: "Weekend",
};
console.log(dayType[day] || "Unknown");

4.10 Loops

while and for are nearly identical to Python:

let i = 0;
while (i < 5) {
  console.log(i);
  i++;
}

// Classic C-style for loop:
for (let i = 0; i < 5; i++) {
  console.log(i);
}

The C-style for(init; condition; update) is JS's traditional counted loop. ES6 added two newer styles:

const arr = ["a", "b", "c"];

// for...of — iterate values (like Python's "for x in arr")
for (const item of arr) {
  console.log(item);     // "a", "b", "c"
}

// for...in — iterate KEYS (rarely what you want for arrays)
for (const i in arr) {
  console.log(i);        // "0", "1", "2" — STRING indices!
}

Critical gotcha: never use for...in to iterate arrays. It iterates string keys, includes inherited properties, and order isn't guaranteed. Use for...of for arrays, or .forEach() / .map() (Section 6).

for...in is for objects (Section 7), and even then, Object.keys() is usually clearer.

break and continue work as you'd expect:

for (let i = 0; i < 10; i++) {
  if (i === 3) continue;     // skip 3
  if (i === 7) break;        // stop at 7
  console.log(i);
}
// 0, 1, 2, 4, 5, 6

4.11 do...while

A loop that runs at least once before checking the condition:

let answer;
do {
  answer = prompt("Type yes:");
} while (answer !== "yes");

Rarely used in modern code. while (true) { ... break ... } is more common when you need this shape.

4.12 The Spread Operator (...)

... in JavaScript means "spread out the contents." It works on arrays, strings, and (since ES2018) objects:

// Spread an array into function arguments
const nums = [3, 7, 1];
Math.max(...nums);              // 7 — same as Math.max(3, 7, 1)

// Combine arrays
const a = [1, 2];
const b = [3, 4];
const combined = [...a, ...b];  // [1, 2, 3, 4]

// Copy an array
const copy = [...nums];          // shallow copy

// Combine objects (ES2018)
const defaults = { theme: "light", fontSize: 14 };
const user = { theme: "dark" };
const settings = { ...defaults, ...user };
// { theme: "dark", fontSize: 14 } — later objects override

In other contexts, the same ... syntax is called rest — it gathers values into an array:

function sum(...nums) {           // collect all args into an array
  return nums.reduce((a, b) => a + b, 0);
}
sum(1, 2, 3, 4);                 // 10

Spread/rest is one of ES6's most used features. You'll see it everywhere.

4.13 Destructuring Assignment

ES6 added a way to unpack values from arrays and objects:

// Array destructuring
const [first, second, third] = [10, 20, 30];
console.log(first, second, third);    // 10 20 30

// Skip values:
const [, , c] = [1, 2, 3];           // c = 3

// Rest:
const [head, ...tail] = [1, 2, 3, 4];
// head = 1, tail = [2, 3, 4]

// Defaults:
const [a = 100, b = 200] = [5];
// a = 5, b = 200

// Object destructuring:
const user = { name: "Alice", age: 30, city: "Lisbon" };
const { name, age } = user;
console.log(name, age);              // "Alice" 30

// Rename while destructuring:
const { name: userName } = user;
console.log(userName);               // "Alice"

// Defaults work here too:
const { country = "Portugal" } = user;
// country = "Portugal" (not in user, so default)

Destructuring is hugely common in modern JS, especially in function parameters:

function greet({ name, age }) {
  console.log(`${name} is ${age}`);
}
greet({ name: "Alice", age: 30 });

This pattern — passing an "options object" and destructuring it inside the function — is the modern alternative to long positional parameter lists.

SECTION 5

Functions

Functions are JavaScript's superpower. Everything is built on them. JS has more ways to declare a function than any sane language needs, and each has subtle differences. We'll cover all of them and the modern recommendations.

5.1 Function Declarations

The traditional form:

function greet(name) {
  return `Hello, ${name}!`;
}

console.log(greet("Alice"));    // "Hello, Alice!"

Function declarations are hoisted — JavaScript scans for them before executing your code, so you can call them before they appear:

greet("Alice");                 // works — function is hoisted

function greet(name) {
  return `Hello, ${name}!`;
}

This is occasionally useful but mostly confusing. Modern code style declares before use.

5.2 Function Expressions

You can assign a function to a variable. The function is then a value:

const greet = function(name) {
  return `Hello, ${name}!`;
};

console.log(greet("Alice"));

Function expressions are not hoisted the same way. The variable greet exists from the top of the scope, but its value is set at the = line. Calling it before that line gives undefined is not a function.

You can also name a function expression for self-reference:

const factorial = function fact(n) {
  return n <= 1 ? 1 : n * fact(n - 1);
};

The inner name fact is only visible inside the function. Outside, only factorial works.

5.3 Arrow Functions

ES6 added a compact alternative — arrow functions:

const square = (n) => {
  return n * n;
};

// Single parameter: parens optional
const square = n => {
  return n * n;
};

// Single expression: braces and `return` can be dropped
const square = n => n * n;

// No parameters: empty parens required
const greet = () => "Hello!";

// Multiple parameters: parens required
const add = (a, b) => a + b;

// Returning an object: wrap in parens (otherwise braces look like a block)
const makeUser = name => ({ name, id: 1 });

Arrow functions are concise — perfect for callbacks:

[1, 2, 3].map(n => n * 2);                  // [2, 4, 6]
[1, 2, 3].filter(n => n > 1);                // [2, 3]
[1, 2, 3].reduce((sum, n) => sum + n, 0);    // 6

But arrows have a critical difference: they don't have their own this. This matters in classes and event handlers. We'll cover this carefully in Section 7. For now: use arrows for short callbacks; use regular function declarations for methods and standalone functions.

5.4 Parameters & Arguments

Default parameter values:

function greet(name = "stranger") {
  console.log(`Hello, ${name}!`);
}
greet();              // "Hello, stranger!"
greet("Alice");       // "Hello, Alice!"

Defaults can reference earlier parameters:

function makeUser(name, displayName = name.toUpperCase()) {
  return { name, displayName };
}
makeUser("alice");    // { name: "alice", displayName: "ALICE" }

Rest parameters — collect any number of arguments into an array:

function sum(...nums) {
  return nums.reduce((a, b) => a + b, 0);
}
sum(1, 2, 3);            // 6
sum(1, 2, 3, 4, 5);      // 15

Default + rest combined:

function greet(greeting = "Hello", ...names) {
  return names.map(n => `${greeting}, ${n}!`).join("\n");
}
greet("Hi", "Alice", "Bob", "Carol");
// "Hi, Alice!\nHi, Bob!\nHi, Carol!"

The legacy arguments object — every regular function has an implicit arguments variable containing all passed arguments:

function legacy() {
  console.log(arguments[0]);          // first argument
  console.log(arguments.length);
}
legacy("a", "b", "c");

arguments is array-LIKE but not an actual array. It doesn't work in arrow functions (one of their key differences). Use ...args instead — it gives you a real array.

5.5 Returning Values

Same as everywhere: return exits the function with a value. Without a return, the function returns undefined:

function noReturn() {
  console.log("running");
}
const result = noReturn();
console.log(result);                  // undefined

You can return early:

function divide(a, b) {
  if (b === 0) return null;
  return a / b;
}

JS functions can only return one value, but you can return an object or array to "return multiple values":

function minMax(arr) {
  return { min: Math.min(...arr), max: Math.max(...arr) };
}

const { min, max } = minMax([3, 1, 4, 1, 5, 9, 2, 6]);
console.log(min, max);    // 1 9

This is the modern pattern — return an object, destructure at the call site. Often clearer than positional tuples.

5.6 Functions are Values (First-Class)

In JS, functions are values. You can store them, pass them, return them:

// Assign:
const fn = function(x) { return x + 1; };

// Store in arrays:
const operations = [
  x => x + 1,
  x => x * 2,
  x => x - 3,
];
operations.forEach(op => console.log(op(10)));   // 11, 20, 7

// Pass as arguments:
function apply(fn, value) {
  return fn(value);
}
apply(x => x * x, 5);                             // 25

// Return from other functions:
function makeMultiplier(factor) {
  return x => x * factor;
}
const double = makeMultiplier(2);
const triple = makeMultiplier(3);
console.log(double(10));                          // 20
console.log(triple(10));                          // 30

This first-class treatment is the foundation of huge parts of JS: array methods (map, filter, reduce), event handlers, callbacks, promises, the whole functional toolkit.

5.7 Closures — JavaScript's Killer Feature

A closure is a function that "remembers" the variables from the scope where it was created, even after that scope is gone.

function makeCounter() {
  let count = 0;
  return function() {
    count++;
    return count;
  };
}

const counter = makeCounter();
console.log(counter());       // 1
console.log(counter());       // 2
console.log(counter());       // 3

What's happening: count is declared inside makeCounter. Normally it would disappear when makeCounter returns. But because the returned function references it, JavaScript keeps it alive. Each call to the returned function modifies the same count.

Each call to makeCounter() creates a separate count:

const counterA = makeCounter();
const counterB = makeCounter();
counterA();   // 1
counterA();   // 2
counterB();   // 1 — independent!

This is the foundation of module patterns, private state, event handlers, partial application, memoization, and a thousand other things. Closures are arguably JavaScript's most important feature.

A practical example — a function that "remembers" its last result:

function memoize(fn) {
  const cache = new Map();
  return function(arg) {
    if (cache.has(arg)) return cache.get(arg);
    const result = fn(arg);
    cache.set(arg, result);
    return result;
  };
}

const slowSquare = (n) => {
  console.log(`computing ${n}^2`);
  return n * n;
};

const fastSquare = memoize(slowSquare);
fastSquare(5);    // computes; logs "computing 5^2"; returns 25
fastSquare(5);    // cached; returns 25; no log
fastSquare(7);    // computes
fastSquare(5);    // cached

The cache Map persists between calls because of closure. Without closures, this pattern would be much uglier.

5.8 Scope and Hoisting

JavaScript has two kinds of scope:

Function scope (legacy, var): variables exist throughout the enclosing function.

Block scope (modern, let/const): variables exist only within {...}.

function example() {
  if (true) {
    var a = 1;
    let b = 2;
  }
  console.log(a);     // 1 — var leaked out
  console.log(b);     // ReferenceError — let is block-scoped
}

Hoisting: var declarations are moved to the top of their function. let/const declarations are hoisted too, but trying to use them before the declaration throws — they're in the "temporal dead zone."

console.log(a);    // undefined (var hoisted, but not initialized)
var a = 5;

console.log(b);    // ReferenceError (TDZ)
let b = 5;

The summary: use let/const, never var, and you'll never need to think about hoisting.

5.9 Pure Functions

A pure function has two properties: 1. Same input → same output, every time. 2. No side effects (doesn't modify external state, doesn't read/write files, doesn't log).

// Pure:
function double(x) {
  return x * 2;
}

// Impure (modifies external state):
let counter = 0;
function incrementCounter() {
  counter++;
  return counter;
}

// Impure (side effect: logs):
function add(a, b) {
  console.log("adding");
  return a + b;
}

Pure functions are easier to test, easier to reason about, and trivially parallelizable. They're the heart of functional programming. Aim for pure functions when you can — but pragmatic JS code mixes pure and impure freely. Don't be dogmatic about it.

5.10 Higher-Order Functions

A function that takes a function as an argument, or returns one, is higher-order. JS's array methods (next section) are higher-order functions. So is the memoize function we wrote above.

The pattern is one of JS's signatures:

// Take a function, transform an array using it:
[1, 2, 3].map(x => x * 2);

// Take a function, run it later:
setTimeout(() => console.log("later"), 1000);

// Take a function, only call it on events:
button.addEventListener("click", () => console.log("clicked"));

This is the JS style. Where Python tends to use comprehensions and explicit loops, JS tends to use methods and callbacks. Get comfortable with it.

SECTION 6

Arrays & Iteration

JavaScript arrays are like Python lists — ordered, mutable, holding any type. But the methods are different (and arguably better in JS). Modern JS code uses these methods heavily; learning them well is high-leverage.

6.1 Creating Arrays

const empty = [];
const nums = [1, 2, 3, 4, 5];
const mixed = [1, "hello", true, null, { x: 1 }];   // any types
const nested = [[1, 2], [3, 4]];

// Array() constructor (rarely needed):
const a = new Array(5);              // [empty × 5] — array of length 5, no values
const b = new Array(1, 2, 3);        // [1, 2, 3]
const c = Array.of(5);               // [5] — single-element array
const d = Array.from("hello");       // ["h", "e", "l", "l", "o"]
const e = Array.from({length: 3}, (_, i) => i * 2);  // [0, 2, 4]

Array.from is a Swiss Army knife — converts any iterable or array-like into an array.

6.2 Accessing & Modifying

const arr = [10, 20, 30, 40, 50];

arr[0];              // 10
arr[arr.length - 1]; // 50 — last element
arr.length;          // 5 — this is a property, not a method

// JS arrays don't have negative indexing like Python:
arr[-1];             // undefined (NOT 50!)
arr.at(-1);          // 50 — the modern `.at()` method (ES2022) does work

arr[1] = 99;         // mutate
arr[100] = "wow";    // creates a sparse array — don't do this

Note: .at(-1) was added in 2022. For older browsers, arr[arr.length - 1] is the traditional way.

Mutating methods (modify the array in place):

const arr = [1, 2, 3];

arr.push(4);          // add to end → [1, 2, 3, 4]
arr.pop();            // remove from end, return it → returns 4, arr is [1, 2, 3]
arr.unshift(0);       // add to start → [0, 1, 2, 3]
arr.shift();          // remove from start → returns 0, arr is [1, 2, 3]

arr.splice(1, 1);             // remove 1 item starting at index 1 → arr is [1, 3]
arr.splice(1, 0, "x", "y");   // insert at index 1 → arr is [1, "x", "y", 3]

arr.sort();           // sort in place (BEWARE — sorts as strings by default!)
arr.reverse();        // reverse in place
arr.fill(0);          // [0, 0, 0, 0]

Non-mutating methods (return a new array, don't touch the original):

const arr = [1, 2, 3];

arr.slice(1);              // [2, 3] — copy from index 1 to end
arr.slice(0, 2);           // [1, 2]
arr.concat([4, 5]);        // [1, 2, 3, 4, 5]
[...arr, 4, 5];            // same — using spread
arr.indexOf(2);            // 1 — index of first match, or -1
arr.includes(2);           // true
arr.join(", ");            // "1, 2, 3"
arr.flat();                // [1, 2, 3] — flatten nested arrays one level
arr.toReversed();          // new reversed array (ES2023, doesn't mutate)
arr.toSorted();            // new sorted array (ES2023)

Distinguishing mutating from non-mutating is a common source of bugs in JS. Memorize the rough split: push/pop/shift/unshift/splice/sort/reverse/fill mutate. Everything else (mostly) returns new.

6.3 Sorting — A Famous Gotcha

.sort() sorts as strings by default. Stunning behavior:

[10, 2, 1].sort();           // [1, 10, 2] — sorted alphabetically by digit!
[100, 25, 5].sort();         // [100, 25, 5]

To sort numerically, pass a comparator function — (a, b) that returns negative if a < b, positive if a > b, zero if equal:

[10, 2, 1].sort((a, b) => a - b);    // [1, 2, 10] — ascending
[10, 2, 1].sort((a, b) => b - a);    // [10, 2, 1] — descending

The a - b trick: subtract them. If a<b the result is negative; the sort interprets that as "a should come first."

For sorting objects:

const users = [
  { name: "Alice", age: 30 },
  { name: "Bob", age: 25 },
  { name: "Carol", age: 35 },
];

users.sort((a, b) => a.age - b.age);
// sorted by age, ascending

users.sort((a, b) => a.name.localeCompare(b.name));
// sorted by name (handles unicode correctly)

6.4 The Iteration Methods — The Bread and Butter

These are the JS array methods you'll use most often. Internalize them.

forEach — do something for each item

[1, 2, 3].forEach((item, index) => {
  console.log(index, item);
});
// 0 1
// 1 2
// 2 3

Like for...of but as a method. Doesn't return anything. Can't be broken out of with break (use for...of if you need early exit).

map — transform each item

const doubled = [1, 2, 3].map(n => n * 2);     // [2, 4, 6]
const names = users.map(u => u.name);           // ["Alice", "Bob", "Carol"]

Returns a new array of the same length, with each item replaced by the function's return value. Use map when you're transforming, not forEach.

filter — keep some, drop others

const evens = [1, 2, 3, 4, 5, 6].filter(n => n % 2 === 0);    // [2, 4, 6]
const adults = users.filter(u => u.age >= 18);

Returns a new array containing only the items for which the function returns truthy.

reduce — fold into a single value

const sum = [1, 2, 3, 4].reduce((acc, n) => acc + n, 0);    // 10
const max = [3, 7, 1, 9].reduce((m, n) => n > m ? n : m);   // 9
const grouped = [1, 2, 3, 4, 5].reduce((groups, n) => {
  const key = n % 2 === 0 ? "even" : "odd";
  groups[key] = groups[key] || [];
  groups[key].push(n);
  return groups;
}, {});
// { odd: [1, 3, 5], even: [2, 4] }

reduce is the most powerful of the three — it can express any transformation. The first argument is a function (accumulator, item) => newAccumulator. The second is the initial value of the accumulator.

Beginners often find reduce confusing. The mental model: "carry a running total (or any data structure) through the array, combining each item into it." Once it clicks, you'll use it often.

find and findIndex — get the first match

const user = users.find(u => u.name === "Bob");        // the object, or undefined
const index = users.findIndex(u => u.name === "Bob");  // index, or -1

some and every — test conditions

[1, 2, 3].some(n => n > 2);       // true — at least one matches
[1, 2, 3].every(n => n > 0);      // true — all match
[1, 2, 3].every(n => n > 2);      // false

Method chaining — the JS style

These methods return arrays, so you can chain them:

const result = [1, 2, 3, 4, 5, 6, 7, 8]
  .filter(n => n % 2 === 0)        // [2, 4, 6, 8]
  .map(n => n * n)                  // [4, 16, 36, 64]
  .reduce((sum, n) => sum + n, 0);  // 120

This reads top-to-bottom as a pipeline. Way clearer than the equivalent with intermediate variables.

6.5 Iterating with for...of

When you need break or continue, or just want a simpler loop:

for (const item of arr) {
  console.log(item);
}

// With index — use entries():
for (const [i, item] of arr.entries()) {
  console.log(i, item);
}

for...of is the modern equivalent of Python's for x in arr:. Use it whenever the method-chaining style would be awkward.

6.6 Spread and Destructuring with Arrays

We covered these in Section 4, but here are array-specific patterns:

// Copy an array
const copy = [...original];

// Concatenate
const all = [...a, ...b, ...c];

// Insert into middle
const before = [1, 2, 3];
const after = [...before.slice(0, 2), "x", ...before.slice(2)];
// [1, 2, "x", 3]

// Remove duplicates using a Set (Section 7)
const unique = [...new Set([1, 2, 2, 3, 3, 4])];   // [1, 2, 3, 4]

// Swap two values
let a = 1, b = 2;
[a, b] = [b, a];
console.log(a, b);   // 2 1

That last one — array destructuring for swapping — is a beautiful little idiom.

6.7 A Real-World Example

Suppose you have an array of orders. Compute the total revenue per customer, then list customers from biggest to smallest.

const orders = [
  { customer: "Alice", amount: 30 },
  { customer: "Bob", amount: 20 },
  { customer: "Alice", amount: 50 },
  { customer: "Carol", amount: 10 },
  { customer: "Bob", amount: 15 },
];

// Group by customer, sum amounts:
const totals = orders.reduce((acc, order) => {
  acc[order.customer] = (acc[order.customer] || 0) + order.amount;
  return acc;
}, {});
// { Alice: 80, Bob: 35, Carol: 10 }

// Convert to array of pairs, sort, take top:
const ranked = Object.entries(totals)
  .map(([name, total]) => ({ name, total }))
  .sort((a, b) => b.total - a.total);

console.log(ranked);
// [{ name: "Alice", total: 80 }, { name: "Bob", total: 35 }, { name: "Carol", total: 10 }]

Five lines do a meaningful chunk of data work. This style — reduce to group, map/sort to transform — is everywhere in JS data processing.

SECTION 7

Objects

JavaScript objects are the foundation of everything non-primitive. Arrays are objects. Functions are objects. Class instances are objects. Even null, technically, used to be objects. (typeof null === "object" — Section 3.)

In JS, "object" means: an unordered collection of key-value pairs, where keys are strings (or symbols) and values are anything.

7.1 Creating Objects

The object literal syntax:

const user = {
  name: "Alice",
  age: 30,
  email: "alice@example.com",
};

Equivalent to Python's dictionary, but with a few differences we'll see.

You can also create an empty object and add to it:

const user = {};
user.name = "Alice";
user["age"] = 30;

Both bracket notation (user["age"]) and dot notation (user.age) work. Bracket is needed when the key is dynamic or has special characters:

const key = "name";
user[key];              // works — dynamic key

const weirdObj = {};
weirdObj["with-dashes"] = 1;       // bracket required
weirdObj["123"] = 1;               // bracket required (key starts with digit)
weirdObj.hello = 1;                // dot works for normal identifiers

7.2 Accessing & Modifying

const user = { name: "Alice", age: 30 };

user.name;                  // "Alice"
user["name"];               // same
user.unknown;               // undefined — no error for missing keys

user.email = "alice@x.com"; // add a new property
delete user.age;            // remove a property
"name" in user;             // true — check existence
"foo" in user;              // false

Key gotcha: dot notation works for identifier-like keys only. Bracket notation works for anything:

const obj = { "first name": "Alice" };
obj.first name;             // SyntaxError
obj["first name"];          // "Alice"

7.3 Shorthand Property Syntax

ES6 added a shorthand for when the property name matches the variable name:

const name = "Alice";
const age = 30;

// Long form:
const user = { name: name, age: age };

// Shorthand:
const user = { name, age };

This is used everywhere in modern JS. When returning multiple values, this pattern is the idiom:

function getUser(id) {
  const name = "Alice";
  const email = "alice@x.com";
  return { name, email };
}

7.4 Computed Property Names

You can use any expression as a key with [...]:

const key = "name";
const user = {
  [key]: "Alice",         // computed at object-creation time
  ["age" + "Group"]: "adult",
};
console.log(user);   // { name: "Alice", ageGroup: "adult" }

7.5 Methods

When a property is a function, we call it a method:

const user = {
  name: "Alice",
  greet: function() {
    return `Hello, I'm ${this.name}`;
  },
};
console.log(user.greet());   // "Hello, I'm Alice"

ES6 added shorthand syntax for method definitions:

const user = {
  name: "Alice",
  greet() {                   // no `function` keyword needed
    return `Hello, I'm ${this.name}`;
  },
};

7.6 this — The Trickiest Topic in JavaScript

this is a special variable inside methods. It refers to the object the method is being called on. But its behavior has subtle rules that surprise everyone.

Rule 1: Method calls

When you call obj.method(), this is obj:

const user = {
  name: "Alice",
  greet() { return `Hello, ${this.name}`; },
};
user.greet();      // "Hello, Alice" — this === user

Rule 2: Detached methods

When you store a method in a variable and call it standalone, this is undefined (in strict mode) or the global object (in sloppy mode):

const user = {
  name: "Alice",
  greet() { return `Hello, ${this.name}`; },
};

const standalone = user.greet;
standalone();      // TypeError or undefined — this is no longer `user`!

This is one of JavaScript's most-bitten footguns. Storing methods and calling them later loses the this binding.

Rule 3: Arrow functions don't have their own this

Arrow functions inherit this from the enclosing scope. This is good — it solves the most common this problem.

const user = {
  name: "Alice",
  hobbies: ["reading", "hiking"],

  printHobbies() {
    // Regular function as callback: `this` gets lost
    this.hobbies.forEach(function(h) {
      console.log(this.name, h);    // this.name is undefined!
    });

    // Arrow function: this is preserved
    this.hobbies.forEach((h) => {
      console.log(this.name, h);    // "Alice reading", "Alice hiking"
    });
  },
};

Rule of thumb: use arrow functions for callbacks (so this stays consistent). Use regular functions (or method shorthand) for actual object methods. Don't use arrow functions for top-level object methods, because they won't have this bound to the object.

Rule 4: Explicit binding with call, apply, bind

You can manually set this:

function greet() { return `Hello, ${this.name}`; }

greet.call({ name: "Alice" });           // "Hello, Alice"
greet.apply({ name: "Bob" });            // "Hello, Bob"

const boundGreet = greet.bind({ name: "Carol" });
boundGreet();                            // "Hello, Carol"

call(thisVal, ...args) and apply(thisVal, argsArray) invoke the function with a chosen this. bind returns a new function with this permanently set.

.bind() is especially useful to "lock in" a method's this:

const user = {
  name: "Alice",
  greet() { return `Hello, ${this.name}`; },
};

const safe = user.greet.bind(user);
safe();         // "Hello, Alice" — works standalone now

Rule 5: new and constructors

When you call a function with new, this is a fresh empty object. (More on this in Section 9 — Classes.)

function User(name) {
  this.name = name;
}
const alice = new User("Alice");
console.log(alice.name);    // "Alice"

That's this. It causes a lot of bugs, especially with callbacks. Modern recommendations:

  • Use arrow functions for callbacks.
  • Use classes (Section 9) instead of manually building objects with methods — classes mostly handle this correctly.
  • When in doubt, use .bind(this) or store this in a const self = this at the top of the method.

7.7 Destructuring Objects

We met destructuring in Section 4. With objects, it shines:

const user = { name: "Alice", age: 30, email: "alice@x.com" };

// Pull out some properties:
const { name, age } = user;

// Rename:
const { name: userName } = user;

// Defaults:
const { country = "Portugal" } = user;

// Nested:
const order = { customer: { name: "Alice" }, total: 50 };
const { customer: { name }, total } = order;
console.log(name, total);    // "Alice" 50

// Rest:
const { name, ...rest } = user;
// name = "Alice", rest = { age: 30, email: "alice@x.com" }

Very common in function parameters:

function createUser({ name, age, email = "" }) {
  return { id: 1, name, age, email };
}

createUser({ name: "Alice", age: 30 });
// { id: 1, name: "Alice", age: 30, email: "" }

7.8 Spread with Objects

ES2018 added ... for objects — copying and merging:

// Copy
const original = { a: 1, b: 2 };
const copy = { ...original };           // { a: 1, b: 2 }

// Merge — later objects override
const defaults = { theme: "light", fontSize: 14 };
const user = { theme: "dark" };
const settings = { ...defaults, ...user };
// { theme: "dark", fontSize: 14 }

// Update a field immutably
const updated = { ...user, theme: "dark" };

Spread does a shallow copy — nested objects are still shared:

const obj = { a: 1, nested: { x: 10 } };
const copy = { ...obj };
copy.nested.x = 999;
console.log(obj.nested.x);   // 999 — same object!

For deep copies, use structuredClone(obj) (modern, built-in) or write recursive logic.

7.9 Iterating Over Objects

To loop over an object's keys, values, or entries:

const user = { name: "Alice", age: 30 };

Object.keys(user);      // ["name", "age"]
Object.values(user);    // ["Alice", 30]
Object.entries(user);   // [["name", "Alice"], ["age", 30]]

// Iterate keys:
for (const key of Object.keys(user)) {
  console.log(key, user[key]);
}

// Iterate entries (often most useful):
for (const [key, value] of Object.entries(user)) {
  console.log(key, value);
}

for...in also iterates object keys, but it includes inherited properties — usually not what you want. Prefer Object.keys() and for...of.

7.10 Map and Set — Modern Alternatives

ES6 introduced Map and Set — proper collection types for cases where objects fall short.

Map — like an object, but keys can be anything

const m = new Map();
m.set("name", "Alice");
m.set(42, "the answer");
m.set({}, "object as key works!");      // can't do this with plain objects

console.log(m.get("name"));             // "Alice"
console.log(m.size);                    // 3
console.log(m.has(42));                 // true
m.delete(42);

// Iterate
for (const [key, value] of m) {
  console.log(key, value);
}

When to use Map over a plain object: - You need non-string keys. - You need to know the size easily (map.size vs Object.keys(obj).length). - You frequently add/remove keys (Maps are faster for this). - You need guaranteed insertion order (objects technically preserve it but Maps are explicit).

For most "string-keyed dictionary" cases, plain objects are still idiomatic.

Set — unique values

const s = new Set([1, 2, 2, 3, 3, 3]);   // duplicates ignored
console.log(s.size);                      // 3
console.log([...s]);                      // [1, 2, 3]

s.add(4);
s.has(2);          // true
s.delete(2);

// Iterate
for (const value of s) {
  console.log(value);
}

Common use: remove duplicates from an array — [...new Set(arr)].

7.11 JSON

JSON (JavaScript Object Notation) is the data interchange format that grew out of JavaScript and became ubiquitous on the web. It looks like JS object syntax but with strict rules: - Keys must be strings, double-quoted. - No trailing commas. - No comments. - No functions, no undefined, no symbols.

const user = { name: "Alice", age: 30, hobbies: ["reading", "hiking"] };

// Object → JSON string
const jsonString = JSON.stringify(user);
// '{"name":"Alice","age":30,"hobbies":["reading","hiking"]}'

JSON.stringify(user, null, 2);
// Pretty-printed with 2-space indent

// JSON string → object
const parsed = JSON.parse(jsonString);
// { name: "Alice", age: 30, hobbies: ["reading", "hiking"] }

JSON.stringify and JSON.parse are the workhorses of saving and loading data, talking to APIs, and persisting state. We'll use them constantly.

JSON.stringify silently drops undefined, functions, and symbols:

JSON.stringify({ a: 1, b: undefined, c: () => {} });
// '{"a":1}'

Bear this in mind when serializing complex objects.

SECTION 8

Strings & Template Literals

We covered the basics of strings in Section 3. This section goes deeper — template tags, escape sequences, the unicode situation, and brief regex coverage.

8.1 Template Literal Recap

Backtick strings can: - Span multiple lines. - Embed expressions with ${...}. - Embed any JS expression, not just variables.

const name = "Alice";
const score = 92;

const message = `
Hello, ${name}!
Your score: ${score}
Result: ${score >= 60 ? "PASS" : "FAIL"}
Doubled: ${score * 2}
`;

Inside ${} you can put any expression — including function calls, ternaries, even other template literals.

8.2 Tagged Templates — Power Mode

You can prefix a template literal with a function. The function receives the static parts and the interpolated values:

function highlight(strings, ...values) {
  return strings.reduce((result, str, i) => {
    const value = values[i] ? `**${values[i]}**` : "";
    return result + str + value;
  }, "");
}

const name = "Alice";
const score = 92;
const msg = highlight`Hello, ${name}! Your score: ${score}.`;
console.log(msg);
// "Hello, **Alice**! Your score: **92**."

Tagged templates power libraries like styled-components, lit-html, GraphQL queries embedded in JS, and many others. They're a niche feature but extremely powerful in the right place.

8.3 Escape Sequences

Inside any string literal:

"\n"       // newline
"\t"       // tab
"\\"       // literal backslash
"\""       // literal double quote (in double-quoted strings)
"\'"       // literal single quote (in single-quoted strings)
"\u{1F680}" // Unicode code point — 🚀

Template literals interpret these the same way:

`Line 1\nLine 2`        // "Line 1\nLine 2" with a newline inside

To include a literal backtick inside a template literal:

`\``                    // "`"

8.4 String Methods

We covered many in Section 3. A few more worth knowing:

"hello world".padStart(15, "-");     // "----hello world"
"5".padStart(3, "0");                // "005"

"  spaced  ".trimStart();            // "spaced  "
"  spaced  ".trimEnd();              // "  spaced"

"abc".codePointAt(0);                // 97 — like Python's ord()
String.fromCodePoint(97);            // "a"   — like Python's chr()

"abc".normalize();                   // Unicode normalization

"--hello--".replace(/-/g, "");       // "hello" — using regex to replace all

8.5 Strings and Unicode

JavaScript strings are sequences of UTF-16 code units, not characters. For most characters (Latin alphabet, common punctuation) this doesn't matter — one code unit = one character. But for emoji and many non-Latin scripts, one logical character is two code units (a "surrogate pair").

"hello".length;          // 5 — easy
"🚀".length;             // 2 — surprise!
"🚀"[0];                 // "\uD83D" — half the emoji, broken

To iterate "real" characters:

[..."🚀hello"];                       // ["🚀", "h", "e", "l", "l", "o"]
[..."🚀hello"].length;                // 6 — correct!

// Or with a for...of loop:
for (const char of "🚀hello") {
  console.log(char);                  // 🚀, h, e, l, l, o
}

Spread (...) and for...of understand surrogate pairs. The legacy .length and s[i] don't.

This is a perpetual source of bugs in text-handling JS code. Use [...str] or Array.from(str) when you need actual characters.

8.6 Regular Expressions — A Brief Introduction

Regex is its own topic (whole books exist). JS has them built-in:

const re = /\d+/;                    // one or more digits
"abc123def".match(re);                // ["123", index: 3, ...]
"abc123def".search(re);               // 3 (index of match)
"abc123def".replace(re, "X");         // "abcXdef"
"abc123def".replace(/\d/g, "X");      // "abcXXXdef" — global flag

re.test("abc123");                    // true

// Capture groups:
const dateRe = /(\d{4})-(\d{2})-(\d{2})/;
const [match, year, month, day] = "Today is 2026-01-15".match(dateRe);
console.log(year, month, day);        // "2026" "01" "15"

// Named groups:
const named = /(?<year>\d{4})-(?<month>\d{2})-(?<day>\d{2})/;
const { groups } = "2026-01-15".match(named);
console.log(groups.year);             // "2026"

Common flags: - g — global (find all matches) - i — case-insensitive - m — multi-line (^ and $ match line starts/ends) - u — unicode mode (handle emoji properly) - s — dot-all (. matches newlines)

Regex is incredibly useful but also notorious for being write-only — you read it once, write it, then can never edit it again. Use regex101.com to debug.

A tip: prefer string methods when they work. .includes(), .startsWith(), .endsWith() are clearer than regex when you don't need pattern matching.

8.7 String Comparison

The < and > operators compare strings character by character using their Unicode values:

"apple" < "banana"          // true
"abc" < "abd"               // true
"Apple" < "apple"           // true — uppercase Latin letters come before lowercase!
"10" < "9"                  // true — STRING comparison, not numeric

For natural alphabetical sort (case-insensitive, locale-aware), use localeCompare:

const words = ["banana", "Apple", "cherry"];
words.sort((a, b) => a.localeCompare(b));
// ["Apple", "banana", "cherry"]

localeCompare handles things like accented characters correctly. For internationalized apps, it's essential.

🏋️ Exercise 8.7

Write a function slugify(text) that converts arbitrary text into a URL slug:

  1. Lowercase the text.

SECTION 9

Classes & Prototypes

JavaScript's OOP system is unique. Other languages use classical inheritance (classes inherit from classes). JS uses prototypal inheritance (objects inherit directly from other objects). ES6 added a class syntax that looks like Java, but underneath it's still prototypes. Understanding both layers makes you a real JS programmer.

9.1 ES6 Classes — The Modern Way

The syntax is familiar from other languages:

class User {
  constructor(name, email) {
    this.name = name;
    this.email = email;
  }

  greet() {
    return `Hello, I'm ${this.name}`;
  }

  static fromString(str) {
    const [name, email] = str.split(",");
    return new User(name, email);
  }
}

const alice = new User("Alice", "alice@x.com");
console.log(alice.greet());                       // "Hello, I'm Alice"

const bob = User.fromString("Bob,bob@x.com");
console.log(bob.greet());                          // "Hello, I'm Bob"

Components: - constructor runs when you call new User(...). Sets up this. - Methods (greet) are defined on instances. - Static methods (fromString) belong to the class itself, not instances. Called as User.fromString(...), not on an instance.

new does four things behind the scenes: 1. Creates a fresh empty object. 2. Sets its prototype to User.prototype. 3. Runs the constructor with this bound to the new object. 4. Returns the new object.

9.2 Inheritance with extends

class Animal {
  constructor(name) {
    this.name = name;
  }
  speak() {
    return "Some sound";
  }
}

class Dog extends Animal {
  constructor(name, breed) {
    super(name);            // call parent constructor
    this.breed = breed;
  }

  speak() {
    return "Woof!";
  }

  describe() {
    return `${this.name} (${this.breed}) says ${this.speak()}`;
  }
}

const rex = new Dog("Rex", "Labrador");
console.log(rex.describe());        // "Rex (Labrador) says Woof!"

super(...) calls the parent's constructor. You must call it before using this in the subclass constructor.

super.method() (no new) calls the parent's version of a method:

class Cat extends Animal {
  speak() {
    return super.speak() + " (but quieter)";    // "Some sound (but quieter)"
  }
}

9.3 Getters & Setters

You can define computed properties that look like fields but run a function:

class Temperature {
  constructor(celsius) {
    this.celsius = celsius;
  }

  get fahrenheit() {
    return this.celsius * 9 / 5 + 32;
  }

  set fahrenheit(value) {
    this.celsius = (value - 32) * 5 / 9;
  }
}

const t = new Temperature(20);
console.log(t.fahrenheit);    // 68 — calls the getter
t.fahrenheit = 100;            // calls the setter
console.log(t.celsius);        // 37.777...

Use sparingly. Getters/setters can hide expensive logic behind innocent-looking property access, which can surprise readers. Real fields > clever getters when they work.

9.4 Private Fields

ES2022 added real private fields, prefixed with #:

class Counter {
  #count = 0;             // private field

  increment() {
    this.#count++;
  }

  get value() {
    return this.#count;
  }
}

const c = new Counter();
c.increment();
c.increment();
console.log(c.value);     // 2
console.log(c.#count);    // SyntaxError — private access outside class

Before private fields, the convention was a leading underscore (_count) — a "soft" privacy that other code politely respects. # fields are actually enforced by the language.

Private methods work the same way: #methodName() { ... }.

9.5 The Prototype Story — What Classes Really Are

This section is the part of JS that surprises programmers from class-based languages. A class in JavaScript is just syntactic sugar over a prototype-based system.

Without class, the same thing looks like this:

function User(name, email) {       // constructor function
  this.name = name;
  this.email = email;
}

User.prototype.greet = function() {
  return `Hello, I'm ${this.name}`;
};

const alice = new User("Alice", "alice@x.com");
console.log(alice.greet());        // "Hello, I'm Alice"

This is the pre-ES6 way of doing OOP in JavaScript. The class syntax is exactly equivalent — just nicer to read.

What's a prototype?

Every JavaScript object has an internal link to another object — its prototype. When you access a property that doesn't exist on the object directly, JS looks at the prototype. Then the prototype's prototype. And so on, up the chain, until it finds it or hits null.

const alice = new User("Alice", "alice@x.com");

alice.name;       // "Alice" — found directly on alice
alice.greet;      // function — not on alice, found on User.prototype
alice.toString;   // function — not on User.prototype, found on Object.prototype

You can inspect the chain:

Object.getPrototypeOf(alice);                  // User.prototype
Object.getPrototypeOf(User.prototype);         // Object.prototype
Object.getPrototypeOf(Object.prototype);       // null

This is prototypal inheritance — objects inheriting from objects, not classes from classes. ES6 class syntax sets up this prototype chain for you behind the scenes.

Why does this matter?

Most of the time, you can ignore the prototype machinery. class works as you'd expect. But:

  • It explains why all arrays magically have .map(), .filter(), etc. — they're on Array.prototype.
  • It enables monkey-patching: Array.prototype.myMethod = function() { ... } adds a method to all arrays. (Don't do this — it pollutes shared global state.)
  • It explains strange behaviors like methods being shared between instances.

Memory-wise: methods live on the prototype, not on each instance. A million instances of User share one greet function. That's efficient.

9.6 The this Pitfalls in Classes

this in methods works as you'd expect — until you start passing methods around as callbacks:

class Timer {
  constructor() {
    this.seconds = 0;
  }

  tick() {
    this.seconds++;
    console.log(this.seconds);
  }

  start() {
    setInterval(this.tick, 1000);
    //          ^ method detached from `this`!
    //            Inside setInterval, `this` will be undefined.
  }
}

const t = new Timer();
t.start();        // TypeError: Cannot read property 'seconds' of undefined

Three ways to fix:

// 1. Arrow function in the call
start() {
  setInterval(() => this.tick(), 1000);
}

// 2. .bind() to lock in this
start() {
  setInterval(this.tick.bind(this), 1000);
}

// 3. Class field with arrow — makes the method auto-bound
class Timer {
  seconds = 0;
  tick = () => {
    this.seconds++;
    console.log(this.seconds);
  };
}

Option 3 (arrow methods as class fields) is the most reliable, but uses one closure per instance (slightly less memory-efficient than regular methods). For most uses, the cost is negligible.

This this-binding issue is the cause of countless React bugs in older codebases. Modern React with hooks avoids it; class components had to fight it constantly.

9.7 When to Use Classes

Honestly, JavaScript culture has moved away from classes over the past decade. Many production codebases use almost no class at all — they use functions and plain objects. Reasons:

  • React shifted from class components to function components with hooks.
  • The this footgun never fully went away.
  • Functional patterns (composing functions, passing data around) work very well in JS.

That said, classes still earn their keep for: - Domain models with clear state and operations: User, Order, ShoppingCart. - Long-lived stateful objects with multiple related methods. - Inheritance hierarchies when they're actually appropriate (rare but useful). - Errors — custom error types (Section 13).

For everything else — utility functions, data transformations, configuration — plain functions and objects are often clearer.

Design a BankAccount class with: - A private #balance (use a real private field). - A constructor(owner, initialBalance = 0). - Methods deposit(amount) and withdraw(amount) that update the balance (throw an error for negative amounts or overdrafts). - A getter balance that returns the current balance. - A static method transfer(from, to, amount) that moves money between two accounts.

Solution 
class BankAccount {
  #balance;

  constructor(owner, initialBalance = 0) {
    if (initialBalance < 0) {
      throw new Error("Initial balance can't be negative");
    }
    this.owner = owner;
    this.#balance = initialBalance;
  }

  get balance() {
    return this.#balance;
  }

  deposit(amount) {
    if (amount <= 0) throw new Error("Deposit must be positive");
    this.#balance += amount;
  }

  withdraw(amount) {
    if (amount <= 0) throw new Error("Withdrawal must be positive");
    if (amount > this.#balance) throw new Error("Insufficient funds");
    this.#balance -= amount;
  }

  static transfer(from, to, amount) {
    from.withdraw(amount);     // throws if insufficient — atomicity
    to.deposit(amount);
  }
}

// Usage
const alice = new BankAccount("Alice", 100);
const bob = new BankAccount("Bob", 50);

alice.deposit(50);
console.log(alice.balance);   // 150

BankAccount.transfer(alice, bob, 30);
console.log(alice.balance);   // 120
console.log(bob.balance);     // 80

try {
  alice.withdraw(999);
} catch (e) {
  console.log(e.message);     // "Insufficient funds"
}
Things this exercise teaches: - Private fields with `#` prevent direct external tampering. - A getter exposes a read-only view of internal state. - Throwing errors signals failure (we'll cover error handling in Section 13). - Static methods belong to the class, not instances — they're "class-level" operations. Note the `transfer` ordering — withdraw first, then deposit. If withdraw fails, no money gets created. If transfer were `deposit(); withdraw()` and withdraw failed afterward, you'd duplicate money. Order matters for correctness. A real banking system would also use database transactions for atomicity, but the principle holds.
SECTION 10

Modules

For most of JavaScript's history, there was no built-in way to split code across files. Browsers loaded scripts globally; if you used <script src="a.js"> and <script src="b.js">, they shared one global namespace. Pollution was rampant.

ES6 finally added modules — a way to split JS into files that explicitly export and import. This section covers ES modules (the modern standard) and CommonJS (the Node.js legacy that you'll still encounter).

10.1 ES Modules — The Modern Standard

A module is a file. Each file has its own scope. Anything not explicitly exported stays private.

math.js:

export const PI = 3.14159;

export function square(n) {
  return n * n;
}

export function cube(n) {
  return n * n * n;
}

app.js:

import { PI, square } from "./math.js";

console.log(PI);              // 3.14159
console.log(square(5));       // 25

Key points: - export exposes a value, function, or class to other modules. - import brings exposed values into another module's scope. - The path must include ./ for relative imports and the file extension (in browsers). - Variables not exported are completely private to the module.

10.2 Named vs Default Exports

Named exports — what we did above. You can have many per file.

// math.js
export const PI = 3.14159;
export function square(n) { return n * n; }
export class Vector { ... }

// app.js — must use the exact names
import { PI, square, Vector } from "./math.js";

You can rename on import:

import { square as sq } from "./math.js";

Default exports — one per file. The "main thing" the module provides.

// user.js
export default class User {
  constructor(name) { this.name = name; }
}

// app.js — can use ANY name when importing the default
import User from "./user.js";       // no curly braces
import Person from "./user.js";     // same thing, different name

You can mix:

// utils.js
export default function main() { ... }
export const VERSION = "1.0";

// app.js
import main, { VERSION } from "./utils.js";

Style debate: named vs default. Defaults are simpler for "one thing per file." Named exports give better tooling (rename refactor, autocomplete, tree-shaking). Most modern codebases favor named exports for everything except React components (where default is convention).

10.3 Importing Everything as a Namespace

// math.js — same as before, named exports

// app.js
import * as math from "./math.js";

console.log(math.PI);
console.log(math.square(5));

* as math bundles all exports into one object. Convenient when you import many things from one module. Slight downside: tree-shaking tools can't always eliminate unused exports.

10.4 Re-exporting

A module can re-export from another module:

// index.js — re-export everything
export { PI, square } from "./math.js";
export { User } from "./user.js";
export { default as Logger } from "./logger.js";

This is the "barrel file" pattern — a single import point for an entire package. Common in library code.

10.5 Side-Effect-Only Imports

import "./styles.css";
import "./polyfill.js";

This runs the module's code but doesn't import any values. Used for CSS files (bundlers process them), polyfills (code that patches global behavior), and any code that needs to execute for its side effects.

10.6 Dynamic import()

The regular import is static — at the top of the file, evaluated once. You can also import dynamically:

async function loadHeavyFeature() {
  const module = await import("./heavy.js");
  module.doStuff();
}

import() returns a Promise (Section 11) that resolves to the module's exports. This enables code splitting — only load the code you need, when you need it.

Useful for: - Loading rarely-used features on demand. - Lazy-loading routes in single-page apps. - Conditional loading based on environment.

10.7 Using ES Modules in the Browser

To use import in a browser-side script, declare the script as type="module":

<script type="module" src="app.js"></script>

Or inline:

<script type="module">
  import { greet } from "./util.js";
  greet();
</script>

Modules behave differently from regular scripts: - They run in strict mode automatically. - They have their own scope (no global pollution). - They're deferred by default — execute after HTML parsing. - They use import/export — no plain <script> cross-talk.

A serving gotcha: for browser modules to work, you usually need to serve files via HTTP, not file://. Use a tiny local server like npx serve or VS Code's Live Server extension.

10.8 CommonJS — Node's Legacy Module System

Before ES modules existed, Node.js invented its own — CommonJS. You'll still see this constantly in older Node code:

// math.js (CommonJS)
const PI = 3.14159;
function square(n) { return n * n; }

module.exports = { PI, square };
// or: module.exports.PI = PI;
//     module.exports.square = square;

// app.js (CommonJS)
const { PI, square } = require("./math");      // no .js needed
console.log(PI, square(5));

Differences from ES modules: - require() is synchronous — it blocks while loading. - It's a regular function, not a special syntax — can be called anywhere, conditionally. - File extension can be omitted.

Node now supports both. New Node code uses ES modules (type: "module" in package.json). Older code (and many libraries) still use CommonJS. They mostly interop, with some friction.

In the browser, only ES modules work (without a bundler).

10.9 Bundlers — The Real-World Reality

In production, you rarely ship raw ES modules. A bundler (Vite, esbuild, Webpack, Rollup) takes your source files and combines them into one (or a few) optimized files. The bundler:

  • Resolves imports and combines files.
  • Strips dead code (tree-shaking).
  • Transpiles modern syntax to older versions for compatibility.
  • Minifies the result.
  • Handles CSS, images, fonts — anything you import.

Bundlers are why a "Hello World" React app downloads 200KB of JS. The pricing of modern web tooling.

We won't cover bundlers in depth here — they're a topic of their own. For learning, the browser's native module support is enough.

SECTION 11

Promises & async/await

Time for the topic that defines modern JavaScript. JS is single-threaded — only one thing runs at a time. Yet web apps fetch data, animate UIs, handle clicks, and run timers simultaneously. How? Through the event loop, callbacks, promises, and now async/await.

This section is dense but essential. If you only deeply learn one thing from this course, make it this.

11.1 The Event Loop — Why JS Doesn't Block

JavaScript runs on a single thread. But operations like "fetch this URL" or "wait 5 seconds" don't tie up that thread. Instead:

  1. JS hands the operation off to the runtime (browser or Node).
  2. The runtime does the work asynchronously.
  3. When done, the runtime puts a callback in a queue.
  4. The event loop runs callbacks from the queue when the main thread is free.
[Synchronous code]  →  runs immediately on main thread
[setTimeout(fn, 1000)]  →  hands off to runtime, queues fn for later
[fetch(url).then(fn)]   →  hands off to runtime, queues fn when response arrives

The single thread + event loop model has consequences: - No blocking calls in long-running code. Don't do CPU-heavy work in the browser — it freezes the UI. - Async operations don't return values immediately. They give you something representing "the future value" — a callback, a Promise, an async result.

11.2 Callbacks — The Original Async

Before Promises, async operations were handled with callbacks:

setTimeout(function() {
  console.log("After 1 second");
}, 1000);

fs.readFile("file.txt", function(err, data) {
  if (err) console.error(err);
  else console.log(data);
});

You pass a function (the "callback") that runs when the operation completes. The runtime calls it for you.

This works but gets ugly fast — the dreaded "callback hell":

fetchUser(userId, function(err, user) {
  if (err) return handleError(err);
  fetchOrders(user.id, function(err, orders) {
    if (err) return handleError(err);
    fetchProducts(orders[0].id, function(err, products) {
      if (err) return handleError(err);
      // The pyramid of doom grows...
    });
  });
});

Promises were invented to fix this.

11.3 Promises — The Modern Async

A Promise is an object representing "the eventual result of an async operation." It has three states: - Pending — the operation hasn't completed. - Fulfilled — succeeded, has a value. - Rejected — failed, has an error.

A Promise is settled once it's either fulfilled or rejected. It can't change state again.

You consume a Promise with .then() and .catch():

fetch("https://api.example.com/data")
  .then(response => {
    console.log("Got response", response);
    return response.json();
  })
  .then(data => {
    console.log("Got data", data);
  })
  .catch(error => {
    console.error("Something failed", error);
  });

Reads like a pipeline: 1. fetch() returns a Promise that resolves to a Response. 2. .then(response => ...) runs when fetch succeeds. Returns a new Promise (from response.json()). 3. .then(data => ...) runs when the JSON parsing succeeds. 4. .catch(error => ...) catches any error in the entire chain.

If any step throws or rejects, the chain skips to the nearest .catch. This is much better than nested callbacks.

11.4 Creating Your Own Promises

Most of the time, you consume promises from APIs (fetch, setTimeout wrappers, etc.). Occasionally, you create them yourself:

function delay(ms) {
  return new Promise((resolve, reject) => {
    setTimeout(() => resolve(`waited ${ms}ms`), ms);
  });
}

delay(1000).then(result => console.log(result));
// after 1 second: "waited 1000ms"

new Promise((resolve, reject) => { ... }) takes an "executor function." Inside: - Call resolve(value) to fulfill the promise. - Call reject(error) to reject it. - Throwing inside the executor also rejects.

You'll rarely need this directly — most async APIs already return Promises. Wrapping callback-based APIs is the main use case.

11.5 async/await — The Game Changer

ES2017 added async/await — syntax that makes Promise-based code look synchronous.

The same chain we wrote earlier becomes:

async function loadData() {
  try {
    const response = await fetch("https://api.example.com/data");
    const data = await response.json();
    console.log("Got data", data);
  } catch (error) {
    console.error("Something failed", error);
  }
}

loadData();

Key concepts: - async before a function declaration makes it an async function. Async functions always return a Promise. - await can only be used inside async functions. It pauses execution until the awaited Promise settles, then evaluates to the resolved value (or throws on rejection). - try/catch works exactly as you'd expect — async errors become catchable like sync ones.

This reads top-to-bottom. The async nature mostly disappears from view. It's by far the dominant style in modern JS.

11.6 Sequential vs Parallel

A common mistake — running things sequentially when they could run in parallel:

// Sequential — total time = sum of all
async function sequential() {
  const a = await fetch(urlA);        // 1 second
  const b = await fetch(urlB);        // 1 second (starts AFTER a done)
  const c = await fetch(urlC);        // 1 second
  // Total: 3 seconds
}

// Parallel — total time = max
async function parallel() {
  const [a, b, c] = await Promise.all([
    fetch(urlA),                       // starts immediately
    fetch(urlB),                       // starts immediately
    fetch(urlC),                       // starts immediately
  ]);
  // Total: ~1 second (all three in flight at once)
}

Promise.all([...]) runs multiple Promises concurrently and waits for all of them. If any reject, the whole thing rejects.

Other combinators: - Promise.allSettled([...]) — waits for all, returns array of {status, value} or {status, reason} for each. - Promise.race([...]) — settles when the first one settles, with that value. - Promise.any([...]) — settles with the first fulfilled value, or rejects if all reject.

11.7 Common Async Patterns

Loading data on a page

async function showUserProfile(userId) {
  const user = await fetch(`/api/users/${userId}`).then(r => r.json());
  document.getElementById("name").textContent = user.name;
  document.getElementById("email").textContent = user.email;
}

A sleep utility

function sleep(ms) {
  return new Promise(resolve => setTimeout(resolve, ms));
}

async function countdown() {
  for (let i = 5; i > 0; i--) {
    console.log(i);
    await sleep(1000);
  }
  console.log("Go!");
}

Retry with backoff

async function fetchWithRetry(url, tries = 3) {
  for (let i = 0; i < tries; i++) {
    try {
      const response = await fetch(url);
      if (!response.ok) throw new Error(`HTTP ${response.status}`);
      return response;
    } catch (e) {
      if (i === tries - 1) throw e;            // last attempt: re-throw
      await sleep(2 ** i * 1000);               // exponential backoff
    }
  }
}

Loading multiple resources

async function loadDashboard() {
  const [user, posts, notifications] = await Promise.all([
    fetch("/api/me").then(r => r.json()),
    fetch("/api/posts").then(r => r.json()),
    fetch("/api/notifications").then(r => r.json()),
  ]);
  return { user, posts, notifications };
}

11.8 The Microtask Queue — Why Order is Tricky

A subtle but important detail. JS has TWO queues for async work:

  • Task queue (or "macrotask"): setTimeout, setInterval, I/O callbacks, UI events.
  • Microtask queue: Promise callbacks (.then, .catch, await resumptions).

After each synchronous task finishes, all microtasks run before the next macrotask. This affects ordering:

console.log("1");

setTimeout(() => console.log("2"), 0);

Promise.resolve().then(() => console.log("3"));

console.log("4");

// Output: 1, 4, 3, 2

Even though setTimeout(0) looks "immediate," the Promise's .then fires first because microtasks have priority.

You rarely need to reason about this directly, but it explains some confusing output in mixed async code.

11.9 Common Mistakes

Forgetting await:

async function bad() {
  const data = fetch("/api/data");      // Promise, not the data!
  console.log(data);                     // logs a Promise object
}

Sequential when you meant parallel:

async function slow() {
  const a = await fetchA();              // ← if these don't depend on each other,
  const b = await fetchB();              //   use Promise.all
}

Unhandled rejection:

async function risky() {
  const data = await fetch("/api/data");
  // if fetch rejects, nothing catches it
}
risky();  // unhandled promise rejection

Always try/catch async work that can fail, or attach .catch().

Treating async like sync for return values:

function getData() {
  fetch("/api/data").then(data => return data);  // ← doesn't work
}
// Use async/await instead:
async function getData() {
  return await fetch("/api/data");
}

For independent operations, always parallelize. It's a free win. A note on `await` at the top level: it's allowed in modules (with `type="module"` or in modern Node), not in regular scripts. The `await fetchAllSequential(...)` above assumes you're inside an async context or a module.
SECTION 12

Iterators & Generators

This section is more advanced. You can skip it on a first read and come back — but knowing how iteration really works in JS unlocks some powerful patterns.

12.1 The Iteration Protocol

You've used for...of on arrays, strings, Maps, Sets. What makes those iterable, and how can you make your own?

The iterable protocol says: any object with a [Symbol.iterator] method that returns an iterator can be used with for...of, spread (...), destructuring, and friends.

An iterator is an object with a .next() method that returns { value, done }:

function makeIterator() {
  let i = 0;
  return {
    next() {
      if (i < 3) return { value: i++, done: false };
      return { value: undefined, done: true };
    }
  };
}

const it = makeIterator();
console.log(it.next());    // { value: 0, done: false }
console.log(it.next());    // { value: 1, done: false }
console.log(it.next());    // { value: 2, done: false }
console.log(it.next());    // { value: undefined, done: true }

To make an object iterable, add a [Symbol.iterator] method that returns an iterator:

class Range {
  constructor(start, end) {
    this.start = start;
    this.end = end;
  }

  [Symbol.iterator]() {
    let current = this.start;
    const end = this.end;
    return {
      next() {
        if (current < end) return { value: current++, done: false };
        return { value: undefined, done: true };
      }
    };
  }
}

for (const n of new Range(1, 5)) {
  console.log(n);    // 1, 2, 3, 4
}

[...new Range(0, 3)];    // [0, 1, 2]

Symbol.iterator is a special key — a unique "well-known" symbol JS uses for this protocol.

12.2 Generators — Iterators Made Easy

Writing next() methods by hand is tedious. Generator functions make iterators trivial:

function* counter() {
  yield 1;
  yield 2;
  yield 3;
}

const gen = counter();
console.log(gen.next());   // { value: 1, done: false }
console.log(gen.next());   // { value: 2, done: false }
console.log(gen.next());   // { value: 3, done: false }
console.log(gen.next());   // { value: undefined, done: true }

// And it's iterable:
for (const n of counter()) {
  console.log(n);   // 1, 2, 3
}

The function* syntax declares a generator. Inside it: - yield value pauses execution and produces a value. - Resumes at that point on the next .next() call. - When the function returns (implicitly or explicitly), the iterator is done.

The Range example becomes:

function* range(start, end) {
  for (let i = start; i < end; i++) {
    yield i;
  }
}

[...range(0, 5)];      // [0, 1, 2, 3, 4]

Way nicer.

12.3 Lazy Evaluation

Generators are lazy — they compute values on demand, not all at once. This makes infinite sequences possible:

function* naturals() {
  let n = 1;
  while (true) {
    yield n++;
  }
}

const gen = naturals();
console.log(gen.next().value);    // 1
console.log(gen.next().value);    // 2
console.log(gen.next().value);    // 3
// ...forever

You couldn't do this with an array — it would crash trying to allocate infinite memory. With a generator, only the requested values exist at any moment.

You can compose generators:

function* take(iter, n) {
  for (const value of iter) {
    if (n-- <= 0) break;
    yield value;
  }
}

[...take(naturals(), 5)];    // [1, 2, 3, 4, 5]

take(naturals(), 5) builds an iterator that yields only the first 5 naturals. The infinite generator never actually generates beyond the 5th item.

12.4 Other Things Generators Can Do

Generators are very powerful — but rarely used in everyday code. A few other features:

Two-way communication — pass values into the generator:

function* echo() {
  while (true) {
    const received = yield;
    console.log("got:", received);
  }
}

const gen = echo();
gen.next();                // prime
gen.next("hello");          // "got: hello"
gen.next("world");          // "got: world"

Delegation with yield*:

function* lettersThenNumbers() {
  yield* "abc";              // yields "a", "b", "c"
  yield* [1, 2, 3];           // yields 1, 2, 3
}

[...lettersThenNumbers()];   // ["a", "b", "c", 1, 2, 3]

Async generators — async function* { ... } yields Promises:

async function* fetchPages(url) {
  let next = url;
  while (next) {
    const response = await fetch(next);
    const data = await response.json();
    yield data;
    next = data.nextPageUrl;
  }
}

for await (const page of fetchPages("/api/posts")) {
  console.log(page);
}

for await...of consumes async iterables — pages of data arriving over time.

12.5 When to Use Generators

Be honest with yourself — most JS programmers go years without writing a generator. They're useful when:

  • You're building lazy data structures (infinite sequences, streams).
  • You're processing large datasets where pulling everything into memory would be wasteful.
  • You're consuming a paginated API (async generators excel here).
  • You're implementing custom iteration protocols on your own classes.

For 90% of code, regular array methods (map, filter, reduce) are clearer and sufficient. Generators are in the toolbox for the cases that need them.

SECTION 13

Errors & Error Handling

We touched on try/catch in Foundations. JavaScript's error system has its own conventions and a few unique features.

13.1 Throwing Errors

function divide(a, b) {
  if (b === 0) {
    throw new Error("Division by zero");
  }
  return a / b;
}

throw halts execution. The error propagates up the call stack until something catches it. If nothing catches it, the program crashes (in Node) or logs an uncaught exception (in the browser).

You can throw anything in JS — strings, numbers, even objects. But always throw Error objects (or subclasses):

throw new Error("Something went wrong");           // ✓ good
throw "Something went wrong";                       // ✗ bad — string, no stack trace
throw 42;                                           // ✗✗ very bad

Why? Error objects carry a stack trace (error.stack) showing where the error originated. Plain strings don't. Debugging without a stack trace is medieval.

13.2 Catching Errors

try {
  const result = divide(10, 0);
} catch (error) {
  console.error("Failed:", error.message);
}

try runs the risky code. catch runs if it throws. The error is whatever was thrown.

You can have an optional finally block that runs no matter what:

try {
  openFile();
  process();
} catch (e) {
  handleError(e);
} finally {
  closeFile();      // always runs — opened file is always closed
}

finally is for cleanup. Releases resources, closes connections, hides loading spinners.

13.3 Built-in Error Types

JavaScript has several built-in error types:

Type Thrown when
Error Generic error
TypeError Operation on the wrong type (null.foo, calling a non-function)
ReferenceError Using an undefined variable
SyntaxError Code can't be parsed
RangeError Value out of range (new Array(-1))
URIError Malformed URI
EvalError Issues with eval() (rare)

You can match specific types in catch:

try {
  someOperation();
} catch (e) {
  if (e instanceof TypeError) {
    // handle type errors specifically
  } else {
    throw e;     // re-throw what we don't recognize
  }
}

13.4 Custom Error Types

Defining your own errors lets you catch them specifically:

class ValidationError extends Error {
  constructor(field, message) {
    super(message);
    this.name = "ValidationError";
    this.field = field;
  }
}

function validateEmail(email) {
  if (!email.includes("@")) {
    throw new ValidationError("email", "Invalid email format");
  }
}

try {
  validateEmail("not-an-email");
} catch (e) {
  if (e instanceof ValidationError) {
    console.error(`Validation failed on ${e.field}: ${e.message}`);
  } else {
    throw e;
  }
}

Set this.name so the error displays clearly. Add any additional fields you need (field, code, etc.).

13.5 Async Errors

async/await integrates with regular try/catch:

async function loadUser(id) {
  try {
    const response = await fetch(`/api/users/${id}`);
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}`);
    }
    return await response.json();
  } catch (error) {
    console.error("Failed to load user:", error);
    return null;
  }
}

For raw Promises (without async/await), use .catch():

fetch("/api/data")
  .then(r => r.json())
  .then(data => use(data))
  .catch(err => handle(err));

Both work. async/await + try/catch is preferred in modern code.

13.6 Beware of "Catch and Ignore"

A common anti-pattern:

try {
  doSomething();
} catch (e) {
  // silence
}

This hides bugs. You can't debug what you don't know broke. Always: 1. Catch only the specific errors you know how to handle. 2. Log or re-throw what you don't recognize.

Acceptable cases for silent catch are rare — e.g., "this is optional functionality; if it fails, just skip it." Even then, log a warning.

13.7 The error.cause Property

Sometimes one error wraps another:

async function loadConfig() {
  try {
    const text = await readFile("config.json");
    return JSON.parse(text);
  } catch (originalError) {
    throw new Error("Failed to load config", { cause: originalError });
  }
}

{ cause: originalError } preserves the underlying error for debugging. In Node and modern browsers, error logs show the chain — your wrapper error followed by the original.

Added in ES2022. Useful for libraries that want to provide clear top-level errors without losing the original detail.

13.8 window.onerror and Unhandled Rejections

In browsers, uncaught errors and unhandled Promise rejections fire global events:

window.addEventListener("error", e => {
  // Uncaught synchronous errors
  console.error("Global error:", e.message, "at", e.filename, e.lineno);
});

window.addEventListener("unhandledrejection", e => {
  // Promises rejected with no .catch
  console.error("Unhandled rejection:", e.reason);
});

Real apps connect these to error reporting services (Sentry, Bugsnag, Rollbar). For learning, you can console-log them.

The DOM

This is where JavaScript becomes interactive. The DOM (Document Object Model) is JavaScript's representation of the HTML page. Manipulating it is how every interactive web feature works.

14.1 What is the DOM?

When a browser parses HTML, it builds a tree of objects representing the page. Each HTML element becomes a node in this tree. JavaScript can read and modify the tree at runtime — adding elements, changing their content, responding to events.

                    document
                       │
                    <html>
                    /     \
                <head>   <body>
                 │       /  |  \
              <title>  <h1> <p> <button>

The DOM API gives you tools to navigate and modify this tree.

14.2 Selecting Elements

// By ID (most common for unique elements)

// By CSS selector — the modern, most flexible way

// Returns a NodeList — array-like but not an actual array

14.3 Reading & Changing Content

const h1 = document.querySelector("h1");

// Text content (most common, safest)
h1.textContent;                       // get
h1.textContent = "New heading";        // set

// innerHTML — interprets HTML tags (DANGEROUS with user input!)

h1.innerHTML = "<em>New</em>";          // set, parsed as HTML

// innerText — what the user actually sees (respects CSS)

14.8 DOM Performance Tips

The DOM is slow compared to plain JS. A few pointers:

  • Batch reads and writes. Reading layout properties (offsetWidth, getBoundingClientRect) forces the browser to compute layout. Mixing reads and writes in a loop is the classic mistake.
  • Build first, attach last. When adding many elements, build them off-DOM (in a DocumentFragment or just a detached node), then attach in one shot.
  • Avoid innerHTML in hot loops. Re-parsing HTML for each item is slow. Build elements with createElement.

A common pattern for building lists efficiently:

const ul = document.querySelector("ul");
const fragment = document.createDocumentFragment();
for (const item of items) {
  const li = document.createElement("li");
  li.textContent = item.name;
  fragment.appendChild(li);
}
ul.appendChild(fragment);    // one DOM operation

For most casual JavaScript, performance isn't a concern. But when rendering thousands of items, these tricks matter.

SECTION 15

Events

Events are how a webpage responds to the user — clicks, keypresses, scrolls, form submissions, page loads, network responses. The event system in JS is rich and worth understanding well.

15.1 addEventListener — The Universal Pattern

The standard way to listen for events:

const button = document.querySelector("button");

button.addEventListener("click", function(event) {
  console.log("Clicked!", event);
});

Parameters: - The event name as a string: "click", "keydown", "submit", etc. - A handler function — receives an Event object.

You can attach multiple listeners to the same element. They all fire on the event:

button.addEventListener("click", logIt);
button.addEventListener("click", animateIt);
button.addEventListener("click", updateState);
// All three run when the button is clicked.

To remove a listener, you need the same function reference:

function handle() { console.log("hi"); }
button.addEventListener("click", handle);
button.removeEventListener("click", handle);

This means you can't easily remove inline arrow function listeners — keep a reference if you'll need to remove.

15.2 The Event Object

Every handler receives an event object with useful info:

button.addEventListener("click", function(event) {
  console.log(event.type);           // "click"
  console.log(event.target);         // the element that was clicked
  console.log(event.currentTarget);  // the element this listener is on
  console.log(event.timeStamp);      // when (in ms since page load)
  console.log(event.clientX, event.clientY);  // mouse position
});

Common properties: - event.target — the actual element that triggered (innermost). - event.currentTarget — the element the listener is attached to (may be an ancestor). - event.preventDefault() — stop the default browser action (e.g., form submission, link navigation). - event.stopPropagation() — stop the event from bubbling up.

15.3 Common Event Types

Mouse events

  • click — button click, link click, anything clickable.
  • dblclick — double-click.
  • mousedown, mouseup — separate from click.
  • mouseenter, mouseleave — hover (don't fire on children).
  • mouseover, mouseout — like enter/leave but fires on children too.
  • mousemove — fires constantly during movement (be careful, can flood handlers).

Keyboard events

  • keydown — key is pressed (fires repeatedly while held).
  • keyup — key is released.
  • keypress — deprecated, don't use.
document.addEventListener("keydown", function(e) {
  console.log(e.key);          // "a", "Enter", "ArrowUp", " " (space)
  console.log(e.code);         // physical key: "KeyA", "Enter", etc.
  console.log(e.ctrlKey, e.shiftKey, e.altKey, e.metaKey);
});

Use e.key for "what character does the user see" and e.code for "what physical key" (matters for games and certain shortcuts).

Form events

  • submit (on <form>) — triggered by submit button or Enter in input. Use preventDefault to handle in JS.
  • change — value changed and lost focus (good for select, checkbox, file).
  • input — value changed at all (fires on every keystroke).
  • focus, blur — element gained/lost focus.

Document / window events

  • DOMContentLoaded — HTML parsed, DOM ready. Most JS should wait for this.
  • load — entire page loaded (including images).
  • resize — window resized.
  • scroll — page scrolled.

15.4 Event Bubbling

When an event fires on an element, it then "bubbles" up through ancestors — parent, grandparent, all the way to document. Each level's listeners get a chance to handle it.

<div id="outer">
  <div id="inner">
    <button>Click me</button>
  </div>
</div>

document.getElementById("outer").addEventListener("click", () => console.log("outer"));
document.getElementById("inner").addEventListener("click", () => console.log("inner"));
document.querySelector("button").addEventListener("click", () => console.log("button"));

// Clicking the button logs:
// button
// inner
// outer

The event "bubbles up" — innermost first, then each ancestor in turn.

You can stop bubbling with event.stopPropagation():

button.addEventListener("click", function(e) {
  e.stopPropagation();
  console.log("button");
  // Now "inner" and "outer" don't fire.
});

Use stopPropagation carefully — sometimes it breaks other code that expected to receive the event.

15.5 Event Delegation

The most useful pattern in DOM programming. Attach one listener to a parent, handle events from any descendant.

Naïve approach — listener per item (slow, doesn't work for items added later):

document.querySelectorAll(".todo").forEach(item => {
  item.addEventListener("click", () => doSomething(item));
});

Delegated approach — one listener:

document.querySelector(".todo-list").addEventListener("click", function(e) {
  const item = e.target.closest(".todo");
  if (!item) return;          // clicked something else
  doSomething(item);
});

e.target.closest(".todo") walks up from the clicked element looking for an ancestor matching .todo. If found, we handle it.

Benefits: - One listener instead of N — less memory. - Works for items added later (after the listener was attached). - Easy to manage.

Used everywhere in real codebases. Learn this pattern.

15.6 Preventing Default Behavior

Some events have default browser behavior. Forms submit and reload the page. Links navigate. Drag-and-drop has built-in behavior.

event.preventDefault() cancels the default. Combined with handling the event yourself, this lets you build single-page apps:

document.querySelector("form").addEventListener("submit", async function(e) {
  e.preventDefault();             // don't reload the page

  const data = Object.fromEntries(new FormData(e.target));

document.querySelectorAll("a.spa-link").forEach(a => {
  a.addEventListener("click", function(e) {
    e.preventDefault();
    navigateInApp(a.href);          // custom client-side navigation
  });
});

15.8 Throttling and Debouncing

Some events fire constantly — mousemove, scroll, resize, input. Running expensive code on every fire is bad for performance. Two techniques:

Debouncing — wait for a pause before running. Useful for search-as-you-type (don't query the server on every keystroke; wait for the user to stop typing).

function debounce(fn, delay) {
  let timer;
  return function(...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

const searchAfterPause = debounce(query => {
  fetch(`/api/search?q=${query}`).then(...);
}, 300);

input.addEventListener("input", e => searchAfterPause(e.target.value));

Throttling — run at most once per interval. Useful for scroll handlers, resize, mousemove.

function throttle(fn, interval) {
  let lastCall = 0;
  return function(...args) {
    const now = Date.now();
    if (now - lastCall < interval) return;
    lastCall = now;
    fn.apply(this, args);
  };
}

window.addEventListener("scroll", throttle(() => {
  console.log("scrolled");
}, 100));

In real code, use these (or a library like Lodash's _.debounce / _.throttle). The naive "run on every event" approach kills performance on high-frequency events.

SECTION 16

Fetch & APIs

Most JavaScript applications talk to a server. The browser provides fetch() — a modern, Promise-based API for HTTP requests. This section covers fetch in depth: GET, POST, error handling, JSON, FormData, AbortController, building a small API client.

16.1 The Basics

const response = await fetch("https://jsonplaceholder.typicode.com/users/1");
const user = await response.json();
console.log(user);

That's it. fetch(url) returns a Promise that resolves to a Response object. .json() reads the body and parses it.

Critical to understand: the Promise resolves as soon as headers arrive — not when the body is downloaded. To get the body, you call .json(), .text(), .blob(), .arrayBuffer(), or .formData() — each returns another Promise.

16.2 The Response Object

const response = await fetch("/api/data");

response.ok;              // true if status is 200-299
response.status;          // 200, 404, 500, etc.
response.statusText;      // "OK", "Not Found", etc.
response.headers;         // Headers object
response.headers.get("content-type");   // "application/json"

// Body — pick one method (consumes the body, can only call once):
await response.json();         // parse as JSON
await response.text();         // raw text
await response.blob();         // binary (for images, files)
await response.formData();     // FormData object
await response.arrayBuffer();  // raw bytes

16.3 The Critical Gotcha — Fetch Doesn't Throw on HTTP Errors

This is the #1 fetch surprise. A 404 or 500 response is NOT a rejection. The Promise resolves normally with response.ok = false. You have to check yourself:

// WRONG — assumes failure throws
try {
  const response = await fetch("/api/data");
  const data = await response.json();
  use(data);
} catch (e) {
  // This only catches network failures (offline, CORS, etc.)
  // It does NOT catch 404, 500, etc.
}

// RIGHT — check response.ok
try {
  const response = await fetch("/api/data");
  if (!response.ok) {
    throw new Error(`HTTP ${response.status}: ${response.statusText}`);
  }
  const data = await response.json();
  use(data);
} catch (e) {
  // Catches both network errors AND HTTP errors
}

This design is intentional but trips up everyone. Most projects wrap fetch in a helper that throws on !ok.

16.4 POST, PUT, DELETE — Beyond GET

By default fetch does GET. For other methods, pass options:

// POST JSON
const response = await fetch("/api/users", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ name: "Alice", email: "alice@x.com" }),
});

// PUT — replace a resource
await fetch("/api/users/42", {
  method: "PUT",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ name: "Alice Updated" }),
});

// PATCH — partial update
await fetch("/api/users/42", {
  method: "PATCH",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ email: "new@x.com" }),
});

// DELETE
await fetch("/api/users/42", { method: "DELETE" });

The Content-Type header tells the server how to parse the body. Forget it, and the server may interpret your JSON as plain text.

16.5 Sending Form Data

For form submissions, you can send FormData directly (no JSON):

const form = document.querySelector("form");
const data = new FormData(form);          // grabs all named inputs
data.append("extra", "value");             // add more fields if needed

await fetch("/api/upload", {
  method: "POST",
  body: data,                              // no Content-Type header — browser sets it
});

FormData is the way to send file uploads — it handles multipart encoding automatically. Don't set Content-Type manually — the browser needs to add the multipart boundary.

16.6 Query Parameters

Building URL query strings by hand is error-prone (escaping &, =, etc.). Use URLSearchParams:

const params = new URLSearchParams({ q: "javascript & tips", page: 1 });
const url = `/api/search?${params}`;
// /api/search?q=javascript+%26+tips&page=1

const response = await fetch(url);

Or build a URL object:

const url = new URL("/api/search", "https://example.com");
url.searchParams.set("q", "javascript");
url.searchParams.set("page", 2);
await fetch(url);

16.7 AbortController — Cancelling Requests

Sometimes a request becomes stale — the user typed something new, navigated away, hit "cancel." You can cancel fetches with an AbortController:

const controller = new AbortController();

fetch("/api/slow", { signal: controller.signal })
  .then(r => r.json())
  .catch(e => {
    if (e.name === "AbortError") console.log("Cancelled");
    else throw e;
  });

// Cancel after 3 seconds
setTimeout(() => controller.abort(), 3000);

The most common real-world use is search-as-you-type — cancel the previous request when the user keeps typing:

let currentController = null;

input.addEventListener("input", async (e) => {
  if (currentController) currentController.abort();
  currentController = new AbortController();

  try {
    const response = await fetch(`/api/search?q=${e.target.value}`, {
      signal: currentController.signal,
    });
    const results = await response.json();
    renderResults(results);
  } catch (e) {
    if (e.name !== "AbortError") console.error(e);
  }
});

Without abort, you risk race conditions — request A returns after request B (which was newer), and you display stale results.

16.8 A Real API Client Helper

Wrap fetch in a small utility to handle the common needs:

async function api(path, options = {}) {
  const { body, ...rest } = options;
  const response = await fetch(path, {
    ...rest,
    headers: {
      "Content-Type": "application/json",
      ...(rest.headers || {}),
    },
    body: body ? JSON.stringify(body) : undefined,
  });

  if (!response.ok) {
    const text = await response.text();
    throw new Error(`${response.status} ${response.statusText}: ${text}`);
  }

  // Return JSON, or null for empty bodies
  const text = await response.text();
  return text ? JSON.parse(text) : null;
}

// Usage:
const user = await api("/api/users/1");
const created = await api("/api/users", { method: "POST", body: { name: "Alice" } });
await api("/api/users/1", { method: "DELETE" });

This is roughly 80% of what libraries like Axios provide. For most apps, a 20-line helper is enough.

16.9 CORS — The Cross-Origin Hurdle

If your JS at https://my-site.com tries to fetch("https://api.other-site.com/data"), the browser blocks it unless the API explicitly allows your origin. This is CORS — Cross-Origin Resource Sharing.

The server needs to send headers like:

Access-Control-Allow-Origin: https://my-site.com

You'll meet CORS errors constantly when building anything that talks to third-party APIs. Symptoms: - "blocked by CORS policy" in the console. - The request itself succeeds (server returns 200), but the browser refuses to give your JS the result.

Solutions vary by situation: - Your own API: configure CORS headers on the server. - Third-party API: only works if they allow your origin. Often you have to proxy through your own backend. - Local development: use a dev server with proxy (Vite has this built in).

CORS exists for security — without it, any random site you visit could read your Gmail. The pain is the price of safety.

SECTION 17

Storage & Modern Browser APIs

The browser is more than just the DOM. There are dozens of APIs for storage, observing changes, intersection with the viewport, geolocation, notifications, and more. This section covers the most important ones — particularly persistence with localStorage and the observer family.

17.1 localStorage — Persistent Key-Value Storage

localStorage saves data in the browser, persistent across reloads and even browser restarts. Each origin (protocol + domain + port) gets its own storage.

// Save
localStorage.setItem("username", "Alice");
localStorage.setItem("theme", "dark");

// Read
localStorage.getItem("username");        // "Alice"
localStorage.getItem("missing");         // null

// Remove
localStorage.removeItem("theme");

// Clear everything
localStorage.clear();

// Iterate
for (let i = 0; i < localStorage.length; i++) {
  const key = localStorage.key(i);
  console.log(key, localStorage.getItem(key));
}

// Or via Object.keys workaround:
Object.keys(localStorage).forEach(k => console.log(k, localStorage[k]));

Critical limitation: localStorage only stores strings. To save objects, JSON.stringify first:

// Save an object
const user = { name: "Alice", prefs: { theme: "dark" } };
localStorage.setItem("user", JSON.stringify(user));

// Read it back
const stored = JSON.parse(localStorage.getItem("user"));
console.log(stored.name);    // "Alice"

A small helper makes this clean:

function saveJSON(key, value) {
  localStorage.setItem(key, JSON.stringify(value));
}
function loadJSON(key, fallback = null) {
  const raw = localStorage.getItem(key);
  if (raw === null) return fallback;
  try {
    return JSON.parse(raw);
  } catch {
    return fallback;
  }
}

Storage limits: browsers typically allow ~5-10 MB per origin. Plenty for preferences and small state; not enough for entire databases.

What localStorage is good for: - User preferences (theme, sidebar collapsed, sort order). - Authentication tokens (with caveats — see "security" below). - Small caches. - Saving drafts ("don't lose what they typed if they refresh").

Security caveat: any JS running on your origin can read everything in localStorage. If you have an XSS vulnerability, the attacker walks away with any tokens stored there. For sensitive auth, prefer HTTP-only cookies.

17.2 sessionStorage — Per-Tab Storage

Same API as localStorage, but data clears when the tab closes:

sessionStorage.setItem("checkout-step", "2");

Useful for state that's meaningful within a single session but shouldn't outlive the tab — wizard progress, transient form data, "you have unsaved changes" warnings.

17.4 IndexedDB — When You Need a Real Database

For larger data — thousands of records, complex queries — localStorage's 5MB limit and synchronous API don't cut it. IndexedDB is a real client-side database, asynchronous, with indexes, transactions, and a much larger quota (often hundreds of MB).

The native API is notoriously verbose:

const request = indexedDB.open("MyDB", 1);
request.onupgradeneeded = (e) => {
  const db = e.target.result;
  db.createObjectStore("users", { keyPath: "id" });
};
request.onsuccess = (e) => {
  const db = e.target.result;
  const tx = db.transaction("users", "readwrite");
  const store = tx.objectStore("users");
  store.add({ id: 1, name: "Alice" });
};

In practice, almost no one uses IndexedDB directly. Wrappers like Dexie or idb make it palatable:

// Using Dexie
import Dexie from "dexie";

const db = new Dexie("MyApp");
db.version(1).stores({ users: "++id, name, email" });

await db.users.add({ name: "Alice", email: "alice@x.com" });
const allUsers = await db.users.toArray();

If you need a client-side database (offline apps, large local data), reach for Dexie. Otherwise, stick with localStorage.

17.5 The Observer Family

The browser has a family of "observer" APIs that watch for changes — to elements scrolling into view, to attributes changing, to element sizes changing. They replaced older, more expensive patterns (polling, scroll handlers everywhere).

IntersectionObserver — react when elements enter the viewport

The classic use case: lazy-loading images, infinite scroll, animating elements when they appear.

const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      console.log(entry.target, "is visible");
      entry.target.classList.add("visible");
      observer.unobserve(entry.target);   // optional — stop watching after first reveal
    }
  });
});

document.querySelectorAll(".lazy").forEach(el => observer.observe(el));

Before IntersectionObserver, you'd attach a scroll listener and compute geometry on every scroll event — slow and complex. IntersectionObserver is efficient and dead simple.

MutationObserver — watch for DOM changes

Detect when something else modifies the DOM:

  subtree: true,          // also descendants
  attributes: true,       // attribute changes
  characterData: true,    // text changes
});

Use cases: - Detecting when third-party code injects content. - Building libraries that need to react to user-added elements. - Browser extensions that inject UI.

Rare in app code; common in libraries.

ResizeObserver — watch element size changes

const observer = new ResizeObserver((entries) => {
  entries.forEach(entry => {
    console.log("Width:", entry.contentRect.width);

});

observer.observe(document.querySelector(".panel"));

Older code used window.resize events, but those fire only on viewport resize — not when an element's parent changes size for other reasons. ResizeObserver works for any element.

Node.js Fundamentals

Until now, every line of JavaScript we wrote ran in a browser. Node.js is the runtime that lets JavaScript run outside the browser — on servers, as command-line tools, as build scripts. This section covers what Node is and how to use it.

18.1 What is Node.js?

Node.js is, basically, Google Chrome's V8 JavaScript engine extracted from the browser, packaged with: - A standard library for file I/O, networking, processes, crypto, etc. - The CommonJS module system (and now ESM support). - A built-in package manager: npm.

It was released in 2009 by Ryan Dahl. The big idea: bring JS's async, non-blocking model to server-side programming.

What this means for you: - The same language now powers your frontend AND your backend. - One ecosystem (npm) covers both. - You can use modern JS features on the server immediately, without browser compatibility worries.

18.2 Installing & Running

Get Node from nodejs.org. The LTS (Long-Term Support) version is the right pick unless you have a reason to be cutting-edge.

Verify:

$ npm --version
10.2.3

Run JS interactively (a REPL):

> .exit

18.6 Modules in Node

Node supports both module systems we covered in Section 10. Quick recap.

CommonJS (Node's traditional, still ubiquitous):

// math.js
function square(n) { return n * n; }
module.exports = { square };

// app.js
const { square } = require("./math");
console.log(square(5));

ES Modules: Add "type": "module" to package.json, or use the .mjs extension:

// app.mjs

console.log(square(5));

New projects today often use ES modules. Many existing libraries still ship CommonJS. The two can interop, with some friction. Picking ESM for new code is generally fine.

18.7 Filesystem — Reading and Writing Files

fs is the filesystem module. Async (modern, Promise-based):

import { readFile, writeFile, readdir } from "node:fs/promises";

// Read a file
const text = await readFile("notes.txt", "utf-8");
console.log(text);

// Write a file
await writeFile("output.txt", "Hello from Node!\n", "utf-8");

// List a directory
const files = await readdir(".");
console.log(files);

The "utf-8" argument decodes bytes as text. Without it, you get a Buffer (raw bytes).

Synchronous versions exist (readFileSync, etc.) — convenient for scripts but blocking. Avoid sync I/O in servers.

18.8 Paths — The path Module

Cross-platform path manipulation:

import path from "node:path";

path.join("data", "users", "alice.json");    // "data/users/alice.json" (or with \ on Windows)
path.resolve("data/users");                    // absolute path
path.basename("/foo/bar.txt");                  // "bar.txt"
path.dirname("/foo/bar.txt");                   // "/foo"
path.extname("bar.txt");                        // ".txt"

// Current file's directory (ESM):
import { fileURLToPath } from "node:url";
const __filename = fileURLToPath(import.meta.url);
const __dirname = path.dirname(__filename);

In CommonJS, __dirname and __filename are magic globals. In ESM, you have to derive them as above (Node has a proposal to add them back natively).

18.9 The Node Event Loop

Node uses the same single-threaded event loop concept as browsers, with some differences: - Optimized for I/O-heavy work (network, files, databases). - File I/O uses a thread pool internally so it doesn't block the main thread. - CPU-heavy work still blocks. For that, use worker threads (a more advanced topic).

The practical takeaway: Node servers handle thousands of concurrent connections with one thread, by never blocking on I/O. For most work, this scales beautifully.

19.2 Routing By Hand

A real server responds differently to different URLs. With the bare http module, you check req.url and req.method yourself:

const server = http.createServer((req, res) => {
  res.setHeader("Content-Type", "application/json");

  if (req.url === "/" && req.method === "GET") {
    res.end(JSON.stringify({ message: "Welcome!" }));
  } else if (req.url === "/users" && req.method === "GET") {
    res.end(JSON.stringify([{ name: "Alice" }, { name: "Bob" }]));
  } else if (req.url === "/users" && req.method === "POST") {
    let body = "";
    req.on("data", chunk => body += chunk);
    req.on("end", () => {
      const user = JSON.parse(body);
      res.end(JSON.stringify({ created: user }));
    });
  } else {
    res.writeHead(404);
    res.end(JSON.stringify({ error: "Not Found" }));
  }
});

This works but gets unwieldy fast. Real Node servers use a framework. The most popular is Express.

19.3 Express — The Standard Web Framework

Install:

Cleaner. Each route has its own handler. req.body automatically parses JSON. res.json() sets the Content-Type and serializes.

19.4 Building a Tiny REST API

Let's build a simple "todos" API. We'll keep data in memory (a real app uses a database — out of scope here).

import express from "express";

const app = express();
app.use(express.json());

// In-memory "database"
let nextId = 1;
let todos = [];

// GET /todos — list all
app.get("/todos", (req, res) => {
  res.json(todos);
});

// GET /todos/:id — one by id
app.get("/todos/:id", (req, res) => {
  const todo = todos.find(t => t.id === Number(req.params.id));
  if (!todo) return res.status(404).json({ error: "Not Found" });
  res.json(todo);
});

// POST /todos — create
app.post("/todos", (req, res) => {
  const { text } = req.body;
  if (!text) return res.status(400).json({ error: "text required" });
  const todo = { id: nextId++, text, done: false };
  todos.push(todo);
  res.status(201).json(todo);
});

// PATCH /todos/:id — update partially
app.patch("/todos/:id", (req, res) => {
  const todo = todos.find(t => t.id === Number(req.params.id));
  if (!todo) return res.status(404).json({ error: "Not Found" });
  Object.assign(todo, req.body);
  res.json(todo);
});

// DELETE /todos/:id
app.delete("/todos/:id", (req, res) => {
  const i = todos.findIndex(t => t.id === Number(req.params.id));
  if (i === -1) return res.status(404).json({ error: "Not Found" });
  const [removed] = todos.splice(i, 1);
  res.json(removed);
});

app.listen(3000, () => console.log("http://localhost:3000"));

19.5 Middleware — Express's Killer Feature

Middleware is a function that processes every request before it reaches your route handlers:

app.use((req, res, next) => {
  console.log(`${req.method} ${req.url}`);
  next();         // pass control to the next middleware/handler
});

app.use((req, res, next) => {
  const auth = req.headers.authorization;
  if (!auth) return res.status(401).json({ error: "Unauthorized" });
  next();
});

app.get("/data", (req, res) => {
  res.json({ secret: "stuff" });
});

Middlewares run in order. Each can: - Inspect the request. - Modify the request or response. - End the response (e.g., reject auth). - Call next() to continue.

Common built-in middlewares: - express.json() — parse JSON bodies. - express.urlencoded() — parse form-encoded bodies. - express.static("public") — serve static files (HTML, CSS, JS, images) from a folder.

Third-party ones are huge: cors, helmet (security headers), morgan (logging), express-rate-limit, compression, etc. Middleware is how Express stays small while supporting everything.

19.6 Serving Static Files

For a tiny full-stack site, Express can serve your HTML/CSS/JS alongside your API:

import express from "express";

const app = express();
app.use(express.static("public"));     // serves files from ./public
app.use(express.json());

app.get("/api/hello", (req, res) => {
  res.json({ message: "Hi from API" });
});

app.listen(3000);

Now http://localhost:3000/index.html serves public/index.html, and http://localhost:3000/api/hello returns JSON.

This is the simplest possible full-stack setup. For real production sites, you'd use a separate frontend build, a CDN, a reverse proxy — but for learning, this works.

19.7 Errors and Promises

Express handles errors via an "error-handling middleware" — a function with four parameters:

app.get("/risky", async (req, res, next) => {
  try {
    const data = await mightFail();
    res.json(data);
  } catch (err) {
    next(err);                    // pass to error handler
  }
});

// Error handler — note 4 params, including err
app.use((err, req, res, next) => {
  console.error(err);
  res.status(500).json({ error: "Server error" });
});

In Express 5 (in active rollout), unhandled Promise rejections in route handlers are automatically forwarded to error middleware — no try/catch needed. Until you're sure you're on v5, the explicit try/catch is safer.

19.8 Beyond Express

Express has been around forever. Newer frameworks exist: - Fastify — Express-like API, much faster. - Hono — modern, runs on Node, Bun, Deno, Cloudflare Workers, etc. - Koa — by the original Express creator, smaller core, async-first. - Next.js — full-stack framework with React, file-based routing. - NestJS — for those who like Angular-style structure.

For learning and most jobs, Express is still the safe default. Knowing it gets you 90% of the way to using any other Node framework.

🏋️ Exercise 19.8

Extend the todos API above with one new endpoint: GET /todos?done=true — filter by completion status. Calling /todos?done=true returns only done todos, /todos?done=false returns only pending ones, /todos returns all.

Hint: query parameters are available as req.query (an object).

Solution 
app.get("/todos", (req, res) => {
  let result = todos;
  if (req.query.done === "true") {
    result = todos.filter(t => t.done);
  } else if (req.query.done === "false") {
    result = todos.filter(t => !t.done);
  }
  res.json(result);
});
Things to notice: - Query params arrive as **strings**, not booleans — `req.query.done` is `"true"` or `"false"` (or `undefined`). - We default to "return all" when the filter is missing. - Other values (`?done=banana`) fall through to the default — return all. A real API might reject unknown values with a 400. Test it: 
$ curl 'http://localhost:3000/todos?done=true'
$ curl 'http://localhost:3000/todos?done=false'
$ curl 'http://localhost:3000/todos'
Real APIs grow more sophisticated filters (`?status=done&category=work&sort=-createdAt`). The query-parsing pattern is the same — read `req.query`, build your filter, return.

20.1 What You're Building

20.2 Requirements in Detail

Let's be precise about behavior, the way a real ticket would be.

Data model. Each task has: a unique id, a title (required), a description (optional), and a status ("todo", "doing", or "done"). The board is an ordered collection of tasks.

Adding a task. Each column has an "+ Add task" affordance. Clicking it reveals an input. Submitting creates a task in that column's status and clears the input. Empty titles are rejected.

Moving a task. Tasks are draggable. Dropping a task on a column moves it to that column's status. The task's position updates immediately and persists.

Editing a task. Clicking a task's title makes it editable in place. Pressing Enter or blurring saves. Escape cancels.

Deleting a task. Each task has a delete control (a small × button). Clicking it removes the task after a confirm.

Counts. Each column header shows the number of tasks currently in it, updating live.

Persistence. All changes save to localStorage immediately. Reloading the page restores the exact board state.

Quote of the day. On load, fetch a quote from a public API and display it in the header. If the fetch fails, show a sensible fallback — never a broken UI.

20.3 Recommended Build Order

Don't try to build it all at once. Here's the order I'd build it in, each step producing something that works before moving on.

Milestone 1 — Static layout. Build the HTML and CSS for three empty columns with headers. No JavaScript yet. Get it looking right first.

Milestone 2 — The data model + render. Write a Task class and a Board that holds tasks. Write a render() function that draws the current tasks into the right columns. Hardcode two or three tasks to test rendering.

Milestone 3 — Adding tasks. Wire up the "add task" inputs. New tasks appear in the right column. Re-render after each add.

Milestone 4 — Persistence. Save the board to localStorage on every change; load it on startup. Now your hardcoded tasks become real saved state.

Milestone 5 — Delete + counts. Add the delete button and the live per-column counts. Both are small once render is solid.

Milestone 6 — Drag and drop. The trickiest part. Make tasks draggable; make columns drop targets; update status on drop.

Milestone 7 — Inline editing. Click a title to edit it. Save on Enter/blur, cancel on Escape.

Milestone 8 — Quote of the day. Add the fetch call and the fallback. This is independent of everything else, so it can come last (or first, as a warm-up).

Building in this order means you always have a working app, and each milestone teaches one thing. If you get stuck on drag-and-drop (milestone 6), you still have a working task board — you can ship without it and add it later.

20.4 Guidance for the Hard Parts

A few pointers on the milestones people get stuck on, without giving away the whole solution.

Rendering cleanly (Milestone 2). The simplest reliable approach: on every change, clear each column's task container and rebuild it from the data. This "wipe and redraw" is less efficient than surgically updating the DOM, but for a board with dozens of tasks it's instant and far simpler to reason about. Don't prematurely optimize. Build each task element with document.createElement (avoid innerHTML with user text — remember XSS from the DOM section).

Persistence (Milestone 4). localStorage only stores strings. Serialize with JSON.stringify(tasks) on save; parse with JSON.parse on load. Wrap the load in a try/catch — corrupt storage shouldn't crash your app. Save after every mutation; a single save() method called at the end of each action keeps it simple.

Drag and drop (Milestone 6). The HTML Drag and Drop API has a specific dance: - Make a task draggable with the draggable="true" attribute. - On the task's dragstart, stash its id (e.g. on a module-level variable, or via event.dataTransfer.setData). - On each column, listen for dragover and call event.preventDefault() — without this, drop never fires (this trips up everyone). - On the column's drop, read the dragged id, update that task's status to the column's status, save, and re-render.

Inline editing (Milestone 7). The classic approach: replace the title element with an <input> (or make it contenteditable) when clicked. Listen for keydown (Enter to save, Escape to cancel) and blur (save). On save, update the task's title in the data and re-render.

The quote fetch (Milestone 8). Use async/await inside an immediately-invoked function or a named init function. Wrap the fetch in try/catch. If it throws (offline, API down, CORS), fall back to a hardcoded quote so the header never looks broken.

20.6 Walkthrough of the Key Decisions

A few things in that solution are worth calling out, because they reflect real engineering judgment you can talk about in an interview.

Why a class for the whole app? TaskFlow bundles state (this.tasks) with the operations on it (add, move, edit, remove, render). That's encapsulation — the same principle behind every framework's component model, done by hand. An interviewer who sees this knows you understand why frameworks organize code the way they do.

Why wipe-and-redraw in render()? Simplicity beats cleverness here. Rebuilding the DOM on each change is trivially fast for this scale and eliminates a whole category of "the UI is out of sync with the data" bugs. Knowing when not to optimize is senior behavior.

Why textContent instead of innerHTML for task titles? Security. If a user names a task <img src=x onerror=alert(1)>, innerHTML would execute it (XSS). textContent treats it as literal text. This is the DOM-security lesson from Section 14, applied.

Why the dragover preventDefault? The Drag and Drop API defaults to not allowing drops. Calling preventDefault in dragover is what signals "this is a valid drop target." Leaving it out is the single most common drag-and-drop bug — worth remembering.

Why wrap storage in try/catch? localStorage can throw (private browsing, quota exceeded, corrupt data). A robust app degrades gracefully instead of crashing. Same for the quote fetch — the fallback means a network failure never produces a broken header.
