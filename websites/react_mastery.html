SECTION 1

Getting Started with React

1.1 What React is, and the problem it solves

Before React, building an interactive web page meant manually reaching into the DOM and changing it: find an element, update its text, add a class, remove a node, wire up an event, and remember to keep every piece consistent as data changed. For anything complex, this became a tangle — the app's state (the data) and the DOM (what's on screen) drifted apart, and bugs bred in the gap. You built exactly this by hand in the JavaScript course, and you felt the friction.

React is a library for building user interfaces out of components, declaratively. Two words in that sentence carry the whole idea:

  • Components — you break the UI into small, reusable, self-contained pieces (a button, a search bar, a comment, a whole page), each managing its own markup and logic. You compose them like functions.
  • Declaratively — instead of writing step-by-step instructions to mutate the DOM ("find this node, change that text"), you describe what the UI should look like for a given state, and React figures out how to make the real DOM match. You write "for this data, show this," and when the data changes, React updates the screen for you.

That second point is the mental shift. In vanilla JavaScript you command the DOM. In React you declare the result and let React reconcile the difference. This is why React apps are easier to reason about: your UI is a function of your state. Change the state, and the UI follows automatically.

1.2 The virtual DOM, briefly

You'll hear "virtual DOM" constantly, so here's the honest, non-mystical version. Updating the real DOM is comparatively slow, and doing it carelessly (re-rendering everything on every change) would be sluggish. React keeps a lightweight JavaScript representation of your UI in memory — the virtual DOM. When your state changes, React builds a new virtual tree, diffs it against the previous one, and applies only the minimal set of real-DOM changes needed. You get the simplicity of "re-describe the whole UI" with performance close to hand-tuned updates. You don't manage any of this — it's just why the declarative model can be fast.

1.3 Setting up with Vite

Modern React projects use a build tool. Vite is the current standard: fast, minimal, and the default recommendation. You'll need Node.js installed (the JavaScript course covered this). Create a project:

npm create vite@latest my-react-app -- --template react
cd my-react-app
npm install
npm run dev

Vite prints a local URL (usually http://localhost:5173). Open it and you'll see a starter page. Leave npm run dev running — it watches your files and hot-reloads the browser the instant you save. This feedback loop is a joy; keep it visible as you work.

The files that matter right now:

  • index.html — the single HTML page; it contains one <div id="root"></div> that React fills.
  • src/main.jsx — the entry point that mounts your app into that root.
  • src/App.jsx — the root component. This is where you'll start.

1.4 Your first component

Open src/App.jsx, delete its contents, and write this:

function App() {
  return <h1>Hello, React!</h1>;
}

export default App;

Save, and the browser updates. You just wrote a component. Look at what it is: a plain JavaScript function that returns some markup. That's the essence of a React component — a function that returns a description of UI. The markup-looking thing it returns is JSX, the subject of Section 2.

A few rules visible already:

  • A component is a function whose name is Capitalized (App, not app). React uses the capital letter to tell your components apart from regular HTML tags.
  • It returns a single piece of UI.
  • It's exported so other files (here, main.jsx) can use it.

1.5 Rendering and the entry point

How does App get onto the page? Look at src/main.jsx:

import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import App from "./App.jsx";

createRoot(document.getElementById("root")).render(
  <StrictMode>
    <App />
  </StrictMode>
);

createRoot(...).render(...) is the one place React connects to the real DOM: it takes the #root div from index.html and renders your <App /> into it. Notice <App /> is used like an HTML tag — that's how you render a component. StrictMode is a development-only helper that surfaces potential problems; leave it on.

You'll rarely touch main.jsx again. From here on you build components and compose them; React handles getting them on screen.

1.6 Composing components

The power comes from combining components. Let's make a couple and nest them:

function Header() {
  return <h1>My App</h1>;
}

function Welcome() {
  return <p>Thanks for stopping by.</p>;
}

function App() {
  return (
    <div>
      <Header />
      <Welcome />
    </div>
  );
}

export default App;

App renders Header and Welcome by using them as tags. This is composition — the same idea as calling functions from functions, but for UI. Notice the parentheses around the returned JSX: when JSX spans multiple lines, wrap it in ( ) so JavaScript reads it as one expression. And notice everything is wrapped in a single <div> — a component must return one top-level element (you'll see a nicer way, fragments, in Section 2).

Already you can feel the model: an app is a tree of components, each a function returning UI, composed together. Every React app you'll ever see — from a to-do list to Facebook — is this idea scaled up.

🏋️ Exercise 1.2

In your own words, what does it mean that React is declarative, and how does that differ from the vanilla-JavaScript DOM manipulation you did before?

Solution

Declarative means you describe what the UI should look like for a given state, and React makes the real DOM match — you don't write the update steps yourself. In vanilla JavaScript you were imperative: you manually found DOM nodes and mutated them (element.textContent = ..., element.classList.add(...)), and you had to keep the DOM in sync with your data by hand. In React, the UI is a function of state: you change the state, and React re-derives and updates the screen for you. Less bookkeeping, fewer sync bugs.

🏋️ Exercise 1.3

What's the significance of the capital letter in a component name like Header? What would happen if you named it header and rendered <header />?

Solution

React uses capitalization to distinguish your components from built-in HTML elements. A capitalized tag (<Header />) tells React to render your component; a lowercase tag (<header />) is treated as the literal HTML <header> element. So naming your component header and writing <header /> would render an empty HTML <header> tag — not your component — and it would silently not work as intended. Always capitalize component names.

SECTION 2

JSX in Depth

2.1 What JSX actually is

JSX is the HTML-looking syntax you've been writing inside components. It is not HTML and it is not a string — it's a syntax extension to JavaScript that describes UI. Your build tool (Vite) compiles it into plain JavaScript function calls before it ever reaches the browser.

This line:

const element = <h1 className="title">Hello</h1>;

compiles to roughly:

const element = React.createElement("h1", { className: "title" }, "Hello");

You almost never write createElement by hand — that's the whole point of JSX — but knowing it's just function calls producing JavaScript objects demystifies everything else. JSX isn't magic markup; it's a pleasant way to write those calls. Because it compiles to expressions, JSX obeys JavaScript rules, which is why the details below matter.

2.2 Embedding JavaScript with { }

Anywhere inside JSX, curly braces { } drop you back into JavaScript. Whatever expression you put inside is evaluated and inserted:

function Greeting() {
  const name = "Arthur";
  const hour = new Date().getHours();
  return (
    <div>
      <h1>Hello, {name}!</h1>
      <p>2 + 2 = {2 + 2}</p>
      <p>It is {hour < 12 ? "morning" : "afternoon"}.</p>
    </div>
  );
}

Anything that produces a value works: variables, arithmetic, function calls, ternaries, property access. Note the word expression — you can put things that evaluate to a value inside { }, but not statements like if or for (those don't return a value). This constraint shapes how you write conditionals and lists in JSX, as you'll see.

2.3 Attributes are camelCase, and className

JSX attributes look like HTML but follow JavaScript naming. The two you'll hit immediately:

  • class becomes className — because class is a reserved word in JavaScript. <div className="card">, not <div class="card">.
  • Multi-word attributes are camelCase — onclick → onClick, tabindex → tabIndex, maxlength → maxLength.

You pass JavaScript values to attributes with { }:

const url = "/logo.png";
const size = 64;
return <img src={url} width={size} alt="Logo" className="logo" />;

String literals use quotes (alt="Logo"); dynamic values use braces (src={url}). Self-closing tags must include the slash: <img />, <br />, <input />.

2.4 Conditional rendering

Because you can only put expressions in JSX, you render conditionally using expressions rather than if statements. Three patterns cover almost everything.

The ternary — choose between two things:

return <p>{isLoggedIn ? "Welcome back!" : "Please log in."}</p>;

Logical && — render something or nothing:

return (
  <div>
    {hasErrors && <p className="error">Something went wrong.</p>}
    {items.length === 0 && <p>No items yet.</p>}
  </div>
);

condition && <JSX /> renders the JSX when the condition is true, and nothing when false. (One gotcha: count && <p>...</p> renders a literal 0 when count is 0, because 0 is falsy but still a renderable value — use count > 0 && ... to be safe.)

An if before the return — for bigger branches, compute first:

function Status({ state }) {
  if (state === "loading") return <Spinner />;
  if (state === "error") return <ErrorBox />;
  return <Content />;
}

Returning early from the component is perfectly idiomatic when whole different UIs are involved.

2.5 Rendering lists with map and key

To render a collection, transform an array into an array of JSX with .map() — the array method you know well from JavaScript:

function TodoList() {
  const todos = ["Learn JSX", "Build a component", "Ship it"];
  return (
    <ul>
      {todos.map((todo) => (
        <li key={todo}>{todo}</li>
      ))}
    </ul>
  );
}

The {todos.map(...)} expression produces an array of <li> elements, which React renders in order. The critical detail is the key prop. React uses key to track which items changed, moved, or were removed between renders, so it can update the list efficiently and correctly. Rules for keys:

  • Every element in a list needs a key.
  • Keys must be unique among siblings and stable across renders.
  • Use a real unique id from your data (todo.id) when you have one — that's ideal.
  • Avoid using the array index as a key when the list can reorder, insert, or delete, because indices shift and React will mis-track items, causing subtle bugs.

Forgetting keys triggers a console warning; ignoring it leads to strange list behavior. Give React a good key and lists just work.

2.6 Fragments — grouping without extra DOM

A component returns one top-level element, but wrapping everything in a <div> litters your DOM with meaningless containers. A fragment groups children without adding a node:

import { Fragment } from "react";

function Info() {
  return (
    <Fragment>
      <h2>Title</h2>
      <p>Body text.</p>
    </Fragment>
  );
}

The shorthand for a fragment is empty tags — <> and </> — which you'll use most of the time:

function Info() {
  return (
    <>
      <h2>Title</h2>
      <p>Body text.</p>
    </>
  );
}

Same result, no wrapper <div> in the output. Reach for a fragment whenever you need to return siblings but don't want an extra element.

2.7 A real example — a product card

Putting the section together into something realistic:

function ProductCard() {
  const product = {
    name: "Mechanical Keyboard",
    price: 129,
    inStock: true,
    tags: ["wireless", "RGB", "hot-swappable"],
  };

  return (
    <div className="card">
      <h2>{product.name}</h2>
      <p className="price">${product.price}</p>

      {product.inStock ? (
        <span className="badge in-stock">In stock</span>
      ) : (
        <span className="badge sold-out">Sold out</span>
      )}

      <ul className="tags">
        {product.tags.map((tag) => (
          <li key={tag}>{tag}</li>
        ))}
      </ul>
    </div>
  );
}

Every tool from this section is here: { } for values, className, a ternary for the stock badge, .map() with a key for the tags. This is exactly what everyday JSX looks like — expressions woven through markup.

🏋️ Exercise 2.4

Why can't you write { if (x) { ... } } directly inside JSX? What do you use instead for a two-way choice, and for whole-component branching?

Solution

JSX braces { } accept only expressions (things that evaluate to a value), and if is a statement, not an expression — it produces no value to insert. For a two-way choice inside JSX, use a ternary: {cond ? <A /> : <B />}. For whole different UIs, use an if with an early return before the JSX: if (state === "loading") return <Spinner />;. Both are expression-friendly ways to branch.

SECTION 3

Components & Props

Components are the nouns of React; props are how you pass information to them. Together they give you reusable, configurable UI. This section is short but foundational — nearly everything you build rests on it.

3.1 Props: passing data into a component

A component that always renders the same thing isn't very useful. Props (short for "properties") let a parent pass data down to a child, exactly like passing arguments to a function. You pass them as attributes when rendering the component, and receive them as a single object parameter:

function Greeting(props) {
  return <h1>Hello, {props.name}!</h1>;
}

function App() {
  return (
    <div>
      <Greeting name="Arthur" />
      <Greeting name="Sam" />
      <Greeting name="Priya" />
    </div>
  );
}

One Greeting component, three different outputs. The parent passes name="Arthur" and the child reads props.name. Props are the mechanism that makes components reusable — the same component, configured differently each time.

3.2 Destructuring props

Reading props.name, props.age, props.email gets repetitive. Because props arrive as a plain object, you destructure them — usually right in the parameter list — which is the idiomatic style you'll see everywhere:

function UserCard({ name, age, email }) {
  return (
    <div className="card">
      <h2>{name}</h2>
      <p>Age: {age}</p>
      <p>{email}</p>
    </div>
  );
}

// used as:
<UserCard name="Arthur" age={31} email="arthur@example.com" />

Note the value types: strings use quotes (name="Arthur"), while numbers, booleans, arrays, objects, and any JavaScript expression use braces (age={31}). age="31" would pass the string "31"; age={31} passes the number.

3.3 Props are read-only

This is a rule React takes seriously: a component must never modify its own props. Props flow down from parent to child, and the child treats them as read-only inputs. Trying to reassign props.name = "x" is a mistake React warns about.

This is the principle of one-way data flow: data moves down the component tree through props. If a child needs to change something, it doesn't edit the prop — it calls a function the parent passed down (you'll do this with events and state in the coming sections). Keeping data flow one-directional is a big part of why React apps are predictable: you can always trace where a piece of data came from — it came from above.

3.4 The children prop

There's one special prop. Whatever you put between a component's opening and closing tags arrives as props.children:

function Card({ children }) {
  return <div className="card">{children}</div>;
}

function App() {
  return (
    <Card>
      <h2>Any content at all</h2>
      <p>Goes here, as children.</p>
    </Card>
  );
}

Card doesn't know or care what's inside it — it just renders children in a styled wrapper. This is the key to building flexible container components (layouts, cards, modals, buttons) that wrap arbitrary content. It's composition at its most powerful: the parent decides the contents, the component decides the frame.

3.5 Default and computed props

You can give props defaults using JavaScript's default parameter syntax, and you can compute values from props inside the component:

function Avatar({ src, size = 48, alt = "" }) {
  return <img src={src} width={size} height={size} alt={alt} />;
}

function Price({ cents }) {
  const dollars = (cents / 100).toFixed(2);   // compute from a prop
  return <span>${dollars}</span>;
}

If <Avatar src="..." /> is rendered without a size, it defaults to 48. Computing derived values (like dollars from cents) right in the component body is normal and encouraged — keep the source data in props and derive the rest.

3.6 Thinking in components

A real skill is looking at a design and seeing the component tree. The heuristic: a component should do one thing. If it grows too big or handles unrelated concerns, split it. A typical page decomposes like this:

App
 └─ Page
     ├─ Header
     │   ├─ Logo
     │   └─ Nav
     ├─ ProductList
     │   └─ ProductCard   (rendered once per product, via map)
     └─ Footer

Data enters at the top and flows down as props; each component renders its slice and passes relevant pieces to its children. ProductList receives an array of products and maps over it, rendering a ProductCard for each and passing that product's data as props. This tree-of-components-with-props-flowing-down is the shape of every React app.

3.7 A real example — composing with props and children

function Badge({ color, children }) {
  return <span className={`badge badge-${color}`}>{children}</span>;
}

function ProductCard({ name, price, status }) {
  return (
    <div className="card">
      <h2>{name}</h2>
      <p className="price">${price}</p>
      {status === "new" && <Badge color="green">New</Badge>}
      {status === "sale" && <Badge color="red">On Sale</Badge>}
    </div>
  );
}

function App() {
  const products = [
    { id: 1, name: "Keyboard", price: 129, status: "new" },
    { id: 2, name: "Mouse", price: 59, status: "sale" },
    { id: 3, name: "Monitor", price: 340, status: "normal" },
  ];

  return (
    <div>
      {products.map((p) => (
        <ProductCard key={p.id} name={p.name} price={p.price} status={p.status} />
      ))}
    </div>
  );
}

Trace the data: App holds the products, maps over them, and passes each product's fields to ProductCard as props; ProductCard conditionally renders a reusable Badge, passing text as children. Small, focused components, configured by props — this is idiomatic React.

🏋️ Exercise 3.2

Explain "one-way data flow" and why props are read-only. If a child needs to change data that lives in the parent, how does that happen (conceptually)?

Solution

One-way data flow means data moves down the component tree: parents pass data to children via props, and children never modify those props (they're read-only inputs). This makes apps predictable — you can always trace a value back up to where it originated.

Instead, the parent passes down a function as a prop, and the child calls that function (typically in response to an event). The parent's function updates the parent's state, and the new data flows back down as props. Data down, events up.

SECTION 4

Class Components: the Original React

Everything so far has used function components — the modern default. But for React's first several years, components with logic were written as classes, and you will absolutely encounter them: in older codebases, in libraries, in tutorials, and in interviews. This section teaches you to read and write class components fluently. You'll use hooks day to day, but understanding classes is what makes the reason for hooks clear later.

Then vs now: From 2013 to early 2019, any component that needed state or lifecycle logic had to be a class. Hooks (React 16.8, Feb 2019) let function components do everything classes could, and the community shifted decisively to functions. Classes are not deprecated and still work perfectly — but new code is written with functions. Learn classes to read the enormous amount of existing React; write functions for new code.

4.1 A function component and its class equivalent

Here is the same simple component, both ways. First, the function version you already know:

function Welcome({ name }) {
  return <h1>Hello, {name}!</h1>;
}

Now the class version:

import { Component } from "react";

class Welcome extends Component {
  render() {
    return <h1>Hello, {this.props.name}!</h1>;
  }
}

Study the differences, because they're the whole pattern:

  • A class component extends React.Component.
  • It defines a render() method that returns the JSX — this is the class's equivalent of the function body's return.
  • Props are accessed through this.props, not a parameter. Because a class instance holds its data on this, the props live at this.props.

Both are used identically: <Welcome name="Arthur" />. From the outside, a caller can't tell whether a component is a function or a class — they compose the same way.

4.2 Why this, and why it's the source of friction

The this keyword is the defining feature — and the defining headache — of class components. In a class, this refers to the component instance, giving you this.props, this.state (Section 5), and your methods. That sounds fine, but JavaScript's this is notoriously slippery: what this refers to depends on how a function is called, not where it's defined. When you pass a class method to an event handler, it can lose its binding to the instance and this becomes undefined. Managing this correctly is a recurring chore in class components — and eliminating it is one of the quiet reasons hooks feel so much lighter. You'll meet this problem head-on in Section 6.

4.3 The anatomy of a class component

A fuller class component, annotated:

import { Component } from "react";

class ProfileCard extends Component {
  // (constructor and state come in Section 5; lifecycle in Section 7)

  // A regular method — a helper the component can use.
  formatName() {
    return this.props.name.toUpperCase();
  }

  // render() is REQUIRED — it returns the component's JSX.
  render() {
    const { role, email } = this.props;   // destructure from this.props
    return (
      <div className="card">
        <h2>{this.formatName()}</h2>
        <p>{role}</p>
        <p>{email}</p>
      </div>
    );
  }
}

Notice render() calls this.formatName() — methods reference each other through this. And you can still destructure this.props at the top of render() for readability, just like function components destructure their parameter. The mental translation is direct: the function body becomes the class's render() method, and the props parameter becomes this.props.

SECTION 5

State: `useState` and `this.state`

Props are data passed in from outside. State is data a component owns and changes over time — a counter's count, whether a menu is open, what's typed in a box. When state changes, React re-renders the component to reflect it. This is where UIs come alive. You'll learn the modern hook first, then its classic counterpart, so the pair is clear.

5.1 useState — the modern way

State in a function component comes from the useState hook. You call it with the initial value; it returns a pair: the current value, and a function to update it.

import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);   // initial value 0

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

Read it piece by piece:

  • useState(0) declares a state variable starting at 0.
  • It returns [count, setCount] — you destructure it into the value (count) and its setter (setCount). The names are yours; the set-prefix convention is universal.
  • Clicking calls setCount(count + 1), which updates the state and tells React to re-render. On the next render, count holds the new value.

That last point is the heart of it: you never reassign count directly (count = count + 1 does nothing useful). You call the setter, and React re-runs the component with the updated value. State changes cause re-renders; direct mutation does not.

5.2 State updates trigger re-renders

When you call a setter, React schedules a re-render of that component. It runs the function again from the top, useState hands back the new value, and the returned JSX reflects it. Your component is a function of its state and props; change either and React re-derives the UI.

function Toggle() {
  const [isOn, setIsOn] = useState(false);
  return (
    <button onClick={() => setIsOn(!isOn)}>
      {isOn ? "ON" : "OFF"}
    </button>
  );
}

Each click flips isOn and re-renders, so the label updates. You describe the UI for each state; React handles the transition.

5.3 State is a snapshot; use functional updates

A subtle but important rule: within a single render, the state value is fixed — a snapshot. This bites when you update based on the previous value multiple times, or inside asynchronous code:

// ⚠️ Buggy: all three read the SAME count from this render
setCount(count + 1);
setCount(count + 1);
setCount(count + 1);   // net effect: +1, not +3

Because count is the same value throughout this render, all three calls compute the same result. The fix is the functional update form — pass a function that receives the latest state:

// ✅ Correct: each receives the up-to-date value
setCount((prev) => prev + 1);
setCount((prev) => prev + 1);
setCount((prev) => prev + 1);   // +3

Rule of thumb: when your new state depends on the old state, use the function form setX(prev => ...). It's always safe, and it's essential in async callbacks where the snapshot would otherwise be stale.

5.4 Multiple state variables and object state

You can call useState as many times as you need — one per independent piece of state, which keeps things clear:

function Form() {
  const [name, setName] = useState("");
  const [age, setAge] = useState(0);
  const [agreed, setAgreed] = useState(false);
  // ...
}

When state is an object or array, remember React state must be treated as immutable — never mutate it in place; create a new object/array. Spread the old one and override what changed:

const [user, setUser] = useState({ name: "", email: "" });

// ✅ Create a new object with the change:
setUser((prev) => ({ ...prev, email: "new@example.com" }));

// ⚠️ Never mutate then set — React may not see the change:
// user.email = "new@example.com"; setUser(user);

The reason: React decides whether to re-render partly by comparing references. Mutating the same object keeps the same reference, so React can miss the update. Always produce a new object/array. (For complex state, useReducer in Section 8 organizes this nicely.)

5.5 this.state and setState — the classic way

Then vs now: Before hooks, state lived on the class instance as this.state, was initialized in the constructor, and was updated with this.setState. This is what you'll see in every class component that holds state.

The same counter, as a class component:

import { Component } from "react";

class Counter extends Component {
  constructor(props) {
    super(props);
    this.state = { count: 0 };        // initialize state
  }

  render() {
    return (
      <div>
        <p>Count: {this.state.count}</p>
        <button onClick={() => this.setState({ count: this.state.count + 1 })}>
          Increment
        </button>
      </div>
    );
  }
}

The key differences from useState:

  • State is a single object (this.state = { count: 0, ... }) holding all of the component's state, initialized in the constructor. (super(props) must be called first — it wires up the base Component.)
  • You read state as this.state.count.
  • You update with this.setState({ ... }), which merges the object you pass into the existing state (it's a shallow merge — you only pass the keys that change) and triggers a re-render.

The functional-update concern exists here too, and for the same reason — this.setState can batch, so when the new value depends on the old you use the function form:

this.setState((prevState) => ({ count: prevState.count + 1 }));

SECTION 6

Events & Forms

Interfaces react to clicks, typing, and submits. This section covers handling events and building forms — and it's where the class-component this problem finally shows its teeth, so the modern approach's simplicity really lands.

6.1 Handling events

You attach event handlers with camelCase props like onClick, onChange, onSubmit, passing a function (not a string, as in HTML):

function Button() {
  function handleClick() {
    alert("Clicked!");
  }
  return <button onClick={handleClick}>Click me</button>;
}

Pass the function by reference — onClick={handleClick} — don't call it with onClick={handleClick()}, which would run it immediately during render. When you need to pass arguments, wrap it in an arrow function so it runs on the event, not on render:

<button onClick={() => deleteItem(item.id)}>Delete</button>

The handler receives a synthetic event — React's cross-browser wrapper around the native DOM event, with the same familiar interface (e.target, e.preventDefault(), etc.):

function handleChange(e) {
  console.log(e.target.value);   // the input's current value
}

6.2 Controlled components — forms the React way

In plain HTML, the DOM holds an input's value. In React, you typically let state hold it, and bind the input to that state. This is a controlled component: the input's value comes from state, and its onChange updates state. State becomes the single source of truth.

import { useState } from "react";

function SearchBox() {
  const [query, setQuery] = useState("");

  return (
    <input
      value={query}                                   // value driven by state
      onChange={(e) => setQuery(e.target.value)}      // every keystroke updates state
      placeholder="Search..."
    />
  );
}

The loop: you type → onChange fires → setQuery updates state → component re-renders → the input shows the new value. Because state always mirrors the input, you can validate, transform, or react to the value anywhere. Every serious React form is built this way.

6.3 Forms with multiple fields

For a form, keep each field in state and handle submission with onSubmit, calling e.preventDefault() to stop the browser's default page reload:

import { useState } from "react";

function SignupForm() {
  const [email, setEmail] = useState("");
  const [password, setPassword] = useState("");

  function handleSubmit(e) {
    e.preventDefault();                 // stop the full-page reload
    console.log("Submitting:", { email, password });
    // ...send to a server, etc.
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="email"
        value={email}
        onChange={(e) => setEmail(e.target.value)}
      />
      <input
        type="password"
        value={password}
        onChange={(e) => setPassword(e.target.value)}
      />
      <button type="submit">Sign up</button>
    </form>
  );
}

onSubmit on the <form> fires when the user submits (via the button or Enter). e.preventDefault() is essential — without it, the browser reloads the page and you lose everything. With state holding every field, the whole form's data is right there when you submit.

6.4 The this-binding problem in class components

Then vs now: In class components, event handlers are methods, and passing a method to onClick detaches it from this — so inside the handler, this becomes undefined and this.setState throws. Solving this was a constant tax in the class era. Function components have no this, so the entire problem simply doesn't exist.

Watch the bug appear in a class component:

import { Component } from "react";

class Toggle extends Component {
  constructor(props) {
    super(props);
    this.state = { on: false };
  }

  handleClick() {
    // ⚠️ `this` is undefined here when called as an event handler!
    this.setState({ on: !this.state.on });
  }

  render() {
    return <button onClick={this.handleClick}>Toggle</button>;
    //                     ^^^^^^^^^^^^^^^^ detached from the instance
  }
}

Clicking throws "Cannot read properties of undefined." The method lost its connection to the instance. The class era had three fixes:

1. Bind in the constructor (the traditional fix):

constructor(props) {
  super(props);
  this.state = { on: false };
  this.handleClick = this.handleClick.bind(this);   // permanently bind
}

2. Use a class field arrow function (the cleaner, later fix) — arrow functions capture this from where they're defined:

handleClick = () => {
  this.setState((prev) => ({ on: !prev.on }));
};

3. Wrap in an arrow at the call site (works, but creates a new function each render):

<button onClick={() => this.handleClick()}>Toggle</button>

That any of this is necessary — that a beginner's first stateful button can crash over a this binding — is exactly the kind of friction hooks removed. Compare the whole thing in a function component:

function Toggle() {
  const [on, setOn] = useState(false);
  return <button onClick={() => setOn((prev) => !prev)}>Toggle</button>;
}

No this, no binding, no constructor — it just works. When people say hooks made React simpler, this is a large part of what they mean.

6.5 A real example — a controlled to-do input

import { useState } from "react";

function TodoInput({ onAdd }) {
  const [text, setText] = useState("");

  function handleSubmit(e) {
    e.preventDefault();
    const trimmed = text.trim();
    if (!trimmed) return;          // ignore empty submissions
    onAdd(trimmed);                // send the value UP to the parent (events up!)
    setText("");                   // clear the input by resetting state
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={text}
        onChange={(e) => setText(e.target.value)}
        placeholder="Add a task..."
      />
      <button type="submit">Add</button>
    </form>
  );
}

Everything connects here: controlled input (value/onChange), form submission with preventDefault, validation (ignore empty), clearing by setting state, and — importantly — calling onAdd (a function prop from the parent) to send the new task up. That's the "data down, events up" pattern from Section 3 made concrete: the parent owns the list, this child reports new items via a callback prop.

SECTION 7

The Lifecycle and Effects

Components don't just render once — they mount (appear), update (re-render when props or state change), and unmount (get removed). Often you need to run code at these moments: fetch data when a component appears, subscribe to something, clean up a timer before it's removed. In the class era these hooks into the component's life were lifecycle methods. In modern React, one hook — useEffect — covers all of it. Understanding the lifecycle is the clearest possible route to understanding useEffect, so we teach them together.

7.1 The class lifecycle methods

Then vs now: Class components expose named methods that React calls at specific moments. Three matter most, and you'll see them constantly in older code. useEffect (modern) replaces all three with a single, more flexible tool.

import { Component } from "react";

class Clock extends Component {
  constructor(props) {
    super(props);
    this.state = { time: new Date() };
  }

  componentDidMount() {
    // Runs ONCE, right after the component first appears.
    // The classic place to start timers, subscriptions, or fetch data.
    this.timerId = setInterval(() => {
      this.setState({ time: new Date() });
    }, 1000);
  }

  componentDidUpdate(prevProps, prevState) {
    // Runs after every re-render (except the first).
    // You compare prevProps/prevState to react to specific changes.
  }

  componentWillUnmount() {
    // Runs right before the component is removed.
    // The place to CLEAN UP — clear timers, cancel subscriptions.
    clearInterval(this.timerId);
  }

  render() {
    return <p>{this.state.time.toLocaleTimeString()}</p>;
  }
}

The three you must recognize:

  • componentDidMount() — runs once after the first render. Start things here: fetch data, set up timers, add subscriptions.
  • componentDidUpdate(prevProps, prevState) — runs after every subsequent render. You compare previous and current values to decide what to do (e.g., refetch when an id prop changed).
  • componentWillUnmount() — runs just before removal. Clean up here, or you leak timers and subscriptions.

The pattern is: set up in componentDidMount, respond to changes in componentDidUpdate, tear down in componentWillUnmount. Notice the cleanup logic (start timer / clear timer) is split across two methods far apart in the file — a real source of bugs, because it's easy to set something up and forget to clean it up. Hold that thought.

7.2 useEffect — the modern replacement

useEffect lets a function component run side effects — anything that reaches outside the pure "compute UI from props/state" job: fetching, timers, subscriptions, manual DOM tweaks, logging. It takes a function (the effect) and a dependency array that controls when the effect runs.

The same clock, modernized:

import { useState, useEffect } from "react";

function Clock() {
  const [time, setTime] = useState(new Date());

  useEffect(() => {
    const timerId = setInterval(() => setTime(new Date()), 1000);

    // The RETURNED function is the cleanup — runs before unmount
    // (and before the effect re-runs). Setup and teardown, together.
    return () => clearInterval(timerId);
  }, []);   // empty deps = run once, on mount

  return <p>{time.toLocaleTimeString()}</p>;
}

Two things make this better than the class version:

  1. Setup and cleanup live together. The effect starts the timer and returns a function that clears it. React runs that returned "cleanup" function when the component unmounts (and before re-running the effect). No more logic split across distant methods — the thing you set up and the thing that tears it down sit side by side. This structure makes it much harder to forget cleanup.
  2. One tool, configured by dependencies, instead of three separate methods.

7.3 The dependency array — the whole game

The second argument to useEffect controls when it runs. This is the concept to nail:

useEffect(() => { /* ... */ });          // no array: runs after EVERY render
useEffect(() => { /* ... */ }, []);      // empty array: runs ONCE (on mount)
useEffect(() => { /* ... */ }, [id]);    // runs on mount + whenever `id` changes
  • No dependency array → the effect runs after every render. Rarely what you want.
  • Empty array [] → runs once after the first render, like componentDidMount. The cleanup runs on unmount, like componentWillUnmount.
  • [a, b] → runs on mount and again whenever any listed value changes. This replaces the "compare prevProps in componentDidUpdate" dance — you just list what the effect depends on, and React re-runs it when those change.

Mapping the class lifecycle onto useEffect:

Class lifecycle useEffect equivalent
componentDidMount useEffect(() => {...}, [])
componentDidUpdate (on specific change) useEffect(() => {...}, [thatValue])
componentWillUnmount the function you return from the effect
all three, for one concern a single useEffect that sets up and returns cleanup

That last row is the deeper win: in classes, one feature (say, a subscription) was scattered across three methods, while one method (say, componentDidMount) mixed several unrelated features. useEffect flips this — each effect groups all the code for one concern (setup + cleanup + when-to-run) in one place. You can have several useEffect calls, one per concern, instead of cramming everything into shared lifecycle methods.

7.4 The dependency rule and common pitfalls

The golden rule: every value from your component that the effect uses must be in the dependency array. React's linter enforces this, and following it prevents the classic bugs.

function Profile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetchUser(userId).then(setUser);
  }, [userId]);   // ✅ userId is used inside, so it's listed
  // ...
}

When userId changes, the effect re-runs and refetches — exactly the behavior you'd have hand-written with componentDidUpdate by comparing prevProps.userId, but automatic.

Two pitfalls to know:

  • The infinite loop. If an effect updates state that's also in its dependency array (or has no array), it re-renders, which re-runs the effect, which updates state again, forever. Fix by narrowing dependencies or using the functional update form of the setter.
  • Missing dependencies. Leaving a used value out of the array leads to stale values inside the effect — it "remembers" an old render's data. Trust the linter; add what it flags.

7.5 Effects for data fetching (the common case)

Fetching when a component mounts is the single most common effect. The shape, with proper loading and error handling:

import { useState, useEffect } from "react";

function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    let cancelled = false;                 // guard against setting state after unmount
    setLoading(true);

    fetch(`/api/users/${userId}`)
      .then((res) => res.json())
      .then((data) => {
        if (!cancelled) { setUser(data); setLoading(false); }
      })
      .catch((err) => {
        if (!cancelled) { setError(err); setLoading(false); }
      });

    return () => { cancelled = true; };    // cleanup: ignore a late response
  }, [userId]);                            // refetch when userId changes

  if (loading) return <p>Loading…</p>;
  if (error) return <p>Something went wrong.</p>;
  return <h1>{user.name}</h1>;
}

The cancelled flag in the cleanup is a real-world detail: if the component unmounts (or userId changes) before the fetch resolves, the cleanup flips the flag so you don't call setState on a gone component. Section 12 revisits data fetching in depth; for now, notice this is componentDidMount + componentDidUpdate + componentWillUnmount all handled by one useEffect.

7.6 A real example — a window-size tracker

A subscription effect, showing setup and cleanup together:

import { useState, useEffect } from "react";

function WindowWidth() {
  const [width, setWidth] = useState(window.innerWidth);

  useEffect(() => {
    function handleResize() {
      setWidth(window.innerWidth);
    }
    window.addEventListener("resize", handleResize);   // setup

    return () => window.removeEventListener("resize", handleResize);  // cleanup
  }, []);   // set up once

  return <p>Window width: {width}px</p>;
}

In a class this would be addEventListener in componentDidMount and removeEventListener in componentWillUnmount — the same listener referenced in two far-apart methods, easy to mismatch. Here they're four lines apart, obviously paired. That co-location is the ergonomic heart of useEffect.

Each effect also groups all the code for one concern, rather than mixing unrelated features into shared lifecycle methods.

SECTION 8

Hooks in Depth

useState and useEffect are the two hooks you'll use most, but the family is larger, and together they let function components do everything — and more cleanly than — classes ever could. This section covers the rules of hooks and the rest of the built-in set, ending with custom hooks, the feature that quietly replaced most of the old code-reuse patterns (Section 9 makes that replacement explicit).

8.1 The Rules of Hooks

Hooks come with two rules. They're not style suggestions — breaking them causes real bugs, and React's linter enforces them:

  1. Only call hooks at the top level of your component or custom hook — never inside loops, conditions, or nested functions. React tracks hooks by call order, so the order must be identical on every render. Putting a useState inside an if can shift the order and corrupt state.
  2. Only call hooks from React functions — function components or other custom hooks. Not from regular JavaScript functions, class methods, or event handlers.

A quick tell: if you're about to write if (x) { const [a] = useState() }, stop — hoist the hook to the top and put the condition inside it or around the value instead.

8.2 useRef — a value that persists without re-rendering

useRef gives you a mutable container whose .current property persists across renders but, crucially, does not trigger a re-render when it changes. Two main uses.

Referencing a DOM element:

import { useRef } from "react";

function TextInput() {
  const inputRef = useRef(null);

  function focusInput() {
    inputRef.current.focus();     // reach the actual DOM node
  }

  return (
    <>
      <input ref={inputRef} />
      <button onClick={focusInput}>Focus</button>
    </>
  );
}

Attaching ref={inputRef} points inputRef.current at the real DOM node, so you can imperatively focus, scroll, or measure it — the escape hatch for the rare times you need the actual element.

Holding a mutable value (like a timer id, or a previous value) that shouldn't cause re-renders:

const timerRef = useRef(null);
timerRef.current = setInterval(...);   // store it; changing it won't re-render

The distinction from state: change state → re-render; change a ref → no re-render. Use state for data that affects the UI; use a ref for behind-the-scenes values.

8.3 useMemo and useCallback — remembering across renders

Because a component re-runs top to bottom on every render, expensive computations and freshly-created functions/objects are rebuilt each time. Two hooks let you memoize — reuse a previous result unless dependencies change.

useMemo caches a computed value:

const sortedItems = useMemo(() => {
  return [...items].sort((a, b) => a.price - b.price);   // expensive
}, [items]);   // recompute only when items changes

useCallback caches a function so it keeps the same identity between renders:

const handleClick = useCallback(() => {
  doSomething(id);
}, [id]);   // same function unless id changes

Why useCallback matters: passing a fresh function to a memoized child (Section 13) on every render would defeat the memoization, since the function looks "new" each time. useCallback keeps it stable. Don't reach for these prematurely — they add complexity, and most components are fast without them. Use them when you have a genuinely expensive computation or a stability requirement (Section 13 shows when).

8.4 useReducer — structured state

When state logic gets complex — multiple sub-values, many kinds of updates — a bunch of useState calls gets unwieldy. useReducer centralizes updates into a single reducer function (the same pattern as Redux, and the same reduce idea from JavaScript): given the current state and an action, return the next state.

import { useReducer } from "react";

function reducer(state, action) {
  switch (action.type) {
    case "increment": return { count: state.count + 1 };
    case "decrement": return { count: state.count - 1 };
    case "reset":     return { count: 0 };
    default:          return state;
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, { count: 0 });

  return (
    <div>
      <p>{state.count}</p>
      <button onClick={() => dispatch({ type: "decrement" })}>−</button>
      <button onClick={() => dispatch({ type: "increment" })}>+</button>
      <button onClick={() => dispatch({ type: "reset" })}>Reset</button>
    </div>
  );
}

useReducer returns the current state and a dispatch function. You call dispatch({ type: "increment" }) to send an action; the reducer computes the new state. This shines when updates are numerous or interdependent: all the state transitions live in one readable, testable function instead of scattered across handlers. It's the natural upgrade from useState for complex state, and it pairs beautifully with Context (Section 10).

8.5 useContext — reading shared data

useContext reads a value from React Context — a way to share data across the tree without passing props through every level. It's important enough to get its own section (Section 10), but the hook itself is simple:

import { useContext } from "react";
import { ThemeContext } from "./ThemeContext";

function Toolbar() {
  const theme = useContext(ThemeContext);   // read the nearest provided value
  return <div className={theme}>...</div>;
}

Any component can grab the context value directly, no matter how deep, instead of receiving it prop-by-prop. Section 10 covers providing and consuming context in full.

8.6 Custom hooks — your own reusable logic

Here's the feature that changes everything. A custom hook is just a function whose name starts with use and that calls other hooks. It lets you extract stateful logic out of a component and reuse it — the modern, clean answer to code reuse that Section 9 will contrast with the old HOC and render-prop patterns.

Extract the window-width logic from Section 7 into a reusable hook:

import { useState, useEffect } from "react";

function useWindowWidth() {
  const [width, setWidth] = useState(window.innerWidth);

  useEffect(() => {
    const handleResize = () => setWidth(window.innerWidth);
    window.addEventListener("resize", handleResize);
    return () => window.removeEventListener("resize", handleResize);
  }, []);

  return width;   // hand back whatever the consumer needs
}

Now any component uses it in one line:

function Header() {
  const width = useWindowWidth();
  return <p>{width < 768 ? "Mobile" : "Desktop"} layout</p>;
}

That's the whole idea: bundle related hook logic (state + effect) into a use-named function, return what consumers need, and call it like a built-in hook. Custom hooks compose — one can call another — and they keep components focused on rendering while logic lives in tidy, testable, shareable units. This single feature is why function components don't need HOCs or render props for most sharing tasks, as you'll see next.

Another common one, a useLocalStorage hook:

function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    const stored = localStorage.getItem(key);
    return stored !== null ? JSON.parse(stored) : initialValue;
  });

  useEffect(() => {
    localStorage.setItem(key, JSON.stringify(value));
  }, [key, value]);

  return [value, setValue];   // same shape as useState — familiar to callers
}

Used as const [name, setName] = useLocalStorage("name", "") — it behaves like useState but persists. Returning the same [value, setter] shape as a built-in hook makes custom hooks feel native.

  1. Only call hooks at the top level (not in loops, conditions, or nested functions). Violation: if (loggedIn) { const [x, setX] = useState(0); } — the conditional can change hook call order between renders and corrupt state.
  2. Only call hooks from React functions (components or custom hooks). Violation: calling useState inside a plain helper function, a class method, or an event handler.

React relies on hooks being called in the same order every render, which both rules protect.

SECTION 9

Reusing Logic: HOCs, Render Props, and Custom Hooks

Here is the section that best repays understanding React's history. Every non-trivial app has cross-cutting logic — behavior needed by many unrelated components: tracking the mouse, subscribing to data, checking authentication, reading window size. Components are easy to reuse; reusing the stateful logic inside them is the hard problem React spent years solving. Before hooks, the two answers were higher-order components and render props. Both work, both are still out there in real codebases, and both were largely superseded by custom hooks. We'll build the same feature three ways so the evolution is unmistakable.

The running example: tracking the mouse position and sharing it with any component that wants it.

9.1 The problem

Say two very different components need the current mouse coordinates — a tooltip that follows the cursor, and a debug panel that displays x/y. The logic (add a mousemove listener, store the position in state, clean up on unmount) is identical, but the components render completely different things. You don't want to copy-paste that logic into both. How do you write it once and share it? Every technique in this section is an answer to exactly this question.

9.2 Higher-Order Components (the classic answer #1)

Then vs now: A higher-order component (HOC) is a function that takes a component and returns a new component with extra props or behavior "wrapped around" it. The convention is a with-prefixed name (withMouse, withAuth, withRouter). HOCs were everywhere from ~2015–2019; libraries like Redux (connect) and React Router (withRouter) popularized them. You'll still meet them constantly.

An HOC is to components what a function that returns a function is to functions. It takes WrappedComponent and returns a new component that computes something and passes it down as a prop:

import { Component } from "react";

// The HOC: a function that takes a component and returns an enhanced one.
function withMouse(WrappedComponent) {
  return class extends Component {
    constructor(props) {
      super(props);
      this.state = { x: 0, y: 0 };
      this.handleMove = this.handleMove.bind(this);
    }

    handleMove(e) {
      this.setState({ x: e.clientX, y: e.clientY });
    }

    componentDidMount() {
      window.addEventListener("mousemove", this.handleMove);
    }

    componentWillUnmount() {
      window.removeEventListener("mousemove", this.handleMove);
    }

    render() {
      // Pass the mouse position down as props, plus any original props.
      return <WrappedComponent {...this.props} mouse={this.state} />;
    }
  };
}

Now any component can be "enhanced" with a mouse prop, without knowing how it's tracked:

function MouseDisplay({ mouse }) {
  return <p>Mouse is at ({mouse.x}, {mouse.y})</p>;
}

// Wrap it — this is the HOC in action:
const MouseDisplayWithMouse = withMouse(MouseDisplay);

// Use the enhanced component:
function App() {
  return <MouseDisplayWithMouse />;
}

withMouse(MouseDisplay) produces a new component that owns the tracking logic and feeds MouseDisplay a mouse prop. Wrap a different component and it gets the same power. The logic is written once and shared — problem solved. But notice the machinery: a function returning a class, spreading {...this.props} so wrapping doesn't swallow the original props, and a new component identity in the tree.

The downsides that eventually bit:

  • Wrapper hell. Real components often need several HOCs: withRouter(withAuth(withTheme(connect(mapState)(MyComponent)))). The component tree fills with wrapper layers that exist only to inject props, and your React DevTools shows a tower of WithX(WithY(...)).
  • Prop collisions. Two HOCs that both inject a prop named data clobber each other silently, and you can't tell from the using component where a prop came from.
  • Indirection. To understand MouseDisplayWithMouse, you have to go find withMouse and read it. The mouse prop appears "from nowhere."

9.3 Render Props (the classic answer #2)

Then vs now: A render prop is a component that takes a function as a prop (often literally called render, or passed as children) and calls that function with the shared data, letting the consumer decide what to render. It rose partly as a reaction to HOC wrapper-hell (~2017–2019). React Router v5's <Route render={...}> and libraries like Downshift used it heavily.

Instead of wrapping a component, you write a component that manages the logic and hands the data to a function you provide:

import { Component } from "react";

class Mouse extends Component {
  constructor(props) {
    super(props);
    this.state = { x: 0, y: 0 };
    this.handleMove = this.handleMove.bind(this);
  }

  handleMove(e) {
    this.setState({ x: e.clientX, y: e.clientY });
  }

  componentDidMount() {
    window.addEventListener("mousemove", this.handleMove);
  }

  componentWillUnmount() {
    window.removeEventListener("mousemove", this.handleMove);
  }

  render() {
    // Call the function passed as `render`, giving it the data.
    return this.props.render(this.state);
  }
}

The consumer passes a function that receives the data and returns JSX:

function App() {
  return (
    <Mouse render={(mouse) => <p>Mouse is at ({mouse.x}, {mouse.y})</p>} />
  );
}

Mouse owns the tracking and calls this.props.render(this.state), delegating the actual UI to whatever function you pass. Same logic reused, different rendering each time. The children-as-a-function variant is common too:

<Mouse>
  {(mouse) => <p>({mouse.x}, {mouse.y})</p>}
</Mouse>
// where the component calls: this.props.children(this.state)

The downsides: render props avoid wrapper components but reintroduce nesting in your JSX — combine several and you get the "pyramid of doom," functions inside functions inside functions:

<Mouse render={(mouse) => (
  <Auth render={(user) => (
    <Theme render={(theme) => (
      <Panel mouse={mouse} user={user} theme={theme} />
    )} />
  )} />
)} />

It works, but it's hard to read, and the data threads awkwardly through the callback layers.

9.4 Custom Hooks (the modern answer)

Then vs now: Hooks (2019) made both patterns largely unnecessary for logic reuse. A custom hook extracts the stateful logic into a plain use-named function — no wrapper component, no render callback, no new tree layer. This is why HOCs and render props faded for this purpose.

The exact same mouse logic, as a hook (you learned to build these in Section 8):

import { useState, useEffect } from "react";

function useMouse() {
  const [position, setPosition] = useState({ x: 0, y: 0 });

  useEffect(() => {
    const handleMove = (e) => setPosition({ x: e.clientX, y: e.clientY });
    window.addEventListener("mousemove", handleMove);
    return () => window.removeEventListener("mousemove", handleMove);
  }, []);

  return position;
}

And any component uses it in a single, obvious line:

function MouseDisplay() {
  const mouse = useMouse();
  return <p>Mouse is at ({mouse.x}, {mouse.y})</p>;
}

Compare this to both classic approaches:

  • No wrapper component — MouseDisplay is a normal component; nothing wraps it, so no wrapper hell and no changed tree.
  • No injected mystery prop — mouse comes from a useMouse() call you can see in the component; its origin is explicit.
  • No render-callback nesting — need several pieces of shared logic? Just call several hooks: const mouse = useMouse(); const user = useAuth(); const theme = useTheme();. Flat, readable, composable. That pyramid of doom becomes three tidy lines.

This is the payoff of understanding the history: hooks aren't just "a new API," they're the solution to the specific pain HOCs and render props caused. Once you've felt wrapper hell and the render-prop pyramid, a custom hook feels like an obvious relief.

9.5 So are HOCs and render props dead?

Not dead — just rarely the right tool for logic reuse anymore. You should still recognize and be able to use them because:

  • Existing code is full of them; you'll maintain HOCs like connect() and withRouter() for years.
  • A few cases still suit them. HOCs remain reasonable for cross-cutting concerns applied uniformly to many components (error boundaries via a wrapper, some styling/theming injectors). Render props still appear in libraries where the consumer must control rendering deeply.
  • Interviews love asking you to explain the difference and why hooks replaced them.

The practical rule mirrors the class-vs-function one: write custom hooks for new logic reuse; read and maintain HOCs and render props where they already live.

🏋️ Exercise 9.1

Solution

A higher-order component is a function that takes a component and returns a new component enhanced with extra props or behavior (e.g., withMouse(Component)). 🏋️ Exercise 9.2

HOC downsides: (1) wrapper hell — stacking several HOCs buries your component under layers of wrapper components (withA(withB(withC(X)))), cluttering the tree; (2) prop collisions / indirection — two HOCs can inject a prop with the same name and clobber each other, and injected props appear "from nowhere," forcing you to hunt down the HOC to understand them.

Render-prop downside: deeply nested callbacks in JSX ("pyramid of doom") when combining several render-prop components.

Custom hooks avoid all of these: no wrapper components, no injected mystery props, and combining several is just several flat useX() calls.

SECTION 10

Context & State Management

As apps grow, two problems appear: passing data through many layers of components ("prop drilling"), and coordinating state that many distant components share. Context solves the first directly and, combined with useReducer, gives you a capable state-management pattern before you ever reach for an external library.

10.1 The prop-drilling problem

Props flow down one level at a time. When a value is needed deep in the tree, every component in between has to accept and forward it, even if it doesn't use it:

function App() {
  const user = { name: "Arthur" };
  return <Page user={user} />;
}
function Page({ user }) {      // doesn't use user — just forwards it
  return <Sidebar user={user} />;
}
function Sidebar({ user }) {   // doesn't use user — just forwards it
  return <Profile user={user} />;
}
function Profile({ user }) {   // finally uses it
  return <p>{user.name}</p>;
}

Page and Sidebar don't care about user; they're just conduits. This is prop drilling — tedious, noisy, and fragile (add a level and you thread the prop through one more component). For widely-needed data like the current user, theme, or language, it doesn't scale.

10.2 Context — the modern way

Context lets a component provide a value that any descendant can read directly, skipping the intermediate layers. Three steps: create the context, provide a value at some ancestor, and consume it wherever needed with useContext.

// 1. Create the context (usually in its own file)
import { createContext } from "react";
export const UserContext = createContext(null);

// 2. Provide a value high in the tree
import { UserContext } from "./UserContext";

function App() {
  const user = { name: "Arthur" };
  return (
    <UserContext.Provider value={user}>
      <Page />
    </UserContext.Provider>
  );
}

// 3. Consume it anywhere below, no prop drilling
import { useContext } from "react";
import { UserContext } from "./UserContext";

function Profile() {
  const user = useContext(UserContext);   // reads the nearest Provider's value
  return <p>{user.name}</p>;
}

Now Page and Sidebar don't mention user at all — Profile reaches up and grabs it. Any component inside the Provider can read the value with one useContext call. When the provided value changes, every consuming component re-renders with the new value.

10.3 The classic Consumer (then vs now)

Then vs now: Before useContext (and in class components, which can't call hooks), you consumed context with a render-prop-style Context.Consumer — the render-prop pattern from Section 9, applied to context. You'll see this in older code.

// Classic consumption — Consumer takes a function child (a render prop):
function Profile() {
  return (
    <UserContext.Consumer>
      {(user) => <p>{user.name}</p>}
    </UserContext.Consumer>
  );
}

Compare that to the modern const user = useContext(UserContext) — one line, no nesting. The Consumer still works and is the only option inside class components, but useContext is far cleaner and is what you'll write. (Class components also had a static contextType shortcut for a single context; recognize it, don't reach for it.)

10.4 Context + useReducer — a real state-management pattern

Context shares data; it doesn't manage it. For shared state that also changes, the powerful combination is Context to distribute + useReducer to update (Section 8). This gives you a centralized store with clean, testable transitions — Redux-like behavior with zero dependencies.

import { createContext, useContext, useReducer } from "react";

const TodosContext = createContext(null);

function todosReducer(state, action) {
  switch (action.type) {
    case "add":    return [...state, { id: action.id, text: action.text, done: false }];
    case "toggle": return state.map((t) => t.id === action.id ? { ...t, done: !t.done } : t);
    case "remove": return state.filter((t) => t.id !== action.id);
    default:       return state;
  }
}

// A provider component wraps the app and exposes state + dispatch via context.
export function TodosProvider({ children }) {
  const [todos, dispatch] = useReducer(todosReducer, []);
  return (
    <TodosContext.Provider value={{ todos, dispatch }}>
      {children}
    </TodosContext.Provider>
  );
}

// A custom hook makes consuming it ergonomic (and hides the context detail).
export function useTodos() {
  const ctx = useContext(TodosContext);
  if (!ctx) throw new Error("useTodos must be used inside <TodosProvider>");
  return ctx;
}

Any component now reads and updates the shared todos with one hook:

function AddTodo() {
  const { dispatch } = useTodos();
  return (
    <button onClick={() => dispatch({ type: "add", id: Date.now(), text: "New task" })}>
      Add
    </button>
  );
}

function TodoCount() {
  const { todos } = useTodos();
  return <p>{todos.length} tasks</p>;
}

Notice the pattern's elegance: useReducer holds the state and transitions, Context distributes them, and a custom hook (useTodos) wraps useContext so components never touch the raw context — they just call useTodos(). This trio (reducer + context + custom hook) is a complete, dependency-free state solution suitable for a great many apps. You'll use exactly this in the capstone.

10.5 When to reach for a state library

Context + reducer covers a lot, but it has limits: every consumer re-renders when the context value changes (so very large or very frequently-updated global state can cause performance issues), and there's no built-in tooling for things like time-travel debugging, middleware, or caching server data. When you outgrow the built-ins, the ecosystem offers:

  • Redux (with Redux Toolkit) — the long-standing choice for large, complex client state, with excellent devtools. (Redux Toolkit is the modern, far-less-boilerplate way to use it; the connect() HOC from Section 9 is the older API you'll still see.)
  • Zustand / Jotai — lighter, hook-first stores that avoid Redux's ceremony.
  • React Query / SWR — not general state managers but server-state libraries: they handle fetching, caching, and syncing remote data, which is what a lot of "state management" actually is. Reach for these for data-fetching (Section 12).

The honest guidance: start with useState, lift state up when needed, reach for Context + reducer when prop drilling hurts, and only add a library when you have a concrete problem it solves. Many production apps never need more than the built-ins plus a server-state library. Don't cargo-cult Redux into a small app.

SECTION 11

Classic Component Patterns

Beyond HOCs and render props, the React community developed a vocabulary of component patterns — recurring ways to structure components for flexibility and reuse. Some predate hooks and have modern equivalents; others are timeless. Knowing them by name helps you read code, communicate with other developers, and recognize the right structure for a problem. This section is a tour of the important ones, old and current.

11.1 Container and Presentational components

Then vs now: A long-popular convention split components into container (or "smart") components that handle data and logic, and presentational (or "dumb") components that just render props. It was especially common in the class + Redux era. Hooks blurred the split — logic now lives comfortably in any component or a custom hook — but the underlying idea (separate data-wrangling from rendering) is still valuable.

The pattern: a presentational component takes data purely as props and renders it, with no state or fetching of its own; a container fetches/computes the data and passes it down.

// Presentational: pure, reusable, easy to test — just props in, UI out.
function UserList({ users }) {
  return (
    <ul>
      {users.map((u) => <li key={u.id}>{u.name}</li>)}
    </ul>
  );
}

// Container: owns the data, renders the presentational component.
function UserListContainer() {
  const [users, setUsers] = useState([]);
  useEffect(() => {
    fetch("/api/users").then((r) => r.json()).then(setUsers);
  }, []);
  return <UserList users={users} />;
}

UserList knows nothing about where users come from — you can reuse it, test it with fake data, or show it in a component gallery. UserListContainer handles the messy part. The modern take: you often don't need a separate container component — a custom hook (const users = useUsers()) captures the "smart" logic and you use it right inside the presentational component. But you'll see the explicit split constantly, and separating concerns remains good instinct.

11.2 Controlled vs Uncontrolled components

You met controlled inputs in Section 6. The formal distinction:

  • Controlled — React state is the single source of truth; the input's value comes from state and onChange updates it. You control it. This is the default recommendation.
  • Uncontrolled — the DOM holds the value; you read it only when needed, via a ref (Section 8), rather than tracking every keystroke in state.
// Uncontrolled: no value/onChange; grab the value with a ref when needed.
function UncontrolledForm() {
  const inputRef = useRef(null);
  function handleSubmit(e) {
    e.preventDefault();
    console.log(inputRef.current.value);   // read straight from the DOM
  }
  return (
    <form onSubmit={handleSubmit}>
      <input ref={inputRef} defaultValue="" />   {/* defaultValue, not value */}
      <button>Submit</button>
    </form>
  );
}

Note defaultValue (sets the initial value without controlling it) instead of value. Uncontrolled inputs are occasionally handy — quick forms, integrating non-React widgets, file inputs (which are always uncontrolled) — but controlled is the default because state-as-source-of-truth makes validation and reactivity trivial.

11.3 Compound components

A compound component is a set of components designed to work together, sharing implicit state through context, so the consumer composes them expressively. You've used the pattern even if you didn't name it — <select> and <option> in HTML are compound. In React, think <Tabs> with <Tab> children:

import { createContext, useContext, useState } from "react";

const TabsContext = createContext(null);

function Tabs({ children, defaultTab }) {
  const [active, setActive] = useState(defaultTab);
  return (
    <TabsContext.Provider value={{ active, setActive }}>
      <div className="tabs">{children}</div>
    </TabsContext.Provider>
  );
}

function Tab({ id, children }) {
  const { active, setActive } = useContext(TabsContext);
  return (
    <button
      className={active === id ? "tab active" : "tab"}
      onClick={() => setActive(id)}
    >
      {children}
    </button>
  );
}

function TabPanel({ id, children }) {
  const { active } = useContext(TabsContext);
  return active === id ? <div className="panel">{children}</div> : null;
}

The consumer composes them naturally, and the shared "which tab is active" state is handled invisibly via context:

<Tabs defaultTab="a">
  <Tab id="a">First</Tab>
  <Tab id="b">Second</Tab>
  <TabPanel id="a">First panel content</TabPanel>
  <TabPanel id="b">Second panel content</TabPanel>
</Tabs>

Compound components give a clean, flexible API — the user arranges the pieces, and the components coordinate behind the scenes. This pattern is very much current and common in component libraries.

11.4 PureComponent and the shape of optimization

Then vs now: Class components had React.PureComponent — a Component variant that automatically skips re-rendering when its props and state haven't shallowly changed — and the manual shouldComponentUpdate(nextProps, nextState) method for custom control. The function-component equivalent is React.memo (Section 13). Same goal, different mechanism.

import { PureComponent } from "react";

// Re-renders only when props/state shallow-change (classic optimization).
class ExpensiveList extends PureComponent {
  render() {
    return <ul>{this.props.items.map((i) => <li key={i.id}>{i.name}</li>)}</ul>;
  }
}

shouldComponentUpdate was the manual version — you returned true/false to decide whether to re-render. Both were the class era's answer to "don't re-render when nothing relevant changed." You'll recognize them in old performance-tuned code; in modern code the same intent is expressed with React.memo, which Section 13 covers.

11.5 Patterns you should recognize but not reach for

A few legacy patterns exist mainly so you can identify them in very old code and know they're superseded:

  • Mixins — the original (pre-2015, React.createClass) way to share logic, via a mixins: [...] array. They caused name clashes and implicit dependencies, were deprecated when ES6 classes arrived, and were a big motivation for HOCs (then render props, then hooks). You'll almost never see them now, but the term comes up in React's history.
  • React.createClass — the way components were defined before ES6 class syntax. Fully obsolete; mentioned only because ancient tutorials use it.
  • propTypes and defaultProps — runtime prop type-checking and default values, attached as static properties (Component.propTypes = {...}). Still functional and common in JavaScript codebases, but TypeScript has largely taken over compile-time prop typing, and function-component defaults now use default parameters ({ size = 48 }). Recognize propTypes; prefer TypeScript or default params in new code.

The through-line of this whole section: React accumulated patterns as it searched for good ways to share and structure logic. Hooks resolved much of that search, but the vocabulary — container/presentational, controlled/uncontrolled, compound, pure/memo — remains how developers talk about component design. Knowing the words makes you fluent in code review and design discussions.

11.6 A real example — recognizing patterns in a component

Read this and name the patterns at play:

function SearchableList({ items }) {          // presentational-ish: data in via props
  const [query, setQuery] = useState("");      // controlled input state
  const filtered = useMemo(
    () => items.filter((i) => i.name.includes(query)),
    [items, query]
  );
  return (
    <>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />  {/* controlled */}
      <ul>{filtered.map((i) => <li key={i.id}>{i.name}</li>)}</ul>
    </>
  );
}

Patterns visible: it's largely presentational (receives items as props), uses a controlled input (value/onChange), and applies memoization for the derived filtered list. A container or a useItems() custom hook would supply items. Being able to narrate a component like this — "controlled input, presentational, memoized derived data" — is exactly the fluency this section builds.

SECTION 12

Data Fetching

Almost every real app talks to a server. You've seen the basic effect-based fetch already; this section makes data fetching robust — handling loading and error states, avoiding common bugs, and understanding why the ecosystem increasingly reaches for dedicated data libraries. As always, we contrast the modern approach with the classic class-component pattern.

12.1 The three states of any fetch

Every request has three possible states, and good UI accounts for all of them: loading (in flight), error (it failed), and success (you have data). Beginners often render only the success case and get blank screens or crashes when things are slow or broken. The disciplined shape tracks all three:

import { useState, useEffect } from "react";

function Repos({ username }) {
  const [repos, setRepos] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    setLoading(true);
    setError(null);

    fetch(`https://api.github.com/users/${username}/repos`)
      .then((res) => {
        if (!res.ok) throw new Error(`HTTP ${res.status}`);   // fetch doesn't throw on 404!
        return res.json();
      })
      .then((data) => setRepos(data))
      .catch((err) => setError(err.message))
      .finally(() => setLoading(false));
  }, [username]);

  if (loading) return <p>Loading…</p>;
  if (error) return <p className="error">Error: {error}</p>;
  return (
    <ul>
      {repos.map((r) => <li key={r.id}>{r.name}</li>)}
    </ul>
  );
}

Two details that catch people out: fetch does not reject on HTTP errors like 404 or 500 — it only rejects on network failure — so you must check res.ok and throw yourself. And render the loading and error branches before the success branch, so you never try to .map() over data you don't have yet.

12.2 The cleanup problem — stale responses

There's a subtle bug in naive fetching. If username changes quickly (or the component unmounts) before a request finishes, a slow earlier response can arrive after a newer one and overwrite it — or you set state on an unmounted component. Guard with a cleanup flag (you saw this in Section 7):

useEffect(() => {
  let active = true;
  setLoading(true);

  fetch(`https://api.github.com/users/${username}`)
    .then((res) => res.json())
    .then((data) => { if (active) setUser(data); })
    .catch((err) => { if (active) setError(err.message); })
    .finally(() => { if (active) setLoading(false); });

  return () => { active = false; };   // ignore this request's result if superseded
}, [username]);

The cleanup sets active = false, so a superseded request's .then becomes a no-op. Modern browsers also support AbortController to actually cancel the request, which data libraries use under the hood.

12.3 Extracting a useFetch hook

Repeating loading/error/data boilerplate in every component is tedious — a perfect job for a custom hook (Section 8). Extract it once:

import { useState, useEffect } from "react";

function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    let active = true;
    setLoading(true);
    setError(null);

    fetch(url)
      .then((res) => {
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        return res.json();
      })
      .then((d) => { if (active) setData(d); })
      .catch((e) => { if (active) setError(e.message); })
      .finally(() => { if (active) setLoading(false); });

    return () => { active = false; };
  }, [url]);

  return { data, loading, error };
}

Now any component fetches in three lines, with all the edge cases handled:

function Repos({ username }) {
  const { data: repos, loading, error } =
    useFetch(`https://api.github.com/users/${username}/repos`);

  if (loading) return <p>Loading…</p>;
  if (error) return <p>Error: {error}</p>;
  return <ul>{repos.map((r) => <li key={r.id}>{r.name}</li>)}</ul>;
}

This is the custom-hook payoff again: the gnarly logic lives once, and components stay focused on rendering.

12.4 The classic pattern — componentDidMount

Then vs now: In class components, the same fetch lived in componentDidMount (initial load) and often componentDidUpdate (refetch when a prop changed) — with the loading/error/data stored in this.state. This is the pattern you'll find throughout older codebases.

import { Component } from "react";

class Repos extends Component {
  state = { repos: [], loading: true, error: null };

  fetchRepos() {
    fetch(`https://api.github.com/users/${this.props.username}/repos`)
      .then((res) => res.json())
      .then((repos) => this.setState({ repos, loading: false }))
      .catch((error) => this.setState({ error: error.message, loading: false }));
  }

  componentDidMount() {
    this.fetchRepos();                       // initial load
  }

  componentDidUpdate(prevProps) {
    if (prevProps.username !== this.props.username) {
      this.setState({ loading: true });
      this.fetchRepos();                     // refetch when username changes
    }
  }

  render() {
    const { repos, loading, error } = this.state;
    if (loading) return <p>Loading…</p>;
    if (error) return <p>Error: {error}</p>;
    return <ul>{repos.map((r) => <li key={r.id}>{r.name}</li>)}</ul>;
  }
}

Compare the effort: the class version splits "fetch on load" and "refetch on change" across two lifecycle methods, and you manually compare prevProps.username. The useEffect version collapses both into one effect with [username] as its dependency. Same behavior; the modern form is tighter and keeps the whole fetch concern in one place — the recurring theme of this course.

12.5 When to use a data-fetching library

Hand-rolled fetching works, but production apps quickly want more: caching (don't refetch data you already have), deduplication (two components asking for the same thing → one request), background refetching, retries, pagination, and keeping the UI in sync as server data changes. Building all that yourself is a lot. Dedicated server-state libraries do it for you:

  • React Query (TanStack Query) — the popular choice; you write useQuery(['repos', username], fetchRepos) and get caching, loading/error states, retries, and refetching essentially for free.
  • SWR — a lighter alternative with the same philosophy (stale-while-revalidate).

The key mental model, increasingly emphasized in modern React: server data is not the same as UI state. Things like "is this menu open" are UI state (useState); things like "the list of repos from the API" are server state, and server-state libraries manage them far better than useState + useEffect can. For learning, build a useFetch by hand (as above) so you understand the mechanics; for real apps, reach for React Query or SWR once fetching gets non-trivial. (There's also a newer direction — React Server Components and framework data loaders, e.g. in Next.js — but that's beyond this course's scope; know the term.)

12.6 A real example — search with debounced-ish fetching

A search box that fetches results when you submit (keeping it simple — fetching on every keystroke would need debouncing):

import { useState } from "react";

function UserSearch() {
  const [username, setUsername] = useState("");
  const [submitted, setSubmitted] = useState("");
  const { data, loading, error } = useFetch(
    submitted ? `https://api.github.com/users/${submitted}` : null
  );

  function handleSubmit(e) {
    e.preventDefault();
    setSubmitted(username.trim());   // trigger the fetch by changing the URL dep
  }

  return (
    <div>
      <form onSubmit={handleSubmit}>
        <input value={username} onChange={(e) => setUsername(e.target.value)} />
        <button>Search</button>
      </form>
      {loading && <p>Loading…</p>}
      {error && <p>Error: {error}</p>}
      {data && <p>{data.name} — {data.public_repos} repos</p>}
    </div>
  );
}

(A production useFetch would skip fetching when url is null; the idea here is that changing submitted changes the URL, which is the effect's dependency, which triggers the fetch. This is the exact shape you'll build on in the capstone.)

🏋️ Exercise 12.1

Name the three states every data fetch can be in, and explain why rendering only one of them is a common beginner bug.

Solution

The three states are loading (request in flight), error (request failed), and success (data available). Beginners commonly render only the success state — immediately trying to display or .map() over the data. This breaks in two ways: while loading, the data isn't there yet (blank screen or a crash from mapping over undefined/empty), and on error there's no feedback to the user. Handling all three — showing a loading indicator, an error message, and finally the data — is what makes a fetch robust.

🏋️ Exercise 12.2

fetch resolved successfully but the server returned a 404. Why won't your .catch fire, and how do you handle HTTP error statuses correctly?

Solution

fetch only rejects on network failures (no connection, DNS failure, etc.) — it treats any HTTP response, including 404 or 500, as a successful promise resolution. So a 404 flows into your .then, not your .catch. To handle it, check the response's ok property (or status) and throw yourself:

.then((res) => {
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
})

Now a 404/500 throws, and your .catch (and error state) handle it as expected.

🏋️ Exercise 12.3

What bug does the let active = true; ... return () => { active = false; } pattern in a fetching useEffect prevent?

Solution

It prevents stale/racing responses and state-updates-after-unmount. If the effect's dependency changes (e.g., a new username) or the component unmounts before a request resolves, an old, slow response could arrive later and either overwrite newer data or call setState on a component that's gone (a warning and a leak). The cleanup sets active = false, so the superseded request's .then/.catch become no-ops — only the most recent, still-mounted effect updates state.

🏋️ Exercise 12.4

SECTION 13

Performance & Optimization

React is fast by default, and the first rule of optimization is don't optimize prematurely — most apps never need any of this. But when you do hit a slow list, a janky interaction, or a heavy computation, you need to understand why React re-renders and which tools control it. This section explains the model and the modern toolset, with a nod to the class-era ancestors.

13.1 When and why components re-render

A component re-renders when: its state changes, its props change, or its parent re-renders. That last one surprises people — when a parent re-renders, all of its children re-render by default, even children whose props didn't change. Usually that's fine (React's rendering is cheap and the DOM diff avoids unnecessary DOM work). It only becomes a problem when a child's render is genuinely expensive or a large subtree re-renders needlessly on a frequent update.

The key correction to a common misconception: a "re-render" means React runs your component function and diffs the result — it does not necessarily touch the real DOM. React only updates the actual DOM where the output changed. So re-renders are not automatically "bad"; they're only worth optimizing when profiling shows they cost something.

13.2 React.memo — skip re-renders when props are unchanged

Then vs now: React.memo is the function-component equivalent of the class era's PureComponent / shouldComponentUpdate (Section 11). It wraps a component so React skips re-rendering it when its props haven't shallowly changed.

import { memo } from "react";

const ExpensiveItem = memo(function ExpensiveItem({ item }) {
  // Only re-renders when `item` changes (shallow comparison of props).
  return <li>{/* ...expensive rendering of item... */}</li>;
});

Now when the parent re-renders but passes the same item, ExpensiveItem is skipped. memo does a shallow comparison of props — so it helps only when props are stable primitives or stable references. Which leads directly to why useMemo/useCallback exist.

13.3 useMemo and useCallback in their real role

You met these in Section 8. Their most important job is keeping references stable so React.memo actually works. Because a component recreates objects and functions on every render, passing a fresh one to a memoized child breaks the memoization — the prop looks new every time.

function Parent({ items }) {
  const [count, setCount] = useState(0);

  // Without useCallback, `onSelect` is a NEW function each render,
  // so a memo'd child sees a "changed" prop and re-renders anyway.
  const onSelect = useCallback((id) => {
    console.log("selected", id);
  }, []);   // stable identity

  // Without useMemo, this array/object is recreated each render.
  const config = useMemo(() => ({ theme: "dark" }), []);

  return (
    <>
      <button onClick={() => setCount(count + 1)}>Count: {count}</button>
      <ExpensiveChild items={items} onSelect={onSelect} config={config} />
    </>
  );
}

Now clicking the counter re-renders Parent, but onSelect and config keep the same identity, so a memoized ExpensiveChild correctly skips re-rendering. useMemo also stands alone for genuinely expensive computations (sorting/filtering large arrays) — cache the result and recompute only when inputs change.

The discipline: React.memo, useMemo, and useCallback are a team. Memoizing a child is pointless if you feed it fresh function/object props every render; stabilizing props is pointless if the child isn't memoized. Reach for the trio together, guided by profiling — not sprinkled everywhere by reflex.

13.4 Keys and lists, revisited for performance

Correct key props (Section 2) aren't just about avoiding warnings — they're a performance feature. Stable, unique keys let React match items across renders and move/update only what changed, instead of tearing down and rebuilding list DOM. Using the array index as a key in a list that reorders or has items inserted/removed forces React to mis-match items, causing both bugs and wasted DOM work. Good keys = correct and fast lists.

13.5 Code splitting with lazy and Suspense

A different kind of performance: initial load. By default your whole app ships as one JavaScript bundle; the bigger it is, the slower the first paint. Code splitting loads parts on demand. React.lazy plus Suspense lets you defer loading a component until it's actually rendered:

import { lazy, Suspense } from "react";

const Dashboard = lazy(() => import("./Dashboard"));   // separate bundle, loaded on demand

function App() {
  return (
    <Suspense fallback={<p>Loading…</p>}>
      <Dashboard />
    </Suspense>
  );
}

lazy(() => import(...)) tells the bundler to split Dashboard into its own chunk, fetched only when it first renders. Suspense shows the fallback while that chunk loads. This is especially valuable for routes (Section 14) — you load each page's code as the user navigates to it, keeping the initial bundle small. (Suspense is also the mechanism newer data-fetching approaches build on.)

13.6 A pragmatic optimization checklist

When something is actually slow, work in this order:

  1. Measure first. Use the React DevTools Profiler to find what's actually re-rendering and how long it takes. Optimize based on data, not guesses.
  2. Fix obvious waste. Bad keys, huge lists rendered all at once (consider virtualization libraries like react-window), expensive work done on every render that could be memoized.
  3. Memoize deliberately. Wrap genuinely expensive children in React.memo, and stabilize the props you pass them with useMemo/useCallback.
  4. Split the bundle. lazy + Suspense for routes and heavy components to speed up initial load.
  5. Reconsider structure. Sometimes lifting state down (moving it to the small component that uses it, so state changes don't re-render a big subtree) beats any memoization.

Above all: don't do any of this without a reason. Premature memoization adds complexity, can introduce subtle bugs (wrong dependency arrays), and often makes code slower to read for zero user benefit. Fast-by-default is a real feature; lean on it until you have a measured problem.

Why does passing an inline onClick={() => doThing()} to a React.memo-wrapped child defeat the memoization, and how do you fix it?

const handleThing = useCallback(() => doThing(), []);

SECTION 14

Routing and Structuring a Real App

A single-page React app still needs multiple "pages" — a home view, a detail view, a settings view — with real URLs you can bookmark, share, and navigate with the back button. Client-side routing provides this without full page reloads. This section covers React Router (the de facto standard) and how to structure a growing app, tying together everything you've learned before the capstone.

14.1 Client-side routing, and why

In a traditional website, clicking a link fetches a whole new HTML page from the server. In a single-page app, the page never reloads — JavaScript swaps the visible components and updates the URL. React Router synchronizes your UI with the URL: it renders different components for different paths, updates the address bar as you navigate, and makes the browser's back/forward buttons work. The result feels like a multi-page site but stays fast and stateful.

Install it into a Vite project:

npm install react-router-dom

14.2 Basic routes

You wrap the app in a router, then declare which component renders for each path:

import { BrowserRouter, Routes, Route } from "react-router-dom";

function App() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/about" element={<About />} />
        <Route path="/users/:username" element={<UserProfile />} />
        <Route path="*" element={<NotFound />} />
      </Routes>
    </BrowserRouter>
  );
}

Reading it: BrowserRouter enables routing for the whole app; Routes picks the one Route whose path matches the current URL and renders its element. The :username segment is a URL parameter (a placeholder), and path="*" is the catch-all for unmatched URLs — your 404 page.

14.3 Navigating — Link and useNavigate

Never use a plain <a href> for internal navigation — it triggers a full page reload, defeating the SPA. Use React Router's Link (which updates the URL without reloading):

import { Link } from "react-router-dom";

function Nav() {
  return (
    <nav>
      <Link to="/">Home</Link>
      <Link to="/about">About</Link>
      <Link to="/users/octocat">Octocat</Link>
    </nav>
  );
}

For navigation triggered by code (after a form submit, say), use the useNavigate hook:

import { useNavigate } from "react-router-dom";

function LoginForm() {
  const navigate = useNavigate();
  function handleSubmit(e) {
    e.preventDefault();
    // ...log in...
    navigate("/dashboard");        // programmatic redirect
  }
  // ...
}

14.4 Reading URL parameters — useParams

The :username in the route becomes available via the useParams hook, so a single route component serves many URLs:

import { useParams } from "react-router-dom";

function UserProfile() {
  const { username } = useParams();     // matches :username in the path
  const { data, loading, error } = useFetch(
    `https://api.github.com/users/${username}`
  );

  if (loading) return <p>Loading…</p>;
  if (error) return <p>Error: {error}</p>;
  return <h1>{data.name}</h1>;
}

Navigate to /users/octocat and username is "octocat"; /users/torvalds and it's "torvalds". One component, driven by the URL — this is how detail pages work everywhere. (Notice useFetch and the loading/error discipline from Section 12 slot right in.)

14.5 Structuring a growing app

There's no single mandated structure, but a widely-used, sane convention scales well. Group by type while the app is small, and consider grouping by feature as it grows:

src/
├── main.jsx              # entry point
├── App.jsx               # routes + layout
├── components/           # reusable, presentational UI (Button, Card, Nav)
│   ├── Button.jsx
│   └── Card.jsx
├── pages/                # one component per route (Home, UserProfile, NotFound)
│   ├── Home.jsx
│   └── UserProfile.jsx
├── hooks/                # custom hooks (useFetch, useLocalStorage)
│   └── useFetch.js
├── context/              # context providers (UserContext, TodosProvider)
│   └── TodosContext.jsx
└── lib/ or utils/        # non-React helpers (formatting, API clients)
    └── api.js

The principles that matter more than the exact folders:

  • Components do one job; split when they grow.
  • Reusable UI goes in components/; route-level components in pages/.
  • Logic lives in hooks (hooks/), not tangled into components.
  • Shared state goes in context/ with a provider and a useX accessor hook (Section 10).
  • Non-React code (API calls, formatting) goes in plain modules — keep it out of components so it's testable and reusable.

This is the structure the capstone uses, and it's representative of real React projects. As apps get large, teams often switch to feature folders (all files for "users" — its page, components, hooks — under features/users/), but the type-based layout above is perfect to start.

15.2 The custom hooks — useFetch and useLocalStorage

Start with the reusable logic (Sections 8 and 12). Create src/hooks/useFetch.js:

import { useState, useEffect } from "react";

export function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState(null);

  useEffect(() => {
    if (!url) return;                     // no url → don't fetch
    let active = true;
    setLoading(true);
    setError(null);

    fetch(url)
      .then((res) => {
        if (!res.ok) throw new Error(res.status === 404 ? "Not found" : `HTTP ${res.status}`);
        return res.json();
      })
      .then((d) => { if (active) setData(d); })
      .catch((e) => { if (active) setError(e.message); })
      .finally(() => { if (active) setLoading(false); });

    return () => { active = false; };     // ignore stale responses
  }, [url]);

  return { data, loading, error };
}

And src/hooks/useLocalStorage.js (from Section 8), so favorites survive a refresh:

import { useState, useEffect } from "react";

export function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    try {
      const stored = localStorage.getItem(key);
      return stored !== null ? JSON.parse(stored) : initialValue;
    } catch {
      return initialValue;
    }
  });

  useEffect(() => {
    localStorage.setItem(key, JSON.stringify(value));
  }, [key, value]);

  return [value, setValue];
}

15.3 The favorites store — Context + reducer + persistence

This is the Section 10 pattern applied for real. Create src/context/FavoritesContext.jsx:

import { createContext, useContext, useReducer, useEffect } from "react";
import { useLocalStorage } from "../hooks/useLocalStorage";

const FavoritesContext = createContext(null);

function favoritesReducer(state, action) {
  switch (action.type) {
    case "add":
      if (state.some((u) => u.id === action.user.id)) return state;   // no duplicates
      return [...state, action.user];
    case "remove":
      return state.filter((u) => u.id !== action.user.id);
    case "clear":
      return [];
    default:
      return state;
  }
}

export function FavoritesProvider({ children }) {
  // Seed the reducer from localStorage, and persist on every change.
  const [stored, setStored] = useLocalStorage("devfinder:favorites", []);
  const [favorites, dispatch] = useReducer(favoritesReducer, stored);

  useEffect(() => { setStored(favorites); }, [favorites, setStored]);

  return (
    <FavoritesContext.Provider value={{ favorites, dispatch }}>
      {children}
    </FavoritesContext.Provider>
  );
}

// Ergonomic accessor hook — components never touch the raw context.
export function useFavorites() {
  const ctx = useContext(FavoritesContext);
  if (!ctx) throw new Error("useFavorites must be used within <FavoritesProvider>");
  return ctx;
}

Notice the whole toolkit converging: a reducer owns the favorites transitions (add/remove/clear, de-duplicated), Context distributes them, a custom hook wraps useLocalStorage to persist, and useFavorites gives components clean access. This is exactly the dependency-free state pattern from Section 10.

Search page (src/pages/Search.jsx) — a controlled form that triggers a fetch by setting the searched username:

import { useState } from "react";

export function Search() {

src/App.jsx composes the router, the provider, and a shared layout:

import { BrowserRouter, Routes, Route, Link } from "react-router-dom";
import { FavoritesProvider } from "./context/FavoritesContext";
import { Search } from "./pages/Search";
import { UserProfile } from "./pages/UserProfile";
import { Favorites } from "./pages/Favorites";

export default function App() {
  return (
    <FavoritesProvider>
      <BrowserRouter>
        <nav>
          <Link to="/">Search</Link> | <Link to="/favorites">Favorites</Link>
        </nav>
        <main>
          <Routes>
            <Route path="/" element={<Search />} />

15.7 The bonus — one feature, the classic way

To seal the then-vs-now lesson, reimplement the search's data-fetching as a class component using an HOC (Sections 4, 7, 9, 12) — the way this would have been written in 2017. You don't ship this; you write it once to feel the contrast:

import { Component } from "react";

// An HOC that injects GitHub-user-fetching props (the classic reuse pattern).
function withGitHubUser(WrappedComponent) {
  return class extends Component {
    state = { user: null, loading: false, error: null };

    componentDidUpdate(prevProps) {
      if (prevProps.username !== this.props.username && this.props.username) {
        this.fetchUser();
      }
    }

    fetchUser() {
      this.setState({ loading: true, error: null });
      fetch(`https://api.github.com/users/${this.props.username}`)
        .then((res) => { if (!res.ok) throw new Error("Not found"); return res.json(); })
        .then((user) => this.setState({ user, loading: false }))
        .catch((error) => this.setState({ error: error.message, loading: false }));
    }

    render() {
      return <WrappedComponent {...this.props} {...this.state} />;
    }
  };
}

Compare it to the hook version (const { data, loading, error } = useFetch(url)): the HOC needs a class, a componentDidUpdate with a manual prevProps comparison, this.state for the three flags, a wrapper component in the tree, and injected props that appear "from nowhere" in the wrapped component. The hook does the same job in one line with none of that. Writing both, back to back, is the fastest way to understand in your bones why React moved to hooks — which is exactly what you set out to learn.

Why the "then and now" mattered

Every modern feature you learned was an answer to a real problem you also saw in its older form. Hooks removed the this-binding tax of classes and the wrapper-hell of HOCs. useEffect collapsed three scattered lifecycle methods into one co-located tool. Custom hooks solved the logic-reuse problem that HOCs and render props had circled for years. You didn't just memorize the modern API — you understand the forces that shaped it, which is what lets you make good judgment calls instead of following rules by rote.
