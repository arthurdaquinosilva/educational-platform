The Ultimate System Design Mastery — A Self-Study Course

A complete course in designing large, real-world software systems. We cover the foundations (scalability, availability, databases, distributed transactions), the canonical case studies (rate limiters, news feeds, CDNs, ride-sharing, chat, Airbnb), and the deeper topics interviewers love (CAP, consensus, sagas, OAuth, the C4 model). By the end, you'll be able to design any system on a whiteboard — and explain your trade-offs.

How to use this course

  • Prerequisites. Some programming experience (any language). Comfort with HTTP and how the web works. Basic understanding of SQL and what a database does. If you've done the previous courses in this series — Foundations, SQL Mastery, JavaScript Mastery — you're more than ready.
  • No setup needed. This course is read-and-think rather than read-and-type. Diagrams, trade-offs, mental models. The occasional pseudocode block isn't meant to be run.
  • You'll need paper or a whiteboard tool. Excalidraw, Miro, Figma, or honest paper and a pencil. System design lives in diagrams; the best way to learn is to draw the systems yourself, not just read about them.
  • Each section follows the same structure: Concept → Why it matters → How it works → Trade-offs → Exercise. The exercises are open-ended ("design X" or "what would you choose, and why") because system design has no single right answer — only better and worse trade-offs.

This course doesn't repeat material from earlier courses (databases, the network stack). Where it overlaps, we go deeper into the distributed-systems-shaped version of the topic.

A note on the case studies: the same handful of systems gets used in interviews everywhere — URL shorteners, news feeds, chat, rate limiters, CDNs. We cover the canonical ones, plus a few less common variations that teach distinct lessons. The goal isn't to memorize answers; it's to build a vocabulary and mental toolkit you can compose for any system.

SECTION 1

The System Design Story

What is "system design," really? Why is it suddenly the hardest interview at every senior engineering job? And what does it actually feel like to design a large system from scratch?

1.1 What System Design Is

System design is the practice of deciding how a software system should be structured at the architectural level — what components exist, how they communicate, where data lives, what fails when, and what trade-offs you're willing to make.

It's different from: - Coding. Coding is "implement this function." System design is "decide which functions even need to exist." - Algorithms. Algorithms are about local efficiency — finding the shortest path, sorting an array. System design is about global efficiency across machines, networks, and time. - Architecture astronomy. Real system design is grounded in concrete trade-offs. Drawing pretty boxes without understanding their costs is the bad-architect stereotype.

A system design problem looks like one of these:

"Design a URL shortener that handles 10,000 requests per second." "Design Twitter's news feed." "Design a notification service that delivers SMS, email, and push to 100 million users." "Design a rate limiter that allows 100 requests per minute per user."

You answer these by sketching the components — load balancers, app servers, databases, caches, queues, CDNs — and explaining why you chose each piece, what fails if it goes down, and how you'd scale it if traffic grew 10×.

There are no "right answers." There are defensible answers and indefensible ones. The job of this course is to teach you to spot the difference.

1.2 Why Companies Care So Much About This

A decade ago, system design interviews were rare — they happened at Google, Amazon, Facebook, and not many other places. Today, every senior engineer interview at every tech company has one, and they often dominate the hiring decision.

Why the change? Three reasons:

  1. The systems got more complex. A 2010 startup ran one Rails app on one server. A 2026 startup runs 50 microservices across 5 regions with a real-time data pipeline, a search cluster, a vector database, and three queues. Even small teams build distributed systems now. Knowing how to design them is no longer specialized — it's table stakes.

  2. Coding interviews stopped distinguishing candidates. Everyone has done LeetCode. Everyone can implement a balanced BST. To find the senior engineers — the ones who can be trusted with architectural decisions — companies needed a different test.

  3. System design correlates with on-the-job performance. Studies and engineering folklore both agree: the engineers who think clearly about systems are the ones who ship working systems. Coding well is necessary but not sufficient.

So if you want senior engineering roles — and especially if you want senior engineering compensation — you have to be good at this.

1.3 A Walk Through the Scaling Journey

The single best way to understand system design is to follow how a system grows from a hundred users to a hundred million. Each milestone introduces a new architectural piece, for a specific reason. We'll cover each piece in depth later — this is just the map.

Stage 0 — One server, one database

       ┌──────────┐
User → │  Server  │ ← runs your app
       │          │
       │ Database │ ← all data on disk
       └──────────┘

A single machine, running your application and your database on the same box. This is fine for your side project, your MVP, an internal tool with 50 users.

Limits: the machine has finite CPU, RAM, disk, and network. As users grow, you hit one of those limits and the app gets slow.

Stage 1 — Split application and database

       ┌──────────┐         ┌──────────┐
User → │   App    │ ←─────→ │ Database │
       │  server  │         │  server  │
       └──────────┘         └──────────┘

Move the database to its own machine. Now the app server doesn't fight the database for memory and CPU. You can scale each one independently.

This is the natural first step. Almost every production system looks at least like this.

Stage 2 — Multiple app servers behind a load balancer

                    ┌────────┐
                ┌─→ │ App #1 │
                │   └────────┘
User → ┌────┐ ──┤   ┌────────┐
       │ LB │   ├─→ │ App #2 │ ←→ ┌──────────┐
       └────┘ ──┤   └────────┘    │ Database │
                │   ┌────────┐    └──────────┘
                └─→ │ App #3 │
                    └────────┘

Add more application servers. A load balancer routes incoming requests across them. If one app server dies, the load balancer skips it; the others keep serving.

This is horizontal scaling: instead of buying a bigger machine ("vertical"), you add more machines ("horizontal"). Horizontal scaling is the foundation of every big system.

Two new properties this gives you: - Capacity — more servers, more concurrent users. - Availability — one server can fail without taking the site down.

Stage 3 — Cache in front of the database

       ┌──────────┐    fast    ┌─────────┐
App  → │  Cache   │ ─────────→ │ Database│
       │ (Redis)  │            └─────────┘
       └──────────┘

The database is now the bottleneck. Every read hits disk, every write hits disk, the queries pile up. Solution: put a fast in-memory cache (Redis, Memcached) in front of it.

When the app needs data, it asks the cache first. If the cache has it, that's the answer (microseconds). If it doesn't, the app queries the database, and stores the result in the cache for next time.

Caches turn 80%+ of database reads into in-memory reads, sometimes 99%+. The database load drops accordingly. We'll cover caching strategies (write-through, write-behind, cache-aside) in Section 6.

Stage 4 — Replicate the database

          writes
App ────────────────→ ┌──────────┐
                      │ Primary  │ ─┐
                      └──────────┘  │ replication
          reads                     │
App ────────────────→ ┌──────────┐ ←┘
                      │ Replica  │
                      └──────────┘

Reads still outweigh writes (typical web app: 100 reads per write). Add read replicas — copies of the database that stay in sync with the primary. The app sends writes to the primary; it sends most reads to a replica.

This multiplies read capacity. It also gives you a backup — if the primary dies, you can promote a replica. We'll cover replication topologies in Section 4.

Stage 5 — Shard the database

Even with caching and read replicas, the primary has to absorb every write. Eventually it can't keep up. The solution: sharding (or partitioning) — split the data across multiple databases.

                                  ┌───────────┐
                                  │ Shard A   │  users 1–10M
                ┌─→ writes ──────→│           │
                │                 └───────────┘
                │                 ┌───────────┐
App → router  ──┼─→ writes ──────→│ Shard B   │  users 10M–20M
                │                 └───────────┘
                │                 ┌───────────┐
                └─→ writes ──────→│ Shard C   │  users 20M–30M
                                  └───────────┘

Each shard handles a slice of the data. Writes are spread across the shards. Capacity scales with shard count.

This is a significant architectural step. Sharding introduces hard problems — how do you query across shards? what if a user moves from shard A to shard B? what about transactions spanning shards? We'll devote Section 4 to it.

Stage 6 — Add a CDN

Users in Tokyo loading content from your San Francisco servers experience hundreds of milliseconds of latency just for the network round-trip. A Content Delivery Network (CDN) caches your static content (images, JavaScript, CSS, videos) at hundreds of "edge" locations around the world. Tokyo users get content from Tokyo edges. Latency drops to single-digit milliseconds.

              edge nodes
       Tokyo ─────────────┐
       Paris ─────────────┤    your origin
       Lagos ─────────────┼───→ servers
       Lima  ─────────────┤
       Sydney ────────────┘

CDNs handle a huge fraction of traffic for any global site. We'll cover them in depth as a case study (Section 13).

Stage 7 — Functional partitioning (microservices)

Your monolithic app — one big codebase that does everything — has become too big to reason about. 80 engineers commit to it. Deployments take 40 minutes. One bug in the checkout code crashes the search feature.

Solution: split the app into separate services, each owning one slice of functionality. A "user service" handles users. A "payment service" handles payments. A "search service" handles search. Each has its own database, its own deployment, its own team.

     ┌──────────────────────────────────────┐
     │            API Gateway               │
     └──────────────────────────────────────┘
        │         │         │         │
        ▼         ▼         ▼         ▼
     ┌─────┐  ┌──────┐  ┌──────┐  ┌──────┐
     │User │  │ Order│  │Search│  │Notif │
     │  svc│  │  svc │  │  svc │  │  svc │
     └─────┘  └──────┘  └──────┘  └──────┘
        │         │         │         │
        ▼         ▼         ▼         ▼
       DB        DB        ES        DB

This is the microservices architecture. It solves coordination problems but creates new operational problems — network calls between services, deployment complexity, distributed tracing, etc. We have a whole section on the monoliths-vs-microservices debate (Section 18).

Stage 8 — Asynchronous processing with queues

Some operations are slow — sending an email, processing a video, charging a credit card. Doing them synchronously inside an HTTP request makes the user wait, ties up your servers, and risks failure if any step is flaky.

Solution: put work on a message queue (Kafka, RabbitMQ, SQS). The web server pushes a job onto the queue and returns immediately. Background workers consume the queue and do the slow work.

    HTTP request                              ┌─────────┐
   ─────────────→ ┌────────┐    ┌───────┐    │ Worker  │
                  │  Web   │ →  │ Queue │ → ┤  Worker  │
   ←───────────── │ server │    │       │    │ Worker  │
        200 OK    └────────┘    └───────┘    └─────────┘
                                  (fast)        (slow)

This gives you decoupling, resilience (workers can retry failed jobs), and load smoothing (a traffic spike fills the queue but doesn't overwhelm the workers).

Most modern systems are heavily queue-based. We cover messaging systems in Section 6.

Stage 9 — Search, analytics, and specialized stores

Your relational database is great for transactional reads/writes but terrible at full-text search. You add Elasticsearch for search. Then you add a data warehouse (Snowflake, BigQuery, Redshift) for analytics, because running analytics queries against your production database is a recipe for outages. Then you add a time-series database (InfluxDB, TimescaleDB) for metrics. Then a vector database (pgvector, Pinecone) for semantic similarity.

Each is a specialized tool for a specific data shape. The lesson: there's no one database that's good at everything. Mature systems use several, each suited to a particular workload.

Stage 10 — Multi-region, edge computing, serverless

The endgame: your service runs in multiple geographic regions, with traffic routed to the nearest one. Static content lives at the edge (CDN). Some compute runs at the edge too (Cloudflare Workers, Vercel Edge Functions). Some workloads run serverlessly (AWS Lambda, Google Cloud Run) — you don't manage servers at all; the cloud provider runs your code on demand and bills you per request.

This is what large web companies look like today. Getting here takes years and dozens of engineers, but the pieces are the same as Stages 0-9 — just more of them, deployed more carefully.

1.4 The Four Constraints That Drive Everything

Every architectural decision in system design is a response to some combination of four constraints:

1. Latency — how long does a request take? Users notice 100ms. Anything over 1 second feels slow. Anything over 10 seconds feels broken.

2. Throughput — how many requests per second can the system handle? A small site handles 10. A medium site handles 1,000. Twitter handles 6,000 tweets per second on average, much more at peaks.

3. Availability — what fraction of time is the system up? "Three nines" (99.9%) means 8.76 hours of downtime per year. "Five nines" (99.999%) means 5.26 minutes. Each extra nine costs roughly an order of magnitude more.

4. Consistency — when one user writes data, when do other users see the new value? Strong consistency means "immediately." Eventual consistency means "soon, but maybe a few seconds." Stronger consistency is more expensive.

Most architectural debates reduce to these four. We'll formalize them in Section 3 (Non-Functional Requirements) and meet them again, constantly, throughout the course.

1.5 Key Principles to Internalize

A few mental models that show up across every chapter.

Trade-offs everywhere. Every decision has a cost. You don't get strong consistency and high availability and low latency — you pick two, or you pick partial versions of each. Senior engineers don't pretend the trade-offs aren't there; they reason about them explicitly.

Failure is the default. Networks fail. Disks fail. Servers crash. With enough machines, something is failing right now. Good systems are designed to expect failure and recover gracefully — not to assume everything works.

Locality beats cleverness. Putting data near the computation that uses it (caches, CDNs, sharding by user region) beats almost any algorithmic improvement. Latency comes from distance and from waiting; both are reduced by locality.

No silver bullet. No database, framework, language, or pattern solves everything. Anyone selling you "the one true architecture" hasn't built enough systems. The right choice depends on your specific problem.

Conway's Law. "Organizations design systems that mirror their communication structures." If you have three teams, you'll end up with three services, whether you wanted them or not. System design and team design are linked.

Start simple, evolve. Most systems don't need to be distributed. Most companies that built microservices first regretted it. Build a monolith. Scale it as far as it'll go. Split out services only when the pain is real. We have a whole section on this debate (Section 18).

SECTION 2

The Interview Flow

Before we go deep on technology, let's talk about how to actually approach a system design problem. There's a structure that works. Without it, you'll panic and ramble. With it, you'll look senior even on questions you haven't seen before.

This section is short on facts and long on process. It applies in interviews, but also in real design meetings, RFC writing, and any time someone says "how would you build X?"

2.1 The 45-Minute Question

A system design interview is typically 45 minutes. That's not much time. Most candidates waste 15 minutes flailing before they find a structure. Don't be that candidate.

A structure that works:

Time Phase What you're doing
0–5 min Requirements Asking clarifying questions, locking down scope
5–10 min Estimation Back-of-the-envelope numbers — QPS, storage, bandwidth
10–20 min High-level design Major components on the board, the request flow
20–35 min Deep dives Pick 2–3 components, go deep — data models, algorithms, scaling
35–42 min Bottleneck hunting Where would this break? What would you change first?
42–45 min Wrap-up Quick recap, things you'd improve given more time

This isn't a formula. Real interviews drift. But if you have this in your head, you'll never be stuck wondering "what should I do next?"

Let's go through each phase.

2.2 Phase 1 — Requirements (5 minutes)

When the interviewer says "design Twitter," they don't mean all of Twitter. They mean some part of it, with some scale, with some specific concerns. Your first job is to figure out which part.

Ask these questions, in roughly this order:

Functional requirements — what does the system do? - "What are the core features I should focus on? Posting tweets, the feed, search, notifications?" - "Are we covering both reads and writes, or focused on one side?" - "What features can I skip? (Ads, DMs, analytics, etc.)"

Scale — how big is this? - "How many users? Daily active, monthly active?" - "How many requests per second?" - "How much data?"

Non-functional requirements — what are the priorities? - "What's our latency budget? Is sub-second OK, or do we need P99 under 100ms?" - "What about availability? Is some downtime OK?" - "Read-heavy or write-heavy?"

Constraints — what's off the table? - "Are we cost-sensitive, or designing for scale at all costs?" - "Multi-region from day one, or single region OK?" - "Are there specific compliance constraints?"

You don't need answers to every question. The point is to clearly establish what you're designing. Write the agreed-upon requirements on the board (or in chat). Refer to them throughout.

A common rookie mistake: diving straight into the architecture without asking anything. The interviewer hands you a vague problem on purpose — they want to see if you'll ask. Asking shows seniority.

Another common mistake: asking too many questions. Five minutes is the budget. Don't spend twenty minutes interviewing the interviewer. Get enough information to start, then start.

2.3 Phase 2 — Back-of-the-Envelope Estimation (5 minutes)

Numbers anchor design. Without them, you're guessing whether something can fit on one server or needs a thousand. Estimation tells you.

You don't need to be exact. You need to be in the right ballpark — within a factor of 10 of reality. This is enough to make design decisions.

A few numbers to memorize:

Quantity ≈
Seconds in a day 86,400 → round to 10⁵
Days in a year 365 → round to 400
1 KB 1,000 bytes
1 MB 10⁶ bytes
1 GB 10⁹ bytes
1 TB 10¹² bytes
L1 cache reference 0.5 ns
Memory reference 100 ns
SSD read 100 µs
Cross-datacenter round trip 100 ms
HDD seek 10 ms

(That last group — the "Latency Numbers Every Programmer Should Know" list — has become an interview classic. Memorize the order of magnitude.)

A worked example. "Design Twitter."

  • 300 million daily active users.
  • Average user posts 2 tweets/day. Total: 600M tweets/day = 600M / 86,400 ≈ 7,000 tweets per second. Peaks maybe 10×: 70,000 tweets per second.
  • Average user reads 100 tweets/day. Total reads: 30 billion / day ≈ 350,000 reads per second. Peaks: 3.5 million.
  • Each tweet is ~280 chars + metadata, say 1 KB. Daily storage: 600M × 1 KB = 600 GB/day ≈ 200 TB/year.

These three numbers (write QPS, read QPS, storage growth) drive most decisions: - 70K writes/sec needs partitioning. One database can't handle that. - 3.5M reads/sec needs heavy caching + replicas. - 200 TB/year is a lot. You'll need cheap object storage for older data.

Estimation takes practice. Two pieces of advice: 1. Round aggressively. Don't compute 86,400 × 365.25 — round to 10⁵ × 400 and call it 4 × 10⁷ seconds/year. 2. Sanity-check. If your answer is "we need 10 million servers," recheck. If it's "we need 0.5 servers," that's also probably wrong.

We'll do estimation in every case study. By the end of the course, it'll feel natural.

2.4 Phase 3 — High-Level Design (10 minutes)

Now you draw boxes. The goal: get the major components and request flow on the board.

Start simple. The first iteration of any design should look like:

[Client] → [Load Balancer] → [App Servers] → [Database]

Even for "design Twitter," start here. Then add. "We'll add caching for the read path. We'll add a write path that goes through a queue for fan-out..."

A good high-level design has 5–10 components. Not 2 (too vague). Not 30 (drowning in detail).

Walk through the happy-path flow in plain English: "The user posts a tweet. The client sends an HTTP POST to the load balancer, which routes to an app server, which writes to the database and pushes a fan-out job to a queue. Workers consume the queue and push the tweet ID into the followers' news feed caches."

This walk-through is critical. It forces you to verify the design works end to end. Many bad designs only become obviously bad when you try to explain how a request flows through them.

A common mistake: jumping into deep-dive details on one component before the high-level structure is sketched. You haven't earned the deep dive yet.

2.5 Phase 4 — Deep Dives (15 minutes)

Now pick 2–3 components and go deep. Which 2–3? Some combination of: - The component that's hardest to get right (often the database — schema, sharding, indexes). - The component the interviewer keeps nudging you toward. - The component that drives the most traffic or cost.

Examples of "going deep":

  • For the database: what's the schema? What are the queries? What index do you need? How do you shard? What's the partition key? How do you handle the partition-hotspot problem?
  • For the cache: cache-aside vs write-through? What TTL? What's the eviction policy? What happens on a cache miss storm?
  • For the queue: at-least-once vs at-most-once delivery? How big do messages get? What if a worker fails halfway through processing?
  • For the API: REST vs gRPC? What are the endpoints? How do you version the API? Pagination?

Don't try to deep-dive everything. Two well-explored components beat seven shallow ones.

Listen to the interviewer. If they say "let's talk more about the feed generation," that's a hint — they want a deep dive on feed generation. Take it. Some candidates push back ("but I also want to talk about authentication"). Don't. The interviewer is steering you toward the most interesting part of the problem; let them.

2.6 Phase 5 — Bottleneck Hunting (7 minutes)

Step back. Where does this design break first? What would you change?

A good way to find bottlenecks: walk through traffic 10× or 100× higher than the original requirements. "At our current scale this works, but if we hit a billion users tomorrow…"

You'll usually find one or more of: - A single database that can't handle the write load → shard it - A single component that's a single point of failure → replicate it - A synchronous step that adds latency → make it async via queue - A read pattern that hits the database too hard → add caching - A spike pattern that overwhelms the system → add rate limiting

Discussing these is valuable. It shows you understand the design isn't done — it has weaknesses, and you know what they are.

2.7 What Interviewers Actually Evaluate

Most candidates think they're being graded on: - Whether they got the "right" answer. - How clever their design is. - Whether they used the most fashionable technology.

They're actually being graded on: - Communication. Can you explain a complex idea clearly? Can you stay organized? - Trade-off awareness. Do you know why you're choosing X over Y? Or are you cargo-culting? - Engineering maturity. Do you ask the right questions? Do you push back when the requirements don't make sense? Do you say "I'd punt on that for v1 and add it in v2"? - Collaboration. Are you easy to work with? Can the interviewer steer you, or do you ignore their hints? - Depth. Can you go from "we use a queue" to actually explaining queue semantics?

Getting "the right answer" matters far less than thinking clearly out loud. A candidate who builds an imperfect but well-reasoned system, explaining their trade-offs as they go, beats a candidate with a perfect design who can't explain it.

2.8 Common Mistakes

A non-exhaustive list:

1. Jumping in without clarifying. Lose 15 minutes building the wrong thing because you didn't ask "are we doing reads or writes?"

2. Overengineering for scale you don't need. Sharding the database for 1,000 users. Microservices for an MVP. Senior engineers don't over-engineer; they pick the right complexity for the requirements.

3. Underengineering for scale you do need. Designing Twitter as a single Rails app on one server. The interviewer told you 300 million users — match the design to the scale.

4. Buzzword bingo. "We'll use Kafka and Kubernetes and gRPC and ML and a service mesh." Why? Because they sound impressive. If you can't explain why each piece earns its place, leave it out.

5. Single point of failure ignored. Drawing one load balancer, one database, one cache. The interviewer will absolutely ask "what happens when that goes down?" Plan for it.

6. Skipping the unsexy parts. Authentication. Monitoring. Backups. These show seniority. Mention them.

7. Drawing the perfect first design. Real systems evolve. Show that you think iteratively: "v1 is the simple thing; v2 adds caching when we hit this bottleneck; v3 shards when we hit this one."

8. Talking past hints. The interviewer says "what if a database write fails halfway?" Don't say "yeah we'd handle that" and move on. They're telling you to discuss it.

9. Silence. Long silent thinking is hard for the interviewer to evaluate. Narrate your thinking out loud, even when you're not sure. "I'm trying to decide between A and B. A means…, B means…, I think I lean A because…"

10. Forgetting non-functional requirements. You drew a nice diagram of components, but you never said anything about latency, availability, or consistency. Those are the constraints that justify the diagram.

2.9 A Worked Mini-Example

"Design a URL shortener like bit.ly."

Requirements (1 minute): - Functional: shorten a long URL into a short one; redirect short URL to long URL. - Scale: assume 100M URLs created per month, 1B URL lookups per day. - Latency: redirect should feel instant. Sub-50ms. - Read-heavy by orders of magnitude.

Estimation (1 minute): - 100M / month = ~40 new URLs per second. - 1B / day = ~12,000 redirect lookups per second. - Storage: each row maybe 500 bytes (short code + long URL + metadata). 100M × 500 B = 50 GB/month, ~600 GB/year. Manageable.

High-level design (3 minutes): - Client → CDN → API gateway → App servers - App servers ↔ Cache (Redis) - App servers ↔ Database (key-value store keyed on short code; long URLs are the values) - Short code generation: hash the long URL, or use a counter encoded in base 62

Deep dive — short code generation (3 minutes): - Option A: hash the URL (MD5, take first 7 chars, base 62). Risk: collisions, plus same URL always shortens to same code (privacy issue if two people shorten the same private URL). - Option B: distributed counter, encoded in base 62. Each app server pre-allocates a range of IDs. Simple, no collisions, ordered. - Discuss the trade-offs. Pick Option B.

Deep dive — read path (2 minutes): - 99% of redirects hit the cache. - Cache miss → DB lookup → cache the result. - DB is read-heavy → add read replicas. - CDN at the edge → for the most popular URLs, the request never even hits your origin.

Bottlenecks (2 minutes): - Cache is a single point of failure → use a Redis cluster. - DB writes are 40/sec, not a problem yet — but if it grew, shard by short code prefix. - Hot URLs (viral content) might cause cache hot spots → use multi-level cache or replicate hot keys.

Wrap-up (1 minute): - Mention: analytics (click tracking — async via queue), custom slugs, expiration, rate limiting against abuse.

SECTION 3

Non-Functional Requirements

Every system has two kinds of requirements:

Functional requirements — what the system does. "Users can post tweets." "The system charges credit cards." These are about features.

Non-functional requirements — how the system does it. "Tweets show up within 200ms." "The system is available 99.99% of the time." "Charging is consistent — no double-charges, ever." These are about quality attributes.

Functional requirements are usually obvious. Non-functional requirements are where senior thinking lives — they're what drive architectural choices. Most of system design is about meeting non-functional requirements while keeping cost and complexity tolerable.

This section gives you the full vocabulary. The next few sections build on it.

3.1 Latency vs Throughput

These are often confused. They're related but different.

Latency is the time for one request to complete, end to end. Measured in milliseconds or microseconds. The user clicks a button; the page updates 150ms later — that's 150ms of latency.

Throughput is the rate of work the system can do. Measured in requests per second (QPS, RPS). "We handle 50,000 requests per second" is throughput.

These can vary independently. A coffee shop with one barista has high latency (each customer waits a long time) but low throughput. A drive-thru with twelve windows has low latency and high throughput. A batch job that processes a million records overnight has high throughput but enormous latency per record.

You optimize them differently: - Lower latency → faster network paths, in-memory caches, simpler algorithms, fewer round trips. - Higher throughput → more servers, parallelism, asynchronous processing, queueing.

Sometimes you trade one for the other. Batching multiple operations together increases throughput but adds latency (you wait for the batch to fill). Removing batches decreases throughput but reduces latency.

Latency percentiles

A single "average latency" number lies. Latency is a distribution. Some requests are fast; some are slow; the slow ones are what users remember.

Real teams measure latency in percentiles:

  • P50 (median) — half of requests are faster than this. The "typical" experience.
  • P95 — 95% of requests are faster. The "slow but not awful" experience.
  • P99 — 99% are faster. The "almost-worst" case.
  • P99.9, P99.99 — the tail. Important for systems with millions of users.

If you serve a billion requests a day and your P99 is 2 seconds, that means 10 million requests per day are slower than 2 seconds. Tail latency is huge in aggregate.

Concrete example. Your service has these latencies: - P50: 50ms - P95: 100ms - P99: 200ms - P99.9: 2,000ms

That P99.9 of 2 seconds means roughly 1 in 1,000 requests are taking 2+ seconds. For most users this is invisible. For your highest-volume customers — making thousands of requests per page — they will consistently hit some of these slow requests, and their experience will feel awful.

This is why tail latency matters disproportionately. Optimizing P99 and P99.9 is harder than optimizing the average, and often more important.

Latency budget

For any user-facing operation, you have a latency budget — the total time the user will tolerate. Decompose it across the components in the request path:

Total budget: 500ms
  ├── Network round trip: 50ms
  ├── Load balancer: 5ms
  ├── App server: 50ms (your code)
  ├── Database queries (3): 150ms
  ├── Cache lookups (5): 25ms
  └── External API call: 200ms

Total: 480ms — fits the budget, with 20ms margin

If any single piece blows the budget, the whole request feels slow. Senior engineers think in budgets.

3.2 Availability — The Nines

Availability is the fraction of time the system is functioning correctly. Expressed as a percentage of "nines":

Availability Downtime per year Downtime per month
99% (two nines) 3.65 days 7.2 hours
99.9% (three nines) 8.76 hours 43.2 minutes
99.99% (four nines) 52.6 minutes 4.3 minutes
99.999% (five nines) 5.26 minutes 26 seconds
99.9999% (six nines) 31.5 seconds 2.6 seconds

Each extra nine costs roughly 10× the engineering investment. The jump from 99.9% to 99.99% is bigger than it looks — you need redundant everything, automated failover, regional replication, and a culture of operational discipline.

Most consumer apps target 99.9% or 99.99%. Critical infrastructure (payment networks, telecom backbones, hospital systems) targets five or six nines.

SLO, SLA, SLI

These three letters get used together constantly.

  • SLI (Service Level Indicator) — what you measure. "P99 latency." "Error rate." "Availability percent over the last 30 days."
  • SLO (Service Level Objective) — your internal target. "P99 < 200ms." "Availability > 99.95%."
  • SLA (Service Level Agreement) — the contractual promise to customers. Usually stricter than internal SLO would suggest? No — actually looser. Companies promise less in writing than they aim for internally, so they don't pay penalties.

Example. AWS's S3 SLA: 99.9% monthly uptime. Their internal SLO is much higher — probably 99.99%+. Their SLI is measured continuously.

Availability is multiplicative across dependencies

If your service depends on three components, each at 99.9% availability, your effective availability is:

0.999 × 0.999 × 0.999 = 0.997 (~99.7%)

Worse than any single component. Every dependency drags down your availability. This is why architects obsess over reducing the number of hard dependencies and adding fallback paths.

The fix: design so a failure in one dependency doesn't cascade. Use timeouts, retries, circuit breakers, graceful degradation. We'll cover these patterns in case studies.

3.3 Reliability

Reliability is the probability that the system performs correctly for some period. Closely related to availability but distinct: availability is about being up; reliability is about doing the right thing while up.

A database that's available 99.99% of the time but corrupts 0.1% of writes is highly available and unreliable.

Key reliability concepts:

  • MTBF (Mean Time Between Failures) — how long the system runs before failing.
  • MTTR (Mean Time To Repair) — how long it takes to fix a failure.
  • Fault tolerance — the system continues operating when a component fails.
  • Graceful degradation — when something fails, the system loses functionality, not data.

A reliable system might still go down — but when it does, it doesn't corrupt data, doesn't lose user requests permanently, doesn't return wrong results.

3.4 Consistency

Consistency is when changes become visible. It's the trickiest non-functional requirement to reason about, especially in distributed systems.

There are several consistency models. Three to know:

Strong consistency

After a write completes, all subsequent reads see the new value. Like a single-machine database.

Easy to reason about. Expensive to provide across multiple machines.

Eventual consistency

After a write, reads might see the old value for a while. The system converges — eventually all replicas have the new value. But "eventually" might be milliseconds, seconds, or longer.

Cheap to provide. Hard to reason about — your code has to handle "I just wrote X but I'm reading X' (the old value)."

DNS is famously eventually consistent. When you update a DNS record, it takes hours to propagate globally. That's an acceptable trade-off for DNS's scale.

Read-your-own-writes

A specific guarantee: a user always sees their own writes immediately, even if other users don't. Compromise between strong and eventual: cheaper than strong everywhere, but masks the surprise of "I posted a tweet and it disappeared."

Most user-facing systems aim for this. Twitter does it. Facebook does it. They sacrifice global consistency but maintain the illusion for the writer.

Linearizability — the strongest form

If write W1 finishes before write W2 starts, every reader sees W1 before W2. It's "as if there was a single global clock." Hardest to achieve in distributed systems. Required for some operations like leader election, distributed locks.

Causal consistency

If event A causes event B, every observer sees A before B. Weaker than linearizability but stronger than eventual. Allows reasonable cause-and-effect reasoning.

The takeaway: consistency is a spectrum, not a binary. Picking where you sit on the spectrum is a design choice with cost implications.

3.5 The CAP Theorem

The single most cited theorem in system design. Eric Brewer, 2000.

CAP says: in a distributed system, when there's a network partition, you can have either Consistency (every read sees the latest write) or Availability (every request gets a response) — but not both.

The "network partition" part matters. The theorem is about what happens during a partition, not always. The acronym is slightly misleading: it should really be "in the presence of P, choose C or A."

Visual:

              Consistency
                  /\
                 /  \
                /    \
               /      \
   (CP)       /        \      (AP)
  Strict    /            \   Highly
 banking   /              \  available
 systems  /                \  social feeds
         /                  \
        /__________________ \
       Availability          Partition tolerance
              (P is always required in distributed systems)

In practice: - CP systems (Consistency + Partition tolerance): traditional databases (PostgreSQL with synchronous replication), distributed consensus systems (ZooKeeper, etcd, Spanner). When a partition happens, reads/writes may fail, but the data stays consistent. - AP systems (Availability + Partition tolerance): DynamoDB (in eventual mode), Cassandra, CouchDB, most NoSQL stores. Always responsive, but may serve stale data during a partition.

CA is mostly a myth — any real distributed system has to tolerate partitions, because networks fail. So the meaningful choice is C-vs-A, when P inevitably happens.

CAP is useful as a vocabulary tool but oversimplifies real systems. Modern wisdom: see PACELC.

3.6 PACELC — The Better Theorem

CAP only describes what happens during a partition. PACELC (Daniel Abadi, 2010) describes the whole picture:

"If there is a Partition (P), how does the system trade off Availability (A) and Consistency (C); Else (E), when the system is running normally, how does it trade off Latency (L) and Consistency (C)?"

Two trade-off pairs: - During partition: A vs C - Normally: L vs C (latency vs consistency)

So a system might be: - PA/EL — sacrifices C both during partition and normally for low latency. Most "AP" NoSQL databases. - PC/EC — strong consistency always. Traditional ACID databases. - PA/EC — eventually consistent during partition, strongly consistent normally. Cosmos DB. - PC/EL — pretty unusual; would mean strict consistency during partition but eventual normally.

The point of PACELC: even without a partition, there's still a real trade-off between latency and consistency. A globally distributed database that wants strong consistency has to do extra round-trips to coordinate — that adds latency.

You'll often see modern databases described in PACELC terms in their docs. Worth understanding.

3.7 Durability

Durability is the guarantee that once a write is acknowledged, it survives failures.

A non-durable system might accept your write, store it in memory, and then lose it when the machine crashes. A durable system writes to disk (often to multiple disks across multiple machines) before acknowledging.

Levels of durability: - No durability — purely in-memory caches. Data is lost on restart. - Single-disk durability — written to one disk. Survives a process crash but not a disk failure. - Replicated durability — written to multiple disks across multiple machines. Survives single-machine failures. - Cross-region durability — replicated across geographic regions. Survives entire datacenter outages.

Higher durability is more expensive (more disks, more network coordination, higher write latency). Banks need cross-region durability. Logging systems often don't.

Important: durability requires the write to be persisted before acknowledgment. A system that says "200 OK" before actually writing to disk is not durable. Performance optimizations sometimes weaken this — be explicit about whether you can.

3.8 Scalability — Vertical vs Horizontal

Scalability is the system's ability to handle more load.

Two ways to scale:

Vertical scaling (scale up) — upgrade the machine. Bigger CPU, more RAM, faster disk. The instance type goes from "m5.large" to "m5.24xlarge."

Pros: simple, no code changes, no distributed-systems headaches.

Cons: hard cap (you can only buy so big a machine), expensive (big machines have nonlinear pricing), single point of failure.

Horizontal scaling (scale out) — add more machines. The system runs on 10 servers instead of 1, then 100, then 1,000.

Pros: virtually unlimited, fault tolerant (lose one machine, the others handle it), cost-efficient.

Cons: requires the system to be designed for distribution — stateless servers, distributed databases, careful coordination.

Modern systems lean heavily on horizontal scaling. The cloud is built around it. But starting with a vertically-scaled monolith is often the right move for small systems — don't go horizontal before you need to.

A key prerequisite for horizontal scaling: statelessness. If each app server stores user session in memory, you can't easily move requests between servers. If sessions live in a central store (Redis, JWT, etc.), any server can handle any request, and you can add/remove servers freely.

3.9 Maintainability and Observability

These don't show up in interviews as often as the others — but they're what separates real engineering teams from amateurs.

Maintainability — how easy is the system to change? Can a new engineer add a feature without breaking three others?

Observability — can you see what's happening inside the system? When something breaks, can you find out why, fast?

Three pillars of observability: - Logs — events with timestamps. "User 42 logged in at 13:42:01." - Metrics — aggregated numbers over time. "P99 latency over the last hour." - Traces — request flow across services. "This user's request hit service A, then B, then C; B took 800ms."

Common tooling: Prometheus + Grafana for metrics, Elasticsearch/Loki for logs, OpenTelemetry/Jaeger/Datadog for traces.

Without observability, debugging production issues feels like an MRI without imaging. You see the patient is sick but can't tell what's wrong. Mature systems invest in observability before they need to.

3.10 Security and Compliance

Briefly — security is its own discipline, but you should at least mention:

  • Authentication — who is the user? (We cover OAuth 2.0 in Section 19.)
  • Authorization — what is this user allowed to do?
  • Encryption — at rest (data on disk) and in transit (TLS).
  • Auditing — who did what, when?
  • Compliance — GDPR, HIPAA, PCI-DSS, SOC2. Each adds specific constraints.

In an interview, briefly mentioning these signals seniority, even if you don't go deep. "I'd put auth behind the API gateway with OAuth, encrypt PII at rest, and log accesses for audit." Twenty seconds, big credibility.

3.11 Costs

The constraint that always wins, eventually. Every architectural choice has a cost: server time, storage, network egress, engineering time.

A few realities: - Network egress is expensive. Data leaving a cloud region (especially to the public internet) costs a lot. Cross-region data transfer is the silent budget killer. - Storage is cheap; access is not. S3 storage is fractions of a cent per GB. Operations (reading, listing) add up if you make millions. - Compute is steady but huge at scale. Cloud bills tend to look like: 60% compute, 20% storage, 15% network, 5% miscellaneous. - Premium tiers cost much more. Spot instances (1/3 the price, can be terminated). On-demand. Reserved (1-3 year commitments, big discount). Senior engineers know which to use where.

In real architecture work, the question is rarely "what's the best design?" but "what's the best design for the budget?" Mentioning cost in an interview is rare and powerful.

3.12 Putting It All Together

When you start a system design problem, write the non-functional requirements explicitly at the top:

- Functional: post tweets, view feed, search
- Scale: 300M DAU, 70K writes/sec peak, 3.5M reads/sec peak
- Latency: P99 < 200ms for reads, < 1s for writes
- Availability: 99.95% (4.4 hours/year downtime)
- Consistency: read-your-own-writes; eventual otherwise
- Durability: tweets must survive a region failure
- Cost: this is YouTube-scale; budget is generous

Now every architectural choice can be justified — "we use eventual consistency because our requirement says so, not because I felt like it."

This is the difference between a junior and a senior engineer in an interview: the senior explicitly grounds choices in stated requirements. The junior makes choices based on what they've heard is "best."

SECTION 4

Scaling Databases

For most systems, the database is the first thing to break under load. CPU you can add; servers you can multiply; but a database has state, and state is hard to scale. This section is the longest in Part 1 because it's the most important.

We'll cover: vertical limits, replication, sharding, indexes, the SQL-vs-NoSQL question, and the specialized stores you'll meet in the case studies.

4.1 The Vertical Limit

Your first instinct should be to not shard. A modern database on a single beefy machine handles a startling amount of traffic. Concretely:

  • A well-tuned PostgreSQL on a 32-core, 256GB RAM box: roughly 10,000 writes/sec, 50,000+ reads/sec for typical workloads.
  • A high-end machine (96 cores, 768GB RAM, NVMe storage): can do 50,000 writes/sec, 500,000 reads/sec.

That covers a substantial number of internet companies. The number-one anti-pattern in junior system design: jumping to sharding before exhausting vertical scaling. Don't.

Vertical scaling limits hit eventually: - CPU — fixed cores per machine. Even AWS's biggest instances cap somewhere around 128 cores. - Memory — can be huge (terabytes) but expensive. - Disk I/O — the real bottleneck for most databases. NVMe helps, but you still cap at some thousands of IOPS per disk. - Network — a single machine's NIC is 10-100 Gbps.

Before you hit any of these, you've probably hit: - Lock contention — too many transactions fighting for the same rows. - Connection limits — each database connection costs memory; thousands of connections become a problem. - Slow queries — usually a missing index, not actual scale.

The first response to "the database is slow" is almost never sharding. It's: find the slow queries, add the missing indexes, fix the N+1 problems, add caching, then add read replicas. Sharding is the last resort.

4.2 Replication

Replication is keeping multiple copies of your data on multiple machines. Two reasons: 1. Availability — if one copy dies, the others continue serving. 2. Read scalability — multiple read replicas can absorb read traffic.

Primary-replica (master-slave) replication

The classic shape:

                writes
   ──────────────────────→  ┌─────────┐
   App                      │ PRIMARY │
                            └─────────┘
                                 │ replicates
                ┌────────────────┼────────────────┐
                ▼                ▼                ▼
            ┌────────┐      ┌────────┐      ┌────────┐
            │ Replica│      │ Replica│      │ Replica│
            └────────┘      └────────┘      └────────┘
                ▲                ▲                ▲
                └────── reads ───┴───── reads ────┘

One primary handles all writes. The primary streams its changes (a log of every modification) to the replicas. Replicas apply the log to stay in sync.

Reads go to replicas, writes go to the primary. This multiplies read capacity (add more replicas) but doesn't help write capacity (still one primary).

Synchronous vs asynchronous replication

When the primary writes data, when does it tell the client "done"?

Synchronous replication — primary waits for at least one replica to confirm before acknowledging the write. Strong durability (the write is safe on at least two machines). High latency (each write waits for a network round trip).

Asynchronous replication — primary acknowledges immediately, then replicates in the background. Fast writes. Risk: if the primary crashes before replication finishes, those writes are lost.

Semi-synchronous — wait for at least one replica out of several. Middle ground.

The choice is a real trade-off. PostgreSQL defaults to async but supports sync. MySQL has all three modes. Most cloud-managed databases (RDS, Cloud SQL) offer per-write configuration.

Replication lag

The gap between when the primary commits a write and when a replica has it. Usually milliseconds. Under load, it grows.

This creates the classic problem: user writes data, immediately reads it back from a replica, doesn't see their own write. They post a tweet, refresh, the tweet is gone (it's on the primary; the replica hasn't received it yet).

Fixes: - Read-your-own-writes from primary — after a write, route that user's reads to the primary for some time. - Versioning — the write returns a version number; the read requires that version or higher. - Wait for replication — write, then wait for confirmation that the data has replicated, then return.

All real systems hit this and handle it somehow.

Multi-primary (multi-leader) replication

Multiple nodes accept writes simultaneously. Each replicates to the others.

   App ─→ Primary A ←──────→ Primary B ←── App
              │                  │
              └────→ Replicas ←──┘

Why? Geographic distribution: writes in Tokyo go to Tokyo's primary; writes in Paris go to Paris's primary. Lower write latency.

Cost: write conflicts. What if user X updates record R at the same moment on Primary A and Primary B? Conflict resolution is a hard problem — last-write-wins, manual reconciliation, conflict-free data types (CRDTs).

Multi-primary is powerful but complicated. Use only when you have a clear reason (typically global write latency).

Leaderless replication

No primary at all. Clients write to multiple replicas; clients read from multiple replicas; clients reconcile differences.

The classic example is Amazon's Dynamo paper, which inspired Cassandra and DynamoDB. The system uses quorum reads and writes: write to W nodes, read from R nodes, where W + R > N (total replicas) to guarantee consistency.

   N = 5 replicas
   W = 3 (write to 3)
   R = 3 (read from 3)
   W + R = 6 > N = 5  → strong consistency on individual keys

If you also need fast operations: W=1, R=1, W+R=2, no consistency guarantees. Tunable per query, depending on the workload.

Cassandra and DynamoDB use this model. Highly available; complex.

4.3 Sharding (Partitioning)

When one primary database can't handle the write load, you split the data across multiple databases. Each handles a subset.

                        ┌───────────────┐
                        │  Router /     │
                        │  Coordinator  │
                        └───────────────┘
                       /        |        \
                      /         |         \
            ┌────────┐    ┌────────┐    ┌────────┐
            │Shard A │    │Shard B │    │Shard C │
            │id 0-99 │    │id 100- │    │id 200- │
            └────────┘    └────────┘    └────────┘

The application (or a routing layer in front) decides which shard each query goes to.

Sharding multiplies write capacity (each shard handles its slice). It's the only way to scale writes horizontally. It also introduces several painful problems.

Choosing a shard key

The single most important decision in sharding. The shard key is the value used to decide which shard data goes to. Common choices: user ID, tenant ID, geographic region.

What makes a good shard key?

  1. Even distribution. If 99% of traffic comes from 1% of users (a "hot tenant"), sharding by user ID doesn't help — one shard handles everything.
  2. Co-location of related data. Queries that need data together should land on the same shard. Otherwise you do cross-shard joins (slow, complex).
  3. Doesn't change. Re-sharding when keys change is hard.

Bad shard keys: - Timestamp — all writes go to the "latest" shard. Hot shard problem. - Auto-incrementing ID — similar problem; everyone hits the latest. - Random UUID — even distribution but kills locality.

Good shard keys (depending on workload): - User ID hash — even distribution; user data stays together. - Tenant ID — for B2B SaaS, all of one company's data is on one shard. - Geographic region — data lives near the users.

Sharding strategies

Range-based sharding. Each shard owns a range. Users 0–999K on shard A, 1M-2M on shard B, etc.

Pros: simple, supports range queries (find all users between X and Y). Cons: hot ranges if data has skew.

Hash-based sharding. Hash the shard key, mod by shard count, that's your shard.

Pros: even distribution. Cons: range queries become impossible (consecutive IDs hash to different shards).

Directory-based sharding. A lookup table maps each key to a shard.

Pros: flexible — you can move keys between shards. Cons: the lookup table becomes a bottleneck.

Consistent hashing. A more sophisticated hash that makes resharding cheap (only a fraction of keys need to move when you add/remove a shard). Used by DynamoDB, Cassandra, Memcached pools.

We'll meet consistent hashing again in the CDN section.

The hot partition problem

Even with a "good" shard key, sometimes one shard becomes the bottleneck. A celebrity user's tweets generate millions of reads. A flash sale concentrates all writes on one product ID's shard.

Mitigations: - Replicate hot data. The celebrity's tweets get cached widely; the flash sale product's inventory is split into "sub-buckets." - Cache aggressively. Most hot reads should never reach the database. - Re-shard. Move the hot key to a dedicated shard. Painful but sometimes necessary.

Cross-shard queries

A query that needs data from multiple shards is expensive. The system has to query each shard, then combine results.

"List all users in Brazil with > 100 followers"
   → query every shard for "Brazil users with > 100 followers"
   → combine results, sort, paginate

This is called a scatter-gather query. Slow, often impossible at scale.

The fix is design-time, not run-time: shard your data so the queries you need don't cross shards. If you need "all Brazilian users" frequently, shard by country.

If you genuinely need cross-shard queries, options include: - A read-only data warehouse (Snowflake, BigQuery) that ingests from all shards and serves analytics. - A search index (Elasticsearch) that has all the data flattened for query. - Tolerating slow queries with explicit scatter-gather logic.

Re-sharding

Eventually you outgrow your shard count and need more shards. This is the nightmare operation: moving data between shards while keeping the system online.

Approaches: - Double-write during migration. Write to old and new shards, gradually shift reads. - Consistent hashing. Originally designed to make resharding cheap; only fractional data needs to move. - Pre-sharding. Start with many "virtual" shards (1024 of them), placed on few physical shards. To scale, redistribute virtual shards across more physical shards. Avoids redoing application logic.

Engineering blogs from Pinterest, Instagram, Stripe, and Shopify document their resharding journeys. Each one is a multi-quarter project. Plan for it once; avoid it twice.

4.4 Indexes

Indexes are the simplest, biggest performance win in databases. They're how a query that would take 30 seconds takes 30 milliseconds. Every system design discussion of databases assumes you know indexes — so let's make sure.

What an index is

A separate data structure (usually a B-tree, sometimes a hash table) that maps a column's values to row locations. The database can find rows by index lookup instead of scanning the whole table.

Without an index, SELECT * FROM users WHERE email = 'x@y.com' scans every row. With an index on email, it's a single lookup.

Trade-offs

  • Reads: indexes make queries on the indexed column much faster.
  • Writes: every write has to update the indexes too. Each index costs you some write performance.
  • Storage: each index uses disk space.
  • Maintenance: indexes need to be rebuilt occasionally.

Don't index everything. Index columns you query on, especially those used in WHERE, JOIN, ORDER BY clauses.

Composite indexes

An index on (country, last_name, first_name) can serve queries that filter on: - country alone - country + last_name - country + last_name + first_name

But NOT queries that filter on last_name alone — the index is ordered by country first.

Composite index ordering is an art. Put the most selective (highest cardinality) columns first, usually. But put the columns that come up in equality filters before those used for ranges.

Covering indexes

If an index contains all the columns a query needs, the database can answer the query from the index alone — no need to read the actual row data. This is a covering index.

-- Index on (user_id, created_at, status)
SELECT status FROM orders WHERE user_id = 42 ORDER BY created_at DESC;
-- Covering index: all needed columns are in the index

Huge speedup for read-heavy workloads.

Index types beyond B-tree

  • Hash indexes — O(1) equality lookups, but useless for range queries. Used in memory-resident databases.
  • Full-text indexes — for searching free-form text. PostgreSQL has built-in; Elasticsearch is dedicated.
  • Geospatial indexes — for "find all points within X km of here." R-tree variants.
  • Bitmap indexes — for low-cardinality columns (gender, status). Great for analytics, slow for OLTP writes.

A real interview almost never goes deep on index internals. You should know what indexes are, why they matter, that they trade write speed for read speed, and that picking columns is a craft.

4.5 SQL vs NoSQL — The Big Picture

This debate has cost many hours of engineering meetings. Here's how to think about it.

What SQL gives you

  • ACID transactions. Atomicity, Consistency, Isolation, Durability. Bank-grade reliability for multi-row updates.
  • Strong schemas. The shape of your data is enforced. Schema migrations are explicit.
  • Powerful queries. Joins across tables, complex WHERE clauses, aggregations.
  • A 50-year-old standard. SQL skills are everywhere. Tooling is mature.
  • Strong consistency by default. When you read, you see the latest write.

What "NoSQL" means

NoSQL is a marketing term covering several different kinds of stores:

  • Document stores (MongoDB, Couchbase) — JSON-like documents. Schema is flexible.
  • Key-value stores (Redis, DynamoDB) — fetch a value by key. Very fast.
  • Wide-column stores (Cassandra, HBase) — like a table but designed for huge scale and high availability.
  • Graph databases (Neo4j, Neptune) — optimized for relationship traversal.

Each gives up some SQL features in exchange for some property — usually horizontal scalability, schema flexibility, or low operational complexity.

When SQL is the right choice

  • You need transactions across multiple rows. "Charge this user's credit card AND record the order AND update inventory." SQL is unmatched here.
  • Your data has clear, stable schemas. A users table with predictable columns isn't a NoSQL use case.
  • You need ad-hoc queries. SQL's flexibility for "show me X by Y joined with Z" is hard to replicate.
  • The team knows it well. Don't pick a database the team can't operate.
  • Default choice for small-to-medium systems. Until you have a clear reason to leave SQL, stay with it.

When NoSQL is the right choice

  • Massive scale with simple queries. DynamoDB or Cassandra at petabyte scale.
  • Flexible schemas. Variable-shape data; many fields per record; rapid schema evolution.
  • Specific access patterns. Graph traversal → graph DB. Time series → time-series DB.
  • High write throughput with eventual consistency tolerance. Logging, analytics events, IoT data.
  • Geographic distribution with local writes. Multi-region Dynamo-style setups.

The middle ground: "NewSQL"

A category of databases trying to give you SQL semantics and horizontal scalability: CockroachDB, Spanner, YugabyteDB, TiDB. They achieve this through distributed consensus (Section 5).

Trade-off: more complex internals, sometimes higher latency. But they're catching on for systems that grew past PostgreSQL's vertical limits.

The honest summary

Most internet companies use both: - SQL (PostgreSQL or MySQL) for transactional data. - A document store or key-value store for cache, sessions, settings. - A search engine (Elasticsearch) for search. - A data warehouse for analytics. - Maybe a graph DB or time-series DB for specific features.

Picking the right tool for each workload is the senior move. "Everything in Postgres" works longer than people think. But beyond a certain scale, polyglot persistence is reality.

4.6 Specialized Stores You Should Know

A whirlwind tour. Each deserves its own course; you should at least know they exist.

Elasticsearch / OpenSearch. Full-text search. Real-time indexing of documents; powerful query DSL. The de facto choice for search across e-commerce, logs (with Kibana), and content sites.

Redis. In-memory key-value store. Microsecond latencies. Used for caching, session storage, rate limiting, pub/sub, leaderboards. Probably the most universally-used non-SQL database.

Memcached. Older, simpler in-memory cache. Used to dominate; now mostly replaced by Redis. Worth knowing for legacy systems.

Cassandra. Wide-column, leaderless, horizontally scalable. Designed at Facebook (Inbox Search). High write throughput, eventually consistent. Used by Netflix, Apple, Discord.

MongoDB. The first widely-adopted document store. Schema-flexible, decent at scale. Often the entry-point database for teams that "want NoSQL."

DynamoDB. Amazon's managed key-value/document store. Massive scale, predictable performance, no operational work. Expensive at large scale but operationally simple.

ClickHouse / DuckDB / BigQuery / Snowflake. Columnar analytical databases. Great for "scan a billion rows and compute aggregates." Bad for OLTP. The analytics half of polyglot persistence.

InfluxDB / TimescaleDB. Time-series databases. Optimized for "append-only data with timestamps and queries by time range." Metrics, sensor data, financial ticks.

Neo4j. Graph database. If your queries are like "friends of friends" or "shortest path between users," graph DBs do it 100× faster than SQL.

Pinecone / Weaviate / pgvector. Vector databases. Store embeddings (numeric arrays from ML models), query by similarity. Critical for AI applications (see our AI Engineering course).

You don't need to know each in depth. You should be able to say "for that workload, I'd reach for X" and explain why.

SECTION 5

Distributed Transactions

A single database can give you ACID transactions easily — it sees all the data, holds locks, makes atomic commitments. Once your data lives across multiple databases (sharded or partitioned across services), atomicity stops being free. This section is about what you do then.

This is the most theoretically dense section in Part 1. Take it slowly. The patterns here power every modern distributed system.

5.1 Why Distributed Transactions Are Hard

Imagine a money transfer: subtract $100 from Alice, add $100 to Bob.

If Alice and Bob are in the same database, this is one transaction:

BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 'alice';
UPDATE accounts SET balance = balance + 100 WHERE id = 'bob';
COMMIT;

If either statement fails, both roll back. Money is never created or lost.

Now imagine Alice's account is in the US database and Bob's is in the EU database. There's no shared transaction. You can:

  1. Subtract $100 from Alice. Then the US database crashes before you tell EU about Bob. Alice lost $100; Bob never got it. Money lost.
  2. Subtract $100 from Alice. Send "add $100" message to EU. EU processes it twice (because of a retry). Money created.
  3. Both succeed. Correct.

The problem isn't theoretical. It's the daily reality of any system spanning multiple data stores.

Why is it hard? Three reasons:

  • No global lock. You can't make both databases pause at the same instant.
  • Networks fail. Messages get lost, duplicated, reordered, delayed.
  • Machines crash. A server can die mid-operation, leaving things in unknown states.

The solutions either accept eventual consistency, or build coordination protocols on top.

5.2 ACID, Revisited

ACID — the gold standard of single-database transactions:

  • Atomicity — all-or-nothing. Either the whole transaction commits, or none of it does.
  • Consistency — the database moves from one valid state to another. Constraints, foreign keys, etc., are preserved.
  • Isolation — concurrent transactions don't interfere visibly with each other.
  • Durability — once committed, the data survives crashes.

The hard ones to provide across databases are A (atomic across machines) and I (isolation across machines).

There are several isolation levels, each weaker than the last:

  • Serializable — strongest. Transactions behave as if executed one at a time.
  • Snapshot isolation — each transaction sees a consistent snapshot of the data.
  • Repeatable read — reads within a transaction are consistent.
  • Read committed — you only see committed data, but two reads of the same row might differ.
  • Read uncommitted — you might see uncommitted data. Used rarely.

These levels have specific anomalies they prevent or allow (dirty reads, non-repeatable reads, phantom reads, write skew). For interviews, knowing that serializable is the strongest and read-committed is the typical default is enough.

5.3 The CAP Theorem (Revisited)

We covered CAP in Section 3, but it's even more relevant here. In the presence of a network partition, you choose between consistency and availability.

For distributed transactions, this means:

  • If you choose consistency (CP) — when a partition happens, transactions fail. Money is never lost or duplicated. Your service has downtime during the partition.
  • If you choose availability (AP) — when a partition happens, transactions succeed locally. They reconcile later. Your service stays up but might temporarily have inconsistent state.

Banks generally choose CP. Social feeds generally choose AP. Pick based on what's more painful: occasional downtime, or occasional inconsistency.

5.4 Two-Phase Commit (2PC)

The classical distributed transaction protocol. A coordinator orchestrates a transaction across multiple participants.

Phase 1 — Prepare

  1. Coordinator says to all participants: "Prepare to commit this transaction."
  2. Each participant does the work locally but doesn't commit. It writes "I'm ready to commit, my decision is final" to a durable log.
  3. Each participant replies "Yes, ready" or "No, can't do it."

Phase 2 — Commit (or Abort)

  1. If all participants said "Yes," coordinator says: "Commit." Everyone commits.
  2. If anyone said "No" (or didn't reply), coordinator says: "Abort." Everyone rolls back.
       Phase 1 (PREPARE)             Phase 2 (COMMIT)

            ┌─→ Database A          ┌─→ Database A (commits)
Coord ──────┼─→ Database B  ────────┼─→ Database B (commits)
            └─→ Database C          └─→ Database C (commits)

        all reply "ready"        all commit, send ACK

What 2PC gives you

Atomicity across participants. Either everyone commits, or everyone aborts. No partial states.

What 2PC costs you

Blocking. If the coordinator crashes between Phase 1 and Phase 2, the participants are stuck. They've prepared but don't know whether to commit. They hold locks until the coordinator comes back.

Latency. Two round trips for every transaction. In low-latency systems, this is unacceptable.

Coordinator as single point of failure. The whole transaction depends on the coordinator. Replicating the coordinator is itself a hard problem.

Doesn't compose well. If your "transaction" calls another service that has its own 2PC, things get hairy fast.

2PC is used in some specific places (XA transactions in enterprise systems, some database internals) but rarely in modern microservices. The cost is too high.

5.5 Three-Phase Commit (3PC)

A fix for 2PC's blocking problem. Adds a "PreCommit" phase between prepare and commit, so participants can independently decide what to do if the coordinator crashes.

In practice, 3PC has its own correctness problems (it assumes synchronous communication and bounded delays — both of which fail in real networks) and is almost never used. Mentioned here so you can recognize it; don't propose it in interviews.

5.6 The Saga Pattern

Modern distributed systems mostly skip 2PC and use sagas: a sequence of local transactions, where each step has a compensating action that undoes it if a later step fails.

Concrete example. Booking a trip: hotel + flight + car rental, each at a different service.

1. Reserve hotel       (local transaction)
2. Reserve flight      (local transaction)
3. Reserve car         (local transaction)

If car reservation fails:
   - Cancel flight     (compensating transaction)
   - Cancel hotel      (compensating transaction)

Each step in the saga is its own transaction; each has a paired "undo" step. If any step fails, you run the compensating actions for the steps that already happened.

There are two flavors:

Orchestration sagas

A central coordinator (the orchestrator) tells each service what to do.

                ┌──────────────┐
                │ Orchestrator │
                └──────┬───────┘
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
    Reserve         Reserve        Reserve
    Hotel           Flight         Car

Pros: easy to understand, easy to monitor. Cons: orchestrator is a single point of coordination.

Choreography sagas

No coordinator. Each service publishes events. Other services listen and react.

   HotelReserved → (flight service hears) → reserves flight
   FlightReserved → (car service hears)   → reserves car
   CarFailed     → (other services hear)  → start compensating

Pros: loose coupling, no central bottleneck. Cons: hard to understand the overall flow; debugging is painful when events fly everywhere.

Most teams start with orchestration sagas (clearer) and graduate to choreography only when the orchestrator becomes a bottleneck.

Saga caveats

Sagas are not atomic. Between steps, the system is in an intermediate state visible to readers. If you query during a saga, you might see the hotel reserved but not the flight.

Sagas are not isolated. Two concurrent sagas can interleave. Step 1 of saga A, step 1 of saga B, step 2 of A, step 2 of B — the order matters and can produce weird results.

Sagas require idempotent operations. A compensating action might run twice (if a retry happens). Cancelling a hotel reservation twice should be the same as cancelling it once.

These are all manageable, but they're real concerns. Sagas trade strict atomicity for practical scalability.

5.7 Eventual Consistency Patterns

Often the right answer isn't 2PC or sagas — it's accepting that the system is eventually consistent and designing around that.

The outbox pattern

A common problem: your service writes to its database AND publishes a message to a queue, atomically. If the database write succeeds and the queue publish fails (or vice versa), you have inconsistency.

The fix: write the message to an outbox table in the same database, inside the same transaction as the business write.

BEGIN;
  INSERT INTO orders (...);
  INSERT INTO outbox (event_type, payload, ...);
COMMIT;

A separate process polls the outbox table and publishes each row to the queue. Once published, it marks the row as sent (or deletes it).

This gives you exactly-once semantics for "write to database AND publish event" — the local transaction is atomic, and the publisher process can retry indefinitely until the event is delivered.

Used everywhere. Probably the single most important pattern in modern event-driven architectures.

Change Data Capture (CDC)

Instead of having your application write to an outbox, capture the database's own transaction log and turn it into events. Debezium is the famous open-source tool; cloud providers offer hosted versions (DynamoDB Streams, MySQL CDC on RDS).

CDC lets you propagate database changes to other systems (search indexes, caches, data warehouses) without your application code knowing about them.

Idempotency

If an operation can be retried (and in any distributed system, it will be), it must be idempotent — calling it twice has the same effect as calling it once.

How to make operations idempotent:

  • Unique idempotency keys. The client sends a unique key with each request. The server stores keys it has seen. If a key comes back, the server returns the original response without re-running the operation.
  • Natural idempotency. "Set the user's name to X" is naturally idempotent — running it twice has the same effect. "Add 100 to the user's balance" is not — running it twice doubles the addition.
  • Conditional updates. "Set status to SHIPPED only if currently PROCESSING." Subsequent calls see status=SHIPPED and do nothing.

Idempotency is a fundamental concept in distributed systems. Every interview asking about retries, queues, or distributed transactions touches it. Internalize it.

At-least-once vs At-most-once vs Exactly-once

These describe message delivery guarantees:

  • At-most-once: the message is delivered zero or one times. May be lost. Cheapest. Used for non-critical things (impression tracking).
  • At-least-once: the message is delivered at least once, possibly more. Duplicates possible. Most common. The application handles dedup via idempotency.
  • Exactly-once: the message is delivered exactly once. In theory, impossible across machines. In practice, achievable through clever combinations of at-least-once delivery + idempotent processing.

When someone says "exactly-once," they usually mean "at-least-once delivery with idempotent consumers." Pure exactly-once across distributed boundaries is a research problem.

5.8 Distributed Consensus

For some operations, you need actual agreement among distributed nodes — leader election, distributed locks, configuration changes. This is the distributed consensus problem.

Consensus algorithms let a group of nodes agree on a value, even if some of them fail. The famous ones:

Paxos

The original (1989, Lamport). Theoretically elegant. Famously hard to implement correctly. Most production systems don't use raw Paxos; they use variants.

Raft

A 2014 redesign of consensus, explicitly aimed at being understandable. The basic idea:

  1. Nodes elect a leader.
  2. The leader receives all writes.
  3. The leader replicates each write to the followers.
  4. Once a majority has acknowledged, the write is considered committed.
  5. If the leader dies, the followers elect a new leader.
                    Leader
                  /   |   \
                 /    |    \
            Follower Follower Follower

Raft is used by etcd, Consul, TiKV, CockroachDB, and many others. If you need distributed consensus, you almost certainly want Raft.

When you need consensus

You don't need consensus for everything. Most operations in a distributed system can tolerate some inconsistency. You need consensus when:

  • Electing a primary/leader in a replication setup.
  • Distributed locks (only one process can hold the lock at a time).
  • Membership decisions (this node is part of the cluster; that one isn't).
  • Configuration changes that must be coordinated.

Don't implement consensus yourself. Use a battle-tested library or service (etcd, ZooKeeper, Consul).

5.9 Distributed Locks

Often you need: "only one process in the world can do X at a time."

Examples: a cron job that should run once per day across N servers; an inventory operation that shouldn't oversell.

A distributed lock is the primitive. Acquire it; do work; release it. If you crash, the lock is released automatically (via expiration).

Implementing with Redis

The simplest practical lock. SET key value NX PX 30000 — set the key if it doesn't exist (NX), with a 30-second expiration (PX). If the SET succeeds, you hold the lock. To release, delete the key (but only if it's still yours).

# Acquire
acquired = redis.set("lock:inventory", my_id, nx=True, px=30000)
if not acquired:
    raise CouldNotAcquireLock

# Do work
process_inventory()

# Release (with a check that we still own it)
if redis.get("lock:inventory") == my_id:
    redis.delete("lock:inventory")

Subtleties: what if your work takes longer than 30 seconds and the lock expires while you're still working? Another process might acquire it. You'd need to extend the lock periodically.

Antirez (the Redis author) proposed Redlock — a more rigorous algorithm using multiple Redis nodes. Whether Redlock is actually correct is contentious (Martin Kleppmann famously argued it isn't). For most practical purposes, single-Redis locks with sensible expiration are fine.

Implementing with ZooKeeper or etcd

These provide consensus-backed lock primitives. More expensive than Redis, more correct under partition. The right choice for high-stakes locks (financial operations, leader election).

The fundamental warning

Distributed locks are not bulletproof. Network partitions, clock skew, and GC pauses can cause two nodes to think they hold the lock simultaneously. For safety-critical operations, prefer designs that don't require mutual exclusion (idempotency, conditional updates, optimistic concurrency).

5.10 Optimistic vs Pessimistic Concurrency

A related concept worth knowing.

Pessimistic concurrency: assume conflicts will happen. Lock first; then operate. Distributed locks are the pessimistic approach.

Optimistic concurrency: assume conflicts won't happen. Operate freely; detect conflict on commit; retry if needed.

The optimistic pattern: each row has a version number. You read the row (and version). You modify in memory. You commit with a condition: "only if the version is still X." If someone else changed it meanwhile (incrementing the version), your commit fails; you re-read and retry.

-- Read
SELECT name, version FROM users WHERE id = 42;
-- Got name="Alice", version=7

-- Modify in app: new name = "Alice Smith"

-- Commit
UPDATE users SET name = 'Alice Smith', version = 8
WHERE id = 42 AND version = 7;
-- If 0 rows affected, someone else updated; retry.

Optimistic concurrency is the default in most modern systems. It's cheaper (no locks held) and scales better. It only loses when conflicts are frequent — then pessimistic locking might be needed.

5.11 Putting It All Together

A typical modern microservices architecture handles distributed state like this:

  • Within a single service, use the database's transactions (ACID, single-DB).
  • Across services, use sagas (orchestration for clarity, choreography for scale).
  • For "write to DB and publish event", use the outbox pattern.
  • For retries, design every operation to be idempotent.
  • For "only one node at a time", use a distributed lock, but only when necessary.
  • For leader election or strict coordination, use a consensus system (etcd, ZooKeeper).
  • For most reads, accept eventual consistency and design the UX around it.

No system uses 2PC across all of this. The cost is prohibitive. The pieces above give you "good enough" distributed transactions, with explicit trade-offs you understand.

SECTION 6

Common Services for Functional Partitioning

Modern systems aren't one big application — they're a collection of specialized services, each doing one thing well. This section tours the most common ones. You'll meet most of them in every case study.

For each: what it does, why you'd use it, the main options, and the trade-offs.

6.1 Load Balancers

The traffic cop in front of your servers. Routes incoming requests across multiple backend servers.

                 ┌─→ Server 1
   Clients ──→ LB ┼─→ Server 2
                 └─→ Server 3

What it gives you

  • Capacity — combine many servers' capacity into one virtual endpoint.
  • High availability — if one server fails, the LB skips it.
  • Health checks — periodically probe servers; remove unhealthy ones from rotation.
  • SSL termination — handle TLS at the LB, plaintext to backends (one less thing for backends to do).

Layer 4 vs Layer 7

Layer 4 load balancer — works at the TCP/UDP level. Sees IP addresses and ports; doesn't look inside the requests. Very fast.

Layer 7 load balancer — works at the HTTP level. Can route based on URL paths, headers, cookies. Slower than L4 but enables sophisticated routing.

In practice, most cloud LBs are L7 (AWS ALB, GCP HTTPS LB). Use L4 (AWS NLB) when you need raw TCP performance or are routing non-HTTP traffic.

Routing algorithms

  • Round-robin — each new request goes to the next server. Simple, decent.
  • Least connections — route to whichever server currently has the fewest active connections. Better for varying request sizes.
  • Random — surprisingly competitive with round-robin in practice.
  • IP hash / session affinity — same client always goes to the same backend. Useful for stateful sessions, but limits the LB's ability to balance.
  • Power of two choices — randomly pick two servers, choose the less-loaded one. Lightweight; nearly as good as least-connections.

For most workloads, round-robin or least-connections is fine. The algorithm matters less than people think.

Sticky sessions — usually a smell

Some apps store session state in the app server's memory. Then they need "sticky sessions" — the LB sends the same user back to the same server. This works but: - Removes the LB's ability to balance freely. - One server crashing loses all its sessions. - Adds complexity.

Better: store session in a shared store (Redis, JWT cookies). Servers become stateless; LB can freely shuffle. The whole architecture gets simpler.

Common LB choices

  • Cloud-managed: AWS ALB/NLB, GCP Load Balancing, Azure Application Gateway. Pay-per-traffic, fully managed.
  • Self-hosted: NGINX, HAProxy, Envoy. More control; you operate them.
  • At-the-edge: Cloudflare, Fastly. Built into the CDN; very close to users.

6.2 API Gateway

A specialized kind of L7 routing in front of your microservices. Handles cross-cutting concerns so individual services don't have to.

                ┌──→ User Service
   Client ─→ API Gateway ─→ Order Service
                └──→ Inventory Service

What it does

  • Routing — /api/users/* goes to user-service; /api/orders/* goes to order-service.
  • Authentication — checks tokens; rejects unauthenticated requests before they hit your services.
  • Rate limiting — protects backends from abuse.
  • Logging and metrics — central observability for all API traffic.
  • Caching — caches common responses.
  • Transformation — converts between protocols (REST → gRPC) or response shapes.
  • API versioning — /v1/... and /v2/... route to different backends.

Without an API gateway, every service implements auth, rate limiting, and logging itself — inconsistently. With one, you have a single place to enforce policies.

Common options

  • AWS API Gateway — fully managed, integrates with Lambda.
  • Kong — open source, popular.
  • Apigee (Google) — enterprise-focused.
  • Envoy / Istio — used as part of a service mesh.
  • Custom — many large companies build their own (Netflix's Zuul → Spring Cloud Gateway).

API gateways are a common point of debate. Done well, they centralize necessary work. Done badly, they become bottlenecks that every team must coordinate with. The decision depends on your team size and complexity.

6.3 Reverse Proxies

A load balancer is a kind of reverse proxy. So is an API gateway. The general term reverse proxy means: a server that accepts client requests on behalf of backend servers, and forwards them.

You'd typically have one for: - TLS termination (decrypt at the proxy; plaintext internally). - Caching common responses. - Hiding the backend topology from clients. - Acting as a single point of policy enforcement.

The line between "load balancer," "API gateway," and "reverse proxy" is fuzzy — they're all reverse proxies with different emphases. NGINX is famously all three.

6.4 Message Queues

A queue lets one service hand work to another asynchronously. Critical for decoupling, resilience, and absorbing traffic spikes.

   Producer ─→ Queue ─→ Consumer(s)

Why queues matter

Without a queue, when service A calls service B: - A waits for B to finish. - If B is slow, A is slow. - If B is down, A's request fails. - A spike in A's traffic immediately becomes a spike in B's load.

With a queue: - A pushes work to the queue and returns immediately. - B consumes at its own pace. - If B is down, work piles up in the queue; B catches up when it recovers. - A spike fills the queue temporarily; B doesn't get overwhelmed.

Common patterns

Background jobs. "Send a confirmation email" is queued; a worker sends it asynchronously. The user's HTTP request returns instantly.

Event-driven architecture. Services publish events; other services subscribe. "OrderPlaced" event → inventory decrements, shipping prepares, analytics records. Each service is decoupled.

Fan-out. One event needs many consumers. Each consumer subscribes independently.

Buffering. Smooth out traffic spikes. Producers push at peak rate; consumers process at sustained rate.

Message queue vs message broker vs streaming platform

The terms overlap but have distinct emphases.

Traditional message queues (RabbitMQ, ActiveMQ, AWS SQS): - Per-message semantics — each message is delivered to one consumer. - Messages are deleted after consumption. - Designed for work queues.

Streaming platforms (Apache Kafka, AWS Kinesis, Apache Pulsar): - Append-only log of messages. - Multiple consumers each read at their own pace. - Messages are retained for a configurable time (hours, days, forever). - Designed for event streaming.

The difference matters. Kafka isn't a drop-in replacement for RabbitMQ. Kafka is better at: - High throughput (millions of messages/sec). - Long retention (replay events from a week ago). - Multiple independent consumers reading the same stream.

RabbitMQ/SQS is better at: - Simple work queues. - Lower latency. - Easier to operate.

Delivery semantics

We covered this in Section 5: at-most-once, at-least-once, "exactly-once." Most modern queues default to at-least-once; idempotent consumers handle the rest.

Common pitfalls

  • Message ordering. Some queues guarantee order per producer; some don't. If order matters, check.
  • Poison messages. A message that crashes the consumer every time it's processed. Without a dead-letter queue, this loops forever. Always have a dead-letter queue.
  • Queue backlog. If consumers can't keep up, the queue grows. Monitor lag and have alerts.
  • Schema evolution. Producers and consumers evolve independently. Either use a schema registry (Confluent Schema Registry) or design messages for backward compatibility.

6.5 Caches

We touched on caches in Section 1. Let's go deeper.

Cache topologies

Local cache — each app server has its own in-process cache. Fastest (zero network hops). Doesn't share state — each server has its own copy.

Distributed cache — a separate cache cluster (Redis, Memcached). All app servers share it. One network hop. Used in most architectures.

CDN / edge cache — at the network edge, close to users. We cover CDNs in their own section.

Cache strategies

Cache-aside (lazy loading). Application asks cache; on miss, queries DB, stores in cache.

app: get user 42 from cache  → miss
app: get user 42 from db     → returns user
app: put user 42 in cache    → done

Most common pattern. Simple. Slight latency on misses.

Write-through. Application writes to both cache and DB synchronously.

app: update user 42  → write to cache AND db (both must succeed)

Cache always up-to-date with DB. Slower writes (write to two places). Useful when you read what you write.

Write-behind (write-back). Application writes to cache; cache asynchronously persists to DB.

app: update user 42  → write to cache (fast)
cache: persists to db (later, in background)

Fastest writes. Risk: cache crashes before persistence → data loss.

Refresh-ahead. Cache proactively refreshes hot keys before they expire. Used when hot data is predictable.

Cache eviction policies

The cache fills up. Some entries must be evicted. Common policies: - LRU (Least Recently Used) — evict the entry that hasn't been read for the longest. The default for most caches. - LFU (Least Frequently Used) — evict the entry that's been read the fewest times. Good when "popularity" matters more than recency. - TTL (Time-To-Live) — entries expire after a fixed duration. Simple, predictable. - Random — surprisingly effective for some workloads.

Cache failure modes

Cache stampede (thundering herd). A hot key expires. A thousand simultaneous requests all miss, all hit the DB at once, all reload the cache. The DB falls over.

Fixes: - Lock-and-fetch — only one request reloads; others wait. - Probabilistic early expiration — refresh slightly before the TTL. - Stale-while-revalidate — serve stale data while refreshing in the background.

Cache inconsistency. DB is updated; cache still has the old value. Most painful in distributed caches.

Fixes: - Write-through (mentioned above) keeps cache in sync. - Cache invalidation on write — write to DB, then explicitly delete from cache. - Short TTLs — accept some inconsistency; it'll resolve in seconds. - Event-based invalidation — the DB publishes change events; the cache subscribes.

There's a famous quote (Phil Karlton): "There are only two hard things in Computer Science: cache invalidation and naming things." Cache invalidation is genuinely hard. Design with it in mind.

Common cache choices

  • Redis — in-memory key-value with rich data types (strings, lists, sets, sorted sets, hashes, streams). The default.
  • Memcached — simpler, just key-value. Faster for simple workloads.
  • Local in-process caches — Caffeine (Java), lru-cache (Node), functools.lru_cache (Python).
  • CDN edge caches — for static and semi-static content.

6.6 Search Engines

Relational databases are great at "find rows matching exact criteria." They're bad at: - Full-text search ("any post containing the phrase 'distributed systems'"). - Fuzzy matching ("did you mean 'system' instead of 'systme'?"). - Relevance scoring ("rank results by how well they match"). - Faceted search ("books, filter by genre, then author, then year").

For these, you use a dedicated search engine. Elasticsearch (and its open-source fork, OpenSearch) is the dominant choice. Lucene-based; distributed; mature.

How it works (lightly)

A search engine maintains an inverted index: for each word, a list of documents containing it.

"the" → doc1, doc3, doc5, doc7, doc8, ...
"system" → doc3, doc7, doc12
"design" → doc7, doc15, doc22

Query "system design" → intersection of "system" and "design" → doc7 (probably ranked first based on relevance).

This is very different from a B-tree index on a SQL column. The inverted index is optimized for "find documents matching a set of terms."

Indexing pipeline

Documents in the operational DB → indexing pipeline → search index.

The pipeline can be: - Synchronous — write to DB and search index in the same transaction (rare; coupled). - Async via queue — DB publishes events; indexer consumes and updates search index. Most common. - Change Data Capture — search engine listens to the DB's transaction log directly.

The search index is always slightly stale (eventually consistent). That's fine for almost all search use cases.

When you need a search engine

  • User-facing search.
  • Log search (the ELK stack: Elasticsearch + Logstash + Kibana).
  • Analytics on free-form text.
  • Faceted product catalogs.

When you don't: - Exact-match lookups (a SQL index is faster and simpler). - Very small datasets (overkill).

6.7 Object Storage

For unstructured data — images, videos, PDFs, backups, anything large — you don't put it in a database. You put it in object storage.

   "my-bucket/photos/alice/2026-06-01.jpg" → binary blob

The data is addressed by a key (a file path-like string). You can read it, write it, delete it, set metadata. You can't query the contents.

Why it's a separate category

  • Storage is cheap. Object storage is 10-100× cheaper per GB than database storage.
  • Scale is unlimited. Practically. AWS S3 stores trillions of objects.
  • High durability. Typical guarantee: "eleven nines" (99.999999999%) of durability. Translated: you'd lose one object every few hundred million years.
  • Direct serving. Clients can read objects directly via signed URLs — no need to proxy through your servers.

The major options

  • Amazon S3 — the original. The industry default.
  • Google Cloud Storage — equivalent on GCP.
  • Azure Blob Storage — equivalent on Azure.
  • MinIO — self-hosted, S3-compatible.
  • Cloudflare R2 — S3-compatible with no egress fees.

S3's API has become a de-facto standard; most tools target it.

Storage classes

S3 (and equivalents) have multiple storage classes with different price/access trade-offs: - Standard — instant access, redundant across availability zones. - Infrequent Access — cheaper storage; higher per-request cost. - Glacier / Archive — much cheaper; retrieval takes hours.

For old data you rarely access, archive tiers cut costs dramatically. You can configure lifecycle policies to move objects automatically.

Patterns

  • Direct upload from client. Client gets a signed URL from your backend, uploads directly to S3. Your servers never see the bytes. Critical for video uploads.
  • Direct download via CDN. Object storage + CDN = your media serving stack.
  • Database stores the URL, not the data. Bytes live in S3; the row in Postgres stores media_url = "s3://my-bucket/path".

This pattern is so universal it's barely worth discussing — but state it explicitly in interviews. It shows you know the right tool for the job.

6.8 Service Discovery

In a microservices architecture, services need to find each other. How does the order-service know where the inventory-service lives?

Static configuration

Each service is configured with the hostnames of its dependencies. Simple, but breaks down when services scale, move, or fail over.

Service registry

A central directory. Services register themselves on startup. Clients query the registry to find services.

                    ┌─────────────────┐
   inventory:8080 ─→│                 │
   payment:8080   ─→│  Service        │←── "where is inventory?"
   shipping:8080  ─→│  Registry       │
                    │                 │── "inventory:8080"
                    └─────────────────┘

Common implementations: - Consul (HashiCorp) - etcd (Kubernetes uses this) - Eureka (Netflix)

DNS-based service discovery

DNS itself is a service registry. Modern container platforms (Kubernetes, ECS) use DNS to expose services. A service named "inventory" gets a DNS record; clients just call "inventory" by name and DNS resolves to the current pod IPs.

This is the modern default. You almost never set up Consul-style service discovery by hand anymore.

6.9 Configuration and Secrets

Apps need configuration: database connection strings, API keys, feature flags. Where does this live?

Bad places: hardcoded in source, committed to git, stored in plaintext on disk.

Better: - Environment variables — fine for simple cases. - Config services — Consul, etcd, Spring Cloud Config. Centralized, versioned. - Secret managers — AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager. Encrypted, audited, with rotation support. - Feature flag services — LaunchDarkly, Unleash, Flagsmith. For things that change at runtime.

In an interview, briefly mentioning "we'd use AWS Secrets Manager for credentials and a feature flag service for runtime toggles" signals operational maturity.

6.10 Observability Stack

We covered this conceptually in Section 3. Practically, in 2026, observability looks like:

  • Metrics — Prometheus (collection) + Grafana (visualization). Or commercial: Datadog, New Relic.
  • Logs — Loki, Elasticsearch (ELK), or commercial (Datadog Logs, Splunk).
  • Traces — OpenTelemetry (the emerging standard), Jaeger, Tempo. Or Datadog APM.

The trend toward OpenTelemetry is real and worth knowing. It's a vendor-neutral standard that combines metrics, logs, and traces. New observability deployments should default to OTel.

SECTION 7

Case Study: Design a URL Shortener

We start the case studies with the most canonical interview question. Designing a URL shortener (think bit.ly, tinyurl, t.co) is the "hello world" of system design. It seems trivial but touches every important concept: ID generation, sharding decisions, caching, hot-key problems, and read-vs-write optimization.

Every section from here on follows the same five-phase structure from Section 2: Requirements → Estimation → High-level Design → Deep Dives → Bottlenecks.

7.1 Requirements

Functional

  • Create a short URL from a long URL. e.g., https://example.com/very/long/path/to/article → https://sh.rt/abc123.
  • Redirect: visiting the short URL sends the user (HTTP 301 or 302) to the original.
  • Optional v2: custom slugs (sh.rt/my-thing), expiration dates, click analytics.

Non-functional

  • Latency: redirects must feel instant. P99 under 100ms.
  • Availability: 99.99% (53 minutes downtime/year). A dead short URL breaks a link in a tweet permanently — high impact.
  • Durability: very high. Once a URL is created, losing it would break every existing link to it.
  • Consistency: read-your-own-writes for the creator. Eventual is fine for others — a 1-second propagation delay is invisible.
  • Read-heavy: redirects vastly outnumber creates.

Constraints

  • Short URLs should be short (7-8 characters is the sweet spot).
  • Short URLs should be non-guessable (don't expose user IDs or sequential counters in plain).
  • The system should be hard to abuse (rate limiting, phishing prevention) — we'll mention but not dwell.

7.2 Estimation

Numbers we need: write QPS, read QPS, storage, bandwidth.

Writes: - Assume 100 million new URLs/month. - 100M / (30 × 86,400 sec) ≈ 40 writes/sec average. - Peak might be 10×: 400 writes/sec peak.

Reads: - Assume each URL gets read 100× on average over its lifetime. - 100M creates/month × 100 reads/URL = 10 billion reads/month. - 10B / (30 × 86,400) ≈ 4,000 reads/sec average. - Peak 10×: 40,000 reads/sec peak.

Read-to-write ratio: ~100:1. Heavily read-skewed. The architecture should optimize the read path.

Storage: - Each entry: short_code (7 chars) + long URL (avg 100 chars) + metadata (created_at, user_id, etc., ~50 bytes) ≈ 200 bytes per row. - 100M URLs/month × 200B = 20 GB/month = ~250 GB/year. Easily fits in one machine. - After 10 years: ~2.5 TB. Still manageable on a single beefy node or a couple of shards.

Bandwidth: - Redirects are ~500 bytes (small HTTP response). - 4,000 reads/sec × 500B = 2 MB/sec average. Trivial.

Conclusion from numbers: - We don't need to shard the database from day one. A single Postgres can handle 400 writes/sec easily. - Reads at 40,000/sec peak ARE significant. They need to hit caches, not the DB directly. - Storage growth is slow.

This shapes our architecture: simple write path, heavily-cached read path, deferred sharding.

Request flows:

Create URL (POST /shorten): 1. Client → LB → App server. 2. App generates a unique short code. 3. App writes {short_code, long_url, ...} to DB primary. 4. App returns the short URL to the client.

The CDN sits in front to cache redirects for the most popular URLs at the edge — for a tweet that goes viral, the CDN absorbs most of the traffic before it hits our backend at all.

7.4 Deep Dive — Generating Short Codes

This is the most interesting subproblem. How do we generate unique, short, non-guessable codes?

Option A: Hash the URL

Take MD5 or SHA-256 of the long URL, encode the first 7 chars in base62.

import hashlib, base64
def shorten(url):
    h = hashlib.sha256(url.encode()).digest()
    return base64.b64encode(h)[:7].decode().replace('/', '_').replace('+', '-')

Pros: - Deterministic: same URL → same short code (saves storage if we dedup). - Stateless: any server can generate.

Cons: - Collisions. SHA-256 truncated to 42 bits gives ~4 trillion possible codes, but the birthday paradox means collisions appear at ~2M URLs. With 100M URLs/month, collisions are guaranteed. - Predictability. If two users shorten the same URL, they get the same code — privacy leak. - Collision handling requires DB lookups on every create, defeating the "stateless" benefit.

Collision handling means checking if the code is already taken, and if so, appending bits and retrying. Hash-based generation isn't as clean as it sounds.

Option B: Counter + base62 encoding

Assign each URL an auto-incrementing integer ID. Encode the ID in base62 (0-9, a-z, A-Z = 62 characters).

ALPHABET = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"
def encode_base62(n):
    if n == 0: return "0"
    result = []
    while n > 0:
        result.append(ALPHABET[n % 62])
        n //= 62
    return ''.join(reversed(result))

7 characters in base62 = 62⁷ ≈ 3.5 trillion possible codes. Plenty.

Pros: - No collisions (counter is unique). - Compact. - Fast.

Cons: - Predictability. Counter values 1, 2, 3, 4 give codes "1", "2", "3", "4" — sequential. Easy to guess what other people have shortened (privacy leak, scraping). - Distributed counter is hard. If you have 10 app servers, who increments the counter? You need a single source of truth or pre-allocated ranges.

The predictability is usually solvable by: - Skipping non-monotonically (each server takes a "stripe": server A gets evens, B gets odds). - XOR-ing the counter with a secret to scramble. - Using a longer code so guessing is impractical.

The distributed-counter problem is solved with range allocation: each app server requests a block of 1000 IDs from a central counter service, then doles them out locally. The counter service is hit once per 1000 URLs, not once per URL.

Option C: UUID-based

Generate a random UUID (128 bits), encode it.

Cons: UUIDs are 22+ characters even base62-encoded. Too long for our "short URL" requirement.

You can truncate, but then you're back to collision territory.

The pragmatic choice

Option B (counter + base62) with range allocation and XOR scrambling. It's: - Collision-free. - Compact (7 chars handles 3.5 trillion URLs). - Easy to scale (range allocation). - Non-guessable enough with light scrambling.

In an interview, this is the answer to lead with. Acknowledge the alternatives and explain why you picked it.

A clever trick — encode user ID into the short code

Sometimes the short code includes the user's shard ID at a known position. Then routing reads is trivial — "code starts with shard prefix X → query shard X." Trades a tiny bit of code length for huge simplicity.

7.5 Deep Dive — Data Model & Schema

Single primary table:

CREATE TABLE short_urls (
    short_code   VARCHAR(10) PRIMARY KEY,
    long_url     TEXT NOT NULL,
    user_id      BIGINT,
    created_at   TIMESTAMP NOT NULL DEFAULT now(),
    expires_at   TIMESTAMP,
    click_count  BIGINT DEFAULT 0
);

CREATE INDEX idx_user_id ON short_urls(user_id);
CREATE INDEX idx_created_at ON short_urls(created_at);

short_code is the primary key — the most common query (SELECT long_url WHERE short_code = ?) is automatic.

For the auto-incrementing counter, a separate table:

CREATE TABLE counter (
    next_id BIGINT NOT NULL
);

App servers atomically increment this in batches (claim a range of 1000).

Why not NoSQL?

  • DynamoDB or Redis would work fine.
  • Key-value access is exactly what we have.
  • Many real URL shorteners use Redis as the primary store, not just a cache.

But Postgres is simpler operationally, supports analytics queries (find URLs by user, count by day), and handles our scale easily. Default to SQL until you have a reason not to.

7.6 Deep Dive — Caching Strategy

Reads are 100× writes. The cache is the most important piece of the architecture.

What to cache: every short code we read, with a moderate TTL (1 hour to 1 day).

Strategy: cache-aside. On read, check cache; on miss, read DB and populate cache.

Capacity calculation: at any time, the "active" set of URLs being clicked is much smaller than the total set. Most clicks concentrate on the popular ones. If our top 10M most-clicked URLs handle 90% of traffic, and each entry is ~200 bytes, that's 2 GB of cache — fits in a single Redis node.

TTL choice: - Shorter TTL = more cache misses, but cache stays small. Good for cold data. - Longer TTL = fewer misses, more memory. Good for hot data. - The right answer depends on the access pattern.

A clever trick: TTL based on popularity. If something's been hit 100 times in the last hour, give it a long TTL. If it was just created and may never be hit again, give it a short TTL.

Cache stampede protection: if a viral URL has its cache entry expire and a million requests hit at once, the DB falls over. Use: - Lock-and-fetch: only one request reloads; others wait briefly. - Probabilistic early refresh: if TTL is at 80% expired, the request triggers a background refresh.

These pay off when a URL goes viral.

7.7 Deep Dive — Read Path Optimization

For maximum throughput, the read path can be optimized further:

1. CDN at the edge. Configure the CDN to cache redirect responses. The top 1% of URLs handle 50%+ of clicks; serving them from edge means those clicks never hit your backend.

This requires returning HTTP 301 (permanent) with appropriate cache headers, allowing the CDN (and browser) to cache the redirect. But beware: a 301 also gets cached by the user's browser, so they won't hit your server again — meaning your click counter undercounts. If accurate analytics matter, use HTTP 302 (temporary) so clicks hit the server every time.

This is a classic trade-off: cacheability vs analytics accuracy. Pick based on requirements.

2. Read replicas. All DB reads go to replicas. The primary is freed up for writes.

3. Specific replica routing. If we use a hashing strategy, we can route specific short codes to specific replicas — improves cache locality within the DB.

7.8 Deep Dive — Write Path

Writes are 400/sec peak — trivial for any modern DB. We don't need to optimize hard. But to keep the system clean:

1. Pre-allocate ID ranges. App servers request blocks of 1000 IDs at a time. The central counter service is hit once per 1000 creates, not once per create. Eliminates write contention.

2. Asynchronous analytics. Don't write click_count incrementally on every read (that would make reads cost a DB write). Instead, queue click events and aggregate them in batches.

3. Idempotency. A user creating the same URL twice in a second should get the same short code (or two different codes, depending on requirements). If "same long URL = same short code," check the DB first. If not, just create a new one.

7.9 Bottleneck Analysis

Where would this design break at 10× scale (1B reads/sec average, 4K writes/sec)?

Database writes (4K/sec): still fine for a single Postgres. Add a write-side read replica for hot reads of recently-created URLs.

Database reads: 1B/sec is way too much for the DB. But the cache layer absorbs ~95%, and the CDN absorbs another large chunk. Maybe only 50K-100K reads/sec actually hit the DB. Fine across replicas.

Cache: with 10× traffic, the cache absorbs 10× more requests. A single Redis can do 200K+ ops/sec, so still single-node. At 100× scale, shard Redis across multiple nodes (consistent hashing).

Counter service: with range allocation, hit once per 1000 IDs. Even 4K writes/sec = 4 counter requests/sec. Trivial.

Storage: 250 GB/year × 10 = 2.5 TB/year. Eventually we'd shard the URL table by short_code prefix.

Hot keys (a single viral URL): the cache layer becomes the bottleneck if 90% of traffic hits one entry. Fix: multi-level cache — local in-app cache for the top 100 entries (zero network hops), then Redis. Or replicate hot keys to multiple Redis nodes.

7.10 Discussion: Other Considerations

A few things to mention in an interview to signal completeness:

  • Custom slugs: users want sh.rt/my-promo instead of sh.rt/abc123. The slug becomes a chosen short code, with collision check against existing codes.
  • Expiration: a background job sweeps expired URLs. Alternatively, mark them inactive in the DB but keep the row for historical purposes.
  • Analytics: clicks streamed to a data pipeline (Kafka → data warehouse). Users see dashboards.
  • Abuse / phishing: integrate with services like Google Safe Browsing. Block URLs that point to known malicious sites. Rate-limit per IP and per user to prevent bulk creation.
  • API rate limiting: applies even more here than to most apps. Anyone shortening URLs at high speed is probably abusing.
  • Migration / data residency: nothing terribly hard given the small dataset.

SECTION 8

Case Study: Design a Rate Limiter

A rate limiter restricts how often a client can do something. Hit our API more than 100 times per minute and you get throttled. They're everywhere: every public API, every login form, every "send password reset" button.

This case study teaches token buckets, sliding windows, and Redis as a coordination primitive.

8.1 Requirements

Functional

  • Limit a client to N requests per time window. e.g., 100 requests per minute per user.
  • Return HTTP 429 ("Too Many Requests") when over the limit, with a Retry-After header.
  • Support multiple rules (per user, per IP, per API key, per endpoint — usually combinations).

Non-functional

  • Low latency: the rate limit check runs on every request. Sub-millisecond budget.
  • Distributed: app servers are behind a load balancer; the limiter must coordinate across them.
  • Available: if the limiter fails, what should happen? Two options — fail open (allow the request) or fail closed (reject). Most public APIs fail open.
  • Approximately accurate: a few requests over the limit is OK. Strict accuracy isn't worth the cost.

8.2 Estimation

  • Suppose 1 billion API requests/day = ~12K req/sec average, 100K peak.
  • Every request triggers at least one limit check.
  • Each check is roughly: 1 Redis op = 0.5ms latency. At 100K req/sec, Redis handles it.
  • State per limited entity: ~50 bytes. For 10M active users: 500MB — fits in one Redis node.

8.3 The Algorithms

Four classic approaches.

Fixed window

Count requests in fixed time windows (e.g., per minute, aligned to clock minutes). Reset to 0 at window boundaries.

12:00:00 - 12:00:59  →  count = 47
12:01:00 - 12:01:59  →  count = 0 (reset)

Implementation (Redis):

INCR key:{user}:{minute}
EXPIRE key:{user}:{minute} 60
if value > 100: reject

Pros: dead simple, one Redis op per request. Cons: boundary bursts — a user can send 100 requests at 12:00:59 and another 100 at 12:01:00, for 200 in two seconds.

Sliding window log

Store the exact timestamp of every request. On each new request: 1. Remove timestamps older than (now - window). 2. Count remaining. 3. If under limit, allow and add new timestamp.

Implementation with Redis sorted sets:

ZREMRANGEBYSCORE key 0 (now - 60_000)   # drop old
ZCARD key                               # count
ZADD key now now                        # add this request

Pros: precisely accurate. Cons: memory cost — must store every timestamp. At 100 req/min per user × 10M users, that's 1B timestamps in memory.

Sliding window counter

A hybrid: keep counters for the current and previous fixed windows, and weight the previous count by how much of the window remains.

At 12:01:30:
  count_current_minute (12:01) = 30
  count_previous_minute (12:00) = 60

  effective = current + previous * (1 - elapsed_in_current / 60)
            = 30 + 60 * (1 - 30/60)
            = 30 + 30 = 60

Pros: approximates sliding window with much less memory (just 2 counters per user). Cons: slight inaccuracy (assumes uniform distribution within the previous window).

This is the sweet spot for most real systems. Cloudflare uses something like this.

Token bucket

A bucket has capacity N tokens. Tokens refill at rate R per second. Each request consumes one token. If the bucket is empty, reject.

Bucket capacity: 100 tokens
Refill rate: 10 tokens/sec

If user sends 100 requests instantly → drains bucket → next request rejected
After 1 second → 10 tokens added → 10 more requests allowed

Pros: allows bursts up to bucket capacity while maintaining average rate. Reflects real user behavior. Cons: slightly more complex to implement atomically.

Token bucket is the algorithm of choice for AWS, Stripe, GitHub. Most user-facing rate limits.

8.4 High-Level Design

                              ┌──────────────┐
                              │              │
   Request ──→ API Gateway ──→│ Rate Limiter │──→ App Servers
                              │              │
                              └──────┬───────┘
                                     │
                                     ▼
                              ┌──────────────┐
                              │  Redis       │
                              │  (cluster)   │
                              └──────────────┘

Three placements possible: - In the API gateway (Kong, Envoy, Cloudflare Workers): nearest the edge; rejects bad traffic earliest. Best default. - As a sidecar/middleware in each app: simple but each app re-implements logic. - As a dedicated service that other services call: clean but adds a network hop.

Modern systems use the gateway approach. Apps just need to know they MIGHT see 429s.

8.5 Distributed Rate Limiting

If we have 10 app servers and a user's limit is 100/minute, each server can't independently allow 100. They must share state.

The shared state lives in Redis. Every check is a Redis op. With pipelining and Lua scripts, a single Redis can handle 100K+ req/sec.

Atomic check-and-increment (Lua script):

local current = redis.call('GET', KEYS[1]) or 0
if tonumber(current) >= tonumber(ARGV[1]) then
    return 0  -- rejected
end
redis.call('INCR', KEYS[1])
redis.call('EXPIRE', KEYS[1], 60)
return 1  -- allowed

Lua runs atomically in Redis — no race condition between check and increment.

8.6 Handling Redis Failure

What if Redis goes down? Options:

  • Fail open: allow all traffic. Public APIs usually pick this — better to over-serve than to outage.
  • Fail closed: reject all traffic. Picked for security-critical operations (login attempts).
  • Local fallback: each app server falls back to its own in-memory limiter when Redis is unreachable. Less accurate but still some protection.

The right answer depends on what you're protecting against.

8.7 Returning 429 With Headers

Standard practice:

HTTP/1.1 429 Too Many Requests
Retry-After: 30
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1717843200

Retry-After tells the client when to retry. X-RateLimit-* are useful headers (not standardized but widely used) so clients can self-throttle proactively.

8.8 Bottleneck Analysis

  • Redis throughput: 100K ops/sec on a single node. At 1M req/sec, need clustering — shard by user_id.
  • Hot users: one user hammering with millions of requests can hotspot a Redis key. Less of a problem because that user gets rate-limited (their own traffic is bounded).
  • Cross-region: each region runs its own Redis. Limits are per-region. Globally consistent rate limiting is much harder (requires cross-region coordination) and usually not worth the cost.

Case Study: Design a Notification Service

A service that sends notifications across multiple channels — push (iOS APNs, Android FCM), SMS (Twilio), email (SendGrid, SES), in-app — at scale. This problem appears in every consumer product. The interesting parts are channel abstraction, retries, user preferences, and traffic bursts.

9.1 Requirements

Functional

  • Send notifications to users via push, SMS, email, in-app, or webhooks.
  • Templating: store templates centrally, fill in per-recipient data.
  • User preferences: respect user opt-outs per channel and category.
  • Scheduled delivery: send at a specific time.
  • Bulk send: "send promo to all users in this segment."

Non-functional

  • Scale: hundreds of millions of notifications per day.
  • Best-effort delivery: not banking; we tolerate occasional drops but should retry hard.
  • Latency: transactional notifications (password reset) within seconds; promotional ones can take minutes.
  • Provider abstraction: must work across changing providers — APNs, FCM, plus we might switch SMS providers.

9.2 Estimation

  • 500M DAU, 5 notifications/user/day = 2.5B notifications/day = ~30K/sec average.
  • Peak (e.g., breaking news, marketing blast): 100× = 3M/sec peak.
  • Average message size: 1 KB. Daily storage: 2.5 TB/day of notification events. Mostly transient.

9.4 Deep Dive — Queue Topology

Why queues? Three reasons: - Smoothing: a marketing blast queues 100M notifications; workers drain at sustainable rate. - Reliability: if a provider is down, messages wait; we don't lose them. - Independent scaling: more SMS traffic? Scale SMS workers without touching push.

Topology: - One topic per (channel × priority): push.high, push.low, email.high, etc. - High-priority topics have more workers; processed first. - Failed messages go to dead-letter queues for human investigation.

9.5 Deep Dive — Template & Preference Management

Templates live in a template service (DB-backed):

{
  "id": "password_reset",
  "channels": ["email", "sms"],
  "subject": "Reset your {{app_name}} password",
  "body": "Click here: {{reset_link}}"
}

User preferences:

{
  "user_id": 42,
  "categories": {
    "marketing": {"email": false, "push": false},
    "transactional": {"email": true, "push": true, "sms": true},
    "social": {"push": true, "email": false}
  }
}

The API checks preferences and skips channels the user opted out of. Templates are versioned; A/B tests use different versions.

9.6 Deep Dive — Provider Abstraction & Retries

Each worker calls an external provider. Common shape:

worker dequeues → render template → call provider API
                                          ↓
                              ┌── success → mark sent
                              └── failure → retry?
                                              ├── transient (5xx, timeout) → retry with backoff
                                              └── permanent (4xx, invalid number) → dead-letter

Exponential backoff (1s, 2s, 4s, 8s...) with jitter to avoid thundering herd. After N retries, dead-letter.

A provider router abstraction: a SMS message comes in; the router picks a provider (Twilio, MessageBird, etc.) based on: - Cost per region. - Provider health (avoid one that's failing). - Carrier-specific routing.

This lets us swap providers without changing application code.

9.7 Deep Dive — Idempotency & Deduplication

A producer might retry sending the same notification ("send password reset"). We don't want to send it 5 times.

Each notification gets an idempotency key from the producer. Before sending: 1. Check if this key has been processed (Redis cache or DB). 2. If yes: skip. 3. If no: send and mark as processed.

The key has a TTL of ~24 hours — long enough to dedupe retries, short enough to not blow up storage.

9.8 Bulk / Campaign Sending

A marketing blast might target 100M users. We can't accept that as a single API call.

The producer specifies a segment (filter): "all users in Brazil, signed up in last 30 days, opted into marketing." A separate campaign service: 1. Reads matching user IDs from the user DB (paginated, in batches). 2. Enqueues notification jobs in chunks. 3. Throttles itself to not overwhelm the queues.

This streaming approach keeps memory bounded.

9.9 Bottlenecks & Failure Modes

  • External provider downtime: APNs is down. We can't send pushes. Queue grows. We need monitoring + fallback (e.g., fall back to SMS for critical messages).
  • Hot queue: a topic for one product line gets a surge. Other topics' SLAs unaffected (separate workers).
  • Cost runaway: SMS is expensive. A bug that sends 100M SMS = $1M bill. Hard quotas per category per day.
  • User complaints: too many notifications drive uninstalls. Implement aggregation ("3 new messages" instead of 3 separate notifications).

Case Study: Design Autocomplete (Typeahead)

When you type "sys" into Google's search box, suggestions appear instantly: "system design," "systemic," "system32." That's typeahead. Same problem powers product search on Amazon, friend search on Facebook, location autocomplete on Maps.

10.1 Requirements

Functional

  • User types a prefix; we return top suggestions.
  • Suggestions ranked by popularity (or other relevance signals).
  • Response within 100ms — autocomplete must feel instant.
  • Suggestions reflect global trends; the source data evolves over time.

Non-functional

  • Scale: 10B searches/day = ~100K queries/sec average, ~1M peak.
  • Latency: P99 < 100ms. Anything slower feels sluggish.
  • Update rate: trending terms appear quickly; daily index update is fine, hourly better.

10.2 Estimation

  • Each user types ~5 characters before clicking a suggestion. Each keystroke triggers a query.
  • 1B searches/day × 5 keystrokes = 5B queries/day = ~60K queries/sec average.
  • Storage: suppose 1M distinct query terms, each with rank info, ~100 bytes = 100 MB. Tiny.

10.3 The Trie Data Structure

The classic approach. A trie (prefix tree) where each node represents a character, and paths from root to nodes spell prefixes.

              root
             /  |  \
            s   a   t
           /|   |   |
          y o   p   o
         /  |   |
        s   p   p
       /        |
      t         l
     /          |
    e           e
   /
  m

At each node, store the top-K completions of that prefix. When user types "sy", walk the trie to that node and return its stored top suggestions.

Pros: super fast lookup (O(prefix length)). Cons: large memory footprint; rebuilding the trie is expensive.

10.4 High-Level Design

                                ┌──────────────────┐
                                │ Search Logs      │
                                │ (Kafka stream)   │
                                └────────┬─────────┘
                                         │
                                         ▼
                                ┌──────────────────┐
                                │ Aggregator       │
                                │ (Spark/Flink)    │
                                │ count terms      │
                                └────────┬─────────┘
                                         │ (hourly batch)
                                         ▼
                                ┌──────────────────┐
                                │ Trie/Index       │
                                │ Builder          │
                                └────────┬─────────┘
                                         │
                                         ▼
   Client ──→ LB ──→ Autocomplete API ──→ Cache (Redis with prebuilt prefix → suggestions)

The hot path: 1. Client sends prefix (each keystroke). 2. API hashes prefix to a Redis key. 3. Redis returns top suggestions (sorted set). 4. API returns to client.

The slow path (every hour): 1. Aggregator reads all search logs from past N days. 2. Counts term frequencies. 3. For each prefix (up to length 5-10), compute top suggestions. 4. Push to Redis.

10.5 Redis Sorted Sets — The Practical Implementation

Instead of an actual trie, use Redis sorted sets keyed by prefix:

prefix:sys      → ZSET { "system design": 8500, "systemic": 1200, "systems": 800, ... }
prefix:syst     → ZSET { "system design": 8500, "systemic": 1200, "systems": 800, ... }
prefix:syste    → ZSET { "system design": 8500, "systemic": 1200, "systems": 800, ... }
prefix:system   → ZSET { "system design": 8500, "system32": 600, ... }

The API does ZREVRANGE prefix:sys 0 9 WITHSCORES — top 10 sorted by score. Sub-millisecond.

Memory cost: for each prefix length 1-10, store top-10 suggestions. With ~1M unique prefixes, ~10MB. Trivial.

10.6 Deep Dive — Ranking

Pure frequency isn't enough. Better signals: - Recency: trending terms (last 24h) bumped up. - Personalization: user's history, location. - Context: current page, app state.

Often a hybrid score: score = log(global_count) + weight * recency_boost + weight * personalization_score.

For personalization, you can't precompute everything — too many users. Compromise: precompute global top-K per prefix; rerank top-K with personalization signals at query time.

10.7 Deep Dive — Updates

Two patterns: - Batch rebuild: every hour, recompute the entire index. Simple, but lag. - Incremental: process search log events in near-real-time, update counters. Fresh, but complex.

Most production systems use a hybrid: incremental for trending terms (real-time), batch rebuild for full re-ranking (daily). The cost-benefit favors batch for most queries.

10.8 Spelling Correction

"Did you mean...?" — adjacent to autocomplete. Implementations use edit distance (Levenshtein) to find close terms. For performance, use BK-trees or precomputed n-grams.

This adds latency. Often returned as a secondary suggestion path: top-5 exact + top-2 corrections.

10.9 Bottlenecks

  • Redis throughput: with sorted sets and pipelining, single Redis handles 200K+ ops/sec. Shard by prefix prefix (first 2 chars) for higher scale.
  • Cache hit rate: ~99% for common prefixes ("th", "the", "ne"). Tail prefixes (rare typed prefixes) might miss; serve from a slower fallback.
  • Cold start: a new product launch with no data. Bootstrap with editor-curated suggestions or external sources.

SECTION 11

Case Study: Design a News Feed (Twitter / Instagram)

The news feed is the iconic system design problem. Twitter's home timeline, Facebook's news feed, Instagram's main feed — they share an architectural shape that took the industry years to figure out. The key insight: at scale, the obvious approach (compute the feed when requested) doesn't work, but neither does the "smart" approach (precompute every user's feed). The answer is hybrid.

11.1 Requirements

Functional

  • Users post content. Their followers see it in their feed.
  • The feed is reverse-chronological (or ranked — we'll discuss).
  • Users can scroll back through older content (pagination).
  • Reasonable freshness: a post should appear in followers' feeds within seconds.

Non-functional

  • 300M DAU.
  • 2 posts/user/day average → 600M posts/day → ~7K posts/sec average, ~70K peak.
  • Each user reads feed 10× per day → 3B feed reads/day → ~35K reads/sec average, 350K peak.
  • Read latency: feed loads in P99 < 200ms.

11.2 The Two Naive Approaches and Why They Fail

Approach A — Pull (fan-out on read)

When user requests feed: 1. Look up everyone they follow. 2. Fetch recent posts from each followee. 3. Merge, sort by time, return.

GET /feed/alice:
  followees = [bob, carol, dave, ... (avg 200 followees)]
  for each f in followees:
    SELECT * FROM posts WHERE user_id = f ORDER BY created_at DESC LIMIT 50
  merge, sort, take top 50

Pros: simple, no precomputed state. Cons: this query is expensive. 200 followees × 50 recent posts each = 10,000 rows scanned per feed request, merged, sorted. At 35K reads/sec, that's 350M row scans/sec. The DB explodes.

Approach B — Push (fan-out on write)

When a user posts, immediately push the post into every follower's feed cache.

POST /tweet from alice:
  followers = [200 people who follow alice]
  for each follower:
    push tweet_id to feed:{follower} cache

When a user requests feed: just read their precomputed feed cache. Fast.

Pros: feed reads are blazingly fast — O(1) Redis lookup. Cons: write amplification. Alice posts; we do 200 writes. Justin Bieber posts; we do 100 million writes. The celebrity problem.

For Bieber, fan-out can take 30+ minutes if done synchronously. By the time the last followers get the post, the next one has already been written.

11.3 The Hybrid Solution

The trick: fan-out on write for normal users, fan-out on read for celebrities.

  • Normal users (e.g., < 10K followers): push (fan-out on write).
  • Celebrities (> 10K followers): pull (fan-out on read). Don't precompute.

When generating Alice's feed: 1. Read precomputed feed cache (has posts from her normal-user followees). 2. For her followees who are celebrities, query their recent posts at request time. 3. Merge both lists, sort, return.

                                       ┌────────────────┐
   POST a tweet ──→ ┌─────────────┐    │ Pushed to all  │
                    │ Normal user?│─Y→│  followers'     │
                    │             │    │  feed caches    │
                    │             │    └────────────────┘
                    │             │
                    │             │    ┌────────────────┐
                    │ Celebrity?  │─Y→ │  NOT pushed —   │
                    └─────────────┘    │  read at        │
                                       │  feed-fetch time│
                                       └────────────────┘

   GET feed → read user's pushed feed cache
            + read recent posts from celebrities they follow
            + merge + sort

This is exactly what Twitter (and similar platforms) does. The threshold for "celebrity" is tunable.

11.4 The Feed Cache

Implemented as Redis sorted sets per user:

feed:alice → ZSET { tweet_id_1: timestamp_1, tweet_id_2: timestamp_2, ... }

Capped at ~1000 tweets per user. Older tweets dropped (users rarely scroll that deep; if they do, fall back to DB).

When Bob (a normal user) posts:

followers_of_bob = [200 user_ids]
for each follower:
  ZADD feed:{follower} {now} {tweet_id}
  ZREMRANGEBYRANK feed:{follower} 0 -1001   # keep newest 1000

When Alice loads feed:

ZREVRANGE feed:alice 0 49 WITHSCORES   # top 50 by score (timestamp)
# then for each celebrity Alice follows:
#   SELECT * FROM posts WHERE user_id = celebrity AND created_at > X
# merge and sort

11.5 Storage Layer

  • Posts table (sharded by user_id): each user's posts stay together.
  • Followers table (sharded by followee_id): finding "all followers of X" is one shard query.
  • Following table (sharded by follower_id): finding "everyone Alice follows" is one shard query.

Two follower tables, indexed in opposite directions, because both queries are common.

11.6 Deep Dive — Fan-out Workers

The fan-out happens asynchronously. When a normal user posts: 1. Post is written to the DB. 2. A PostCreated event is published to Kafka. 3. Fan-out workers consume the event: a. Look up followers of the poster. b. For each, push to their feed cache. 4. Workers parallelize across followers.

Why async? Avoids slowing down the original write. The user's tweet appears immediately in their own feed (read-your-own-writes); follower feeds update within seconds.

11.7 Deep Dive — Ranking (Beyond Chronological)

Twitter and Facebook moved away from pure chronological order years ago. The "algorithmic feed" ranks by predicted engagement.

Ranking signals: - Time of post (newer is better, with decay). - Predicted user interest (ML model: would they like/comment/dwell?). - Author affinity (do they interact with this author often?). - Content type (some users prefer videos, some text).

The mechanic: candidate generation (give me ~500 candidate posts) → ranking (score each, sort).

Candidate generation comes from the feed cache + celebrity pulls (as before). Ranking is an ML inference on each candidate. Real systems run inference on small per-candidate features at < 1ms each, parallelized.

This adds latency. P99 for ranked feeds is harder to achieve than for chronological.

11.8 Bottleneck Analysis

  • Celebrity write traffic: Justin Bieber posts, 100M followers. If we did fan-out on write, that's 100M Redis writes. Pulled at read time instead — much cheaper.
  • Hot accounts on read: when a celebrity posts, millions of users will all pull from the same celebrity-posts cache. Replicate hot caches across multiple Redis nodes.
  • Feed cache memory: 300M users × 1000 tweet_ids × 16 bytes = 4.8 TB. Sharded across many Redis nodes.
  • DB writes: 7K posts/sec is easy. But each post fanned out to 200 followers = 1.4M cache writes/sec. Need a Redis cluster.

Case Study: Design a Chat / Messaging Service

WhatsApp, Slack, Discord, Telegram. Real-time messaging at scale. The technically interesting parts are persistent connections, message ordering, and offline delivery.

12.1 Requirements

Functional

  • 1-to-1 and group chats.
  • Real-time message delivery.
  • Persist messages (history available on new device).
  • Read receipts, typing indicators, presence ("X is online").
  • Offline delivery (push notifications when user is away).

Non-functional

  • Hundreds of millions of users, billions of messages per day.
  • End-to-end latency: < 200ms for online users.
  • Message ordering: messages in a chat appear in correct order.
  • Reliable delivery: messages aren't lost (at-least-once with dedup).

12.2 Estimation

  • 500M DAU, 50 messages/user/day = 25B messages/day = ~300K/sec, peak 1M+.
  • Storage per message ~500 bytes including metadata. 25B × 500B = 12.5 TB/day.

12.3 The Architecture

                                   │              │
                                   ▼              ▼
                          ┌────────────┐   ┌─────────────┐
                          │ Postgres / │   │   Kafka     │
                          │ Cassandra  │   │  (fan-out)  │

                          └────────────┘          │
                                                  ▼
                                          ┌──────────────┐

The hot path: 1. User A's client sends message via WebSocket to its Realtime Gateway. 2. Gateway forwards to Message Service. 3. Message Service writes to DB, publishes to Kafka. 4. Fanout worker reads Kafka, looks up recipients, pushes to their Realtime Gateways via internal RPC. 5. Recipients' Gateways send to clients via WebSocket.

12.4 Persistent Connections — WebSockets

Each user has a persistent connection to a Realtime Gateway. The Gateway maps user_id → connection.

Scale problem: with 500M users, even 10% online = 50M connections. A single server handles maybe 100K connections. Need 500+ gateway nodes.

When a fanout worker wants to push to user X: 1. Look up which Gateway holds X's connection (Redis or service discovery). 2. RPC to that Gateway. 3. Gateway pushes to the client.

Connection routing: the LB in front of Gateways needs to be sticky — once connected, the user stays on the same gateway. A separate routing service (e.g., a hash of user_id → gateway) ensures consistent placement.

12.5 Message Ordering

For a 1-to-1 chat, ordering within the chat must be consistent. Two approaches:

Per-chat counter: each chat has a monotonically increasing sequence number. When you send, you get the next number. Clients sort by sequence.

Lamport timestamps: similar idea but using logical clocks. More robust to clock skew.

For groups, things get tricky. The "true" order of messages depends on whose clock you trust. Most systems use server-assigned timestamps — the server stamps each message on receipt. Order is what the server saw, not absolute physical time.

This is fine for human-readable chat. For more correctness-critical applications (collaborative editing), CRDTs or operational transforms are used.

12.6 Storage Schema

One row per message in a wide-column store like Cassandra or DynamoDB:

Partition key: chat_id
Clustering key: timestamp DESC, message_id
Columns: sender_id, content, attachments, ...

Partitioning by chat_id means all messages for one chat are co-located. Queries like "last 50 messages in chat X" are single-partition (fast).

A separate user_chats table:

Partition key: user_id
Columns: chat_id, last_message_id_seen, joined_at, ...

This drives the user's chat list ("show all chats I'm in").

12.7 Read Receipts & Presence

Presence (online/offline): each gateway updates a Redis key presence:user_id = online on connect, removes on disconnect. Friends query Redis to see status.

Read receipts: when a user reads a message, the client sends "read:up_to_msg_id=X." The server updates user_chats.last_message_id_seen = X and notifies other chat members.

Typing indicators: very ephemeral. Sent over WebSocket, never persisted. The Gateway forwards "user X is typing in chat Y" to other recipients.

12.8 Offline Delivery

When user is offline (no WebSocket): 1. Message lands in DB normally. 2. Fanout finds user is not connected. 3. Falls back to push notification via APNs/FCM. 4. When user comes back online, client requests "messages since I was last here."

The "since when" is tracked by the user_chats table per chat. Catch-up is straightforward.

12.9 Encryption

For end-to-end encryption (WhatsApp-style), the server NEVER sees plaintext. Clients encrypt with recipient's public key; recipient decrypts.

Server can still: - Route messages. - Persist (encrypted) blobs. - See metadata (who's messaging whom, when, how much).

E2EE is a security topic; we mention but don't deep-dive. Signal Protocol is the de facto standard. Most consumer messaging apps use it now.

12.10 Bottlenecks

  • Connection scaling: 500+ gateway servers. Use cheap, high-connection-count tech (Go, Node, Erlang/Elixir).
  • Group fan-out: a 100K-member group chat. One message → 100K pushes. Spread the fan-out across workers; user-facing latency under 1s.
  • Storage cost: 12.5 TB/day. Use tiered storage — recent messages on SSD, old messages on cheap object storage.
  • Message search: indexed via Elasticsearch (async). Per-user search index.

SECTION 13

Case Study: Design a CDN

CDNs (Cloudflare, Akamai, Fastly, CloudFront) are what make the modern web fast. They cache content at thousands of edge locations worldwide. A user in Lima loading an image gets it from a server in Lima, not from your origin in Virginia.

13.1 Requirements

Functional

  • Cache HTTP responses at edge locations.
  • Serve cached content; miss → fetch from origin.
  • Invalidate cache on content updates.
  • Support custom rules (cache lifetime, headers to respect, etc.).

Non-functional

  • Massive scale: serves a meaningful fraction of all internet traffic.
  • Sub-50ms first byte: from any populated area.
  • High availability: edge nodes are everywhere; failures are local.

13.2 Estimation

  • A large CDN: ~300 cities, ~5000 edge nodes total.
  • Cache miss rate: ~5% (95% of requests served from edge).
  • Edge node capacity: 10-100 Gbps each.

13.3 The Architecture

              Users (Lima, Lisbon, Lagos, Lahore, ...)
                            │
                            ▼
                ┌───────────────────────┐
                │       Anycast IPs     │  ← single IP routed to nearest edge
                └───────────────────────┘
                            │
              ┌─────────────┼─────────────┐
              ▼             ▼             ▼
         ┌──────────┐  ┌──────────┐  ┌──────────┐
         │  Edge    │  │  Edge    │  │  Edge    │
         │  Lima    │  │  Lisbon  │  │  Lagos   │
         └────┬─────┘  └────┬─────┘  └────┬─────┘
              │             │             │
              └─────────────┼─────────────┘
                            │  cache miss
                            ▼
                   ┌────────────────┐
                   │  Origin        │
                   │  (your servers)│
                   └────────────────┘

A request flow: 1. User queries cdn.example.com. DNS returns an anycast IP — many edges advertise this same IP; routing protocols send the user to the nearest one. 2. The nearest edge checks its cache. 3. Hit: serves immediately. 4. Miss: fetches from origin (with intermediate "shield" tiers possible), caches, serves.

13.4 Anycast Routing

A clever networking trick: multiple edge nodes globally advertise the same IP. BGP routing automatically delivers each user to the topologically-closest one.

The user types cdn.example.com; DNS returns one IP; the user's packets follow the shortest route, which by definition leads to the closest edge.

This avoids the latency and complexity of GeoDNS (where DNS itself does the routing). Anycast is the dominant approach now.

13.5 Cache Hierarchy

A request that misses the local edge often hits a mid-tier cache (also called "shield") before reaching origin:

   Edge → Mid-tier → Origin

The mid-tier consolidates traffic from many edges. Reduces origin load further.

13.6 Consistent Hashing for Cache Placement

Within an edge node cluster, content distribution uses consistent hashing.

Naive: node = hash(key) mod N. Problem: adding/removing a node changes most key mappings → massive cache invalidation.

Consistent hashing: nodes and keys are placed on a hash ring. Each key goes to the next node clockwise. Adding/removing a node only re-maps keys near that node's position. ~1/N of keys move, not all of them.

              0
              │
       node1 ─┤
              │
     key2 ────┤
              │
       node2 ─┤   (key2 goes to node2, the next clockwise)
              │
     key3 ────┤
              │
       node3 ─┤   (adding node4 here only affects key2)
              │
            359

Used by every distributed cache: CDNs, memcached pools, DynamoDB's internals.

13.7 Cache Invalidation

The hardest problem in CDNs. When you update content (new version of a JS file), how do you invalidate the cached copies at thousands of edges?

Approaches: - TTL-based: cache for N seconds; updates appear after TTL expires. Simple but slow. - Purge API: customer calls API; CDN propagates invalidation messages to all edges. Takes seconds-to-minutes. - Versioned URLs: include a hash in the URL (/app.abc123.js). Updates change the URL; old cached copies remain but no one fetches them.

Versioned URLs are the gold standard. Static assets get filename hashes. The "current version" URL is set in the HTML. CDN never sees a "current" name change — only new URLs appearing.

13.8 Pushed vs Pulled Cache

Pulled (default): edge fetches from origin on demand. Cache fills naturally as users request content.

Pushed (pre-warming): you proactively push content to edges before users request it. Useful for big launches (a new game; everyone will download).

Most CDNs are primarily pull-based, with optional push features.

13.9 Beyond Static Content

Modern CDNs do more than cache static files: - Edge compute (Cloudflare Workers, Fastly Compute@Edge, Vercel Edge Functions). Run code at the edge. - Image optimization: resize, reformat (WebP/AVIF), compress, on the fly. - DDoS protection: filter malicious traffic at the edge before it reaches origin. - API caching: cache API responses with smart invalidation. - A/B testing routing: split users to different backends at the edge.

The edge has become a programmable layer of the internet. It changed in the late 2010s; serverless compute at the edge is now a major architectural pattern.

13.10 Bottlenecks

  • Cache hit rate: 95% is achievable for static content. For dynamic content, harder — but smart caching of API responses helps.
  • Hot content storm: a viral video. All edges fetch from origin almost simultaneously. Mitigation: request collapsing (only one fetch per edge per object) + mid-tier caching.
  • Cold start at new edge locations: cache is empty; everything misses. Pre-warm with predicted-popular content.

SECTION 14

Case Study: Design a Distributed Cache

A standalone Redis instance handles a lot — but eventually a single node isn't enough memory or throughput. Distributed caches spread the data across many nodes. Memcached, Redis Cluster, and the patterns underneath them appear in every large-scale system.

14.1 Requirements

Functional

  • Key-value store. SET, GET, DELETE, INCREMENT.
  • Optional features: TTL, sorted sets, pub/sub.
  • Linear scalability: doubling nodes ≈ doubling capacity.

Non-functional

  • Latency: sub-millisecond reads, sub-2ms writes.
  • Availability: surviving single-node failures.
  • Consistency: usually weak (eventual is OK).

14.2 Estimation

  • 100M cache entries × 1 KB = 100 GB. Fits in ~10 nodes with 16 GB RAM each (with replication, more).
  • 1M ops/sec total. Each node handles 100K. 10 nodes suffice for the throughput.

14.3 Topology — Sharding

A distributed cache shards keys across nodes. Consistent hashing (covered in CDN section) is the standard approach.

                 Hash ring (0..2^32)
                       │
                   ────┼────
                  /    │    \
              n1 ┘     │      └ n2
                       │
              n4 ─     ─ n3

   key K → hash(K) → maps to position → next node clockwise

Adding/removing nodes only moves ~1/N of keys. Clients embed the hash ring (or fetch it from a coordinator) and compute the destination node themselves — no extra hop.

Virtual nodes

Real implementations don't place each physical node once on the ring. They place it many times (100s of virtual positions). This: - Smooths the load distribution. - Reduces rebalancing impact when nodes change.

Most consistent-hash implementations use vnodes; you don't typically configure them per-call.

14.4 Replication

Each shard has replicas for availability. Two flavors:

Primary-replica: each shard has one primary (handles writes) and N replicas (handle reads). Reads can hit any. Failover when primary dies.

Multi-primary: all nodes accept writes. Inconsistencies are reconciled via vector clocks or last-write-wins. Cassandra and Dynamo use this.

Most caches use primary-replica or no replication at all (cache is recoverable; losing some is OK).

14.5 Multi-Level Caches

           ┌────────────────────┐
           │ L1: in-process     │  ← fastest, smallest, per-instance
           │   (Caffeine, etc.) │
           └─────────┬──────────┘
                     │ miss
                     ▼
           ┌────────────────────┐
           │ L2: distributed    │  ← shared across instances
           │   (Redis cluster)  │
           └─────────┬──────────┘
                     │ miss
                     ▼
           ┌────────────────────┐
           │ L3: origin DB      │
           └────────────────────┘

Different latencies, different cost-per-byte: - L1: ~nanoseconds, ~1 GB per instance. - L2: ~milliseconds, ~100 GB cluster. - L3: ~10ms, terabytes.

A request walks down the levels on miss. Multi-level caching is a major reason Twitter/Facebook scale at all.

14.6 Hot Key Problem

A single cache key gets hammered (a viral celebrity, a featured product). The node holding it becomes a bottleneck.

Mitigations: - Local L1 cache: serve hot keys from in-process memory, never hit Redis. - Replicate hot keys: copy to multiple nodes, route requests across replicas. - Add jitter to TTLs: prevent the "many keys expire at once" stampede.

14.7 Cache Coherence

If two nodes have the same key cached, and one is updated, how does the other learn?

Most caches don't try. Each node has independent TTLs; the stale copy goes away naturally. For higher coherence: - Pub/sub invalidation: writer broadcasts "key X invalidated"; all caches drop it. - Versioning: cached values include a version; clients check freshness.

Hard correctness here is expensive. Most systems tolerate brief inconsistency.

14.8 Bottlenecks

  • Hot key: discussed above.
  • Rebalancing during topology change: adding a node moves keys. During the move, requests for moving keys may miss. Use consistent hashing + slot-by-slot migration to minimize.
  • Total memory ceiling: cluster as big as memory. For huge datasets, consider on-disk caches (RocksDB-backed).

Discussion **Client-side (Memcached):** - Pros: one network hop. Fastest. - Cons: clients need the hash ring config. Changes require client updates. Doesn't auto-rebalance from client view. **Server-side (Redis Cluster):** - Pros: clients don't need to know topology. Easier to add/remove nodes. - Cons: an extra MOVED redirect on first miss. SECTION 15

Case Study: Design a Ride-Sharing Service (Uber/Lyft)

Real-time matching of riders to nearby drivers. The defining challenge is geospatial queries at scale — "find me drivers within 2km, who are available, sorted by ETA." This case study introduces Geohashes, S2 cells, and live location updates.

15.1 Requirements

Functional

  • Driver app reports location every few seconds while on duty.
  • Rider requests pickup at their location.
  • System finds nearby available drivers, sends ride request.
  • One driver accepts; ride begins. Rider tracks driver's approach.

Non-functional

  • Match latency: under 5 seconds from rider request to driver assignment.
  • Location freshness: driver locations within 5 seconds of real position.
  • Scale: 10M concurrent drivers globally; 100M ride requests/day.

15.2 Estimation

  • 10M drivers, location update every 5 sec → 2M location writes/sec.
  • 100M rides/day = ~1200 requests/sec average, 10K peak.
  • Each match query: 1km radius, ~50 candidate drivers to score.

15.3 Geospatial Indexing

The core problem: "find all points within radius R of (lat, lng)." Doing this with a standard SQL query and Pythagoras would scan the whole driver table. We need spatial indexes.

Geohash

Encode each location as a string. Nearby points share string prefixes.

Location (37.7749, -122.4194) → geohash "9q8yyk8ytp"
Nearby points share prefix:
  "9q8yyk8ytp" — same exact cell
  "9q8yyk8y" — same 8-char cell
  "9q8yy" — same 5-char cell (larger area)

Query: "find drivers with geohash starting with 9q8yy" — single index lookup returns all in that cell.

Caveat: cells have fixed sizes; a search radius rarely matches cell boundaries. Real implementations search 9 cells (the user's + 8 neighbors) to avoid edge artifacts.

S2 cells

Google's S2 library uses a hierarchical hexagonal grid on the sphere. More mathematically rigorous than Geohash. Used by Uber, Foursquare, and others.

Each location maps to an S2 cell ID at various zoom levels. Querying "all rides within X km" becomes "all rides in S2 cells covering this radius." Fast.

Pick Geohash or S2 based on tooling. They solve the same problem at similar performance.

Other options

  • R-trees: PostGIS's choice. Great for irregular shapes.
  • QuadTrees: simpler hierarchy. Useful for 2D games.

For ride-sharing, Geohash/S2 in Redis sorted sets is the most common production choice.

15.5 Location Updates

Drivers ping their location every 5 seconds via WebSocket. The Location Service: 1. Receives the update. 2. Updates Redis: GEOADD drivers lng lat driver_id. 3. Acks the driver.

Redis has built-in geo commands (GEOADD, GEORADIUS, GEOSEARCH). Under the hood it uses Geohash + sorted sets. Fast.

For 2M writes/sec, Redis (sharded by driver_id) handles it. Multiple Redis nodes, each holding drivers in a region.

15.6 Match Flow

  1. Rider requests pickup. Their location lat/lng is sent.
  2. Matching Service queries Redis: GEOSEARCH drivers FROMLONLAT ... BYRADIUS 2km. Returns ~50 nearby drivers.
  3. Filter for "available" status (cached separately).
  4. Score candidates: distance, driver rating, surge demand, ETA (need to compute road distance, not just crow-flies).
  5. Pick top driver, send ride request via their WebSocket.
  6. Driver accepts (or declines, in which case try the next candidate).
  7. Trip begins.

The whole flow must complete in 5 seconds. Critical path: Redis query (10ms) + scoring (50ms) + driver round-trip (1-2s for them to accept).

15.7 Trip Lifecycle — A Saga

Once a trip starts, it has multiple stages, each touching different services: 1. Accept → reserve driver, reserve rider. 2. Pickup → record meeting. 3. In progress → meter running, route tracking. 4. Drop-off → calculate fare, charge rider, credit driver, update ratings.

Each is a step in a saga. Compensating actions if any fail (refund the charge, refund the driver's credit). Idempotency throughout.

15.8 Pricing & Surge

When demand outpaces supply in an area, prices rise (surge). Implemented by a separate service that monitors supply-demand ratio per region, publishes a multiplier.

The pricing engine consumes location streams (where drivers/riders are concentrated), computes surge per geo-cell, publishes to the matching service.

Surge calculation is a complex topic — too aggressive and riders churn; too gentle and supply collapses. Real systems use ML.

15.9 Bottlenecks

  • Location write rate: 2M/sec. Sharded Redis. Each region's Redis handles its drivers.
  • Hot cells: airport, downtown core. Many drivers in one cell → hot key. Use finer-grained subdivision or replicate.
  • WebSocket connections: 10M concurrent driver connections. Need many gateway nodes.
  • Cross-region coordination: rare — a ride happens within a city. So sharding by region works cleanly.

Case Study: Design Airbnb (Booking System)

A reservation system. Hotels.com, Airbnb, Booking.com, restaurant reservation systems — they share key challenges: search with complex filters, calendar consistency, atomic booking under contention.

16.1 Requirements

Functional

  • Search listings by date range, location, filters (price, capacity, amenities).
  • View listing details.
  • Book a listing for specified dates.
  • Cancellation, modification.

Non-functional

  • No double-bookings: a property is booked for at most one party per date range.
  • Search latency: P99 < 500ms.
  • Scale: 10M listings worldwide, 100M searches/day, 1M bookings/day.

16.2 Estimation

  • 100M searches/day = ~1K/sec average, ~10K peak.
  • 1M bookings/day = ~12/sec average, peak ~100/sec.
  • Search is read-heavy; bookings are correctness-critical writes.

16.3 Architecture

                          │
                  ┌───────▼────────┐
                  │  API Gateway   │
                  └───────┬────────┘
                          │
              ┌───────────┼───────────┐
              ▼           ▼           ▼
          ┌──────┐   ┌────────┐  ┌─────────┐
          │Search│   │Listing │  │Booking  │
          │ Svc  │   │ Svc    │  │ Svc     │
          └──┬───┘   └───┬────┘  └────┬────┘
             │           │            │

       ┌─────────┐  ┌────────┐  ┌─────────┐
       │Elastic- │  │Postgres│  │Postgres │

16.4 Search

Elasticsearch is the right tool for "find listings matching criteria." Listings are indexed with all searchable attributes: - Geo (city, coordinates). - Capacity, rooms. - Amenities. - Price range. - Available date ranges.

For "available between June 1-5," index pre-computed availability windows. When a booking changes, update the index.

This index is eventually consistent (updated via Kafka events). A property booked seconds ago might still appear in search results briefly — but the booking flow will catch the conflict.

16.5 The Booking Problem

Two users see the same listing as available. Both click "Book." Race condition.

Naive approach (broken):

1. read availability
2. check no overlap
3. write booking

Between 1 and 3, the other user does the same. Both succeed. Double-booking.

Fix 1 — pessimistic locking: SELECT ... FOR UPDATE locks the row during the booking. The second user waits, then sees the updated state.

BEGIN;
  SELECT * FROM listings WHERE id = X FOR UPDATE;
  -- check availability
  INSERT INTO bookings ...;
COMMIT;

Works but limits booking throughput per listing. For a popular listing during peak hours, queuing up.

Fix 2 — optimistic concurrency: each booking attempt includes a version. The insert is conditional on no overlap existing.

INSERT INTO bookings (listing_id, start_date, end_date, ...)
SELECT $listing_id, $start, $end, ...
WHERE NOT EXISTS (
    SELECT 1 FROM bookings
    WHERE listing_id = $listing_id
      AND tsrange(start_date, end_date) && tsrange($start, $end)
);

If 0 rows inserted → conflict; tell user "no longer available." If 1 row → success.

Postgres's GIST index on tsrange (or daterange) makes this overlap check fast. This is the modern preferred approach: no locks held during the transaction.

16.6 Soft Holds

UX nuance: when a user starts the checkout flow, we don't want someone else to book the property out from under them.

Soft hold: when checkout starts, place a 10-minute exclusive hold. If they complete checkout, the booking goes through. If they abandon, the hold expires and the property is available again.

Implementation: a separate holds table (or Redis with TTL). Booking creation checks for active holds by other users.

This is a UX win, but it adds complexity. Smaller systems skip holds and rely on optimistic locking + clear error messages.

16.7 Pricing

Listings have a base price; the actual price depends on: - Date (weekend, holiday). - Length of stay. - Demand. - Discounts (last-minute, long stays).

The pricing service computes a price for a (listing, dates) tuple on demand. Cache results — same query, same answer for the day.

16.8 Payments

Tied to booking. Use the saga pattern (Section 5): 1. Hold listing. 2. Authorize payment (don't capture yet). 3. Confirm booking. 4. Capture payment.

If 4 fails: cancel booking, release hold. If 3 fails: release the auth.

Real systems integrate Stripe or similar; the saga becomes "wait for the payment provider's webhook before confirming."

16.9 Reviews and Ratings

Eventually consistent. After a stay: - Guest can review host. - Host can review guest. - Reviews published after both submit (or after deadline).

Stored in their own service. Computed aggregate scores (average rating, count) for fast display on listings — refreshed periodically, not real-time.

16.10 Bottlenecks

  • Hot listings: a viral listing gets 1000s of booking attempts. Most fail immediately due to overlap. The DB row update load is bounded by successful bookings (low).
  • Search latency: Elasticsearch must scale by sharding. Often sharded by region.
  • Multi-region: real Airbnb operates globally. Bookings shard by listing's region; each region has its own DB and search cluster. Cross-region search aggregates.

Case Study: Design a Video Streaming Service (YouTube/Netflix)

The case study with the most moving parts. Video at scale involves upload pipelines, transcoding, adaptive bitrate streaming, CDN distribution, recommendations. Each is its own discipline.

17.1 Requirements

Functional

  • Upload videos.
  • Process: generate multiple resolutions/bitrates.
  • Stream playback on various devices.
  • Search, recommendations, comments.

Non-functional

  • Massive storage: petabytes.
  • Massive bandwidth: terabytes/sec at peak.
  • Low playback latency: video starts in <2 sec.
  • Adapts to viewer bandwidth: smooth playback on poor connections.

17.2 Estimation

  • YouTube-scale: 500 hours of video uploaded per minute (publicly stated figure).
  • Each minute of HD video: ~100 MB. So uploads: 500 × 60 × 100 MB/min = 3 TB/min upload.
  • Storage: years of upload. Hundreds of petabytes.
  • Reads (playback): billions of hours/day. Tens of terabits/sec of bandwidth at peak.

The economics: storage is cheap; transcoding compute is expensive; bandwidth (via CDN) is the biggest cost.

17.3 The Upload Pipeline

Chunked uploads

Don't upload a 4 GB video as one HTTP request. Split into 5 MB chunks. Resume on failure. Verify with checksums.

The browser handles this with the resumable upload protocol (used by Google, AWS, etc.).

Transcoding

The original is in some source format. Streaming needs: - Multiple resolutions (240p for slow connections, 4K for big screens). - Multiple bitrates per resolution. - Segmented (10-second chunks) for HLS/DASH protocols.

Modern transcoding uses FFmpeg + custom pipelines on GPUs (huge cost saver). Each upload triggers parallel jobs producing N variants.

Thumbnails

Generated alongside transcoding. Multiple frames, eventually one chosen by ML for the "main" thumbnail. Stored as small images.

17.4 Adaptive Bitrate Streaming (HLS/DASH)

The breakthrough that made internet video work.

Old: progressive download. Single bitrate. Bad connection = stutter forever.

New: client downloads a "manifest" listing all available bitrates. Player downloads segments at one bitrate. If bandwidth drops, it switches to a lower bitrate for the next segment. Smooth experience even on flaky networks.

HLS (HTTP Live Streaming, Apple): .m3u8 manifest, .ts segments. The dominant standard. DASH (MPEG-DASH): .mpd manifest, various segment formats. More flexible but less universal.

Most platforms ship both. The player picks based on device support.

A typical setup: - 6-8 quality variants. - 6-10 second segments. - Manifest is tiny (few KB), reloaded periodically for live streams.

17.5 CDN Distribution

Video is the heaviest content on the web. CDN caches segments at the edge. A popular video → entire library cached at every edge. A long-tail video → fetched from origin on first request, cached for next viewers.

Cache hit rate is everything. Even 1% cache miss rate at YouTube scale means terabytes/sec of origin egress (very expensive). Major platforms invest huge engineering effort in maximizing hit rates.

Tricks: - Predict popular content: pre-warm CDN before a big launch. - Localized caching: a video popular in Brazil stays in Brazilian edges; less needed in Tokyo. - Edge servers vs ISP partnerships: large platforms negotiate cache appliances inside ISPs (Netflix's Open Connect, YouTube's GGC).

17.6 Metadata Storage

Separate from video bytes. A videos table:

video_id, title, description, owner_id, length, upload_date,
view_count, status, ...

Plus tables for likes, comments, channels, playlists, captions, etc. Standard SQL territory.

View counts: streamed as events to a real-time aggregator (counts updated every few seconds). Displayed count is a delayed approximation.

17.7 Search and Discovery

Elasticsearch for text search. ML for recommendations: - "Up next" recommendations from a ranking model. - Personalized home page. - Trending feeds.

This is its own field — recommender systems at YouTube/Netflix scale are major research areas. We mention but don't deep-dive.

17.8 Live Streaming — A Different Beast

The above is video on demand (VOD). Live streaming is different: - Source: a live encoder (camera, OBS). - Latency target: sub-30-seconds (5 seconds for "low-latency live"). - Cannot pre-cache.

Live streams use the same HLS/DASH segments, but segments are generated continuously and pushed through the pipeline in near-real-time. CDN serves them moments after they're created.

For truly real-time (Zoom-style), HLS is too slow; protocols like WebRTC are used. Different architecture entirely.

17.9 Bottlenecks

  • Storage cost: petabytes. Tiered: hot in fast storage; cold in archival.
  • Transcoding compute: expensive. Use spot instances + GPUs.
  • CDN egress: dominant cost. Optimize hit rate.
  • Hot videos: a viral upload. Spread across CDN edges; replicate at origin.

Reasons 1. **Adaptive bitrate**: switch quality during playback. 2. **Seeking is cheap**: jump to minute 30 = download from that segment, not from the beginning. 3. **Live streams**: continuous segment generation works for live; one big file doesn't. 4. **DRM and encryption**: per-segment encryption keys. 5. **Bandwidth resilience**: short segments fail in isolation; one bad segment doesn't kill the whole stream. 6. **Cache-friendliness**: each segment is independently cacheable at the CDN. SECTION 18

Monoliths vs Microservices

The most-debated architectural choice in modern software. The pendulum has swung both ways multiple times. This section gives you the honest version of the debate.

18.1 What Each Term Means

Monolith: one application, one codebase, one deployment unit. May have many features and subsystems internally, but ships and runs as a single artifact.

Microservices: many small applications, each in their own codebase, each independently deployed, communicating over the network (typically HTTP/gRPC).

There's a spectrum. A modular monolith has clear internal module boundaries but ships as one piece. Macroservices are large services (10-20 of them) instead of small ones (100+). The debate isn't binary.

18.2 What Microservices Promise

  • Independent deployment: each team ships their service on their own schedule.
  • Independent scaling: scale the busy services without scaling the calm ones.
  • Tech diversity: each service can pick its language, database, framework.
  • Smaller blast radius: a bug in one service doesn't crash the others.
  • Team autonomy: teams own their services end-to-end.

These are real benefits. Many large companies (Amazon, Netflix, Uber, Spotify) operate at huge scale partly because they've embraced microservices.

18.3 What Microservices Actually Cost

Every benefit has a hidden cost.

Network calls instead of function calls. What was a 1-microsecond function call is now a 1-millisecond network call. With chains of services, latency compounds.

Distributed debugging. A request flows through 12 services. When it fails, where? You need distributed tracing infrastructure (Jaeger, OpenTelemetry).

Data consistency. Each service has its own DB. Cross-service transactions become sagas (Section 5). Complex.

Operational overhead. 50 services means 50 deployments, 50 dashboards, 50 alerts pages, 50 different on-call rotations. The infra team grows.

Testing. Integration tests across services are slow and brittle. You add contract testing, service virtualization, complex CI pipelines.

Network failures. Networks fail more often than function calls. Every service-to-service hop needs timeouts, retries, circuit breakers.

Versioning. Service A v1 vs A v2 deployed simultaneously, both being called by Services B, C, D. You can't atomically update everything.

18.4 When Each Wins

Monoliths win when

  • Small team (< ~30 engineers).
  • Early-stage product, fast iteration over feature stability.
  • Domain boundaries unclear (you don't know yet what services should exist).
  • Operations team is small.
  • Latency is critical and intra-service calls would compound.

Microservices win when

  • Large organization (multiple teams, each with their own roadmap).
  • Different parts of the system have very different scale needs (one service handles 1M QPS, another 10 QPS).
  • Strong DevOps maturity (CI/CD, observability, on-call).
  • Tech diversity is genuinely valuable (one team uses Python, another Go, another Java).
  • Failure isolation matters more than latency.

18.5 The "Distributed Monolith" Trap

The worst architecture: microservices' costs without their benefits. Symptoms: - Services that can't be deployed independently (changes to A require coordinated deployment with B). - Shared database across services. - Synchronous chains: A calls B calls C calls D for one user request. - One bug crashes everything anyway (because they're all coupled).

How you end up here: split a monolith into services without first cleaning up its coupling. The new services maintain all the old dependencies, but now they're remote calls.

The fix is hard. Re-monolithize the parts that should never have been split. Decouple data ownership. Re-establish service boundaries that match team boundaries, not file structure.

18.6 The Modern Consensus (2026)

After 15 years of microservices hype, the mature view:

  • Start with a monolith. Almost always. Even at companies that grow huge, the early monolith years are a feature, not a bug.
  • Modularize within the monolith. Clear module boundaries, dependency rules. This is the prerequisite for any later split.
  • Extract services when the pain is real. When a specific module has a different scaling need, ownership conflict, or technology need.
  • Aim for "macro" not "micro" services. 10 large services with clear boundaries beats 100 tiny ones in 90% of cases.

This is approximately the path Stripe, Shopify, GitHub, and many others have followed.

18.7 Service Mesh

If you DO go full microservices, the infra layer has to do a lot: - Service discovery. - Load balancing. - Mutual TLS. - Circuit breaking. - Observability.

Doing this in each service's code is repetitive and error-prone. A service mesh (Istio, Linkerd) handles it for you. A sidecar proxy attaches to each service; the proxies handle the networking. Your service code knows nothing about TLS, retries, or auth.

Powerful but adds operational complexity. Only worth it past a certain service count (~50+ services).

🏋️ Exercise 18.7

Your startup has 20 engineers. You currently have one monolithic Rails app. The CTO is hearing from peers about microservices and wants to migrate. What's your recommendation?

Argument Push back. At 20 engineers, you almost certainly don't have the team size or operational maturity to benefit from microservices. The likely outcome of a migration: - Months of work to extract services. - A distributed monolith because boundaries aren't clean yet. - Slower velocity for years. - Eventually unwinding back to a clearer modular monolith. OAuth 2.0 and OpenID Connect

How modern web/mobile apps authenticate users. "Sign in with Google" works because of OAuth + OIDC. Every system design with authentication should mention these. We'll cover the basics — enough that you can explain them in interviews.

19.1 The Problem

A user wants to use your app. The app needs to know who they are. Several options:

  1. Roll your own (username + password). You store password hashes; you handle signup, login, password resets, MFA. Tedious and risky.
  2. Federated identity (sign in with Google/Apple/GitHub). The user already has an account elsewhere; you delegate authentication.

OAuth 2.0 + OpenID Connect is the standard for option 2.

19.2 Key Distinction — Authentication vs Authorization

  • Authentication: WHO is this user? (Identity verification.)
  • Authorization: WHAT can this user do? (Permissions.)

OAuth 2.0 is fundamentally about authorization — granting an app permission to access something on a user's behalf ("yes, this app can read my Google contacts").

OpenID Connect (OIDC) is built on top of OAuth, adding authentication — the protocol layer that says "here's verifiably who the user is."

In practice, most "sign in with X" buttons use OIDC. OAuth 2.0 alone is for API permission delegation (your app reading my Drive files).

19.3 The Roles

  • Resource Owner: the user.
  • Client: your app.
  • Authorization Server: the identity provider (Google, Auth0, your own).
  • Resource Server: holds the data (Google's API, your backend, etc.).

19.4 The Flows

OAuth defines several flows. Two matter most.

Authorization Code Flow (the main one)

Used for server-side web apps and mobile apps.

1. User clicks "Sign in with Google" in your app.
2. Your app redirects to Google with: client_id, redirect_uri, scope, state.
3. User logs in to Google, consents to permissions.
4. Google redirects back to your app with an "authorization code."
5. Your app exchanges code + client_secret for an access_token (+ id_token + refresh_token).
6. Your app uses access_token to call Google APIs.

   User           Your App           Google
    │   1. click   │                   │
    │ ─────────────►                   │
    │              │  2. redirect      │
    │              │ ─────────────────►│
    │   3. login & consent             │
    │ ◄─────────────────────────────── │
    │              │  4. code          │
    │              │ ◄─────────────────│
    │              │  5. exchange      │
    │              │    code + secret  │
    │              │ ─────────────────►│
    │              │                   │
    │              │  ◄ access_token   │
    │              │    + id_token     │
    │              │                   │

The state parameter prevents CSRF: a value your app generates and verifies on return.

PKCE (Proof Key for Code Exchange)

Adds a layer for public clients (mobile apps, SPAs) that can't keep a client_secret secret. Client generates a random code_verifier, sends its SHA-256 hash (code_challenge) with the initial request. On code exchange, sends the original code_verifier. Server verifies the hash matches.

PKCE is now considered mandatory for public clients. Use it.

Other flows (don't use, but recognize)

  • Implicit flow: returned tokens directly in the URL. Deprecated.
  • Resource Owner Password Credentials: user gives your app their password. Deprecated.
  • Client Credentials: machine-to-machine, no user. Still useful for server-to-server.

19.5 Tokens

OAuth dispenses three kinds of tokens:

Access token: short-lived (minutes to an hour). Sent with API calls in Authorization: Bearer <token> header. The actual permission grant.

Refresh token: long-lived (days to months). Stored securely; used to get new access tokens without re-prompting the user.

ID token (OIDC specific): a JWT containing user identity claims. Used to authenticate the user, not for API calls.

Access tokens can be opaque (a random string the auth server validates) or JWT (self-validating).

19.6 JWT — JSON Web Tokens

A JWT is a string with three base64-encoded parts: header.payload.signature.

header.{
  "alg": "RS256",
  "typ": "JWT"
}

payload.{
  "sub": "user_123",
  "iss": "https://accounts.example.com",
  "aud": "your_app",
  "exp": 1717920000,
  "iat": 1717916400,
  "email": "user@example.com"
}

signature.<RSA signature over the above, by the auth server's key>

The signature lets the receiver verify the JWT is genuine and untampered, without calling the auth server. The receiver downloads the auth server's public key once (cached); validation is purely cryptographic.

This is huge for performance. JWTs scale beautifully — no token-validation round trips.

Caveats: - JWTs can't be "logged out" — they're valid until expiration. Use short lifetimes (15 min). - Don't put secrets in JWTs; they're not encrypted, just signed. - Refresh tokens compensate: long-lived refresh + short-lived access.

19.7 OpenID Connect = OAuth + Identity

OIDC adds, on top of OAuth: - A standardized openid scope. - An ID token in the response. - Userinfo endpoint for getting profile data. - Discovery endpoint (.well-known/openid-configuration) so apps can configure themselves.

When you "Sign in with Google," your app receives an ID token (a JWT) saying "this user is sub=12345, email=user@gmail.com, verified by Google." That's authentication.

19.8 In Real Systems

A typical setup: - API Gateway validates the JWT on each request (cheap — local crypto). - Backend services trust the gateway's validation; they receive the user ID in a header. - Users authenticate via an identity provider (Auth0, Okta, Cognito, or self-hosted Keycloak).

For interviews, this level of detail is plenty. "We'd use OAuth 2.0 with PKCE for our mobile app, validate JWTs at the API gateway, with refresh tokens for long-lived sessions." Twenty seconds, clearly senior.

🏋️ Exercise 19.8

Why is the access token short-lived and the refresh token long-lived, instead of just having one long-lived token?

Answer If access tokens were long-lived AND used everywhere, leaking one (XSS attack, accidental log, malicious extension) would give an attacker permanent access. By making access tokens short-lived (15 min): - An attacker who steals one has only 15 min of access. - The actual long-lived secret (refresh token) is used rarely and from controlled environments. Refresh tokens are: - Used only by the official client (your app's server, not browser code). - Often single-use (one refresh → new access + new refresh). - Revocable (server tracks them; can blacklist). It's defense in depth: even if the short-lived token is leaked, the damage is small. If the refresh token is leaked, you have time to revoke it. This pattern (rotate frequently used credentials; protect rarely used ones) shows up everywhere in security.

20.3 The Rules

C4's discipline:

  1. One level per diagram. Don't mix container-level boxes with component-level ones.
  2. Same diagram for the same level. All "Container" diagrams of your system look similar.
  3. Notation is open. Use any tool. Brown's diagrams use specific symbology but the levels matter more.
  4. Words matter. Each box has a name and a one-line description.

20.4 In Interviews

A subtle but real tip: announce which C4 level you're drawing. "Let me start with the system context — who uses this and what external systems are involved. Then I'll go to containers and detail the components."

This: - Shows the interviewer you have a framework. - Gives them a roadmap of your answer. - Prevents the common mistake of drawing component-level detail before establishing context.

You don't have to use C4 specifically. But the idea — multiple zoom levels, each clearly distinguished — is what makes architecture diagrams readable.

20.5 Tools

  • Mermaid / PlantUML: text-based, version-controlled.
  • Diagrams.net / Lucidchart: traditional UI.
  • structurizr: C4-native, by the model's author.

In interviews you'll likely use a virtual whiteboard. Practice on Excalidraw — it has a clean style and works in a browser.

SECTION 21

Two-Phase Commit, in Depth

We touched on 2PC in Section 5. Worth a closer look because it appears in interview questions on database internals, distributed databases, and "why is your system AP-not-CP."

21.1 The Algorithm in Detail

Recall: - Phase 1 (Prepare): coordinator asks all participants if they can commit. - Phase 2 (Commit/Abort): if all said yes, commit; otherwise abort.

State diagram per participant:

        START
          │
          ▼
       WORKING
          │
          ▼  (receives PREPARE)
       PREPARED  ── if rejects → ABORT (and reply NO)
          │
          ▼  (receives COMMIT)
       COMMITTED

When a participant is PREPARED, it has done all the work locally but not committed. It has logged the prepared state to durable storage. From this point, it MUST honor the coordinator's decision — it can no longer abort on its own.

21.2 Failure Modes

This is where 2PC gets brittle.

Coordinator fails after sending PREPARE, before sending COMMIT

The participants are stuck. They've prepared (logged it durably), but don't know whether to commit or abort. They hold locks. Other transactions are blocked.

When the coordinator recovers, it consults its log. If it had decided to commit, it tells everyone. If it had decided to abort, same.

But what if the coordinator never recovers? The participants are stuck indefinitely. This is the "blocking problem."

Operational mitigations: - Coordinator log is heavily durable. - Coordinator is replicated (Paxos-style). - Participants have escape hatches (manual intervention).

Participant fails after PREPARE

Coordinator hits a timeout. Coordinator decides what to do: - If at least one other participant said NO → abort. - If all said YES → safe to commit; the failed participant will COMMIT when it recovers (consulting its log).

The failed participant, on recovery, looks at its log: "I'm PREPARED on transaction X." Asks coordinator for the decision. Commits or aborts accordingly.

Coordinator AND a participant fail

Worst case. The remaining participants may be unable to determine the correct outcome. Operator intervention.

21.3 Why 2PC Still Exists

Given its problems, why bother?

Strict atomicity is genuinely valuable for: - Financial transactions across two databases. - Distributed databases that promise ACID across nodes (Spanner, CockroachDB). - Two-system updates that MUST stay in sync.

In these cases, the cost is acceptable. Other approaches (sagas) don't give true atomicity.

21.4 Where You'll Encounter 2PC

  • XA transactions: standardized protocol for 2PC across heterogeneous resources (multiple databases, message queues). Supported by enterprise databases.
  • Spanner / CockroachDB: distributed SQL databases use 2PC internally (over Paxos for coordinator durability) for multi-shard transactions.
  • Distributed cache + DB updates: rarely. Most projects use the outbox pattern instead.

Modern systems generally avoid XA. The cost is too high. But knowing what it is helps you discuss the trade-offs.

21.5 Two-Phase Commit vs Sagas

2PC Saga
Atomicity Yes (true atomicity) No (eventual; intermediate states visible)
Isolation Yes (locks held) No (interleaving possible)
Latency High (two round trips, locks) Low (each step is local)
Failure handling Blocking on coordinator fail Compensating actions
Use case Financial transactions, distributed DB Microservices, business workflows

Sagas dominate modern microservice architectures. 2PC dominates inside distributed databases (where the coordinator is itself replicated reliably).

🏋️ Exercise 21.5

A user attempts to transfer money between two of their accounts at the same bank, but the accounts live in DIFFERENT databases (legacy: checking is in DB1, savings is in DB2). Walk through how you'd implement this. 2PC? Saga? Something else?

Discussion This is the classic "do you need atomicity" case. Money cannot be created or lost; both updates must succeed or neither must. **Saga approach**: 1. Debit checking (DB1). 2. Credit savings (DB2). 3. If step 2 fails → compensating action: credit checking back. Problem: between step 1 and 2 (or after step 2 fails but before compensation), the user sees an inconsistent state. For a *user-facing* transfer, they'd see "$0 in checking, $0 in savings" briefly. Confusing but not catastrophic if eventually resolved. For internal accounting? It's fine. Most banking ledgers ARE saga-style under the hood, with a higher-level invariant (total bank assets unchanged) verified by daily reconciliation. **2PC approach**: - True atomicity. Both update or neither. - Cost: both databases support XA, you pay the coordination overhead. **Outbox pattern** (likely the best practical answer): 1. In DB1, atomically: debit checking AND insert "pending transfer to savings" into outbox table. 2. A worker reads outbox, calls DB2 to credit savings. 3. On success, marks outbox row as done. 4. On failure, retries indefinitely (DB1 transaction succeeded; we don't lose data, we just keep trying to apply it). 5. Reconciliation job verifies all outbox entries eventually completed; alerts if any are stuck. The outbox approach gives us: - Atomicity of (debit + intent to credit) in DB1. - Eventually-consistent credit in DB2. - No locks held across systems. - Retryable failures. This is roughly how every modern payment system handles cross-database state. Real banks use sophisticated variants, but the pattern is the same.
SECTION 22

Capstone Project: Write a Real Design Document

You've worked through the fundamentals and eleven case studies. You can reason about scale, consistency, caching, sharding, and trade-offs. Now you'll produce the artifact that actually matters for a system-design portfolio: a complete, written design document for a system of your choice — the kind of document senior engineers write before building anything, and the kind of work sample that gets people hired into senior roles.

Unlike a coding capstone, the deliverable here is a document, not a program. That's exactly right for system design: the skill being demonstrated is thinking and communicating about architecture, and a well-written design doc proves it better than any code could.

22.1 What You're Producing

A design document (roughly 1,500–3,000 words plus diagrams) for one non-trivial system. It follows the same five-phase structure this course taught, turned into a written artifact:

  1. Requirements — functional and non-functional, with explicit numbers.
  2. Estimation — back-of-envelope capacity, traffic, storage.
  3. High-level design — the architecture, with a diagram.
  4. Deep dives — 2-3 components explored in detail, with the real trade-offs.
  5. Bottlenecks & scaling — where it breaks and how you'd address it.

This is precisely what a senior engineer produces at the start of a real project, and precisely what a system-design interview asks you to do out loud. Writing it down forces the clarity that talking can paper over.

Why this makes a strong portfolio piece

  • It demonstrates senior-level thinking. Junior engineers write code; senior engineers make and justify architectural decisions. A good design doc signals seniority.
  • It's exactly what interviews test. Having written several polished design docs means you walk into a system-design interview having already done the thing, calmly, on paper.
  • It shows communication skills. The ability to explain a complex system clearly in writing is rare and highly valued. Most engineers are bad at it.
  • It's public evidence. Published on your GitHub or blog, it's a work sample any hiring manager can read.

22.2 Choosing a System

Pick something you find genuinely interesting and that has real scale challenges. Good choices (each exercises different muscles):

  • A URL shortener (if you want to start with the one we fully worked — but go deeper than Section 7).
  • A ride-sharing service (geospatial, real-time matching).
  • A chat application (WebSockets, ordering, presence).
  • A social media feed (fan-out, the celebrity problem).
  • A video streaming platform (storage, CDN, transcoding).
  • A collaborative document editor (like Google Docs — operational transforms/CRDTs, hard).
  • A stock trading platform (low latency, strong consistency, correctness-critical).
  • A food delivery app (matching, geospatial, real-time tracking, payments).
  • A multiplayer game backend (state sync, latency, cheating prevention).
  • A ticketing system (like Ticketmaster — the flash-sale / high-contention problem).

Pick one that isn't identical to a case study we did, so you're genuinely designing rather than recalling. The ticketing system and collaborative editor are especially good "I designed this myself" choices because they have distinctive hard problems.

22.3 The Structure to Follow

Write these sections, in this order. This is the document outline.

1. Problem statement & requirements

Open with one or two sentences on what the system does. Then:

  • Functional requirements — what it must do, as a bulleted list. Be specific: "users can post messages," "messages appear for all channel members in real-time," etc.
  • Non-functional requirements — the qualities: expected scale (DAU, QPS), latency targets (P99), availability target (how many nines), consistency needs, durability needs.
  • Explicit scope — what you're not designing. "Out of scope: payments, moderation, mobile push." Scoping is a senior move; it shows judgment.

2. Estimation

Do the back-of-envelope math (Section 2's skill):

  • Users → requests per second (average and peak).
  • Read-to-write ratio.
  • Storage growth per day / year.
  • Bandwidth.

Show your working. The numbers drive every architecture decision that follows, so make them explicit. State your assumptions ("assume 10M DAU, each posting twice a day").

3. High-level architecture

  • A diagram — boxes and arrows showing the major components (clients, load balancer, services, databases, caches, queues). ASCII, Excalidraw, Mermaid, whatever — but include one.
  • A prose walkthrough of the main request flows: "when a user posts a message, the request goes to... then... then..."
  • Justify your major technology choices briefly (why SQL vs NoSQL, why a queue here).

4. Deep dives (pick 2-3)

Choose the 2-3 most interesting or most challenging parts and go deep. For each:

  • Explain the specific problem.
  • Present the options.
  • Choose one and justify it with the trade-offs.

Examples of good deep-dive topics: the data model and sharding strategy; the caching approach and invalidation; how real-time delivery works; how you handle the celebrity/hot-key problem; how you ensure consistency for a critical operation.

This is the heart of the document — where you demonstrate real depth rather than surface-level box-drawing.

5. Bottlenecks & scaling

Honest analysis of where the design strains:

  • What breaks first as traffic grows 10×?
  • What are the single points of failure?
  • What's the scaling plan for each bottleneck?
  • What did you consciously trade off, and why is that acceptable?

Ending with the limitations shows maturity. No real system is perfect; pretending yours is signals inexperience.

22.4 Guidance on Writing It Well

Lead with the requirements, always. The most common mistake — in documents and interviews — is jumping to architecture before establishing what's needed. Numbers first; boxes second.

Make the numbers explicit. "High traffic" is meaningless. "40,000 reads/sec at peak" drives real decisions. Every architectural choice should trace back to a requirement or an estimate.

Justify, don't just assert. Not "we use Redis." Rather "we use Redis for the session cache because sessions are read on every request, need sub-millisecond latency, and can tolerate being rebuilt if lost." The why is the value.

Show the trade-offs. Every decision costs something. "We shard by user_id, which co-locates a user's data and makes their queries fast, at the cost of making cross-user analytics queries expensive — which we accept because those run offline in the warehouse." That sentence demonstrates senior thinking.

Include at least one diagram. A system-design document without a picture is hard to follow. It doesn't need to be pretty — clarity beats polish.

Be honest about limitations. The bottleneck analysis is where you prove you understand your own design deeply. Don't hide the weaknesses; explain them and how you'd address them.

Keep it readable. Short paragraphs. Clear headings. A reader should be able to skim the structure and dive into any section. This is a communication exercise as much as a technical one.

22.6 Make It a Series

One design document is good. Three or four is a genuinely impressive portfolio. Consider writing design docs for several different systems over time — each one exercises different concepts:

  • A read-heavy system (feed, URL shortener) — teaches caching and fan-out.
  • A write-heavy system (analytics ingestion, logging) — teaches queues and partitioning.
  • A real-time system (chat, multiplayer, ride-share) — teaches WebSockets, presence, geospatial.
  • A consistency-critical system (payments, ticketing, trading) — teaches transactions, locking, and the hard consistency problems.

A GitHub repo (or a simple blog) titled "System Design Notebook" with four polished design docs is a portfolio piece that will genuinely impress senior engineers and hiring managers. It shows range, depth, and the ability to communicate — exactly what distinguishes a strong candidate.

22.7 Publishing It

  • Put your design doc(s) in a public GitHub repository as markdown, or on a personal blog.
  • Include the diagrams (commit the images, or use Mermaid which renders on GitHub).
  • Write a short intro README framing the collection: "Design documents I've written to practice and demonstrate system design. Each follows a requirements → estimation → design → deep-dives → bottlenecks structure."
  • Link it from your résumé and LinkedIn.

When a hiring manager for a senior or mid-level backend role sees that you've independently written several clear, rigorous design documents, it changes the conversation. You're no longer claiming you understand system design — you're demonstrating it.

That's your capstone. Not code — a document, or a series of them, that proves you can think about systems the way senior engineers do, and communicate that thinking clearly. That skill, demonstrated in writing, is one of the most valuable things you can show.

Practice case studies. A few per week, on paper, without looking anything up. Pick ones from this course. Pick new ones — "design Stack Overflow," "design Google Docs," "design a stock trading platform." Each one trains a slightly different mental muscle.

You now have the principles. The technologies will come and go, and you'll learn each one in days because the foundations are solid.

Build things. Read code. Argue about architecture. Be willing to be wrong, especially in front of senior engineers — that's how you learn the things you can't read.
