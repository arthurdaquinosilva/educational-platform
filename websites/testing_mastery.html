Writing code is half the job; proving it works — and keeps working — is the other half. Automated tests are what let you ship with confidence, refactor without fear, and collaborate without breaking each other's work. This course teaches testing from first principles: what to test, how to write good tests, how to test tricky things (async code, APIs, UI), test-driven development, and how testing fits into real teams and pipelines. The examples use JavaScript/TypeScript with Vitest (a fast, modern test runner whose API is essentially identical to Jest) plus React Testing Library and Playwright — but the principles transfer to every language and framework. Every test example in this course was actually run.

How to use this course

  • Run the tests yourself. Testing is a hands-on skill; reading about it isn't enough. Set up the tiny project in Section 2 and run every example — watch tests pass, then deliberately break the code and watch them fail. That red-to-green loop is the whole game.
  • Each section follows Concept → Code → Real example → Exercises with solutions. Attempt every exercise before opening the solution.
  • Focus on the "why," not just the syntax. Matchers and mocking APIs differ between tools; the judgment — what's worth testing, what makes a test good, when to mock — is what makes you valuable, and it's universal.
  • The stack here is JavaScript/TypeScript, but if you work in Python, Go, or anything else, the concepts map directly (pytest, Go's testing, JUnit — same ideas, different syntax).

By the end you'll be able to write clear unit tests, test asynchronous and UI code, use test doubles well, practice TDD, reason about coverage honestly, wire tests into CI, and — most importantly — decide what deserves a test and what makes it good.

SECTION 1

Why Test?

Before any syntax, you need to believe testing is worth your time — because half of testing well is wanting to. This section makes the case and gives you the mental map for everything that follows.

1.1 What a test actually is

An automated test is just code that runs your code and checks it did the right thing. That's it:

function add(a, b) {
  return a + b;
}

// A test: call the code, assert the result.
if (add(2, 3) !== 5) {
  throw new Error("add is broken!");
}

That's a real (if primitive) test. A test framework like Vitest gives you nicer ways to express "call the code, assert the result," plus a runner that finds your tests, executes them, and reports what passed and failed. But strip away the tooling and every test is that same shape: arrange some inputs, act by running the code, assert the outcome.

1.2 The real reason to test — confidence

People think tests exist to "find bugs." They do, but that undersells them. The deeper value is confidence to change code. Consider what happens without tests: you write a function, manually click through the app to check it works, and move on. Six weeks later you change something nearby — and you have no idea what you just broke. So either you re-test everything by hand (slow, error-prone) or you cross your fingers and ship (scary). Both get worse as the codebase grows, until people are afraid to touch working code.

A good test suite flips that. Change whatever you want, run the tests, and in seconds you know whether you broke anything. This is why testing's biggest payoff isn't the day you write the test — it's every day after, when the test quietly guards that behavior while you and your teammates change everything around it. Tests are what let a codebase stay malleable as it grows.

1.3 What tests give you, concretely

Beyond confidence, tests deliver several things at once:

  • Regression protection — a bug you fix and cover with a test can't silently come back; the test fails if it does.
  • Executable documentation — a well-written test shows exactly how a function is meant to be used and what it returns, and unlike comments, it can't go out of date (if it did, it'd fail).
  • Design pressure — code that's hard to test is usually badly designed (too coupled, doing too much). Writing tests reveals those problems early, nudging you toward cleaner structure (Section 5).
  • Faster debugging — when a test fails, it points at the specific behavior that broke, instead of you hunting through the whole app.
  • Safe collaboration — on a team, tests are how you trust code you didn't write and let others trust yours.

1.4 The types of tests

Tests come in layers, distinguished by how much they exercise at once:

  • Unit tests — test one small piece (a function, a class, a component) in isolation. Fast, focused, numerous. When one fails, you know exactly where the problem is.
  • Integration tests — test several pieces working together (a function that uses a database, a few modules cooperating). Slower, but they catch problems in the seams between units that unit tests miss.
  • End-to-end (E2E) tests — drive the whole application the way a user does (click a button in a real browser, expect the page to update). The most realistic and the most confidence-giving per test, but also the slowest and most fragile.

Each layer catches different bugs. A unit test proves your discount math is right; an integration test proves it's saved correctly; an E2E test proves a user can actually check out. You need a mix.

1.5 The testing pyramid (and trophy)

How much of each layer? The classic guidance is the testing pyramid: many fast unit tests at the base, fewer integration tests in the middle, and just a handful of slow E2E tests at the top. The reasoning is economic — unit tests are cheap to write, fast to run, and precise when they fail, so you can afford lots of them; E2E tests are expensive and flaky, so you use them sparingly for critical user journeys.

A popular modern refinement is the testing trophy, which puts the most weight on integration tests — the argument being that integration tests hit the sweet spot of realistic-enough to catch real bugs while still reasonably fast, and that they give the most confidence per unit of effort for typical web apps. You don't have to pick a camp. The shared, durable lesson is: lots of fast low-level tests, fewer slow high-level ones, and choose the level that gives you the most confidence for the least cost. We'll build all three layers in this course.

1.6 The cost side — testing honestly

Testing isn't free, and pretending otherwise leads to bad tests. Tests are code you have to write and maintain; a test that breaks every time you refactor (without catching real bugs) is a liability, not an asset. The goal isn't "test everything" or chase a coverage number — it's to test the things that matter in ways that stay robust. Throughout this course you'll learn not just how to write tests but how to write good ones and how to recognize bad ones (Section 14). Held that way, testing pays for itself many times over. Held badly, it becomes busywork people resent. The difference is judgment — which is what we're really here to build.

SECTION 2

Your First Tests

Enough theory — let's write and run real tests. This section gets you a working test setup and teaches the anatomy of a test so the rest of the course has a foundation to build on.

2.1 Setting up Vitest

In a JavaScript/TypeScript project (one with a package.json), install Vitest as a dev dependency:

npm install --save-dev vitest

Add a script to package.json so you can run tests easily:

{
  "scripts": {
    "test": "vitest run",
    "test:watch": "vitest"
  }
}

vitest run runs the suite once (what you'd use in CI); plain vitest starts watch mode, re-running relevant tests every time you save — the tight feedback loop you'll live in while developing. Vitest finds any file named *.test.js, *.test.ts, *.spec.js, etc., automatically. (If you know Jest, nearly everything here is identical — Vitest deliberately mirrors Jest's API, so the skills transfer both ways.)

2.2 The anatomy of a test

Here's a complete, real test file. Say we have a function to test:

// math.js
export function add(a, b) {
  return a + b;
}

The test file:

// math.test.js
import { describe, it, expect } from "vitest";
import { add } from "./math.js";

describe("add", () => {
  it("adds two positive numbers", () => {
    expect(add(2, 3)).toBe(5);
  });

  it("handles negative numbers", () => {
    expect(add(-1, -4)).toBe(-5);
  });
});

Four pieces to learn:

  • describe(name, fn) — groups related tests under a label. Optional but keeps output organized. You nest them for structure.
  • it(name, fn) (also spelled test) — defines one test case. The name should describe the behavior ("adds two positive numbers"), so a failure reads like a sentence about what broke.
  • expect(value) — wraps the actual value you got.
  • .toBe(expected) — a matcher that asserts something about it (here, strict equality). If the assertion fails, the test fails with a helpful message showing expected vs actual.

Read it aloud: "it adds two positive numbers — expect add(2, 3) to be 5." Good tests read like specifications.

2.3 Running them

npm test

You'll see each test file, a ✓ or ✗ per test, and a summary. A passing run looks like:

 ✓ math.test.js (2 tests) 3ms

 Test Files  1 passed (1)
      Tests  2 passed (2)

Now make it fail on purpose — change add to a - b and re-run. Vitest shows you exactly which test failed, the expected value, and the actual value:

 ✗ math.test.js > add > adds two positive numbers
   expected -1 to be 5

That precise, immediate feedback — this behavior, this expected value, this actual value — is the core loop of testing. Get comfortable running tests constantly; in watch mode they run as you type.

2.4 The AAA pattern — structure inside a test

Inside a test, the clearest structure is Arrange, Act, Assert (AAA):

import { it, expect } from "vitest";
import { createCart } from "./cart.js";

it("adds an item to the cart", () => {
  // Arrange — set up the inputs and starting state
  const cart = createCart();
  const item = { id: "a1", price: 10 };

  // Act — perform the single action under test
  cart.add(item);

  // Assert — check the outcome
  expect(cart.items).toHaveLength(1);
  expect(cart.total).toBe(10);
});

Separating the three phases (even just with blank lines) makes a test instantly readable: what's the setup, what's the one thing being tested, and what should be true afterward. A test that's hard to split into AAA is often testing too much at once. Each test should act once and verify one behavior — not necessarily one expect, but one logical outcome.

2.5 What makes a good test name

Test names are the first thing you read when something fails, so make them describe behavior, not implementation:

// ❌ vague — tells you nothing when it fails
it("works", () => { /* ... */ });
it("test add", () => { /* ... */ });

// ✅ describes the behavior and the case
it("returns the sum of two numbers", () => { /* ... */ });
it("throws when dividing by zero", () => { /* ... */ });
it("applies a 10% discount for orders over $100", () => { /* ... */ });

A good name lets you understand a failure without reading the test body. A common, readable convention is "it [does X] when [condition]." Treat the name as a one-line spec of the behavior you're locking in.

2.6 A real example — testing a small function

// slugify.js
export function slugify(text) {
  return text
    .toLowerCase()
    .trim()
    .replace(/[^\w\s-]/g, "")
    .replace(/\s+/g, "-");
}

// slugify.test.js
import { describe, it, expect } from "vitest";
import { slugify } from "./slugify.js";

describe("slugify", () => {
  it("lowercases and hyphenates a simple title", () => {
    expect(slugify("Hello World")).toBe("hello-world");
  });

  it("strips punctuation", () => {
    expect(slugify("Hello, World!")).toBe("hello-world");
  });

  it("trims surrounding whitespace", () => {
    expect(slugify("  spaced out  ")).toBe("spaced-out");
  });

  it("collapses multiple spaces into one hyphen", () => {
    expect(slugify("a   b")).toBe("a-b");
  });
});

Notice how the tests double as documentation: reading them tells you exactly what slugify does with punctuation, whitespace, and casing. Each test arranges an input, acts by calling slugify, and asserts the output — and each covers one behavior, so a failure names the exact rule that broke. This is what everyday unit testing looks like.

🏋️ Exercise 2.2

What is the AAA pattern, and why does it make tests easier to read? What does it suggest if a test is hard to fit into AAA?

Solution

AAA = Arrange, Act, Assert: first set up inputs and state (Arrange), then perform the single action under test (Act), then check the outcome (Assert). SECTION 3

Assertions & Matchers

A test is only as good as its assertions. Matchers are the vocabulary of assertions — the different ways to say "I expect this to be true." Choosing the right matcher makes tests clearer and failures more informative. This section is a practical tour of the ones you'll actually use.

3.1 Equality: toBe vs toEqual

The single most important distinction in all of testing assertions:

import { it, expect } from "vitest";

it("toBe checks identity for primitives", () => {
  expect(2 + 2).toBe(4);           // ✅ primitives compared by value
  expect("ab").toBe("ab");         // ✅
});

it("toEqual checks deep structural equality for objects", () => {
  expect({ a: 1 }).toEqual({ a: 1 });      // ✅ same shape and values
  expect({ a: 1 }).toBe({ a: 1 });         // ❌ different objects in memory!
  expect([1, 2]).toEqual([1, 2]);          // ✅ deep array equality
});
  • toBe uses Object.is (essentially ===): same value for primitives, same reference for objects. Two objects with identical contents are not toBe each other, because they're different objects in memory.
  • toEqual recursively compares contents — the right choice for objects and arrays.

This trips up nearly every beginner: use toBe for numbers, strings, booleans; use toEqual for objects and arrays. Reaching for toBe on an object gives a confusing failure even when the contents match. (There's also toStrictEqual, which is like toEqual but also checks for undefined properties and exact types — handy when the distinction matters.)

3.2 Truthiness and nullish

expect(value).toBeTruthy();      // value is truthy
expect(value).toBeFalsy();       // value is falsy
expect(value).toBeNull();        // strictly null
expect(value).toBeUndefined();   // strictly undefined
expect(value).toBeDefined();     // not undefined

Prefer the specific matcher when you can: toBeNull() gives a clearer failure than toBeFalsy() when you actually mean null. Reserve toBeTruthy/toBeFalsy for when any truthy/falsy value is genuinely acceptable — otherwise assert the exact value.

3.3 Numbers

expect(2 + 2).toBe(4);
expect(10).toBeGreaterThan(5);
expect(10).toBeGreaterThanOrEqual(10);
expect(3).toBeLessThan(5);

// Floating point — NEVER use toBe (0.1 + 0.2 !== 0.3 in floating point!):
expect(0.1 + 0.2).toBeCloseTo(0.3);       // ✅ handles float imprecision

That last one matters: 0.1 + 0.2 is 0.30000000000000004 in floating-point arithmetic, so toBe(0.3) fails. Any time you compare computed decimals, use toBeCloseTo, not toBe.

3.4 Strings

expect("hello world").toContain("world");        // substring
expect("hello world").toMatch(/^hello/);          // regex
expect("hello").toHaveLength(5);

toContain checks a substring; toMatch tests a regular expression — great for asserting a value has the right shape without pinning the exact string.

3.5 Arrays and objects

expect([1, 2, 3]).toContain(2);                   // array includes a value
expect([1, 2, 3]).toHaveLength(3);
expect({ name: "Ada", age: 36 }).toHaveProperty("name");
expect({ name: "Ada", age: 36 }).toHaveProperty("age", 36);   // property + value
expect({ a: 1, b: 2 }).toMatchObject({ a: 1 });   // subset match

toMatchObject is especially useful: it checks that an object contains the given properties without requiring you to specify every field — perfect when you only care about part of a larger result.

3.6 Exceptions

To assert that code throws, you must wrap it in a function so the matcher can invoke it and catch the error:

function divide(a, b) {
  if (b === 0) throw new Error("Cannot divide by zero");
  return a / b;
}

it("throws on divide by zero", () => {
  expect(() => divide(1, 0)).toThrow();                       // throws anything
  expect(() => divide(1, 0)).toThrow("Cannot divide by zero"); // message contains
  expect(() => divide(1, 0)).toThrow(/divide by zero/);        // message matches regex
});

The () => divide(1, 0) wrapper is essential — if you wrote expect(divide(1, 0)), the error would throw before expect runs, crashing the test instead of being asserted. Passing a string or regex checks the error message. Testing the error paths is as important as testing the happy path — arguably more, since errors are where things go wrong.

3.7 Negation and async matchers

Any matcher can be negated with .not:

expect(add(2, 2)).not.toBe(5);
expect([1, 2]).not.toContain(3);

And for promises, resolves/rejects let you assert on async results (Section 7 goes deeper):

await expect(Promise.resolve(42)).resolves.toBe(42);
await expect(Promise.reject(new Error("nope"))).rejects.toThrow("nope");

3.8 Choosing the right matcher

The guiding principle: the more specific the matcher, the better the failure message. expect(result).toBe(5) failing tells you "expected 5, got 4" — instantly useful. expect(result > 0).toBe(true) failing only tells you "expected true, got false" — you've thrown away the actual number. So assert on the value, not a boolean you computed from it:

// ❌ throws away information on failure
expect(user.age > 18).toBe(true);
expect(list.includes("a")).toBe(true);

// ✅ specific matchers → informative failures
expect(user.age).toBeGreaterThan(18);
expect(list).toContain("a");

3.9 A real example — well-matched assertions

import { describe, it, expect } from "vitest";

function parsePrice(input) {
  const value = Number.parseFloat(input.replace(/[$,]/g, ""));
  if (Number.isNaN(value)) throw new Error(`Invalid price: "${input}"`);
  return Math.round(value * 100) / 100;
}

describe("parsePrice", () => {
  it("parses a dollar-formatted string", () => {
    expect(parsePrice("$1,234.50")).toBe(1234.5);
  });

  it("rounds to two decimal places", () => {
    expect(parsePrice("9.999")).toBeCloseTo(10.0, 2);
  });

  it("throws on non-numeric input, including the bad value in the message", () => {
    expect(() => parsePrice("abc")).toThrow(/Invalid price: "abc"/);
  });

  it("is not fooled into returning NaN", () => {
    expect(parsePrice("$5")).not.toBeNaN();
  });
});

Each assertion uses the most specific matcher for the job — toBe for an exact number, toBeCloseTo for a rounded decimal, toThrow with a regex for the error path (checking the message includes the offending input), and .not.toBeNaN() for the edge case. When any of these fails, the message tells you precisely what went wrong.

🏋️ Exercise 3.1

Explain the difference between toBe and toEqual. For each of these, say which to use: comparing two numbers, comparing two objects with the same contents, comparing two strings, comparing two arrays.

Solution

toBe compares with Object.is/=== — same value for primitives, same reference for objects (two distinct objects with equal contents are not toBe). toEqual does a deep, structural comparison of contents, which is what you want for objects and arrays.

  • Two numbers → toBe (primitive value comparison).
  • Two objects with the same contents → toEqual (toBe would fail — different references).
  • Two strings → toBe (primitive).
  • Two arrays → toEqual (deep comparison of elements).

🏋️ Exercise 3.2

Why does expect(0.1 + 0.2).toBe(0.3) fail, and what should you use instead? Why must toThrow be given a function, e.g. expect(() => f()).toThrow()?

Solution

0.1 + 0.2 evaluates to 0.30000000000000004 due to floating-point representation, so it's not exactly 0.3 and toBe fails. Use toBeCloseTo(0.3), which compares up to a tolerance and ignores tiny float imprecision.

toThrow needs a function because it must call the code itself inside a try/catch to observe the throw. If you wrote expect(f()), f() would execute and throw before expect runs, so the error would crash the test rather than being caught and asserted. Wrapping it as () => f() hands expect a function it can invoke safely.

Rewrite these to use more specific matchers: expect(user.age > 18).toBe(true) and expect(items.includes("x")).toBe(true). Why is the specific version better?

SECTION 4

Organizing Tests

As a test suite grows, structure keeps it readable and maintainable. This section covers grouping, setup and teardown, test isolation (the property that makes suites trustworthy), and the small tools for focusing and skipping tests.

4.1 Grouping with describe

describe blocks group related tests and can nest to mirror your code's structure:

import { describe, it, expect } from "vitest";

describe("ShoppingCart", () => {
  describe("add()", () => {
    it("increases the item count", () => { /* ... */ });
    it("increases the total", () => { /* ... */ });
  });

  describe("remove()", () => {
    it("decreases the item count", () => { /* ... */ });
    it("does nothing for a missing item", () => { /* ... */ });
  });
});

The nesting produces readable output (ShoppingCart > add() > increases the total) and organizes tests by the method or behavior they cover. Don't over-nest — one or two levels is usually plenty — but grouping by the unit under test and then by method or scenario is a clean, common convention.

4.2 Setup and teardown hooks

Tests often need the same setup. Rather than repeat it, use lifecycle hooks:

import { describe, it, expect, beforeEach, afterEach, beforeAll, afterAll } from "vitest";
import { createCart } from "./cart.js";

describe("ShoppingCart", () => {
  let cart;

  beforeEach(() => {
    cart = createCart();      // fresh cart before EACH test
  });

  it("starts empty", () => {
    expect(cart.items).toHaveLength(0);
  });

  it("holds an added item", () => {
    cart.add({ id: "a", price: 5 });
    expect(cart.items).toHaveLength(1);
  });
});
  • beforeEach runs before every test in its scope — the workhorse, used to create fresh state so tests don't interfere.
  • afterEach runs after every test — for cleanup (closing connections, restoring mocks, clearing data).
  • beforeAll/afterAll run once for the whole block — for expensive setup you can safely share (starting a server, opening a database).

The key reason beforeEach exists: each test should start from a clean, known state. Creating a fresh cart before each test means the "starts empty" test can't be polluted by an item another test added.

4.3 Test isolation — the property that makes suites trustworthy

This is the most important idea in this section: tests must be independent. A test should pass or fail based only on its own code — never on whether another test ran first, or in what order. Shared mutable state is the enemy:

// ❌ BAD — tests share state and depend on order
const cart = createCart();   // one cart for all tests

it("adds an item", () => {
  cart.add({ id: "a", price: 5 });
  expect(cart.items).toHaveLength(1);   // passes...
});

it("starts empty", () => {
  expect(cart.items).toHaveLength(0);   // ❌ FAILS — the item from the previous test is still here
});

// ✅ GOOD — fresh state per test via beforeEach
let cart;
beforeEach(() => { cart = createCart(); });

When tests share state, they become order-dependent and flaky: they pass together but fail when run alone, or break mysteriously when you add a test in the middle. Isolated tests can run in any order (and in parallel, which Vitest does by default), and a failure means that behavior is broken — not "some earlier test left a mess." Whenever you see let x; beforeEach(() => { x = fresh(); }), that's isolation being protected. Make every test able to run entirely on its own.

4.4 Focusing and skipping

While developing, you often want to run or ignore specific tests:

it.only("run just this one", () => { /* ... */ });   // runs ONLY this test in the file
it.skip("temporarily disabled", () => { /* ... */ }); // skipped, but reported
it.todo("write this test later");                     // a placeholder reminder

describe.only("focus this whole group", () => { /* ... */ });
  • .only — run just this test/group, ignoring the rest. Invaluable for focusing on one failure. Never commit .only — it silently disables your other tests in CI; many teams add a lint rule to catch it.
  • .skip — disable a test but keep it visible in the report (better than commenting it out, which hides it). Use for a known-broken test you'll fix soon.
  • .todo — a named placeholder for a test you plan to write, so it shows up as a reminder.

4.5 Parameterized tests with it.each

When the same logic should be checked against many input/output pairs, don't copy-paste — use a table:

import { it, expect } from "vitest";
import { slugify } from "./slugify.js";

it.each([
  ["Hello World", "hello-world"],
  ["Hello, World!", "hello-world"],
  ["  spaced  ", "spaced"],
  ["Multiple   Spaces", "multiple-spaces"],
])("slugify(%s) → %s", (input, expected) => {
  expect(slugify(input)).toBe(expected);
});

it.each runs the test body once per row, with the values substituted in — turning four near-identical tests into one concise, data-driven table. The %s placeholders put the actual values into each test's name, so failures still read clearly. This keeps large sets of examples maintainable and makes it trivial to add another case (just add a row).

4.6 A real example — a well-organized suite

import { describe, it, expect, beforeEach } from "vitest";
import { createCart } from "./cart.js";

describe("ShoppingCart", () => {
  let cart;

  beforeEach(() => {
    cart = createCart();       // isolation: fresh cart per test
  });

  describe("when new", () => {
    it("has no items", () => {
      expect(cart.items).toHaveLength(0);
    });
    it("has a zero total", () => {
      expect(cart.total).toBe(0);
    });
  });

  describe("add()", () => {
    it("adds an item", () => {
      cart.add({ id: "a", price: 10 });
      expect(cart.items).toHaveLength(1);
    });

    it.each([
      [[{ id: "a", price: 10 }], 10],
      [[{ id: "a", price: 10 }, { id: "b", price: 5 }], 15],
    ])("totals %j to %i", (items, expectedTotal) => {
      items.forEach((i) => cart.add(i));
      expect(cart.total).toBe(expectedTotal);
    });
  });
});

Everything from this section is here: nested describe blocks mirror the cart's structure, beforeEach guarantees isolation with a fresh cart per test, behaviors are grouped logically, and it.each handles the multi-case total calculation as a table. The suite reads like a specification of how a cart behaves — organized, isolated, and easy to extend.

🏋️ Exercise 4.1

Why is test isolation important? Give a concrete example of a bug caused by tests sharing state, and explain how beforeEach fixes it.

Solution

Isolation means each test's result depends only on its own code, not on other tests or their order. Without it, tests become flaky and order-dependent.

Concrete bug: two tests share one cart object. Test A adds an item and asserts the count is 1 (passes). Test B asserts the cart "starts empty" — but the item from Test A is still there, so B fails (or passes/fails depending on order). The suite might pass when run in one order and fail in another, or a test passes with the group but fails alone.

beforeEach fixes it by recreating fresh state before every test: beforeEach(() => { cart = createCart(); }). Now each test starts from a clean, known cart, so no test can pollute another, tests can run in any order (and in parallel), and a failure means that behavior is genuinely broken.

🏋️ Exercise 4.2

What's the difference between beforeEach and beforeAll? When would using beforeAll for setup be risky?

Solution

beforeEach runs before every test in scope; beforeAll runs once before all tests in the block. Use beforeEach for fresh per-test state; use beforeAll for expensive setup you can safely share (e.g., starting a server or opening a DB connection once).

beforeAll is risky when the shared thing holds mutable state that tests modify: because it's created once and reused, one test's changes leak into the next, breaking isolation and creating order-dependence — exactly the flakiness beforeEach prevents. Rule of thumb: share via beforeAll only things that are effectively read-only or externally reset; anything a test mutates should be re-created in beforeEach.

Solution

SECTION 5

Writing Testable Code

Here's a secret that reframes everything: the hardest part of testing is usually the code, not the tests. When code is a pain to test, that pain is almost always a signal that the code is poorly structured. Learning to write testable code will make you a better programmer even setting tests aside — because the same properties that make code testable (small, focused, loosely coupled, predictable) make it easier to understand and change. This section is about that design feedback loop.

5.1 Pure functions — the easiest thing to test

A pure function depends only on its inputs and produces only a return value — no reading or writing of external state, no side effects. Pure functions are trivial to test: give inputs, check the output, done.

// ✅ PURE — output depends only on inputs; nothing external touched
function calculateTotal(items, taxRate) {
  const subtotal = items.reduce((sum, item) => sum + item.price, 0);
  return subtotal * (1 + taxRate);
}

// Testing it is effortless:
expect(calculateTotal([{ price: 100 }], 0.1)).toBeCloseTo(110);

Contrast with an impure version that reaches outside itself:

// ❌ IMPURE — reads a global, mutates state, does I/O
let currentTaxRate = 0.1;               // external dependency
function calculateTotalBad(items) {
  const subtotal = items.reduce((sum, item) => sum + item.price, 0);
  console.log("Calculating...");        // side effect
  return subtotal * (1 + currentTaxRate); // depends on hidden global
}

The impure one is harder to test: its result depends on a global you must set up (and reset, or it leaks between tests), and it has a side effect (logging) that's awkward to verify or silence. The more of your logic you can express as pure functions, the more of your code is effortlessly testable. This is why functional-leaning code tends to be well-tested code.

5.2 Separate logic from side effects

Real programs must do impure things — fetch data, write files, update the DOM. The trick isn't to avoid side effects; it's to keep them at the edges and push the logic into pure functions in the middle. Compare:

// ❌ tangled — decision logic mixed with I/O, hard to test in isolation
async function processOrder(orderId) {
  const order = await db.getOrder(orderId);            // I/O
  let discount = 0;
  if (order.total > 100) discount = order.total * 0.1; // logic buried in I/O
  if (order.customer.isVIP) discount += 5;
  await db.saveDiscount(orderId, discount);            // I/O
}

// ✅ separated — pure decision, thin I/O shell
// Pure and trivially testable — no database needed:
function calculateDiscount(order) {
  let discount = 0;
  if (order.total > 100) discount = order.total * 0.1;
  if (order.customer.isVIP) discount += 5;
  return discount;
}

// The I/O shell just wires things together:
async function processOrder(orderId) {
  const order = await db.getOrder(orderId);
  const discount = calculateDiscount(order);   // ← pure core
  await db.saveDiscount(orderId, discount);
}

Now the interesting part — the discount rules, where bugs live — is a pure function you can test exhaustively with plain objects, no database required. The processOrder shell that does I/O is thin and boring, and what testing it needs (mocking db) is minimal. Pushing logic out of I/O and into pure functions is the single highest-leverage move for testability.

5.3 Dependency injection

When code does need a dependency (a database, a clock, a random source, an API client), inject it rather than hard-coding it. Injection means passing the dependency in, so a test can pass a fake one:

// ❌ hard-coded dependency — impossible to test without a real clock/DB
function createUser(name) {
  return { name, id: Math.random(), createdAt: new Date() };  // non-deterministic!
}

That function is unpredictable — Math.random() and new Date() give different results every run, so you can't assert on the output. Inject the sources of nondeterminism:

// ✅ dependencies injected — fully controllable in a test
function createUser(name, { generateId, now }) {
  return { name, id: generateId(), createdAt: now() };
}

// In a test, pass predictable fakes:
const user = createUser("Ada", {
  generateId: () => "fixed-id",
  now: () => new Date("2024-01-01"),
});
expect(user).toEqual({ name: "Ada", id: "fixed-id", createdAt: new Date("2024-01-01") });

Now the function is deterministic under test — you control the id and the clock, so you can assert exactly. Dependency injection is how you make code that uses external things still testable: the real dependency in production, a controlled fake in tests. (Mocking, Section 6, is another way to supply fakes — but injecting is often cleaner because it makes dependencies explicit.)

5.4 The design feedback loop

Notice the pattern across these examples: hard-to-test code is badly designed code. If a function is painful to test, it's usually telling you something:

  • "I need a huge amount of setup to test this" → the function is doing too much; split it.
  • "I can't test this without a real database/network" → the logic is tangled with I/O; separate them.
  • "The result changes every run" → hidden nondeterminism; inject the clock/random/id source.
  • "I have to test five behaviors in one test" → the function has too many responsibilities.

This is why some developers write tests first (Section 8) — not just to have tests, but because the act of testing forces good design. Even if you test after, treat testing friction as a design smell worth listening to. Testable code and well-designed code are very nearly the same thing — which is why learning to test makes you a better engineer overall, not just someone who writes tests.

5.5 A real example — refactoring for testability

Start with a hard-to-test function and fix it:

// ❌ Before: nondeterministic, does I/O, logic buried
async function sendWelcomeIfNew(userId) {
  const user = await api.getUser(userId);
  const daysSinceSignup = (Date.now() - user.signupTime) / 86400000;
  if (daysSinceSignup < 1) {
    await emailService.send(user.email, "Welcome!");
    return true;
  }
  return false;
}

The decision ("is this user new?") is the logic worth testing, but it's trapped between two I/O calls and depends on Date.now(). Extract the pure decision:

// ✅ After: pure decision extracted and injected
function isNewUser(user, now) {
  const daysSinceSignup = (now - user.signupTime) / 86400000;
  return daysSinceSignup < 1;
}

async function sendWelcomeIfNew(userId, { api, emailService, now = Date.now }) {
  const user = await api.getUser(userId);
  if (isNewUser(user, now())) {
    await emailService.send(user.email, "Welcome!");
    return true;
  }
  return false;
}

Now isNewUser is pure and exhaustively testable with plain data and a fixed now — the actual business rule is locked down without any mocking. And sendWelcomeIfNew takes its dependencies as arguments, so a test can inject fake api/emailService and a fixed clock. The refactor that made it testable also made it clearer and more modular — the design feedback loop paying off.

SECTION 6

Test Doubles: Mocks, Stubs & Spies

Real code has dependencies — it calls APIs, queries databases, sends emails, reads clocks. In a unit test you don't want to hit a real API (slow, flaky, has side effects). Test doubles are stand-ins for real dependencies: fake objects you control. This section covers what they are, how to create them, and — most importantly — the judgment of when to use them, because over-mocking is one of the most common ways to write bad tests.

6.1 The vocabulary

"Mock" is used loosely to mean any test double, but the precise terms describe different jobs:

  • Stub — a fake that returns canned answers. "When asked for the user, return this fixed object." Used to control inputs to the code under test.
  • Spy — a wrapper that records how it was called (arguments, how many times) while optionally still doing the real thing. Used to verify interactions.
  • Mock — a fake pre-programmed with expectations about how it should be called, which the test then verifies. In practice, a fake that both returns canned answers and records calls.

Vitest (like Jest) provides these through vi.fn() and vi.spyOn(), and the distinctions blur in everyday use — but knowing the intent behind each keeps your tests clear about what they're actually checking.

6.2 Mock functions with vi.fn()

vi.fn() creates a mock function — a fake function that records every call and can be told what to return:

import { it, expect, vi } from "vitest";

it("records calls and returns a canned value", () => {
  const fn = vi.fn();
  fn.mockReturnValue(42);          // stub: control what it returns

  const result = fn("a", "b");

  expect(result).toBe(42);
  expect(fn).toHaveBeenCalled();               // spy: it was called
  expect(fn).toHaveBeenCalledTimes(1);
  expect(fn).toHaveBeenCalledWith("a", "b");   // ...with these arguments
});

A vi.fn() is stub and spy in one: mockReturnValue makes it return canned data, and matchers like toHaveBeenCalledWith let you assert how it was called. This is the core tool for supplying and verifying dependencies — e.g., passing a vi.fn() as a callback and checking it was invoked correctly.

6.3 Spying on existing methods with vi.spyOn

When you want to watch (or override) a method on an existing object, vi.spyOn wraps it:

import { it, expect, vi, afterEach } from "vitest";

const logger = {
  save(message) { /* ...writes to disk... */ return "saved"; },
};

it("spies on a real method", () => {
  const spy = vi.spyOn(logger, "save");

  logger.save("hello");

  expect(spy).toHaveBeenCalledWith("hello");   // verify the interaction
  spy.mockRestore();                            // restore the original
});

By default spyOn still calls the real method (recording the call); add .mockReturnValue(...) or .mockImplementation(...) to replace its behavior. Crucially, restore spies after use (mockRestore, or a global afterEach(() => vi.restoreAllMocks())) so the override doesn't leak into other tests — an isolation concern from Section 4.

6.4 Mocking whole modules

Sometimes a module (an API client, the file system) needs replacing entirely. vi.mock intercepts imports of a module and substitutes fakes:

import { it, expect, vi } from "vitest";
import { getWeather } from "./weather.js";
import { fetchJson } from "./http.js";

vi.mock("./http.js", () => ({
  fetchJson: vi.fn(),               // replace the real fetchJson with a mock
}));

it("returns the temperature from the API", async () => {
  fetchJson.mockResolvedValue({ temp: 20 });   // stub the async response

  const temp = await getWeather("Paris");

  expect(temp).toBe(20);
  expect(fetchJson).toHaveBeenCalledWith("https://api.weather.com/Paris");
});

Now getWeather runs its real logic, but its HTTP call is intercepted and returns your canned { temp: 20 } — no network, fast and deterministic. mockResolvedValue is the async cousin of mockReturnValue (it resolves a promise). Module mocking is powerful but heavy-handed; prefer dependency injection (Section 5) when you can, and reach for vi.mock when injecting isn't practical.

6.5 The danger — over-mocking

Here's the judgment that separates good testers from cargo-cultists: mocking has real costs, and over-mocking produces tests that pass while the code is broken. Every mock is an assumption about how a dependency behaves. If that assumption is wrong — or drifts out of date when the real dependency changes — your test happily passes against a fantasy:

// ⚠️ Over-mocked: this test barely tests anything real
it("processes the order", () => {
  const db = { save: vi.fn(), load: vi.fn().mockReturnValue({ total: 100 }) };
  const validator = { validate: vi.fn().mockReturnValue(true) };
  const calculator = { calc: vi.fn().mockReturnValue(90) };

  processOrder("id", { db, validator, calculator });

  expect(db.save).toHaveBeenCalled();   // we only checked that mocks called mocks
});

That test mocks everything the function touches, so it never exercises real logic — it just verifies that the function calls its collaborators, which is testing the implementation, not the behavior. If calc actually has a bug, this test won't catch it (the mock returns 90 regardless). Guidelines for mocking well:

  • Mock at the boundaries — external systems you don't own or that are slow/nondeterministic (network, database, time, filesystem, third-party APIs). Don't mock your own pure logic; run it for real.
  • Prefer real objects when they're cheap. If a dependency is a fast pure function, use the real thing — a real calculateTotal gives a truer test than a mock returning a fixed number.
  • Mock what you're not testing, run what you are. In a test of processOrder's logic, the database is a fair mock (it's I/O), but the discount calculation should run for real if that's part of what you're verifying.
  • Fewer mocks = more real coverage. If a test needs to mock five things, that's often a design smell (Section 5) — the function is too coupled.

The mantra: mock the edges, test the middle for real. A test drowning in mocks often proves only that your mocks agree with each other.

6.6 A real example — mocking only the boundary

import { describe, it, expect, vi } from "vitest";

// The unit under test: fetches a user (I/O) then applies REAL logic.
async function getGreeting(userId, { fetchUser }) {
  const user = await fetchUser(userId);          // boundary — will be mocked
  const hour = user.localHour;
  const timeOfDay = hour < 12 ? "morning" : hour < 18 ? "afternoon" : "evening";  // real logic
  return `Good ${timeOfDay}, ${user.name}!`;
}

describe("getGreeting", () => {
  it("greets by name and time of day", async () => {
    // Mock ONLY the boundary (the fetch); let the greeting logic run for real.
    const fetchUser = vi.fn().mockResolvedValue({ name: "Ada", localHour: 9 });

    const greeting = await getGreeting("u1", { fetchUser });

    expect(greeting).toBe("Good morning, Ada!");     // real logic verified
    expect(fetchUser).toHaveBeenCalledWith("u1");     // boundary interaction verified
  });

  it("says evening after 6pm", async () => {
    const fetchUser = vi.fn().mockResolvedValue({ name: "Bob", localHour: 20 });
    expect(await getGreeting("u2", { fetchUser })).toBe("Good evening, Bob!");
  });
});

This is mocking done right: the fetchUser boundary (I/O we don't want in a unit test) is a mock returning canned data, but the actual logic — computing the time of day and formatting the greeting — runs for real and is genuinely verified. Change the time-of-day thresholds and break them, and these tests fail. Mock the edge, test the middle for real.

Solution
  • Stub — a fake that returns canned answers, used to control the inputs the code under test receives (e.g., "when asked, return this user").
  • Spy — a wrapper that records how it was called (args, call count), optionally still calling the real thing, used to verify interactions.

  expect(result).toBe(99);                       // returns the callback's result
  expect(transform).toHaveBeenCalledWith(7);      // called with the right arg
  expect(transform).toHaveBeenCalledTimes(1);
});

vi.fn().mockReturnValue(99) acts as a stub (canned return) and a spy (records the call), letting you assert both the returned value and that transform was invoked once with 7.

🏋️ Exercise 6.3

Explain the danger of over-mocking. Why can a heavily-mocked test pass while the code is broken?

Solution

Over-mocking replaces so many of a function's collaborators with fakes that the test stops exercising real behavior — it only checks that the function calls its (mocked) dependencies. The danger: every mock encodes an assumption about how a dependency behaves, and mocks return whatever you told them regardless of the real logic.

So a heavily-mocked test can pass while the code is broken because the real logic never runs: if the actual calculation or dependency has a bug, the mock still returns its canned value, so the test sees the "expected" result and passes. It's testing the implementation's wiring (that mocks called mocks), not the actual behavior. It also breaks when the real dependency changes but the mock doesn't — passing against an outdated fantasy. The fix: mock only the edges (I/O, external systems) and run your real logic.

🏋️ Exercise 6.4

State the "mock the edges, test the middle" principle. For a function that reads from a database, applies discount rules, and writes back, what should you mock and what should you run for real?

Solution

The principle: mock the boundaries/external systems (things that are slow, nondeterministic, have side effects, or you don't own), and run your own logic for real so the test actually verifies behavior.

For that function: - Mock the database reads and writes — they're I/O (slow, stateful, external), so you stub the read to return known data and spy on the write to verify what was saved. - Run for real the discount-rule logic — that's the core behavior you care about, so it should execute against the (mocked) input data and be asserted directly. Mocking the discount calculation would make the test meaningless (it'd just return a fixed number).

So: stub db.load to return a known order, let the real discount rules compute, and assert both the computed discount and that db.save was called with it.

SECTION 7

Testing Asynchronous Code

Most real code is asynchronous — it fetches data, reads files, waits for timers. Async tests have a special hazard: if you don't wait for the async work correctly, your test can finish (and pass) before the assertion ever runs. This section covers testing promises, async/await, timers, and mocked API calls, plus the pitfalls unique to async testing.

7.1 The core rule — return or await the promise

The number-one async testing mistake is a test that finishes before its assertion. The fix: make the test async and await the work, so the framework waits for it:

import { it, expect } from "vitest";

async function fetchDouble(n) {
  return Promise.resolve(n * 2);      // stand-in for real async work
}

it("doubles asynchronously", async () => {   // ← async test function
  const result = await fetchDouble(21);        // ← await the work
  expect(result).toBe(42);
});

Without async/await, this test would kick off fetchDouble and immediately end — the expect might run against a pending promise, or not run at all, and the test would pass without checking anything. Always await (or return) the async work in a test. A test that forgets this is worse than no test: it's a false sense of safety.

7.2 Asserting on resolved and rejected promises

Two clean ways to assert async outcomes. First, await and assert normally (shown above). Second, the resolves/rejects matchers (Section 3):

it("resolves to the value", async () => {
  await expect(fetchDouble(5)).resolves.toBe(10);
});

it("rejects with an error", async () => {
  const failing = Promise.reject(new Error("boom"));
  await expect(failing).rejects.toThrow("boom");
});

Note the await before expect even with resolves/rejects — it's what makes the test wait. Both styles are fine; await expect(...).resolves reads nicely for one-liners, while const x = await ...; expect(x)... is clearer when you assert several things about the result.

7.3 Testing that async code rejects

Testing the failure path of async code is essential and easy to get subtly wrong. Don't do this:

// ❌ BROKEN — if loadUser resolves, the test passes without catching anything
it("throws for a missing user", async () => {
  try {
    await loadUser("nonexistent");
  } catch (err) {
    expect(err.message).toBe("Not found");
  }
});

If loadUser doesn't throw (a bug!), the catch never runs, no assertion executes, and the test passes — hiding the very bug you're testing for. Use rejects instead, which asserts that a rejection happens:

// ✅ correct — fails if it does NOT reject
it("throws for a missing user", async () => {
  await expect(loadUser("nonexistent")).rejects.toThrow("Not found");
});

If you must use try/catch, add expect.assertions(1) at the top so the test fails when no assertion runs — but rejects is cleaner and less error-prone.

7.4 Mocking async dependencies (fetch/API)

Unit-testing code that calls an API means mocking the network (Section 6) — you don't want real HTTP in a unit test. Use mockResolvedValue/mockRejectedValue (the async versions of mockReturnValue):

import { it, expect, vi } from "vitest";

async function getUserName(id, { fetchUser }) {
  const user = await fetchUser(id);
  return user.name.toUpperCase();
}

it("uppercases the fetched name", async () => {
  const fetchUser = vi.fn().mockResolvedValue({ name: "ada" });   // fake resolved promise

  const name = await getUserName("u1", { fetchUser });

  expect(name).toBe("ADA");
  expect(fetchUser).toHaveBeenCalledWith("u1");
});

it("propagates a fetch failure", async () => {
  const fetchUser = vi.fn().mockRejectedValue(new Error("network down"));  // fake rejection

  await expect(getUserName("u1", { fetchUser })).rejects.toThrow("network down");
});

mockResolvedValue(x) makes the mock return Promise.resolve(x); mockRejectedValue(e) returns Promise.reject(e). Together they let you test both the success and failure paths of async code deterministically, with no network. Testing the failure path is just as important — that's where unhandled rejections and bad error handling hide.

7.5 Controlling time with fake timers

Code that uses setTimeout, setInterval, or debouncing is painful to test if you actually wait — a test shouldn't take 3 real seconds. Fake timers let you fast-forward time instantly:

import { it, expect, vi, beforeEach, afterEach } from "vitest";

function debounce(fn, delay) {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), delay);
  };
}

beforeEach(() => vi.useFakeTimers());
afterEach(() => vi.useRealTimers());

it("only calls the function once after the delay", () => {
  const fn = vi.fn();
  const debounced = debounce(fn, 100);

  debounced();
  debounced();
  debounced();                     // called 3x rapidly
  expect(fn).not.toHaveBeenCalled();   // ...but not yet

  vi.advanceTimersByTime(100);     // fast-forward 100ms instantly

  expect(fn).toHaveBeenCalledTimes(1);   // debounced to a single call
});

vi.useFakeTimers() replaces the real timer functions with controllable fakes; vi.advanceTimersByTime(ms) jumps the clock forward, firing any timers due — instantly, no real waiting. This makes time-based logic (debounce, throttle, retries, polling) fast and deterministic to test. Remember to restore real timers in afterEach so they don't leak.

7.6 A real example — testing an async function's success and failure

import { describe, it, expect, vi } from "vitest";

// Fetches a repo, applies real logic (formatting), with the fetch injected.
async function describeRepo(name, { fetchRepo }) {
  const repo = await fetchRepo(name);
  if (!repo) throw new Error(`Repo "${name}" not found`);
  return `${repo.name} has ${repo.stars} star${repo.stars === 1 ? "" : "s"}`;
}

describe("describeRepo", () => {
  it("formats a repo with multiple stars", async () => {
    const fetchRepo = vi.fn().mockResolvedValue({ name: "vitest", stars: 12000 });
    await expect(describeRepo("vitest", { fetchRepo })).resolves.toBe("vitest has 12000 stars");
  });

  it("uses the singular for exactly one star", async () => {
    const fetchRepo = vi.fn().mockResolvedValue({ name: "tiny", stars: 1 });
    expect(await describeRepo("tiny", { fetchRepo })).toBe("tiny has 1 star");
  });

  it("throws when the repo is missing", async () => {
    const fetchRepo = vi.fn().mockResolvedValue(null);
    await expect(describeRepo("ghost", { fetchRepo })).rejects.toThrow('Repo "ghost" not found');
  });

  it("propagates network errors", async () => {
    const fetchRepo = vi.fn().mockRejectedValue(new Error("timeout"));
    await expect(describeRepo("x", { fetchRepo })).rejects.toThrow("timeout");
  });
});

This covers the full async surface: the happy path (with real formatting logic, including the singular/plural edge case), a not-found case that should throw, and a network failure that should propagate — each awaited properly so the assertions actually run, and each using a mocked boundary so there's no real network. Testing success and both failure modes is what makes async code trustworthy.

🏋️ Exercise 7.1

What is the most common async-testing mistake, and what's the consequence? Show the wrong and right way to test that await getValue() returns 42.

Solution

The most common mistake is not awaiting (or returning) the async work, so the test finishes before the assertion runs. The consequence is a false pass: the expect may never execute (or runs against a pending promise), so the test passes without actually checking anything — worse than no test, because it gives false confidence.

});

// ✅ right — async test that awaits the work

  const v = await getValue();
  expect(v).toBe(42);
});
// or: await expect(getValue()).resolves.toBe(42);

🏋️ Exercise 7.2

Why is this test for a rejection broken, and how do you fix it? javascript it("throws", async () => { try { await risky(); } catch (e) { expect(e.message).toBe("fail"); } });

Solution

It's broken because if risky() doesn't reject (the bug you're guarding against), the catch block never runs, so no assertion executes and the test passes — hiding the failure. It only works when the code already behaves; it can't detect the regression it's meant to catch.

Fix with the rejects matcher, which asserts that a rejection actually occurs:

it("throws", async () => {
  await expect(risky()).rejects.toThrow("fail");
});

Now the test fails if risky() resolves instead of rejecting. (If you must keep try/catch, add expect.assertions(1) so the test fails when no assertion runs.)

🏋️ Exercise 7.3

What problem do fake timers solve? Sketch how you'd test that a function calls its callback after a 1000ms setTimeout, without waiting one real second.

Solution

Fake timers solve the problem that time-based code (setTimeout, setInterval, debounce, polling) would make tests slow and nondeterministic if you waited for real time. They replace the real timer functions with controllable fakes you can fast-forward instantly.

import { it, expect, vi, beforeEach, afterEach } from "vitest";

afterEach(() => vi.useRealTimers());

  setTimeout(cb, 1000);

  expect(cb).not.toHaveBeenCalled();   // not yet
  vi.advanceTimersByTime(1000);        // jump the clock 1s — instantly
  expect(cb).toHaveBeenCalledTimes(1); // now it fired
});

vi.useFakeTimers() installs the fakes, vi.advanceTimersByTime(1000) fires timers due at 1000ms with no real waiting, and vi.useRealTimers() restores them afterward so they don't leak into other tests.

SECTION 8

Test-Driven Development

Test-Driven Development (TDD) inverts the usual order: you write the test first, watch it fail, then write the code to make it pass. It sounds backwards, but it's a powerful design discipline used widely in industry. This section teaches the rhythm, walks a real example, and — honestly — discusses where it helps and where it doesn't.

8.1 The red-green-refactor cycle

TDD is a tight loop of three steps:

  1. Red — write a small test for behavior that doesn't exist yet. Run it; it fails (red). This proves the test can fail and defines what "done" means.
  2. Green — write the simplest code that makes the test pass. Not elegant — just passing. Run it; it goes green.
  3. Refactor — now that a passing test protects you, clean up the code (and tests) without changing behavior. Run tests again; still green.

Then repeat for the next small piece of behavior. The cycle is deliberately tiny — minutes, sometimes seconds — so you're always a step away from working code. The mantra: red, green, refactor.

8.2 Why write the test first?

Writing the test before the code has several non-obvious benefits:

  • It forces you to define "done" precisely before you start — you must state the expected behavior as a concrete assertion, which clarifies your thinking.
  • It guarantees the test actually tests something. A test written after code, against code that already works, might pass for the wrong reasons; a test you've seen fail first is proven to detect the behavior.
  • It drives better design (Section 5). Because you write the usage before the implementation, you naturally design clean, easy-to-call interfaces — you feel the awkwardness of a bad API immediately, as its first user.
  • It keeps you focused. You write only enough code to pass the current test, avoiding speculative over-engineering ("YAGNI" — you aren't gonna need it).
  • You always have a safety net. Because tests come first and you refactor under green, the code is covered from the very first line.

8.3 A worked TDD example — a FizzBuzz-like rule

Let's build a describeNumber function TDD-style: it returns "Fizz" for multiples of 3, "Buzz" for multiples of 5, "FizzBuzz" for both, and the number as a string otherwise.

Cycle 1 — Red. Write the first failing test:

import { it, expect } from "vitest";
import { describeNumber } from "./describeNumber.js";

it("returns the number as a string by default", () => {
  expect(describeNumber(1)).toBe("1");
});

Run it — it fails (the function doesn't exist). Green: write the simplest thing:

export function describeNumber(n) {
  return String(n);
}

Green. Cycle 2 — Red. Add the next behavior:

it("returns Fizz for multiples of 3", () => {
  expect(describeNumber(3)).toBe("Fizz");
});

Fails. Green: just enough to pass:

export function describeNumber(n) {
  if (n % 3 === 0) return "Fizz";
  return String(n);
}

Cycle 3 — Red/Green. Add Buzz for multiples of 5, then FizzBuzz for both, each with a failing test first and the minimal code after. After a few cycles you arrive at:

export function describeNumber(n) {
  if (n % 15 === 0) return "FizzBuzz";
  if (n % 3 === 0) return "Fizz";
  if (n % 5 === 0) return "Buzz";
  return String(n);
}

Refactor. With all tests green, you can safely tidy up (e.g., build the string from parts) knowing the tests will catch any mistake. The point isn't FizzBuzz — it's the rhythm: each behavior is defined by a failing test, then made to pass minimally, then cleaned up under the protection of green tests. You never write code that isn't demanded by a test.

8.4 Where TDD shines — and where it doesn't

TDD is a tool, not a religion. Being honest about its fit:

It shines when: - The requirements are clear and you can state expected behavior up front (algorithms, business rules, parsers, calculations, well-specified functions). - You're fixing a bug — write a failing test that reproduces it first, then fix; you've proven the fix and prevented regression in one move. (This is TDD almost everyone agrees on.) - The logic is complex enough that thinking in small verified steps helps.

It's harder / less valuable when: - You're exploring or prototyping — when you don't yet know what you're building, writing tests for it first is premature; you'd be locking down a design you're still discovering. - The behavior is hard to express as a quick assertion up front (some UI, some integration-heavy code). - You're doing throwaway spikes or research code.

The balanced view: TDD is a genuinely valuable discipline that many excellent engineers swear by, and it's not the only legitimate way to work — plenty of great, well-tested code is written test-after. What matters most is that the code ends up well-tested; TDD is one reliable path there, especially for clear-spec logic and bug fixes. Learn it, practice it, and apply it where it earns its keep rather than dogmatically everywhere. Even if you don't do strict TDD daily, the habit of writing a failing test for each bug before fixing it is worth adopting universally.

8.5 A real example — TDD for a bug fix

The most universally-endorsed use of TDD. Suppose users report that parseTags("a,,b") returns ["a", "", "b"] — an empty tag from the double comma. TDD the fix:

Red — write a test that reproduces the bug (it fails against the current code):

import { it, expect } from "vitest";
import { parseTags } from "./parseTags.js";

it("ignores empty segments from consecutive commas", () => {
  expect(parseTags("a,,b")).toEqual(["a", "b"]);   // currently returns ["a","","b"] → fails
});

Green — fix the code until it passes:

export function parseTags(input) {
  return input
    .split(",")
    .map((t) => t.trim())
    .filter((t) => t.length > 0);   // ← the fix: drop empties
}

Refactor — tidy if needed, tests still green. Now the bug is fixed and permanently guarded: if anyone reintroduces it, this test fails. Writing the failing test first guaranteed you actually reproduced the bug (not just imagined it) and that your fix truly addresses it. Adopt this for every bug you fix, even if you TDD nothing else.

SECTION 9

Testing React Components

UI components need testing too, but how you test them matters enormously. This section teaches component testing with React Testing Library (RTL) — the standard tool — and, more importantly, the philosophy that makes UI tests valuable instead of brittle: test what the user experiences, not how the component is built.

9.1 The guiding philosophy

React Testing Library is built on one principle, worth memorizing: the more your tests resemble the way your software is used, the more confidence they give you. In practice that means:

  • Find elements the way a user would — by their visible text, their label, their role (button, heading) — not by CSS classes, component internals, or test-specific IDs when avoidable.
  • Interact the way a user would — click, type — not by calling component methods or poking at state.
  • Assert on what the user would see — text on screen, elements present or gone — not on internal state variables or which functions were called.

This philosophy is what separates robust UI tests from brittle ones. A test tied to implementation (state, class names, method names) breaks every time you refactor, even when the UI still works perfectly. A test tied to behavior (what renders, what happens on click) keeps passing through refactors and fails only when the user-visible behavior actually breaks — which is exactly when you want it to fail.

9.2 Setup

RTL runs your components in a simulated DOM (jsdom). A typical setup with Vitest:

npm install --save-dev @testing-library/react @testing-library/user-event \
  @testing-library/jest-dom jsdom

Configure Vitest to use the jsdom environment (in vitest.config.js, set test.environment = "jsdom"), and import the jest-dom matchers (which add DOM-specific assertions like toBeInTheDocument). Then you're ready to render components in tests.

9.3 Rendering and querying

The basic shape: render the component, screen to query the resulting DOM, and assert:

import { describe, it, expect } from "vitest";
import { render, screen } from "@testing-library/react";

function Greeting({ name }) {
  return <h1>Hello, {name}!</h1>;
}

describe("Greeting", () => {
  it("displays the name", () => {
    render(<Greeting name="Ada" />);
    expect(screen.getByText("Hello, Ada!")).toBeInTheDocument();
  });

  it("shows the greeting as a heading", () => {
    render(<Greeting name="Bob" />);
    expect(screen.getByRole("heading")).toHaveTextContent("Hello, Bob!");
  });
});

render(<Greeting name="Ada" />) mounts the component into jsdom; screen queries that DOM. The queries are the heart of RTL, and there's a recommended priority order (most user-like first):

  • getByRole — find by accessibility role (button, heading, textbox), optionally with a name. Preferred — it's how assistive tech and users perceive the UI.
  • getByLabelText — form fields by their label. Preferred for inputs.
  • getByText — visible text content.
  • getByTestId — a data-testid attribute. The escape hatch, for when nothing user-facing identifies the element. Use sparingly.

Querying by role and text (not class names) is what keeps tests aligned with user experience.

9.4 getBy vs queryBy vs findBy

Three query variants for three situations — a common source of confusion:

  • getBy... — expects the element to exist now; throws if not found. Use for things that should be present.
  • queryBy... — returns null if not found instead of throwing. Use to assert something is absent: expect(screen.queryByText("Error")).not.toBeInTheDocument().
  • findBy... — returns a promise that waits for the element to appear (for async UI). Use with await when content shows up after a delay or data load: await screen.findByText("Loaded").

The rule: getBy for "is here," queryBy for "is not here," findBy for "will be here soon." Using getBy to check absence throws (wrong); using getBy for async content fails because it checks too early (use findBy).

9.5 Simulating user interaction

Use @testing-library/user-event to interact like a real user (it fires realistic event sequences). Interactions are async, so await them:

import { describe, it, expect } from "vitest";
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

describe("Counter", () => {
  it("increments when the button is clicked", async () => {
    const user = userEvent.setup();
    render(<Counter />);

    expect(screen.getByText("Count: 0")).toBeInTheDocument();

    await user.click(screen.getByRole("button", { name: "Increment" }));

    expect(screen.getByText("Count: 1")).toBeInTheDocument();
  });
});

Notice what this test does not do: it never touches count state directly or checks setCount. It finds the button the way a user would (by its role and label), clicks it, and asserts the visible result ("Count: 1"). If you rewrote Counter to use useReducer instead of useState, this test would still pass — because the user-visible behavior is unchanged. That's the payoff of testing behavior over implementation.

9.6 Testing forms and typing

function SearchForm({ onSearch }) {

The input is found by its label (getByLabelText) — user-like and it nudges you toward accessible markup. user.type types character by character (firing real events), and the test asserts the callback got the typed value. Testing a form this way verifies the whole user flow: find field → type → submit → correct result.

9.7 What NOT to do — testing implementation details

The anti-pattern to avoid:

// ❌ brittle — reaches into implementation
it("sets state on click", () => {
  const { container } = render(<Counter />);
  // querying by class name — breaks if you restyle
  const btn = container.querySelector(".increment-btn");
  // asserting on internal state or instance — breaks if you refactor
});

Tests like this break when you rename a class, switch state management, or restructure the component — even though the UI still works. They create friction that makes people stop refactoring (or stop testing). Always prefer role/label/text queries and assertions on rendered output. If a component is hard to test without reaching inside, that's often a design signal (Section 5) — maybe logic should be extracted into a pure function and tested separately, leaving the component test to cover just the rendering and interaction.

9.8 A real example — a small interactive component

import { describe, it, expect, vi } from "vitest";
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { useState } from "react";

  const [text, setText] = useState("");
  return (
    <div>
      <label htmlFor="todo">New todo</label>
      <input id="todo" value={text} onChange={(e) => setText(e.target.value)} />

        onClick={() => { if (text.trim()) { onAdd(text.trim()); setText(""); } }}

        Add
      </button>
    </div>
  );
}

describe("TodoInput", () => {
  it("adds a trimmed todo and clears the input", async () => {
    const user = userEvent.setup();

    render(<TodoInput onAdd={onAdd} />);

    const input = screen.getByLabelText("New todo");
    await user.type(input, "  buy milk  ");
    await user.click(screen.getByRole("button", { name: "Add" }));

    expect(onAdd).toHaveBeenCalledWith("buy milk");   // trimmed
    expect(input).toHaveValue("");                     // cleared after adding
  });

  it("does not add an empty todo", async () => {
    const user = userEvent.setup();
    const onAdd = vi.fn();

});

These tests read like a description of how a user uses the component: find the labeled input, type into it, click Add, and check the visible/observable outcomes (the callback fired with trimmed text, the field cleared, or — for the empty case — nothing happened). Nothing depends on how TodoInput is implemented internally, so the tests are robust to refactors and fail only when real behavior breaks.

🏋️ Exercise 9.1

State React Testing Library's guiding philosophy. Why does testing by CSS class or internal state produce brittle tests, while testing by role/text produces robust ones?

Solution

RTL's philosophy: the more your tests resemble the way the software is actually used, the more confidence they give — so query and interact the way a user does (by visible text, labels, roles; by clicking and typing) and assert on what the user sees.

Testing by CSS class or internal state is brittle because those are implementation details: renaming a class, restyling, or switching from useState to useReducer changes them without changing what the user experiences — so the test breaks even though the UI still works (a false failure), creating friction that discourages refactoring. Testing by role/text ties the test to user-visible behavior, which only changes when the actual behavior changes — so the test survives refactors and fails precisely when something a user would notice breaks. That's exactly when you want a failure.

SECTION 10

Integration Testing

Unit tests verify pieces in isolation, but real bugs love the seams between pieces — the places where modules connect, data crosses a boundary, or an assumption on one side doesn't match the other. Integration tests exercise several units working together, catching exactly the bugs that isolated unit tests (with their mocks) can miss. This section covers what integration testing is, how to do it, and why many teams find it the highest-value layer.

10.1 What integration testing is (and why it matters)

An integration test verifies that multiple units cooperate correctly — for example, a service that uses a repository that uses a database, tested together rather than with everything mocked. The motivation is a real weakness of pure unit tests: when you mock a unit's every dependency (Section 6), you test it against your assumptions about those dependencies. If an assumption is wrong — the database returns dates as strings, the API nests data one level deeper than you thought, two modules disagree about a field name — every unit test can pass while the assembled system is broken.

Integration tests catch these contract mismatches by running the real pieces together. They trade some speed and isolation for realism: slower than unit tests, a failure spans more code (so slightly harder to localize), but far more likely to catch the bugs that actually reach production. This is why the "testing trophy" (Section 1) weights them heavily — they hit a sweet spot of confidence per effort.

10.2 A simple integration test — modules together

You don't need a database to write integration tests; testing your own modules together (without mocking the seams between them) is already integration testing:

// priceItems.js — pure logic
export function priceItems(items, taxRate) {
  const subtotal = items.reduce((s, i) => s + i.price * i.qty, 0);
  return Math.round(subtotal * (1 + taxRate) * 100) / 100;
}

// cart.js — uses priceItems
import { priceItems } from "./priceItems.js";
export function createCart(taxRate) {
  const items = [];
  return {
    add(item) { items.push(item); },
    total() { return priceItems(items, taxRate); },   // real dependency, not mocked
  };
}

// cart.integration.test.js — tests cart + priceItems TOGETHER
import { describe, it, expect } from "vitest";
import { createCart } from "./cart.js";

describe("cart (integration with pricing)", () => {
  it("computes a tax-inclusive total across multiple items", () => {
    const cart = createCart(0.1);
    cart.add({ name: "Pen", price: 2, qty: 3 });
    cart.add({ name: "Pad", price: 5, qty: 1 });
    // 2*3 + 5*1 = 11, *1.1 = 12.1 — exercises BOTH modules for real
    expect(cart.total()).toBe(12.1);
  });
});

This test deliberately does not mock priceItems — it lets the cart use the real pricing logic, verifying the two modules agree on the data shape ({ price, qty }) and produce the right combined result. If someone changed priceItems to expect quantity instead of qty, a mocked unit test might miss it, but this integration test would fail. Not mocking the seam is the whole point.

10.3 Integration testing with a database

For code that touches a database, the realistic approaches, roughly from fastest to most faithful:

  • An in-memory database — e.g., SQLite in-memory (from the SQL course), or an in-memory fake of your data layer. Fast, no external service, real query behavior. Great default for integration tests of data logic.
  • A real database in a container — spin up the actual database (Postgres, etc.) in Docker for the test run (tools like Testcontainers automate this). Slower but maximally faithful — it catches database-specific behavior an in-memory fake wouldn't.
  • A test database — a dedicated schema/instance reset between tests.

The critical discipline whichever you choose: reset state between tests (Section 4's isolation, now at the data layer) — typically by wrapping each test in a transaction that rolls back, or truncating tables in beforeEach — so tests don't pollute each other. A rough shape:

import { describe, it, expect, beforeEach, afterAll } from "vitest";
import { createUserRepo } from "./userRepo.js";
import { openTestDb } from "./testDb.js";

describe("userRepo (integration)", () => {
  let db, repo;
  beforeEach(async () => {
    db = await openTestDb();        // fresh in-memory DB per test → isolation
    repo = createUserRepo(db);
  });
  afterAll(async () => { await db?.close(); });

  it("saves and retrieves a user by id", async () => {
    const created = await repo.create({ name: "Ada" });   // real INSERT
    const found = await repo.findById(created.id);         // real SELECT
    expect(found).toMatchObject({ name: "Ada" });          // round-trip through the DB
  });
});

The value: this exercises the real SQL, the real mapping between rows and objects, and the real repository code together — catching schema/column mismatches and query bugs that a mocked db never would.

10.4 Integration testing an API

For a web server, integration tests hit your route handlers with real requests (using a library like Supertest) — exercising routing, middleware, validation, and handler logic together, typically against a test database:

// pseudo-shape of an API integration test
import { describe, it, expect } from "vitest";
import request from "supertest";
import { app } from "./app.js";

describe("POST /users", () => {
  it("creates a user and returns 201 with the created record", async () => {
    const res = await request(app)
      .post("/users")
      .send({ name: "Ada", email: "ada@example.com" });

    expect(res.status).toBe(201);
    expect(res.body).toMatchObject({ name: "Ada" });
    expect(res.body.id).toBeDefined();
  });

  it("returns 400 for a missing name", async () => {
    const res = await request(app).post("/users").send({ email: "x@y.com" });
    expect(res.status).toBe(400);
  });
});

This drives the whole request pipeline — parsing, validation, the handler, the response — as a client would, without spinning up a real network server. It's one of the highest-value tests for a backend: it verifies the contract your API actually exposes.

10.5 Balancing the layers

Integration tests are powerful but not a silver bullet — hold them in balance with the other layers:

  • Use unit tests for detailed logic and edge cases (a pricing function's dozen rules) — fast and precise; you don't want to route every edge case through the database.
  • Use integration tests for the seams and critical flows (does the service actually save the right thing? does the API return the right status?) — where contract bugs live.
  • Mock only the truly external in integration tests (third-party payment APIs, email providers) — keep your own pieces real, since their integration is what you're testing.

The combination is what gives real confidence: units prove each rule is right, integration proves the rules are wired together correctly. A bug that slips through units because "all my mocks agreed" is exactly what an integration test catches.

🏋️ Exercise 10.1

What kinds of bugs do integration tests catch that unit tests (with mocks) can miss? Explain the term "contract mismatch."

Solution

Integration tests catch bugs in the seams between units — where modules connect and must agree on data shapes, types, names, and behavior. Pure unit tests mock each dependency, so they verify a unit against your assumptions about its collaborators; if those assumptions are wrong, every unit test passes but the assembled system fails.

A contract mismatch is exactly that kind of disagreement across a boundary: two pieces have incompatible expectations about their shared interface — e.g., one module returns { qty } but the other reads { quantity }, the database returns dates as strings but the code expects Date objects, or an API nests data one level deeper than the caller assumes. Each side's unit tests (with mocks matching its own assumption) pass, but running the real pieces together reveals the mismatch. Integration tests, by not mocking the seam, surface these.

🏋️ Exercise 10.2

In the cart + pricing example, why is it important that the integration test does not mock priceItems? What bug would that catch that a mocked unit test wouldn't?

Solution

Not mocking priceItems is the point: the test exercises the real interaction between cart and priceItems, verifying they agree on the data shape and produce the correct combined result. Mocking priceItems would make the test assert only that the cart calls it — against a fake that returns whatever you programmed — so it wouldn't verify the real pricing or the real contract between the modules.

Concretely, it catches a contract mismatch: if priceItems were changed to read item.quantity while the cart still adds items with qty, the real pricing would compute wrong totals (or NaN). The integration test — running both for real — fails and exposes this. A mocked unit test of the cart would still pass, because the mock doesn't care about the field name, hiding the bug until production.

🏋️ Exercise 10.3

When integration-testing database code, why must you reset state between tests, and what are two ways to do it? What should you still mock in an integration test?

Solution

You must reset state between tests to preserve isolation at the data layer: a shared, persistent database means one test's inserts/updates leak into the next, causing order-dependent, flaky tests (Section 4) — e.g., a "no users exist" test failing because a previous test created one.

Two ways to reset: 1. Transaction rollback — wrap each test in a database transaction and roll it back afterward, so changes never persist. 2. Truncate/recreate — clear the relevant tables (or spin up a fresh in-memory database) in beforeEach.

What to still mock in an integration test: the truly external systems you don't own — third-party APIs like payment or email providers — since you don't want real charges/emails and their behavior isn't what you're testing. Keep your own pieces (your modules, your database access) real, because verifying their integration is the whole purpose.

SECTION 11

Code Coverage & Test Quality

Once you have tests, a natural question is "how much of my code is tested?" Code coverage answers that — but it's the single most misused metric in testing. This section explains what coverage measures, what it genuinely tells you, what it dangerously doesn't, and how to think about test quality beyond any number.

11.1 What coverage measures

Coverage tools run your test suite while tracking which lines, branches, and functions of your code actually executed. They then report percentages. Vitest has coverage built in:

npx vitest run --coverage

You'll get a table like:

File          | % Stmts | % Branch | % Funcs | % Lines |
--------------|---------|----------|---------|---------|
cart.js       |   100   |    75    |   100   |   100   |
discount.js   |   62.5  |    50    |   66.6  |   62.5  |

The common coverage types, from weakest to strongest signal:

  • Statement/line coverage — did each line run at least once? The most common and the weakest.
  • Branch coverage — did each branch of every if/? :/switch go both ways? Stronger, because it catches untested conditional paths.
  • Function coverage — was each function called?

Branch coverage is the one to watch: 100% line coverage with 60% branch coverage means whole logical paths (the else, the error case) are never tested.

11.2 What coverage genuinely tells you

Coverage's real value is finding what you haven't tested. If a file shows 40% coverage, you know large parts are unexercised — untested code that could be silently broken. Low coverage is a reliable red flag; the uncovered lines highlighted in a report are a useful to-do list of gaps. Used this way — as a gap-finder — coverage is genuinely helpful. Look at which lines are red, not just the percentage: an uncovered error handler or edge branch is a real hole worth filling.

11.3 What coverage dangerously does NOT tell you

Here's the crucial insight that separates thoughtful engineers from metric-chasers: coverage tells you code ran, not that it's correct. A line can be executed by a test that asserts nothing meaningful about it. Consider:

function calculateDiscount(total) {
  return total * 0.9;   // 10% off — but what if the rule is wrong?
}

// This test gives 100% coverage...
it("runs the discount function", () => {
  calculateDiscount(100);         // ← executes the line
  expect(true).toBe(true);        // ← ...but asserts NOTHING about the result
});

That test hits 100% coverage while verifying nothing. The function could return garbage and the test would still pass. This is why coverage percentage is a terrible goal. When a number becomes a target, people optimize the number: they write tests that execute code without truly checking it, add assertion-free tests, or test trivial getters to pad the stat — producing high coverage and low confidence. High coverage is necessary but not sufficient for a good suite: you can't test what you don't run, but running isn't testing.

11.4 The 100% coverage trap

Chasing 100% coverage is usually a mistake:

  • Diminishing returns — going from 0→70% covers your important logic; 90→100% often means testing trivial code (simple getters, defensive branches that can't happen) at high effort for low value.
  • It incentivizes bad tests — a mandated 100% gate pressures developers to write meaningless tests just to color the last lines green.
  • It creates false confidence — "100% covered" sounds bulletproof but can hide a suite full of weak assertions.

A healthier stance: aim for high coverage of the code that matters (core logic, tricky branches, edge cases) and don't sweat the last few percent of trivial code. Many good teams target something like 70–85% as a floor that flags gaps, not a ceiling to max out — and they care far more about what is covered and how well it's asserted than the number itself.

11.5 Mutation testing — measuring test quality

If coverage measures what your tests run, mutation testing measures whether they'd actually catch bugs. A mutation-testing tool deliberately introduces small bugs ("mutants") into your code — changing > to >=, + to -, true to false, deleting a line — then runs your suite against each mutant. If a test fails, the mutant is "killed" (good — your tests caught the bug). If all tests still pass, the mutant "survived" (bad — your tests didn't notice a real change in behavior).

Original:  if (total > 100) discount = 0.1;
Mutant:    if (total >= 100) discount = 0.1;   // did any test catch this?

A surviving mutant reveals a real weakness: your tests execute that line (100% coverage!) but don't actually pin down its behavior. Mutation testing is slower and more advanced (tools like Stryker for JavaScript), so it's not an everyday habit for most teams — but understanding it sharpens how you think: a good test is one that would fail if the code were wrong. That's the true measure of test quality, and it's what coverage can't see.

11.6 A real example — same coverage, very different quality

// The code under test:
function grade(score) {
  if (score >= 90) return "A";
  if (score >= 80) return "B";
  if (score >= 70) return "C";
  return "F";
}

// ❌ 100% line coverage, LOW quality — weak assertions, misses boundaries
it("returns grades", () => {
  expect(grade(95)).toBeDefined();   // "defined"? that's almost nothing
  expect(grade(85)).toBeDefined();
  expect(grade(50)).toBeDefined();
  // every line ran, but a bug (e.g., >= 90 → > 90) would NOT be caught
});

// ✅ Same coverage, HIGH quality — exact values AND boundaries
import { it, expect } from "vitest";
it.each([
  [90, "A"], [89, "B"],   // boundary: is 90 an A? is 89 a B?
  [80, "B"], [79, "C"],
  [70, "C"], [69, "F"],
  [100, "A"], [0, "F"],
])("grade(%i) is %s", (score, expected) => {
  expect(grade(score)).toBe(expected);
});

Both suites report ~100% coverage, but only the second would catch a bug. It asserts exact grades and — crucially — tests the boundaries (90 vs 89, 80 vs 79), which is exactly where off-by-one bugs (>= vs >) hide. A mutation test would kill mutants against the second suite and let them survive against the first. This is the whole lesson: coverage counts execution; quality comes from strong assertions on the cases that matter, especially boundaries.

🏋️ Exercise 11.1

What is the difference between line/statement coverage and branch coverage? Why can 100% line coverage still hide untested logic?

Solution

Line/statement coverage measures whether each line (or statement) executed at least once during the tests. Branch coverage measures whether each branch of every conditional (if/else, ternary, switch, &&/||) was taken in both directions.

100% line coverage can still hide untested logic because a line can run without all its conditional paths being exercised. For example, if (x) doA(); — a single test with x truthy runs the line (100% line coverage for it) but never tests the case where x is falsy. Whole logical paths (the implicit else, error branches, edge conditions) can be unexercised even at 100% line coverage. That's why branch coverage is a stronger signal: it catches those missing paths.

SECTION 12

Testing in the Real World

Everything so far has been about writing tests. This section is about living with them on a real project and team: running them automatically in CI, catching problems before they're committed, deciding what deserves a test, dealing with legacy code that has none, and maintaining a suite over time so it stays an asset rather than a burden.

12.1 Continuous Integration — tests that run themselves

Tests only protect you if they actually run. Continuous Integration (CI) runs your test suite automatically on every push and pull request, so broken code is caught before it merges. This is where testing delivers its biggest team payoff: no one can accidentally merge a change that breaks the suite, because the CI check blocks it.

A minimal GitHub Actions workflow:

# .github/workflows/test.yml
name: Tests
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test          # vitest run — fails the build if any test fails

When a test fails, the pull request shows a red ✗ and (typically) can't be merged until it's green. This is the mechanism that turns "we have tests" into "our tests actually protect the main branch." A few CI practices worth knowing: run tests on a clean checkout (so you catch missing-dependency bugs), make the build fail on any test failure (never merge red), and keep the suite fast enough that people don't dread the wait (parallelize, and reserve slow E2E tests for a smaller/less-frequent job if needed).

12.2 Pre-commit hooks — catching issues even earlier

You can catch problems before code even reaches CI, using pre-commit hooks — scripts that run automatically when you git commit. Tools like Husky make this easy: run linting and the relevant tests on commit, and reject the commit if they fail.

// package.json — with Husky + lint-staged
{
  "lint-staged": {
    "*.{js,ts}": ["eslint --fix", "vitest related --run"]
  }
}

A common, pragmatic setup: on commit, run the linter and only the tests related to the changed files (fast), and leave the full suite to CI. This gives near-instant feedback and keeps obviously-broken code from ever being committed — without making every commit wait for the entire suite. Keep hooks fast, or people will bypass them with --no-verify.

12.3 What to test — and what not to

Not everything deserves a test; spending testing effort wisely is a real skill. A practical guide:

Do test: - Business logic and rules — calculations, validations, pricing, permissions. This is where bugs are costly and behavior is precise. Highest priority. - Edge cases and boundaries — empty inputs, zero, negatives, off-by-one boundaries, error conditions. Bugs cluster here. - Anything that's broken before — every bug fix gets a regression test (Section 8). - Critical user paths — login, checkout, the things that must never break, via integration/E2E.

Usually don't bother testing: - Trivial code — simple getters/setters, one-line pass-throughs with no logic. Testing them is noise. - Third-party libraries — you don't test that React or the standard library works; assume they're tested. Test your use of them. - Framework/config boilerplate — generated code, plain configuration. - Purely visual/styling details better checked by eye (though critical layout can warrant a test).

The guiding question isn't "can I test this?" but "what's the risk and cost if this breaks, versus the cost of the test?" Test where the value is high (complex, important, error-prone logic) and skip where it's low (trivial, unlikely-to-break, cheaply-verified-by-hand). This risk-based judgment is exactly the kind of thing that distinguishes experienced engineers.

12.4 Testing legacy code

Real jobs often mean inheriting a large codebase with few or no tests — and code that wasn't written to be testable (Section 5). You can't stop everything to add tests everywhere, so use targeted strategies:

  • Characterization tests — when you don't even know exactly what the code is supposed to do, write tests that capture what it currently does (its actual outputs for given inputs). These "pin" the existing behavior so you can refactor safely: you're not asserting correctness, you're locking in current behavior to detect unintended changes. Once pinned, you can improve the code and know instantly if you altered its behavior.
  • Test before you change — before modifying an untested area, add tests around it first (characterization or, if you understand the intent, proper tests), so your change has a safety net.
  • Add tests as you touch code — don't try to retrofit tests everywhere at once. Whenever you fix a bug or add a feature in a region, add tests for that region. Coverage grows organically where it matters (the code being actively changed).
  • Extract and test — when you need to change tangled code, extract the piece you're touching into a testable pure function (Section 5), test that, then change it. You improve testability incrementally.

The mindset: legacy code without tests is changed fearfully; adding even a few characterization tests around the area you're working in converts that fear into confidence, one region at a time.

12.5 Maintaining tests over time

A test suite is code, and like all code it needs maintenance or it rots:

  • Keep tests fast. A slow suite that takes 20 minutes gets run less and trusted less. Keep unit tests in milliseconds; push slow things (real DB, E2E) into separate, less-frequent jobs. Speed is a feature.
  • Fix or delete flaky tests immediately. A test that fails intermittently (Section 14) is worse than no test — it trains people to ignore red, eroding trust in the whole suite. Quarantine and fix it, or remove it.
  • Refactor tests too. Extract shared setup into helpers/fixtures, remove duplication, keep names clear. Tests you can read and change are tests you'll keep.
  • Delete tests that no longer earn their place. Tests for removed features, or redundant tests that only duplicate others, are maintenance cost with no benefit. A leaner, meaningful suite beats a bloated one.
  • Treat a failing test as information, not an annoyance. When a legit test fails, either the code has a bug (fix the code) or the behavior legitimately changed (update the test deliberately) — never just "make it pass" by weakening the assertion without thought.

The goal is a suite people trust and rely on — fast, green-means-good, red-means-real. That trust is the whole point; a suite nobody trusts provides no confidence no matter how many tests it has.

CI is the authoritative safety net (runs the full suite on a clean checkout); pre-commit hooks are a fast first line of defense that shortens the feedback loop, so you don't wait for CI to learn you broke something trivial.

SECTION 13

End-to-End Testing

At the top of the testing pyramid sit end-to-end (E2E) tests: tests that drive your entire application the way a real user does — a real browser, real clicks, real navigation — and verify the whole system works together. They give the most realistic confidence of any test, and they're also the slowest and most fragile. This section covers what E2E testing is, the tools, what a test looks like, the flakiness problem, and — importantly — when E2E is worth it.

Note: the E2E examples below are illustrative of the Playwright API and structure. Unlike the earlier sections (whose examples were all executed), these require a running browser and web server, so treat them as faithful templates to adapt rather than copy-paste-runnable snippets.

13.1 What E2E testing is

An E2E test exercises your application as a black box, through its real user interface, across the full stack. For a web app, that means: launch a real (or headless) browser, navigate to a URL, click buttons, fill forms, and assert on what appears on the page — with the actual frontend, backend, and (usually) a real database all running. Where a unit test checks one function and an integration test checks a few modules, an E2E test checks the entire user journey: "can a user actually sign up, log in, add an item to the cart, and check out?"

Because it uses the real UI and real backend, an E2E test catches things no lower test can: a broken deployment, a frontend/backend contract mismatch, a misconfigured route, a button that's covered by another element, a login flow that breaks across pages. It's the closest thing to a robot user clicking through your app.

13.2 The tools

The dominant E2E tools for web apps are Playwright (modern, from Microsoft, cross-browser, fast, increasingly the default) and Cypress (popular, great developer experience). Both let you script browser interactions in JavaScript/TypeScript. Playwright can drive Chromium, Firefox, and WebKit, run headless in CI, and auto-wait for elements. Setup is roughly:

npm init playwright@latest

which scaffolds a config and an example test, and installs the browser binaries.

13.3 A sample E2E test

A Playwright test reads like a description of a user session:

// e2e/checkout.spec.js  (illustrative — requires the app running)
import { test, expect } from "@playwright/test";

test("a user can log in and check out", async ({ page }) => {
  // Navigate to the real app
  await page.goto("http://localhost:3000/login");

  // Fill and submit the login form, like a user
  await page.getByLabel("Email").fill("ada@example.com");
  await page.getByLabel("Password").fill("correcthorse");
  await page.getByRole("button", { name: "Log in" }).click();

  // Assert we reached the dashboard
  await expect(page.getByRole("heading", { name: "Welcome, Ada" })).toBeVisible();

  // Add an item and check out
  await page.getByRole("button", { name: "Add to cart" }).first().click();
  await page.getByRole("link", { name: "Cart" }).click();
  await page.getByRole("button", { name: "Checkout" }).click();

  // Assert the order confirmation appears
  await expect(page.getByText("Order confirmed")).toBeVisible();
});

Notice the query style is the same user-centric philosophy as React Testing Library (Section 9) — getByLabel, getByRole, getByText. That's not a coincidence; the "find things the way a user would" principle applies at every level. The difference is that this runs against the real, fully-deployed app in a real browser, not a single component in jsdom. await expect(...).toBeVisible() is Playwright's assertion, which auto-waits for the element (retrying until it appears or times out) — crucial for a real app where things load asynchronously.

13.4 The flakiness problem

E2E tests' biggest weakness is flakiness — tests that sometimes pass and sometimes fail without any code change. Because they involve real browsers, real timing, network, animations, and async loading, timing issues creep in: the test clicks a button before it's ready, or asserts on content before it has loaded. Flaky E2E tests are a genuine plague — they erode trust (Section 12) and waste time on false alarms. Mitigations:

  • Auto-waiting / web-first assertions — modern tools like Playwright automatically wait for elements to be actionable/visible and retry assertions until a timeout. Rely on these instead of fixed sleeps.
  • Never use arbitrary sleep(2000) — hard-coded waits are the classic flakiness cause: too short and it's flaky, too long and it's slow. Wait for a condition (an element visible, a network response), not a fixed time.
  • Control the environment — use stable test data, reset state between tests, and stub truly-external services (payment sandboxes) so runs are deterministic.
  • Isolate tests — each E2E test should set up its own state and not depend on another test's leftovers (the same isolation principle from Section 4, now at the whole-app level).

Even with care, E2E tests are more prone to flakiness than lower tests — which is a big reason you keep them few.

13.5 When E2E is worth it

E2E tests are expensive: slow to run (seconds to minutes each), slow to write, and higher-maintenance (UI changes break them, flakiness demands attention). So the pyramid's advice is to have relatively few, reserved for the cases where they earn their cost:

  • Critical user journeys — the flows that must work and would be catastrophic if broken: sign-up, login, checkout/payment, the core action of your product. A handful of E2E tests guarding these is worth a lot.
  • Smoke tests — a few tests that verify the app boots and the main pages load after a deploy, catching gross breakage (a bad build, a broken route, a failed integration) that unit tests can't.
  • Cross-cutting flows — behavior that spans multiple pages/systems and can't be verified any lower.

What E2E is not for: detailed logic and edge cases. Testing a pricing function's twelve branches through the browser would be absurdly slow and flaky — that belongs in fast unit tests. Push detail down to unit tests; reserve E2E for a few high-value, full-journey confidence checks. The right instinct: many unit tests, some integration tests, and a small, curated set of E2E tests covering the paths you most fear breaking.

An E2E test exercises the entire application as a real user does — through the real UI in a real browser, across the full stack (frontend + backend + usually a real database), verifying a complete user journey end to end. 🏋️ Exercise 13.2

What is test flakiness, why are E2E tests especially prone to it, and why is sleep(2000) a bad way to handle timing? What should you do instead?

Solution

Flakiness is when a test sometimes passes and sometimes fails without any change to the code — a nondeterministic result. E2E tests are especially prone because they involve real browsers, real network, real async loading, animations, and timing: the test might interact with or assert on the page before it's finished loading/rendering, so the outcome depends on unpredictable timing.

sleep(2000) (a fixed, arbitrary wait) is bad because it guesses at timing: if it's too short the test is still flaky (the page might not be ready), and if it's long enough to be safe it makes every run needlessly slow — and it's brittle across machines/CI where timing differs. Instead, wait for a condition, not a duration: use auto-waiting / web-first assertions (e.g., Playwright's expect(...).toBeVisible() retries until the element appears or a timeout hits), or explicitly wait for a specific element or network response. This makes tests both faster (proceed as soon as ready) and more reliable (don't proceed until actually ready).

SECTION 14

Testing Best Practices & Anti-Patterns

This final teaching section pulls everything together into the judgment that separates a suite that helps from one that hurts. Good tests share a handful of qualities; bad tests share a handful of smells. Knowing both — and being able to recognize them in your own code — is what makes you someone who tests well, not just someone who writes tests.

14.1 The qualities of a good test (FIRST)

A widely-cited checklist for good unit tests is FIRST:

  • Fast — runs in milliseconds. Fast tests get run constantly; slow ones get skipped. Speed keeps the feedback loop tight.
  • Isolated / Independent — depends on no other test and no shared mutable state; can run alone or in any order (Section 4). Isolation is what makes results trustworthy.
  • Repeatable — same result every time, on any machine, regardless of date, network, or environment. No hidden dependence on the clock, randomness, or external services (Section 5's dependency injection is how you achieve this). A repeatable test is the opposite of a flaky one.
  • Self-validating — has a clear pass/fail via assertions; you never manually inspect output to decide if it passed. The test decides.
  • Timely — written close to the code (ideally alongside or before it, per TDD), not bolted on months later when gaps have already shipped.

Beyond FIRST, the highest-order quality: a good test would fail if the code were wrong (Section 11). If breaking the code wouldn't break the test, the test isn't doing its job.

14.2 Test behavior, not implementation

The most important single principle, echoed throughout this course: test what the code does, not how it does it. Assert on observable outputs and behavior (return values, what's rendered, what a collaborator was called with), not on internal details (private variables, internal method calls, class names). Behavior-focused tests survive refactoring — you can completely rewrite the internals, and if behavior is preserved, the tests stay green. Implementation-focused tests break on every refactor, punishing you for improving your code and eventually training you to stop refactoring (or stop testing). The litmus test: "If I refactor this code without changing what it does, will this test still pass?" If no, the test is too coupled to implementation.

14.3 Common anti-patterns and test smells

Recognize these — they show up constantly:

  • The assertion-free test — runs code but asserts nothing meaningful (expect(true).toBe(true), or no expect at all). Provides coverage and zero protection (Section 11). Every test must make a meaningful assertion.
  • Testing implementation details — asserting on internals/private methods/class names; breaks on refactor (14.2).
  • The brittle/over-specified test — asserts far more than it cares about (the exact whole object when only one field matters, exact call counts that don't matter), so trivial unrelated changes break it. Assert on what the behavior actually requires; use toMatchObject for partial matches.
  • Flaky tests — intermittent pass/fail from timing, order-dependence, real network, or uncontrolled randomness/time. Worse than no test: they erode trust in the whole suite until people ignore red. Fix (usually via isolation, auto-waiting, or injecting the clock/random) or delete.
  • Over-mocking — mocking so much that tests only verify mocks calling mocks, passing while real code is broken (Section 6).
  • Interdependent tests — one test depends on another running first (shared state, execution order). Break isolation → order-dependent flakiness (Section 4).
  • The slow test — a "unit" test that hits a real DB/network/filesystem, dragging the suite down. Push slowness to integration/E2E layers; keep units fast.
  • Testing the framework/library — writing tests that really just verify React or the standard library works. Test your logic, not your dependencies (Section 12).
  • Copy-paste tests — near-identical tests duplicated instead of parameterized. Use it.each (Section 4) and shared setup.

14.4 Practical do's and don'ts

Distilled guidance:

  • Do give tests descriptive, behavior-focused names; a failure should read like a sentence about what broke (Section 2).
  • Do test edge cases and boundaries deliberately — empty, zero, negative, off-by-one, error paths — that's where bugs live (Section 11).
  • Do keep each test focused on one behavior; a test that checks ten things is hard to read and localizes failures poorly.
  • Do write a regression test for every bug you fix (Section 8).
  • Do keep tests simple — test code should be more obvious than production code, not clever. If a test needs its own complex logic, that logic can have bugs too.
  • Don't put logic (conditionals, loops computing the expected value) in tests when you can avoid it — hard-code expected values so the test is a plain, checkable statement of fact.
  • Don't chase a coverage number; chase meaningful assertions on code that matters (Section 11).
  • Don't leave .only committed or ignore flaky/failing tests (Sections 4, 12).

14.5 A real example — smelly vs clean

// ❌ SMELLY — assertion-free, implementation-coupled, over-specified, unfocused
it("test user", () => {
  const svc = new UserService(new RealDatabase());   // slow: real DB in a "unit" test
  svc._cache = {};                                    // reaching into a private field
  const u = svc.createUser("Ada", "ada@x.com");
  // asserts the ENTIRE object incl. volatile fields → brittle
  expect(u).toEqual({ id: u.id, name: "Ada", email: "ada@x.com", createdAt: u.createdAt });
  svc.deleteUser(u.id);                               // no assertion — just runs code
});

// ✅ CLEAN — fast, isolated, focused, behavior-based, meaningful assertions
import { describe, it, expect, vi } from "vitest";
import { createUser } from "./userService.js";

describe("createUser", () => {
  it("returns a user with the given name and email", () => {
    const user = createUser("Ada", "ada@example.com", {
      generateId: () => "u1",          // injected → repeatable
      now: () => new Date("2024-01-01"),
    });
    expect(user).toMatchObject({ name: "Ada", email: "ada@example.com" }); // only what matters
  });

  it("rejects an invalid email", () => {
    expect(() => createUser("Ada", "not-an-email", { generateId: () => "u1", now: () => new Date() }))
      .toThrow(/invalid email/i);      // behavior (error path), meaningful assertion
  });
});

The clean version embodies FIRST (fast — no real DB; isolated; repeatable via injected id/clock; self-validating), tests behavior not internals (no private-field poking), asserts only what matters (toMatchObject), stays focused (one behavior each), and covers an edge case (invalid email). That's the whole course in one contrast: not just having tests, but having good ones.

🏋️ Exercise 14.4

If it can't be made reliable and isn't worth the effort, delete it — an honest gap is better than a test that trains people to ignore failures. Never leave it silently flaky.

15.1 The design — testable from the start

CartKit is deliberately structured for testability (Section 5): pure logic in the middle, the single side effect injected at the edge.

src/
  cart.js       pure cart core — add / remove / setQuantity / itemCount / subtotal
  discount.js   pure discount math — percent & fixed, clamped at 0, rounded to cents
  checkout.js   the async boundary (validateCode) + checkout() integration

The only external dependency in the whole system is a discount-code lookup, and it's injected as fetchCode. That one choice is what makes the async parts fast and deterministic to test — we pass a vi.fn() in tests and the real API call in production. Here's the core:

// src/cart.js — pure cart operations (excerpt)
export function createCart() { return { lines: [] }; }

export function addItem(cart, item, qty = 1) {
  if (qty <= 0) throw new Error("Quantity must be positive");
  if (typeof item.price !== "number" || item.price < 0) throw new Error("Item needs a non-negative price");
  const existing = cart.lines.find((l) => l.id === item.id);
  if (existing) existing.qty += qty;
  else cart.lines.push({ id: item.id, name: item.name, price: item.price, qty });
  return cart;
}

export function subtotal(cart) {
  const raw = cart.lines.reduce((sum, l) => sum + l.price * l.qty, 0);
  return Math.round(raw * 100) / 100;
}

// src/discount.js — pure discount math (excerpt)
export function applyDiscount(amount, discount) {
  if (!discount) return round(amount);
  let result;
  if (discount.type === "percent") {
    if (discount.value < 0 || discount.value > 100) throw new Error("Percent must be 0..100");
    result = amount * (1 - discount.value / 100);
  } else if (discount.type === "fixed") {
    if (discount.value < 0) throw new Error("Fixed discount must be non-negative");
    result = amount - discount.value;
  } else {
    throw new Error(`Unknown discount type: ${discount.type}`);
  }
  return round(Math.max(0, result));   // never below zero
}

// src/checkout.js — the async boundary + integration
import { subtotal } from "./cart.js";
import { applyDiscount } from "./discount.js";

export async function validateCode(code, { fetchCode }) {   // ← boundary injected
  if (!code) return null;
  const discount = await fetchCode(code.trim().toUpperCase());
  return discount ?? null;
}

export async function checkout(cart, code, deps) {
  const sub = subtotal(cart);
  const discount = code ? await validateCode(code, deps) : null;
  const total = applyDiscount(sub, discount);            // real cart + real discount math
  return { subtotal: sub, discount, total };
}

Notice: checkout runs the real cart and real discount logic — only the fetchCode lookup is ever mocked. That's "mock the edges, test the middle for real" (Section 6) baked into the architecture.

15.2 Unit tests — the cart core (isolation, edge cases, errors)

// test/cart.test.js (excerpt)
import { describe, it, expect, beforeEach } from "vitest";
import { createCart, addItem, removeItem, setQuantity, itemCount, subtotal } from "../src/cart.js";

const PEN = { id: "pen", name: "Pen", price: 1.5 };

describe("cart core", () => {
  let cart;
  beforeEach(() => { cart = createCart(); });   // isolation: fresh cart per test (Section 4)

  describe("addItem", () => {
    it("increments quantity when the same item is added again", () => {
      addItem(cart, PEN, 1);
      addItem(cart, PEN, 2);
      expect(cart.lines).toHaveLength(1);       // merged, not duplicated
      expect(itemCount(cart)).toBe(3);
    });
    it("throws on non-positive quantity", () => {
      expect(() => addItem(cart, PEN, 0)).toThrow(/positive/i);   // error path
    });
    it("throws on a negative price", () => {
      expect(() => addItem(cart, { id: "x", name: "X", price: -1 })).toThrow(/price/i);
    });
  });

  describe("setQuantity", () => {
    it("removes the item when set to 0 or less", () => {
      addItem(cart, PEN, 2);
      setQuantity(cart, "pen", 0);
      expect(cart.lines).toHaveLength(0);        // edge behavior
    });
    it("throws for an unknown id", () => {
      expect(() => setQuantity(cart, "ghost", 2)).toThrow(/no item/i);
    });
  });
});

Thirteen unit tests cover the happy paths, the merge behavior, the quantity-to-zero edge, and every error condition — each in isolation via beforeEach, each asserting an exact outcome.

15.3 Exhaustive pure-logic tests — discounts (tables + boundaries)

Because applyDiscount is pure, we can hammer it with a data table (it.each) and nail the boundaries — exactly where discount bugs hide:

// test/discount.test.js (excerpt)
import { describe, it, expect } from "vitest";
import { applyDiscount } from "../src/discount.js";

describe("applyDiscount", () => {
  it("returns the amount unchanged when there is no discount", () => {
    expect(applyDiscount(100, null)).toBe(100);
  });

  it.each([
    [100, 10, 90],
    [50, 100, 0],       // 100% off → 0 (boundary)
    [99.99, 0, 99.99],  // 0% off → unchanged (boundary)
  ])("percent: %d at %d%% → %d", (amount, value, expected) => {
    expect(applyDiscount(amount, { type: "percent", value })).toBe(expected);
  });

  it.each([
    [100, 15, 85],
    [10, 25, 0],        // fixed larger than amount → clamped to 0, never negative (boundary)
  ])("fixed: %d minus $%d → %d", (amount, value, expected) => {
    expect(applyDiscount(amount, { type: "fixed", value })).toBe(expected);
  });

  it("throws on an out-of-range percent", () => {
    expect(() => applyDiscount(100, { type: "percent", value: 150 })).toThrow(/0\.\.100/);
  });
  it("throws on an unknown discount type", () => {
    expect(() => applyDiscount(100, { type: "bogus", value: 1 })).toThrow(/unknown discount type/i);
  });
});

The boundaries (0%, 100%, an oversized fixed discount clamped to zero) are the whole point — these are the cases a weak suite skips and a strong one insists on (Section 11).

15.4 Async + mocks + integration — checkout

The richest file: it mocks only the boundary, tests both success and failure paths, and finishes with true integration tests that run the real cart and discount math together.

// test/checkout.test.js (excerpt)
import { describe, it, expect, vi } from "vitest";
import { validateCode, checkout } from "../src/checkout.js";
import { createCart, addItem } from "../src/cart.js";

describe("validateCode (async, mocked boundary)", () => {
  it("normalizes the code and returns the discount from the lookup", async () => {
    const fetchCode = vi.fn().mockResolvedValue({ type: "percent", value: 10 });
    const discount = await validateCode("  save10 ", { fetchCode });
    expect(discount).toEqual({ type: "percent", value: 10 });
    expect(fetchCode).toHaveBeenCalledWith("SAVE10");        // trimmed + upper-cased
  });

  it("returns null for an empty code without calling the lookup", async () => {
    const fetchCode = vi.fn();
    expect(await validateCode("", { fetchCode })).toBeNull();
    expect(fetchCode).not.toHaveBeenCalled();               // spy: no wasted call
  });

  it("propagates a lookup failure", async () => {
    const fetchCode = vi.fn().mockRejectedValue(new Error("service down"));
    await expect(validateCode("SAVE10", { fetchCode })).rejects.toThrow("service down");
  });
});

describe("checkout (integration: cart + discount + async validation)", () => {
  function sampleCart() {
    const cart = createCart();
    addItem(cart, { id: "pen", name: "Pen", price: 1.5 }, 2);  // 3.00
    addItem(cart, { id: "pad", name: "Pad", price: 4.0 }, 1);  // 4.00
    return cart;                                                // subtotal 7.00
  }

  it("applies a validated percent code to the real subtotal", async () => {
    const fetchCode = vi.fn().mockResolvedValue({ type: "percent", value: 50 });
    const result = await checkout(sampleCart(), "HALF", { fetchCode });
    expect(result).toEqual({ subtotal: 7, discount: { type: "percent", value: 50 }, total: 3.5 });
  });

  it("never charges below zero with an oversized fixed discount", async () => {
    const fetchCode = vi.fn().mockResolvedValue({ type: "fixed", value: 999 });
    const result = await checkout(sampleCart(), "MEGA", { fetchCode });
    expect(result.total).toBe(0);
  });
});

The integration tests are the payoff: checkout runs the real subtotal and real applyDiscount, with only fetchCode mocked. If the cart and discount modules ever disagreed about shapes or math, these would catch it — the "all my mocks agreed" bug that pure unit tests miss (Section 10).

15.6 Your capstone challenge

Build your own version, ideally test-first (Section 8). The full CartKit source and tests ship with this course as a reference, but you'll learn far more by writing it yourself. Suggested path:

  1. Set up a fresh project with Vitest (npm i -D vitest), scripts for test and coverage.
  2. TDD the pure core. Write a failing test for addItem, make it pass, refactor; repeat for removeItem, setQuantity, subtotal. Include edge cases and error paths from the start.
  3. TDD the discount logic with it.each tables, insisting on the boundaries (0%, 100%, over-clamp, invalid inputs).
  4. Design for injection. Make the code-lookup a dependency you pass in, then test validateCode with vi.fn() — success, unknown code, empty code, and a rejected lookup.
  5. Write the integration test for checkout: real cart + real discount, only the lookup mocked. Assert the full result object.
  6. Run coverage. Then look at which lines (if any) are uncovered and decide whether each gap is a meaningful case worth a test — not to hit a number, but to catch real bugs.

The through-lines worth carrying with you:

  • Testing is about confidence, not bug-hunting. The real payoff is every day after you write a test, when it lets you and your team change code without fear. Tests are what keep a codebase malleable as it grows.
  • A good test would fail if the code were wrong. This one sentence is the master test of test quality. Coverage counts execution; this counts protection. Chase meaningful assertions — especially on boundaries — not a coverage number.
  • Test behavior, not implementation. Assert on what the code does, not how. Behavior-focused tests survive refactors and fail only when something real breaks; implementation-coupled tests punish you for improving your code.
  • Testable code is well-designed code. Testing friction is a design smell. Pure functions in the middle, side effects and dependencies injected at the edges — that structure makes code both easy to test and easy to understand.
  • Mock the edges, run the middle for real. Mock the network, database, clock, and third-party APIs; run your own logic. Over-mocking produces tests that pass while the code is broken.
  • Match the layer to the job. Many fast unit tests for detailed logic and edge cases, integration tests for the seams where contract bugs live, a few E2E tests for the critical journeys you most fear breaking.
  • Trust is the whole point. A suite people trust — fast, isolated, repeatable, green-means-good, red-means-real — is worth more than a large suite riddled with flaky or meaningless tests. Fix or delete anything that erodes that trust.

Where to go next. Testing is a craft you deepen by doing. Add a real test suite to a project you already have — start with the business logic and every bug you fix. Try strict TDD on a well-specified feature and feel how it shapes your design. Wire your tests into CI so they protect your main branch automatically. Explore your language's ecosystem beyond this course's stack (pytest, Go's testing, JUnit, RSpec) — the concepts transfer directly; only the syntax changes. And when a suite feels painful, ask whether the code is telling you something about its design.

The habit that compounds fastest: every time you fix a bug, write the failing test first. It costs a minute, proves your fix, and guards against regression forever. Do that consistently and your suite grows exactly where it matters, one real bug at a time.
