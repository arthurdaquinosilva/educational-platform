SECTION 1

Beyond the Basics: What Makes a Script "Advanced"

Before the techniques, the mindset. What separates a throwaway script from a professional tool isn't cleverness — it's robustness, clarity, and predictability. This section sets the frame: the anatomy of a serious script, choosing your shell, and the standards the rest of the course upholds.

1.1 From "works once" to "works always"

A beginner's script is written to work this time, on this input, on this machine. A professional script is written to work every time, on unexpected input, on other machines, and to fail clearly when it can't. The gap between them is entirely about handling the cases that aren't the happy path: the missing file, the argument with a space, the command that fails halfway, the interrupt signal, the empty variable. Everything in this course is, in one way or another, about closing that gap. The advanced scripter's habit of mind is "what happens when this goes wrong?" — asked at every line.

Concretely, a professional script:

  • Fails fast and loudly when something is wrong, rather than limping onward in a broken state.
  • Cleans up after itself — no leftover temp files, even if interrupted.
  • Quotes everything so spaces and special characters don't break it.
  • Validates its input and tells the user clearly how to use it.
  • Is readable — a teammate (or you in six months) can understand it.
  • Is testable — its behavior can be verified automatically.

1.2 Anatomy of a professional script

Here's the skeleton the rest of this course fills in — the shape of a serious Bash script (this ran):

#!/usr/bin/env bash
#
# greet — a tiny example of professional script structure.
# Usage: greet [-u] NAME

set -euo pipefail            # strict mode (Section 2)

# --- constants ---
readonly SCRIPT_NAME="${0##*/}"     # basename of this script

# --- functions ---
usage() {
  cat <<EOF
Usage: $SCRIPT_NAME [-u] NAME
  -u   shout the greeting in uppercase
EOF
}

main() {
  local upper=0
  # (option parsing goes here — Section 7)
  local name="${1:-world}"
  local message="Hello, $name!"
  [[ "$upper" -eq 1 ]] && message="${message^^}"    # uppercase (Section 4)
  printf '%s\n' "$message"
}

main "$@"                     # pass all arguments to main

Notice the structure even in this trivial example: a descriptive header comment, strict mode, constants marked readonly, a usage function, logic wrapped in a main function, and main "$@" at the bottom passing the real arguments in. Wrapping logic in main (rather than leaving it at the top level) keeps variables scoped, makes the script readable top-to-bottom, and — crucially — means the whole script is parsed before anything runs (so a syntax error near the end can't leave you half-executed). This is a small habit with big payoff.

1.3 Choosing your shell — bash vs POSIX sh

A foundational decision: which shell your script targets.

  • Bash is the most common interactive shell on Linux and macOS (though macOS ships an old version). It has powerful features — arrays, associative arrays, [[ ]] tests, advanced parameter expansion, process substitution — that make scripting far nicer. Most of this course uses Bash.
  • POSIX sh is the portable standard that every Unix-like system provides (often as dash, a minimal, fast shell). It lacks Bash's niceties but runs anywhere — critical for system scripts, container init scripts, and cross-platform tools.

The practical rule: use Bash for your own scripts and tooling (you get the good features, and Bash is almost always available), and write portable POSIX sh only when you specifically need maximum portability — a script that must run in minimal environments (Alpine containers, embedded systems, other Unixes). The dangerous mistake is putting #!/bin/sh at the top and then using Bash-only features ("bashisms") — it works on systems where /bin/sh is Bash and breaks on systems where it's dash. Section 13 covers portability in depth, tested in a real POSIX shell. For now: declare the shell you actually mean. If you use Bash features, your shebang must say bash.

1.4 The shebang, precisely

The first line — the shebang — tells the system which interpreter to run:

#!/usr/bin/env bash     # portable: finds bash via PATH (recommended)
#!/bin/bash             # explicit path (works if bash is there; less portable)
#!/bin/sh               # POSIX shell — ONLY if you write portable sh

#!/usr/bin/env bash is generally preferred because it locates bash through the user's PATH rather than assuming a fixed location (Bash lives at /bin/bash on Linux but /usr/local/bin/bash or elsewhere on other systems). The rule that prevents the most pain: the shebang must match the features you use. Write #!/usr/bin/env bash if you use any Bash feature. Reserve #!/bin/sh for scripts you've deliberately kept POSIX-compliant. Mismatching them is the single most common portability bug in shell scripting.

🏋️ Exercise 1.2

Why is it recommended to wrap a script's logic in a main function called with main "$@" at the bottom, rather than writing the logic at the top level?

Solution

Wrapping logic in a main function (and calling main "$@" at the very bottom) has several benefits:

  1. The whole script is parsed before anything runs. Because execution only begins at the final main "$@" line, Bash reads and parses the entire file first. A syntax error anywhere — even near the end — is caught before any command executes, so you can't get halfway through a partial run. (Without this, Bash executes line by line and could run several commands before hitting a later parse error.)
  2. Variable scoping. Variables declared local inside main (and other functions) don't leak into the global environment, keeping things clean and avoiding accidental clobbering.
  3. Readability. The reader sees the high-level flow in main and can read the script top-to-bottom (helpers defined first, main orchestrating them, invocation last).
  4. Testability. Logic organized into functions (including main) can be sourced and tested individually.

main "$@" specifically forwards all the script's original arguments, unmodified and correctly quoted, into the function.

🏋️ Exercise 1.3

Explain the most common portability bug involving the shebang line. Why does #!/bin/sh with Bash-only features "work on my machine" but break elsewhere?

Solution

The most common portability bug is writing #!/bin/sh at the top but then using Bash-only features ("bashisms") — such as arrays, [[ ]], or advanced parameter expansion.

It "works on my machine" but breaks elsewhere because /bin/sh is not the same program on every system. The fix is to make the shebang match the features used: if the script uses any Bash feature, its shebang must be #!/usr/bin/env bash (or #!/bin/bash). Reserve #!/bin/sh strictly for scripts written to be POSIX-compliant. Declaring the shell you actually mean prevents the mismatch entirely.

SECTION 2

Strict Mode & Robust Error Handling

The foundation of a reliable script is how it behaves when a command fails. By default, the shell is dangerously forgiving — it plows ahead after errors. This section makes your scripts strict: failing fast, catching unset variables, and handling errors deliberately. This is the single highest-leverage upgrade you can make to your scripts, and every example here ran.

2.1 The default shell is too forgiving

By default, a shell script continues executing after a command fails. Consider the danger:

#!/usr/bin/env bash
cd "$build_dir"          # if this FAILS (dir doesn't exist)...
rm -rf ./*               # ...this runs in whatever directory you were in!

If cd fails (typo, missing directory), the script doesn't stop — it runs rm -rf ./* in the current directory instead of the intended one. This exact pattern has destroyed real systems. The default behavior — ignore failures and continue — is almost never what you want in a script that does real work. Strict mode fixes this.

2.2 Strict mode: set -euo pipefail

The near-universal preamble for robust Bash scripts (introduced in the DevOps course; here we go deeper on its subtleties):

set -euo pipefail
  • set -e (errexit) — exit immediately if a command fails (returns non-zero). No more plowing ahead after cd fails.
  • set -u (nounset) — treat referencing an unset variable as an error. Catches typos: $bulid_dir (misspelled) would silently be empty without -u; with it, the script errors.
  • set -o pipefail — a pipeline's exit status is that of the last command to fail, not just the last command. Without it, false | true succeeds (status 0) because true was last; with it, the pipeline correctly reports failure.

Together they transform the shell from "ignore errors and continue" to "stop the moment something is wrong." Add them to the top of essentially every Bash script you write. But -e in particular has subtleties you must understand, or it will surprise you.

2.3 The subtleties of set -e — where it does not fire

set -e is not magic, and knowing its blind spots is what separates people who think their script is safe from people whose scripts are safe. -e does not trigger an exit in several cases:

set -e

# 1. Commands in a condition are EXEMPT (that's how if/while work):
if grep -q pattern file; then ... fi     # grep failing is fine here — it's a test
grep -q pattern file && do_something      # left side of && is a test — no exit on fail

# 2. Commands whose result is negated or tested:
! some_command                           # negated — won't exit
some_command || handle_error             # part of || — won't exit (you're handling it)

# 3. Commands in a pipeline other than the last (WITHOUT pipefail):
false | true                             # exits 0 without pipefail — false is "hidden"

# 4. Functions called in a condition lose -e INSIDE them (a notorious gotcha):
check() { false; echo "still runs!"; }   # if check is called in `if check; then`,
if check; then ... fi                    # -e is disabled inside check → "still runs!" prints

The critical takeaways: -e is suppressed for any command that's part of a test (an if/while condition, an operand of &&/||, or negated with !) — which is by design, since those are supposed to be allowed to "fail." And most infamously, set -e is disabled inside a function when that function is called in a conditional context. These aren't bugs; they're the defined behavior. Knowing them means you don't rely on -e where it won't help, and you add explicit checks where it matters.

2.4 Explicit error handling — beyond -e

Because -e has gaps, robust scripts also handle errors explicitly. The core patterns:

# Handle a specific command's failure with || :
command || { echo "command failed" >&2; exit 1; }

# Provide a fallback / default action:
result=$(some_command) || result="default"

# A reusable die() helper — print an error and exit:
die() { echo "Error: $*" >&2; exit 1; }
[[ -f "$config" ]] || die "config file not found: $config"

# Check a required command exists before using it:
command -v jq >/dev/null 2>&1 || die "jq is required but not installed"

The die() helper (print to stderr, exit non-zero) is one of the most useful idioms in shell scripting — it makes validation read cleanly: [[ condition ]] || die "message". Note errors go to stderr (>&2), keeping them separate from a script's normal output so that output can be piped or captured without the error messages polluting it. Explicit handling is how you cover the cases -e misses and how you give useful error messages instead of a bare non-zero exit.

2.5 The ERR trap — react to any failure

For centralized error handling, Bash offers the ERR trap, which runs a command whenever a command fails (subject to the same rules as -e):

#!/usr/bin/env bash
set -euo pipefail

# Run this handler on ANY failing command: report where it happened.
trap 'echo "Error on line $LINENO (exit $?)" >&2' ERR

process_data      # if this fails, the trap fires with the line number

The ERR trap fires on the same conditions -e acts on, and is invaluable for logging: you can report the line number ($LINENO), the failed command, and the exit code — turning a silent failure into a diagnostic message. It's the foundation of good error reporting in larger scripts, and it pairs with the EXIT trap (next section) for cleanup. Together, strict mode + explicit die() checks + an ERR trap give you defense in depth: fail fast by default, handle known failure points explicitly, and report anything that slips through.

Explain each of the three options in set -euo pipefail. For pipefail specifically, give an example where its absence hides a failure.

Solution
  • set -e (errexit) — the script exits immediately if any command returns a non-zero (failure) status, instead of continuing. Prevents "plowing ahead" after a failure (e.g., running destructive commands after a failed cd).
  • set -u (nounset) — referencing an unset variable is treated as an error rather than expanding to an empty string. Catches typos and missing variables (e.g., a misspelled $bulid_dir errors instead of silently becoming empty).

🏋️ Exercise 2.2

set -e does not always cause an exit on failure. Name three situations where a failing command will not trigger set -e, and explain why the function-in-a-conditional case is especially surprising.

Solution

Situations where set -e does not trigger an exit on a failing command:

  1. Commands used as a condition — in an if/while/until test, or as the left operand of &&/||, or negated with !. E.g., if grep -q x file; then — grep failing is the point of the test, so -e is suppressed. (cmd || fallback and ! cmd likewise won't exit.)
  2. Non-final commands in a pipeline (without pipefail) — e.g., in false | true, the false failing doesn't trigger -e; only the pipeline's final status matters, which is true's success.
  3. Commands whose failure is otherwise "consumed" — e.g., a command in a $(...) used in an assignment that's part of a tested construct, or any command explicitly followed by || something.

The function-in-a-conditional case is especially surprising because when a function is called in a conditional context (e.g., if my_func; then), set -e is disabled for all the commands inside that function, not just the call itself. So a function that fails partway will keep executing its remaining commands rather than aborting:

check() { false; echo "still runs!"; }   # `false` doesn't stop the function
if check; then ...; fi                    # prints "still runs!" — -e is off inside check

This is defined POSIX/Bash behavior, but it surprises people who assume -e protects code inside every function unconditionally. The lesson: don't rely on -e inside functions called conditionally — add explicit checks.

SECTION 3

Traps & Signal Handling

A robust script cleans up after itself — even when interrupted. If your script creates a temp file and the user hits Ctrl-C, that temp file shouldn't be left behind. Traps let a script react to signals and to its own exit, running cleanup or handling interruptions gracefully. This is a hallmark of professional scripts, and every example here ran.

3.1 Signals — how the outside world interrupts your script

A signal is an asynchronous notification sent to a process. The ones that matter for scripting:

  • INT (SIGINT) — sent when the user presses Ctrl-C. Default action: terminate.
  • TERM (SIGTERM) — the polite "please shut down" signal (what kill sends by default, what orchestrators send to stop containers). Default action: terminate.
  • HUP (SIGHUP) — terminal closed / "hang up." Often used to signal "reload config."
  • EXIT — not a real signal, but a pseudo-signal Bash treats like one: it fires whenever the script exits for any reason (normal end, exit, or a fatal error). This is the key to reliable cleanup.
  • KILL (SIGKILL, kill -9) — cannot be trapped or ignored; it force-kills immediately. Nothing your script can do about it, by design.

By default these terminate your script abruptly — mid-operation, temp files intact, no cleanup. Traps let you intervene.

3.2 The trap command

trap registers a command (or function) to run when a signal or pseudo-signal occurs:

trap 'command to run' SIGNAL_NAME [MORE_SIGNALS...]

# Run cleanup when the script exits for ANY reason:
trap 'rm -f "$tmpfile"' EXIT

# React to Ctrl-C specifically:
trap 'echo "Interrupted!"; exit 130' INT

# Handle several signals with one handler:
trap 'echo "shutting down"; cleanup' INT TERM

The syntax is trap 'handler' SIGNALS. The handler is a command string (often a call to a cleanup function). You can trap multiple signals with one statement. The single most valuable trap is on EXIT, because it runs no matter how the script ends — making it the reliable place for cleanup.

3.3 Cleanup on EXIT — the essential pattern

The canonical use of traps: guarantee that temporary resources are cleaned up, whether the script succeeds, fails, or is interrupted. (This ran.)

#!/usr/bin/env bash
set -euo pipefail

# Create a temp file and register cleanup IMMEDIATELY, before doing work.
tmpfile="$(mktemp)"
trap 'rm -f "$tmpfile"' EXIT

echo "Working with temp file: $tmpfile"
# ... do work that writes to $tmpfile ...
# Even if the next line fails, or the user hits Ctrl-C, or we exit normally,
# the EXIT trap fires and removes the temp file.
process_stuff > "$tmpfile"

Because the EXIT trap fires on every exit path — success, error (thanks to set -e), or signal — the temp file is always removed. The discipline: register the cleanup trap immediately after creating the resource, so there's no window where an early failure leaks it. For multiple resources, a cleanup function is cleaner:

cleanup() {
  rm -f "$tmpfile" "$lockfile"
  [[ -n "${tmpdir:-}" ]] && rm -rf "$tmpdir"
}
trap cleanup EXIT

This is the pattern that distinguishes a tidy tool from one that litters /tmp with leftover files every time it's interrupted.

3.4 Graceful shutdown and interrupt handling

For long-running scripts (servers, watchers, batch jobs), you often want to handle INT/TERM deliberately — finish the current item, log a message, then exit cleanly:

#!/usr/bin/env bash
set -euo pipefail

running=1
trap 'echo "Received shutdown signal, finishing current task..."; running=0' INT TERM

while [[ "$running" -eq 1 ]]; do
  do_one_unit_of_work
  # On Ctrl-C, the trap sets running=0; the loop finishes its current
  # iteration and then exits cleanly instead of dying mid-task.
done
echo "Shut down cleanly."

Here the trap doesn't exit immediately — it sets a flag that the main loop checks, allowing the current unit of work to complete before shutting down. This is exactly how well-behaved services handle TERM from an orchestrator (Section 10 of the DevOps course): catch the signal, stop accepting new work, finish what's in flight, exit. By convention, a script killed by a signal exits with code 128 + signal_number (so Ctrl-C/INT, signal 2, gives 130) — worth mimicking with exit 130 if you exit directly from an INT handler.

3.5 A real example — safe temp workspace

Tying traps together with a realistic pattern — a script that needs a whole temp directory and guarantees its removal (this ran):

#!/usr/bin/env bash
set -euo pipefail

workdir="$(mktemp -d)"                    # a temp DIRECTORY
trap 'rm -rf "$workdir"' EXIT             # guaranteed cleanup, any exit path

echo "Using workspace: $workdir"
# Do multi-step work safely inside the throwaway workspace:
echo "data" > "$workdir/input.txt"
tr 'a-z' 'A-Z' < "$workdir/input.txt" > "$workdir/output.txt"
cat "$workdir/output.txt"
# On exit — success, error, or Ctrl-C — the whole workdir is removed.

mktemp -d creates a uniquely-named temp directory (avoiding collisions and race conditions from hardcoded names like /tmp/mydir), and the EXIT trap guarantees the entire tree is cleaned up. This is the safe, professional way to do scratch work: unique temp location, immediate cleanup trap, all work confined inside. No leftovers, no collisions, no matter how the script ends.

🏋️ Exercise 3.1

What is special about the EXIT pseudo-signal, and why is it the ideal place for cleanup? Why should you register the cleanup trap immediately after creating a resource rather than later?

Solution

The EXIT pseudo-signal is special because it fires whenever the script exits for any reason — normal completion, an explicit exit, or a fatal error (including one triggered by set -e) — and after signal handlers run. (It's not a real OS signal; Bash triggers it on every exit path.)

This makes it the ideal place for cleanup because you only need one trap to guarantee cleanup regardless of how the script ends: you don't have to enumerate every possible exit or catch every error individually. Register trap cleanup EXIT once and the cleanup runs on success, on failure, and (after their handlers) on interrupts.

You should register the cleanup trap immediately after creating the resource (e.g., right after mktemp) so there is no window in which the resource exists but the trap isn't set. If you created a temp file and then did other work before setting the trap, a failure or interrupt in that gap would exit the script and leak the temp file — the very thing the trap is meant to prevent. Create-then-trap, back to back, closes that gap.

🏋️ Exercise 3.2

Explain the "set a flag" pattern for handling INT/TERM in a long-running loop. Why is it often better than exiting immediately from the signal handler? What exit code convention applies to a script terminated by a signal?

Solution

The "set a flag" pattern traps INT/TERM with a handler that doesn't exit but instead sets a variable (e.g., running=0); the main loop checks that flag each iteration and exits cleanly once it's set:

It's often better than exiting immediately because it allows the script to finish the current unit of work and shut down gracefully rather than being killed mid-operation. Dying instantly could leave data half-written, a resource locked, or a task partially done. By deferring the exit to a clean point in the loop, you stop accepting new work but complete what's in flight — exactly how well-behaved services respond to TERM from an orchestrator (finish in-flight requests, then stop).

Exit code convention: a script terminated by a signal exits with 128 + signal_number. Since INT (Ctrl-C) is signal 2, that's 130; TERM is signal 15, giving 143. If you exit directly from an INT handler, it's conventional to exit 130 to reflect this.

Why is mktemp -d preferred over a hardcoded temp directory like /tmp/myapp? Write the two-line pattern for a guaranteed-cleaned-up temp directory.

Solution

mktemp -d is preferred over a hardcoded path like /tmp/myapp because: - Uniqueness / no collisions — mktemp -d creates a directory with a guaranteed-unique random name, so two concurrent runs (or a leftover from a previous run) don't collide or interfere.   INT  (Ctrl-C, sig 2)   TERM (kill/orchestrator stop, sig 15)   HUP (terminal closed / reload)

  trap 'rm -f "$tmp"' EXIT            # cleanup on any exit

Signal exit code = 128 + signal number (INT→130, TERM→143).

SECTION 4

Parameter Expansion & Variable Mastery

Bash's parameter expansion is a small language for transforming variables — providing defaults, stripping prefixes and suffixes, substituting text, slicing, and changing case — all without spawning external commands like sed or cut. Mastering it makes scripts shorter, faster, and more robust. Every expansion here was run.

4.1 Why parameter expansion matters

Beginners reach for external tools to manipulate strings: echo "$path" | sed 's/.*\///' to get a filename, echo "$name" | tr a-z A-Z to uppercase. Each external command spawns a process — slow in a loop, and more code to get wrong. Bash can do most string manipulation natively with parameter expansion, which is faster (no subprocess), more portable within Bash, and often clearer once you know the syntax. It's one of the biggest markers of shell fluency. The general form is ${variable...} with operators inside the braces.

4.2 Defaults and required values

Providing fallbacks and enforcing required variables:

${var:-default}    # use $var, or "default" if var is unset OR empty
${var:=default}    # use $var, or SET var to "default" and use it
${var:?message}    # use $var, or error with "message" and exit if unset/empty
${var:+alternate}  # use "alternate" if var IS set, else empty (the inverse)

# Examples (ran):
name="${1:-world}"                     # default a positional arg
: "${CONFIG:?CONFIG must be set}"      # require a variable, clean error if missing
port="${PORT:-3000}"                   # optional config with a default

The :- form is the everyday default-provider. The :? form is the clean way to require a variable (it aborts with your message under set -u), replacing verbose if [[ -z ... ]] checks. The :+ form is the useful inverse ("if this is set, use that"), handy for conditionally adding flags. Note the : makes them treat empty-string the same as unset; without the colon (${var-default}) they only act on truly unset variables.

4.3 Stripping prefixes and suffixes

Removing parts of a string by pattern — the native replacement for basename, dirname, and extension manipulation:

${var#pattern}     # remove SHORTEST match of pattern from the START
${var##pattern}    # remove LONGEST match from the START
${var%pattern}     # remove SHORTEST match from the END
${var%%pattern}    # remove LONGEST match from the END

# Examples (ran), with path="/home/user/archive.tar.gz":
"${path##*/}"      # archive.tar.gz   (longest */ from start = basename)
"${path%/*}"       # /home/user       (shortest /* from end   = dirname)
"${path%.*}"       # /home/user/archive.tar  (drop last extension)
"${path##*.}"      # gz               (the last extension)

The mnemonic: # points to the left on the keyboard (start of string), % points to the right (end); doubled (##/%%) means greedy (longest match). These four cover the vast majority of path and filename manipulation — ${path##*/} is basename, ${path%/*} is dirname, and ${file%.*}/${file##*.} split a name from its extension — all with no external process.

4.4 Substitution, slicing, length, and case

The rest of the toolkit:

${var/find/replace}    # replace FIRST occurrence of find with replace
${var//find/replace}   # replace ALL occurrences (doubled slash = global)
${var/#prefix/repl}    # replace only if at the start
${var/%suffix/repl}    # replace only if at the end

${var:offset}          # substring from offset to end
${var:offset:length}   # substring: length chars from offset
${#var}                # LENGTH of the string (number of characters)

${var^^}               # UPPERCASE all       (Bash 4+)
${var,,}               # lowercase all        (Bash 4+)
${var^}                # uppercase first char only

# Examples (ran):
s="hello world hello"
"${s//hello/hi}"       # hi world hi        (global replace)
"${s:6}"               # world hello        (substring from index 6)
"${s:0:5}"             # hello              (first 5 chars)
"${#s}"                # 17                 (length)
"${s^^}"               # HELLO WORLD HELLO

With these you can replace text (// for global — the native sed 's///g' for simple cases), extract substrings by position, get a string's length, and change case — all inline. The case-modification operators (^^, ,,) are Bash 4+; the rest are widely available. Together with the prefix/suffix strippers, parameter expansion handles the great majority of string work you'd otherwise shell out for.

4.5 Arithmetic expansion

Related and essential: $(( )) evaluates integer arithmetic:

count=$((count + 1))          # increment
total=$(( (a + b) * 2 ))      # arithmetic with precedence
(( count++ ))                 # C-style increment (as a statement)
if (( count > 10 )); then ... # arithmetic comparison — cleaner than [[ ]] for numbers

# Examples (ran):
x=5; y=3
echo $(( x * y ))             # 15
echo $(( x % y ))             # 2  (modulo)
(( x > y )) && echo "x is bigger"

$(( )) does integer math (Bash has no built-in floats — use awk or bc for those, Section 9). The (( )) command form is for evaluating/comparing without capturing a value — if (( n > 10 )) reads more naturally than [[ "$n" -gt 10 ]] for numeric comparisons, and (( count++ )) is the idiomatic increment. Inside (( )) you don't need $ on variable names ((( x * y )), not (( $x * $y ))), and you get C-style operators.

Arrays & Associative Arrays

Beginners cram lists into space-separated strings and pay for it with quoting bugs. Bash has real arrays — indexed and associative (key/value) — which handle spaces correctly, express intent clearly, and unlock patterns like counting and lookup tables. This section covers both, with the quoting rules that make them safe. Every example ran.

5.1 Why arrays (and the trap of fake ones)

A common beginner mistake is storing a list as a plain string: files="a.txt b.txt c.txt" then looping with for f in $files. This appears to work — until a filename contains a space (my file.txt), at which point the unquoted expansion splits it into two "files" and everything breaks. Real arrays store each element distinctly, so spaces inside an element are preserved. Using proper arrays instead of space-separated strings is one of the clearest signs of a scripter who has been bitten by word-splitting and learned the lesson.

5.2 Indexed arrays

Ordered lists indexed by integer (starting at 0):

# Declaration / assignment
fruits=("apple" "banana" "cherry")        # literal array
fruits+=("date")                          # append an element
fruits[10]="fig"                          # sparse assignment is allowed

# Access
echo "${fruits[0]}"                       # apple  (single element — BRACES REQUIRED)
echo "${fruits[@]}"                       # all elements
echo "${#fruits[@]}"                      # count of elements
echo "${!fruits[@]}"                      # the INDICES (keys)

# Iterate correctly — ALWAYS quote "${arr[@]}"
for fruit in "${fruits[@]}"; do
  echo "fruit: $fruit"
done

The essential syntax: ${array[i]} for one element (the braces are mandatory — $array[0] does not work, it's $array followed by literal [0]), ${array[@]} for all elements, ${#array[@]} for the count, and ${!array[@]} for the indices. The golden rule for iteration: always write "${arr[@]}" with the quotes and @ — this expands to each element as a separate, properly-quoted word, preserving spaces within elements. (${arr[*]} joins them into one string; "${arr[@]}" keeps them separate — you almost always want @.)

5.3 Associative arrays (key/value maps)

Bash 4+ supports associative arrays — dictionaries mapping string keys to values. They must be declared with declare -A:

declare -A color                          # REQUIRED before use
color["apple"]="red"
color["banana"]="yellow"
color["grape"]="purple"

echo "${color[apple]}"                     # red  (look up by key)
echo "${#color[@]}"                        # 3    (number of pairs)
echo "${!color[@]}"                        # the KEYS
echo "${color[@]}"                         # the VALUES

# Iterate over key/value pairs:
for fruit in "${!color[@]}"; do
  echo "$fruit is ${color[$fruit]}"
done

# Check if a key exists:
if [[ -v color["apple"] ]]; then echo "have apple"; fi

Associative arrays turn awkward lookups into clean, O(1) key access. The pattern for key in "${!map[@]}" iterates keys (note the !), and ${map[$key]} fetches the value. The most common use is counting — tallying occurrences of things (word frequencies, log levels, error types) by using the thing as a key and incrementing the value. Forgetting declare -A is the #1 associative-array bug: without it, Bash treats the subscripts as arithmetic and silently misbehaves.

5.4 A real example — counting with an associative array

The killer app for associative arrays: counting occurrences. Here we tally how often each word appears (this ran):

#!/usr/bin/env bash
set -euo pipefail

declare -A count
words=(apple banana apple cherry banana apple)

for w in "${words[@]}"; do
  count["$w"]=$(( ${count["$w"]:-0} + 1 ))    # increment, defaulting to 0
done

# Print the tally
for w in "${!count[@]}"; do
  printf '%-8s %d\n' "$w" "${count[$w]}"
done

The key trick is ${count["$w"]:-0} — when a word is seen for the first time, its count is unset, so :-0 defaults it to 0 before incrementing (avoiding a set -u error). This counting pattern — associative array keyed on the item, value is the tally — is one of the most useful in all of shell scripting, the native equivalent of a hash-map counter in any other language. It appears constantly in log analysis, data summarization, and deduplication.

5.5 Passing arrays to functions

A subtlety worth knowing: arrays don't pass cleanly as normal arguments (the elements just flatten into the argument list). Two robust approaches:

# Approach 1: expand the array into arguments, rebuild inside:
process() {
  local items=("$@")                       # capture all args as an array
  echo "got ${#items[@]} items"
}
process "${myarray[@]}"                     # expand array into the call

# Approach 2 (Bash 4.3+): pass by name with a nameref:
process_ref() {
  local -n arr="$1"                         # nameref to the caller's array
  echo "count: ${#arr[@]}"
}
process_ref myarray                         # pass the NAME, not the expansion

Approach 1 (expand with "${arr[@]}", capture with local items=("$@")) is the portable, common choice. Approach 2 uses a nameref (local -n) to reference the caller's array by name — cleaner for large arrays or when you need to modify the original, but Bash 4.3+ only. Either way, the naive process $myarray (unquoted, no [@]) passes only the first element — a common bug. Arrays need "${arr[@]}" to travel intact.

SECTION 6

Functions, Scope & Building Libraries

Functions turn a sprawling script into readable, reusable, testable pieces. But shell functions have quirks — around scope, arguments, and how they "return" data — that trip up people coming from other languages. This section covers writing solid functions and packaging them into libraries you can share across scripts. Every example ran.

6.1 Defining functions and local scope

A function groups commands under a name. The critical discipline is local: by default, all shell variables are global, so a variable set inside a function leaks out and can clobber others. Declaring variables local confines them:

greet() {
  local name="$1"              # local — confined to this function
  local greeting="Hello, $name"
  echo "$greeting"
}

# WITHOUT local, this would be a bug:
counter=0
increment() { counter=$((counter + 1)); }   # intentionally global here
increment; increment
echo "$counter"                 # 2

# But accidental globals bite:
bad() { temp="oops"; }          # 'temp' leaks to the global scope!

Always declare function-internal variables local unless you specifically intend to modify a global. Shell's default-global scoping is a frequent source of bugs where a helper function silently overwrites a variable the caller was using (imagine two functions both using i as a loop counter). local is the fix, and using it everywhere is a mark of careful scripting. Note local also has a subtle interaction with set -e on the same line (local x="$(cmd)" masks cmd's exit status) — declare, then assign separately if the command's success matters: local x; x="$(cmd)".

6.2 Arguments and returning status

Functions receive arguments exactly like scripts do — $1, $2, $@, $#:

add() {
  local a="$1" b="$2"
  echo $(( a + b ))            # "return" a value by PRINTING it (see 6.3)
}

# Functions "return" only an exit STATUS (0-255) via `return`:
is_valid_port() {
  local port="$1"
  [[ "$port" =~ ^[0-9]+$ ]] && (( port >= 1 && port <= 65535 ))
  # the last command's status IS the function's return status
}

if is_valid_port "8080"; then echo "valid"; fi

Two things to internalize. First, $1, $2, $@, $# inside a function refer to the function's arguments, not the script's. Second — the big one — return only sets an exit status (0–255), not a value. A function's return N communicates success/failure, exactly like a command's exit code, and is what if my_func; then tests. To return actual data, you print it (next section). is_valid_port above "returns" true/false purely via the exit status of its last command — the idiomatic way to write a predicate function.

6.3 Returning data — command substitution vs global

Since return can't hand back a string or number, functions produce data in one of two ways:

# Way 1: PRINT the result, capture with command substitution (most common):
get_timestamp() { date '+%Y-%m-%d'; }
today="$(get_timestamp)"        # capture the function's stdout

# Way 2: assign to a variable (a global, or a nameref parameter):
compute() {
  local -n out="$1"             # nameref: caller passes a variable NAME
  out="computed value"
}
compute result                  # pass the name; function fills it
echo "$result"

Way 1 (print and capture with $(...)) is the standard, most portable pattern: the function writes its result to stdout, the caller captures it with command substitution. The one caveat: the function must print only its result to stdout (send logs/errors to stderr with >&2), or the capture will include noise. Way 2 (nameref output parameter) avoids a subshell (faster in tight loops, and can return multiple values) but is Bash 4.3+ and less obvious. For most code, print-and-capture is the right, readable choice.

6.4 Building a library

Functions used across multiple scripts belong in a library — a file of function definitions that other scripts load with source (or its POSIX alias .):

# lib/logging.sh — a reusable library (no shebang needed; it's sourced, not run)
# shellcheck shell=bash            # tells ShellCheck the target shell (not a shebang; doesn't run)
log_info()  { printf '[INFO]  %s\n' "$*"; }
log_warn()  { printf '[WARN]  %s\n' "$*" >&2; }
log_error() { printf '[ERROR] %s\n' "$*" >&2; }
die()       { log_error "$@"; exit 1; }

# In a script that uses it:
#!/usr/bin/env bash
set -euo pipefail
source "$(dirname "$0")/lib/logging.sh"     # load the library

log_info "starting up"
die "something went wrong"                   # from the library

source file (or . file) runs the file's contents in the current shell, so its function definitions become available — the shell equivalent of import. A library file typically has no shebang and no top-level executing code — just definitions — because it's meant to be sourced, not executed directly. (Add a # shellcheck shell=bash directive comment instead of a shebang so ShellCheck knows the target shell — it's a comment, so it doesn't execute; and lint a script that sources libraries with shellcheck -x so it follows the source.) The "$(dirname "$0")/lib/..." locates the library relative to the script, so it works regardless of where the script is called from. Building a small library of your common helpers (logging, die, validation) and sourcing it across scripts is exactly how teams keep shell tooling consistent and DRY.

6.5 A real example — a sourced library in action

Putting it together — a library plus a script that uses it, both run:

# --- mathlib.sh (the library) ---
is_number()  { [[ "$1" =~ ^-?[0-9]+$ ]]; }        # predicate: status only
max() {                                            # data: printed
  local m="$1"; shift
  local n
  for n in "$@"; do (( n > m )) && m="$n"; done
  printf '%s\n' "$m"
}

# --- script.sh (uses the library) ---
#!/usr/bin/env bash
set -euo pipefail
source ./mathlib.sh
is_number "42"  && echo "42 is a number"
is_number "4x2" || echo "4x2 is not a number"
biggest="$(max 3 9 2 7)"
echo "max is $biggest"

is_number is a predicate — it returns only a status, used directly in &&/||. max returns data — it prints the result, captured with $(max ...). This cleanly separates the two function idioms (status-returning predicates vs data-returning producers) and shows a library sourced and used. It's the model for organizing any nontrivial shell project: helpers in libraries, scripts that source them and orchestrate.

🏋️ Exercise 6.1

Why must you use local for variables inside functions? Give a concrete bug that omitting local can cause. What's the issue with local x="$(some_command)" under set -e?

Solution

You must use local because shell variables are global by default — a variable assigned inside a function without local persists in the global scope after the function returns and can clobber variables elsewhere.

Concrete bug: suppose the main script uses a loop variable i, and calls a helper that also uses i in its own loop without declaring it local:

for i in 1 2 3; do
  helper           # helper does: for i in a b c; do ...; done  (no local!)
  echo "$i"        # prints "c" every time — helper overwrote the caller's i
done

The helper's loop overwrites the caller's i, so the outer loop breaks. Declaring local i inside helper isolates it and fixes the bug. (Similarly, a helper setting temp="oops" silently creates/overwrites a global temp.)

The local x="$(some_command)" issue under set -e: combining the declaration and command-substitution assignment on one line masks the command's exit status, because the exit status of the whole statement becomes that of local (which succeeds), not of some_command. So if some_command fails, set -e won't catch it. The fix is to declare and assign separately when the command's success matters:

local x
x="$(some_command)"    # now a failure in some_command triggers set -e

🏋️ Exercise 6.2

In shell, what does return actually communicate, and how is that different from returning a value in most languages? Show the two ways a function can hand back a computed string, and note a caveat for each.

Solution

return communicates only an exit status — an integer 0–255 — signaling success (0) or failure (non-zero), exactly like a command's exit code. This differs from most languages, where return hands back an arbitrary value (string, number, object). A shell function's return cannot hand back data; it only reports status (which is what if my_func; then tests).

Two ways to hand back a computed string:

  1. Print it, capture with command substitution: bash make_name() { printf '%s\n' "user_$1"; } name="$(make_name 42)" Caveat: the function must write only its result to stdout — any logging/diagnostics must go to stderr (>&2), or they'll be captured too. Also runs in a subshell (minor overhead).

  2. Write into a variable via a nameref output parameter (Bash 4.3+): bash make_name() { local -n out="$1"; out="user_$2"; } make_name name 42 # pass the variable's NAME Caveat: requires Bash 4.3+ and namerefs, is less obvious to readers, and you must avoid name collisions between the nameref and the variable passed in. (Alternatively, assign to a plain global — but that reintroduces scope/clobbering concerns.)

Print-and-capture is the standard, portable, readable choice.

🏋️ Exercise 6.3

What does source lib.sh (or . lib.sh) do, and how does it differ from running ./lib.sh? Why does a library file typically have no shebang and no top-level executing code?

Solution

source lib.sh (or . lib.sh) executes the file's contents in the current shell, so any functions and variables it defines become available in the calling script's environment — the shell's equivalent of import/include.

How it differs from ./lib.sh: running ./lib.sh executes the file in a new, separate subshell (child process). Any functions or variables it defines exist only in that child and vanish when it exits — the parent script gains nothing. So to use a library's functions, you must source it (run it in your own shell), not execute it as a subprocess.

A library typically has no shebang and no top-level executing code because it's meant to be sourced, not run directly: - No shebang — the shebang matters only when a file is executed as a program; a sourced file runs under the caller's already-chosen interpreter, so a shebang is irrelevant (and could mislead someone into running it standalone). - No top-level executing code — sourcing runs everything at the file's top level in the caller's shell. If the library did work (or called exit) at the top level, sourcing it would trigger that work (or terminate the caller) as a side effect. A pure library should only define functions/constants, doing nothing until the caller invokes them.

Option & Argument Parsing with `getopts`

A professional command-line tool accepts options (-v, --output file) and arguments the way standard Unix tools do. Hand-rolling that parsing is error-prone; Bash's built-in getopts does it properly. This section covers positional arguments, getopts for short options, building a usage/help, and validating input. Every example ran.

7.1 Positional arguments and shift

The basics your tool builds on. A script's arguments are $1, $2, …; $# is the count; $@ is all of them; $0 is the script name:

echo "script: $0"
echo "first arg: $1"
echo "arg count: $#"
echo "all args: $@"

# shift removes $1 and moves everything down ($2 becomes $1, etc.):
first="$1"; shift          # now $@ is everything AFTER the first arg
echo "remaining: $@"

# Iterate over all positional args:
for arg in "$@"; do echo "arg: $arg"; done

shift is the key tool for consuming arguments one at a time — after shift, the old $2 is $1, and $# decreases by one. It's how you peel off a leading argument (like a subcommand) and pass the rest along. Always iterate with "$@" (quoted) so arguments with spaces stay intact — the same quoting rule as arrays.

7.2 getopts — parsing short options

getopts is a Bash built-in that parses standard single-dash options (-v, -o value) in a loop. You give it an option string describing valid options; a : after a letter means that option takes an argument:

#!/usr/bin/env bash
set -euo pipefail

verbose=0
output=""

# "vo:h" → -v (flag), -o (needs an argument), -h (flag).
while getopts "vo:h" opt; do
  case "$opt" in
    v) verbose=1 ;;
    o) output="$OPTARG" ;;          # $OPTARG holds the option's argument
    h) usage; exit 0 ;;
    \?) usage; exit 1 ;;            # \? matches any INVALID option
  esac
done
shift $((OPTIND - 1))               # discard the parsed options, leaving positionals

echo "verbose=$verbose output=$output"
echo "remaining positional args: $*"

The mechanics: getopts "vo:h" opt sets opt to each option letter in turn; the option string "vo:h" declares -v and -h as flags and -o as taking an argument (the trailing :). When an option takes an argument, getopts puts it in $OPTARG. An invalid option makes opt the literal ? (matched by \?). After the loop, shift $((OPTIND - 1)) discards all the parsed options, leaving only the positional arguments in $@. This is the standard, robust way to parse options — it handles bundled flags (-vh), -o value and -ovalue, and the -- end-of-options marker automatically.

7.3 Building usage and required-option validation

A real tool documents itself and validates its inputs:

#!/usr/bin/env bash
set -euo pipefail
readonly SCRIPT_NAME="${0##*/}"

usage() {
  cat <<EOF
Usage: $SCRIPT_NAME [-v] -o OUTPUT INPUT
  -o OUTPUT   output file (required)
  -v          verbose
  -h          show this help
EOF
}

verbose=0 output=""
while getopts "vo:h" opt; do
  case "$opt" in
    v) verbose=1 ;;
    o) output="$OPTARG" ;;
    h) usage; exit 0 ;;
    \?) usage >&2; exit 1 ;;
  esac
done
shift $((OPTIND - 1))

# Validate: required option present?
[[ -n "$output" ]] || { echo "Error: -o OUTPUT is required" >&2; usage >&2; exit 1; }
# Validate: exactly one positional argument (INPUT)?
[[ $# -eq 1 ]] || { echo "Error: exactly one INPUT required" >&2; exit 1; }
input="$1"

echo "input=$input output=$output verbose=$verbose"

The usage function (printed via a here-doc) is the tool's built-in documentation, shown on -h or on error. After parsing, the script validates: the required -o is present, and exactly one positional INPUT remains. Sending usage/errors to stderr (>&2) on the error path keeps them off stdout. This structure — parse with getopts, then validate — is the backbone of every well-behaved CLI tool.

7.4 A note on long options

getopts handles only short options (-v). Long options (--verbose, --output=file) aren't supported by the Bash built-in. Options if you need them:

  • Provide short options and document them — often sufficient, and standard for many tools.
  • Hand-roll a while/case loop over "$@", matching --verbose, --output, etc., using shift to consume them. More code, full control.
  • Use the external getopt (GNU, no s) which supports long options — but it's less portable and trickier.
# Hand-rolled long-option parsing (the portable way to get --long options):
while [[ $# -gt 0 ]]; do
  case "$1" in
    -v|--verbose) verbose=1; shift ;;
    -o|--output)  output="$2"; shift 2 ;;      # consume flag AND its value
    --output=*)   output="${1#*=}"; shift ;;    # --output=file form
    -h|--help)    usage; exit 0 ;;
    --)           shift; break ;;               # end of options
    -*)           echo "unknown option: $1" >&2; exit 1 ;;
    *)            break ;;                        # first positional — stop
  esac
done

For most tools, getopts with short options is the right, clean choice. Reach for the hand-rolled loop only when long options genuinely matter — it uses the same shift-based consuming, with case matching each option form (note shift 2 to consume a flag and its argument, and ${1#*=} to extract the value from --output=file).

In a getopts option string like "vo:h", what does each character mean, and what is the role of $OPTARG? What does shift $((OPTIND - 1)) accomplish and why is it necessary?

Solution

In the option string "vo:h": - v — declares -v as a valid flag (no argument). - o: — declares -o as an option that takes an argument (the trailing colon : means "requires an argument"). - h — declares -h as a valid flag.

(A leading : in the whole string would enable "silent" error handling; without it, getopts prints its own error messages for unknown options / missing arguments.)

$OPTARG holds the argument value of an option that takes one. When getopts processes -o file.txt, it sets opt=o and OPTARG=file.txt, so you read the value from $OPTARG (e.g., output="$OPTARG").

shift $((OPTIND - 1)) discards all the options getopts just parsed, leaving only the positional arguments in $@. getopts sets OPTIND to the index of the next unparsed argument; subtracting 1 gives the number of option-words consumed, and shifting by that many removes them. It's necessary so that after option parsing, $1, $2, … and "$@" refer to the tool's actual positional arguments (files, etc.), not the options — letting you then read/validate them cleanly.

Write the option-parsing skeleton for a tool that accepts -v (verbose flag), -n COUNT (an option with an argument), and -h (help), using getopts. Include handling of an invalid option.

Solution

  cat <<EOF

  -n COUNT   number of items
  -h         show this help

}

SECTION 8

Text Processing I: `grep`, `sed` & Regular Expressions

Text processing is the beating heart of shell scripting — extracting, filtering, and transforming lines of text. This section covers regular expressions (the pattern language underneath it all), grep (find matching lines), and sed (transform streams). Section 9 then covers awk for structured/columnar data. Every command here ran.

8.1 Regular expressions — the pattern language

Regexes describe text patterns. The core syntax (with an important Unix wrinkle — two "flavors"):

.        any single character          ^        start of line
*        zero or more of previous      $        end of line
[abc]    any one of a, b, c            [^abc]   any char NOT a, b, c
[a-z]    any char in the range         \        escape a special char

Basic (BRE) vs Extended (ERE) — these need backslashes in BRE, but not in ERE:
  +      one or more        (BRE: \+   ERE: +)
  ?      zero or one        (BRE: \?   ERE: ?)
  |      alternation (or)   (BRE: \|   ERE: |)
  ()     grouping           (BRE: \(\) ERE: ())
  {n,m}  repetition count   (BRE: \{\} ERE: {})

The Unix wrinkle that confuses everyone: tools have two regex dialects. Basic Regular Expressions (BRE) — the default for grep and sed — require backslashes before +, ?, |, (, ), {, } to give them their special meaning. Extended Regular Expressions (ERE) — enabled with grep -E or sed -E — treat those characters as special without backslashes, which is far more readable. The practical advice: use ERE (-E) whenever your pattern needs +, ?, |, or grouping — it saves a thicket of backslashes and matches the regex syntax you know from other languages.

8.2 grep — finding matching lines

grep prints lines matching a pattern. The flags you'll use constantly:

grep "pattern" file           # lines matching pattern
grep -i "error" log           # -i  case-insensitive
grep -v "debug" log           # -v  INVERT — lines NOT matching
grep -n "TODO" *.js           # -n  show line numbers
grep -r "func" src/           # -r  recursive through directories
grep -c "error" log           # -c  COUNT matching lines
grep -E "warn|error" log      # -E  extended regex (alternation)
grep -o "[0-9]+" file         # -o  print only the MATCHED text, not whole line
grep -l "TODO" *.md           # -l  just list filenames that match
grep -q "pattern" file        # -q  quiet — no output, just exit status (for `if`)

The two most underused-yet-powerful: grep -o prints only the matched portion (one per line), turning grep into an extractor — grep -oE '[0-9]+' pulls every number out of a file. And grep -q produces no output, only an exit status, which is exactly what you want in a condition: if grep -q "pattern" file; then. Combined with -i, -v, -c, and -r, these cover the overwhelming majority of line-filtering tasks.

8.3 sed — stream editing

sed transforms text as it streams through. Its dominant use is substitution, but it does much more:

# Substitution: s/find/replace/[flags]
sed 's/old/new/'  file        # replace FIRST "old" on each line
sed 's/old/new/g' file        # g = GLOBAL — every occurrence on the line
sed 's/old/new/gi' file       # g + i (case-insensitive)
sed -E 's/[0-9]+/NUM/g' file  # -E for extended regex

# Using capture groups (\1, \2 refer to \(...\) groups; ERE: (...)):
sed -E 's/(\w+)@(\w+)/\2 user \1/' file   # rearrange around captured groups

# Other operations:
sed -n '5,10p' file           # -n + p: print ONLY lines 5-10 (ranges)
sed '/^#/d' file              # DELETE lines matching a pattern (comments here)
sed '2d' file                 # delete line 2
sed 's/^/  /' file            # indent every line (insert at start)

The workhorse is s/find/replace/ with the g flag for all-occurrences (without it, only the first match per line changes). Capture groups — \(...\) in BRE or (...) in ERE — remember matched text that you reference as \1, \2 in the replacement, enabling rearrangement (swapping fields, reformatting). sed also deletes lines (/pattern/d), prints ranges (-n '5,10p'), and more. It reads from stdin or files and writes to stdout — a true stream editor.

8.4 In-place editing and the portability trap

sed -i edits a file in place (modifying it directly) — powerful and dangerous:

sed -i 's/old/new/g' file          # GNU sed: edit file in place
sed -i.bak 's/old/new/g' file      # edit in place, keep a .bak backup (safer)

A critical portability gotcha: sed -i behaves differently on GNU sed (Linux) versus BSD sed (macOS). GNU sed -i takes an optional suffix attached (-i.bak or bare -i); BSD sed -i requires a suffix argument (-i '' for none, as a separate word). So sed -i 's/.../.../' works on Linux but errors on macOS, and sed -i '' 's/.../.../' works on macOS but misbehaves on Linux. This trips up everyone who writes a script on one and runs it on the other. Safe approaches: always make a backup (-i.bak, which works on GNU), or avoid -i entirely by writing to a temp file and moving it (sed '...' file > tmp && mv tmp file) — portable and safer, since a failed sed won't corrupt the original.

8.5 A real example — extracting and transforming

Combining the tools on realistic data (all ran). Say we have log lines and want the unique IP-like tokens and a count:

# Extract all IPv4-looking addresses, one per line, with grep -oE:
grep -oE '[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+' access.log

# Count how many lines contain "ERROR" (case-insensitive):
grep -ic error app.log

# Redact emails in a file with sed (replace user part), keep a backup:
sed -E 's/([a-zA-Z0-9._]+)@/[REDACTED]@/g' users.txt

# Pipeline: find ERROR lines, extract the code after "code=", tally is Section 9's job
grep ERROR app.log | grep -oE 'code=[0-9]+' | sed 's/code=//'

grep -oE 'pattern' is the extraction workhorse — it pulls just the matching substrings. sed -E 's/.../.../' reshapes them. Chaining grep (filter) → grep -o (extract) → sed (reshape) is a fundamental text-processing pipeline; when the data is columnar (fields in each line), awk (Section 9) takes over and does it more elegantly. But for pattern-based extraction and substitution, grep and sed are the tools.

What is the difference between Basic (BRE) and Extended (ERE) regular expressions, and how do you enable ERE in grep and sed? Why is ERE usually preferable when your pattern uses +, ?, |, or grouping?

Solution

BRE (Basic Regular Expressions) is the default dialect for grep and sed. In BRE, the metacharacters +, ?, |, (, ), {, } are literal unless escaped with a backslash — so "one or more" is \+, alternation is \|, grouping is \(...\), etc.

ERE is usually preferable when a pattern uses +, ?, |, or grouping because in BRE those all require backslashes, producing hard-to-read "backslash soup" (e.g., BRE \(foo\|bar\)\+ vs ERE (foo|bar)+). ERE is cleaner, less error-prone, and matches the regex conventions you already know — so reach for -E whenever your pattern needs those constructs.

SECTION 9

Text Processing II: `awk` in Depth

Where grep filters and sed substitutes, awk understands structure — it splits each line into fields and lets you compute with them. It's a small, complete programming language purpose-built for columnar and tabular text: extracting columns, computing sums and averages, tallying with associative arrays, and reformatting. Mastering awk is a genuine force-multiplier. Every example here ran (in mawk, a standard POSIX awk).

9.1 The awk model — patterns and actions

An awk program is a series of pattern { action } rules. For each input line, awk checks each pattern; if it matches, it runs the action. This tiny model is deceptively powerful:

awk 'pattern { action }' file

awk '/error/ { print }' file        # pattern only: print lines matching /error/
awk '{ print $1 }' file             # action only (no pattern): runs on EVERY line
awk '$3 > 100 { print $1 }' file    # pattern is an expression: field 3 > 100
awk 'NR == 1 { print }' file        # print only the first line

If you give only a pattern, the default action is { print } (print the whole line). If you give only an action, it runs on every line. The pattern can be a /regex/, a comparison ($3 > 100), or a special variable test (NR == 1). This pattern-action structure is what makes awk read like a series of rules: "for lines like this, do that."

9.2 Fields and built-in variables

awk's superpower is automatic field splitting. Each line is split (by default on whitespace) into fields $1, $2, …, with $0 being the whole line:

# Given "alice 30 engineer", awk sees $1=alice, $2=30, $3=engineer
awk '{ print $1, $3 }' file        # print columns 1 and 3

# Key built-in variables:
#   $0   the whole line          NF   Number of Fields on this line
#   $1.. individual fields        NR   Number of Records (line number so far)
#   FS   input Field Separator    OFS  Output Field Separator

awk '{ print NR, $0 }' file        # prepend line numbers
awk '{ print $NF }' file           # $NF = the LAST field (NF is a number, $NF is that field)
awk -F',' '{ print $2 }' file      # -F',' sets the field separator to comma (CSV)
awk -F':' '{ print $1 }' /etc/passwd   # first field of colon-separated data

The built-ins to know cold: NF (number of fields on the current line — so $NF is the last field), NR (record/line number, running count), and FS/OFS (input/output field separators). Set the field separator with -F (-F',' for CSV, -F':' for /etc/passwd-style data) or by assigning FS. This automatic splitting is why awk beats cut/sed for columnar data: you just refer to $2 or $NF directly.

9.3 BEGIN, END, and accumulation

awk has special BEGIN and END blocks that run once, before and after processing all lines — perfect for initialization and summaries:

# Sum a column and print the total at the end:
awk '{ sum += $1 } END { print "total:", sum }' numbers.txt

# Average:
awk '{ sum += $1; n++ } END { printf "avg: %.2f\n", sum/n }' numbers.txt

# BEGIN for setup / headers:
awk 'BEGIN { print "Name Score"; OFS="\t" } { print $1, $2 }' data.txt

# Count matching lines (like grep -c, but flexible):
awk '/ERROR/ { count++ } END { print count }' log.txt

BEGIN { } runs once before any input (initialize variables, print headers, set FS); END { } runs once after all input (print totals, averages, summaries). Between them, awk accumulates: variables persist across lines (and auto-initialize to 0/empty), so sum += $1 totals a column and n++ counts lines, with the END block reporting the result. This accumulate-then-summarize pattern is awk's bread and butter for turning rows of data into statistics. Note printf (like C's) for formatted output — %.2f for two decimals, essential since this is where you finally get real number formatting.

9.4 Associative arrays in awk — grouping and tallying

Just like Bash, awk has associative arrays — and they're even more central here, powering group-by operations:

# Tally occurrences of each value in column 1 (group-by count):
awk '{ count[$1]++ } END { for (k in count) print k, count[k] }' data.txt

# Sum a value (column 2) grouped by a key (column 1):
awk '{ total[$1] += $2 } END { for (k in total) print k, total[k] }' sales.txt

# Count log lines by level (field 1):
awk '{ level[$1]++ } END { for (l in level) printf "%-6s %d\n", l, level[l] }' app.log

This is the single most powerful awk pattern: array[$key]++ to count, or array[$key] += $value to sum, grouped by a field — then for (k in array) in the END block to print the results. It's a complete group-by aggregation in one line, the shell equivalent of SELECT key, COUNT(*)/SUM(x) GROUP BY key. Any time you need "how many of each" or "total per category," this is the tool. (Iteration order with for (k in array) isn't guaranteed — pipe to sort if you need ordering.)

9.5 A real example — a log summary in one line

Bringing it together on realistic log data (this ran). Given lines like ERROR /api/users 503 120ms:

# Report: request count and average latency per status code
awk '{
  status = $3
  latency = $4; sub(/ms/, "", latency)     # strip "ms" suffix
  count[status]++
  total[status] += latency
}
END {
  print "STATUS  COUNT  AVG_MS"
  for (s in count)
    printf "%-7s %-6d %.1f\n", s, count[s], total[s]/count[s]
}' requests.log

In a few lines, awk parses each record into fields, strips a suffix with sub(), accumulates a count and a latency sum grouped by status code, and prints a formatted per-status summary with averages — a genuine analytics query over a log file, with no database and no other tools. This is why awk is beloved: tasks that would be dozens of lines in another language, or awkward pipelines of cut/sort/uniq, become a short, readable awk program. When your data has fields, awk is almost always the right tool.

Solution

awk's pattern { action } model: awk reads input line by line, and for each line checks each rule's pattern; if the pattern matches (or is true), it executes that rule's action. A program is a series of such rules.

  • Only a pattern (no action): the default action { print } runs — the whole matching line is printed. E.g., awk '/error/' prints lines containing "error".
  • Only an action (no pattern): the action runs on every line. E.g., awk '{ print $1 }' prints field 1 of every line.

The variables: - $0 — the entire current line (record). - $NF — the last field on the line. ($ followed by the value of NF.) - NF — the Number of Fields on the current line (a count). So the last field is $NF, the second-to-last $(NF-1). - NR — the Number of Records processed so far, i.e., the current line number (running count across all input).

(Patterns can be /regex/, comparisons like $3 > 100, or tests on these variables like NR == 1.)

SECTION 10

Pipelines, Process Substitution & Redirection Mastery

Redirection and pipelines are how shell commands connect. Beyond > and |, Bash offers file-descriptor manipulation, process substitution, here-documents, and here-strings — tools that let you wire commands together in powerful ways. Mastering them is what lets you compose small tools into exactly the data flow you need. Every example here ran.

10.1 Redirection and file descriptors

Every process has three standard streams, each with a number (file descriptor): 0 = stdin, 1 = stdout, 2 = stderr. Redirection connects these to files or each other:

command > file          # stdout to file (overwrite)
command >> file         # stdout to file (append)
command < file          # stdin FROM file
command 2> errors.log   # stderr (fd 2) to a file
command 2>&1            # stderr to wherever stdout currently goes (MERGE them)
command > out 2>&1      # stdout to file, then stderr to same place (order matters!)
command &> out          # Bash shorthand: BOTH stdout and stderr to file
command 2>/dev/null     # discard stderr (silence errors)
command >/dev/null 2>&1 # discard ALL output

The subtle one is 2>&1 — "make fd 2 (stderr) point to wherever fd 1 (stdout) currently points." Order matters: > out 2>&1 sends both to out (redirect stdout to the file, then point stderr at the same place), whereas 2>&1 > out sends stderr to the original stdout (the terminal) and only stdout to the file — because at the moment 2>&1 runs, stdout hadn't been redirected yet. Reading redirections left-to-right and remembering "&N means current target of fd N" untangles this classic confusion. &> is the convenient Bash shorthand for "both."

10.2 exec and persistent redirection

exec (without a command) applies redirections to the rest of the script, not just one command — useful for logging everything a script does:

#!/usr/bin/env bash
# Redirect ALL of this script's output to a log file from here on:
exec > script.log 2>&1
echo "this goes to script.log, not the terminal"

# Or open a file descriptor for repeated use:
exec 3> output.txt       # open fd 3 for writing
echo "line" >&3          # write to fd 3
exec 3>&-                # close fd 3

exec > file 2>&1 at the top of a script redirects everything the script subsequently prints into a log — a common pattern for scripts that run unattended (cron jobs, deploy scripts). You can also open your own file descriptors (exec 3> file) to write to a specific stream repeatedly without reopening the file each time, closing them with exec 3>&-. This is advanced but occasionally exactly what you need.

10.3 Process substitution — treating command output as a file

Process substitution — <(command) — lets you use a command's output wherever a filename is expected. It's one of Bash's most useful and least-known features:

# Compare the output of two commands (diff needs FILES, not pipes):
diff <(sort file1) <(sort file2)

# Feed a command's output to something that reads a file argument:
while read -r line; do echo "got: $line"; done < <(grep ERROR log)

# Multiple inputs at once:
paste <(cut -f1 data) <(cut -f3 data)

<(command) runs command and presents its output as a temporary filename (a /dev/fd/... path) that the outer command reads as if it were a real file. This solves a common frustration: tools like diff, comm, and paste want file arguments, not piped input — process substitution lets you feed them the output of arbitrary commands. The while read ... done < <(command) form is also the correct way to loop over command output without the subshell variable-loss problem (Section 11). There's also >(command) for the output direction (rarer). Process substitution is the tool that unblocks "but this command needs a file, and I have a pipeline."

10.4 Here-documents and here-strings

Ways to feed multi-line or fixed input to a command without a separate file:

# Here-document: feed a block of text as stdin, until the delimiter word:
cat <<EOF
Hello, $USER
Today is $(date +%A)
EOF
# ↑ variables and command substitution ARE expanded

# Quote the delimiter to DISABLE expansion (literal text):
cat <<'EOF'
This $variable is literal — not expanded.
EOF

# <<- strips leading TABS (lets you indent the here-doc in code):
cat <<-EOF
    indented in source, tabs stripped in output
EOF

# Here-string: feed a single string as stdin:
grep "pattern" <<< "$myvariable"       # instead of echo "$myvariable" | grep

A here-document (<<DELIM ... DELIM) feeds a block of text to a command's stdin — ideal for usage messages, config templates, and multi-line input. By default it expands $variables and $(commands); quoting the delimiter (<<'EOF') makes it literal (essential when the text contains $ you don't want expanded). <<- strips leading tabs so you can indent. A here-string (<<<) feeds a single string as stdin — cmd <<< "$var" is cleaner than echo "$var" | cmd and avoids a subprocess and pipe. These eliminate the need for temp files just to hand some text to a command.

10.5 A real example — composing a pipeline

Bringing redirection, process substitution, and here-strings together (all ran):

# Log everything to a file while a script runs, timestamped:
exec > >(while read -r l; do echo "$(date +%T) $l"; done) 2>&1
echo "this line gets a timestamp prefix in the log"

# Find lines in file A that aren't in file B (both sorted on the fly):
comm -23 <(sort a.txt) <(sort b.txt)

# Process a variable's lines without a temp file or a lost-variable subshell:
total=0
while read -r n; do total=$((total + n)); done <<< "$numbers"
echo "total: $total"

Each shows a composition: >(...) sends output through a filter (adding timestamps) before the file; <(sort ...) feeds sorted streams to comm which needs files; <<< "$numbers" loops over a variable's lines while keeping total in the current shell (no subshell, so the value survives — Section 11's lesson). This compositional power — wiring commands together via descriptors and substitutions — is what lets the shell act as glue between small tools, building exactly the data flow a task requires without intermediate files.

Solution

The three standard file descriptors: - 0 = stdin (standard input) - 1 = stdout (standard output — normal output) - 2 = stderr (standard error — diagnostics/errors)

SECTION 11

Working with Files, Loops & Robust Iteration

Iterating over files and lines is the most common thing scripts do — and the most common place they break, because filenames and input contain spaces, newlines, and other surprises. This section covers robust iteration: reading files line by line correctly, handling any filename safely, and avoiding the subtle subshell and word-splitting bugs that plague naive loops. Every example ran.

11.1 The right way to read a file line by line

The correct incantation for reading a file line by line is specific, and every part matters:

while IFS= read -r line; do
  echo "line: $line"
done < file.txt

Three critical pieces:

  • read -r — the -r (raw) flag stops read from interpreting backslashes as escapes. Always use -r unless you specifically want backslash processing; without it, a line containing \ is mangled. This alone fixes a huge class of bugs.
  • IFS= — setting the Internal Field Separator to empty for this command prevents read from stripping leading and trailing whitespace from the line. Without it, indentation and trailing spaces are lost.
  • < file.txt — redirect the file into the loop's stdin. (Redirecting after done feeds the whole loop.)

Memorize while IFS= read -r line; do ... done < file as a unit — it reads each line exactly, whitespace and backslashes intact. The common naive alternatives (for line in $(cat file)) are broken: they word-split on spaces (so each word becomes an iteration, not each line) and glob-expand. Never loop over $(cat file) to process lines — use the while read form.

11.2 The subshell trap in pipelines

A notorious bug: piping into a while loop runs the loop in a subshell, so variables set inside don't survive:

count=0
grep ERROR log | while read -r line; do
  count=$((count + 1))       # modifies count in a SUBSHELL
done
echo "$count"                 # prints 0! — the subshell's count is gone

# FIX with process substitution (loop runs in the CURRENT shell):
count=0
while read -r line; do
  count=$((count + 1))
done < <(grep ERROR log)
echo "$count"                 # correct count

When you pipe command | while ..., the while loop is part of the pipeline and runs in a subshell — a child process. Any variable it sets (like count) changes only the child's copy, which vanishes when the pipeline ends, so the parent sees the original value (0 here). This surprises everyone once. The fix is process substitution (Section 10): while read ...; done < <(command) redirects the command's output into the loop without a pipe, so the loop runs in the current shell and its variables persist. Whenever a while read loop needs to set variables you use later, feed it with < <(...), not ... |.

11.3 Iterating over files safely — globs and find -print0

Looping over files has its own pitfalls (spaces and special characters in names). The safe approaches:

# Glob directly — handles spaces correctly, quote the variable:
for file in *.txt; do
  [[ -e "$file" ]] || continue      # guard: skip if no matches (glob stays literal)
  echo "processing: $file"
done

# nullglob makes a non-matching glob expand to NOTHING (not itself):
shopt -s nullglob
for file in *.log; do echo "$file"; done    # loop body skipped if no .log files
shopt -u nullglob

# For recursive or complex finds, use find -print0 with read -d '':
find . -name '*.txt' -print0 | while IFS= read -r -d '' file; do
  echo "found: $file"
done

Globbing (for file in *.txt) is actually safe for spaces as long as you quote "$file" — each match is a separate iteration regardless of spaces in names. Two gotchas: a glob that matches nothing expands to the literal pattern (*.txt as text), so either guard with [[ -e "$file" ]] || continue or enable shopt -s nullglob (which makes non-matching globs vanish). For recursive traversal, the bulletproof method is find ... -print0 | while IFS= read -r -d '' file — find -print0 separates filenames with a null byte (the one character that can't appear in a filename), and read -d '' reads up to that null, so filenames containing spaces, tabs, or even newlines are handled perfectly. This null-delimited pattern is the gold standard for "do something to every file matching X."

11.4 mapfile / readarray — slurp lines into an array

Bash 4+ can read an entire file (or command output) into an array in one step:

mapfile -t lines < file.txt          # each line → an array element (-t strips newlines)
echo "${#lines[@]} lines"
echo "first: ${lines[0]}"

# From a command (with process substitution):
mapfile -t files < <(find . -name '*.sh')
for f in "${files[@]}"; do echo "$f"; done

mapfile -t array (also spelled readarray) reads lines into an array, one line per element, with -t trimming the trailing newlines. It's the clean way to get a file's lines into an array for random access, counting, or processing — no loop needed for the reading. Combined with process substitution (< <(command)), it captures command output line-by-line into an array safely. For simply loading lines to work with as a collection, mapfile -t is more convenient and less error-prone than a manual read loop.

11.5 A real example — safe batch processing

A realistic pattern: process every matching file, robustly, counting results (this ran):

#!/usr/bin/env bash
set -euo pipefail
shopt -s nullglob                      # non-matching globs vanish

processed=0
for file in ./data/*.csv; do
  # Safe even if the filename has spaces (quoted throughout):
  lines=$(wc -l < "$file")
  echo "processing $file ($lines lines)"
  processed=$((processed + 1))
done
echo "processed $processed files"

# Alternative for recursive: null-delimited, keeps count in current shell:
count=0
while IFS= read -r -d '' f; do
  count=$((count + 1))
done < <(find ./data -name '*.csv' -print0)
echo "found $count CSV files recursively"

This shows the two robust patterns side by side: a quoted glob loop (for file in ./data/*.csv, with nullglob so it's a no-op when there are no matches) for a single directory, and the null-delimited find -print0 + read -d '' fed via process substitution for recursive traversal (keeping count in the current shell). Both quote every variable, both handle any filename. This is what "robust iteration" looks like: no assumptions about filenames, no lost variables, no word-splitting.

🏋️ Exercise 11.2

Explain the subshell bug when piping into a while read loop. Why does a variable set inside not survive, and how does process substitution fix it?

Solution

The subshell bug: when you write command | while read ...; do ...; done, the while loop is the right-hand side of a pipeline. In Bash (by default), each stage of a pipeline runs in a separate subshell (a child process). So any variable the loop sets or modifies is changed only in that child subshell:

count=0
grep ERROR log | while read -r l; do count=$((count+1)); done
echo "$count"     # 0 — the increments happened in a subshell that has exited

The variable doesn't survive because a child process cannot alter its parent's variables; when the subshell running the loop finishes, its modified count is discarded, and the parent's original count (0) is what remains.

Process substitution fixes it by feeding the loop without a pipe:

count=0
while read -r l; do count=$((count+1)); done < <(grep ERROR log)
echo "$count"     # correct

Here < <(grep ERROR log) redirects the command's output into the loop's stdin, so the while loop runs in the current shell (it's no longer a pipeline stage). Its variable changes persist after the loop. (Other fixes exist — shopt -s lastpipe, or restructuring — but < <(...) is the standard idiom.)

🏋️ Exercise 11.3

Why is find ... -print0 | while IFS= read -r -d '' file the gold standard for iterating over files? What specifically does -print0 / -d '' protect against, and what does nullglob do for a plain glob loop?

Solution

find ... -print0 | while IFS= read -r -d '' file is the gold standard because it correctly handles every possible filename, including ones with spaces, tabs, or even newlines in them.

What -print0 / -d '' protect against: normally find separates results with newlines, and a while read splits on newlines — but a filename can legally contain a newline, which would break such a loop (one file appearing as two). -print0 makes find separate filenames with a null byte (\0) — the only character that cannot appear in a filename. read -d '' sets the read delimiter to that null byte, so each read consumes exactly one complete filename up to the null. Together with IFS= (no whitespace trimming) and -r (no backslash mangling), this reads every filename intactly, immune to spaces, tabs, and newlines. (The alternative is guarding with [[ -e "$f" ]] || continue inside the loop.)

  NEVER: for line in $(cat file)  → word-splits & globs, iterates WORDS not lines.

Subshell trap: cmd | while read ...; do var=...; done   → var LOST (loop in subshell)
  FIX: while read ...; do var=...; done < <(cmd)   → runs in current shell, var survives

Iterate files safely:
  for f in *.txt; do [[ -e "$f" ]] || continue; ... "$f"; done   # quote $f!
  shopt -s nullglob   → non-matching glob vanishes (else expands to literal pattern)
  recursive gold standard (any filename, even newlines):
    while IFS= read -r -d '' f; do ...; done < <(find . -name '*.txt' -print0)

Slurp into array (Bash 4+): mapfile -t arr < file    (-t strips newlines)
  mapfile -t files < <(find . -name '*.sh')
SECTION 12

Debugging, Profiling & Defensive Scripting

Even careful scripts have bugs. This section covers the tools to find them — Bash's tracing and syntax-checking, and above all ShellCheck, the static analyzer that catches whole categories of shell bugs automatically — plus the defensive patterns that prevent bugs in the first place. The ShellCheck examples here were run against real scripts, and its findings are real.

12.1 Tracing execution with set -x

When a script misbehaves, the fastest way to see what it's actually doing is set -x (xtrace), which prints each command as it executes, with variables expanded:

set -x                       # turn on tracing
name="world"
echo "Hello, $name"
set +x                       # turn it off

# Output (to stderr):
# + name=world
# + echo 'Hello, world'
# Hello, world

set -x shows each command after expansion (so you see the real values, not $name but world), prefixed with +. Turn it on around the suspicious region and off after (set +x). You can also run a whole script traced without editing it: bash -x script.sh. To make the trace more informative, customize PS4 (the trace prefix): PS4='+ ${BASH_SOURCE}:${LINENO}: ' adds the file and line number to every traced line — turning the trace into a precise execution log. set -x is the single most useful debugging tool in shell; when confused, trace.

12.2 Syntax-checking without running: bash -n

Before running a script that does something destructive, check its syntax without executing it:

bash -n script.sh            # parse only — report syntax errors, run NOTHING

bash -n (noexec) reads and parses the script, reporting any syntax errors (unclosed quotes, missing fi/done, etc.) without executing a single command. It's a cheap safety check — especially valuable for scripts that modify files or systems, where you don't want to discover a syntax error halfway through. Run it as a pre-flight check. (It catches syntax errors, not logic errors or runtime failures — for those, you need tracing and testing.)

12.3 ShellCheck — the essential linter

The single highest-value tool for shell quality is ShellCheck, a static analyzer that reads your script and flags bugs, pitfalls, and bad practices — many of which you'd otherwise discover only in production. It catches the exact classes of error this course has warned about. Run it on a script:

shellcheck script.sh

Consider this buggy script — every issue below was flagged by ShellCheck for real:

#!/usr/bin/env bash
echo "Processing $file"          # SC2154: file is referenced but never assigned
for f in $(ls *.txt); do          # SC2045: iterating ls is fragile; SC2035: use ./*.txt
  cp $f /backup                    # SC2086: unquoted $f — breaks on spaces
done
if [ $count = 0 ]; then           # SC2086: unquoted $count; may break if empty
  echo "none"
fi

ShellCheck flags each problem with a code (e.g., SC2086 for unquoted variables — the most common real bug), an explanation, and often a suggested fix. The most valuable findings it produces constantly:

  • SC2086 — unquoted variable that will word-split/glob (cp $f → cp "$f"). This is the most common shell bug, and ShellCheck catches every instance.
  • SC2046 — unquoted command substitution with the same problem.
  • SC2164 — cd without checking it succeeded (cd "$dir" || exit).
  • SC2155 — local x="$(cmd)" masking the exit status (Section 6!).
  • SC2045 — iterating over ls output instead of globbing.

Running ShellCheck on every script — locally and in CI — is the highest-leverage habit in shell scripting. It encodes the accumulated wisdom of the community about what goes wrong, and turns "I hope this is correct" into "a tool verified the common pitfalls are absent." Treat a clean ShellCheck run as a baseline requirement, not a nicety.

12.4 Defensive scripting patterns

Beyond tools, defensive habits prevent bugs before they happen. The essentials, gathered:

  • Quote every variable expansion — "$var", "${arr[@]}", "$(cmd)". Unquoted expansions word-split and glob; quoting is the #1 defense. (ShellCheck's SC2086 enforces this.)
  • Use strict mode — set -euo pipefail at the top (Section 2).
  • Check cd and other critical commands — cd "$dir" || die "cannot cd". Never assume a cd succeeded.
  • Prefer [[ ]] over [ ] in Bash — [[ ]] is safer (no word-splitting inside, supports &&, =~ regex) and doesn't need every variable quoted (though quoting anyway is fine).
  • Set defaults for variables that might be unset — "${var:-default}", especially under set -u.
  • Validate inputs early — check files exist, arguments are present, numbers are numeric, before doing work.
  • Fail with clear messages to stderr — the die() helper (Section 2).

These habits, applied consistently, eliminate most shell bugs at the source. The through-line of this entire course is here: assume nothing about inputs or the environment, quote everything, fail fast and loud, and let tools (ShellCheck, bash -n, tests) verify what you can't hold in your head.

12.5 A real example — before and after

A fragile script, and its hardened version (both real; ShellCheck flagged the first, passed the second):

# BEFORE — fragile (ShellCheck flags SC2086, SC2164, and SC2045):
#!/bin/bash
cd $1                             # unquoted; unchecked
for f in $(ls); do                # parsing ls
  mv $f processed_$f              # unquoted — breaks on spaces
done

# AFTER — robust (ShellCheck clean):
#!/usr/bin/env bash
set -euo pipefail
cd "$1" || { echo "cannot cd to $1" >&2; exit 1; }   # quoted + checked
for f in *; do                    # glob, not ls
  [[ -e "$f" ]] || continue        # skip if empty dir
  mv -- "$f" "processed_$f"        # quoted; -- ends options
done

The "after" version quotes every expansion, checks the cd, globs instead of parsing ls, guards the empty case, and uses -- to prevent filenames starting with - from being read as options. It's the same logic, made robust — and a clean ShellCheck run confirms the common pitfalls are gone. This before/after is the whole course in miniature: the difference between a script that works in a demo and one you'd trust in production.

🏋️ Exercise 12.2

ShellCheck's SC2086 is its most common finding. What is the bug it flags, why is it dangerous, and how do you fix it? Give an example.

Solution

SC2086 flags an unquoted variable expansion (or unquoted command substitution) — e.g., cp $file dest instead of cp "$file" dest.

Why it's dangerous: an unquoted $var undergoes word-splitting and glob expansion: - Word-splitting: if the value contains whitespace, it's split into multiple arguments. file="my report.txt"; cp $file dest becomes cp my report.txt dest — three arguments, so it tries to copy two files named my and report.txt. Broken (and potentially destructive). - Globbing: if the value contains *, ?, [, it's expanded against filenames, so the value silently changes. - With an empty/unset value, the argument can disappear entirely, changing the command's meaning (e.g., [ $x = 1 ] becomes [ = 1 ], a syntax error).

The fix: quote the expansion — "$var", "${arr[@]}", "$(cmd)":

cp "$file" dest          # instead of  cp $file dest

Quoting makes the value a single argument with spaces and glob characters preserved literally. Quoting every expansion (unless you specifically want splitting) is the #1 defensive habit in shell scripting, and SC2086 catches every place you forgot.

SECTION 13

Portability & POSIX Compliance

Sometimes a script must run everywhere — in a minimal container, on a BSD, on an embedded system, wherever /bin/sh might be dash, busybox, or an old shell rather than Bash. Writing portable POSIX shell means avoiding Bash-only features ("bashisms") and sticking to the standardized subset. This section covers what's portable, what isn't, and how to verify portability — by actually running scripts under a POSIX shell, which is exactly what we did here.

13.1 When portability matters (and when it doesn't)

Be clear-eyed about this: most of your scripts don't need to be POSIX-portable. If a script runs on your machines and CI where Bash is present, use Bash and enjoy its features (everything in this course so far). Portability matters in specific situations:

  • Minimal environments — Alpine Linux containers (common in Docker) ship busybox/ash, not Bash. Container init and entrypoint scripts often must be POSIX sh.
  • System scripts — things that run as /bin/sh (some init systems, package maintainer scripts) get a POSIX shell, which may be dash.
  • Cross-platform tools — a script meant to run on Linux, macOS, and BSD can't assume a modern Bash (macOS ships Bash 3.2 from 2007 for licensing reasons).
  • Maximum reach — installers and bootstrap scripts that must run on anything.

When one of these applies, you write POSIX sh. Otherwise, write Bash. The mistake is either extreme: needlessly crippling every script to POSIX, or blindly assuming Bash where it isn't guaranteed. Match the tool to the target.

13.2 Common bashisms and their POSIX alternatives

The features you must give up (and what to use instead) when writing portable sh:

# Arrays — NOT in POSIX. Use space-separated + set, or multiple variables.
arr=(a b c)              # bashism → no portable array; restructure

# [[ ]] — NOT POSIX. Use [ ] (test).
[[ "$x" == "y" ]]        # bashism → [ "$x" = "y" ]   (single =, and quote!)

# == in test — use = :
[ "$x" == "y" ]          # bashism → [ "$x" = "y" ]

# Brace expansion — NOT POSIX:
echo {1..5}             # bashism → use seq: for i in $(seq 1 5)

# $'...' ANSI-C quoting, ${var^^} case conversion — NOT POSIX:
${name^^}               # bashism → use tr: printf '%s' "$name" | tr a-z A-Z

# local — technically not POSIX (but widely supported; dash has it)
# function keyword — use name() { } form (portable), not `function name`

# Process substitution <(...) and here-strings <<< — NOT POSIX:
diff <(cmd1) <(cmd2)    # bashism → use temp files
cmd <<< "$var"          # bashism → printf '%s\n' "$var" | cmd

The big ones to remember: no arrays, [ ] not [[ ]] (with single = for equality), no brace expansion ({1..5}, {a,b}), no <<< or <(...), no ${var^^}/${var,,} case conversion. POSIX shell is a smaller language — you lean more on external tools (tr, seq, expr, temp files) to make up for missing built-ins. It's more verbose, which is precisely why you only use it when you must.

13.3 Verifying portability — run it under a POSIX shell

Reasoning about portability is unreliable; testing it is definitive. The key technique: run your script under an actual POSIX shell — dash is the standard choice (it's what /bin/sh is on Debian/Ubuntu, and it's strictly POSIX, so it rejects bashisms):

dash script.sh          # run under dash — bashisms will fail here
sh script.sh            # /bin/sh (dash on Debian/Ubuntu) — same effect

This isn't hypothetical — here's a real demonstration that was actually run. A script using a bashism, executed under dash, fails; the POSIX version succeeds:

# bashism.sh — uses [[ ]] and arrays
arr=(one two three)
[[ "${#arr[@]}" -eq 3 ]] && echo "ok"
#   Under bash:  prints "ok"
#   Under dash:  Syntax error: "(" unexpected   ← the array assignment fails

# portable.sh — POSIX equivalent
set -- one two three          # positional params as the "array"
[ "$#" -eq 3 ] && echo "ok"
#   Under bash:  prints "ok"
#   Under dash:  prints "ok"   ← works everywhere

Running the bashism version under dash produces a real syntax error (the arr=(...) array assignment is invalid POSIX); the portable version — using the positional parameters (set --) as a stand-in for an array and [ ] with a single = — runs identically under both bash and dash. This is the way to verify portability: don't guess, run it under dash (or add it to CI). ShellCheck also helps — it can warn about bashisms when you tell it the target is sh (via a #!/bin/sh shebang or # shellcheck shell=sh).

13.4 The checkbashisms approach and shebang honesty

Two more portability practices:

  • Set the shebang honestly (Section 1, revisited): a portable script gets #!/bin/sh, and then you must actually keep it POSIX — which running under dash verifies. A Bash script gets #!/usr/bin/env bash. The shebang is a promise; portability testing keeps you honest to it.
  • ShellCheck with sh target flags bashisms automatically: give the script a #!/bin/sh shebang (or # shellcheck shell=sh) and ShellCheck will warn on every Bash-only feature (arrays, [[ ]], <<<, etc.). This catches bashisms statically, before you even run under dash. (There's also a dedicated checkbashisms tool from Debian's devscripts, but ShellCheck's sh mode covers the same ground and you likely already have it.)

The combined workflow for a script that must be portable: write it in POSIX style, give it #!/bin/sh, run ShellCheck (which now flags bashisms), and run it under dash to confirm. Three cheap checks that together guarantee your "portable" script actually is.

SECTION 14

Testing Shell Scripts with Bats

Shell scripts deserve tests as much as any other code — arguably more, since their failure modes (a bad rm, a broken deploy) are so consequential. Bats (Bash Automated Testing System) is the standard framework for testing shell scripts and functions. This section covers writing and running real Bats tests — and every test shown here was actually executed with Bats.

14.1 Why test shell scripts

Scripts are code, and untested code breaks. The case for testing shell specifically is strong: shell scripts often do destructive or high-stakes things (delete files, deploy systems, modify configuration), their failure modes are silent and severe, and their many edge cases (empty input, spaces in filenames, missing files, non-zero exits) are exactly the kind of thing tests pin down. A test suite lets you refactor a gnarly script with confidence, documents the expected behavior, and catches regressions before they reach production. If a script matters enough to be reused, it matters enough to test.

14.2 Bats basics

Bats runs test files (conventionally *.bats) containing @test blocks. Each @test is a named test; inside, you run commands and assert on their results:

#!/usr/bin/env bats

@test "addition works" {
  result="$(echo $((2 + 3)))"
  [ "$result" -eq 5 ]
}

@test "grep finds a match" {
  run grep "hello" <<< "hello world"
  [ "$status" -eq 0 ]
}

A .bats file has a #!/usr/bin/env bats shebang and one or more @test "description" { ... } blocks. A test passes if every command in it succeeds (and the block's final status is 0) and fails if any command returns non-zero — so a bare [ ... ] assertion that's false fails the test. You run the suite with bats file.bats, and Bats reports each test as ok/not ok (TAP format) with a summary.

14.3 The run helper, $status, and $output

Bats's key helper is run, which executes a command without letting its non-zero exit fail the test, capturing the results for you to assert on:

@test "command exits non-zero on bad input" {
  run my_script --bad-flag
  [ "$status" -eq 1 ]                    # $status = the exit code
  [ "$output" = "Error: unknown flag" ]  # $output = combined stdout+stderr
}

@test "output contains expected text" {
  run echo "hello world"
  [[ "$output" == *"world"* ]]           # partial match
  [ "${lines[0]}" = "hello world" ]      # lines[] = output split by line
}

run command executes it and sets three things: $status (the exit code), $output (all output as one string), and $lines[] (output split into an array of lines). Crucially, run suppresses the exit code from failing the test, so you can assert on a non-zero status — essential for testing error paths (a script that's supposed to exit 1 on bad input). Without run, a failing command would abort the test; with run, you capture and check its behavior. This is how you test both success and failure cases.

14.4 setup, teardown, and testing a real script

Bats provides setup (run before each test) and teardown (run after each test) functions — ideal for creating and cleaning up test fixtures:

#!/usr/bin/env bats

setup() {
  TEST_DIR="$(mktemp -d)"                # fresh temp dir per test
  echo "apple"  > "$TEST_DIR/fruits.txt"
  echo "banana" >> "$TEST_DIR/fruits.txt"
}

teardown() {
  rm -rf "$TEST_DIR"                      # clean up after each test
}

@test "counts lines in the file" {
  run wc -l < "$TEST_DIR/fruits.txt"
  [ "$status" -eq 0 ]
  [ "$output" -eq 2 ]
}

@test "finds a specific fruit" {
  run grep -c "apple" "$TEST_DIR/fruits.txt"
  [ "$output" -eq 1 ]
}

setup runs before every @test (creating a clean fixture — here a temp directory with a known file), and teardown runs after every test (removing it), so tests are isolated — each starts fresh and leaves no trace, and their order doesn't matter. This is the same discipline as unit testing in any language (Testing course, 12): arrange a known state, act, assert, clean up. To test your own script, you'd source its functions or run the script as a command and assert on $status/$output.

14.5 A real example — a tested function library

Here's a real library and its Bats suite, both actually run (the suite passes):

# strutil.sh — functions under test
# shellcheck shell=bash
slugify() { printf '%s' "$1" | tr '[:upper:] ' '[:lower:]-'; }
is_email() { [[ "$1" =~ ^[^@]+@[^@]+\.[^@]+$ ]]; }

#!/usr/bin/env bats
# test_strutil.bats

setup() { source ./strutil.sh; }          # load the functions to test

@test "slugify lowercases and dashes spaces" {
  run slugify "Hello World"
  [ "$output" = "hello-world" ]
}

@test "is_email accepts a valid address" {
  run is_email "a@b.com"
  [ "$status" -eq 0 ]
}

@test "is_email rejects an invalid address" {
  run is_email "not-an-email"
  [ "$status" -eq 1 ]
}

setup sources the library so its functions are callable; each test uses run to invoke a function and asserts on $output (for data-returning functions like slugify) or $status (for predicates like is_email) — the two function idioms from Section 6, now under test. Running bats test_strutil.bats executes all three and reports them passing. This is the complete testing loop for shell: functions in a library, a .bats suite that sources and exercises them, run in CI on every change (Section 7 of the DevOps course — CI runs your tests). Tested shell is trustworthy shell.

🏋️ Exercise 14.2

What does Bats's run helper do, and what are $status, $output, and $lines? Why is run essential for testing a command that is supposed to fail?

Solution

run command executes the given command and captures its results instead of letting it affect the test's flow. After run, it populates: - $status — the command's exit code. - $output — the command's combined output (stdout + stderr) as a single string. - $lines — the output split into an array of lines (${lines[0]}, ${lines[1]}, …), for asserting on individual lines.

You then assert on these, e.g., [ "$status" -eq 0 ] or [[ "$output" == *"expected"* ]].

run is essential for testing a command that's supposed to fail because, without it, a command returning non-zero would immediately fail the @test (a non-zero status in the block is treated as a test failure). If you're testing an error path — e.g., "the script must exit 1 and print an error on bad input" — you want the command to return non-zero, and you need to assert that it did. run suppresses that non-zero exit from failing the test and stashes it in $status, so you can check [ "$status" -eq 1 ] and inspect $output. It's what lets you test failure behavior deliberately, not just success.

🏋️ Exercise 14.3

What are setup and teardown for in Bats, and why do they make tests reliable? Sketch how you would test a function slugify from a library file.

Solution

setup runs before each @test; teardown runs after each @test. They're for fixtures — establishing a known starting state before a test and cleaning it up afterward (e.g., creating a temp directory with known files in setup, rm -rf-ing it in teardown).

They make tests reliable by ensuring isolation: because each test gets a fresh setup and a clean teardown, tests don't depend on each other or on leftover state, and their execution order doesn't matter. Each test starts from the same known baseline and leaves nothing behind to contaminate the next — eliminating flaky, order-dependent failures. (This is the standard arrange/act/assert/cleanup discipline of unit testing.)

Sketch for testing slugify from a library:

#!/usr/bin/env bats

setup() { source ./strutil.sh; }        # load the function under test

@test "slugify lowercases and hyphenates" {
  run slugify "Hello World"
  [ "$status" -eq 0 ]
  [ "$output" = "hello-world" ]
}

@test "slugify handles already-clean input" {
  run slugify "abc"
  [ "$output" = "abc" ]
}

setup sources the library so slugify is available in each test; each @test uses run slugify ... to capture $output/$status and asserts the result. Run with bats test_strutil.bats.

15.4 Cleanup done right — a scoping lesson (Sections 3, 6, 11)

logstat buffers its input into temp files (so it can scan the data several times). Those must be cleaned up on every exit path. The first draft put the temp variable inside main as a local — and it had a real bug worth dwelling on:

With local tmp inside main, the EXIT trap rm -f "$tmp" fired after main returned, when tmp was already out of scope — so under set -u it errored with tmp: unbound variable. The trap outlives the function's local scope.

The fix — and the professional pattern — is to keep cleanup state at script scope, set the trap once, early, and guard it:

# Script-level so the EXIT trap always sees them (a local would be gone
# by the time the trap fires after main returns).
TMPFILES=()
cleanup() { [[ ${#TMPFILES[@]} -gt 0 ]] && rm -f "${TMPFILES[@]}"; return 0; }
trap cleanup EXIT

new_tmp() { local t; t="$(mktemp)"; TMPFILES+=("$t"); printf '%s' "$t"; }

An array (Section 5) collects every temp file created; cleanup removes them all on EXIT (Section 3), guarded so it's safe when empty (Section 11), and new_tmp is a small producer function (Section 6). This is exactly the kind of subtle bug — a trap referencing an out-of-scope local — that separates a script that seems fine from one that is fine. It surfaced only because the tool was tested and its stderr checked.

15.5 Option parsing and validation (Section 7)

Standard getopts, then validate:

main() {
  local level_filter="" top_n=5
  while getopts ":l:n:h" opt; do
    case "$opt" in
      l) level_filter="$OPTARG" ;;
      n) top_n="$OPTARG" ;;
      h) usage; exit 0 ;;
      :)  die "option -$OPTARG requires an argument (see -h)" ;;
      \?) die "unknown option -$OPTARG (see -h)" ;;
    esac
  done
  shift $((OPTIND - 1))

  is_uint "$top_n" || die "-n requires a non-negative integer, got: $top_n"

  local input="${1:--}"                          # default to stdin ("-")
  if [[ "$input" != "-" ]]; then
    [[ -f "$input" ]] || die "not a file: $input"
    [[ -r "$input" ]] || die "cannot read: $input"
  fi
  # ...
}

The leading : in ":l:n:h" enables custom error handling: : catches a missing option-argument, \? catches an unknown option — each reported cleanly via die. 15.6 The analysis — awk group-by (Section 9)

The heart of the tool is awk aggregation. Counting entries by level with percentages is a single associative-array pass:

awk '
  { level[$2]++; total++ }
  END {
    for (l in level)
      printf "  %-7s %6d  (%5.1f%%)\n", l, level[l], 100 * level[l] / total
  }
' "$data" | sort -k2 -rn

And the top-N messages combine awk (build the message from fields 3..NF, tally it), sort -rn, head, and a robust while read loop:

awk '{
  msg = ""
  for (i = 3; i <= NF; i++) msg = msg (i > 3 ? " " : "") $i

}
END { for (m in count) printf "%d\t%s\n", count[m], m }' "$data" \
  | sort -rn | head -n "$top_n" \

This is the course in a few lines: awk for field-aware group-by counting (Section 9), a pipeline composing small tools (Section 10), and while IFS=$'\t' read -r for safe line-by-line consumption (Section 11). 🏋️ Exercise 15.1

The first version of logstat declared its temp-file variable as local inside main, and the EXIT trap rm -f "$tmp" produced tmp: unbound variable under set -u. Explain precisely why, and why moving the cleanup state to script scope fixes it.

Solution

Why it failed: the EXIT trap runs when the script exits. When main finishes and control returns to the top level (main "$@" being the last line), the script then exits and the EXIT trap fires — but at that moment main has already returned, so any variable declared local inside main is out of scope (destroyed when the function returned). The trap's rm -f "$tmp" therefore references an unset variable, and because set -u (nounset) is on, referencing an unset variable is a fatal error → tmp: unbound variable. The trap outlives the function's local scope. (On paths where exit was called from within main, the local was still in scope, which is why the bug appeared inconsistently — a classic intermittent scoping bug.)

Why script scope fixes it: declaring the cleanup state at script (global) scope — e.g., TMPFILES=() at the top — means it remains in scope for the entire lifetime of the script, including when the EXIT trap runs after main returns. The trap can always see it. Setting the trap once, early, and guarding the cleanup ([[ ${#TMPFILES[@]} -gt 0 ]]) makes it safe on every exit path — before any temp file is created, after several are, on success, on error, or on signal. The general lesson: variables an EXIT trap depends on must not be function-locals (unless you're certain you only ever exit from inside that function); keep trap-referenced state at script scope.

🏋️ Exercise 15.2

logstat uses awk for its counting rather than grep | sort | uniq -c pipelines. Referencing the group-by pattern, explain why awk is the better tool here, and what the for (l in level) / END structure accomplishes.

Solution

Why awk is better here: the task is group-by aggregation — count entries per level, sum/compute per group, and format the result with percentages. awk does all of this in a single pass with an associative array keyed by the field (level[$2]++), directly accessing the level field ($2) thanks to automatic field splitting, and then formatting output with printf (percentages to one decimal). A grep | sort | uniq -c pipeline can count occurrences but is clumsier: it requires isolating the field first (another awk/cut), can't easily compute percentages or an error rate in the same pass, needs sort before uniq -c, and produces output you'd then have to reformat. awk expresses the whole SELECT level, COUNT(*) GROUP BY level (plus percentage math) as one short, readable program — fewer processes, one pass, full control over formatting.

What for (l in level) / END accomplishes: as awk reads each line, level[$2]++ accumulates a count per distinct level into the associative array level (and total++ tracks the grand total). These persist across all lines. The END block runs once after all input is consumed, and for (l in level) iterates over every key (each distinct level) in the array, printing that level, its count, and its percentage (100 * level[l] / total). So the pattern is: accumulate per-group during the pass, then emit the per-group results at the end — the essence of group-by aggregation. (Iteration order isn't guaranteed, which is why the output is piped to sort.)
