SECTION 1

What Is DevOps?

Before any tools, you need the mental model: what problem DevOps solves, why it exists, and the shape of the pipeline everything else in this course fills in.

1.1 The problem DevOps solves

Historically, software was split between two teams with opposing incentives. Developers were rewarded for change — shipping new features. Operations was rewarded for stability — keeping the servers up. Developers would "throw code over the wall" to ops, who had to run software they didn't write and didn't fully understand. When something broke, each side blamed the other: "it works on my machine" met "your code broke production." Releases were rare, huge, terrifying events — done at 2 a.m. on weekends, often failing, often rolled back by hand.

DevOps is the response: a culture and set of practices that unite development and operations so the people who build software also share responsibility for running it. The cultural core is captured in a phrase from Amazon: "you build it, you run it." When the same team owns a service from code to production to on-call, incentives align — you write more operable software because you're the one who gets paged when it breaks. DevOps isn't a job title or a single tool; it's this shared-ownership culture, enabled by automation.

1.2 What DevOps actually is

DevOps rests on a few reinforcing ideas:

  • Automate everything repeatable. Manual steps are slow, error-prone, and don't scale. Builds, tests, deployments, and infrastructure setup should be scripted so they run identically every time, by machine, on demand.
  • Ship small and often. Instead of one giant risky release a quarter, deploy small changes continuously. Small changes are easier to review, safer to deploy, and trivial to debug when something breaks (you know exactly what changed).
  • Keep everything reproducible. The same code, config, and environment should produce the same result anywhere — your laptop, CI, staging, production. Reproducibility is what kills "works on my machine."
  • Make everything reversible. Any deploy can be rolled back quickly. If you can undo a change in seconds, deploying stops being scary.
  • Measure and watch. You can't operate what you can't see. Logs, metrics, and alerts tell you whether a change helped or hurt, and warn you of trouble before users complain.

These aren't independent — they compound. Automation enables frequent shipping; reproducibility enables safe automation; observability enables confident change. Together they turn deployment from a rare ordeal into a routine, boring, safe non-event — which is exactly the goal. In healthy DevOps, "boring" is the highest praise.

1.3 The deployment pipeline — the mental model

Nearly everything in this course fits into one picture: the path a change takes from a developer's commit to running in production. A typical pipeline:

  commit  →  CI (build + test)  →  package (artifact/image)  →
      deploy to staging  →  verify  →  deploy to production  →  monitor
        │                                                          │
        └──────────────── roll back if unhealthy ─────────────────┘

Each stage has a job: CI proves the change is good (compiles, passes tests); packaging turns it into a deployable artifact; deployment puts it on servers, usually staging first; verification (health checks, smoke tests) confirms it's working; monitoring watches it over time; and rollback is the escape hatch when something's wrong. The two big halves have names: Continuous Integration (CI) — automatically build and test every change — and Continuous Delivery/Deployment (CD) — automatically move validated changes toward (and into) production. Hold this pipeline in your head; every later section is filling in one of these boxes.

1.4 CI, Continuous Delivery, and Continuous Deployment

Three terms that sound alike and are constantly confused:

  • Continuous Integration (CI) — every change is automatically built and tested as soon as it's pushed, so integration problems surface immediately instead of piling up. (Section 7.)
  • Continuous Delivery (CD) — every change that passes CI is automatically prepared and kept ready to deploy at any moment, but the final push to production is a manual decision (a button someone clicks). (Section 8.)
  • Continuous Deployment (CD) — goes one step further: every change that passes all automated checks is deployed to production automatically, no human gate. (Section 8.)

The difference between the two CDs is just that last step: delivery keeps you always ready to ship with a human pressing the button; deployment ships automatically. Continuous Deployment demands the most mature testing and monitoring (you're trusting automation to release to users), which is why many teams do Continuous Delivery — automated everything, with a human approving the final production release.

1.5 Why it matters — the business case

This isn't process for its own sake. Research (notably the long-running State of DevOps / DORA studies) consistently finds that teams practicing these ideas ship faster and more reliably — the old assumption that speed and stability trade off against each other turns out to be false. The four widely-cited metrics ("DORA metrics") that capture this: deployment frequency (how often you ship), lead time for changes (commit → production), change failure rate (what fraction of deploys cause problems), and time to restore service (how fast you recover from an incident). High-performing teams do well on all four at once — they deploy often, quickly, with few failures, and recover fast. That combination is the practical goal of everything in this course: change safely and frequently, because the two reinforce each other rather than compete.

SECTION 2

The Command Line & Scripting for Ops

Automation is the foundation of DevOps, and the shell is where automation begins. This section builds on the Linux course (09) to focus on operational scripting: writing scripts that are safe, predictable, and reusable — the kind you'd trust to build and deploy real software. Every script here was run in a real shell.

2.1 Why scripting is the bedrock

Every automated step in a pipeline — build, test, deploy, health-check, rollback — is, at bottom, a script the machine runs instead of a human. Before fancy tools, the skill underneath all of DevOps is writing a script that does a task the same way every time, checks whether it worked, and stops loudly if it didn't. Master this and everything else (CI, CD, containers) is just orchestration around scripts. A build in GitHub Actions is a list of shell commands; a deploy is a script; a health check is a script. The tooling is scaffolding; the scripts do the work.

2.2 The safety preamble — set -euo pipefail

The single most important habit in ops scripting. By default, a shell script keeps going after a command fails — which in a deploy script is catastrophic (imagine "build the app" fails but "ship it to production" runs anyway). Start every serious Bash script with:

#!/usr/bin/env bash
set -euo pipefail

# now the script is strict:
#   -e  exit immediately if any command fails (non-zero exit)
#   -u  error on use of an unset variable (catches typos in $VARNAME)
#   -o pipefail  a pipeline fails if ANY command in it fails, not just the last

Without -e, failures are silently ignored and the script barrels on. Without -u, a typo'd variable expands to empty string (turning rm -rf "$DIR/" into rm -rf "/" if DIR is unset — a real way people have wiped servers). Without pipefail, build | tee log reports success even if build failed, because tee succeeded. Together, these three make a script fail fast and loud instead of limping forward in a broken state — exactly what you want when the script controls your production systems.

2.3 Exit codes — how scripts communicate success and failure

Every command returns an exit code: 0 means success, any non-zero means failure. This is how the shell (and CI systems, and orchestrators) know whether a step worked. You check the last command's code with $?, and your own scripts should exit with a meaningful code:

#!/usr/bin/env bash
set -euo pipefail

npm run build
echo "build exit code: $?"     # 0 if the build succeeded

# Make your script report failure explicitly:
if ! curl -fsS http://localhost:3000/health > /dev/null; then
  echo "Health check failed!" >&2   # errors go to stderr
  exit 1                             # non-zero → callers know it failed
fi
echo "Healthy."
exit 0

This matters enormously in pipelines: a CI job or deploy step is considered failed precisely when its script exits non-zero. If your deploy script exits 0 even though the deploy broke, the pipeline thinks everything's fine and moves on. Writing correct exit codes — exit 1 on failure, exit 0 on success — is how you make automation trustworthy. (Note >&2 sends error messages to stderr, keeping them separate from normal output — important when logs are parsed by machines.)

2.4 Variables, arguments, and environment

Operational scripts are parameterized — the same deploy script works for staging or production based on inputs. Two sources of input: command-line arguments and environment variables.

#!/usr/bin/env bash
set -euo pipefail

# Positional arguments: $1, $2, ...  ($0 is the script name)
ENVIRONMENT="${1:-staging}"        # default to "staging" if no arg given

# Environment variables, with a required check:
: "${DEPLOY_TOKEN:?DEPLOY_TOKEN must be set}"   # -u + this errors clearly if missing

echo "Deploying to: $ENVIRONMENT"

"${1:-staging}" uses the first argument or defaults to staging. "${DEPLOY_TOKEN:?message}" aborts with your message if the variable is unset — the clean way to demand required config (Section 9 covers config and secrets properly). Always quote your variable expansions ("$VAR", not $VAR) — unquoted variables split on spaces and break on paths with spaces, a classic source of bugs. Environment variables are the standard way pipelines and containers pass configuration into scripts, so getting comfortable reading them is essential.

2.5 Idempotency — safe to run twice

A crucial operational property: an idempotent script produces the same result whether you run it once or ten times. This matters because ops scripts get re-run — a deploy retries after a network blip, a setup script runs on every boot. A non-idempotent script that appends a line to a config file adds it again on every run (corrupting the file); an idempotent one checks first, or writes the whole file fresh. Compare:

# ❌ NOT idempotent — appends every run, duplicating the line
echo "export PATH=/opt/app/bin:\$PATH" >> ~/.bashrc

# ✅ idempotent — only adds if absent
LINE="export PATH=/opt/app/bin:\$PATH"
grep -qxF "$LINE" ~/.bashrc || echo "$LINE" >> ~/.bashrc

# ✅ idempotent — mkdir -p succeeds whether or not the dir exists
mkdir -p /opt/app/releases

Aim for idempotency in anything that sets up or changes state. It's what makes automation safe to retry, which you'll need constantly — pipelines re-run, deploys get triggered twice, servers reboot. "Can I safely run this again?" should always be "yes."

2.6 A real example — a small build-and-check script

Here's an operational script tying it together: it builds an app, verifies the build produced output, and reports clearly. (This exact script was run.)

#!/usr/bin/env bash
set -euo pipefail

APP_DIR="${1:-.}"                 # where the app lives (default: current dir)
DIST_DIR="$APP_DIR/dist"

echo "==> Building app in $APP_DIR"
cd "$APP_DIR"

# Simulate a build step (in a real app: npm run build, go build, etc.)
mkdir -p "$DIST_DIR"             # idempotent
echo "console.log('hello');" > "$DIST_DIR/bundle.js"

# Verify the build actually produced the expected artifact:
if [[ ! -f "$DIST_DIR/bundle.js" ]]; then
  echo "Build failed: bundle.js not found" >&2
  exit 1
fi

BUNDLE_SIZE=$(wc -c < "$DIST_DIR/bundle.js")
echo "==> Build OK: bundle.js ($BUNDLE_SIZE bytes)"
exit 0

Every principle is here: the strict preamble, a parameter with a default, an explicit verification that fails loud (exit 1) if the artifact is missing, mkdir -p for idempotency, quoted variables, and a clear success exit. This is the shape of the real build/deploy/health-check scripts you'll write throughout the pipeline — and the shape CI systems expect: do the work, verify it, exit with the right code.

🏋️ Exercise 2.1

What does set -euo pipefail do, option by option, and why is each important in a deploy script? Give a concrete disaster each prevents.

Solution
  • -e — exit immediately if any command returns non-zero (fails). Prevents: a deploy script whose "build" step fails but which then runs the "ship to production" step anyway, deploying a broken/empty build. With -e, it stops at the failed build.
  • -u — treat use of an unset variable as an error instead of expanding it to empty string. Prevents: a typo or missing variable turning rm -rf "$DIR/" into rm -rf "/" (because $DIR was empty) — a classic way servers get wiped. With -u, the script errors on the unset variable instead.

Together they make the script fail fast and loud rather than continuing in a broken state — essential when the script controls production systems.

🏋️ Exercise 2.2

What is an exit code, and why is it critical in a CI/CD pipeline? What's wrong with a deploy script that always exits 0?

Solution

An exit code is the number a command/script returns when it finishes: 0 means success, any non-zero means failure. It's the standard way programs communicate their outcome to whatever ran them — the shell, a CI job, or an orchestrator checks it to decide whether a step succeeded.

It's critical in CI/CD because pipelines use exit codes to decide whether to proceed or stop: a job/step is considered failed exactly when its command exits non-zero, halting the pipeline and preventing a bad change from advancing. Exit codes are how automation "knows" whether each stage worked.

A deploy script that always exits 0 is dangerous because it reports success even when the deploy actually failed. The pipeline, seeing 0, believes everything is fine and continues — promoting a broken deploy to production, skipping rollback, marking the release green. The script has effectively lied to the automation, defeating the safety the pipeline is supposed to provide. Scripts must exit 1 (or another non-zero code) on real failures.

🏋️ Exercise 2.3

What does it mean for a script to be idempotent, and why does it matter operationally? Rewrite this non-idempotent line to be idempotent: bash echo "127.0.0.1 myapp.local" >> /etc/hosts

Solution

Idempotent means running the script multiple times produces the same result as running it once — re-running causes no additional/duplicate changes. It matters operationally because ops scripts get re-run all the time: pipelines retry after failures, deploys get triggered twice, setup scripts run on every boot. A non-idempotent script that appends on each run corrupts state (duplicate lines, growing files); an idempotent one is safe to retry, which automation constantly requires.

The given line appends the host entry on every run, duplicating it. Idempotent version — only add if it's not already present:

LINE="127.0.0.1 myapp.local"

grep -qxF "$LINE" /etc/hosts checks (quietly, whole-line, fixed-string) whether the entry already exists; the || echo ... >> appends it only if it doesn't. Running this repeatedly leaves exactly one copy of the line.

SECTION 3

Version Control & Git Workflows for Delivery

Git isn't just for saving code — in DevOps it's the source of truth that drives the entire pipeline. A push triggers CI; a tag triggers a release; the state of a branch decides what's in production. This section is about using Git as a delivery mechanism: the branching workflows teams use, how versions and releases work, and the increasingly central idea that Git should describe not just your code but your deployments.

3.1 Git as the trigger for everything

In a modern pipeline, Git events drive automation. Pushing to a branch kicks off CI; merging to the main branch may trigger a deploy; creating a tag may cut a release. This is why the workflow — how your team uses branches and merges — matters operationally: it defines when things build, test, and ship. Your branching strategy isn't just a collaboration convention; it's the control flow of your delivery pipeline. Choose it deliberately.

3.2 Branching workflows

Two dominant approaches, with a real trade-off:

Trunk-based development — everyone commits to (or very quickly merges into) a single main branch ("trunk"), in small, frequent increments. Branches are short-lived (hours to a day). This is the workflow most associated with high-performing DevOps: small, frequent integration means fewer merge conflicts, faster feedback, and a main branch that's always close to deployable. Its demands: strong automated testing (since main must stay releasable) and often feature flags to hide unfinished work that's already merged.

Git Flow (and similar long-lived-branch models) — separate long-lived branches for development, releases, and hotfixes, with features developed on their own branches and merged through a more ceremonial process. It offers structure for scheduled, versioned releases (think packaged software with distinct versions), but its long-lived branches cause painful merge conflicts and slow integration — the opposite of "ship small and often."

The modern DevOps lean is strongly toward trunk-based (or a lightweight feature-branch flow) for continuously-deployed web services, because frequent integration is what enables frequent, safe deployment. Git Flow still fits software with explicit versioned releases. The principle underneath: the longer a branch lives, the more it diverges and the riskier its merge — so keep branches short and integrate often.

3.3 Feature branches and pull requests

Even trunk-based teams usually use short-lived feature branches with pull requests (PRs) as the integration gate:

git switch -c feature/add-health-endpoint   # branch off main
# ... make small changes, commit ...
git push -u origin feature/add-health-endpoint
# open a Pull Request → CI runs automatically → teammate reviews → merge to main

The PR is where CI and human review meet: opening it triggers the automated build/test (Section 7), and a required passing check plus an approval gate merging. This is the everyday unit of change in DevOps — a small branch, a PR, green CI, a review, a merge that (often) deploys. Keeping each PR small makes review easier, CI faster, and any resulting problem trivial to trace to one change.

3.4 Tags and semantic versioning

A tag marks a specific commit as a named point — almost always a release:

git tag -a v1.4.0 -m "Release 1.4.0"
git push origin v1.4.0        # pushing the tag often triggers a release pipeline

Releases are conventionally named with Semantic Versioning (SemVer): MAJOR.MINOR.PATCH (e.g., 2.5.1):

  • MAJOR — incremented for incompatible/breaking changes (users must adapt).
  • MINOR — new functionality added in a backward-compatible way.
  • PATCH — backward-compatible bug fixes only.

So 1.4.2 → 1.5.0 signals new features that won't break you; 1.5.0 → 2.0.0 warns of breaking changes. SemVer gives everyone downstream a contract: the version number itself tells them how risky an upgrade is. In pipelines, tags are common release triggers — pushing v2.0.0 builds and publishes that exact version — which is why a clean tagging discipline matters operationally.

3.5 GitOps — Git as the source of truth for deployments

A powerful modern idea: GitOps extends "Git drives everything" all the way to infrastructure and deployments. The desired state of your system — which version runs where, what infrastructure exists, what config is applied — is declared in a Git repository, and automation continuously makes reality match what Git says. Deploying becomes a Git commit/merge; rolling back becomes reverting a commit; the repo is the single auditable record of what should be running.

The benefits are exactly the DevOps values made concrete: every change is version-controlled, reviewed (via PR), and auditable (Git history shows who changed what, when, and why); rollback is git revert; and there's no drift between "what we think is deployed" and "what's actually deployed," because Git is authoritative. You don't need a fancy GitOps tool to adopt the mindset: keep your deploy config, infrastructure definitions, and environment settings in version control, and change production by changing the repo. That single discipline brings review, history, and reversibility to operations.

3.6 A real example — a release-tagging helper

A small script that validates a version string against SemVer and creates an annotated tag (this exact validation logic was run):

#!/usr/bin/env bash
set -euo pipefail

VERSION="${1:?Usage: release.sh vMAJOR.MINOR.PATCH}"

# Enforce a SemVer-shaped tag (vX.Y.Z) before tagging — fail fast on typos.
if [[ ! "$VERSION" =~ ^v[0-9]+\.[0-9]+\.[0-9]+$ ]]; then
  echo "Invalid version '$VERSION' — expected vMAJOR.MINOR.PATCH (e.g. v1.4.0)" >&2
  exit 1
fi

echo "Tagging release $VERSION"
# git tag -a "$VERSION" -m "Release $VERSION"     # (real repo)
# git push origin "$VERSION"                       # triggers the release pipeline
echo "Would create and push tag $VERSION"

The regex ^v[0-9]+\.[0-9]+\.[0-9]+$ enforces the vX.Y.Z shape, so a mistyped tag (v1.4, 1.4.0, v1.4.0-oops) is rejected before it pollutes your release history — an example of the "fail fast on bad input" habit from Section 2 applied to release management. In a real repo the commented git tag/git push lines do the work; pushing the tag is what typically triggers the versioned release build.

🏋️ Exercise 3.2

Explain Semantic Versioning: what each of MAJOR, MINOR, PATCH means, and what upgrading from 1.4.2 to 2.0.0 versus to 1.5.0 tells a user.

Solution

Semantic Versioning (SemVer) formats a version as MAJOR.MINOR.PATCH: - MAJOR — incremented for incompatible/breaking changes; users may have to change their code to upgrade. - MINOR — incremented for new, backward-compatible functionality; safe to upgrade, adds features. - PATCH — incremented for backward-compatible bug fixes only; safe, no new features.

So upgrading: - 1.4.2 → 1.5.0 (minor bump) tells the user: new features were added, but nothing should break — a safe upgrade. - 1.4.2 → 2.0.0 (major bump) tells the user: there are breaking changes — they must review and possibly adapt their code before upgrading.

The value is that the version number itself communicates upgrade risk — a contract with everyone downstream about how cautious to be.

Solution

GitOps makes a Git repository the source of truth for the desired state of your system — which versions run where, what infrastructure exists, what config is applied — and uses automation to continuously make the real system match what Git declares. Deploying becomes committing/merging a change to the repo.

It delivers auditability because every change to what's running goes through Git: the commit history records who changed what, when, and why, and changes are reviewed via pull request before taking effect — a complete, reviewable audit trail with no undocumented manual changes. It delivers reversibility because reverting to a previous state is just reverting to a previous Git commit; since Git is authoritative, there's no drift between intended and actual state.

Rolling back a bad deployment in GitOps: git revert the commit that introduced it (or reset the declared state back to the previous known-good version) and push/merge — the automation then reconciles the running system back to that prior state. Rollback is an ordinary, auditable Git operation rather than a manual scramble.

SECTION 4

Build & Package: Artifacts and Dependencies

Between "code in a repo" and "software running on a server" sits a crucial step: turning source into a deployable artifact. This section covers reproducible builds, dependency management, and the "works on my machine" problem — which, unsolved, motivates the containers you'll meet in Section 5.

4.1 From source to artifact

Source code isn't what runs in production — a build runs. Building transforms source into a deployable artifact: compiled binaries, a bundled/minified JavaScript app, a .jar, a container image. The artifact is the unit you deploy, and a key DevOps principle governs it: build once, deploy many times. You build a single artifact, then promote that same artifact through staging and into production — never rebuild separately for each environment. Rebuilding per environment risks subtle differences (different dependency versions, different build machines) that mean the thing you tested isn't the thing you shipped. One artifact, tested once, deployed everywhere: that's how you guarantee production runs exactly what you verified.

4.2 Reproducible builds and lockfiles

For "build once, deploy everywhere" to be trustworthy, builds must be reproducible — the same source must produce the same artifact regardless of who builds it or when. The biggest threat is dependencies: if your build fetches "the latest compatible version" of each library, two builds a week apart can pull different code and behave differently. The fix is a lockfile — a file that pins the exact version (and often a checksum) of every dependency, direct and transitive:

package.json        →  declares  "express": "^4.18.0"   (a RANGE — "4.18 or compatible")
package-lock.json   →  pins      express 4.18.2 (+ every sub-dependency, exact + hashed)

package.json says what you want (a range); the lockfile records exactly what you got, down to every transitive dependency. Commit the lockfile and install from it (npm ci rather than npm install in CI) so every build — on your laptop, in CI, in the container — uses byte-for-byte identical dependencies. Other ecosystems have the same concept (poetry.lock, Cargo.lock, go.sum). Without a committed lockfile, "reproducible build" is a fiction and "works on my machine" is inevitable.

4.3 Separating build-time from run-time

A useful distinction: some dependencies and steps are needed only to build the app, others only to run it. Compilers, bundlers, test frameworks, and type checkers are build-time — needed to produce the artifact, useless (and bloat, and attack surface) in production. The runtime, the framework, and production libraries are run-time — needed by the running app. Keeping these separate (e.g., dependencies vs devDependencies in Node) keeps your production artifact lean: you don't ship the test framework and the TypeScript compiler to your servers. This distinction becomes concrete and powerful with multi-stage container builds (Section 5), where a heavy build stage produces the artifact and a slim final stage carries only what's needed to run.

4.4 The "works on my machine" problem

The oldest problem in deployment: code runs perfectly on the developer's laptop and fails on the server. The causes are always environmental differences: a different language runtime version, a missing system library, different OS, an environment variable set locally but not in production, a dependency that was installed globally on the dev's machine but never declared. The developer's environment has accumulated state that the build doesn't capture, so "the same code" runs in a different context and behaves differently.

Everything in this section chips away at it — reproducible builds and lockfiles pin the dependencies — but they don't pin the runtime and OS. That remaining gap (the language version, system libraries, OS-level differences) is exactly what containers solve by packaging the app together with its entire environment into one artifact that runs identically everywhere. Section 4's disciplines make dependencies reproducible; Section 5's containers make the whole environment reproducible. That's the through-line: the more of the environment you capture in a versioned, reproducible artifact, the less room "works on my machine" has to exist.

4.5 A real example — a reproducible-install check

An operational check that a project is set up for reproducible builds (this logic was run):

#!/usr/bin/env bash
set -euo pipefail

# A reproducible Node build requires a committed lockfile; CI should use `npm ci`.
if [[ ! -f package-lock.json ]]; then
  echo "No package-lock.json — builds are NOT reproducible. Commit a lockfile." >&2
  exit 1
fi

echo "Lockfile present. In CI, install with 'npm ci' (exact, from lockfile) — not 'npm install'."

# npm ci is stricter than npm install: it installs EXACTLY the lockfile, and fails
# if package.json and the lockfile disagree — catching drift before it ships.
echo "npm ci:"
echo "  - installs the exact dependency tree from package-lock.json"
echo "  - errors if package.json and lockfile are out of sync"
echo "  - deletes node_modules first for a clean, deterministic install"
exit 0

The check enforces the core reproducibility requirement: a committed lockfile. The distinction it teaches — npm ci (clean, exact, from the lockfile, fails on drift) versus npm install (may update the lockfile, resolves ranges) — is exactly why CI uses npm ci: it guarantees the build uses the precise dependencies you committed and tested, not whatever happens to resolve today. Same idea, different names, in every ecosystem.

SECTION 5

Containers with Docker

Containers are the technology that finally killed "works on my machine" — and they're the backbone of modern deployment. A container packages your app together with its entire environment into one portable, versioned artifact that runs identically on your laptop, in CI, and in production. This section covers the container model, Docker, writing a Dockerfile, images versus containers, and the multi-stage build pattern that produces lean production images.

Hands-on note: this environment has no Docker daemon, so the images below weren't built here — the Dockerfiles are correct, idiomatic templates to run with docker build on a machine that has Docker. (The commands are exactly what you'd use; nothing about them is pseudo-code.)

5.1 What a container is

A container is a lightweight, isolated package that bundles an application with everything it needs to run — code, runtime, system libraries, and dependencies — so it runs the same everywhere. Crucially, it does this without shipping a whole operating system: containers share the host's OS kernel and isolate only at the process level. That's the key difference from a virtual machine (VM): a VM virtualizes an entire machine including its own guest OS (heavy — gigabytes, slow to start), while a container virtualizes just the application's environment on top of the shared host kernel (light — megabytes, starts in milliseconds). You can run dozens of containers where you'd fit a handful of VMs.

The payoff for the "works on my machine" problem (Section 4): because the container carries its own runtime, libraries, and OS userland, the environmental differences that caused failures simply can't occur — the environment travels with the app. The container that passed your tests is bit-for-bit the container that runs in production.

5.2 Images vs containers

Two terms people conflate, but the distinction is fundamental:

  • An image is a blueprint — a read-only, built, versioned template containing your app and its environment. It's the artifact you build once and store.
  • A container is a running instance of an image — a live process. You can start many containers from one image, like many objects from one class.

The analogy that sticks: image is to container as class is to object (or as a program on disk is to a running process). You docker build an image once, push it to a registry (a store of images — Docker Hub, GitHub Container Registry, a cloud registry), then anywhere that needs to run the app pulls that image and starts a container from it. The image is the "build once, deploy many" artifact from Section 4 — and it captures the whole environment, not just dependencies.

5.3 The Dockerfile

A Dockerfile is a recipe for building an image — a sequence of instructions, each producing a layer. A straightforward one for a Node service:

# Start FROM a base image (a minimal Node runtime on a slim Linux)
FROM node:20-slim

# Set the working directory inside the image
WORKDIR /app

# Copy dependency manifests FIRST (for layer caching — see 5.4), then install
COPY package.json package-lock.json ./
RUN npm ci --omit=dev            # exact, reproducible, production deps only

# Now copy the rest of the source
COPY . .

# Document the port the app listens on
EXPOSE 3000

# The command that runs when a container starts
CMD ["node", "server.js"]

Key instructions: FROM picks a base image (start from an official, slim one); WORKDIR sets the directory; COPY brings files in; RUN executes build commands (like installing dependencies) at build time; EXPOSE documents the port; CMD defines what runs when the container starts. You build and run it with:

docker build -t myapp:1.4.0 .        # build an image, tag it with a version
docker run -p 3000:3000 myapp:1.4.0  # run a container, map host:container ports

Note the tag myapp:1.4.0 — tagging images with versions (ideally your SemVer/Git tag from Section 3) is how "build once, deploy many" and rollback work: you deploy myapp:1.4.0, and rolling back is just running myapp:1.3.0 again.

5.4 Layers and caching

Each Dockerfile instruction creates a layer — a cached filesystem diff. Docker reuses cached layers on rebuild until it hits an instruction whose inputs changed, then rebuilds that layer and everything after. This drives a critical ordering rule: put things that change rarely early, and things that change often late. That's why the Dockerfile above copies package.json/lockfile and runs npm ci before copying the rest of the source:

COPY package.json package-lock.json ./   # changes rarely
RUN npm ci --omit=dev                     # slow — but cached unless deps change
COPY . .                                  # changes on every code edit

Because dependencies change far less often than source code, this ordering means a normal code change reuses the cached (slow) dependency-install layer and only re-runs the fast final copy — turning minute-long rebuilds into second-long ones. Get the order wrong (copy all source first, then install) and every code change busts the cache and reinstalls all dependencies. Layer ordering is one of the highest-leverage Docker skills. (A .dockerignore file — like .gitignore — keeps node_modules, .git, and secrets out of the build context, making builds faster and images cleaner.)

5.5 Multi-stage builds — lean production images

The build-time vs run-time distinction from Section 4 becomes powerful here. A naive image includes all the build tools (compilers, dev dependencies) in the final image — bloated and a larger attack surface. A multi-stage build uses one stage to build and a second, slim stage that copies only the finished artifact:

# ---- Stage 1: build (has the full toolchain) ----
FROM node:20 AS build
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci                       # ALL deps, incl. build tools
COPY . .
RUN npm run build                # produces ./dist

# ---- Stage 2: runtime (slim; only what's needed to RUN) ----
FROM node:20-slim AS runtime
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci --omit=dev            # production deps only
COPY --from=build /app/dist ./dist   # copy ONLY the built artifact from stage 1
EXPOSE 3000
CMD ["node", "dist/server.js"]

The final image contains only the slim runtime, production dependencies, and the built dist — none of the compilers, dev dependencies, or source-only files from the build stage. The result is dramatically smaller (faster to push/pull, faster to start) and more secure (fewer tools for an attacker to exploit). COPY --from=build is the magic: it reaches into the earlier stage and takes just the artifact. This pattern — heavy build stage, slim runtime stage — is standard practice for production images.

5.6 A real example — a review checklist you can run

Since we can't build images here, here's an operational habit made concrete: a script that lints a Dockerfile for common production mistakes (this logic was run against the Dockerfiles above).

#!/usr/bin/env bash
set -euo pipefail
DOCKERFILE="${1:?Usage: check-dockerfile.sh <path>}"

warn() { echo "WARNING: $1" >&2; }

# 1. Pinned base image? ':latest' (or no tag) makes builds non-reproducible.
if grep -Eq '^FROM [^ ]+:latest' "$DOCKERFILE" || grep -Eq '^FROM [^:]+$' "$DOCKERFILE"; then
  warn "Base image uses :latest (or no tag) — pin a version for reproducible builds."
fi
# 2. Dependencies copied before source? (layer-cache ordering)
if grep -qE '^COPY \. \.' "$DOCKERFILE" && ! grep -qE 'package.*\.json' "$DOCKERFILE"; then
  warn "Copies all source without copying manifests first — busts the dependency cache."
fi
# 3. Reproducible install? npm ci, not npm install.
if grep -qE 'npm install' "$DOCKERFILE"; then
  warn "Uses 'npm install' — prefer 'npm ci' for reproducible image builds."
fi
echo "Checked $DOCKERFILE."

Nothing here needs a Docker daemon — it's just enforcing the section's lessons (pin the base image, copy manifests before source for caching, use npm ci) as an automated check, the same "fail loud on bad input" habit from Section 2 applied to image builds. Real teams use dedicated linters (like hadolint) for this, but the point stands: the qualities of a good image are checkable.

SECTION 6

Docker Compose & Multi-Service Local Environments

Real applications are rarely a single process — a web app needs a database, maybe a cache, maybe a background worker. Docker Compose lets you define and run a whole multi-container application from one declarative file, giving every developer an identical, one-command local environment. This section covers Compose, the services/networks/volumes model, and how it delivers "dev/prod parity."

Hands-on note: no Docker daemon here, so docker compose up wasn't run — but the Compose files below are real and their YAML was validated (parsed and linted) in this environment. Compose files are YAML, and getting the YAML right is most of the battle.

6.1 The multi-service problem

Your app (Section 5) is one container. But it needs PostgreSQL, and Redis for caching. Starting each by hand — right versions, right ports, right network so they can talk, right environment variables, in the right order — is tedious and error-prone, and every developer would do it slightly differently (recreating "works on my machine" at the environment level). You want one command that brings up the entire stack, identically, for everyone. That's Docker Compose: a single YAML file declaring all the services, and docker compose up to run them together.

6.2 The Compose file

docker-compose.yml declares your services declaratively:

services:
  web:
    build: .                     # build the app image from the local Dockerfile
    ports:
      - "3000:3000"              # host:container
    environment:
      - DATABASE_URL=postgres://app:secret@db:5432/appdb
      - REDIS_URL=redis://cache:6379
    depends_on:
      - db
      - cache

  db:
    image: postgres:16           # pull an official image
    environment:
      - POSTGRES_USER=app
      - POSTGRES_PASSWORD=secret
      - POSTGRES_DB=appdb
    volumes:
      - db-data:/var/lib/postgresql/data   # persist data across restarts

  cache:
    image: redis:7

volumes:
  db-data:                       # a named volume, managed by Docker

docker compose up reads this and starts all three containers wired together. Notice web reaches the database at the host db (postgres://...@db:5432) — not localhost. That's the key Compose feature: services can address each other by service name.

6.3 The model — services, networks, volumes

Three concepts:

  • Services — each entry under services: is a container (or a set of replicas) defined by an image (or a build) plus its config. web, db, and cache are services.
  • Networks — Compose automatically puts all services on a shared private network and gives each a DNS name equal to its service name. That's why web connects to db by the hostname db — Compose resolves it. Services are isolated from the outside except for the ports you explicitly publish.
  • Volumes — named, Docker-managed storage that persists beyond a container's life. Containers are ephemeral: delete one and its filesystem is gone. Databases need their data to survive restarts and redeploys, so you mount a volume (db-data) at the database's data directory. This is the crucial distinction between stateless app containers (throw them away freely) and stateful data that must be persisted in volumes (or, in production, a managed database).

6.4 Dev/prod parity

Compose's biggest operational value is environment parity — every developer, and CI, runs the same stack with the same versions, wired the same way, from the same file in version control. No more "it works for me but not for you because you have Postgres 14 and I have 16." A new team member runs docker compose up and has the entire application stack running in minutes, identical to everyone else's. This extends the "capture the environment in a versioned artifact" theme from single containers to the whole system: the Compose file is a checked-in, reproducible description of your local environment.

A caveat worth stating honestly: Compose is superb for local development and testing, and it can run simple production deployments, but large-scale production orchestration (many machines, auto-scaling, self-healing, rolling updates across a cluster) is the domain of orchestrators like Kubernetes. Compose's job in most shops is giving developers and CI a faithful, one-command local replica of the production stack. The mental model — declare services, let them talk over a network, persist state in volumes — carries directly into those bigger orchestrators.

6.5 A real example — validating a Compose file

Compose files are YAML, and most Compose breakage is either invalid YAML or a missing key. Here's a validator run in this environment against the file above (it actually executed):

# validate_compose.py — parse and sanity-check a docker-compose.yml (no daemon needed)
import sys, yaml

with open("docker-compose.yml") as f:
    doc = yaml.safe_load(f)          # fails loudly on invalid YAML

services = doc.get("services", {})
assert services, "no services defined"

for name, svc in services.items():
    # every service must either build or pull an image
    assert "build" in svc or "image" in svc, f"service '{name}' has neither build nor image"

# any named volume used by a service must be declared at the top level
declared = set((doc.get("volumes") or {}).keys())
for name, svc in services.items():
    for mount in svc.get("volumes", []):
        vol = mount.split(":")[0]
        if not vol.startswith((".", "/")):      # a named volume, not a bind mount
            assert vol in declared, f"service '{name}' uses undeclared volume '{vol}'"

print(f"OK: {len(services)} services, volumes {sorted(declared)}")

This catches the mistakes that actually bite — malformed YAML, a service with no image or build, a volume referenced but never declared — before you ever try to start the stack. It embodies the recurring habit: validate the declarative config before acting on it. (Teams also run docker compose config, which parses and normalizes the file, as a CI check.)

A database service needs a volume because its data (the actual records) must survive restarts, redeploys, and container replacement — losing it would be catastrophic. Mounting a volume at the database's data directory keeps the data safe across the container's life. A stateless web container doesn't need one because it holds no data worth preserving between runs — it just processes requests and can be thrown away and recreated at will (that disposability is exactly what makes scaling and deployment easy). The distinction — stateless app containers vs stateful data in volumes/managed databases — is fundamental.

Solution

Dev/prod parity means the development (and CI) environment closely matches production — same service versions, same wiring, same configuration — so code behaves the same way in each and "works in dev but not prod" failures are minimized.

SECTION 7

Continuous Integration

Continuous Integration (CI) is the first automated gate in your pipeline: every change is built and tested automatically, the moment it's pushed, so problems surface immediately instead of accumulating. This section covers what CI does and why, the anatomy of a CI workflow (using GitHub Actions), and the practices that make CI fast and trustworthy. The workflow YAML here was validated in this environment.

7.1 What CI is and the problem it solves

Before CI, developers worked in isolation for days or weeks, then merged everything together at the end — "integration" was a dreaded, days-long event where everyone's divergent changes collided ("integration hell"). Continuous Integration flips this: everyone integrates frequently (Section 3's trunk-based idea), and every integration is automatically verified by building the project and running the tests. Because changes are small and checked constantly, conflicts and breakages are caught within minutes of being introduced, while the change is small and fresh in the author's mind — not discovered weeks later buried under other work.

The essence: CI automatically builds and tests every change, giving fast feedback on whether it's safe to integrate. A push opens a pull request; CI runs; a red result blocks the merge; a green result (plus review) allows it. This is the mechanism that keeps the main branch continuously healthy and close to deployable — the precondition for continuous delivery.

7.2 What a CI pipeline does on every change

A typical CI run, triggered by every push/PR, does some or all of:

  • Checkout the code at the pushed commit.
  • Install dependencies reproducibly (from the lockfile — npm ci, Section 4).
  • Lint / format-check — catch style and obvious errors fast.
  • Type-check (if applicable) — e.g., tsc --noEmit.
  • Run the test suite (Section 12's tests) — the core of CI.
  • Build — produce the artifact, proving it can be built.
  • Report a pass/fail status back to the pull request.

The ordering usually goes fast-and-cheap first (lint, type-check) so quick failures report in seconds, with the slower steps (full test suite, build) after. The whole run's success is a single pass/fail signal on the commit — the green check or red X you see on a PR.

7.3 The anatomy of a GitHub Actions workflow

GitHub Actions defines CI in a YAML file under .github/workflows/. Its structure — workflow → jobs → steps — is representative of nearly every CI system:

# .github/workflows/ci.yml
name: CI

on:                          # WHAT triggers this workflow
  push:
    branches: [main]
  pull_request:

jobs:                        # a workflow has one or more JOBS (run in parallel by default)
  test:
    runs-on: ubuntu-latest   # the machine/runner to execute on
    steps:                   # a job is an ordered list of STEPS
      - uses: actions/checkout@v4          # a reusable action: check out the code
      - uses: actions/setup-node@v4        # install Node
        with:
          node-version: 20
          cache: npm                       # cache dependencies between runs (speed)
      - run: npm ci                        # reproducible install
      - run: npm run lint                  # fast checks first
      - run: npm run typecheck
      - run: npm test                      # the test suite
      - run: npm run build                 # prove it builds

The pieces: on declares the triggers (here, pushes to main and any pull request); jobs are units of work that run on fresh runners (runs-on); each job's steps run in order, either using a prebuilt action (uses:) or running a shell command (run:). If any step exits non-zero (Section 2's exit codes!), the job fails and the workflow reports red. This maps directly onto the pipeline mental model: CI is just an automated, hosted runner executing your build-and-test scripts on every change.

7.4 Matrix builds and caching

Two practical features you'll use constantly:

Matrix builds run the same job across multiple configurations in parallel — e.g., testing on several language versions or operating systems:

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
        node: [18, 20, 22]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "${{ matrix.node }}" }
      - run: npm ci
      - run: npm test

This one job definition runs six combinations (2 OSes × 3 Node versions) in parallel, verifying your code works across all of them — invaluable for libraries that must support many environments.

Caching saves and restores slow-to-produce files (like downloaded dependencies) between runs, so CI doesn't re-download everything every time. cache: npm in setup-node (shown above) caches the npm store keyed on your lockfile, cutting install time dramatically. Speed matters enormously: CI you wait ten minutes for gets run less and trusted less. Fast CI keeps the feedback loop tight.

7.5 CI best practices

What makes CI trustworthy (echoing themes from the Testing course):

  • Keep it fast. Order cheap checks first; cache dependencies; parallelize with jobs/matrix. Slow CI erodes the habit of running it.
  • Make failures block. A red CI must prevent merging (branch protection rules) — otherwise the signal is meaningless and broken code reaches main.
  • Keep main always green. The whole point is a continuously-healthy main branch; a persistently-red main means people stop trusting CI and merge over failures.
  • Fix flaky CI immediately. A test or step that fails intermittently (Testing course, Section 14) trains people to ignore red — the worst outcome. Quarantine and fix it.
  • Run the same checks locally. Developers should be able to run the exact lint/test/build commands CI runs, so they catch issues before pushing (pre-commit hooks help here).
  • One clear signal. The PR should show a single, unambiguous pass/fail so anyone can see at a glance whether a change is safe.

CI is the foundation everything else builds on: you can't do continuous delivery or deployment without trustworthy automated verification of every change first.

7.6 A real example — a validated CI workflow

The ci.yml from 7.3 and the matrix example from 7.4 were both parsed and linted in this environment to confirm they're valid, well-formed workflows (invalid YAML is the most common CI-setup failure). A small structural check like the one below is itself worth running in a repo:

# check_workflow.py — sanity-check a GitHub Actions workflow (ran here)

wf = yaml.safe_load(open(".github/workflows/ci.yml"))

# NOTE: YAML parses the bare key `on:` as the boolean True, so check both.
assert ("on" in wf) or (True in wf), "workflow has no triggers"
assert "jobs" in wf and wf["jobs"], "workflow defines no jobs"

    assert "runs-on" in job, f"job '{name}' has no runner (runs-on)"
    assert job.get("steps"), f"job '{name}' has no steps"

That on:-parses-as-True quirk is a genuinely useful thing to know — YAML treats the unquoted word on as a boolean, so tooling that inspects workflows has to account for it. The check confirms the workflow has triggers, at least one job, and that every job has a runner and steps — the structural minimum for a workflow that will actually run.

🏋️ Exercise 7.2

Describe the workflow → jobs → steps structure of a GitHub Actions (or any CI) pipeline. How does the exit code of a step determine the run's outcome?

Solution
  • Workflow — the whole automated pipeline, defined in a YAML file, with triggers (on:) that say when it runs (e.g., on push or pull request).
  • Jobs — a workflow contains one or more jobs, each running on a fresh runner (a machine, via runs-on). Jobs run in parallel by default (and can fan out via a matrix).

Exit codes determine the outcome: each step is a command, and if a step exits non-zero (fails — Section 2), that step fails, which fails its job, which makes the workflow report a red/failed status. If every step exits zero, the job passes; if all jobs pass, the workflow is green. This is why correct exit codes matter so much: CI's entire pass/fail signal is built from the exit codes of the underlying scripts/commands. A required-passing workflow then gates merging the pull request.

SECTION 8

Continuous Delivery & Deployment

CI proves a change is good; Continuous Delivery/Deployment (CD) gets that validated change to users. This section covers the CD half of the pipeline: environments and promotion, the deployment pipeline end to end, the delivery-vs-deployment distinction in practice, and a real, runnable deploy script with health-gated promotion.

8.1 From CI to CD

CI ends with a validated artifact (Section 4) — tested, built, ready. CD is everything that moves that artifact toward production: deploying it to environments, verifying it there, and promoting it onward. The guiding principle from Section 4 holds throughout: build once, deploy many — the same artifact that passed CI is the one promoted through staging and into production, never rebuilt. CD automates the path so that shipping a change is a routine, low-risk, push-button (or fully automatic) operation rather than a manual ceremony.

8.2 Environments and promotion

Software is deployed through a series of environments, each a complete running copy of the app used for a different purpose:

  • Development — where developers run and test locally (your Compose stack, Section 6).
  • Staging (a.k.a. pre-prod) — a production-like environment for final verification: same setup as production, real-ish data, where you catch problems before users do.
  • Production — the live environment real users hit.

A change is promoted through these in order: validated in CI, deployed to staging, verified there, then promoted to production. The reason for staging is risk reduction — it's a rehearsal on a production-like stage, so environment-specific problems (config, integrations, scale) surface where they're safe. The discipline: the same artifact moves up the chain, gaining confidence at each stage, so by the time it reaches production it's been exercised in a near-identical environment.

8.3 The deployment pipeline, end to end

Putting CI and CD together, an automated pipeline on a merge to main might:

merge to main
   │
   ├─ CI: install → lint → test → build         (Section 7)
   ├─ package: build & tag the image  app:<git-sha>   (Section 5)
   ├─ deploy to STAGING (run the new image)
   ├─ verify: health checks + smoke tests        ← gate
   ├─ [Continuous Delivery]  wait for manual approval
   │  [Continuous Deployment] no wait — proceed automatically
   ├─ deploy to PRODUCTION
   ├─ verify: health checks
   └─ on failure at any deploy step → ROLL BACK to previous image

Every box is something from earlier sections: CI (7), a tagged artifact (4/5), scripted deploys and health checks (2), and rollback (next). The pipeline is the assembly of these pieces into an automated flow from merge to monitored production. Note the verify gates: a deploy isn't "done" when the new version starts — it's done when health checks confirm it's actually serving correctly. A deploy that starts but fails its health check should trigger rollback, not celebration.

8.4 Delivery vs deployment, in practice

The distinction from Section 1 becomes concrete at the "wait for approval" box:

  • Continuous Delivery keeps production always deployable — the pipeline runs all the way up to production and then stops for a human to approve the final release (a manual gate, often required for regulated industries, high-stakes systems, or teams building confidence). The release is one click, on demand, from a known-good artifact.
  • Continuous Deployment removes that gate — passing all automated checks is the release; the change flows to production with no human in the loop.

The choice depends on how much you trust your automated verification. Continuous Deployment requires excellent tests, health checks, monitoring, and fast rollback (because there's no human safety net), which is why many teams practice Continuous Delivery — full automation with a human pressing the final button — and graduate to Continuous Deployment as their safety nets mature. Both are legitimate; the shared win is that the artifact is always ready and releasing is routine.

8.5 A real example — a health-gated deploy script

This deploy script models the core CD safety pattern — deploy, then verify with a health check, and only declare success if healthy — and it ran in this environment against a tiny local server. This is the heart of a real deploy step.

#!/usr/bin/env bash
set -euo pipefail

ENVIRONMENT="${1:?Usage: deploy.sh <environment> <version>}"
VERSION="${2:?Usage: deploy.sh <environment> <version>}"
HEALTH_URL="${HEALTH_URL:-http://localhost:3000/health}"

echo "==> Deploying $VERSION to $ENVIRONMENT"

# 1. "Release" the new version (in reality: run the new image / update the service).
#    Here we just record it, standing in for the orchestrator's release step.
echo "$VERSION" > current_version.txt

# 2. Health-gate: poll the health endpoint, retrying, before declaring success.
echo "==> Verifying health at $HEALTH_URL"
ATTEMPTS=5
for i in $(seq 1 "$ATTEMPTS"); do
  if curl -fsS "$HEALTH_URL" > /dev/null 2>&1; then
    echo "==> Healthy on attempt $i — deploy of $VERSION to $ENVIRONMENT SUCCESS"
    exit 0
  fi
  echo "   attempt $i/$ATTEMPTS: not healthy yet, retrying..."
  sleep 1
done

# 3. Never went healthy → fail loudly so the pipeline triggers rollback (Section 10).
echo "==> Health check FAILED after $ATTEMPTS attempts — deploy unhealthy" >&2
exit 1

The pattern is the important part: a deploy isn't complete when the process starts, it's complete when verification passes. The script retries the health check (real services take a moment to warm up — Section 10's readiness idea), succeeds only on a healthy response, and exits non-zero if it never becomes healthy — which is exactly what a pipeline needs to decide "this deploy failed, roll back." Waiting-for-condition (not sleep 30 and hope) is the same lesson as the async tests in the Testing course. This is a genuine, if minimal, deploy step.

The typical environments are development (local, where developers build and test), staging / pre-production (a production-like environment for final verification), and production (the live environment real users hit). A change is promoted through them in order.

🏋️ Exercise 8.2

In a deployment pipeline, what is a "verify gate," and why should a deploy be considered complete only when health checks pass rather than when the new version starts? What should happen if the health check fails?

A verify gate is a checkpoint after a deploy where automated health checks and/or smoke tests confirm the newly-deployed version is actually running and serving correctly before the pipeline proceeds (e.g., before promoting to the next environment).

A deploy should be considered complete only when health checks pass, not when the new version merely starts, because a process starting is not the same as it working: it might start and then crash, fail to connect to its database, be misconfigured, or return errors. SECTION 9

Configuration & Secrets

The same artifact runs in dev, staging, and production (Section 4's "build once, deploy many") — but it must behave differently in each: different database URLs, different API keys, different log levels. That difference is configuration, and getting it right — especially secrets — is where a lot of real incidents (and breaches) happen. This section covers separating config from code, the twelve-factor approach, and handling secrets safely. The config-loading code here was run.

9.1 Separate config from code

The foundational rule: configuration must live outside your code, not baked into it. If your database password or API endpoint is hard-coded, you'd need a different build per environment (breaking "build once, deploy many") and you'd have secrets sitting in your source history forever. Instead, the same artifact reads its settings from the environment it runs in. This is the core of the widely-cited Twelve-Factor App guidance: store config in the environment. Config is everything that varies between deploys — database URLs, credentials, hostnames, ports, feature flags, log levels — and it belongs outside the code, injected at runtime.

The practical test the twelve-factor methodology offers: could you open-source your codebase this instant without exposing any secrets? If the answer is no — if there are passwords or keys in the code — your config isn't properly separated. A clean codebase contains no environment-specific values and no secrets; it reads them all from outside.

9.2 Environment variables — the standard mechanism

The dominant way to inject config is environment variables — key/value pairs the operating system provides to a process. They're language-agnostic, supported everywhere (shells, containers, CI, cloud platforms), and require no special files in the image. Your code reads them at startup:

// config.js — read ALL config from the environment, with validation and defaults
function required(name) {
  const value = process.env[name];
  if (value === undefined || value === "") {
    throw new Error(`Missing required environment variable: ${name}`);
  }
  return value;
}

export const config = {
  port: Number(process.env.PORT ?? 3000),          // default for optional config
  databaseUrl: required("DATABASE_URL"),           // required — fail fast if absent
  logLevel: process.env.LOG_LEVEL ?? "info",
  environment: process.env.NODE_ENV ?? "development",
};

Two habits make this robust: validate required config at startup (fail immediately and loudly if a required variable is missing — the same fail-fast principle as Section 2, applied so a misconfigured deploy crashes on boot rather than mysteriously misbehaving later), and provide sensible defaults for optional config. In local development, tools load variables from a .env file (which is never committed — see 9.4); in production, the platform/orchestrator injects them. Because the mechanism is identical (process.env), the same code works everywhere.

9.3 Config vs secrets — a crucial distinction

Not all config is equally sensitive. Regular config (port, log level, feature flags, non-secret URLs) is harmless if seen. Secrets — passwords, API keys, tokens, private keys, database credentials — grant access and must be protected: never logged, never committed, never exposed. Both are injected as config, but secrets demand extra care because leaking one is a security incident. The rest of this section is mostly about that extra care, because secret mishandling is one of the most common and damaging operational mistakes.

9.4 Never commit secrets

The cardinal rule: secrets never go in version control. Once a secret is committed, it's in the Git history forever — even if you delete it in a later commit, it's still recoverable from history, and if the repo was ever pushed anywhere or is public, assume it's compromised. This happens constantly: scanners crawl public repos for committed AWS keys and exploit them within minutes. Protections:

  • .gitignore your secret files — .env, key files, credential files. The .env file holds your local secrets and must be listed in .gitignore so it's never committed. Commit a .env.example with the names of required variables but no values, so others know what to set.
  • Never hard-code secrets in source — read them from the environment (9.2).
  • Scan for leaked secrets — tools (git-secrets, trufflehog, GitHub's secret scanning) detect committed credentials; many teams run them in CI or as pre-commit hooks.
  • If a secret leaks, rotate it immediately — deleting the commit isn't enough; the only real fix is to revoke and replace the exposed credential, because you must assume it was captured.

9.5 Managing secrets properly

Beyond "don't commit them," where do secrets live? Options, roughly increasing in sophistication:

  • Environment variables injected by the platform — CI systems and cloud platforms have encrypted "secrets" stores (GitHub Actions secrets, the platform's secret manager) that inject values as environment variables at runtime, never exposing them in code or logs. This is the baseline good practice.
  • Dedicated secret managers — services like HashiCorp Vault, AWS Secrets Manager, or cloud key vaults store secrets encrypted, control access per-identity, audit every access, and can rotate secrets automatically. The app fetches secrets at runtime with a short-lived credential. This is the mature approach for anything sensitive at scale.

Two more essential practices: least privilege — each service gets only the specific secrets and permissions it needs, nothing more, so a compromise is contained; and rotation — secrets are changed periodically (and immediately on any suspected leak), so a stolen credential has a limited useful life. And a rule that catches people constantly: keep secrets out of logs. Accidentally logging a request that contains an auth token, or printing config (including secrets) at startup, quietly leaks credentials into log files that many people can read. Redact sensitive fields before logging (Section 13).

9.6 A real example — validated config loading

This config loader — reading and validating environment variables, failing fast on missing required ones — ran in this environment:

// config.js (Node) — ran with different env vars to show behavior
function required(name) {
  const v = process.env[name];
  if (v === undefined || v === "") throw new Error(`Missing required env var: ${name}`);
  return v;
}
const config = {
  port: Number(process.env.PORT ?? 3000),
  databaseUrl: required("DATABASE_URL"),
  logLevel: process.env.LOG_LEVEL ?? "info",

// Print config for demo — but REDACT anything secret-shaped:
const safe = { ...config, databaseUrl: config.databaseUrl.replace(/:\/\/.*@/, "://***:***@") };

Run with DATABASE_URL set, it loads and prints config with the credentials in the URL redacted (://***:***@) — never logging the raw secret. Run without DATABASE_URL, it throws Missing required env var: DATABASE_URL at startup and exits non-zero — a misconfigured deploy fails immediately and visibly instead of running in a broken state. Both behaviors (fail-fast validation, redacted logging) are exactly the disciplines this section is about.

Configuration must live outside the code because settings differ per environment (database URLs, credentials, log levels) while the artifact is the same everywhere. If config were hard-coded, you'd need a different build per environment — directly breaking "build once, deploy many" (Section 4), since the whole point is that one validated artifact runs in dev, staging, and production, differing only in the config injected at runtime. Externalized config lets the identical artifact behave correctly in each environment. (It also keeps secrets out of source.)

The twelve-factor litmus test: could you open-source your codebase right now without exposing any secrets? If not — if there are passwords, keys, or environment-specific values embedded in the code — then config is not properly separated. A clean codebase contains no secrets and no per-environment values; it reads them all from the environment.

🏋️ Exercise 9.2

Distinguish regular config from secrets. Why is committing a secret to Git dangerous even if you delete it in a later commit, and what must you do if a secret leaks?

Solution

Regular config (port, log level, feature flags, non-sensitive URLs) is harmless if seen. Secrets (passwords, API keys, tokens, private keys, database credentials) grant access and must be protected — never logged, committed, or exposed — because leaking one is a security incident.

Committing a secret is dangerous even if deleted later because Git retains full history: the secret remains recoverable from earlier commits forever, and if the repository was ever pushed/shared (or is public), you must assume the secret was already captured — scanners actively crawl public repos for credentials and exploit them within minutes. Deleting it in a new commit does not remove it from history.

If a secret leaks, you must rotate it immediately — revoke and replace the exposed credential (and update wherever it's used). Removing it from the repo/history is not sufficient, because you have to assume it was already captured; only revoking the credential actually closes the exposure.

🏋️ Exercise 9.3

Name three practices for managing secrets safely, and explain "least privilege" and why keeping secrets out of logs matters.

Solution

Three safe-secret-management practices (any three): - Never commit secrets — .gitignore secret files (.env), commit a value-less .env.example, read secrets from the environment, and scan for leaks. - Inject via encrypted stores — use the platform's secret store (e.g., CI/cloud secrets) or a dedicated secret manager (Vault, AWS Secrets Manager) that stores secrets encrypted, controls and audits access, and injects them at runtime. - Rotate secrets — change them periodically and immediately on any suspected leak, limiting a stolen credential's useful life. - Validate/fail fast on missing config at startup.

Least privilege means each service/identity is granted only the specific secrets and permissions it actually needs, nothing more. This contains compromise: if one service or credential is breached, the attacker gains access only to that narrow scope, not everything.

Keeping secrets out of logs matters because logs are widely readable and long-lived; accidentally logging a request containing a token, or printing full config (including secrets) at startup, silently leaks credentials into log files where many people (and systems) can see them — a common, easily-overlooked breach vector. Sensitive fields must be redacted before logging.

Deployment Strategies

How you replace the running version matters as much as what you deploy. Swap versions carelessly and users see downtime or errors; do it well and they never notice. This section covers the deployment strategies (recreate, rolling, blue-green, canary), zero-downtime deploys, health checks and readiness, and rollback — the safety mechanism that makes frequent deployment tolerable. 10.1 The problem — replacing a running version

When you deploy version 2 over version 1, there's a moment of transition, and how you manage it determines whether users experience downtime or errors. The naive approach — stop v1, start v2 — means the app is down during the swap, and if v2 fails to start, it's down until you fix it. For anything users depend on, that's unacceptable. Deployment strategies are the established patterns for transitioning between versions safely, trading off downtime, risk, and resource cost.

10.2 The strategies

Four common patterns, from simplest/riskiest to safest:

  • Recreate — stop all instances of v1, then start v2. Simple, but causes downtime during the swap and offers no gradual safety. Acceptable only where brief downtime is fine (internal tools, maintenance windows).
  • Rolling — replace instances incrementally: update a few at a time while the rest keep serving, until all run v2. No downtime (there are always healthy instances serving), and it needs no extra capacity. Downsides: during the roll, v1 and v2 run simultaneously (they must be compatible), and rolling back mid-deploy is slower. This is a common default.
  • Blue-green — run two full environments: "blue" (current, live) and "green" (new version). Deploy v2 to green, verify it fully, then switch all traffic from blue to green at once (a router/load-balancer change). Near-instant cutover, instant rollback (switch traffic back to blue), and you verify the new version completely before any user hits it. Cost: you need double the infrastructure during the deploy.
  • Canary — release v2 to a small subset of users/traffic first (say 5%), watch its metrics and errors closely, and if it's healthy, gradually increase the percentage until it's 100%. This limits the blast radius of a bad deploy — only a few users are affected if v2 is broken — and catches problems real synthetic tests miss, at the cost of more sophisticated traffic routing and monitoring.

The through-line: safer strategies reduce how many users a bad deploy can hurt and how fast you can undo it, usually by spending more resources or complexity. Choose based on how critical the service is and how much risk each deploy carries.

10.3 Zero-downtime deployment

Rolling, blue-green, and canary all aim for zero-downtime deployment — releasing without users ever seeing an interruption. Achieving it requires that at every moment during the deploy, some healthy instances are serving requests, and that traffic is only sent to instances confirmed ready. That's why zero-downtime depends on two things: a way to run old and new versions concurrently during the transition, and reliable health/readiness checks (next) so the system only routes traffic to instances that are actually able to serve. Done right, you deploy in the middle of the day, under load, and no user notices — which is the ultimate enabler of "ship small and often."

10.4 Health checks and readiness

Orchestrators and load balancers decide whether to send traffic to an instance based on health checks — endpoints your app exposes that report its status. Two distinct kinds, and the distinction matters:

  • Liveness — "is the process alive and not wedged?" If a liveness check fails, the orchestrator restarts the instance (it's broken).
  • Readiness — "is this instance ready to serve traffic right now?" A newly-started app might be alive but not yet ready — still connecting to its database, warming caches, loading config. Until readiness passes, the load balancer must not send it traffic. This is what enables zero-downtime rolling deploys: a new instance receives traffic only once it reports ready, so users never hit a half-started instance.
// health endpoints (Node) — the shape orchestrators expect; this server ran
const http = require("http");
let ready = false;
// Simulate startup work (DB connect, cache warm) completing after a moment:
setTimeout(() => { ready = true; }, 500);

http.createServer((req, res) => {
  if (req.url === "/health/live") {           // liveness: is the process up?
    res.writeHead(200); res.end("alive");
  } else if (req.url === "/health/ready") {    // readiness: safe to send traffic?
    res.writeHead(ready ? 200 : 503);
    res.end(ready ? "ready" : "not ready");
  } else {
    res.writeHead(200); res.end("hello");
  }
}).listen(3000);

The /health/ready endpoint returns 503 until startup finishes, then 200. A load balancer polling it holds traffic back until the instance is genuinely ready — exactly the gate that makes rolling/blue-green/canary deploys safe. (Your deploy script from Section 8 was a simpler version of this same "verify health before declaring success" idea.)

10.5 Rollback — the safety net

The single most important operational capability: fast rollback. No matter how careful your process, some bad deploys reach production — the honest goal isn't zero failures, it's fast recovery (recall the DORA metric "time to restore service"). Rollback means quickly returning to the previous known-good version. Because you followed "build once, deploy many" (Section 4) and tagged your artifacts (Section 5), rollback is simply redeploying the previous artifact — run app:1.3.0 again instead of the broken app:1.4.0. Blue-green makes it instant (switch traffic back to blue); with GitOps (Section 3) it's a git revert.

Why this matters so much: rollback is what makes frequent deployment psychologically and practically safe. If undoing a bad deploy takes hours of manual scrambling, every deploy is terrifying and teams deploy rarely. If it takes 30 seconds, a bad deploy is a minor, recoverable event — so you deploy often and fearlessly. The capability to reverse a change quickly is precisely what enables the courage to make changes quickly. Build, test, and rehearse your rollback path with the same care as your deploy path — a rollback you've never tested is not a rollback you can rely on.

Infrastructure as Code

So far you can build an artifact, containerize it, and deploy it — but onto what? Servers, networks, load balancers, and databases have to exist first, and how you create and manage them is its own discipline. Infrastructure as Code (IaC) applies the same principles you've used for application code — version control, review, reproducibility, automation — to the infrastructure itself. This section covers the concept, the declarative model, and Terraform's workflow.

11.1 The problem — click-ops and snowflake servers

The old way to set up infrastructure was manual: click around a cloud console, SSH into servers and run commands, configure things by hand. This "click-ops" has the same problems manual deployment did, amplified. It's not reproducible (nobody remembers the exact 40 clicks that produced the production database); it drifts (servers get tweaked over time until each is a unique "snowflake" no one fully understands); it's not reviewable or auditable (changes leave no record); and it doesn't scale (setting up a second identical environment by hand is error-prone and slow). When a hand-built server dies, recreating it exactly is a nightmare because its configuration lived only in someone's memory and a history of ad-hoc changes.

11.2 What Infrastructure as Code is

Infrastructure as Code means defining your infrastructure — servers, networks, load balancers, databases, DNS records, permissions — in machine-readable configuration files that are version-controlled, and using tools to create and update the real infrastructure from those files. Your infrastructure becomes code: reviewed via pull request, versioned in Git, reproducible on demand, and created by automation rather than by hand.

The benefits are exactly the DevOps values applied to infra: reproducibility (spin up an identical environment from the same files, every time — dev, staging, prod, or a fresh region), version control and auditability (every infrastructure change is a reviewed, recorded commit — you can see who changed what and why, and roll back), consistency (no more snowflakes — environments defined by the same code are the same), and automation (infrastructure provisioning becomes part of the pipeline). IaC is what makes "reproducible environment" true all the way down to the servers, not just the app.

11.3 Declarative vs imperative

The dominant IaC approach is declarative: you describe the desired end state — "there should be two web servers, a load balancer, and a database" — and the tool figures out how to make reality match, creating what's missing, changing what differs, and leaving alone what's already correct. This contrasts with an imperative approach (a script listing the exact steps: "create server, then create another, then..."), which is brittle and hard to re-run safely.

Declarative IaC has a superpower: idempotency (Section 2's concept, at the infrastructure level). You can apply the same configuration repeatedly and the tool only makes changes needed to reach the declared state — apply it when everything already matches and it does nothing. This is what makes IaC safe to run continuously and safe to use for detecting/correcting drift: the config is the single source of truth for what infrastructure should exist, and the tool continuously reconciles reality to it (the same idea as GitOps in Section 3, applied to infrastructure).

11.4 Terraform's model and workflow

Terraform is a widely-used, cloud-agnostic IaC tool. Its model has a few core concepts:

  • Providers — plugins for a specific platform (AWS, Google Cloud, Azure, and many others) that know how to create that platform's resources.
  • Resources — the infrastructure objects you declare (a server, a network, a DNS record), written in HCL (HashiCorp Configuration Language):
# main.tf — declares desired infrastructure (TEMPLATE — not applied in this course)
provider "aws" {
  region = "us-east-1"
}

resource "aws_instance" "web" {
  ami           = "ami-0abcdef1234567890"
  instance_type = "t3.micro"
  tags = {
    Name = "web-server"
  }
}
  • State — Terraform records what it has created in a state file, its map between your config and the real-world resources. This is how it knows, on the next run, what already exists and what needs to change. (In teams, state is stored remotely and shared, not on one laptop.)

The workflow is two commands that embody "see before you change":

terraform plan     # show exactly what WILL change (create/modify/destroy) — a dry run
terraform apply    # make it so — reconcile reality to the declared config

plan computes and displays the diff between the declared state and reality without changing anything — you review it like a code diff before applying. apply executes that plan, creating/updating/destroying resources to match your config. This plan-then-apply loop is what makes IaC safe: you never change infrastructure blind — you see the exact consequences first, review them, then apply. Rolling back is editing the config back (a reviewed commit) and applying again.

11.5 Why IaC matters

Tie it back to everything: IaC completes the reproducibility story. Containers (Section 5) made the application environment reproducible; IaC makes the infrastructure it runs on reproducible. Together, your entire system — app, environment, and infrastructure — is defined in version-controlled files, reviewable and rebuildable from scratch. If a region goes down, you can recreate your whole environment elsewhere from code. If you need a new staging environment, it's an apply away. Infrastructure changes get the same safety (review, history, rollback) as code changes. This is the endpoint of the DevOps automation journey: nothing about your system exists only as manual, undocumented, unreproducible state.

SECTION 12

Cloud & Hosting Fundamentals

Modern software runs on the cloud — someone else's data centers, rented on demand. You don't need to master any single provider, but you need the mental model: the service tiers, the core building blocks (compute, storage, networking), and how traffic actually reaches your app (load balancers, DNS, TLS, CDNs). This section is the conceptual map that orients you in any cloud.

12.1 What "the cloud" is

The cloud is on-demand access to computing resources — servers, storage, networking, databases — over the internet, rented from a provider (AWS, Google Cloud, Azure, and many smaller platforms) instead of owned. The shift it enabled is profound: instead of buying physical servers (huge upfront cost, weeks to provision, you own the maintenance), you rent exactly what you need, pay for what you use, and scale up or down in minutes. This elasticity — grow to handle a traffic spike, shrink when it passes — plus not having to run your own data center is why nearly all new software is cloud-hosted. The trade-offs are ongoing cost at scale and dependence on the provider, but the flexibility and speed usually win.

12.2 The service tiers — IaaS, PaaS, Serverless

Cloud offerings sit on a spectrum of how much the provider manages versus how much control you keep:

  • Infrastructure as a Service (IaaS) — you rent raw building blocks (virtual machines, storage, networks) and manage the OS, runtime, and app yourself. Most control and flexibility; most operational responsibility. (E.g., a plain cloud VM.)
  • Platform as a Service (PaaS) — you deploy your application and the provider manages the servers, OS, scaling, and infrastructure underneath. You give it your code (or container); it runs it. Much less ops work; less control. (E.g., app-hosting platforms.)
  • Serverless / Functions as a Service (FaaS) — you provide just functions or small units of code; the provider runs them on demand, scales them automatically (including to zero when idle), and you pay only for actual execution. Least ops, least control, and a different design model. There are still servers — you just don't manage them.

The trade-off is consistent: moving up the stack (IaaS → PaaS → serverless) trades control for reduced operational burden. More management by the provider means less work for you but fewer knobs to turn. Teams choose based on how much control they need versus how much ops they want to own — and often mix tiers.

12.3 The core building blocks

Whatever the tier, three fundamental resources underlie everything:

  • Compute — the processing power that runs your code: virtual machines, containers, or serverless functions. This is where your app actually executes.
  • Storage — where data lives. Two broad kinds: object storage (for files/blobs — images, backups, static assets; cheap, massively scalable) and block/database storage (for structured data your app queries). Storage is separate from compute so data persists when compute instances come and go (recall stateless containers, Section 6).
  • Networking — how everything connects and how traffic reaches you: virtual private networks, firewalls/security groups (controlling what can talk to what), and the public-facing entry points. Networking is also your first line of security — restricting access so only intended traffic reaches each component (least privilege, Section 9, at the network level).

Almost every cloud service is a specialized form of compute, storage, or networking. Recognizing which one you're looking at cuts through the overwhelming menu of provider offerings.

12.4 How traffic reaches your app

Four pieces turn "my app runs on a server somewhere" into "users can reach it reliably and securely":

  • DNS (Domain Name System) — translates human names (yourapp.com) into the IP addresses of your servers. It's the internet's phone book; a DNS record points your domain at your infrastructure (often at a load balancer).
  • Load balancer — sits in front of multiple instances of your app and distributes incoming traffic across them. It's essential for scaling (spread load over many instances) and reliability (route around unhealthy instances — using the health checks from Section 10). It's the single entry point behind which your instances hide, and the thing blue-green/canary deploys re-point.
  • TLS / HTTPS — encrypts traffic between users and your app so it can't be read or tampered with in transit. HTTPS (HTTP over TLS) is mandatory for any real service; certificates (often auto-provisioned and renewed now) prove your identity and enable the encryption. Terminating TLS is commonly the load balancer's job.
  • CDN (Content Delivery Network) — a globally-distributed cache that serves your static content (images, JS, CSS) from locations physically near each user, making the app faster worldwide and offloading traffic from your servers. Requests for cached content are answered by the nearest CDN edge instead of traveling to your origin.

Put together: a user types your domain → DNS resolves it to your load balancer → the connection is secured with TLS → the load balancer routes to a healthy compute instance → static assets come fast from a CDN. That path is the same in essence on every cloud.

12.5 A real deploy-target perspective

You don't have to start on a hyperscale cloud. For a small service like the ones in this course, PaaS platforms (which take your container or repo and run it, handling TLS, scaling, and a URL for you) are often the pragmatic choice — they collapse most of 12.4 into a managed default, letting you deploy a real, HTTPS-served app in minutes. As you grow, you move down toward IaaS for control, or adopt orchestration (Kubernetes) for scale. The point isn't which provider — it's that the concepts (tiers, compute/storage/networking, DNS/LB/TLS/CDN) are universal. Learn the model once and every provider's console becomes navigable: you're always looking at some arrangement of these same building blocks.

The three core building blocks are: - Compute — the processing power that runs your code (virtual machines, containers, serverless functions) — where the app executes. - Storage — where data lives: object storage (files/blobs — images, backups, static assets; cheap, massively scalable) and block/database storage (structured data the app queries). - Networking — how everything connects and how traffic reaches you (virtual networks, firewalls/security groups, public entry points); also a first line of security.

Almost every cloud service is a specialized form of one of these three.

Storage is kept separate from compute because compute instances are ephemeral — they come and go (scaling up/down, being replaced on deploy, failing and restarting — like stateless containers in Section 6). If data lived on the compute instance, it would be lost whenever that instance went away. Separating storage means data persists independently of any particular compute instance, so instances can be freely created and destroyed while the data remains safe. This separation is what makes elastic scaling and safe deployment possible.

Trace how a user's request reaches your app, naming the roles of DNS, load balancer, TLS, and CDN.

Solution

The path of a request:

  1. DNS (Domain Name System) — the user's browser looks up your domain name (yourapp.com) and DNS translates it to an IP address, typically pointing at your load balancer. DNS is the internet's "phone book" mapping names to addresses.
  2. TLS / HTTPS — the connection is secured with TLS, encrypting traffic between the user and your app so it can't be read or tampered with in transit (HTTPS = HTTP over TLS). A certificate proves your identity; TLS is often terminated at the load balancer.
  3. Load balancer — sits in front of multiple app instances and distributes incoming traffic across them, routing only to healthy instances (using health checks) and enabling scaling and reliability. It's the single public entry point behind which your instances hide.
  4. Compute instance — the load balancer forwards the request to a healthy instance of your app, which processes it and responds.
  5. CDN (Content Delivery Network) — for static content (images, JS, CSS), a globally-distributed cache serves it from an edge location physically near the user, making responses faster and offloading that traffic from your origin servers.

So: user → DNS resolves the name → connection secured by TLS → load balancer routes to a healthy compute instance → static assets served fast from the nearest CDN edge.

Observability: Logging, Metrics & Monitoring

You've deployed your app — but is it working? Not "did it start," but: is it serving users correctly, right now, under load? Observability is how you know. It's the practice of instrumenting systems so you can understand their internal state from the outside — through logs, metrics, and traces — and it's what turns "I hope production is fine" into "I can see that production is fine (and I'll be alerted the moment it isn't)." This section covers the three pillars, structured logging, the key metrics, and monitoring/alerting. The logging and metrics code here ran.

13.1 Why observability — you can't operate blind

The DevOps principle "measure and watch" (Section 1) lives here. Once your code runs on servers you can't see, you need a way to answer: Is it up? Is it fast? Is it erroring? How many users? Where's the bottleneck? Without instrumentation, you're flying blind — you find out about problems when angry users report them, and you debug by guessing. Observability gives you eyes: it's the ability to understand what your system is doing internally by looking at the outputs it emits. The goal is to detect problems before users do, diagnose them quickly when they happen, and understand your system's normal behavior so you can spot the abnormal. The classic framing is three pillars: logs, metrics, and traces.

13.2 Logs — and why structured logging wins

Logs are timestamped records of discrete events — "request received," "database query failed," "user logged in." They're your detailed narrative of what happened. But how you log matters enormously for operability. Traditional logs are unstructured text:

2026-01-15 10:32:01 ERROR Failed to process order 12345 for user alice: timeout

That's readable by a human but painful for machines to search and aggregate. Structured logging emits logs as structured data (usually JSON), with fields instead of a prose sentence:

// A tiny structured logger (this ran) — one JSON object per line
function log(level, message, fields = {}) {
  console.log(JSON.stringify({
    timestamp: new Date().toISOString(),
    level,
    message,
    ...fields,
  }));
}

log("error", "Failed to process order", {
  orderId: 12345, userId: "alice", reason: "timeout", durationMs: 3021,
});
// → {"timestamp":"...","level":"error","message":"Failed to process order",
//    "orderId":12345,"userId":"alice","reason":"timeout","durationMs":3021}

Why structured wins operationally: log aggregation systems can index, filter, and query these fields — "show me all error logs where reason is timeout in the last hour," "count errors by userId," "alert if durationMs > 5000." That's impossible (or fragile regex-hell) with prose logs. Structured logs turn your logs from a text file you grep in a panic into a queryable dataset. Two rules: include context (IDs, durations, relevant fields — enough to understand the event without the code in front of you) and never log secrets (Section 9 — redact tokens, passwords, credentials before they hit the log).

13.3 Metrics and the golden signals

Metrics are numeric measurements aggregated over time — request rate, error count, response time, CPU usage, memory. Unlike logs (individual events), metrics are the quantitative view: trends and totals you can graph and alert on. You could track hundreds, but a famous, focused starting point is the four golden signals (from Google's SRE practice) — the handful that best tell you if a user-facing service is healthy:

  • Latency — how long requests take (and watch the slow tail, not just the average — the 95th/99th percentile reveals pain the mean hides).
  • Traffic — how much demand: requests per second, throughput.
  • Errors — the rate of failing requests (5xx responses, exceptions). A rising error rate is often your earliest signal of trouble.
  • Saturation — how "full" your resources are: CPU, memory, disk, connection pools. High saturation predicts imminent problems (you're running out of headroom).

Watching these four tells you most of what you need about a service's health: is it slow (latency), overwhelmed (traffic/saturation), or broken (errors)? A common way to expose metrics is a /metrics endpoint the monitoring system scrapes:

// A minimal metrics endpoint (this ran) — counters exposed for scraping
let requestCount = 0, errorCount = 0;
// ...increment requestCount on each request, errorCount on failures...
function metrics() {
  return [
    `app_requests_total ${requestCount}`,
    `app_errors_total ${errorCount}`,
  ].join("\n");
}
// GET /metrics  → the monitoring system scrapes this periodically and stores the series

13.4 Monitoring, alerting, and dashboards

Collecting logs and metrics is useless if no one looks. Monitoring is the system that continuously gathers this telemetry, stores it, and lets you visualize and alert on it:

  • Dashboards visualize your metrics over time — graphs of the golden signals, so you can see trends, spot anomalies, and understand normal behavior. A glance tells you if things look right.
  • Alerting automatically notifies you (page, message, email) when a metric crosses a threshold — error rate spikes, latency climbs, disk fills, or the health check fails. This is what lets you find out about problems before (or as) users do, instead of after.

The art of alerting is alerting on symptoms that matter, not noise. Too many alerts (especially non-actionable ones) cause alert fatigue — people start ignoring alerts, and miss the real one. Good alerts are actionable (something a human should do), tied to user impact (alert that the service is erroring, not that one CPU briefly hit 80%), and rare enough to still command attention. Alert on "users are being affected," and let dashboards carry the rest.

13.5 Tracing — following a request

The third pillar, briefly. In systems made of many services, a single user request may hop through several of them. Distributed tracing tags each request with an ID and records its journey across services, so you can see the whole path and where the time went — invaluable for debugging "why is this endpoint slow?" when the answer is buried three services deep. For a single service it's less critical, but as systems grow into multiple services, tracing becomes the way you understand cross-service behavior. Logs tell you what happened in one place; metrics tell you how much/how often; traces tell you where a request went and where it spent its time.

SECTION 14

Reliability & Operations

The final discipline: keeping systems reliable over time, and responding well when — not if — they fail. This section covers the Site Reliability Engineering (SRE) mindset, service level objectives and error budgets, incident response and blameless postmortems, and the operational practices (scaling, backups, security basics) that separate systems that survive from those that don't. It's the operational maturity that everything else serves.

14.1 The SRE mindset — reliability as a feature

Site Reliability Engineering (SRE) is a discipline that applies engineering rigor to operations — treating reliability as a first-class feature you design and measure, not an afterthought you hope for. Its central, liberating insight: 100% reliability is the wrong target. Chasing perfect uptime is infinitely expensive and unnecessary — users can't even tell the difference between 99.99% and 100%, and the last fraction of a percent costs more than all the rest combined. Instead, you choose an appropriate reliability target for each service and manage to it deliberately. This reframes operations from "never fail" (impossible, paralyzing) to "fail rarely enough, and recover fast" (achievable, and it's what actually matters to users). Reliability becomes something you budget and trade off like any other engineering concern.

14.2 SLIs, SLOs, and error budgets

The SRE way to make reliability measurable:

  • SLI (Service Level Indicator) — a measurement of some aspect of service health, e.g., "the percentage of requests that succeed" or "the percentage served in under 200ms." It's a number derived from your metrics (Section 13).
  • SLO (Service Level Objective) — your target for an SLI, e.g., "99.9% of requests succeed over 30 days." The SLO is the reliability goal you commit to.
  • Error budget — the flip side of the SLO: if your target is 99.9% success, then 0.1% is your allowed failure — your "budget" for errors/downtime in the period. This is the powerful part: an error budget turns reliability into a resource you spend. As long as you're within budget, you can ship features fast and take risks; if you exhaust the budget (too many failures), you slow down and focus on stability until you're back on target. It aligns the classic dev-vs-ops tension (Section 1!) — velocity vs stability — into one shared, numeric agreement: ship fast while meeting the SLO. (And an SLA — Service Level Agreement — is a contractual promise to customers, usually looser than your internal SLO, with penalties if breached.)

The error-budget idea is quietly profound: it makes "how much can we risk?" an objective question with a data-driven answer, ending the endless argument between "ship more" and "stabilize more."

14.3 Incident response

When something breaks in production (an incident), how you respond determines the damage. Mature incident response has structure:

  • Detect — monitoring/alerting (Section 13) surfaces the problem quickly, ideally before users are widely affected.
  • Respond — someone is on-call (responsible for responding to alerts) and engages; for serious incidents, a clear incident commander coordinates so response isn't chaotic.
  • Mitigate first, diagnose later — the priority is restoring service, not immediately finding root cause. Often the fastest mitigation is your Section 10 safety net: roll back the recent deploy, failover, or scale up — stop the bleeding, understand it afterward. (This is the DORA "time to restore" metric in action.)
  • Communicate — keep stakeholders/users informed (a status page) during the incident.
  • Recover and learn — restore full service, then do a postmortem.

The single most important reflex: when in doubt during an incident, roll back. The recent change is the most likely culprit, and reverting to a known-good state is usually the fastest path to restoration — which is exactly why you invested in fast, tested rollback (Section 10).

14.4 Blameless postmortems

After a significant incident, you write a postmortem — a document capturing what happened, the timeline, the impact, the root cause, and (most importantly) what you'll change so it can't happen the same way again. The crucial adjective is blameless: the postmortem focuses on systemic causes and improvements, not on blaming the individual who pushed the button. This isn't just kindness — it's effectiveness. When people fear blame, they hide mistakes, and you lose the information you need to actually fix the system. A blameless culture treats incidents as learning opportunities: the person who caused an outage is usually the one who best understands how to prevent the next one, and punishing them just guarantees the next person conceals theirs. The goal is a system that's a little more resilient after every incident — you can't achieve that if fear suppresses the truth. Ask "how did the system let this happen and how do we improve it?", never "whose fault was it?"

14.5 Keeping systems reliable — the operational essentials

A grab-bag of practices that keep production healthy over time:

  • Scaling — handle growing load by scaling horizontally (more instances behind the load balancer — the usual cloud approach, enabled by stateless app design) rather than only vertically (a bigger single machine, which has a ceiling). Autoscaling adds/removes instances based on demand.
  • Backups and disaster recovery — regularly back up data and test that you can actually restore it (an untested backup is not a backup — a rule people learn the hard way). Disaster recovery is your plan for major failures: how you'd bring the system back, and how fast. IaC (Section 11) makes this dramatically easier — recreate infrastructure from code.
  • Redundancy — eliminate single points of failure: run multiple instances, across multiple availability zones, so one failure doesn't take you down. This is designing for failure — assume components will fail and ensure the system survives when they do.
  • Security basics — least privilege (Section 9) everywhere, keep dependencies and systems patched, encrypt data in transit (TLS, Section 12) and at rest, and never trust input. Security is an operational responsibility, not a separate team's problem.
  • Capacity and cost — watch that you have headroom (saturation, Section 13) and that you're not wildly overspending; the cloud's elasticity cuts both ways.

The unifying mindset across all of this: design for failure. Everything fails eventually — servers, networks, dependencies, deploys. Reliable systems don't assume things won't break; they're built so that when things break, the system degrades gracefully, recovers fast, and loses nothing important. That mindset — plus the automation, observability, and reversibility from every prior section — is what operations maturity actually means.

14.6 A note on culture

You've now seen the full technical arc — but DevOps began (Section 1) as a culture, and it ends there too. All these practices depend on shared ownership ("you build it, you run it"), blameless learning, and treating operations as everyone's concern rather than a wall to throw code over. The tools enable the culture, but the culture is what makes the tools matter. A team with mediocre tooling and a healthy culture of ownership and learning will out-operate a team with the best tools and a blame-driven culture every time.

  push: { branches: [main] }

jobs:
  ci:

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4

      - run: npm run lint
      - run: npm test

    runs-on: ubuntu-latest
    environment: production

      - run: ./scripts/deploy.sh production "${GITHUB_SHA}"

15.7 Deploy & rollback (these ran against the live app)

The heart of safe delivery (Sections 8, 10). deploy.sh records the current version as "previous," releases the new one, then polls readiness — succeeding only if healthy, exiting non-zero (→ triggering rollback) if not. rollback.sh restores the previous known-good version.

# scripts/deploy.sh (essence) — health-gated, records previous for rollback
cp "$CURRENT_FILE" "$PREVIOUS_FILE" 2>/dev/null || true
echo "$VERSION" > "$CURRENT_FILE"
for i in $(seq 1 10); do
  if curl -fsS "$HEALTH_URL" >/dev/null 2>&1; then echo "deploy SUCCESS"; exit 0; fi
  sleep 1
done
echo "unhealthy — roll back!" >&2; exit 1

Verified behavior (ran against the live Beacon service): deploying v1.4.0 succeeded and recorded it current; deploying v1.4.1 succeeded and recorded v1.4.0 as previous; rollback.sh restored v1.4.0; and deploying with the server down retried ten times and exited 1 — exactly the signal that triggers the workflow's rollback step. The rollback path was tested, not just written — as Section 10 insists.

🏋️ Exercise 15.1

Beacon's deploy.sh polls /health/ready (not /health/live) before declaring success, and records the previous version before releasing the new one. Explain why each of those two choices is correct, tying each to a course concept.

Solution

Polling /health/ready (not /health/live): readiness answers "is this instance ready to serve traffic right now?" while liveness only answers "is the process up?" (Section 10). A deploy should be declared successful only when the new version can actually serve users — it may be alive but still warming up (connecting to its database, loading config), during which it must not receive traffic. Gating on readiness ensures the deploy is judged "done" only when the service genuinely works (Section 8's verify-gate: a deploy is complete when health checks pass, not when the process starts). Gating on liveness would wrongly call a not-yet-ready instance a success.

Recording the previous version before releasing the new one: this makes rollback possible (Section 10). Because of "build once, deploy many" with tagged artifacts (Sections 4–5), rolling back means redeploying the previous known-good version — but you must know what that version was. Saving the current version as "previous" before overwriting it means that if the new deploy fails its health gate, rollback.sh can restore the exact prior version. It's the concrete mechanism behind "fast, tested rollback is the safety net that makes frequent deployment safe."

🏋️ Exercise 15.2

The Dockerfile is presented as a template that wasn't built, while deploy.sh was actually run against the app. Why is being explicit about that distinction itself a DevOps value? Relate it to how you should treat an un-verified deploy in real operations.

Solution

Being explicit about what was verified versus what is an unrun template is a DevOps value because operations is fundamentally about not trusting unverified claims of success. The entire course insists a deploy isn't "done" until health checks prove it works (Section 8), that you test your rollback path rather than assume it works (Section 10), and that exit codes must honestly report failure rather than always claiming success (Section 2). Presenting an un-built Dockerfile as if it were verified would be exactly the dishonesty those practices exist to prevent — a green checkmark that doesn't reflect reality. Labeling it "template, not run" is the documentation equivalent of a truthful exit code.

In real operations, an un-verified deploy should be treated as not-yet-successful — you don't assume it worked because it "should." You gate it behind health checks and smoke tests, watch its metrics, and keep rollback ready, only trusting it once verification passes. Claiming success you haven't confirmed is how broken changes reach users; honest distinction between "verified" and "assumed" is a core operational discipline, in code, in pipelines, and in communication.
