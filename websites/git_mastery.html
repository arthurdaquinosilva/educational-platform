The Ultimate Git Mastery — A Self-Study Course

A structured, hands-on Git course that takes you from your first commit to confident, everyday version control — and the workflows real teams rely on. All commands work on macOS, Linux, and Windows (Git Bash).

How to use this course

  • Open a terminal and follow along. Git is muscle memory — typing the commands is what teaches you.
  • Each section follows the same shape: Concept → Commands → Real example → Exercises with solutions.
  • Don't skip the exercises. Try them yourself first; only check the solution when stuck.
  • Create a throwaway directory (mkdir ~/git-practice) and break things freely. You can always rm -rf and start over.
SECTION 1

Getting Started

1.1 Introduction

Git is the most popular version control system in the world. It tracks your project's history, lets you undo mistakes, and lets teams work on the same code without overwriting each other.

By the end of this course you will be able to:

  • Create snapshots of your work (commits) and navigate between them.
  • Branch off to try ideas without breaking your main code.
  • Merge work back together, with or without conflicts.
  • Collaborate with others via remote repositories (GitHub, GitLab, etc.).
  • Rewrite history when you need to clean up your commits.

1.2 What is Git?

A version control system (VCS) tracks changes to files over time so you can:

  • See what changed, when, and by whom.
  • Go back to any previous version.
  • Try experimental changes without losing your stable work.
  • Collaborate with others on the same files.

Git specifically is:

  • Distributed. Every developer has a full copy of the project history. You don't need a server to work — you can commit offline, on a plane, anywhere.
  • Fast. Operations happen locally, so commits, diffs, and log views are instant.
  • Snapshot-based. Most VCSes track changes to files. Git tracks snapshots of the whole project. This sounds wasteful but is actually clever — and it's why Git's branching is so cheap.

Git was created by Linus Torvalds in 2005 to manage the Linux kernel. He named it after himself — "git" is British slang for an unpleasant person.

1.3 Using Git

There are several ways to use Git:

  • Command line — what we'll use. This is what every other interface calls under the hood, so learning it makes everything else clearer.
  • GUI tools — GitKraken, Sourcetree, GitHub Desktop. Friendlier visualization, but they hide what's happening.
  • IDE integration — VS Code, IntelliJ, Sublime all have built-in Git. Great for daily work once you understand the underlying commands.

We'll focus on the command line because it's universal, scriptable, and forces you to understand Git instead of memorizing button locations.

1.4 Installing Git

macOS:

# Easiest: install via Homebrew
brew install git

# Or: install Xcode Command Line Tools (also installs git)
xcode-select --install

Linux (Ubuntu/Debian):

sudo apt update
sudo apt install git

Windows: Download "Git for Windows" from git-scm.com. This installs git, plus a Bash shell called Git Bash that gives you a Unix-like terminal — use it instead of cmd.exe for this course.

Verify the install:

git --version
# Should print something like: git version 2.45.0

1.5 Configuring Git

Before your first commit, Git needs to know who you are. These settings get attached to every commit.

git config --global user.name "Your Name"
git config --global user.email "you@example.com"

Set your default editor (the program that opens when Git needs you to write a commit message):

git config --global core.editor "code --wait"   # VS Code
git config --global core.editor "vim"            # vim
git config --global core.editor "nano"           # nano

Set the default branch name for new repositories (modern convention is main):

git config --global init.defaultBranch main

Make sure line endings don't get mangled across operating systems:

# On macOS / Linux:
git config --global core.autocrlf input

# On Windows:
git config --global core.autocrlf true

View all your settings:

git config --list
git config --global -e   # opens the config file in your editor

The three config levels:

  • --system — for every user on this machine. Lives in /etc/gitconfig.
  • --global — for your user, across all repos. Lives in ~/.gitconfig.
  • (no flag, inside a repo) — for that one repo. Lives in .git/config.

More specific overrides less specific. So you can set a different email on a work repo while keeping a personal email globally.

1.6 Getting Help

When you forget a flag (you will, constantly):

git help config           # opens the full manual
git config --help         # same thing
git config -h             # short, one-screen reference

-h (lowercase) is your friend — it gives a concise summary you can scan in 5 seconds. The full man pages are excellent but long.

1.7 Cheat Sheet — Section 1

git --version                  # check installed version
git config --global user.name "..."
git config --global user.email "..."
git config --global init.defaultBranch main
git config --list              # show all config
git help <command>             # full manual
git <command> -h               # short help
SECTION 2

Creating Snapshots

The fundamental Git operation: taking snapshots of your project. Every section after this builds on these commands.

2.1 Initializing a Repository

A Git repository is just a regular folder with a hidden .git directory inside. To turn a folder into a repository:

mkdir my-project
cd my-project
git init

You'll see: Initialized empty Git repository in .../my-project/.git/

The .git folder is where Git stores everything — all your history, all your branches, all your config for this repo. Don't touch it manually. Deleting .git removes the entire history; the actual files in your folder are untouched.

To see this hidden folder:

ls -la              # macOS/Linux: lists hidden files
dir /a              # Windows cmd

2.2 Git Workflow

Git has three "states" your files can be in:

  Working Directory  →  Staging Area  →  Repository
       (edits)            (git add)       (git commit)
  1. Working directory — your actual files. Edit, delete, rename freely.
  2. Staging area (also called the "index") — a list of changes you've marked as ready to commit. Think of it as a draft of your next snapshot.
  3. Repository — the permanent history. Once committed, a snapshot is preserved forever (almost — see Section 6).

The staging area is what confuses most beginners. Why two steps? Because real commits are deliberate. You might edit 5 files but only want to commit 2 of them as one logical change. Staging lets you pick.

2.3 Staging Files

Add files to the staging area with git add:

git add file1.txt              # one file
git add file1.txt file2.txt    # multiple files
git add *.js                   # pattern
git add .                      # everything in current directory and below
git add -A                     # all changes anywhere in the repo

Check what's staged:

git status

You'll see something like:

Changes to be committed:
  new file:   file1.txt

Changes not staged for commit:
  modified:   file2.txt

2.4 Committing Changes

Once you've staged what you want, commit:

git commit -m "Initial commit"

-m provides the message inline. Without it, Git opens your editor for a longer message:

git commit

This is preferred for non-trivial commits because you can write a multi-line message:

Add user authentication

- Hashes passwords with bcrypt
- Adds /login and /signup routes
- Sets session cookies

The first line is the subject (under 50 chars, imperative mood: "Add", "Fix", "Remove" — not "Added" or "Adds"). Then a blank line, then the body explaining why.

2.5 Committing Best Practices

A good commit is:

  • Small. One logical change per commit. Don't mix "fix bug" with "refactor".
  • Atomic. The project should build and tests should pass at every commit.
  • Descriptive. "Fix bug" tells you nothing in 6 months. "Fix off-by-one error in pagination" does.
  • Imperative subject. "Add login button" not "Added" or "Adds" — completes the sentence "This commit will...".

These habits feel pedantic at first, but pay off enormously when you're hunting a regression months later with git bisect or trying to understand why someone made a change.

2.6 Skipping the Staging Area

For small changes you're sure about, -a stages all modified (already-tracked) files and commits in one step:

git commit -am "Fix typo in README"

Note: -a does not add new (untracked) files. You still need git add for those.

2.7 Removing Files

To delete a file from both the working directory and the repository:

git rm file.txt
git commit -m "Remove file.txt"

If you've already deleted the file manually with rm or your file explorer:

rm file.txt        # or just delete it in your file manager
git add file.txt   # stage the deletion (yes, "add" stages deletes too)
# or:
git add -A         # stage all changes including deletions

To remove a file from the repository but keep it on disk (e.g., you committed a config file by mistake):

git rm --cached config.json
git commit -m "Stop tracking config.json"
# Now add config.json to .gitignore so it doesn't come back

2.8 Renaming or Moving Files

git mv old-name.txt new-name.txt
git commit -m "Rename file"

This is exactly the same as:

mv old-name.txt new-name.txt   # rename via OS
git add old-name.txt new-name.txt

Git tracks renames by detecting content similarity — if a deleted file's contents match a new file's contents, Git infers a rename.

2.9 Ignoring Files

You almost never want to commit: - Build outputs (dist/, build/, *.class, *.o) - Dependencies (node_modules/, vendor/, __pycache__/) - IDE config (.vscode/, .idea/) - OS files (.DS_Store, Thumbs.db) - Secrets (.env, API keys, credentials)

Create a .gitignore file at the root of your repo:

# Dependencies
node_modules/

# Build output
dist/
*.log

# OS
.DS_Store
Thumbs.db

# IDE
.vscode/
.idea/

# Secrets — IMPORTANT
.env

Lines starting with # are comments. Trailing / means "directory". * matches any number of characters (except /).

Then commit it:

git add .gitignore
git commit -m "Add .gitignore"

Important: .gitignore only ignores files Git isn't already tracking. If you accidentally committed secrets.env first, adding it to .gitignore won't remove it. You'd need:

git rm --cached secrets.env
echo "secrets.env" >> .gitignore
git commit -m "Stop tracking secrets.env"

For boilerplate .gitignore files for any language, see github.com/github/gitignore — copy what fits your project.

🏋️ Exercise 2.9

Create a new repository. Make a folder called bin/ and put a file app.exe in it. Also create README.md. Configure .gitignore so bin/ is ignored. Verify with git status that only README.md and .gitignore show up.

Solution 
mkdir test-ignore && cd test-ignore
git init
mkdir bin
echo "fake binary" > bin/app.exe
echo "# My Project" > README.md
echo "bin/" > .gitignore
git status
# Should show: README.md, .gitignore (untracked); bin/ is ignored.

2.10 Short Status

git status is verbose. For a compact view:

git status -s

Output uses two-character codes:

 M file1.txt    (modified, not staged)
M  file2.txt    (modified, staged)
MM file3.txt    (modified, staged AND modified again after staging)
A  file4.txt    (newly added, staged)
?? file5.txt    (untracked)
D  file6.txt    (deleted, staged)

Left column = staging area. Right column = working directory. Once you internalize this, git status -s becomes your default.

2.11 Viewing Staged and Unstaged Changes

git status tells you which files changed. To see what changed:

git diff                  # unstaged changes (working dir vs staging)
git diff --staged         # staged changes (staging vs last commit)
git diff HEAD             # all changes (working dir vs last commit)

--cached is a synonym for --staged — they do the same thing.

Limit to one file:

git diff file.txt

Reading a diff:

diff --git a/file.txt b/file.txt
index 1234abc..5678def 100644
--- a/file.txt          # the "before" version
+++ b/file.txt          # the "after" version
@@ -1,3 +1,4 @@        # @@ chunk header: lines 1-3 became lines 1-4
 unchanged line
-removed line          # lines starting with - were removed
+added line            # lines starting with + were added
 another unchanged

2.12 Visual Diff Tools

Reading diffs in the terminal is fine for small changes. For big ones, configure a visual tool:

git config --global diff.tool vscode
git config --global difftool.vscode.cmd 'code --wait --diff $LOCAL $REMOTE'
git config --global difftool.prompt false

# Then:
git difftool
git difftool --staged

Other popular options: Meld, Beyond Compare, KDiff3, P4Merge.

2.13 Viewing History

The commit log is your project's story:

git log                   # full history, newest first
git log --oneline         # one commit per line
git log --oneline --reverse  # oldest first
git log -3                # last 3 commits
git log --stat            # show files changed per commit
git log -p                # show full diff of each commit

git log --oneline becomes your default — it gives you the commit hash and message in one line:

a3f1b9c (HEAD -> main) Add user authentication
9d2e7a4 Fix typo in README
0c1f8d3 Initial commit

That short hash on the left (a3f1b9c) is enough to identify the commit anywhere.

2.14 Viewing a Commit

To see the details of one specific commit:

git show a3f1b9c          # by hash
git show HEAD             # most recent commit
git show HEAD~1           # one before HEAD
git show HEAD~3           # three before HEAD

To see just one file in a past commit:

git show HEAD~2:file.txt

HEAD is Git's name for "where you currently are" — usually the latest commit on your current branch. HEAD~N means N commits back.

2.15 Unstaging Files

You staged something you didn't mean to. To unstage (keeping your changes):

git restore --staged file.txt

(In older Git: git reset HEAD file.txt. Both still work; restore is the newer, clearer command.)

2.16 Discarding Local Changes

You modified a file and want to throw away the changes (back to last commit):

git restore file.txt          # discard changes in working directory
git restore .                 # discard all working directory changes

This is destructive — the changes are gone. Use with care. If the file was previously untracked (new), restore won't remove it; use rm for that, or git clean:

git clean -fd     # remove all untracked files and directories. CAREFUL.
git clean -n      # dry run: show what would be removed

2.17 Restoring a File to an Earlier Version

If you've already committed a bad change and want to bring back the old version of a file:

git restore --source=HEAD~1 file.txt

This puts the version from 1 commit ago into your working directory. The file is now modified — stage and commit it normally to make the restoration permanent.

2.18 Creating Snapshots with VS Code

VS Code has built-in Git support — the Source Control panel (Cmd+Shift+G / Ctrl+Shift+G) shows changes, lets you stage hunks of a file with a click, and commit. Useful for visual people, but learn the commands first so you know what's happening.

2.19 Cheat Sheet — Section 2

git init                          # create new repo
git status                        # what's changed
git status -s                     # short version
git add <file>                    # stage a file
git add .                         # stage everything
git commit -m "message"           # commit staged changes
git commit -am "message"          # stage + commit tracked files
git rm <file>                     # remove file (and stage deletion)
git mv <old> <new>                # rename
git diff                          # unstaged changes
git diff --staged                 # staged changes
git log --oneline                 # compact history
git show <hash>                   # details of a commit
git restore --staged <file>       # unstage
git restore <file>                # discard local changes
SECTION 3

Browsing History

Now that you can make commits, you need to navigate the history that builds up. This section is everything you can ask Git about your past.

3.1 Introduction

Every commit has: - A unique hash (e.g., a3f1b9c2...) — 40 hex chars, but the first 7 are enough to identify it. - An author (name + email). - A timestamp. - A parent (the commit before it). - A message. - A snapshot of every tracked file at that moment.

The tools in this section help you query all of that.

3.2 Getting a Repository

For these exercises, clone any public repo you like — or any repo you have lying around:

git clone https://github.com/some/repo.git
cd repo

If you don't have one, just keep practicing in your sandbox repo from Section 2.

3.3 Viewing the History

git log
git log --oneline
git log --oneline --stat       # + which files changed and how many lines
git log --oneline --patch      # + the actual diff
git log --graph --oneline      # ASCII branch graph (useful with branches)

The graph view becomes critical once you have branches:

* a3f1b9c (HEAD -> main) Merge feature branch
|\
| * 9d2e7a4 (feature) Add login page
| * 7c1d2b8 Add /login route
|/
* 0c1f8d3 Initial commit

3.4 Filtering the History

Show commits matching some criteria:

git log --oneline -3                       # last 3 commits
git log --author="Alice"                   # by author
git log --before="2026-01-01"              # before a date
git log --after="2 weeks ago"              # human-readable dates work
git log --grep="bug"                       # commit messages containing "bug"
git log -S"functionName"                   # commits that added/removed this string
git log --oneline -- file.txt              # commits that touched this file

-S is incredibly powerful — it searches the content of changes, not commit messages. Use it when hunting for "when did this function get added?"

3.5 Formatting the Log Output

Build your own format with placeholders:

git log --pretty=format:"%h %an %ar %s"
# a3f1b9c Alice 2 days ago Add user authentication
# 9d2e7a4 Bob   3 days ago Fix typo in README

Useful placeholders: - %h short hash · %H full hash - %an author name · %ae author email - %ad author date · %ar author date relative - %s subject (commit message first line) - %d ref names (branches, tags)

There are color codes too, but the default --oneline is usually enough.

3.6 Aliases

You'll type some commands hundreds of times. Save typing with aliases:

git config --global alias.lg "log --oneline --graph --all"
git config --global alias.st "status -s"
git config --global alias.unstage "restore --staged"

# Then:
git lg
git st
git unstage file.txt

These live in your global config and work in every repo.

3.7 Viewing a Commit

git show a3f1b9c                    # full diff + metadata
git show a3f1b9c --stat             # just which files changed
git show a3f1b9c:path/to/file.txt   # the file's content at that commit

HEAD relative references: - HEAD — current commit - HEAD~1 or HEAD^ — one before - HEAD~5 — five before - HEAD~1^ — same as HEAD~2

3.8 Viewing the Changes Across Commits

Compare two commits:

git diff HEAD~3..HEAD              # changes in last 3 commits
git diff HEAD~3..HEAD -- file.txt  # just for one file
git diff branch1..branch2          # branch comparison

The .. syntax means "from … to …".

3.9 Checking Out a Commit

To time-travel — see your project as it was at a specific commit:

git checkout a3f1b9c

This puts you in a state called detached HEAD. You can look around, run the code, but if you commit, those commits are not on any branch and easy to lose. Go back to normal:

git switch main         # or whichever branch you came from

(Older Git uses git checkout main for the same thing. switch and restore were introduced in 2019 to clarify the overloaded checkout.)

3.10 Finding Bugs Using Bisect

The killer feature. If you know a bug exists now but didn't exist 50 commits ago, git bisect does a binary search of your history to find the exact commit that introduced it.

git bisect start
git bisect bad                    # current commit is bad
git bisect good a3f1b9c           # this old commit was good

# Git checks out the middle commit. Test it.
# If it's bad:
git bisect bad
# If it's good:
git bisect good

# Repeat. Each step halves the range.
# After log2(N) steps you have the culprit.

git bisect reset                  # finish, go back to where you started

For 1000 commits, you find the bad one in 10 steps. This single command has saved me hours, many times.

3.11 Finding Contributors Using Shortlog

git shortlog -sn
# 142  Alice
#  87  Bob
#  12  Carol

Number of commits per author. Useful for "who knows this codebase".

3.12 Viewing the History of a File

git log -- file.txt
git log -p -- file.txt        # with diffs
git log --follow -- file.txt  # follow through renames

--follow is essential — without it, the history stops at the file's last rename.

3.13 Restoring a Deleted File

You deleted a file 5 commits ago and now want it back:

# Find when it was deleted
git log --diff-filter=D --summary | grep deleted

# Bring it back from before the delete
git checkout <commit-before-deletion>^ -- path/to/file.txt

The ^ (or ~1) is "one before". -- separates revisions from paths.

3.14 Finding the Author of a Line Using Blame

When you want to know "who wrote this line and why?":

git blame file.txt              # author + commit per line
git blame -L 10,20 file.txt     # lines 10-20 only
git blame -e file.txt           # show email instead of name

blame has a bad name — it's diagnostic, not accusatory. Most often you blame to find the commit that introduced a line, then git show <hash> to read the commit message and understand the intent.

3.15 Tagging

A tag is a permanent label on a specific commit. Used almost exclusively for marking releases:

git tag v1.0.0                          # lightweight tag on current commit
git tag v1.0.0 a3f1b9c                  # tag a specific commit
git tag -a v1.0.0 -m "Release 1.0.0"    # annotated tag (preferred)
git tag                                 # list all tags
git show v1.0.0                         # see what's at this tag
git tag -d v1.0.0                       # delete tag (locally)

Lightweight vs annotated: annotated tags (-a) store an author, date, and message — they're real Git objects. Lightweight tags are just pointers. Use annotated for releases; lightweight for temporary local bookmarks.

3.16 Cheat Sheet — Section 3

git log --oneline --graph --all       # branchy history
git log --author=name --since="..."   # filter
git log --grep="..." -S"text"         # search messages and content
git log --pretty=format:"..."         # custom format
git show <hash>                       # commit detail
git checkout <hash>                   # time-travel (read-only)
git switch main                       # back to normal
git bisect start / good / bad         # binary-search for bugs
git blame file                        # line-level history
git tag -a v1.0 -m "..."              # mark a release
SECTION 4

Branching

The heart of Git. Branching lets you work on multiple things in parallel without them stepping on each other.

4.1 What are Branches?

A branch is a pointer to a commit. That's it. Not a copy of files, not a clone of the repo — just a moveable label.

main:    a3f1b9c — 9d2e7a4 — 7c1d2b8        ← "main" points here
                                  ↑
                              (current)

feature:  a3f1b9c — 9d2e7a4 — 7c1d2b8 — fe1ab2c   ← "feature" points here
                                  ↑
                              shared with main

When you commit on feature, the feature pointer moves forward but main stays put. This is why branches are essentially free in Git — you're not copying files, just making a new pointer.

4.2 Working with Branches

git branch                          # list local branches; * marks current
git branch -a                       # include remote branches
git branch feature                  # create branch (but don't switch)
git switch feature                  # switch to it
git switch -c feature               # create AND switch in one step
git branch -d feature               # delete (safe — refuses if unmerged)
git branch -D feature               # force delete
git branch -m old-name new-name     # rename

(Older Git: git checkout -b feature was the create-and-switch idiom. switch -c is the modern equivalent.)

The * in git branch shows where you are:

  feature
* main

4.3 Comparing Branches

What's on feature that isn't on main?

git log main..feature              # commits in feature, not in main
git diff main..feature             # actual code differences
git diff main..feature -- file.txt # just one file

4.4 Stashing

You're working on something, and you need to switch branches right now — but you don't want to commit your half-finished work. Stash it:

git stash                          # save current changes (working dir + staged)
git stash push -m "WIP: login fix" # with a message

# do other stuff, switch branches, whatever

git stash list                     # see your stashes
# stash@{0}: WIP: login fix
# stash@{1}: WIP: earlier thing

git stash pop                      # apply most recent and remove from stash list
git stash apply stash@{1}          # apply specific one, keep in list
git stash drop stash@{1}           # delete a stash without applying
git stash clear                    # delete all stashes

The stash is a stack — pop takes the top. Use apply + drop if you want to apply something other than the top.

git stash only stashes tracked files by default. To include untracked:

git stash -u

4.5 Merging

Combining work from one branch into another.

git switch main
git merge feature           # bring feature's changes into main

Git tries to merge automatically. If the two branches changed different things, it just works. If they changed the same lines, you get a conflict (covered in 4.8).

There are several merge strategies:

4.6 Fast-Forward Merges

If main hasn't moved since you branched off, Git just slides the main pointer forward — no merge commit needed.

Before:
main:     A — B
              \
feature:       C — D       (main is at B; feature added C, D)

After: git merge feature (fast-forward)
main:     A — B — C — D    (main pointer just moved forward)

Result: linear history, no extra merge commit. Some teams love this; others want to preserve the fact that work happened on a branch.

Force a merge commit even if fast-forward is possible:

git merge --no-ff feature

4.7 Three-Way Merges

When both branches have moved since they diverged, Git creates a new "merge commit" that has two parents:

Before:
        A — B — E             ← main
             \
              C — D            ← feature

After: git merge feature
        A — B — E — M          ← main (M is the merge commit)
             \     /
              C — D

Commit M records "I'm the result of merging feature into main." The history preserves the branching structure.

4.8 Merge Conflicts

If both branches edited the same lines, Git can't decide which version wins. You'll see:

Auto-merging file.txt
CONFLICT (content): Merge conflict in file.txt
Automatic merge failed; fix conflicts and then commit the result.

Open the file. Git inserts conflict markers:

<<<<<<< HEAD
This is the line from main
=======
This is the line from feature
>>>>>>> feature

Edit the file to what you want (delete the markers), then:

git add file.txt
git commit                # finishes the merge

To bail out of the merge entirely:

git merge --abort

4.9 Graphical Merge Tools

When conflicts get complex, use a visual tool:

git config --global merge.tool vscode
git config --global mergetool.vscode.cmd 'code --wait $MERGED'

# When you have conflicts:
git mergetool

Or use VS Code's built-in conflict UI — click "Accept Current Change", "Accept Incoming", "Accept Both", or just edit manually.

4.10 Aborting a Merge

git merge --abort

Brings everything back to the state before git merge. Safe anytime before you finalize the merge with a commit.

4.11 Undoing a Faulty Merge

You finished the merge, but it was wrong. Two options:

Option A — Reset (rewrites history; only safe if not pushed):

git reset --hard HEAD~1     # go back one commit

Option B — Revert (safe even if pushed):

git revert -m 1 HEAD        # creates a new commit that undoes the merge

-m 1 tells Git which parent of the merge to use as the "mainline" (the one to keep). Almost always 1.

The golden rule: once you push, prefer revert over reset. More on this in Section 6.

4.12 Squash Merging

Take all the messy little commits on a feature branch and merge them into main as a single clean commit:

git switch main
git merge --squash feature
git commit -m "Add login feature"

Result: main shows one tidy commit instead of "WIP", "more work", "fix", "actually fix it", "ok working now". Common pattern for PR-based workflows.

Downside: you lose the granular history of how the feature was built. For exploratory work, that's a feature. For a long-running effort, you might want to keep the commits.

4.13 Rebasing

rebase is the alternative to merge. Instead of creating a merge commit, rebase moves your branch's commits on top of another branch — replaying them one by one.

Before:
        A — B — E              ← main
             \
              C — D             ← feature

After: git switch feature && git rebase main
        A — B — E — C' — D'    ← feature (C', D' are new commits with same changes)

Note C' and D' — they have new hashes because their parent changed. The original C and D still exist briefly (recoverable from reflog), but for practical purposes they're gone.

Why use it? Linear history. No merge commits cluttering the log. git log --oneline reads as a clean sequence.

When to use it: - ✅ On your local, unpushed branches. - ✅ To keep a long-running feature branch up to date with main. - ❌ On commits that have been pushed and pulled by others. You'll rewrite their history under their feet.

The Golden Rule: never rebase commits that exist outside your local repository.

git switch feature
git rebase main
# (resolve any conflicts; git add; git rebase --continue)
git switch main
git merge feature             # now this is a fast-forward

If you hit conflicts during rebase:

# edit the conflicting file, then:
git add file.txt
git rebase --continue

# or, give up:
git rebase --abort

4.14 Cherry Picking

Take one specific commit from another branch and apply it to yours:

git cherry-pick a3f1b9c

Useful when: - You wrote a bug fix on a feature branch and want to apply it to main now, without merging the rest of the feature. - Backporting a fix to an old release branch.

4.15 Picking a File from Another Branch

You don't even need a commit — grab a single file as it exists on another branch:

git checkout feature -- path/to/file.txt
# This stages the file. Commit normally.

(Modern syntax: git restore --source=feature -- path/to/file.txt)

🏋️ Exercise 4.15

Create a branch experiment, make 2 commits on it, then realize you only want to bring one of those commits back to main. How do you do it?

Solution 
git switch main
git log experiment --oneline    # find the hash of the commit you want
git cherry-pick <hash>
You can also cherry-pick a range: 
git cherry-pick start-hash..end-hash

4.16 Cheat Sheet — Section 4

git branch                          # list
git switch -c feature               # create + switch
git switch main
git merge feature                   # merge into current
git merge --squash feature          # combine into one commit
git merge --no-ff feature           # force merge commit
git merge --abort                   # bail out
git rebase main                     # replay current onto main
git cherry-pick <hash>              # apply one commit
git stash / pop / list / drop       # temporary save
git branch -d feature               # delete branch
SECTION 5

Collaboration

Working with other people requires remote repositories — copies of the project on a server everyone can reach. GitHub, GitLab, and Bitbucket are the most popular hosts, but the Git mechanics are identical regardless.

5.1 Workflows

Two common patterns:

Centralized workflow: - One shared remote repo. - Everyone clones it, pushes branches, opens PRs. - Common in companies. Simple.

Forking workflow: - The main repo is owned by someone (e.g., an open-source maintainer). - Contributors fork (make a copy on the server) and push to their fork. - Open a PR from your fork to the main repo. - Common in open source.

Both use the same commands; the difference is permissions.

5.2 Creating a GitHub Repository

On github.com → New Repository → fill in name, public/private, click Create. Don't initialize with a README if you're pushing an existing local repo — it'll cause a divergence on the first push.

To connect a local repo to GitHub:

git remote add origin https://github.com/you/repo.git
git push -u origin main
  • origin is the conventional name for "the main remote" — you can use anything.
  • -u (upstream) sets the default remote+branch so future git push and git pull know what to do.

5.3 Cloning a Repository

git clone https://github.com/user/repo.git
git clone https://github.com/user/repo.git my-folder    # into a specific folder

This: 1. Downloads everything (all branches, all history). 2. Sets up origin to point at the source URL. 3. Checks out the default branch.

5.4 Fetching

fetch downloads new commits from the remote but doesn't change your working files.

git fetch                     # all remotes, all branches
git fetch origin              # just origin
git fetch origin main         # just origin's main

After fetching, you can see what's new with:

git log HEAD..origin/main --oneline   # commits on remote that you don't have

Then either merge or rebase:

git merge origin/main
# or
git rebase origin/main

5.5 Pulling

pull = fetch + merge (by default) or fetch + rebase (with --rebase).

git pull                      # fetch + merge
git pull --rebase             # fetch + rebase (cleaner history)

Make --rebase the default for your repo:

git config pull.rebase true

Or globally:

git config --global pull.rebase true

This is a controversial setting — some teams prefer merge commits. Pick a policy and stick with it.

5.6 Pushing

Send your local commits to the remote:

git push                      # push current branch to its upstream
git push origin feature       # push feature to origin
git push -u origin feature    # push and set upstream (first time)

If the remote has commits you don't have, push is rejected:

! [rejected]        main -> main (fetch first)

The fix: git pull first, resolve any conflicts, then git push.

Never use --force (-f) on shared branches. It overwrites the remote history, deleting commits other people may have based work on. If you absolutely must force-push (e.g., to clean up your own feature branch after a rebase), use:

git push --force-with-lease   # safer: refuses if someone else pushed since

5.7 Storing Credentials

GitHub deprecated password auth in 2021. Options:

Personal Access Token (PAT): generate one in GitHub Settings → Developer Settings → Personal Access Tokens. Use it as your password. Git will cache it via the OS keychain on macOS/Windows.

SSH keys: generate ssh-keygen -t ed25519, add the public key to GitHub Settings → SSH Keys. Use SSH URLs (git@github.com:user/repo.git instead of https://...). No tokens to manage.

GitHub CLI: brew install gh && gh auth login — handles credentials for you.

Most developers settle on SSH for convenience.

5.8 Sharing Tags

Tags don't get pushed automatically:

git push origin v1.0.0          # one tag
git push origin --tags          # all tags
git push --follow-tags          # push tags reachable from pushed commits

To delete a tag from the remote:

git push origin --delete v1.0.0

5.9 Releases

A release is a GitHub/GitLab feature built on top of tags. Push a tag, then on GitHub → Releases → Draft New Release → pick the tag → add release notes and any binary attachments.

From the CLI:

gh release create v1.0.0 --notes "Initial release"

5.10 Sharing Branches

To make a local branch available on the remote:

git switch -c new-feature
# work, commit
git push -u origin new-feature

To check out a branch someone else pushed:

git fetch
git switch some-feature       # Git auto-creates local branch tracking origin/some-feature

List remote branches:

git branch -r

Delete a remote branch (e.g., after merging a PR):

git push origin --delete old-feature

5.11 Collaboration Workflow

Typical day-to-day:

# Start of the day — get latest changes
git switch main
git pull

# Start a feature
git switch -c add-search

# Work, commit, work, commit
git add .
git commit -m "Add search input"
git add .
git commit -m "Wire search to API"

# Push your branch
git push -u origin add-search

# Open a Pull Request on GitHub
# Wait for review, address feedback (more commits + push)
# After PR is merged, clean up:
git switch main
git pull
git branch -d add-search

5.12 Pull Requests

A pull request (PR) — also called merge request on GitLab — is a request to merge your branch into another. It's not a Git feature; it's a platform feature. PRs add:

  • A web UI for code review (line-by-line comments).
  • A discussion thread.
  • CI/CD checks (tests must pass).
  • Required approvals before merging.

The actual merge happens with git merge under the hood when you click "Merge". Most teams configure GitHub to use squash merge or rebase merge for clean linear history.

Best practices: - Keep PRs small. <400 lines is the sweet spot for review quality. - One concern per PR. - A clear title and description with context. - Self-review before requesting reviewers.

5.13 Resolving Conflicts

When a PR can't auto-merge, you'll see "This branch has conflicts that must be resolved" on GitHub.

git switch your-branch
git fetch origin
git merge origin/main          # or: git rebase origin/main
# resolve conflicts in the editor
git add <resolved-files>
git commit                     # if merging
git rebase --continue          # if rebasing
git push                       # push the resolution
# (use --force-with-lease if you rebased)

5.14 Issues, Labels, Milestones

GitHub features for tracking work:

  • Issues: bug reports, feature requests, tasks. Each has a number (#123) referenceable in commits ("Fixes #123" in a commit message auto-closes the issue when merged to main).
  • Labels: colored tags (bug, enhancement, good first issue).
  • Milestones: group issues into a release or sprint.

These don't affect Git itself, but they're how teams coordinate. Worth learning the conventions in whatever team you join.

5.15 Contributing to Open Source

The standard flow:

  1. Fork the repo on GitHub (button top-right). Creates a copy under your account.
  2. Clone your fork locally: git clone https://github.com/you/repo.git.
  3. Add the upstream remote so you can pull in changes from the original: bash git remote add upstream https://github.com/original-owner/repo.git
  4. Create a feature branch: git switch -c fix-something.
  5. Make changes, commit, push to your fork: git push -u origin fix-something.
  6. Open a PR from your branch to the original repo. Use the GitHub UI button "Compare & pull request".

5.16 Keeping a Forked Repository Up to Date

Your fork doesn't auto-sync with upstream. Periodically:

git fetch upstream
git switch main
git merge upstream/main         # or rebase
git push origin main            # update your fork on GitHub

GitHub now offers a "Sync fork" button in the UI that does this from the web.

5.17 Cheat Sheet — Section 5

git clone <url>
git remote -v                       # list remotes
git remote add origin <url>
git fetch                           # download, don't merge
git pull                            # fetch + merge
git pull --rebase                   # fetch + rebase
git push                            # send commits
git push -u origin branch           # first push, set upstream
git push --force-with-lease         # safer force-push
git push origin --tags              # push tags
git push origin --delete branch     # delete remote branch
git branch -r                       # list remote branches
git remote add upstream <url>       # for forks
SECTION 6

Rewriting History

The advanced stuff. You can edit, reorder, combine, and remove past commits. With great power comes great risk — this section comes with the most important rule in Git.

6.1 Why Rewrite History?

Real-world reasons:

  • You committed a password by mistake.
  • Your commit message has a typo.
  • You made 10 messy "WIP" commits and want them as 1 clean one.
  • You accidentally committed to the wrong branch.
  • A reviewer wants you to split one big commit into smaller ones.

Rewriting history is what makes Git feel like a writer's tool: drafts can be cleaned up before final publication.

6.2 The Golden Rule of Rewriting History

Never rewrite history that has been shared.

If you've pushed a commit and someone else has pulled it, rewriting that commit creates a divergence. Their next pull will get confused, and "fixing" it usually means losing work.

Safe to rewrite: - Commits only in your local repo. - Commits on your personal feature branch that no one else uses (and that you control).

Dangerous to rewrite: - Anything on main / develop / release after pushing. - Branches other people are working from.

If you must change shared history, communicate first — "Everyone, I'm rewriting feature-X, please re-clone or hard reset."

6.3 Example of a Bad History

* a3f1b9c WIP
* 9d2e7a4 more stuff
* 7c1d2b8 fix
* 0c1f8d3 fix again
* fe2ab1c WIP 2
* 8d4c5e6 ok working
* 1a2b3c4 oops
* 5f6e7d8 actual feature start

Useless for code review or git bisect. Let's learn the tools to fix it.

6.4 Undoing Commits

You committed something, then realized you didn't want to. There are three "reset" modes:

git reset --soft HEAD~1     # undo commit, keep changes staged
git reset --mixed HEAD~1    # undo commit, keep changes (unstaged) — DEFAULT
git reset --hard HEAD~1     # undo commit AND throw away the changes
  • --soft — "I committed too soon" — keeps everything staged, ready to re-commit.
  • --mixed — "I want to redo from staging" — keeps the changes but unstages them.
  • --hard — "I want this never to have happened" — changes are gone.

--hard is destructive. Always check git status first to make sure you don't have uncommitted work you care about.

6.5 Reverting Commits

If the commit has been pushed, revert instead of reset. Revert creates a new commit that undoes a previous one — no history rewrite.

git revert a3f1b9c              # creates a new commit undoing a3f1b9c
git revert HEAD                 # undo the most recent commit
git revert --no-commit a3f1b9c  # stage the inverse but don't commit (lets you combine)

Result history:

* d9e0f1a Revert "Add feature X"
* a3f1b9c Add feature X
* ...

Both commits exist; readers see what was done and what was undone.

6.6 Recovering Lost Commits

You ran git reset --hard and lost commits. Don't panic. Git keeps a reflog — a record of where HEAD has been, even when the underlying commits are no longer in the visible history.

git reflog
# a3f1b9c HEAD@{0}: reset: moving to HEAD~1
# 9d2e7a4 HEAD@{1}: commit: Add feature
# ...

The commits still exist for ~90 days (default garbage collection interval). To get one back:

git reset --hard 9d2e7a4
# or, to bring it back as a branch:
git branch recovered 9d2e7a4

reflog has saved my career multiple times. If you ever think you lost work, check the reflog before you panic.

6.7 Amending the Last Commit

You committed, then realized you forgot something:

# Make your edit, then:
git add forgotten-file.txt
git commit --amend
# Opens editor; edit message if you want, save, done.

# Or amend without changing the message:
git commit --amend --no-edit

Amend creates a new commit replacing the previous one. So this rewrites history — safe locally, dangerous if pushed.

6.8 Amending an Earlier Commit

For anything older than HEAD, use interactive rebase:

git rebase -i HEAD~5

Git opens an editor with the last 5 commits:

pick a3f1b9c Add feature X
pick 9d2e7a4 Fix typo
pick 7c1d2b8 Refactor login
pick 0c1f8d3 Add tests
pick fe2ab1c Update docs

Change pick to edit next to the commit you want to modify:

edit a3f1b9c Add feature X
pick 9d2e7a4 Fix typo
...

Save and quit. Git replays commits up to that one, then pauses. You can now amend:

# make changes
git add .
git commit --amend
git rebase --continue

Git replays the rest of the commits on top of your changes.

6.9 Dropping Commits

Same setup as 6.8 — git rebase -i HEAD~5. Change pick to drop (or just delete the whole line):

pick a3f1b9c Add feature X
drop 9d2e7a4 Fix typo
pick 7c1d2b8 Refactor login

The dropped commit is removed. Subsequent commits are replayed without it — which may cause conflicts if they depended on the dropped change.

6.10 Rewording Commit Messages

Change pick to reword:

reword a3f1b9c Add feature X
pick 9d2e7a4 Fix typo

Git pauses on each reword to let you edit the message.

6.11 Reordering Commits

In the interactive rebase editor, just rearrange the lines. Git replays in the new order.

pick 7c1d2b8 Refactor login
pick a3f1b9c Add feature X
pick 9d2e7a4 Fix typo

Beware: reordering commits that touch the same lines often causes conflicts. Tools like git mergetool help.

6.12 Squashing Commits

Combine multiple commits into one. Change pick to squash (or s):

pick a3f1b9c Add feature X
squash 9d2e7a4 WIP
squash 7c1d2b8 fix
squash 0c1f8d3 more fixes

This squashes the bottom three into the first. Git opens an editor combining all the commit messages so you can write a single coherent one.

Use fixup (or f) instead of squash to discard the squashed commits' messages and keep just the first one's.

This is the cleanup move for "10 WIP commits → 1 clean commit before opening a PR".

6.13 Splitting a Commit

Same rebase setup. Mark the commit as edit. When Git pauses:

git reset HEAD^               # undo the commit but keep changes
# Now stage piece by piece:
git add file1.txt
git commit -m "First logical change"
git add file2.txt
git commit -m "Second logical change"
git rebase --continue

One commit became two — useful when a reviewer says "this PR does too much".

6.14 Rewriting History — A Worked Example

Starting messy history:

* a3f1b9c Update docs
* 9d2e7a4 fix
* 7c1d2b8 fix
* 0c1f8d3 WIP
* fe2ab1c Add login feature

We want:

* X Add login feature
* Y Update docs

git rebase -i HEAD~5

Mark up:

pick fe2ab1c Add login feature
fixup 0c1f8d3 WIP
fixup 7c1d2b8 fix
fixup 9d2e7a4 fix
pick a3f1b9c Update docs

Save. Done. Clean history.

If anything goes wrong:

git rebase --abort

…and you're back to where you started. Always safe.

6.15 Cheat Sheet — Section 6

git reset --soft HEAD~1            # undo commit, keep staged
git reset --mixed HEAD~1           # undo commit, unstage (default)
git reset --hard HEAD~1            # undo commit, lose changes (DANGER)
git revert <hash>                  # undo via new commit (safe)
git commit --amend                 # edit last commit
git rebase -i HEAD~N               # interactive rebase last N commits
#   in the editor: pick / reword / edit / squash / fixup / drop
git rebase --continue              # resume after fixing
git rebase --abort                 # bail out
git reflog                         # recovery log of all HEAD movements
SECTION 7

Capstone Project: Run a Real Git Workflow

You've learned snapshots, history, branching, collaboration, and rewriting history. Now you'll put all of it together in a single, realistic workflow — the kind you'll run every day as a professional developer. This capstone isn't a program you build; it's a workflow you perform, start to finish, exactly as you would on a real team.

Unlike the other courses' capstones, the "artifact" here is your demonstrated ability — a GitHub repository whose commit history and pull requests show you can work like a professional. That history is genuinely something to show an employer: it proves you understand branching, code review, conflict resolution, and clean history — the things that separate someone who "knows some Git commands" from someone who can work on a team.

7.1 What You're Doing

You'll simulate the full lifecycle of a small feature on a team project:

  1. Create a repository and push it to GitHub.
  2. Set up a main branch with an initial project.
  3. Work on two feature branches — as if you were two different developers.
  4. Open pull requests for each.
  5. Deliberately create a merge conflict and resolve it.
  6. Use interactive rebase to clean up messy commits before merging.
  7. Merge everything cleanly and tag a release.

By the end, your repo's history and PR list look like real professional work. You'll have exercised every important Git skill in a realistic sequence.

Why this makes a good portfolio piece

  • Recruiters and engineers look at your GitHub. A repo with clean commits, meaningful messages, real branches, and resolved PRs signals competence immediately.
  • It demonstrates workflow, not just commands. Anyone can memorize git commit. Showing you can run a branch-based workflow with code review and conflict resolution is what teams actually need.
  • Conflict resolution is a fear-point for beginners. Demonstrating you can handle it confidently sets you apart.

7.2 The Scenario

You're building a simple project — let's use a small "quotes" collection (a text file or tiny script; the content barely matters, the workflow is the point). You'll play two roles:

  • Developer A, adding a "quote of the day" feature.
  • Developer B, adding an "author index" feature.

Both developers touch the same file, which is what creates the realistic conflict you'll resolve.

You can do this entirely solo — you're just switching branches to simulate two people. On a real team, these would be two colleagues, but the Git mechanics are identical.

7.3 The Full Workflow — Step by Step

Work through these in order. Type every command yourself; don't copy-paste blindly — the point is to build muscle memory.

Part 1 — Set up the project

mkdir quotes-project && cd quotes-project
git init

Create a starting file, quotes.md:

# Famous Quotes

1. "The only way to do great work is to love what you do." — Steve Jobs
2. "Simplicity is the ultimate sophistication." — Leonardo da Vinci

Commit it:

git add quotes.md
git commit -m "Initial commit: add quotes collection"

Create a repository on GitHub (empty, no README), then connect and push:

git remote add origin https://github.com/YOUR-USERNAME/quotes-project.git
git branch -M main
git push -u origin main

You now have a project on main, on GitHub. This is your starting point.

Part 2 — Developer A's feature branch

git switch -c feature/quote-of-the-day

Add a new section to quotes.md — a "Quote of the Day" heading at the top:

# Famous Quotes

## Quote of the Day
> "The only way to do great work is to love what you do." — Steve Jobs

1. "The only way to do great work is to love what you do." — Steve Jobs
2. "Simplicity is the ultimate sophistication." — Leonardo da Vinci

Commit it — but deliberately make two messy commits, so you can clean them up later with rebase:

git add quotes.md
git commit -m "add qotd"
# make a tiny follow-up edit (fix a typo, adjust formatting)
git add quotes.md
git commit -m "fix"

Push the branch:

git push -u origin feature/quote-of-the-day

Part 3 — Clean up commits with interactive rebase

Those two commits (add qotd and fix) are messy. A professional squashes them into one clean commit before merging. This is where interactive rebase (Section 6) pays off:

git rebase -i main

In the editor, squash the second commit into the first, and rewrite the message to something clear:

Add "Quote of the Day" section to quotes collection

Force-push the cleaned-up branch (safely — it's your own feature branch):

git push --force-with-lease

Your feature branch now has one clean, well-described commit instead of two sloppy ones. This is exactly the discipline teams expect.

Part 4 — Open a pull request

On GitHub, open a pull request from feature/quote-of-the-day into main. Write a real description: what the change does, why. Then merge it (on a real team, a colleague would review first).

After merging, update your local main:

git switch main
git pull origin main

Part 5 — Developer B's feature branch (and the conflict)

Now play Developer B. Branch from the current main:

git switch -c feature/author-index

Add an "Authors" section — but put it in the same top area of quotes.md that Developer A modified. This is what creates the conflict:

# Famous Quotes

## Authors Featured
- Steve Jobs
- Leonardo da Vinci

## Quote of the Day
> "The only way to do great work is to love what you do." — Steve Jobs

1. "The only way to do great work is to love what you do." — Steve Jobs
2. "Simplicity is the ultimate sophistication." — Leonardo da Vinci

Commit and push:

git add quotes.md
git commit -m "Add authors featured section"
git push -u origin feature/author-index

Part 6 — Create and resolve a conflict

Here's the realistic part. While Developer B was working, imagine main changed underneath them. Simulate that: switch to main and make a conflicting change directly (in reality this would be someone else's merged PR):

git switch main

Edit quotes.md — change the Quote of the Day to a different quote:

## Quote of the Day
> "Simplicity is the ultimate sophistication." — Leonardo da Vinci

Commit it:

git add quotes.md
git commit -m "Update quote of the day"
git push origin main

Now Developer B's branch is based on old main, and both changed the same area. Rebase B's branch onto the updated main:

git switch feature/author-index
git rebase main

Git stops with a conflict. Open quotes.md — you'll see the conflict markers:

<<<<<<< HEAD
## Quote of the Day
> "Simplicity is the ultimate sophistication." — Leonardo da Vinci
=======
## Quote of the Day
> "The only way to do great work is to love what you do." — Steve Jobs
>>>>>>> Add authors featured section

Resolve it by hand — decide what the correct combined result should be (keep the updated quote from main, keep your authors section), and delete the conflict markers. Then:

git add quotes.md
git rebase --continue

You just resolved a merge conflict during a rebase — the single most intimidating Git operation for beginners, and you did it. Push the rebased branch:

git push --force-with-lease

Part 7 — Merge and tag a release

Open a pull request for feature/author-index, review, and merge it on GitHub. Then update local main and tag a release:

git switch main
git pull origin main
git tag -a v1.0.0 -m "First release: quote of the day and author index"
git push origin v1.0.0

You now have a repository with: - A clean commit history. - Two merged feature branches via pull requests. - A resolved merge conflict in the history. - A tagged release.

That's a professional Git workflow, start to finish.

7.4 What Each Step Demonstrated

Worth understanding, and worth explaining in an interview.

Feature branches (Parts 2, 5). Real teams never commit directly to main. Every change happens on a branch, gets reviewed, then merges. Demonstrating this shows you understand team workflow, not just solo hacking.

Interactive rebase to clean history (Part 3). The difference between add qotd / fix / fix again and one clear Add "Quote of the Day" section commit is the difference between an amateur and a professional history. Clean history makes debugging (Section 3's git bisect, git blame) actually work.

--force-with-lease instead of --force (Parts 3, 6). After rebasing, you must force-push because history changed. --force-with-lease is the safe version — it refuses to overwrite if someone else pushed in the meantime, preventing you from clobbering a colleague's work. Using it shows you rewrite history carefully.

Pull requests (Parts 4, 7). The unit of code review. Even solo, using PRs builds the habit and produces a reviewable record of every change.

Conflict resolution during rebase (Part 6). This is the skill beginners fear most. Showing you can rebase onto updated main, resolve conflicts by hand, and continue — calmly — is a strong signal of real competence.

Tagging releases (Part 7). Marking versioned points in history. Shows you understand that software ships in versions.

7.5 Make It Yours — Extensions

Once you've run the base workflow, deepen it:

  • Add a real .gitignore and a README.md with a project description and a contribution guide.
  • Protect the main branch on GitHub (settings → branches) so direct pushes are blocked and PRs are required. Now you must use the workflow.
  • Use GitHub Issues — open an issue for each feature, reference it in your PR ("Closes #1"), and watch it auto-close on merge.
  • Try git bisect — introduce a "bug" in one commit, then use bisect to find which commit introduced it (Section 3).
  • Cherry-pick — practice moving a single commit from one branch to another with git cherry-pick.
  • Contribute to a real open-source project. Find a beginner-friendly repo (look for "good first issue" labels), fork it, make a small fix, and open a real PR. This is the ultimate demonstration — an actual contribution to real software, visible on your GitHub profile forever.

That last one is the real graduation. A merged pull request to a project you don't own is worth more than any tutorial. Employers love it.

7.6 Documenting It

Your repository is the portfolio piece here, so make it presentable:

  • Write a clear README.md explaining what the project is and, crucially, noting that it demonstrates a professional Git workflow (feature branches, PRs, conflict resolution, clean history, tagged releases).
  • Keep your commit messages clean and meaningful — they're on display.
  • Pin the repo on your GitHub profile.

When an employer looks at your GitHub and sees a repo with real branches, thoughtful PRs, a resolved conflict, and a clean history, they see someone who can drop into a team and work immediately. That's exactly the signal you want to send.

That's your capstone. You ran a complete, professional Git workflow — the same sequence of operations you'll perform every day as a developer. You branched, reviewed, rebased, resolved a conflict, and shipped. That's not "knowing Git commands." That's working like a professional.

🏋️ Final Exercise

Run the entire workflow in 7.3 yourself, start to finish, on a real GitHub repository. Don't skip the conflict — that's the most valuable part. When you're done, add at least one extension from 7.5.

Then, for the real challenge: find a beginner-friendly open-source project and open a genuine pull request. A single merged PR to a real project is the strongest Git portfolio evidence there is.

You've reached the end. Here's what you can now do:

  1. Take snapshots of your work — commit deliberately, with good messages.
  2. Browse history — log, diff, blame, bisect, find any change.
  3. Branch fearlessly — try ideas in isolation, merge when ready.
  4. Resolve conflicts without panic.
  5. Collaborate via remotes — clone, pull, push, fork, PR.
  6. Rewrite history when needed — and know when not to.

What to do next

  • Use Git daily. This is muscle memory. Even on solo projects, commit, branch, merge.
  • Read other repos' histories. Open a project you like on GitHub, click "Commits", read the messages. You'll absorb conventions.
  • Set up SSH keys with GitHub if you haven't. Stop typing passwords/tokens.
  • Configure aliases for your most-used commands. Mine: git lg (graph log), git st (status short), git unstage.
  • Learn git bisect. Most developers never do, and it's the single highest-value command in the toolkit.
  • Look into hooks (.git/hooks/) — automatic scripts that run on commit/push. Powerful for code formatting and pre-push checks.

Beyond the basics

  • GitHub Actions / GitLab CI — automate tests, builds, deploys when you push.
  • Git internals — the book Pro Git (free at git-scm.com/book) has a "Git Internals" chapter that's genuinely illuminating. Once you understand that everything is a hash-named blob, tree, or commit, every command makes sense.
  • Conventional commits — a structured format (feat:, fix:, docs:) that enables automated changelogs.
  • Trunk-based development vs Git Flow — two opposing branching strategies, both common. Read both, pick what fits your team.

Good luck. Git is one of those tools where every hour invested pays for itself a hundred times over.
