SECTION 1

The Web & HTML

1.1 What is HTML?

HTML stands for HyperText Markup Language. Three words, each important:

  • HyperText — text with links to other text. The defining innovation. You click a word and end up somewhere else entirely.
  • Markup — annotations around content that say "this is a heading", "this is a list", "this is a link". The browser interprets the markup and renders accordingly.
  • Language — there are rules, a grammar, a standard.

HTML is not a programming language. It has no variables, no loops, no conditionals. It's a declarative description of a document's structure.

1.2 A Tiny Bit of History

Understanding HTML's history explains all its quirks.

  • 1989 — Tim Berners-Lee, working at CERN, proposes a "hypertext information management system" to share research papers across computers. He invents HTTP, URLs, the first browser, and HTML — basically the entire Web.
  • 1991 — First HTML spec. 18 tags total. No images, no styles. Just text and links.
  • 1993 — Mosaic browser adds <img>. The Web has pictures.
  • 1995–1999 — The "Browser Wars". Netscape and Microsoft race to add features, each inventing their own tags (<blink>, <marquee>). HTML becomes a mess.
  • 1997 — HTML 4. The W3C tries to standardize. Introduces CSS as a separate concern.
  • 2000 — XHTML 1.0. The W3C decides HTML should be reformulated as strict XML — every tag must close, every attribute must be quoted. Developers grumble; many adopt it anyway.
  • 2004 — A group of browser makers (Apple, Mozilla, Opera) form WHATWG, frustrated with the W3C's direction. They start work on what becomes HTML5.
  • 2008 — HTML5 draft. New semantic tags (<header>, <article>, <nav>), native video/audio, canvas, local storage.
  • 2014 — HTML5 finalized.
  • 2019 → today — HTML is now a "living standard" maintained by WHATWG. There's no "HTML6" — the standard just continuously evolves. The W3C effectively ceded control.

Why this matters: HTML's forgiving nature — <p> doesn't strictly need </p>, tags can be lowercase or uppercase, attributes can be unquoted — is a deliberate design choice from the 1990s, baked in forever for backward compatibility. The Web never breaks old pages.

1.3 How a Browser Renders HTML

When you load a page, roughly this happens:

  1. Browser fetches the HTML file over HTTP.
  2. It parses the HTML, building a DOM (Document Object Model) — a tree of elements.
  3. It fetches and parses CSS, building the CSSOM (CSS Object Model).
  4. DOM + CSSOM combine into a render tree — what to draw.
  5. Layout calculates positions and sizes.
  6. Paint draws pixels to the screen.
  7. JavaScript can modify the DOM/CSSOM, triggering re-renders.

Each step is more expensive than the last. This is why "minimize reflow/repaint" became a performance mantra.

SECTION 2

Document Structure

2.1 The DOCTYPE

The first line of every HTML document:

<!DOCTYPE html>

This tells the browser "render in standards mode" — using current specs. Without it (or with an old DOCTYPE), browsers fall into quirks mode, emulating bugs from the 1990s for compatibility with ancient sites.

Historical note: in HTML 4 and XHTML, DOCTYPEs were monstrous URLs:

<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.01//EN" "http://www.w3.org/TR/html4/strict.dtd">

HTML5 mercifully simplified this to just <!DOCTYPE html>. It's not even technically HTML — it's a holdover from SGML that triggers standards mode and is otherwise ignored. Always use it.

2.2 The <html> Element

The root of every page. Always set the language:

<html lang="en">

The lang attribute helps: - Screen readers pronounce content correctly. - Search engines index by language. - Browser translation tools detect when to offer translation.

Use BCP 47 language codes: en, en-US, pt-BR, ja, zh-Hans.

2.3 The <head>

Metadata about the document — not displayed to users:

<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <meta name="description" content="A page about cats.">
  <title>My Page Title</title>
  <link rel="stylesheet" href="styles.css">
  <link rel="icon" href="favicon.ico">
  <script src="script.js" defer></script>
</head>

2.4 The Charset Declaration

<meta charset="UTF-8">

Tells the browser how to decode bytes into characters. UTF-8 is the modern universal encoding — supports every language, every emoji. Without this, accented characters and non-ASCII text may display as mojibake.

This should be the very first thing inside <head>. Browsers buffer the first 1024 bytes looking for it.

2.5 The Viewport Meta Tag

<meta name="viewport" content="width=device-width, initial-scale=1.0">

Crucial for mobile. Without it, phone browsers pretend to be 980px wide and shrink your page to fit, making it tiny and unreadable. With it, the page renders at the actual device width.

This is the single most important line for responsive design. Add it to every page.

2.6 The <title>

<title>My Page Title</title>

What appears in: - The browser tab. - Bookmarks. - Search engine results.

Keep it under ~60 characters. Make each page's title unique and descriptive.

2.7 Meta Tags

<meta name="description" content="Brief page summary, 150-160 chars.">
<meta name="author" content="Your Name">
<meta name="robots" content="index, follow">

<meta name="twitter:description" content="Brief summary">
<meta name="twitter:image" content="https://example.com/preview.jpg">

Without these, your links look bare on social media. With them, you get rich previews.

2.8 The XHTML Detour (1999–2010)

For about a decade, the Web's official direction was XHTML — HTML reformulated as strict XML:

  • Every tag must close (<br />, not <br>).
  • Every attribute must have a quoted value.
  • Tag names must be lowercase.
  • Documents must be well-formed XML or browsers should refuse to display them.

The intent was good: stricter HTML would be easier to parse and process programmatically. But it failed for two reasons:

  1. Browsers refused to enforce it. They kept being lenient because breaking the Web was unacceptable.
  2. WHATWG forked off and made HTML5, which embraced HTML's forgiving nature and added features XHTML never would have.

XHTML lost. But its influence persists — modern HTML5 style guides still recommend lowercase tags, quoted attributes, and explicit closing tags because they're objectively clearer.

Why use these instead of <span style="font-weight: bold">?

3.4 <strong> vs <b>, <em> vs <i> — the great confusion

These pairs render visually the same by default but mean different things:

Tag Meaning When to use
<strong> Strong importance Warnings, key terms
<b> Stylistic bold without importance Product names, keywords
<em> Stress emphasis (changes meaning when spoken) "I never said that"
<i> Alternate voice — foreign words, technical terms Schadenfreude, Latin names

In practice, most developers use <strong> and <em> exclusively.

SECTION 4

Links & Navigation

The "HT" in HTML. Hypertext — clickable text that takes you somewhere else.

4.1 The Anchor Tag

<a href="https://example.com">Visit Example</a>

href (hypertext reference) is the destination. The content between the tags is what users click.

A real link, clickable:

<a href="https://example.com">Visit Example.com</a>
Result

4.2 Absolute vs Relative URLs

Absolute — complete URL, works from anywhere:

<a href="https://example.com/page.html">External link</a>

Root-relative — starts with /, relative to the site root:

<a href="/about.html">About</a>
<a href="/blog/post.html">Blog Post</a>

Document-relative — relative to the current page:

<a href="about.html">About (same folder)</a>
<a href="../index.html">Up one folder</a>
<a href="blog/post.html">Down one folder</a>

Rule of thumb: use root-relative (/about.html) for internal links — they don't break when you reorganize pages.

4.3 Fragment Identifiers

Link to a specific point within a page using #:

<!-- On the same page -->
<a href="#section-3">Jump to Section 3</a>
...
<h2 id="section-3">Section 3</h2>

<!-- On a different page -->
<a href="article.html#conclusion">Read the conclusion</a>

#top (special) takes you to the top of the page.

4.4 Link Targets

<a href="https://example.com" target="_blank">Opens in new tab</a>

target values: - _self — same window (default) - _blank — new tab/window - _parent — parent frame (rarely used) - _top — top frame (rarely used)

Security: when using target="_blank" to external sites, add rel="noopener noreferrer":

<a href="https://untrusted.com" target="_blank" rel="noopener noreferrer">Link</a>

Without noopener, the opened page can access your page's window.opener and redirect it — a phishing vector. Modern browsers default to noopener behavior, but adding it explicitly is still best practice.

4.6 The rel Attribute

Describes the relationship between current page and the link target:

SECTION 5

Images & Media

5.1 The <img> Tag

<img src="cat.jpg" alt="A black cat sitting on a windowsill">

src — the image source (URL or path). alt — alternative text. Always include this.

The alt attribute serves three purposes: 1. Screen readers read it aloud. 2. If the image fails to load, the browser displays the alt text. 3. Search engines use it to understand image content.

If an image is purely decorative (a divider, a flourish), use alt="" (empty). Don't omit it — that's different. alt="" tells screen readers "skip this".

When an image is missing, the browser shows the alt text in its place — useful! (the URL below is intentionally broken to demonstrate):

<img src="not-a-real-image.jpg" alt="A black cat sitting on a windowsill"
     style="border:1px dashed #aaa; padding:12px; color:#999;">
Result
<img src="logo.png" alt="">    <!-- Decorative, screen readers skip -->
<img src="logo.png">           <!-- Bad: screen reader will read the URL -->

5.2 Width and Height

<img src="photo.jpg" alt="A landscape" width="800" height="600">

Always include width and height (as HTML attributes, not just CSS). This lets the browser reserve space before the image loads, preventing "layout shift" — the annoying jump when text rearranges itself as images load in.

You can still resize with CSS (width: 100%; height: auto). The HTML attributes set the aspect ratio even when CSS overrides the size.

5.4 The <picture> Element

For serving different images based on conditions (format support, screen size):

<picture>
  <source srcset="cat.avif" type="image/avif">
  <source srcset="cat.webp" type="image/webp">

5.5 Responsive Images with srcset

Serve different sizes for different screen densities:

<img src="cat-400.jpg"
     srcset="cat-400.jpg 400w,
             cat-800.jpg 800w,
             cat-1600.jpg 1600w"
     sizes="(max-width: 600px) 400px, 800px"
     alt="A cat">

The browser picks the optimal size based on viewport width, device pixel ratio, and bandwidth. A user on a phone gets the 400px version; someone on a 4K display gets 1600px.

This was a huge HTML5-era addition. Before srcset, everyone shipped enormous images to phones.

5.6 Lazy Loading

<img src="below-fold.jpg" alt="..." loading="lazy">

Images marked loading="lazy" don't download until the user scrolls near them. Great for long pages with many images — load only what's visible.

The first few images on the page (above the fold) should stay as loading="eager" (default) — lazy-loading them delays your first impression.

7.5 The Dark Era: Table Layouts (Historical)

From roughly 1996 to 2008, designers used <table> for page layout:

<!-- Don't do this! Historical only -->
<table border="0" cellpadding="0" cellspacing="0" width="100%">
  <tr>
    <td colspan="3"><!-- header --></td>
  </tr>
  <tr>
    <td width="200"><!-- sidebar --></td>
    <td><!-- main content --></td>
    <td width="200"><!-- ads --></td>
  </tr>
  <tr>
    <td colspan="3"><!-- footer --></td>
  </tr>
</table>

Why? Because CSS layout in the late 90s was broken. float was buggy, position was inconsistent across browsers, and there was no flexbox. Tables were the only thing that worked reliably across IE and Netscape.

The consequences: - HTML and visual layout tangled together. Hard to maintain. - Accessibility nightmare. Screen readers tried to read tables as data, getting hopelessly confused. - Slow rendering. Browsers had to know the entire table before rendering anything. - Impossible to make responsive. A 3-column table doesn't gracefully become 1 column on mobile.

When CSS layout finally got reliable (around 2008 with broader browser support for float-based grids, then 2015 with flexbox, then 2017 with Grid), table layouts died — and good riddance.

There's still one place table layout survives: HTML emails. Email clients have terrible CSS support, so designers still use tables for email layout to this day. Welcome to 1999.

7.6 When to Use Tables Today

Use <table> for: - Spreadsheet-like data (comparisons, schedules, statistics) - Anything you'd put in a spreadsheet - Pricing tables (controversial — some are layouts, some are data)

Do NOT use <table> for: - Page layout (use Flexbox/Grid) - Form layout (use Flexbox/Grid) - Card galleries (use Flexbox/Grid)

A good test: would this data make sense in Excel or Google Sheets? If yes, it's a table.

SECTION 9

Accessibility

Accessibility (often abbreviated a11y — "a" + 11 letters + "y") means making the Web usable for people with disabilities. About 1 in 7 people worldwide has some form of disability — visual, motor, hearing, cognitive. The Web should work for them too.

This isn't optional kindness. In many countries it's legally required for businesses (ADA in the US, EAA in the EU). It's also just good engineering — most accessibility wins also help SEO and mobile UX.

9.1 Semantic HTML Is Most of the Job

If you use HTML elements for their intended purpose, you've done 80% of the work:

  • <button> for buttons (not <div onclick>).
  • <a href> for navigation (not <button onclick>).
  • <input type="checkbox"> for checkboxes (not styled <div>).
  • Heading hierarchy (<h1> → <h2> → <h3>, no skipping).
  • <label> for every form input.
  • Real lists (<ul>, <ol>) for lists, not <br>-separated text.

Each native HTML element comes with built-in keyboard, focus, and screen reader behavior. Custom widgets have to reimplement all of it (badly).

9.2 The alt Attribute

Every <img> needs alt:

<img src="cat.jpg" alt="A black cat curled up on a red blanket">
<img src="decorative-line.png" alt="">     <!-- decorative -->
<img src="logo.png" alt="The Paper Cat">    <!-- logo -->

Writing alt text: - Describe what the image shows, not "image of" or "picture of" (screen readers already say "image"). - For complex images (charts, diagrams), provide a longer description nearby in the page text. - Decorative images get alt="".

9.3 Keyboard Navigation

Every interactive element must work without a mouse. Test by pressing Tab repeatedly — can you reach every link, button, and form field? Is the focus visible?

<button>I'm focusable for free</button>
<a href="/about">Me too</a>
<input type="text">              <!-- focusable -->

<div onclick="...">Bad - not focusable, not keyboard-accessible</div>
<div role="button" tabindex="0" onclick="...">Better but still flawed</div>
<button onclick="...">Correct</button>

If you must make a non-button focusable, use tabindex="0". Don't use tabindex with positive numbers — it breaks the natural tab order in confusing ways. Use only 0 (in tab order) or -1 (focusable by JS, not in tab order).

9.4 Focus Styles

Browsers add an outline around focused elements. Designers often remove it. Don't.

/* DON'T: */
*:focus { outline: none; }

/* DO: replace with a custom style */
*:focus-visible {
  outline: 2px solid blue;
  outline-offset: 2px;
}

:focus-visible shows the outline only for keyboard focus (not mouse clicks), which is what most designers actually want.

9.5 ARIA: When HTML Isn't Enough

ARIA = Accessible Rich Internet Applications. A set of attributes that let you add accessibility metadata to elements that don't have it natively.

The first rule of ARIA: don't use ARIA. Use the right HTML element. Only reach for ARIA when there's no native equivalent.

Most useful ARIA attributes:

<!-- Describe a non-obvious purpose -->
<button aria-label="Close dialog">×</button>

<!-- Link content to an explanation -->
<input type="password" aria-describedby="pw-hint">
<p id="pw-hint">Must be at least 8 characters.</p>

<!-- Indicate state -->
<button aria-pressed="true">Bold</button>
<details aria-expanded="false">...</details>   <!-- native details handles this -->

<!-- Hide decorative content from screen readers -->
<span aria-hidden="true">🎉</span> Celebration!

<!-- Live regions: announce dynamic updates -->
<div aria-live="polite">Cart has 3 items</div>
<div aria-live="assertive">Error: form invalid</div>

<!-- Custom roles (when no semantic HTML fits) -->
<div role="alert">Saved successfully!</div>
<nav role="navigation">...</nav>   <!-- redundant; <nav> already has this role -->

9.6 Color & Contrast

Text must have sufficient contrast against its background. The WCAG (Web Content Accessibility Guidelines) standard:

  • AA (the legal minimum): 4.5:1 for normal text, 3:1 for large text (18pt+).
  • AAA (ideal): 7:1 for normal text, 4.5:1 for large text.

Use a contrast checker (webaim.org/resources/contrastchecker, or Chrome DevTools' built-in checker). Light gray text on white is a designer favorite and an accessibility crime.

Also: never use color alone to convey meaning. Red error text? Add an icon or word "Error". Red/green for required/optional? Add an asterisk.

9.7 Form Accessibility Checklist

  • Every input has a <label>.
  • Required fields are marked with required (and visually with an asterisk).
  • Errors are announced (use aria-live or role="alert").
  • Error messages are linked with aria-describedby.
  • Submit button is reachable by Tab.
  • Form validation feedback isn't color-only.
<form>
  <label for="email">Email address <span aria-hidden="true">*</span></label>
  <input type="email" id="email" name="email" required
         aria-required="true"
         aria-describedby="email-error email-hint">
  <p id="email-hint">We'll never share your email.</p>
  <p id="email-error" role="alert"></p>
  <button type="submit">Submit</button>
</form>

9.8 Skip Links

For long pages, provide a "skip to main content" link as the first focusable element:

<body>
  <a href="#main" class="skip-link">Skip to main content</a>
  <header>...</header>
  <main id="main">...</main>
</body>

.skip-link {
  position: absolute;
  top: -100px;
  left: 0;
}
.skip-link:focus {
  top: 0;
}

Visible only when focused. Keyboard users press Tab once, see the link, hit Enter, skip the giant nav.

9.9 Testing Accessibility

  • Keyboard test: unplug your mouse. Can you use the whole site?
  • Screen reader: macOS has VoiceOver (Cmd+F5). Windows has NVDA (free) or Narrator. Try navigating with eyes closed.
  • Axe DevTools: browser extension that scans for common issues.
  • Lighthouse: Chrome's built-in audit includes accessibility (Cmd+Opt+I → Lighthouse tab).

No tool catches everything. Manual testing matters. But automated tools catch the obvious failures fast.

SECTION 10

Embedded & Advanced

10.1 <iframe>

Embed another HTML page inside this one:

<iframe src="https://example.com" width="600" height="400" title="Example site"></iframe>

Used for: - Embedded videos (YouTube, Vimeo). - Embedded maps (Google Maps). - Embedded forms (Google Forms, Typeform). - Embedded payment widgets (Stripe).

Security: iframes can be dangerous because the embedded site has its own JS. Use sandbox to restrict:

<iframe src="..." sandbox="allow-scripts allow-same-origin"></iframe>

sandbox with no value disables everything. Add specific permissions as needed.

Also use loading="lazy" for off-screen iframes — same as images:

<iframe src="..." loading="lazy"></iframe>

10.2 <canvas>

A 2D drawing surface controlled by JavaScript:

<canvas id="myCanvas" width="500" height="300"></canvas>

<script>
  const ctx = document.getElementById('myCanvas').getContext('2d');
  ctx.fillStyle = 'blue';
  ctx.fillRect(50, 50, 100, 100);
</script>

Used for: - Games - Data visualizations (Chart.js, D3 can render to canvas) - Image filters and editing - Generative art

<canvas> is bitmap — once drawn, pixels are pixels. No DOM tree inside. No automatic accessibility. For accessibility, add a text fallback inside the tag:

<canvas>This chart shows sales growing 30% in Q3.</canvas>

10.4 The <template> Element

Inert HTML that's parsed but not rendered. Used by JavaScript to clone for dynamic content:

<template id="user-card">

    <h2 class="name"></h2>
    <p class="email"></p>
  </div>
</template>

<script>
  const tpl = document.getElementById('user-card');
  const clone = tpl.content.cloneNode(true);
  clone.querySelector('.name').textContent = 'Alice';
  clone.querySelector('.email').textContent = 'alice@example.com';
  document.body.appendChild(clone);
</script>

Cleaner than building strings of HTML in JavaScript.

10.5 Custom Elements (Web Components)

You can define your own HTML tags:

<my-greeting name="World"></my-greeting>

<script>
  class MyGreeting extends HTMLElement {
    connectedCallback() {
      this.innerHTML = `<p>Hello, ${this.getAttribute('name')}!</p>`;
    }
  }
  customElements.define('my-greeting', MyGreeting);
</script>

Custom elements must contain a hyphen in their name — that's how browsers distinguish them from native elements.

Combined with Shadow DOM (encapsulated styling) and HTML Templates, this is the foundation of Web Components — a native browser standard for reusable components. Used by Google (Lit), Salesforce (LWC), and any team that wants framework-agnostic components.

10.6 The <script> Tag — Modern Best Practices

<!-- Run after HTML parses (recommended for most scripts) -->
<script src="app.js" defer></script>

<!-- Run as soon as downloaded, possibly blocking parsing (use sparingly) -->
<script src="critical.js"></script>

<!-- Don't block parsing; run when ready (no guaranteed order) -->
<script src="analytics.js" async></script>

<!-- Modern ES modules -->
<script type="module" src="app.js"></script>

Where to put scripts: - In <head> with defer: modern recommendation — parses in parallel, runs after HTML is built. - At end of <body>: the old workaround — ensured DOM was ready. Now obsolete; defer is better. - In <head> without defer: only for scripts that must run before HTML continues (rare).

10.7 Web Storage

Not a tag, but a key HTML5 API:

// localStorage — persists forever (until cleared)
localStorage.setItem('username', 'alice');
const name = localStorage.getItem('username');
localStorage.removeItem('username');

// sessionStorage — cleared when tab closes
sessionStorage.setItem('cart', JSON.stringify(items));

Replaces cookies for client-side data storage (in most cases). Storage holds strings only — use JSON.stringify / JSON.parse for objects.

10.8 The Modern Web Platform

HTML is now the glue for a vast set of browser APIs:

  • Geolocation API — get user's location.
  • Notification API — show desktop notifications.
  • Service Workers — background scripts for offline support, push notifications.
  • WebSockets — real-time two-way communication.
  • WebRTC — peer-to-peer video/audio (Zoom, Google Meet).
  • Web Audio API — synthesize and process sound.
  • WebGL — 3D graphics in <canvas>.
  • WebAssembly — run compiled C/Rust/etc. in the browser.
  • Web Bluetooth, Web USB, Web NFC — direct hardware access.

These are JavaScript APIs, not HTML elements, but they define what "the modern Web" means. HTML provides the structure; these APIs provide the capabilities.

Capstone Project: Build a Multi-Page Website

11.1 What You're Building

A three-page website for a fictional local business — a community bakery called Rising Sun Bakery. Three pages, linked together with navigation:

  1. Home (index.html) — a welcome, a hero section, featured products, opening hours.
  2. Menu (menu.html) — the bakery's offerings, organized in tables and sections.
  3. Contact (contact.html) — a contact form and location details.

The whole thing is HTML only — no CSS, no JavaScript. That's intentional. This capstone proves you can structure a real, multi-page, semantic, accessible website using the raw material of the web. Styling is the next course's job. A well-structured unstyled page is the foundation everything else builds on — and it's genuinely what employers want to see you understand, because messy HTML can't be rescued by any amount of CSS.

Why this makes a good portfolio piece

  • It's a complete website, not a single page — multiple pages, working navigation between them, a coherent whole.
  • It demonstrates semantic HTML: proper use of header, nav, main, section, article, footer — not a soup of divs.
  • It's accessible: proper headings hierarchy, alt text, labeled form controls, landmark regions. This matters enormously to employers and is where most self-taught developers are weak.
  • It's deployable. Push to GitHub Pages and you have a live, multi-page site with a real URL.

11.2 Requirements in Detail

Shared across all pages: - A consistent header containing the business name and a nav with links to all three pages. - A main element wrapping each page's unique content. - A consistent footer with copyright and basic info. - Correct document structure: <!DOCTYPE html>, lang attribute, charset, viewport, a descriptive <title> unique to each page. - Navigation that works — clicking links moves between the actual pages.

Home page (index.html): - A hero section welcoming visitors. - A "featured products" section with at least three products, each with an image (with alt text) and a short description. Use article or section appropriately. - An opening-hours section — a table is a reasonable choice here.

Menu page (menu.html): - The offerings organized into categories (breads, pastries, drinks) using section and headings. - At least one table presenting items with prices. - Semantic grouping so a screen-reader user can navigate by heading.

Contact page (contact.html): - A contact form with: name, email, subject (a dropdown), message (a textarea), and a submit button. Every control properly labeled. - The business address and hours. - Fake but realistic details.

Accessibility requirements (all pages): - Exactly one <h1> per page, then a logical heading hierarchy (no skipping levels). - Every image has meaningful alt text (or alt="" if purely decorative). - Every form control has an associated <label>. - Navigation is in a <nav>; main content in <main>; each in the right landmark. - The page is understandable and navigable with the HTML alone — no reliance on visual styling to convey meaning.

11.3 Recommended Build Order

Milestone 1 — One complete page skeleton. Build index.html with the full document structure: doctype, head with meta and title, and a body containing empty header, nav, main, footer. Get the skeleton right before adding content.

Milestone 2 — Navigation. Add the three-link nav to index.html. Create menu.html and contact.html as copies of the skeleton. Wire the nav so all three link to each other. Now you can click between three (mostly empty) pages.

Milestone 3 — Home content. Fill in the home page: hero, featured products with images, opening hours table.

Milestone 4 — Menu content. Fill in the menu page: categorized sections, item tables with prices.

Milestone 5 — Contact content. Build the contact form with all controls properly labeled, plus the address details.

Milestone 6 — Accessibility pass. Go through every page: check heading hierarchy, alt text, form labels, landmarks. Fix anything missing. This is where a good project becomes a great one.

Building one page fully skeletal first, then duplicating, means you get the shared structure right once and reuse it.

11.4 Guidance for the Hard Parts

Consistent structure across pages (Milestones 1-2). Since there's no CSS or templating, you're literally copying the header/nav/footer into each file by hand. That's normal for hand-written HTML. Get one page's shared parts perfect, then copy them into the others and change only the main content and the <title>. (Real sites use templates or a framework to avoid this repetition — you'll learn those later. For now, hand-copying teaches you the structure.)

Marking the current page in nav. A nice accessibility touch: on each page, mark the nav link to the current page with aria-current="page". On the home page, its own link gets aria-current="page"; on the menu page, the menu link does; and so on.

The featured products (Milestone 3). Each product is a natural article (a self-contained piece of content) or a section. Give each an image with real alt text ("A golden sourdough loaf with a crackled crust"), a heading, and a description. Since there's no CSS, they'll stack vertically — that's fine.

The opening hours and menu tables. Use proper table structure: thead with th headers, tbody with the data. Add a <caption> describing the table. This is what makes tables accessible.

The contact form (Milestone 5). Every control needs a <label> with a for attribute matching the control's id. The subject dropdown is a <select> with <option>s. The message is a <textarea>. Group related fields in a <fieldset> with a <legend> if it aids clarity. Since there's no backend, the form won't actually submit anywhere — set action="#" or leave it; the point is correct structure.

The accessibility pass (Milestone 6). Read each page top to bottom imagining you can't see it — only the HTML structure. Can you tell what's a heading, what's navigation, what's the main content? Is every image described? Is every input labeled? If yes, you've built accessible HTML.

11.6 Walkthrough of the Key Decisions

Worth understanding and worth explaining in an interview.

Why article for products but section for page regions? An article is self-contained content that would make sense on its own (a product, a blog post, a comment). A section is a thematic grouping within a page. Each product is a self-contained article; the "Featured This Week" grouping is a section. Choosing correctly shows you understand semantics, not just tags.

Why aria-labelledby on sections? It ties each landmark section to its heading, so assistive technology announces "Featured This Week region" when a screen reader user navigates to it. It's the accessible way to name a region using a heading you're already displaying.

Why scope on table headers? scope="col" and scope="row" tell screen readers which cells a header describes. Without them, a table is just a grid of disconnected cells to a non-visual user. With them, the reader can say "Saturday: 7 AM – 4 PM" correctly.

Why aria-current="page"? It programmatically marks which nav link is the current page — an accessibility win that visual styling alone (bold, color) can't convey to a screen reader.

Why <address> for contact info? It's the semantic element specifically for contact information. tel: and mailto: links make the phone and email actionable on mobile devices.

Why does the form have no real action? There's no backend in this project — it's pure HTML. The form structure is correct and complete; wiring it to actually send requires a server (a later course). The point here is demonstrating correct, accessible form structure.
