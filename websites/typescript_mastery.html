TypeScript is JavaScript with a type system bolted on top — and it has quietly become the default for serious JavaScript work. It catches a whole class of bugs before your code ever runs, turns your editor into a tireless assistant, and makes large codebases sane to work in. This course takes you from "what even is a type?" to writing fully-typed, production-grade TypeScript, including how to type React components and how to migrate an existing JavaScript project. Prerequisite: solid JavaScript (the JavaScript Mastery course is plenty). It pairs especially well with the React course. Every code example here is checked by the TypeScript compiler.

How to use this course

  • Let the compiler teach you. TypeScript's error messages are the point — they're the feedback loop. Set up the playground in Section 1 and try the broken examples so you see the red squiggles yourself.
  • Each section follows the same rhythm: Concept → Code → Real example → Exercises with solutions. Attempt every exercise before opening the solution.
  • Types are about catching mistakes early. As you learn each feature, keep asking "what bug does this prevent?" That question is the soul of TypeScript, and it turns a pile of syntax into a set of tools you actually reach for.
  • Don't fight the compiler. When it complains, it's usually right — it found something you'd have hit at runtime. Learning to read its errors is half the skill.

By the end you'll be able to type functions, objects, classes, and generics; model complex data with unions and narrowing; wield advanced and utility types; configure the compiler; type React apps; and take an untyped JavaScript project and make it type-safe.

SECTION 1

Why TypeScript, and Getting Started

1.1 The problem TypeScript solves

JavaScript lets you write this and says nothing until it explodes at runtime:

function getName(user) {
  return user.name.toUpperCase();
}

getName({ nmae: "Ada" });   // typo — runtime crash: Cannot read properties of undefined
getName("hello");            // runs, returns "HELLO" — but was that intended?
getName();                   // runtime crash

Every bug there is invisible until the code executes — and maybe only in production, on a path your tests missed. JavaScript has no idea what shape user is supposed to be, so it can't warn you.

TypeScript adds a static type system to JavaScript. You annotate what shapes your data and functions expect, and a compiler checks those expectations before the code runs, flagging mismatches in your editor as you type. The same function in TypeScript:

function getName(user: { name: string }): string {
  return user.name.toUpperCase();
}

getName({ nmae: "Ada" });   // ❌ compile error: 'nmae' doesn't exist; did you mean 'name'?
getName("hello");            // ❌ compile error: string is not { name: string }
getName();                   // ❌ compile error: expected 1 argument

None of those bad calls compile. You find the mistakes in your editor, in seconds, instead of in production. That's the whole value proposition: move errors from runtime to compile time.

1.2 What TypeScript actually is

Three facts that clear up most confusion:

  • TypeScript is a superset of JavaScript. Every valid JavaScript program is already valid TypeScript. You can adopt it gradually, file by file, and any JS you know still works.
  • Types are erased. TypeScript compiles to plain JavaScript by stripping out the type annotations — the types exist only at development time to check your code, and vanish in the output. The browser and Node never see them; they run ordinary JavaScript.
  • It's a checker, not a runtime. TypeScript doesn't make your code faster or add features to how it runs. It adds a safety net over development — catching mistakes, powering autocomplete, and documenting intent.

That "types are erased" point is worth sitting with: the type system is a compile-time-only overlay. It cannot check anything at runtime (a value arriving from the network could still be the wrong shape), which is why later sections cover validating external data. But for the code you write, it's an enormous safety upgrade.

1.3 The other huge benefit — tooling

Types don't just catch errors; they make your editor genuinely intelligent. Because the editor knows the shape of everything, you get precise autocomplete (it knows user. can be .name or .email), inline documentation, safe rename-across-the-project, and instant feedback as you type. On a real codebase this is transformative — you navigate and refactor with confidence instead of grepping and hoping. Many developers say the tooling alone is worth it, before you even count the bugs prevented.

1.4 Setting up

The fastest way to experiment is the online TypeScript Playground (typescriptlang.org/play) — it compiles as you type and shows errors and the emitted JavaScript. Keep it open while you read.

For a real project, install the compiler with npm (you know npm from the JavaScript course):

npm install --save-dev typescript      # add TypeScript to a project
npx tsc --init                          # create a tsconfig.json (compiler settings)

tsc is the TypeScript compiler. It reads your .ts files, type-checks them, and emits .js. The tsconfig.json it generates configures how (Section 11 covers it in depth). To compile:

npx tsc                 # compile the whole project per tsconfig.json
npx tsc --noEmit        # type-check only, don't produce JS (great for CI)
npx tsc --watch         # re-check on every save

In a modern setup you often let a build tool (like Vite) handle the compiling and use tsc --noEmit purely as a type-checker. Either way, the mental model is: write .ts, the compiler checks it and produces .js.

1.5 Your first types — annotations

You add a type with a colon after a name. Start with variables:

let username: string = "Ada";
let age: number = 36;
let isActive: boolean = true;

age = "old";     // ❌ Type 'string' is not assignable to type 'number'.

The last line is the point: once age is a number, TypeScript won't let you put a string in it. The annotation is a promise the compiler enforces everywhere that variable is used.

1.6 Type inference — TypeScript is smart

Here's the part beginners miss: you usually don't need to write types at all. TypeScript infers them from your values:

let username = "Ada";      // inferred as string
let age = 36;              // inferred as number
let tags = ["a", "b"];     // inferred as string[]

age = "old";               // ❌ still an error — inferred number is enforced

username is a string even though you didn't say so, because you assigned it one. Good TypeScript leans on inference — you annotate where it adds information (function parameters, object shapes, public APIs) and let the compiler infer the obvious rest. Over-annotating everything is a beginner habit; the idiomatic style is to type the boundaries and let inference fill the middle.

Where inference can't help — most importantly, function parameters — you must annotate, because TypeScript can't guess what a caller will pass:

function double(n: number) {   // parameter MUST be annotated
  return n * 2;                // return type inferred as number
}

1.7 A first taste of the payoff

type User = { name: string; age: number };

function describe(user: User): string {
  return `${user.name} is ${user.age}`;
}

const ada = { name: "Ada", age: 36 };
describe(ada);                          // ✅ fine

describe({ name: "Bob" });              // ❌ missing 'age'
describe({ name: "Bob", age: "40" });   // ❌ age must be number
ada.naem;                               // ❌ typo caught — did you mean 'name'?

Every commented error is caught in your editor before you run anything, and while typing user. you'd get autocomplete offering name and age. Multiply this across thousands of lines and you see why TypeScript took over. The rest of this course is learning to describe your data precisely enough that the compiler can catch this much — and more.

🏋️ Exercise 1.1

In one or two sentences: what does it mean that "TypeScript types are erased," and what practical consequence does that have for data coming from an API?

Solution

"Types are erased" means TypeScript compiles to plain JavaScript by removing all type annotations — the types exist only at development/compile time to check your code and are gone at runtime. The practical consequence: TypeScript cannot check data at runtime, so a value arriving from an API might not actually match the type you declared for it. The compiler trusts your annotation; if the server sends a different shape, TypeScript won't catch it — you must validate external data yourself (covered later). Types protect the code you write, not the data you receive.

🏋️ Exercise 1.2

Given let count = 10; (no annotation), does count = "ten"; compile? Why or why not? What does this tell you about inference?

Solution

It does not compile — TypeScript reports that string is not assignable to number. Even without an explicit annotation, TypeScript inferred count as number from the initial value 10, and it enforces that inferred type just as strictly as a written one. This shows inference isn't a weaker form of typing: let count = 10 and let count: number = 10 give you the same protection, so you can often skip the annotation and let TypeScript figure it out.

🏋️ Exercise 1.3

Why must you annotate function parameters even though TypeScript can infer the types of variables and return values?

Solution

TypeScript infers a variable's type from the value you assign, and a function's return type from what it returns — it has the information right there. But it cannot see what a caller will pass into a parameter; that's determined elsewhere, potentially in many places. With no value to infer from, an un-annotated parameter would fall back to any (no checking). So you annotate parameters to tell the compiler what the function expects, which is exactly what lets it flag bad calls. (Return types can be annotated too, for documentation and to catch mistakes, but they're often left to inference.)

SECTION 2

Basic Types

TypeScript's power starts with a vocabulary of types to describe your values. This section covers the primitives and the handful of special types you'll use constantly, including the crucial any/unknown distinction that separates safe TypeScript from TypeScript-in-name-only.

2.1 The primitives

The three you'll use most mirror JavaScript's primitives:

let title: string = "Hello";
let count: number = 42;          // one number type — ints and floats alike
let done: boolean = false;

TypeScript has one number type for all numeric values (no separate int/float). There are also bigint and symbol for their JavaScript counterparts, rarely needed early. As Section 1 stressed, you'd usually let these be inferred (let title = "Hello"); the annotations here are to show the type names.

2.2 Arrays and tuples

An array type is the element type followed by []:

let names: string[] = ["Ada", "Bob"];
let scores: number[] = [90, 85, 100];
let flags: boolean[] = [true, false];

names.push("Cy");        // ✅
names.push(42);          // ❌ number not assignable to string

The array knows its element type, so pushing the wrong thing is an error, and .map(), .filter(), etc. all stay type-aware. (An alternate syntax Array<string> means the same thing; string[] is more common.)

A tuple is a fixed-length array where each position has its own type — perfect for "a pair" or "a row":

let point: [number, number] = [10, 20];
let entry: [string, number] = ["age", 36];

entry = [36, "age"];     // ❌ wrong order of types

Tuples power patterns like React's useState return ([value, setter]) — the reason const [count, setCount] = useState(0) knows count is a number and setCount is a function is that useState returns a typed tuple.

2.3 any — the escape hatch you should avoid

any turns off type checking for a value. It's assignable to and from anything, and accessing any property on it is allowed — which means it silently reintroduces every bug TypeScript exists to prevent:

let data: any = "hello";
data = 42;                 // fine
data.foo.bar.baz();        // no error — but crashes at runtime
data();                    // no error — but crashes at runtime

any is a hole in your type safety. It's occasionally necessary (interfacing with truly dynamic data, or during migration), but every any is a place bugs can hide. Treat it as a code smell, and reach for unknown instead whenever you can.

2.4 unknown — the safe alternative to any

unknown is the type-safe counterpart to any. It can hold any value (like any), but TypeScript won't let you do anything with it until you've narrowed it to a specific type (Section 5). This forces you to check before you use:

let value: unknown = fetchSomething();

value.toUpperCase();       // ❌ can't — TypeScript doesn't know it's a string

if (typeof value === "string") {
  value.toUpperCase();     // ✅ inside the check, it's known to be a string
}

This is exactly what you want for values whose type you genuinely don't know yet — API responses, JSON.parse results, caught errors. unknown says "could be anything, so prove what it is before using it," whereas any says "could be anything, do whatever you want, good luck." Prefer unknown; avoid any.

2.5 void and never

void is the return type of a function that doesn't return a value:

function log(message: string): void {
  console.log(message);      // returns nothing
}

never represents a value that can never occur — the return type of a function that never finishes normally (it always throws or loops forever), and, more usefully, the type of a branch that should be impossible:

function fail(message: string): never {
  throw new Error(message);   // never returns
}

never looks obscure now, but it's the secret to exhaustiveness checking in Section 5 — making the compiler force you to handle every case of a union. File it away.

2.6 null, undefined, and strict null checking

JavaScript's null and undefined are types in TypeScript. Their behavior depends on one hugely important compiler setting, strictNullChecks (on by default in strict mode — always keep it on):

let name: string = null;    // ❌ (with strict null checks) null isn't a string

With strict null checks on, a string cannot secretly be null or undefined — you must say so explicitly with a union (Section 5):

let name: string | null = null;    // ✅ explicitly allowed to be null

This eliminates the "cannot read property of undefined/null" class of bugs — the single most common JavaScript runtime error — by making nullability visible in the type and forcing you to handle it. It's one of TypeScript's biggest wins, and why strict mode is non-negotiable.

2.7 Literal types

A type can be a specific value, not just a general category:

let direction: "left" | "right";    // only these two exact strings
direction = "left";                  // ✅
direction = "up";                    // ❌ not "left" or "right"

let dice: 1 | 2 | 3 | 4 | 5 | 6;    // only these numbers

These literal types, combined with unions (Section 5), let you model "one of a fixed set of options" precisely — a status that's only "active" | "inactive" | "pending", an HTTP method that's only "GET" | "POST" | .... This is far safer than a plain string, because typos and invalid values become compile errors.

2.8 A real example — describing a small domain

type Status = "todo" | "in-progress" | "done";   // literal union

let taskTitle: string = "Write docs";
let priority: number = 2;
let tags: string[] = ["writing", "urgent"];
let status: Status = "todo";
let assignee: string | null = null;              // may be unassigned
let coordinates: [number, number] = [0, 0];      // a tuple

status = "finished";     // ❌ not a valid Status — typo caught
tags.push(99);           // ❌ tags holds strings
assignee = "Ada";        // ✅ now assigned

Notice how much the types document the domain: anyone reading this knows a status is one of three exact strings, tags are strings, and an assignee might be absent. The types are precise enough that the compiler catches invalid states — which is the entire game.

🏋️ Exercise 2.1

Declare a variable id that can hold either a number or a string, and a variable role that can only be "admin", "editor", or "viewer". Then show one assignment to each that fails.

Solution 
let id: number | string;
let role: "admin" | "editor" | "viewer";

id = 42;          // ✅
id = "u_42";      // ✅
id = true;        // ❌ boolean is neither number nor string

role = "admin";   // ✅
role = "owner";   // ❌ not one of the allowed literals

number | string is a union allowing either type; the literal union "admin" | "editor" | "viewer" restricts role to exactly those three strings, so "owner" is a compile error.

🏋️ Exercise 2.2

Explain the difference between any and unknown. Rewrite this to be type-safe: let x: any = getData(); x.toFixed(2);

Solution

any disables type checking — you can do anything to an any value with no errors, which lets runtime bugs slip through. unknown also holds any value, but TypeScript forbids using it until you narrow it to a known type, forcing a safety check first.

Type-safe rewrite:

let x: unknown = getData();
if (typeof x === "number") {
  x.toFixed(2);      // ✅ only called once we know x is a number
}

With unknown, calling x.toFixed(2) directly is an error; the typeof check narrows x to number inside the block, where the call is safe.

🏋️ Exercise 2.3

What bug does strictNullChecks prevent? Given let user: string | null, write code that safely uppercases it only when it's present.

Solution

strictNullChecks prevents the extremely common "cannot read property of null/undefined" runtime error, by refusing to let null/undefined hide inside other types — you must declare nullability explicitly and handle it.

let user: string | null = getUser();

if (user !== null) {
  console.log(user.toUpperCase());   // ✅ inside the check, user is string
}
// or concisely:
console.log(user?.toUpperCase());     // optional chaining: undefined if user is null

Because user might be null, TypeScript won't let you call .toUpperCase() on it directly; narrowing with a check (or optional chaining) is required, which is exactly the safety you want.

🏋️ Exercise 2.4

When is a tuple the right choice over a regular array? Give the type for a value representing an RGB color (three numbers), and explain why a tuple fits better than number[].

Solution

A tuple fits when you have a fixed number of elements with specific meanings/positions, rather than a variable-length list of same-kind items.

let color: [number, number, number] = [255, 128, 0];   // R, G, B

A tuple is better than number[] here because RGB is exactly three values in a known order. number[] would allow [255] or [1,2,3,4,5] — any length — and wouldn't capture that position 0 is red, 1 is green, 2 is blue. The tuple encodes the fixed shape, so wrong lengths and (with distinct types) wrong orders become compile errors.

SECTION 3

Functions

Functions are where types earn their keep — they're the contracts between the pieces of your program. Typing a function's inputs and output means every call site is checked, and every change ripples out as compile errors instead of runtime surprises.

3.1 Parameter and return types

Annotate each parameter, and optionally the return type:

function add(a: number, b: number): number {
  return a + b;
}

add(2, 3);          // ✅ 5
add(2, "3");        // ❌ "3" is not a number
add(2);             // ❌ expected 2 arguments

The return type (: number here) is often inferred and can be omitted — TypeScript sees you return a + b and knows it's a number. Many developers still annotate return types on public functions as documentation and to catch mistakes where the body accidentally returns the wrong thing. Parameters, as always, must be annotated.

3.2 Optional and default parameters

A ? marks a parameter optional (callers may omit it; inside, its type includes undefined):

function greet(name: string, greeting?: string): string {
  return `${greeting ?? "Hello"}, ${name}`;   // handle the missing case
}

greet("Ada");                // ✅ "Hello, Ada"
greet("Ada", "Welcome");     // ✅ "Welcome, Ada"

A default value makes a parameter optional and supplies a fallback (and TypeScript infers its type from the default):

function greet(name: string, greeting = "Hello"): string {
  return `${greeting}, ${name}`;
}

Optional parameters must come after required ones. Use a default when you have a sensible fallback; use ? when "absent" is a meaningful state you'll handle.

3.3 Rest parameters

Type a rest parameter as an array of the element type:

function sum(...nums: number[]): number {
  return nums.reduce((total, n) => total + n, 0);
}

sum(1, 2, 3, 4);       // ✅ 10
sum(1, "2");           // ❌ all must be numbers

...nums collects any number of arguments into a number[], fully type-checked.

3.4 Function types

A function is a value, so it has a type — written with an arrow. This matters when you pass functions as arguments (callbacks) or store them:

// A variable holding a function:
let operation: (a: number, b: number) => number;
operation = (a, b) => a + b;      // ✅ params inferred from the type!
operation = (a, b) => a > b;      // ❌ returns boolean, not number

// A function that takes a callback:
function applyTwice(value: number, fn: (n: number) => number): number {
  return fn(fn(value));
}
applyTwice(3, (n) => n * 2);      // ✅ 12

Notice a lovely payoff: when you assign (a, b) => a + b to a variable already typed as (a: number, b: number) => number, you don't re-annotate a and b — TypeScript infers them from the target type. This "contextual typing" is why callbacks in typed APIs (like array .map) get their parameter types for free.

3.5 void return, and a subtlety

A function that returns nothing has return type void:

function logError(msg: string): void {
  console.error(msg);
}

A useful subtlety: a void-typed callback slot accepts functions that return a value (the value is just ignored). This is why array.forEach(item => list.push(item)) is fine even though push returns a number — forEach expects a void callback and ignores the return. You don't need to worry about it often; just know void means "return value won't be used," not "must literally return nothing."

3.6 Function overloads

Occasionally a function behaves differently based on argument types, and you want each shape of call typed precisely. Overloads let you declare multiple signatures above one implementation:

function parseInput(input: string): string[];
function parseInput(input: number): number[];
function parseInput(input: string | number): string[] | number[] {
  if (typeof input === "string") return input.split("");
  return [input];
}

const a = parseInput("hi");     // a is string[]
const b = parseInput(42);       // b is number[]

The two signatures on top are what callers see; the third (with the union) is the actual implementation and is not directly callable. Overloads are relatively rare — a union or generic often reads better — but you'll meet them in library type definitions, so recognize the pattern: several signatures, one implementation.

3.7 A real example — typed helpers

type User = { name: string; age: number; email?: string };

function createUser(name: string, age: number, email?: string): User {
  return email ? { name, age, email } : { name, age };
}

function formatUsers(users: User[], separator = ", "): string {
  return users.map((u) => u.name).join(separator);
}

const users = [createUser("Ada", 36), createUser("Bob", 40, "bob@x.com")];
formatUsers(users);            // ✅ "Ada, Bob"
formatUsers(users, " | ");     // ✅ "Ada | Bob"
createUser("Cy");              // ❌ age is required

Every function here is a checked contract: createUser guarantees it returns a valid User, formatUsers guarantees it takes User[], and the optional email/separator are handled. The types make the helpers self-documenting and misuse-proof.

🏋️ Exercise 3.1

Write a function multiply that takes two numbers and returns their product, with a return type annotation. Then write greetAll that takes a string[] and an optional greeting (default "Hi"), returning a single string greeting everyone.

Solution 
function multiply(a: number, b: number): number {
  return a * b;
}

function greetAll(names: string[], greeting = "Hi"): string {
  return names.map((n) => `${greeting}, ${n}`).join("; ");
}

greetAll(["Ada", "Bob"]);            // "Hi, Ada; Hi, Bob"
greetAll(["Ada"], "Welcome");        // "Welcome, Ada"

multiply annotates both params and the return; greetAll uses a default parameter (greeting = "Hi"), which makes it optional and infers its type as string.

🏋️ Exercise 3.2

Declare a variable transform whose type is "a function taking a string and returning a number." Assign a valid function to it, then show an assignment that fails and explain why.

Solution 
let transform: (s: string) => number;

transform = (s) => s.length;      // ✅ s inferred as string; returns number
transform = (s) => s.toUpperCase(); // ❌ returns string, not number

The type (s: string) => number describes the function's shape. The first assignment matches (returns s.length, a number), and s is inferred from the target type. The second fails because toUpperCase() returns a string, violating the declared => number return type.

🏋️ Exercise 3.3

What's the difference between greeting?: string and greeting: string = "Hi" in a parameter list? When would you choose each?

Solution

Both make the parameter optional for callers, but:

  • greeting?: string — the parameter may be omitted; if it is, its value is undefined, and its type inside the function is string | undefined, so you must handle the missing case.
  • greeting: string = "Hi" (default) — if omitted, it's automatically "Hi"; inside the function its type is just string (never undefined).

Choose a default when you have a sensible fallback value and don't want to handle undefined. Choose ? when "absent" is a distinct, meaningful state you intend to handle differently (e.g., "no filter provided" vs "an empty filter").

SECTION 4

Objects, Interfaces & Type Aliases

Most real data is objects, so describing object shapes is the heart of everyday TypeScript. You'll learn object types, the two ways to name them (interface and type), and the modifiers that make shapes precise.

4.1 Object types

You can annotate an object's shape inline:

function printUser(user: { name: string; age: number }): void {
  console.log(`${user.name}, ${user.age}`);
}

printUser({ name: "Ada", age: 36 });        // ✅
printUser({ name: "Ada" });                  // ❌ missing 'age'
printUser({ name: "Ada", age: 36, x: 1 });   // ❌ 'x' isn't in the type

Two things to notice: all listed properties are required by default, and TypeScript rejects excess properties on object literals (the x: 1 error) — a nice guard against typos in keys. Inline object types get verbose fast, though, so you name them.

4.2 Type aliases

A type alias gives a name to any type — object shapes, unions, primitives, anything:

type User = {
  name: string;
  age: number;
};

type ID = string | number;         // alias for a union
type Point = [number, number];     // alias for a tuple

function printUser(user: User): void {
  console.log(user.name);
}

Now User is reusable and self-documenting. Type aliases are the general-purpose naming tool — they work for every kind of type, not just objects.

4.3 Interfaces

An interface is the other way to name an object shape. For plain objects, it's nearly interchangeable with a type alias:

interface User {
  name: string;
  age: number;
}

function printUser(user: User): void {
  console.log(user.name);
}

Same usage, same checking. So which do you use? The guidance most teams follow: use interface for object shapes (especially public APIs and things meant to be extended), and type for everything else (unions, tuples, function types, mapped/conditional types — which interfaces can't express). Two concrete differences:

  • Interfaces can be extended and merged (below); type aliases use intersections instead.
  • Type aliases can name any type (unions, primitives, tuples); interfaces only describe object/function shapes.

Honestly, for a simple object shape either is fine and consistency matters more than the choice. Pick one convention and hold to it.

4.4 Optional and readonly properties

A ? makes a property optional; readonly forbids reassignment after creation:

interface User {
  readonly id: number;      // can't be changed once set
  name: string;
  email?: string;           // may be absent
}

const u: User = { id: 1, name: "Ada" };   // email omitted — fine
u.name = "Ada L.";                          // ✅
u.id = 2;                                   // ❌ id is readonly
u.email?.toUpperCase();                     // handle possibly-undefined

readonly encodes immutability in the type (great for ids and config), and ? models genuinely optional data. Accessing an optional property gives you string | undefined, so the compiler makes you handle the missing case — usually with optional chaining (?.).

4.5 Index signatures

When you don't know the property names ahead of time — a dictionary/map-like object — an index signature types the keys and values:

interface StringMap {
  [key: string]: string;         // any string key → string value
}

const headers: StringMap = {};
headers["Content-Type"] = "application/json";   // ✅
headers["Accept"] = 42;                          // ❌ value must be string

This says "any string key maps to a string value." Use it for genuine dictionaries; for objects with known keys, list the properties explicitly (it's more precise). Often a better modern choice is the Record utility type (Section 9): Record<string, string> means the same thing more readably.

4.6 Extending and composing shapes

Interfaces extend other interfaces to build bigger shapes from smaller ones:

interface Animal {
  name: string;
}

interface Dog extends Animal {
  breed: string;
}

const d: Dog = { name: "Rex", breed: "Lab" };   // needs both

Type aliases achieve the same via intersection (&), which combines shapes:

type Animal = { name: string };
type Dog = Animal & { breed: string };

Both give you a Dog requiring name and breed. Extension/intersection is how you model "is-a" relationships and share common fields across related types without repetition.

4.7 A real example — modeling a domain with interfaces

interface Product {
  readonly id: string;
  name: string;
  price: number;
  tags?: string[];
}

interface DiscountedProduct extends Product {
  discountPercent: number;
}

function priceAfterDiscount(p: DiscountedProduct): number {
  return p.price * (1 - p.discountPercent / 100);
}

const item: DiscountedProduct = {
  id: "p1",
  name: "Keyboard",
  price: 100,
  discountPercent: 20,
};

priceAfterDiscount(item);        // ✅ 80
item.id = "p2";                  // ❌ readonly
priceAfterDiscount({ id: "p2", name: "Mouse", price: 50 });  // ❌ missing discountPercent

The types describe the domain precisely: a product has a readonly id, optional tags, and a discounted product is a product plus a discount. The compiler enforces every rule — required fields present, readonly respected, correct shape passed to functions.

SECTION 5

Unions, Intersections & Narrowing

This section is where TypeScript starts to feel powerful. Union types model "one of several possibilities," and narrowing is how you safely work with them — teaching the compiler which possibility you're in. Master this and you can model real, messy data precisely and handle it safely. Discriminated unions, the crown jewel here, are one of TypeScript's best ideas.

5.1 Union types

A union (|) means a value is one of several types:

let id: string | number;
id = "abc";      // ✅
id = 123;        // ✅
id = true;       // ❌ neither string nor number

function format(value: string | number): string {
  return value.toString();   // ✅ .toString() exists on BOTH
}

You can only use members/operations valid for every member of the union. Both string and number have .toString(), so that's allowed — but value.toUpperCase() would error, because number doesn't have it. To use string-only methods, you must first prove the value is a string. That proving is narrowing.

5.2 Narrowing with type guards

Narrowing is convincing the compiler that, within a block, a value is a more specific type than its declared union. The tools are ordinary JavaScript checks that TypeScript understands:

typeof — for primitives:

function format(value: string | number): string {
  if (typeof value === "string") {
    return value.toUpperCase();    // ✅ here, value is string
  }
  return value.toFixed(2);         // ✅ here, value must be number
}

Inside the if, TypeScript knows value is a string, so string methods are allowed; after the if (the only other possibility), it knows it's a number. This is control-flow analysis — the compiler tracks what each branch implies.

instanceof — for classes:

function handle(err: Error | string): string {
  if (err instanceof Error) return err.message;   // err is Error here
  return err;                                       // err is string here
}

in — checks whether a property exists, to distinguish object shapes:

type Dog = { bark: () => void };
type Cat = { meow: () => void };

function speak(animal: Dog | Cat): void {
  if ("bark" in animal) animal.bark();   // narrowed to Dog
  else animal.meow();                     // narrowed to Cat
}

Truthiness — narrows out null/undefined:

function greet(name: string | null): string {
  if (name) return `Hi, ${name}`;   // name is string here (null filtered out)
  return "Hi, stranger";
}

These four — typeof, instanceof, in, and truthiness — cover most narrowing. The key mental model: write a normal runtime check, and TypeScript narrows the type inside the branch to match.

5.3 Discriminated unions — the big idea

Here's the pattern that makes complex state genuinely safe. A discriminated union is a union of object types that all share a common literal property — the discriminant — which tells them apart. It's the idiomatic way to model "a value that is one of several distinct kinds, each with its own data."

type LoadingState = { status: "loading" };
type SuccessState = { status: "success"; data: string[] };
type ErrorState   = { status: "error"; message: string };

type RequestState = LoadingState | SuccessState | ErrorState;

Every member has a status property with a distinct literal type. Now switch on that discriminant, and TypeScript narrows each case to the exact shape — giving you access to data only in the success case, message only in the error case:

function render(state: RequestState): string {
  switch (state.status) {
    case "loading":
      return "Loading…";
    case "success":
      return state.data.join(", ");    // ✅ data exists here
    case "error":
      return state.message;             // ✅ message exists here
  }
}

Try to use state.data in the "loading" case and it won't compile — that field doesn't exist there. This models "impossible states are impossible": you can't accidentally read data while loading, because the type says there is none. Discriminated unions replace flimsy patterns like "a bunch of optional fields and hope the right ones are set" with airtight modeling. You'll use them constantly — for state, for events, for API results, for anything with distinct variants.

5.4 Exhaustiveness checking with never

Remember never from Section 2? Here's its payoff. Add a default case that assigns the value to a never, and the compiler will error if you ever add a new variant and forget to handle it:

function render(state: RequestState): string {
  switch (state.status) {
    case "loading": return "Loading…";
    case "success": return state.data.join(", ");
    case "error":   return state.message;
    default:
      const _exhaustive: never = state;   // ✅ compiles only if all cases handled
      return _exhaustive;
  }
}

If every case is handled, state in the default has been narrowed to never (nothing left), so assigning it to a never variable is fine. But the day someone adds { status: "cancelled" } to the union, state in default is now CancelledState, which isn't assignable to never — compile error, pointing right at the switch. The compiler forces you to handle the new case. This turns "did I update every switch?" from a manual hunt into an automatic guarantee. It's one of the most valuable patterns in TypeScript.

5.5 Intersection types

Where a union is "one or the other," an intersection (&) is "both at once" — it combines types into one that has everything from each:

type Timestamps = { createdAt: Date; updatedAt: Date };
type Identifiable = { id: string };

type Entity = Identifiable & Timestamps;   // has id, createdAt, AND updatedAt

const e: Entity = {
  id: "1",
  createdAt: new Date(),
  updatedAt: new Date(),
};

Intersections shine for mixing in shared fields: define Timestamps and Identifiable once, then intersect them into every entity type that needs them. It's the composition tool that pairs with unions — unions for "which kind is it?", intersections for "combine these capabilities."

5.6 A real example — a typed events system

type ClickEvent = { type: "click"; x: number; y: number };
type KeyEvent   = { type: "keypress"; key: string };
type ScrollEvent = { type: "scroll"; delta: number };

type AppEvent = ClickEvent | KeyEvent | ScrollEvent;

function handleEvent(event: AppEvent): string {
  switch (event.type) {
    case "click":    return `Click at (${event.x}, ${event.y})`;
    case "keypress": return `Key: ${event.key}`;
    case "scroll":   return `Scrolled ${event.delta}px`;
    default:
      const _exhaustive: never = event;
      return _exhaustive;
  }
}

handleEvent({ type: "click", x: 10, y: 20 });    // ✅
handleEvent({ type: "click", key: "a" });         // ❌ click has no 'key'

This is production-grade modeling: each event kind carries exactly its own data, the handler is narrowed per case, invalid combinations don't compile, and if you add a new event type, the exhaustiveness check makes you handle it. Discriminated unions plus exhaustiveness is a pattern you'll reach for again and again.

What's the difference between A | B and A & B? Give a one-line example where each is the right tool.

SECTION 6

Generics

Generics are TypeScript's tool for writing reusable, type-safe code that works over many types without resorting to any. They feel abstract at first, but the motivation is concrete: you want a function or container to work with any type while preserving that type's information. Once it clicks, you'll see generics everywhere — because the standard library and every good library are built on them.

6.1 The problem generics solve

Say you want a function that returns the first element of an array. Without generics you're stuck:

function first(arr: any[]): any {
  return arr[0];
}

const n = first([1, 2, 3]);      // n is 'any' — we LOST the fact it's a number
n.toUpperCase();                  // no error, but crashes at runtime

Using any "works" but throws away the type — n should be number, but it's any, so all safety is gone. You could write a separate firstNumber, firstString, etc., but that's absurd. Generics solve exactly this: capture the type as a parameter and reuse it.

6.2 Generic functions

A type parameter (conventionally T) is declared in angle brackets and used like a normal type inside the function:

function first<T>(arr: T[]): T | undefined {
  return arr[0];
}

const n = first([1, 2, 3]);        // T inferred as number → n: number | undefined
const s = first(["a", "b"]);       // T inferred as string → s: string | undefined
n?.toFixed(2);                      // ✅ n is number-aware

<T> says "this function works for some type T, which I'll figure out from the argument." When you call first([1, 2, 3]), TypeScript infers T = number, so the return is number | undefined — the type flows through, fully preserved. One function, every element type, zero any. You rarely pass T explicitly; inference handles it.

6.3 Generics with multiple parameters

You can have several type parameters, and they can relate to each other:

function pair<K, V>(key: K, value: V): [K, V] {
  return [key, value];
}

const p = pair("age", 36);         // p: [string, number]

function mapValues<T, U>(arr: T[], fn: (item: T) => U): U[] {
  return arr.map(fn);
}

const lengths = mapValues(["a", "bb"], (s) => s.length);   // number[]

In mapValues, T is the input element type and U is what the callback produces — TypeScript threads both through, so mapping strings to their lengths correctly yields number[]. This is exactly how array .map is typed. Generics let relationships between types be expressed and preserved.

6.4 Constraints — extends

Sometimes a generic shouldn't accept literally any type — it needs the type to have certain capabilities. Constraints (extends) limit what T can be:

function longest<T extends { length: number }>(a: T, b: T): T {
  return a.length >= b.length ? a : b;
}

longest("hello", "hi");           // ✅ strings have .length → returns "hello"
longest([1, 2, 3], [1]);          // ✅ arrays have .length
longest(10, 20);                  // ❌ numbers have no .length

T extends { length: number } means "T can be any type, as long as it has a length property." Inside the function you can safely use .length, and callers are restricted to types that have it. Constraints are how you write generic code that still does something with the values — you promise the compiler the type has what you need. A very common constraint is T extends keyof O (Section 8) for type-safe property access.

6.5 Generic interfaces and types

Types and interfaces can be generic too — this is how you build reusable containers:

interface Box<T> {
  value: T;
}

const numberBox: Box<number> = { value: 42 };
const stringBox: Box<string> = { value: "hi" };

type ApiResponse<T> = {
  data: T;
  status: number;
  ok: boolean;
};

const userResponse: ApiResponse<{ name: string }> = {
  data: { name: "Ada" },
  status: 200,
  ok: true,
};

Box<T> is a container parameterized by what it holds; ApiResponse<T> wraps any data payload with common response fields. This is the pattern behind Array<T>, Promise<T>, Map<K, V>, and countless library types — a reusable shape with a "hole" you fill with a specific type. You provide the type argument (Box<number>) or let it be inferred.

6.6 Default type parameters

Like function parameters, type parameters can have defaults:

interface Container<T = string> {
  items: T[];
}

const a: Container = { items: ["x"] };          // T defaults to string
const b: Container<number> = { items: [1, 2] };  // T explicitly number

Defaults let a generic be convenient in the common case while still customizable — you'll see them in library types constantly.

6.7 A real example — a type-safe result wrapper

Generics plus discriminated unions (Section 5) combine into a genuinely useful pattern — a Result type that represents "success with a value, or failure with an error," preserving the value's type:

type Result<T> =
  | { ok: true; value: T }
  | { ok: false; error: string };

function tryParseJSON<T>(text: string): Result<T> {
  try {
    return { ok: true, value: JSON.parse(text) as T };
  } catch {
    return { ok: false, error: "Invalid JSON" };
  }
}

const result = tryParseJSON<{ name: string }>('{"name":"Ada"}');
if (result.ok) {
  console.log(result.value.name);    // ✅ value is { name: string } here
} else {
  console.log(result.error);          // ✅ error available here
}

Result<T> is generic (works for any success type) and a discriminated union (narrowed by ok), so after if (result.ok) you get the typed value, and in the else you get the error — the type flows all the way through. This "typed success-or-failure" pattern is a common, elegant alternative to throwing exceptions, and it shows generics and unions working together.

SECTION 7

Classes in TypeScript

You met classes in JavaScript and React. TypeScript supercharges them with types, access control, and features that make object-oriented code safer and more expressive. Even if you write mostly functions, you'll meet typed classes in libraries, frameworks, and back-end code — so it pays to know them well.

7.1 Typed class fields and methods

A TypeScript class annotates its fields and methods just like everything else:

class Point {
  x: number;        // field declarations with types
  y: number;

  constructor(x: number, y: number) {
    this.x = x;
    this.y = y;
  }

  distanceTo(other: Point): number {
    return Math.hypot(this.x - other.x, this.y - other.y);
  }
}

const p = new Point(3, 4);
p.distanceTo(new Point(0, 0));    // ✅ 5
p.x = "nope";                      // ❌ x is number

Fields are declared with their types at the top; the constructor and methods are typed like ordinary functions. Compared to a JavaScript class, you get checked field types, checked method signatures, and autocomplete on instances.

7.2 Access modifiers

TypeScript adds access modifiers that control visibility — enforcing encapsulation at compile time:

class BankAccount {
  public owner: string;         // accessible everywhere (the default)
  private balance: number;       // only inside this class
  protected id: string;          // this class and subclasses

  constructor(owner: string, balance: number) {
    this.owner = owner;
    this.balance = balance;
    this.id = crypto.randomUUID();
  }

  deposit(amount: number): void {
    this.balance += amount;      // ✅ private is reachable inside the class
  }

  getBalance(): number {
    return this.balance;
  }
}

const acct = new BankAccount("Ada", 100);
acct.deposit(50);           // ✅
acct.balance;               // ❌ 'balance' is private
  • public (default) — accessible anywhere.
  • private — accessible only within the class. Callers can't read or write it, so you control access through methods.
  • protected — like private, but also accessible in subclasses.

This is real encapsulation: balance can only change through deposit, so you can enforce invariants (e.g., "never go negative"). (There's also JavaScript's runtime-enforced #private fields; TypeScript's private is compile-time only but more flexible. Both are common.)

7.3 Parameter properties — a shorthand

The pattern "declare a field, then assign it from a constructor parameter" is so common that TypeScript has a shorthand: put an access modifier on a constructor parameter, and it automatically becomes a field:

class Point {
  constructor(
    public x: number,
    public y: number,
  ) {}                       // x and y are declared AND assigned automatically
}

const p = new Point(3, 4);
p.x;    // ✅ 3

That tiny class is equivalent to the longer version in 7.1 — the public x: number parameter declares the field, and TypeScript assigns this.x = x for you. This parameter property shorthand removes a lot of boilerplate and is idiomatic in TypeScript classes. It works with private and protected too.

7.4 readonly fields

Combine readonly with a modifier for fields that can be set once (in the constructor) and never changed:

class User {
  constructor(
    public readonly id: string,
    public name: string,
  ) {}
}

const u = new User("u1", "Ada");
u.name = "Ada L.";    // ✅
u.id = "u2";          // ❌ id is readonly

readonly id is perfect for identifiers and other create-once values — the compiler guarantees they never change after construction.

7.5 Implementing interfaces

A class can implement an interface, promising to provide everything the interface declares — the compiler checks that it does:

interface Serializable {
  serialize(): string;
}

class Config implements Serializable {
  constructor(private settings: Record<string, string>) {}

  serialize(): string {
    return JSON.stringify(this.settings);
  }
}

implements Serializable means "this class guarantees a serialize() method." If you forget it or get its signature wrong, the class won't compile. Interfaces thus act as contracts that classes fulfill — you can write functions accepting any Serializable, and any conforming class works. This is how you get polymorphism without tight coupling.

7.6 Abstract classes

An abstract class is a base class that can't be instantiated directly and can declare abstract methods that subclasses must implement:

abstract class Shape {
  abstract area(): number;          // no body — subclasses must provide it

  describe(): string {              // a concrete method subclasses inherit
    return `This shape has area ${this.area()}`;
  }
}

class Circle extends Shape {
  constructor(private radius: number) {
    super();
  }
  area(): number {                  // required by the abstract base
    return Math.PI * this.radius ** 2;
  }
}

new Circle(2).describe();     // ✅ "This shape has area 12.56…"
new Shape();                  // ❌ can't instantiate an abstract class

Abstract classes sit between interfaces and concrete classes: they can provide shared implementation (describe) while requiring subclasses to fill in the specifics (area). Use them when related classes share behavior but each must supply certain pieces.

7.7 A real example — a small typed hierarchy

interface Payable {
  calculatePay(): number;
}

abstract class Employee implements Payable {
  constructor(
    public readonly id: string,
    public name: string,
    protected baseSalary: number,
  ) {}

  abstract calculatePay(): number;         // each type computes pay differently

  describe(): string {
    return `${this.name} (#${this.id})`;
  }
}

class SalariedEmployee extends Employee {
  calculatePay(): number {
    return this.baseSalary / 12;           // monthly
  }
}

class Contractor extends Employee {
  constructor(id: string, name: string, private hourlyRate: number, private hours: number) {
    super(id, name, 0);
  }
  calculatePay(): number {
    return this.hourlyRate * this.hours;
  }
}

const staff: Employee[] = [
  new SalariedEmployee("e1", "Ada", 120000),
  new Contractor("c1", "Bob", 80, 160),
];

staff.forEach((e) => console.log(e.describe(), e.calculatePay()));

Everything from this section is here: parameter properties (public readonly id), access modifiers (protected baseSalary, private hourlyRate), an interface contract (implements Payable), an abstract base with shared behavior (describe) and a required method (calculatePay), and subclasses supplying their own logic. The Employee[] array holds different concrete types uniformly, and calling calculatePay() dispatches to each — type-safe polymorphism.

SECTION 8

Advanced Types

This section covers the type-level tools that make TypeScript feel almost magical — operators that compute new types from existing ones. You won't reach for these daily as a beginner, but understanding them unlocks the utility types in Section 9, lets you read sophisticated library types, and lets you write types that stay correct as your data evolves. Take it slowly; each tool is small.

8.1 keyof — the keys of a type

keyof T produces a union of the property names of T:

interface User {
  name: string;
  age: number;
  email: string;
}

type UserKey = keyof User;      // "name" | "age" | "email"

let k: UserKey = "name";        // ✅
k = "phone";                    // ❌ not a key of User

keyof turns "the keys of this object type" into a usable union. On its own it's modest; combined with generics it enables type-safe property access — a function that can get any property of any object, with the return type correct per key:

function getProp<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user: User = { name: "Ada", age: 36, email: "a@x.com" };
const name = getProp(user, "name");    // string (T[K] where K="name")
const age = getProp(user, "age");      // number
getProp(user, "phone");                // ❌ "phone" isn't a key of User

K extends keyof T constrains the key to actual keys, and T[K] (next topic) is the type of that specific property. This is the canonical example of keyof + generics, and it's exactly how typed libraries offer safe get/pick helpers.

8.2 Indexed access types — T[K]

You can look up the type of a property with bracket notation at the type level:

interface User {
  name: string;
  age: number;
  address: { city: string; zip: string };
}

type Name = User["name"];            // string
type Age = User["age"];              // number
type City = User["address"]["city"]; // string — nested lookup
type NameOrAge = User["name" | "age"]; // string | number

User["name"] is the type of the name property. This keeps derived types in sync with the source: if you change User["age"] to a string, everything using that indexed type updates automatically. It's how you avoid duplicating type information — reference it instead of restating it.

8.3 typeof (the type operator)

You've used typeof at runtime (Section 5). At the type level, typeof captures the type of an existing value — handy for deriving a type from a constant instead of writing it twice:

const defaultConfig = {
  theme: "dark",
  fontSize: 14,
  showSidebar: true,
};

type Config = typeof defaultConfig;
// { theme: string; fontSize: number; showSidebar: boolean }

function applyConfig(config: Config): void { /* ... */ }

typeof defaultConfig derives the Config type from the actual object, so the type and the default value can never drift apart — change the object, and Config follows. This pairs beautifully with keyof: keyof typeof defaultConfig is "theme" | "fontSize" | "showSidebar".

8.4 Mapped types — transforming every property

A mapped type builds a new object type by iterating over the keys of another and transforming each. The syntax echoes a for...in loop, at the type level:

interface User {
  name: string;
  age: number;
}

// Make every property optional:
type PartialUser = {
  [K in keyof User]?: User[K];
};
// → { name?: string; age?: number }

// Make every property readonly:
type ReadonlyUser = {
  readonly [K in keyof User]: User[K];
};

// Turn every property into a boolean (e.g., a "which fields changed" flag set):
type Flags = {
  [K in keyof User]: boolean;
};
// → { name: boolean; age: boolean }

[K in keyof User] iterates each key K; User[K] is that key's type (indexed access), and you can add or remove modifiers (?, readonly). Mapped types are how you transform a whole shape at once — and they're exactly what the built-in utility types (Section 9) are made of. Partial<T>, Readonly<T>, and friends are just mapped types you don't have to write.

8.5 Conditional types

A conditional type chooses between two types based on a condition, using a ternary-like extends ? : at the type level:

type IsString<T> = T extends string ? "yes" : "no";

type A = IsString<string>;    // "yes"
type B = IsString<number>;    // "no"

More usefully, they let types react to their inputs. A common pattern extracts a piece of another type with infer:

type ElementType<T> = T extends (infer U)[] ? U : T;

type X = ElementType<number[]>;    // number  (unwrapped the array)
type Y = ElementType<string>;      // string  (not an array → unchanged)

infer U says "capture whatever the array's element type is, and call it U." Conditional types power the fancier utility types (ReturnType, Parameters, Awaited) — you rarely write them by hand early on, but recognizing extends ? : and infer lets you read advanced library types instead of being mystified by them.

8.6 Template literal types

Types can be built from string templates, mirroring template literals at the type level:

type Color = "red" | "green" | "blue";
type Shade = "light" | "dark";

type ColorVariant = `${Shade}-${Color}`;
// "light-red" | "light-green" | "light-blue" | "dark-red" | ... (all 6 combos)

let v: ColorVariant = "dark-blue";    // ✅
v = "medium-blue";                     // ❌ not a valid combination

Template literal types combine string unions into new precise string types — useful for typed CSS class names, event names (on${Capitalize<EventName>}), API routes, and anywhere strings follow a pattern. They're a niche but delightful tool for making "stringly-typed" APIs actually type-safe.

8.7 A real example — deriving types instead of duplicating them

// One source of truth — the actual default values:
const themeDefaults = {
  primaryColor: "#333",
  fontSize: 16,
  spacing: 8,
};

// Derive types from it — never restate the shape:
type Theme = typeof themeDefaults;              // the object's type
type ThemeKey = keyof typeof themeDefaults;      // "primaryColor" | "fontSize" | "spacing"

// A type-safe updater for any single theme property:
function updateTheme<K extends ThemeKey>(key: K, value: Theme[K]): void {
  // value is exactly the right type for that key
  console.log(`Setting ${key} to ${value}`);
}

updateTheme("fontSize", 18);        // ✅ value must be number
updateTheme("primaryColor", "#fff"); // ✅ value must be string
updateTheme("fontSize", "big");      // ❌ fontSize is a number
updateTheme("margin", 4);            // ❌ "margin" isn't a theme key

No type is written twice: Theme comes from the values via typeof, ThemeKey from keyof typeof, and updateTheme uses a constrained generic with indexed access (Theme[K]) so each key demands the right value type. Change themeDefaults and every derived type updates automatically. This "derive, don't duplicate" discipline is what advanced types are for.

🏋️ Exercise 8.4

What does the conditional type type Unwrap<T> = T extends Promise<infer U> ? U : T do? Give the result for Unwrap<Promise<string>> and Unwrap<number>.

The infer U captures the Promise's resolved type in the extends clause, and the ? : chooses between "the unwrapped type" and "the original type." This is essentially how the built-in Awaited<T> works.

SECTION 9

Utility Types

TypeScript ships a toolbox of utility types — generic types that transform other types — built from the mapped and conditional types you just met. They save you from re-describing shapes by hand and express common transformations concisely. You'll use these constantly, so this section is a practical tour of the ones that matter, grouped by what they do.

9.1 Reshaping objects: Partial, Required, Readonly

These transform every property of an object type at once:

interface User {
  id: string;
  name: string;
  email: string;
}

type PartialUser = Partial<User>;    // all properties optional
// { id?: string; name?: string; email?: string }

type RequiredUser = Required<User>;  // all properties required (removes ?)
type ReadonlyUser = Readonly<User>;  // all properties readonly

Partial<T> is the workhorse — it's exactly what you want for update functions, where a caller supplies only the fields they're changing:

function updateUser(id: string, changes: Partial<User>): void {
  // changes may contain any subset of User's fields
}

updateUser("u1", { name: "Ada L." });          // ✅ just the name
updateUser("u1", { email: "a@x.com", name: "A" }); // ✅ several
updateUser("u1", { role: "admin" });            // ❌ 'role' isn't a User field

Required<T> does the opposite (strips ?), and Readonly<T> makes an immutable view — all three are just the mapped types from Section 8 with friendly names.

9.2 Selecting properties: Pick and Omit

These build a smaller type from a larger one — invaluable for deriving DTOs, form models, and API shapes from a central type:

interface User {
  id: string;
  name: string;
  email: string;
  passwordHash: string;
}

// Keep only certain fields:
type PublicUser = Pick<User, "id" | "name">;
// { id: string; name: string }

// Remove certain fields:
type SafeUser = Omit<User, "passwordHash">;
// { id: string; name: string; email: string }

// A "create" input: everything except the server-generated id:
type CreateUserInput = Omit<User, "id">;

Pick<T, Keys> keeps the listed keys; Omit<T, Keys> drops them. These are how you avoid defining nearly-identical types repeatedly — derive CreateUserInput from User with Omit<User, "id">, and if User gains a field, the input type updates automatically. Derive, don't duplicate (the Section 8 mantra) in one line.

9.3 Building maps: Record

Record<Keys, Value> builds an object type with the given keys, each of the given value type — the clean way to type dictionaries and lookups:

type Role = "admin" | "editor" | "viewer";

// A permission flag for every role:
type Permissions = Record<Role, boolean>;
// { admin: boolean; editor: boolean; viewer: boolean }

const perms: Permissions = {
  admin: true,
  editor: true,
  viewer: false,
};                          // ❌ if you forget a role — all keys required

// An open-ended string dictionary:
type Cache = Record<string, number>;
const hits: Cache = {};
hits["home"] = 5;           // ✅

Record<Role, boolean> is both more readable than an index signature and stricter: because Role is a finite union, TypeScript requires an entry for every role — forget one and it won't compile. For open-ended keys, Record<string, V> behaves like an index signature. This is the go-to for "a value per key."

9.4 Filtering unions: Exclude, Extract, NonNullable

These operate on union types:

type Status = "active" | "inactive" | "banned" | "pending";

type Activeish = Exclude<Status, "banned" | "pending">;  // "active" | "inactive"
type Bad = Extract<Status, "banned" | "pending">;         // "banned" | "pending"

type MaybeString = string | null | undefined;
type DefinitelyString = NonNullable<MaybeString>;          // string

Exclude<T, U> removes members of a union; Extract<T, U> keeps only the matching ones; NonNullable<T> strips null and undefined. They're handy for deriving sub-sets of a union without restating it — e.g., "all statuses except the terminal ones."

9.5 Function-related: ReturnType, Parameters, Awaited

These extract types from functions and promises, using the conditional/infer machinery of Section 8 under the hood:

function createUser(name: string, age: number) {
  return { id: "x", name, age, createdAt: new Date() };
}

type User = ReturnType<typeof createUser>;
// { id: string; name: string; age: number; createdAt: Date }

type Args = Parameters<typeof createUser>;    // [name: string, age: number]

type Data = Awaited<Promise<string>>;          // string (unwraps the Promise)
type FetchResult = Awaited<ReturnType<typeof fetchUser>>;  // resolved type of an async fn

ReturnType<typeof fn> is especially useful: instead of separately declaring a type and a function that returns it, you write the function and derive the type from it — one source of truth. Parameters grabs the argument tuple; Awaited unwraps a Promise (or nested promises). These show up constantly when typing around existing functions and async code.

9.6 A real example — deriving a whole API layer from one type

// The single source of truth for a user entity:
interface User {
  id: string;
  name: string;
  email: string;
  passwordHash: string;
  createdAt: Date;
}

// Everything else is DERIVED — no shape is written twice:

// What the API returns publicly (no secrets):
type PublicUser = Omit<User, "passwordHash">;

// What you send to create one (server fills id + createdAt):
type CreateUserInput = Omit<User, "id" | "createdAt" | "passwordHash"> & {
  password: string;
};

// What you send to update one (any subset of the editable fields):
type UpdateUserInput = Partial<Pick<User, "name" | "email">>;

// A lookup of users by id:
type UserMap = Record<string, PublicUser>;

function createUser(input: CreateUserInput): PublicUser { /* ... */ return {} as PublicUser; }
function updateUser(id: string, changes: UpdateUserInput): void { /* ... */ }

createUser({ name: "Ada", email: "a@x.com", password: "secret" });  // ✅
updateUser("u1", { name: "Ada L." });                                // ✅
updateUser("u1", { passwordHash: "x" });                             // ❌ not updatable

Every type here comes from User via utility types — PublicUser omits the secret, CreateUserInput shapes the create payload, UpdateUserInput allows partial edits of just the editable fields, UserMap indexes them. Add a field to User and the derived types adjust automatically, with the compiler flagging anywhere that now needs attention. This is utility types at their best: a whole, consistent API surface derived from one definition.

🏋️ Exercise 9.4

All three derive a narrower union from an existing one without restating its members.

Modules & Declaration Files

Real projects span many files and use many libraries. This section covers how TypeScript handles modules (importing/exporting across files with types), how it understands JavaScript libraries through declaration files, and the @types ecosystem that types the whole npm world. This is the glue that makes TypeScript work at project scale.

10.1 Modules — exporting and importing types

TypeScript uses standard ES modules (from the JavaScript course), and types travel through them just like values. You export and import types the same way you export and import functions:

// user.ts
export interface User {
  id: string;
  name: string;
}

export type Role = "admin" | "user";

export function createUser(name: string): User {
  return { id: crypto.randomUUID(), name };
}

// app.ts
import { createUser } from "./user.js";
import type { User, Role } from "./user.js";

const u: User = createUser("Ada");
let r: Role = "admin";

Two things to note. First, import type { ... } is a type-only import — it imports things used only as types. It's good practice because it makes intent clear and lets the compiler erase the import entirely from the output (types don't exist at runtime). Second, module specifiers: in modern Node ESM you include the .js extension even when importing a .ts file (because that's what the compiled output will be) — your tsconfig settings (Section 11) determine the exact rules, but import type and clear module boundaries are universal good practice.

10.2 export type and re-exporting

You can mark exports as type-only at the source too, and re-export types to build a clean public surface:

// types.ts
export type { User, Role } from "./user.js";   // re-export types from one place

// A "barrel" file that gathers a module's public API:
export { createUser } from "./user.js";
export type { User } from "./user.js";

Gathering exports into an index/barrel file lets consumers write import { User, createUser } from "./domain" instead of reaching into individual files. Marking type-only exports keeps runtime and type concerns cleanly separated.

10.3 Declaration files (.d.ts)

A declaration file (.d.ts) contains only type declarations — no implementation. It describes the shape of code without providing it. You meet them in two ways.

First, the compiler can generate them for your own library, so consumers get types without seeing your source:

// with "declaration": true in tsconfig, math.ts →
// math.d.ts:  export declare function add(a: number, b: number): number;

Second, and more importantly, .d.ts files are how plain JavaScript libraries get types. A library written in JS has no types of its own, but a declaration file can describe its API so TypeScript understands it. The keyword declare says "this thing exists somewhere (implemented in JS); here's its type":

// my-lib.d.ts — describing a JS library's shape
declare function greet(name: string): string;

declare module "some-untyped-lib" {
  export function doThing(x: number): void;
}

You'll rarely write these early on, but you'll benefit from them constantly — because of the @types ecosystem.

10.4 @types and DefinitelyTyped

Here's the practical magic. DefinitelyTyped is a huge community repository of declaration files for thousands of JavaScript libraries, published on npm under the @types/ scope. When a library doesn't ship its own types, you install its types separately:

npm install lodash              # the JS library (no built-in types)
npm install --save-dev @types/lodash   # its community type declarations

Now import _ from "lodash" is fully typed — autocomplete, checking, the works — even though lodash itself is plain JavaScript. Many modern libraries bundle their own types (you'll see a types field in their package.json, or the library is written in TypeScript), so no @types package is needed. The rule of thumb: import a library, and if TypeScript complains it "could not find a declaration file," try npm install -D @types/that-library. If none exists, you can write a minimal .d.ts yourself (next).

10.5 Ambient declarations and quick escape hatches

Sometimes you use something with no types at all — an untyped library, a global injected by a script, a non-code import. Ambient declarations let you tell TypeScript "trust me, this exists":

// globals.d.ts

// A global variable injected at runtime (e.g., by a <script> or build):
declare const APP_VERSION: string;

// A global function on window:
interface Window {
  gtag?: (...args: unknown[]) => void;
}

// Let TypeScript accept importing a file type it doesn't know:
declare module "*.svg" {
  const src: string;
  export default src;
}

declare const APP_VERSION: string tells TypeScript a global named APP_VERSION exists and is a string, so you can use it without an import. The declare module "*.svg" pattern is how you make imports like import logo from "./logo.svg" type-check in a bundler setup. These are escape hatches — use them sparingly, but they're invaluable for bridging the typed and untyped worlds. For a truly untyped module you just need to use, a one-line declare module "untyped-lib"; (which types it as any) unblocks you, at the cost of safety for that import.

10.6 A real example — a small typed module boundary

// api/types.ts — the shared type vocabulary
export interface Repo {
  id: number;
  name: string;
  stars: number;
}

export type SortOrder = "asc" | "desc";

// api/client.ts — implementation importing the types
import type { Repo, SortOrder } from "./types.js";

export async function fetchRepos(user: string, order: SortOrder): Promise<Repo[]> {
  const res = await fetch(`https://api.github.com/users/${user}/repos`);
  const data = (await res.json()) as Repo[];
  return data.sort((a, b) => (order === "asc" ? a.stars - b.stars : b.stars - a.stars));
}

// app.ts — the consumer, importing from a clean boundary
import { fetchRepos } from "./api/client.js";
import type { Repo } from "./api/types.js";

const repos: Repo[] = await fetchRepos("octocat", "desc");

Types live in one module (types.ts), the implementation imports them with import type, and the consumer depends on a small, clear public surface. This is how typed projects stay organized at scale — explicit module boundaries with types flowing across them. (Notice the as Repo[] on the fetch result — a reminder from Section 1 that external data isn't actually checked at runtime; Section 13 covers validating it properly.)

What is import type and why prefer it for imports used only as types? What happens to such an import in the compiled JavaScript?

Solution

import type { X } from "..." is a type-only import — it brings in X for use purely as a type (annotations, generics), not as a runtime value. Prefer it because it makes intent explicit and guarantees the import is fully erased from the compiled JavaScript: since types don't exist at runtime, a type-only import produces no import statement in the output, avoiding unnecessary runtime dependencies and potential circular-import issues. A regular import of something used only as a type usually gets erased too, but import type states the intent clearly and removes ambiguity for the compiler and bundlers.

SECTION 11

The Compiler & `tsconfig.json`

Everything so far has been the language. This section is about the tool — the compiler and its configuration file, tsconfig.json. Understanding it is what lets you set up a project correctly, tune how strict the checking is, and read the settings in any codebase you join. You don't need to memorize every option, but you should know the ones that matter and, above all, why strict mode is the setting that makes TypeScript worth using.

11.1 What tsc does, recapped

The compiler tsc has two jobs: type-check your code (report errors) and emit JavaScript (strip types, optionally down-level modern syntax for older runtimes). npx tsc --init scaffolds a tsconfig.json that configures both. In many modern setups a bundler (Vite, esbuild) does the actual JS emit for speed, and you run tsc --noEmit purely as the type-checker (in your editor and in CI). Either way, tsconfig.json is the control panel.

11.2 The shape of tsconfig.json

A typical modern config:

{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "lib": ["ES2022", "DOM"],
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "outDir": "./dist",
    "rootDir": "./src",
    "noEmit": false
  },
  "include": ["src"],
  "exclude": ["node_modules", "dist"]
}

compilerOptions holds the settings; include/exclude say which files to compile. Let's walk the options that matter.

11.3 strict — the one setting that matters most

"strict": true

strict: true is non-negotiable. It's an umbrella that turns on a family of stricter checks, and without it you're using a much weaker, less useful TypeScript. The most important members of the family:

  • strictNullChecks — null/undefined aren't silently allowed in every type (Section 2). This alone eliminates the "cannot read property of undefined" bug class. The single most valuable flag.
  • noImplicitAny — a value TypeScript can't infer becomes an error instead of silently any. It forces you to type things rather than accidentally opting out of checking (Section 1's "parameters must be annotated" is this flag at work).
  • strictFunctionTypes, strictBindCallApply, strictPropertyInitialization, and others tighten specific corners.

Always start a project with strict: true. Turning strictness off to silence errors is almost always the wrong move — the errors are real. If you're migrating a large JS codebase (Section 14), you might enable strict flags gradually, but the destination is full strict mode.

11.4 target, module, and lib

These three describe the environment your code runs in:

  • target — which JavaScript version to emit. ES2022 is a safe modern default; a lower target (e.g., ES2017) makes the compiler down-level newer syntax for older runtimes. Modern Node and browsers handle recent targets fine.
  • module — the module system for the output: NodeNext/ESNext for ES modules (the modern default), CommonJS for older Node. Pair module and moduleResolution consistently.
  • lib — which built-in type definitions are available. ["ES2022", "DOM"] gives you modern JavaScript APIs plus browser globals (document, window, fetch). For a Node-only project you'd drop "DOM". If you use fetch and get "cannot find name 'fetch'," your lib (or @types/node) is the place to look.

These usually come preset by your project template (Vite, create-next-app, etc.), and you rarely change them by hand — but knowing what they control means you can fix environment mismatches when they arise.

11.5 Other useful options

A handful you'll actually touch:

  • esModuleInterop: true — smooths importing CommonJS modules with import syntax. Almost always on.
  • skipLibCheck: true — skips type-checking inside .d.ts files of dependencies. Speeds up builds and avoids errors in third-party types you can't fix. Commonly on.
  • outDir / rootDir — where emitted JS goes, and the source root.
  • noUnusedLocals / noUnusedParameters — flag dead variables. Nice for cleanliness (some find them noisy).
  • paths / baseUrl — set up import aliases like @/components instead of ../../../components:
"baseUrl": ".",
"paths": { "@/*": ["src/*"] }

(Path aliases also need your bundler/runtime to understand them; the tsconfig entry only handles the type side.)

11.6 Running and checking

The commands you'll use:

npx tsc                 # compile the project per tsconfig
npx tsc --noEmit        # type-check only — the standard CI/pre-commit check
npx tsc --watch         # re-check continuously as you edit

The most important habit: run tsc --noEmit in CI (and ideally a pre-commit hook), so no type error ever reaches your main branch. Combined with strict: true, this is what makes TypeScript a genuine safety net rather than optional decoration — the build fails if the types don't check.

🏋️ Exercise 11.1

Why is strict: true considered essential? Name two specific checks it enables and the bug each prevents.

Solution

strict: true turns on the family of strict checks that make TypeScript genuinely safe; without it you get a much weaker checker that lets many bugs through. Two key members:

  1. strictNullChecks — prevents null/undefined from silently inhabiting other types, eliminating the extremely common "cannot read property of null/undefined" runtime crash by forcing you to handle nullability explicitly.
  2. noImplicitAny — turns values TypeScript can't infer a type for (like un-annotated function parameters) into errors instead of silent any, preventing whole regions of your code from accidentally losing all type-checking.

Starting with strict: true (and keeping it) is the difference between real safety and TypeScript-in-name-only.

🏋️ Exercise 11.2

What do target and lib control, and how would they differ between a browser front-end and a Node-only back-end?

Solution
  • target sets which JavaScript version the compiler emits (down-leveling newer syntax if the target is older). Both a browser app and a Node app can use a modern target like ES2022 on current runtimes; you'd lower it only to support older environments.
  • lib sets which built-in type definitions are available. A browser front-end includes "DOM" (so document, window, etc. are typed): ["ES2022", "DOM"]. A Node-only back-end omits "DOM" (there's no document) and typically relies on @types/node for Node globals: ["ES2022"].

So the main difference is lib: include DOM for the browser, drop it for Node.

🏋️ Exercise 11.3

What does npx tsc --noEmit do, and why is running it in CI valuable even when a bundler already compiles your code?

Solution

npx tsc --noEmit runs the TypeScript compiler purely as a type-checker — it reports all type errors but produces no JavaScript output. It's valuable in CI because bundlers like Vite or esbuild typically strip types without fully type-checking them (for speed) — they'll happily build code that has type errors. Running tsc --noEmit as a separate CI step (and locally/pre-commit) ensures the whole project actually type-checks, so a type error fails the build and never reaches the main branch. Paired with strict: true, this is what turns TypeScript into an enforced safety net rather than optional annotations.

SECTION 12

TypeScript with React

TypeScript and React are a superb pair — types catch the exact mistakes that plague React code (wrong props, misused state, bad event handlers), and the editor tooling makes building components a pleasure. This section shows how to type the React you learned in the React course: props, state, hooks, events, and generic components. (Set up a typed React project with npm create vite@latest my-app -- --template react-ts — the -ts template wires everything up.)

12.1 Typing props

The most common thing you'll type in React is a component's props. Define a type (or interface) for the props and annotate the parameter:

interface GreetingProps {
  name: string;
  age?: number;              // optional prop
}

function Greeting({ name, age }: GreetingProps) {
  return <h1>Hello, {name}{age !== undefined && `, age ${age}`}</h1>;
}

// usage — checked at every call site:
<Greeting name="Ada" />;            // ✅
<Greeting name="Ada" age={36} />;   // ✅
<Greeting age={36} />;              // ❌ missing required 'name'
<Greeting name={42} />;             // ❌ name must be a string

This is a huge upgrade over untyped React: pass the wrong prop, forget a required one, or misspell one, and it's a compile error in your editor — not a silent bug or a runtime warning. The props type also drives autocomplete when you use the component.

12.2 Typing children and events

Two React-specific types come up constantly. For components that wrap content, type children with React.ReactNode (the type of "anything renderable"):

interface CardProps {
  title: string;
  children: React.ReactNode;    // any renderable content
}

function Card({ title, children }: CardProps) {
  return (
    <div className="card">
      <h2>{title}</h2>
      {children}
    </div>
  );
}

For event handlers, React provides typed event types so e is fully typed:

function SearchBox() {
  function handleChange(e: React.ChangeEvent<HTMLInputElement>) {
    console.log(e.target.value);    // value is known to be a string
  }
  function handleSubmit(e: React.FormEvent) {
    e.preventDefault();
  }
  return (
    <form onSubmit={handleSubmit}>
      <input onChange={handleChange} />
    </form>
  );
}

React.ChangeEvent<HTMLInputElement> types a change event on an input, so e.target.value is correctly a string. Often you don't even write these — if you inline the handler (onChange={(e) => ...}), TypeScript infers e's type from the JSX, so you get it for free. Annotate explicitly only when the handler is a separate named function.

12.3 Typing hooks

The hooks from the React course are all generic, and usually infer their types:

import { useState } from "react";

const [count, setCount] = useState(0);        // inferred: number
const [name, setName] = useState("");          // inferred: string

useState(0) infers count: number and setCount as a function taking a number — the tuple typing from Section 2 in action. You provide an explicit type argument when the initial value doesn't tell the whole story — most often when a state starts null but will hold an object, or holds a union:

interface User { id: string; name: string; }

const [user, setUser] = useState<User | null>(null);   // explicit: could be a User later
const [status, setStatus] = useState<"idle" | "loading" | "done">("idle");

Here useState<User | null>(null) tells TypeScript the state will eventually hold a User even though it starts null — without the type argument it would infer just null. useRef, useReducer, and useContext follow the same idea: infer when possible, annotate the type parameter when the initial value is insufficient. A typed useReducer is especially nice — the reducer's discriminated-union action type (Section 5) gives you exhaustive, checked dispatch calls.

12.4 Typing a custom hook

Custom hooks are just functions, so you type them like any function — annotate parameters and let the return be inferred (or annotate it for clarity):

import { useState, useEffect } from "react";

interface FetchState<T> {
  data: T | null;
  loading: boolean;
  error: string | null;
}

function useFetch<T>(url: string): FetchState<T> {
  const [state, setState] = useState<FetchState<T>>({
    data: null,
    loading: true,
    error: null,
  });

  useEffect(() => {
    let active = true;
    fetch(url)
      .then((res) => res.json())
      .then((data: T) => active && setState({ data, loading: false, error: null }))
      .catch((e: Error) => active && setState({ data: null, loading: false, error: e.message }));
    return () => { active = false; };
  }, [url]);

  return state;
}

// usage — the generic flows through:
interface Repo { id: number; name: string; }
const { data, loading } = useFetch<Repo[]>("/api/repos");   // data: Repo[] | null

The generic useFetch<T> from Section 6 shines here: the caller specifies what shape they expect (useFetch<Repo[]>), and data is typed as Repo[] | null all the way through. This is the payoff of combining generics with React — reusable, fully-typed data-fetching logic.

12.5 Generic components

Components can be generic too, for reusable UI that works over any data type — a typed list, table, or select:

interface ListProps<T> {
  items: T[];
  renderItem: (item: T) => React.ReactNode;
}

function List<T>({ items, renderItem }: ListProps<T>) {
  return <ul>{items.map((item, i) => <li key={i}>{renderItem(item)}</li>)}</ul>;
}

// usage — T inferred from items, so `user` in renderItem is typed:
interface User { id: string; name: string; }
const users: User[] = [{ id: "1", name: "Ada" }];

<List items={users} renderItem={(user) => user.name} />;   // user: User ✅

List<T> works for any element type, and because T is inferred from items, the renderItem callback's parameter is correctly typed (user: User) — pass a number[] and renderItem would receive a number. Generic components are how libraries build reusable, type-safe primitives.

12.6 A real example — a fully-typed component

import { useState } from "react";

interface Todo {
  id: number;
  text: string;
  done: boolean;
}

interface TodoItemProps {
  todo: Todo;
  onToggle: (id: number) => void;      // typed callback prop
}

function TodoItem({ todo, onToggle }: TodoItemProps) {
  return (
    <li onClick={() => onToggle(todo.id)} style={{ textDecoration: todo.done ? "line-through" : "none" }}>
      {todo.text}
    </li>
  );
}

function TodoApp() {
  const [todos, setTodos] = useState<Todo[]>([]);
  const [text, setText] = useState("");

  function addTodo() {
    if (!text.trim()) return;
    setTodos((prev) => [...prev, { id: Date.now(), text, done: false }]);
    setText("");
  }

  function toggle(id: number) {
    setTodos((prev) => prev.map((t) => (t.id === id ? { ...t, done: !t.done } : t)));
  }

  return (
    <div>
      <input value={text} onChange={(e) => setText(e.target.value)} />
      <button onClick={addTodo}>Add</button>
      <ul>{todos.map((t) => <TodoItem key={t.id} todo={t} onToggle={toggle} />)}</ul>
    </div>
  );
}

Everything is typed and checked: the Todo shape, the TodoItemProps (including the onToggle callback signature), useState<Todo[]> for the list, the inferred event on the input, and the typed toggle/addTodo functions. Pass a wrong prop to TodoItem, forget an id, or misuse the callback, and it's a compile error. This is what "React with TypeScript" feels like — the same components you already know, with a safety net that catches mistakes as you type.

🏋️ Exercise 12.2

Why do you usually not need to annotate useState(0), but you do need useState<User | null>(null)?

Solution

useState(0) gives TypeScript enough information: from the initial value 0 it infers the state type as number, so count is number and setCount accepts numbers — no annotation needed. useState(null), however, would infer the type as just null, which is wrong if the state will later hold a User. So you supply an explicit type argument, useState<User | null>(null), telling TypeScript the state can be a User or null even though it starts null. Rule: annotate when the initial value doesn't represent the full range of types the state will hold (starts null/undefined, or is a union).

🏋️ Exercise 12.3

When would you make a component generic? Sketch the props type for a reusable Select<T> that takes options: T[], a way to get each option's label, and an onSelect callback.

Solution

Make a component generic when it should work over any data type while preserving that type — reusable list/table/select primitives where the item type varies by usage.

interface SelectProps<T> {
  options: T[];
  getLabel: (option: T) => string;
  onSelect: (option: T) => void;
}

function Select<T>({ options, getLabel, onSelect }: SelectProps<T>) {
  return (
    <select onChange={(e) => onSelect(options[e.target.selectedIndex])}>
      {options.map((opt, i) => <option key={i}>{getLabel(opt)}</option>)}
    </select>
  );
}

Because T is inferred from options, both getLabel and onSelect receive the correct element type — pass User[] and they get a User. One component, type-safe for every element type.

SECTION 13

Typing Real-World Code

The examples so far have been clean. Real code is messier — data arrives from networks in unknown shapes, errors are of unknown types, and third-party values slip past the compiler. This section is about the practical discipline that separates robust TypeScript from code that merely looks typed: handling the boundary between your safe, typed world and the untyped chaos outside it.

13.1 The core problem — external data is a lie

Recall from Section 1 that types are erased and can't check anything at runtime. That has a sharp consequence for data you fetch:

interface User {
  id: string;
  name: string;
}

async function getUser(id: string): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  return res.json();       // typed as User — but this is a PROMISE, not a guarantee
}

res.json() returns Promise<any>, and you've asserted it's a User. But if the server returns { id: 1, fullName: "Ada" } (number id, wrong key), TypeScript is none the wiser — it trusts you. Your typed code is now built on a false premise, and the bug surfaces somewhere far away. The type annotation on external data is a hope, not a check. Recognizing this is the beginning of writing real-world TypeScript.

13.2 unknown at the boundary, then narrow

The disciplined first step: treat incoming data as unknown (Section 2) rather than blindly asserting a type, which forces you to check before use:

async function getUserSafe(id: string): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  const data: unknown = await res.json();       // honest: we don't know yet

  if (
    typeof data === "object" && data !== null &&
    "id" in data && "name" in data &&
    typeof (data as Record<string, unknown>).id === "string"
  ) {
    return data as User;      // now we've actually checked the shape
  }
  throw new Error("Invalid user data from API");
}

This is safer than the blind version, but the manual checking is verbose and easy to get wrong. It's fine for one or two fields; for real payloads you want a cleaner tool — which is why the ecosystem reaches for validation libraries.

13.3 Runtime validation — where types meet reality

The professional solution is a schema validation library (Zod is the popular choice) that checks data at runtime and gives you a TypeScript type from the same schema — closing the gap between compile-time types and runtime reality:

import { z } from "zod";

// Define the schema once:
const UserSchema = z.object({
  id: z.string(),
  name: z.string(),
  email: z.string().email().optional(),
});

// Derive the TypeScript type FROM the schema — one source of truth:
type User = z.infer<typeof UserSchema>;   // { id: string; name: string; email?: string }

async function getUser(id: string): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  const data: unknown = await res.json();
  return UserSchema.parse(data);   // validates at runtime; throws if the shape is wrong
}

UserSchema.parse(data) actually checks the data at runtime and returns a correctly-typed User — or throws if the server lied. And z.infer derives the static type from the schema, so your type and your validation can never drift apart. This pattern — validate untrusted data at the boundary, derive the type from the validator — is how production TypeScript apps stay honest about external data. You don't need to master Zod here; just know that "parse, don't assume" is the goal, and a schema library is the tool.

13.4 Typing errors — catch gives you unknown

In modern TypeScript, the variable in a catch clause is unknown (because JavaScript lets you throw anything, not just Error objects). So you must narrow before using it:

try {
  doSomethingRisky();
} catch (err) {
  // err is unknown — you can't just do err.message
  if (err instanceof Error) {
    console.error(err.message);      // ✅ narrowed to Error
  } else {
    console.error("Unknown error:", err);
  }
}

This is unknown + narrowing (Sections 2 and 5) applied to error handling. A common helper makes it tidy:

function getErrorMessage(err: unknown): string {
  return err instanceof Error ? err.message : String(err);
}

Never assume a caught value is an Error — narrow it. This matters especially with async/await, where a rejected promise or a thrown non-Error can land in your catch.

13.5 Type assertions — as, and when it's a lie

The as keyword is a type assertion — you overriding the compiler: "trust me, this is that type." You've seen it (data as User). It's sometimes necessary, but it's an unchecked claim, and a wrong assertion is a lie that hides a bug:

const el = document.getElementById("app") as HTMLCanvasElement;   // asserting a specific element
const value = someUnknown as string;                               // asserting away unknown

const x = "hello" as unknown as number;   // 🚩 the "double assertion" — almost always wrong

Guidelines: prefer narrowing (a real runtime check) over asserting whenever possible, since narrowing is verified and assertions are not. Reserve as for cases where you genuinely know more than the compiler (e.g., a DOM element you're certain of, or after you've validated data). Treat the as unknown as X double-assertion as a loud warning sign that something is wrong. Every as is a small hole in the type system — use them deliberately, not to silence errors.

13.6 Type predicates — teaching the compiler your own checks

When you write a reusable validation function, a type predicate (x is T) tells the compiler that a true return means the value is that type — so calls to it narrow, just like typeof:

interface User { id: string; name: string; }

// The return type `data is User` is the type predicate:
function isUser(data: unknown): data is User {
  return (
    typeof data === "object" && data !== null &&
    "id" in data && "name" in data
  );
}

const data: unknown = JSON.parse(someJson);
if (isUser(data)) {
  console.log(data.name);    // ✅ narrowed to User by the predicate
}

Without the data is User predicate, isUser would just return boolean and wouldn't narrow anything. With it, TypeScript trusts the function as a custom type guard. This lets you build your own reusable guards for domain types — the manual cousin of what a schema library automates. (There are also assertion functions, function assertIsUser(data): asserts data is User, which throw instead of returning a boolean.)

13.7 A real example — a safe API boundary

interface Repo {
  id: number;
  name: string;
  stars: number;
}

function isRepo(x: unknown): x is Repo {
  return (
    typeof x === "object" && x !== null &&
    typeof (x as Record<string, unknown>).id === "number" &&
    typeof (x as Record<string, unknown>).name === "string" &&
    typeof (x as Record<string, unknown>).stars === "number"
  );
}

function getErrorMessage(err: unknown): string {
  return err instanceof Error ? err.message : String(err);
}

async function fetchRepos(user: string): Promise<Repo[]> {
  try {
    const res = await fetch(`https://api.github.com/users/${user}/repos`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);

    const data: unknown = await res.json();
    if (!Array.isArray(data) || !data.every(isRepo)) {
      throw new Error("Unexpected API response shape");
    }
    return data;   // ✅ verified Repo[] — safe to use downstream
  } catch (err) {
    console.error("Failed to fetch repos:", getErrorMessage(err));
    return [];
  }
}

This is real-world TypeScript in miniature: incoming data starts as unknown, a type predicate (isRepo) verifies the shape before it's trusted, HTTP errors are handled, and the caught error is narrowed via a helper rather than assumed to be an Error. Everything downstream of fetchRepos gets a genuine Repo[], not a hopeful assertion. (In a production app, isRepo would typically be a Zod schema — same idea, less hand-written checking.)

Solution

🏋️ Exercise 13.3

What is a type predicate, and how does function isString(x: unknown): x is string differ from function isString(x: unknown): boolean?

Solution

A type predicate is the special return type x is T that tells the compiler a true result means the argument is type T, so calling the function narrows the value in the branch where it returns true.

  • (x: unknown): boolean — returns just a boolean with no narrowing effect: even inside if (isString(v)), v is still unknown, so you can't safely use it as a string.

The predicate is what makes a custom validation function usable as a real type guard, exactly like the built-in typeof/instanceof checks.

🏋️ Exercise 13.4

When is a type assertion (as) appropriate, and when is it a red flag? What does x as unknown as Y signal?

Solution

A type assertion is appropriate when you genuinely know more than the compiler about a value's type and can't easily prove it with a runtime check — e.g., asserting a specific DOM element (getElementById(...) as HTMLCanvasElement) you're certain exists, or narrowing a value after you've already validated it. It's a red flag when used to silence an error you don't understand, or to force an incompatible type — because assertions are unchecked, a wrong one hides a real bug that types were supposed to catch.

x as unknown as Y (a double assertion) signals that x and Y are so unrelated that TypeScript refuses a direct as, so the code is bypassing the type system entirely. It's almost always a sign of a design problem or a genuine type mismatch that should be fixed properly (e.g., by validating/narrowing) rather than asserted away.

SECTION 14

Migrating JavaScript to TypeScript

Most TypeScript you'll write in your career won't be greenfield — it'll be adding types to an existing JavaScript codebase, or working in one that's mid-migration. TypeScript was designed for this: it's a superset, so you can adopt it incrementally, file by file, without stopping the world. This final teaching section covers how to do that sanely.

14.1 The mindset — gradual, not big-bang

The cardinal rule: don't rewrite everything at once. A big-bang migration — flip every file to .ts and try to fix thousands of errors simultaneously — is miserable and risky. Because TypeScript is a superset and can be configured to allow JavaScript alongside TypeScript, you migrate incrementally: get the tooling in place, convert a file at a time, and tighten strictness as you go. The codebase keeps working the whole time. Patience here is a feature, not a compromise.

14.2 Step 1 — set up TypeScript alongside JavaScript

Add TypeScript and a permissive initial config that lets .js and .ts coexist:

{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "allowJs": true,        // let TS compile/understand .js files too
    "checkJs": false,       // don't type-check .js yet (start quiet)
    "strict": false,        // start lenient — we'll tighten later
    "outDir": "./dist",
    "skipLibCheck": true
  },
  "include": ["src"]
}

allowJs: true is the key that makes coexistence possible — TypeScript will happily process your existing .js files. Starting with strict: false and checkJs: false means adding TypeScript produces few or no immediate errors, so you can integrate the tooling without a wall of red. You'll ratchet these up later.

14.3 Step 2 — convert file by file

Now migrate one file at a time, typically starting with leaf modules (utilities and types with few dependencies) and working up toward the entry points. For each file:

  1. Rename .js → .ts (or .jsx → .tsx).
  2. Fix the errors that appear — usually adding parameter types, handling null/undefined, and declaring object shapes.
  3. Commit. Move to the next file.

Converting leaves first means that when you later convert a file that imports them, its dependencies are already typed, so you get better inference and fewer mysteries. The codebase stays shippable after every single file — that's the whole point of incremental migration.

14.4 Step 3 — use any deliberately as a temporary bridge

During migration, any is a legitimate temporary tool (this is the one time Section 2's "avoid any" relaxes). When converting a file surfaces a gnarly type you don't want to solve right now, annotating it any (or better, unknown and a TODO) lets you finish the file and keep moving, rather than getting stuck:

// A pragmatic temporary bridge during migration:
function processLegacyData(data: any) {   // TODO: type properly once we tackle this module
  return data.whatever.deeply.nested;
}

The discipline is that these are debts to repay, not permanent choices. Many teams grep for any (or use an ESLint rule) to track and gradually eliminate them. Using any to finish the migration is fine; leaving any scattered forever defeats the purpose. Prefer unknown where you can, since it at least forces a check later.

14.5 Step 4 — type JS without converting it, via JSDoc

Sometimes you can't rename a file to .ts yet (build constraints, risk), but you still want types. TypeScript can type-check plain .js files using JSDoc comments — you get real checking without changing the file's language:

// still a .js file, but typed via JSDoc

/** @param {string} name @param {number} age @returns {string} */
function describe(name, age) {
  return `${name} is ${age}`;
}

/** @type {{ id: string; tags: string[] }} */
const record = { id: "1", tags: [] };

Turn on checkJs: true (or add // @ts-check at the top of a file), and TypeScript checks these JSDoc types just like annotations. This is a great intermediate step — you get type safety on legacy .js files, catching bugs before committing to renaming them. Many codebases live comfortably in this JSDoc-typed state for a long time.

14.6 Step 5 — ratchet up strictness

Once a good chunk is converted, tighten the compiler gradually, fixing the new errors each flag surfaces:

// Enable these one at a time, fixing errors between each:
"noImplicitAny": true,        // then...
"strictNullChecks": true,     // (usually the biggest, most valuable batch of fixes)
"strict": true                // finally the full umbrella

Enabling flags one at a time keeps each batch of errors manageable, instead of thousands at once. strictNullChecks is typically the largest and most valuable step — it surfaces all the places null/undefined weren't handled. The destination is always the same: full strict: true, checkJs on, and no stray any. You may take weeks or months to get there on a big codebase, and that's fine — every step leaves it safer than before.

14.7 Common pitfalls, and do's and don'ts

A few things that trip people up:

  • Don't turn strict off (or sprinkle // @ts-ignore) permanently to make errors vanish — you're throwing away the safety you're paying for. Do use any/unknown as tracked, temporary bridges.
  • Don't try to type third-party JS libraries by hand first — check for @types/* packages (Section 10); they usually exist.
  • Don't over-engineer types during migration (elaborate generics, conditional types). Do get things typed first; refine later. A plain, correct type beats a clever, wrong one.
  • Do convert leaf/utility files first and commit often, so the codebase stays shippable and reviews stay small.
  • Do treat the first strictNullChecks pass as bug-finding — many of the errors it raises are real latent bugs, not false alarms.

The overarching do: let the migration be boring and incremental. TypeScript's superset design exists precisely so you never have to choose between "all JavaScript" and "all TypeScript" — you can live in a productive in-between and move at whatever pace your team can sustain.

15.2 The domain types

Start with the single source of truth. Notice how the DTOs are derived, never restated (Section 9):

// src/types.ts

export type Priority = "low" | "medium" | "high";
export type Status = "todo" | "in-progress" | "done";

export interface Task {
  readonly id: string;          // set by the store, never by callers
  readonly createdAt: Date;     // set by the store
  title: string;
  priority: Priority;
  status: Status;
  tags: string[];
}

// What a caller supplies to CREATE a task: everything except store-managed fields.
export type CreateTaskInput = Omit<Task, "id" | "createdAt">;

// What a caller supplies to UPDATE: any subset of the editable fields.
export type UpdateTaskInput = Partial<Pick<Task, "title" | "priority" | "status" | "tags">>;

The readonly ids can't be tampered with; Priority and Status are precise literal unions (no stray strings); and CreateTaskInput/UpdateTaskInput come straight from Task via Omit, Pick, and Partial — add a field to Task and both DTOs update automatically. This is the "derive, don't duplicate" discipline made concrete.

15.3 The generic store

A reusable, type-safe container for any entity with a string id — generics with a constraint (Section 6), returning a Result<T> discriminated union (Section 5) instead of throwing:

// src/store.ts

export interface Entity {
  readonly id: string;
}

export type Result<T> =
  | { ok: true; value: T }
  | { ok: false; error: string };

export class Store<T extends Entity> {
  private readonly items = new Map<string, T>();

  add(item: T): void {
    this.items.set(item.id, item);
  }

  get(id: string): Result<T> {
    const item = this.items.get(id);
    return item
      ? { ok: true, value: item }
      : { ok: false, error: `No item with id "${id}"` };
  }

  replace(id: string, next: T): Result<T> {
    if (!this.items.has(id)) return { ok: false, error: `No item with id "${id}"` };
    this.items.set(id, next);
    return { ok: true, value: next };
  }

  remove(id: string): boolean {
    return this.items.delete(id);
  }

  all(): T[] {
    return [...this.items.values()];
  }

  where(predicate: (item: T) => boolean): T[] {
    return this.all().filter(predicate);
  }

  get size(): number {
    return this.items.size;
  }
}

Store<T extends Entity> works for any type with an id — tasks here, but equally users, repos, anything — with full type safety on every method. Lookups return a Result<T>: callers must check .ok before touching .value, so "not found" is impossible to ignore. This is the typed, exception-free style from Section 13.

15.4 The TaskVault core

Here's where the course culminates — a discriminated-union action type and an exhaustive reducer:

// src/taskStore.ts

import { Store, type Result } from "./store.js";
import type { Task, Priority, Status, CreateTaskInput, UpdateTaskInput } from "./types.js";

let counter = 0;
function nextId(): string {
  counter += 1;
  return `task_${counter}`;
}

// Every mutation is one of these, tagged by `type` — the reducer switches on it.
export type TaskAction =
  | { type: "create"; input: CreateTaskInput }
  | { type: "update"; id: string; changes: UpdateTaskInput }
  | { type: "toggleDone"; id: string }
  | { type: "remove"; id: string };

export class TaskVault {
  private readonly store = new Store<Task>();

  create(input: CreateTaskInput): Task {
    const task: Task = { id: nextId(), createdAt: new Date(), ...input };
    this.store.add(task);
    return task;
  }

  update(id: string, changes: UpdateTaskInput): Result<Task> {
    const found = this.store.get(id);
    if (!found.ok) return found;                       // propagate the error Result
    return this.store.replace(id, { ...found.value, ...changes });
  }

  toggleDone(id: string): Result<Task> {
    const found = this.store.get(id);
    if (!found.ok) return found;
    const status: Status = found.value.status === "done" ? "todo" : "done";
    return this.store.replace(id, { ...found.value, status });
  }

  remove(id: string): boolean {
    return this.store.remove(id);
  }

  // The reducer — exhaustiveness enforced by assigning to `never` in default.
  dispatch(action: TaskAction): Result<Task> | boolean {
    switch (action.type) {
      case "create":     return { ok: true, value: this.create(action.input) };
      case "update":     return this.update(action.id, action.changes);
      case "toggleDone": return this.toggleDone(action.id);
      case "remove":     return this.remove(action.id);
      default: {
        const _exhaustive: never = action;   // ← compile error if a case is unhandled
        return _exhaustive;
      }
    }
  }

  byStatus(status: Status): Task[] { return this.store.where((t) => t.status === status); }
  byPriority(priority: Priority): Task[] { return this.store.where((t) => t.priority === priority); }
  withTag(tag: string): Task[] { return this.store.where((t) => t.tags.includes(tag)); }
  all(): Task[] { return this.store.all(); }

  // Stats typed with Record over the literal unions → must cover every value.
  stats(): { total: number; byStatus: Record<Status, number>; byPriority: Record<Priority, number> } {
    const byStatus: Record<Status, number> = { todo: 0, "in-progress": 0, done: 0 };
    const byPriority: Record<Priority, number> = { low: 0, medium: 0, high: 0 };
    for (const t of this.store.all()) {
      byStatus[t.status] += 1;
      byPriority[t.priority] += 1;
    }
    return { total: this.store.size, byStatus, byPriority };
  }
}

Trace the concepts: create accepts a CreateTaskInput and the store fills the readonly fields; update/toggleDone return a Result and propagate the not-found error by returning it; dispatch switches on the discriminated action.type and the default branch's const _exhaustive: never = action guarantees every action variant is handled; and stats uses Record<Status, number> / Record<Priority, number>, which the compiler requires to contain an entry for every literal.

The exhaustiveness guarantee is not hypothetical. Add a new action — say | { type: "archive"; id: string } — to TaskAction without adding a case "archive", and the compiler stops you cold:

taskStore.ts: error TS2322: Type '{ type: "archive"; id: string; }'
  is not assignable to type 'never'.

That error points straight at the reducer, forcing you to handle the new case. The same protection applies to stats: add "blocked" to Status, and the Record<Status, number> initializer fails to compile until you add the blocked: 0 entry. The types make forgetting impossible.

15.5 The runnable demo

// src/index.ts

import { TaskVault, type TaskAction } from "./taskStore.js";

const vault = new TaskVault();

const a = vault.create({ title: "Write TS course", priority: "high", status: "todo", tags: ["writing"] });
const b = vault.create({ title: "Review PR", priority: "medium", status: "todo", tags: ["code", "review"] });
vault.create({ title: "Water plants", priority: "low", status: "in-progress", tags: ["home"] });

// Update returns a Result — must check .ok before using .value:
const upd = vault.update(a.id, { status: "in-progress", tags: ["writing", "urgent"] });
console.log("update ok?", upd.ok, upd.ok ? upd.value.tags : upd.error);

vault.toggleDone(b.id);

// Drive mutations through the typed reducer:
const actions: TaskAction[] = [
  { type: "create", input: { title: "Ship it", priority: "high", status: "todo", tags: [] } },
  { type: "toggleDone", id: a.id },
  { type: "update", id: b.id, changes: { priority: "low" } },
];
for (const action of actions) vault.dispatch(action);

console.log("done tasks:", vault.byStatus("done").map((t) => t.title));
console.log("high priority:", vault.byPriority("high").map((t) => t.title));

// A missing id yields an error Result rather than throwing:
const missing = vault.update("task_999", { title: "nope" });
console.log("missing update:", missing.ok ? "ok" : missing.error);

console.log("stats:", JSON.stringify(vault.stats(), null, 2));

Compile and run it:

npx tsc         # type-checks under strict AND emits JS to ./dist (or your outDir)
node dist/index.js

You'll see the updates, queries, the handled "not found" case, and the derived stats print out — a fully-typed core doing real work, with the compiler having verified every line first.

15.7 Extending it

The one idea to keep

If you remember a single thing from this course, make it this: make illegal states unrepresentable. Every technique here — literal unions instead of loose strings, discriminated unions with exhaustiveness, readonly for immutable fields, Result types instead of hidden exceptions, deriving DTOs so they can't drift, strictNullChecks so null is visible — serves that one goal. When your types are precise enough that the wrong thing doesn't compile, you've moved a whole category of bugs from "found in production" to "impossible to write." That is the entire point of TypeScript, and you now know how to do it.
