Complete SQL Mastery — A Self-Study Course

A structured, hands-on SQL course modeled on Mosh Hamedani's Complete SQL Mastery curriculum. Primary dialect: MySQL 8.0. PostgreSQL differences are called out in PG Notes where relevant.

How to use this course

  • Set up your environment first (Section 1). You need to be able to run the SQL, not just read it.
  • Each section follows the same shape: Concept → Syntax → Example → Exercises with solutions.
  • Don't skip the exercises. Try them yourself first; only check the solution when you're stuck or done.
  • The course uses three sample databases — sql_store, sql_invoicing, and sql_hr — which you'll create in Section 1. Every example references these.
  • Sections 16–20 are the “pro” track. They go past the core curriculum into window functions, query diagnostics, application integration, migrations, and scaling — the skills that separate writing SQL from engineering database systems.
SECTION 1

Getting Started

1.1 Introduction

Welcome. By the end of this course you will be able to:

  • Read data from one or many related tables.
  • Insert, update, and delete data safely.
  • Aggregate and summarize large datasets.
  • Write complex queries (subqueries, joins, EXISTS).
  • Design a normalized relational database from scratch.
  • Add indexes for performance and grants for security.

SQL is one of the highest-leverage skills you can have as a developer or analyst. The syntax is small, but the thinking in sets is what takes practice — which is what these exercises are for.

1.2 What is SQL?

SQL (Structured Query Language) is the language used to talk to relational databases. A relational database stores data in tables: rows and columns, like spreadsheets, but with rules and relationships.

Key distinction: - A database is the data itself (the tables and rows). - A DBMS (Database Management System) is the software that runs databases — MySQL, PostgreSQL, SQL Server, Oracle, SQLite.

SQL is mostly standardized across DBMSes, but each has its own small dialect. We use MySQL.

1.3 Installing MySQL on Mac / Windows

You have three paths — pick one:

Option A — Online sandbox (recommended to start, zero install): Go to db-fiddle.com, set the database on the left to "MySQL 8.0". You get a Schema panel (for CREATE TABLE / INSERT) and a Query panel. Perfect for everything in this course up to Section 9 or so.

Option B — Install locally on Mac: 1. Install Homebrew if you don't have it: see brew.sh 2. brew install mysql 3. brew services start mysql 4. Install MySQL Workbench (GUI) from mysql.com/products/workbench

Option C — Install locally on Windows: 1. Download the MySQL Installer from dev.mysql.com/downloads/installer 2. Choose "Developer Default" — this installs MySQL Server + Workbench + sample data. 3. During setup, set a root password and remember it.

PG Note: If you want to use PostgreSQL instead, install PostgreSQL + pgAdmin (or use the online sandbox at sqliteonline.com → pick PostgreSQL). About 90% of this course works unchanged; I'll note the differences.

SECTION 2

Retrieving Data From a Single Table

All examples in this section use the sql_store database. Run USE sql_store; first.

2.1 The SELECT Statement

The SELECT statement is the query. Its full skeleton:

SELECT   column_list      -- which columns
FROM     table_name       -- from which table
WHERE    condition        -- filter rows (optional)
ORDER BY column           -- sort (optional)
LIMIT    n;               -- cap rows (optional)

The simplest form:

SELECT * FROM customers;

* means "all columns". This returns every row of customers. Note the semicolon — SQL statements end with ;. (Optional if you only run one, but get in the habit.)

Comments:

-- Single-line comment
/* multi-line
   comment */

2.2 The SELECT Clause

You can list specific columns, do math, alias columns, and remove duplicates.

-- Specific columns
SELECT first_name, last_name FROM customers;

-- Math expressions
SELECT first_name, points, points + 10 FROM customers;
SELECT first_name, points, points * 10 + 100 FROM customers;

-- Aliases with AS (AS is optional, but clearer)
SELECT first_name, points, points * 10 + 100 AS discount_factor
FROM customers;

-- Aliases with spaces: use quotes
SELECT first_name AS "Given Name" FROM customers;

-- DISTINCT removes duplicate rows in the result
SELECT DISTINCT state FROM customers;

Operator precedence: * and / happen before + and -. Use parentheses to force order:

SELECT points * (10 + 100) AS result FROM customers;

🏋️ Exercise 2.2

Return all products with a new column called new_price that is 10% higher than unit_price.

Solution 
SELECT
  name,
  unit_price,
  unit_price * 1.1 AS new_price
FROM products;

2.3 The WHERE Clause

WHERE filters rows. It runs before SELECT decides which columns to return.

SELECT * FROM customers WHERE points > 3000;

Comparison operators: >, >=, <, <=, =, != (or <>).

SELECT * FROM customers WHERE state = 'VA';
SELECT * FROM customers WHERE state != 'VA';
SELECT * FROM customers WHERE birth_date > '1990-01-01';

String/date literals use single quotes. Strings are case-insensitive by default in MySQL.

🏋️ Exercise 2.3

Get the orders placed this year (or any year — use 2019 against the sample data).

Solution 
SELECT * FROM orders WHERE YEAR(order_date) = 2019;
-- Or:
SELECT * FROM orders WHERE order_date >= '2019-01-01';

2.4 The AND, OR, and NOT Operators

Combine conditions:

-- Both must be true
SELECT * FROM customers
WHERE birth_date > '1990-01-01' AND points > 1000;

-- Either one
SELECT * FROM customers
WHERE birth_date > '1990-01-01' OR points > 1000;

-- NOT negates
SELECT * FROM customers
WHERE NOT (birth_date > '1990-01-01' OR points > 1000);

Precedence: AND is evaluated before OR. Use parentheses to be explicit:

SELECT * FROM customers
WHERE birth_date > '1990-01-01'
   OR (points > 1000 AND state = 'VA');

🏋️ Exercise 2.4

From order_items, get items in order #6 where the total price (quantity × unit_price) is greater than 30.

Solution 
SELECT *
FROM order_items
WHERE order_id = 6 AND quantity * unit_price > 30;

2.5 The IN Operator

Use IN when you'd otherwise repeat OR for the same column:

-- Verbose
SELECT * FROM customers WHERE state = 'VA' OR state = 'FL' OR state = 'GA';

-- Clean
SELECT * FROM customers WHERE state IN ('VA','FL','GA');

-- NOT IN works too
SELECT * FROM customers WHERE state NOT IN ('VA','FL','GA');

🏋️ Exercise 2.5

Get products with quantity_in_stock of 49, 38, or 72.

Solution 
SELECT * FROM products WHERE quantity_in_stock IN (49, 38, 72);

2.6 The BETWEEN Operator

BETWEEN is inclusive on both ends:

SELECT * FROM customers WHERE points BETWEEN 1000 AND 3000;
-- Equivalent to: points >= 1000 AND points <= 3000

SELECT * FROM customers
WHERE birth_date BETWEEN '1990-01-01' AND '2000-01-01';

2.7 The LIKE Operator

Pattern match on strings.

  • % = any number of characters (including zero)
  • _ = exactly one character
SELECT * FROM customers WHERE last_name LIKE 'b%';    -- starts with b
SELECT * FROM customers WHERE last_name LIKE '%b%';   -- contains b
SELECT * FROM customers WHERE last_name LIKE '%y';    -- ends with y
SELECT * FROM customers WHERE last_name LIKE '_____y';-- 5 chars + y
SELECT * FROM customers WHERE last_name LIKE 'b____y';-- b, 4 chars, y

🏋️ Exercise 2.7

Get customers whose addresses contain trail or avenue, and whose phone numbers end with 9.

Solution 
SELECT *
FROM customers
WHERE (address LIKE '%trail%' OR address LIKE '%avenue%')
  AND phone LIKE '%9';

2.8 The REGEXP Operator

REGEXP is LIKE on steroids — full regular expressions.

SELECT * FROM customers WHERE last_name REGEXP 'field';      -- contains 'field'
SELECT * FROM customers WHERE last_name REGEXP '^field';     -- starts with
SELECT * FROM customers WHERE last_name REGEXP 'field$';     -- ends with
SELECT * FROM customers WHERE last_name REGEXP 'field|mac';  -- 'field' OR 'mac'
SELECT * FROM customers WHERE last_name REGEXP '[gim]e';     -- ge, ie, or me
SELECT * FROM customers WHERE last_name REGEXP '[a-h]e';     -- ae..he

Key meta-characters: ^ start, $ end, | or, [abc] class, [a-f] range.

PG Note: PostgreSQL uses ~ (case-sensitive) or ~* (case-insensitive) for regex: WHERE last_name ~* 'field'.

🏋️ Exercise 2.8

Find customers whose: a) first names are ELKA or AMBUR b) last names end with EY or ON c) last names start with MY or contain SE d) last names contain B followed by R or U

Solution 
SELECT * FROM customers WHERE first_name REGEXP 'elka|ambur';
SELECT * FROM customers WHERE last_name  REGEXP 'ey$|on$';
SELECT * FROM customers WHERE last_name  REGEXP '^my|se';
SELECT * FROM customers WHERE last_name  REGEXP 'b[ru]';

2.9 The IS NULL Operator

NULL means "unknown / missing". You cannot compare with =. Use IS NULL / IS NOT NULL.

SELECT * FROM customers WHERE phone IS NULL;
SELECT * FROM customers WHERE phone IS NOT NULL;

This is a common bug source: WHERE phone = NULL returns zero rows, always. NULL is not equal to anything, not even another NULL.

🏋️ Exercise 2.9

Get orders that haven't been shipped.

Solution 
SELECT * FROM orders WHERE shipped_date IS NULL;

2.10 The ORDER BY Clause

Sort the result.

SELECT * FROM customers ORDER BY first_name;            -- ascending (default)
SELECT * FROM customers ORDER BY first_name DESC;       -- descending
SELECT * FROM customers ORDER BY state, first_name;     -- sort by state, then within each state by first name
SELECT * FROM customers ORDER BY state DESC, first_name DESC;

You can sort by columns not in the SELECT list, by aliases, or by expressions:

SELECT first_name, last_name, 10 AS points FROM customers ORDER BY points;
SELECT *, quantity * unit_price AS total_price
FROM order_items
WHERE order_id = 2
ORDER BY total_price DESC;

🏋️ Exercise 2.10

From order #2 in order_items, sort items by total price (quantity × unit_price) descending.

Solution 
SELECT *, quantity * unit_price AS total_price
FROM order_items
WHERE order_id = 2
ORDER BY total_price DESC;

2.11 The LIMIT Clause

Cap how many rows you return — essential for big tables.

SELECT * FROM customers LIMIT 3;        -- first 3
SELECT * FROM customers LIMIT 6, 3;     -- skip 6, return next 3 (pagination)

LIMIT is always last in MySQL: after ORDER BY.

SELECT * FROM customers ORDER BY points DESC LIMIT 3;  -- top 3 by points

PG Note: PostgreSQL also supports LIMIT n OFFSET m syntax: LIMIT 3 OFFSET 6. MySQL accepts this too.

🏋️ Exercise 2.11

Get the top 3 most-loyal customers (highest points).

Solution 
SELECT * FROM customers ORDER BY points DESC LIMIT 3;

Section 2 — Recap

You can now query a single table. The full clause order is:

SELECT … FROM … WHERE … ORDER BY … LIMIT …;

But the logical execution order is different — FROM → WHERE → SELECT → ORDER BY → LIMIT. The database first picks the table, filters rows, then picks columns and sorts. Knowing this prevents many beginner mistakes.

SECTION 3

Retrieving Data From Multiple Tables

This is where SQL gets powerful. Real databases split data across tables to avoid duplication, and joins glue them back together.

3.1 Inner Joins

The most common join. "Inner" because rows only appear in the result when there's a match on both sides.

SELECT *
FROM orders
INNER JOIN customers ON orders.customer_id = customers.customer_id;

INNER is optional — JOIN alone means inner join. But being explicit is clearer.

Qualify column names when the same name exists in both tables:

SELECT order_id, orders.customer_id, first_name, last_name
FROM orders
JOIN customers ON orders.customer_id = customers.customer_id;

Use table aliases to keep it short:

SELECT order_id, o.customer_id, first_name, last_name
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id;

Once you alias, you must use the alias (not the original name) in the rest of the query.

🏋️ Exercise 3.1

Join order_items with products to show order_id, product name, quantity, and the unit_price from order_items (not the catalog).

Solution 
SELECT oi.order_id, p.name, oi.quantity, oi.unit_price
FROM order_items oi
JOIN products p ON oi.product_id = p.product_id;

3.2 Joining Across Databases

You can join tables from different databases by prefixing with the database name:

USE sql_store;

SELECT *
FROM order_items oi
JOIN sql_inventory.products p ON oi.product_id = p.product_id;

(Only works if both databases live on the same MySQL server.)

3.3 Self Joins

You can join a table with itself. Classic example: employees and their managers in sql_hr.

USE sql_hr;

SELECT
  e.employee_id,
  e.first_name AS employee,
  m.first_name AS manager
FROM employees e
JOIN employees m ON e.reports_to = m.employee_id;

Each row of e is matched with the row of m that's their manager. Anya (the CEO) has no manager, so she won't appear in this INNER JOIN. (Outer joins handle that — coming up.)

3.4 Joining Multiple Tables

You can chain joins:

USE sql_store;

SELECT
  o.order_id,
  o.order_date,
  c.first_name,
  c.last_name,
  os.name AS status
FROM orders o
JOIN customers c     ON o.customer_id = c.customer_id
JOIN order_statuses os ON o.status      = os.order_status_id;

🏋️ Exercise 3.4

In sql_invoicing, list every payment with its client name and payment method name.

Solution 
USE sql_invoicing;

SELECT
  p.date,
  p.amount,
  c.name      AS client,
  pm.name     AS payment_method
FROM payments p
JOIN clients c          ON p.client_id      = c.client_id
JOIN payment_methods pm ON p.payment_method = pm.payment_method_id;

3.5 Compound Join Conditions

When the relationship needs more than one column (composite keys), use AND:

SELECT *
FROM order_items oi
JOIN order_item_notes oin
  ON oi.order_id   = oin.order_id
 AND oi.product_id = oin.product_id;

3.6 Implicit Join Syntax

You'll see this in older code:

SELECT *
FROM orders o, customers c
WHERE o.customer_id = c.customer_id;

Avoid it. It works, but if you forget the WHERE clause you get a Cartesian product — every row of A paired with every row of B. With 10 customers and 6 orders, that's 60 rows of nonsense. Stick with explicit JOIN ... ON.

3.7 Outer Joins

INNER JOIN drops rows without a match. Outer joins keep them.

  • LEFT JOIN: keep all rows from the left table, even unmatched.
  • RIGHT JOIN: keep all rows from the right table.
  • FULL OUTER JOIN: keep both. (Not in MySQL; emulate with UNION of LEFT and RIGHT.)
-- All customers, with their orders if any (customers without orders show NULL order_id)
SELECT c.customer_id, c.first_name, o.order_id
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id;

In the sample data, customers 1, 3, 4, 9 have no orders — they appear with NULL for order_id.

Rule of thumb: always write LEFT JOINs (avoid RIGHT). Read left-to-right: "keep everything from this table, attach matches from that one."

🏋️ Exercise 3.7

Show all products and the total quantity ordered. Include products that were never ordered.

Solution 
SELECT p.product_id, p.name, oi.quantity
FROM products p
LEFT JOIN order_items oi ON p.product_id = oi.product_id;
(In Section 5 we'll add `SUM` and `GROUP BY` to actually sum quantities.)

3.8 Outer Join Between Multiple Tables

You can mix outer joins:

SELECT
  c.customer_id,
  c.first_name,
  o.order_id,
  sh.name AS shipper
FROM customers c
LEFT JOIN orders o   ON c.customer_id = o.customer_id
LEFT JOIN shippers sh ON o.shipper_id  = sh.shipper_id
ORDER BY c.customer_id;

Important: once you LEFT JOIN, subsequent INNER JOINs on that side will re-filter and drop the unmatched rows. Use LEFT JOIN consistently for everything you want preserved.

3.9 Self Outer Joins

Combine self join with outer join — fixes the "CEO is missing" problem from 3.3:

USE sql_hr;

SELECT
  e.first_name AS employee,
  m.first_name AS manager
FROM employees e
LEFT JOIN employees m ON e.reports_to = m.employee_id;

Now Anya appears with NULL manager.

3.10 The USING Clause

When the join column has the same name in both tables, shorten with USING:

-- Verbose:
SELECT *
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id;

-- With USING:
SELECT *
FROM orders o
JOIN customers c USING (customer_id);

Multiple columns:

JOIN order_item_notes USING (order_id, product_id)

3.11 Natural Joins

SELECT *
FROM orders
NATURAL JOIN customers;

The DB guesses the join columns by matching names. Avoid this. Implicit magic = silent bugs.

3.12 Cross Joins

Every row of A paired with every row of B — Cartesian product. Useful for generating combinations (sizes × colors).

SELECT s.name AS shipper, p.name AS product
FROM shippers s
CROSS JOIN products p;

3.13 Unions

UNION combines rows from two queries (joins combine columns).

SELECT order_id, order_date, 'Active'   AS status FROM orders WHERE order_date >= '2019-01-01'
UNION
SELECT order_id, order_date, 'Archived' AS status FROM orders WHERE order_date <  '2019-01-01';

Rules: - Same number of columns in each SELECT. - Column names taken from the first query. - UNION removes duplicates; UNION ALL keeps them (and is faster).

🏋️ Exercise 3.13

Produce a customer report classifying loyalty by points: <2000 = Bronze, 2000–2999 = Silver, ≥3000 = Gold. Sort by first name.

Solution (UNION version — there's a cleaner CASE version later) 
SELECT customer_id, first_name, points, 'Bronze' AS type
FROM customers WHERE points < 2000
UNION
SELECT customer_id, first_name, points, 'Silver' AS type

When you're new, drawing the relationships on paper helps. For each join, ask: "What's the left table, what's the right, and what column connects them?" If you can answer that, the SQL writes itself.

SECTION 4

Inserting, Updating, and Deleting Data

You've been reading. Time to write.

4.1 Column Attributes

Open customers in your client and look at the columns:

  • Data type — INT, VARCHAR(50), DATE, DECIMAL(4,2), etc.
  • PK — primary key (uniquely identifies the row).
  • NN — NOT NULL (column must have a value).
  • AI — AUTO_INCREMENT (DB generates the value).
  • Default — value used when you don't supply one.

These attributes determine what you can insert.

4.2 Inserting a Row

INSERT INTO customers
VALUES (DEFAULT, 'John', 'Smith', '1990-01-01', NULL, 'address', 'city', 'CA', DEFAULT);

DEFAULT lets the DB fill in customer_id (auto-increment) and points (default 0). NULL works for nullable columns.

Better: specify columns explicitly. Then column order doesn't matter and you can skip the optional ones.

INSERT INTO customers (first_name, last_name, address, city, state)
VALUES ('John', 'Smith', '123 Main', 'San Diego', 'CA');

4.3 Inserting Multiple Rows

Comma-separate the value tuples:

INSERT INTO shippers (name) VALUES
('Shipper1'), ('Shipper2'), ('Shipper3');

4.4 Inserting Hierarchical Rows

Insert a row, then insert child rows that reference it. Use LAST_INSERT_ID() to grab the auto-generated id:

INSERT INTO orders (customer_id, order_date, status)
VALUES (1, '2019-01-02', 1);

INSERT INTO order_items
VALUES (LAST_INSERT_ID(), 1, 1, 2.95),
       (LAST_INSERT_ID(), 2, 1, 3.95);

PG Note: PostgreSQL uses RETURNING:

INSERT INTO orders (...) VALUES (...) RETURNING order_id;

4.5 Creating a Copy of a Table

INSERT ... SELECT copies rows from one table into another:

CREATE TABLE orders_archived AS SELECT * FROM orders;

This creates the table structure and copies data — but does not copy primary key / auto-increment attributes. Fine for backups.

To insert filtered rows into an existing table:

INSERT INTO orders_archived
SELECT * FROM orders WHERE order_date < '2019-01-01';

SECTION 5

Summarizing Data

When you stop looking at individual rows and start computing over groups of them. This is the leap from "querying" to "analytics."

5.1 Aggregate Functions

Built-in functions that compute over a set of rows:

Function Returns
MAX(col) maximum value
MIN(col) minimum value
AVG(col) average
SUM(col) sum
COUNT(col) count of non-null values
COUNT(*) count of rows 
USE sql_invoicing;

SELECT
  MAX(invoice_total)  AS highest,
  MIN(invoice_total)  AS lowest,
  AVG(invoice_total)  AS average,
  SUM(invoice_total)  AS total,
  COUNT(*)            AS total_records,
  COUNT(payment_date) AS count_of_payments
FROM invoices
WHERE invoice_date > '2019-07-01';

Notes: - Aggregates ignore NULL. - COUNT(col) counts only non-null values of that column; COUNT(*) counts every row. - COUNT(DISTINCT col) counts unique values.

5.2 The GROUP BY Clause

GROUP BY says "compute the aggregate per group" instead of one number for the whole table.

-- Total sales per client
SELECT
  client_id,
  SUM(invoice_total) AS total_sales
FROM invoices
GROUP BY client_id
ORDER BY total_sales DESC;

Group by multiple columns — one row per combination:

SELECT
  state,
  city,
  SUM(invoice_total) AS total_sales
FROM invoices i
JOIN clients c USING (client_id)
GROUP BY state, city;

The rule: every column in SELECT must either be (a) inside an aggregate function, or (b) listed in GROUP BY. Otherwise the result is meaningless ("which row of name do you want me to show for this group?").

🏋️ Exercise 5.2

Per payment method, get the total amount paid in 2019.

Solution 
SELECT

  SUM(p.amount) AS total
FROM payments p

5.3 The HAVING Clause

WHERE filters rows before grouping. HAVING filters groups after grouping.

-- Wrong: can't use aggregates in WHERE
-- SELECT client_id, SUM(invoice_total) AS total
-- FROM invoices
-- WHERE SUM(invoice_total) > 500   -- ERROR
-- GROUP BY client_id;

-- Right:
SELECT client_id, SUM(invoice_total) AS total
FROM invoices
GROUP BY client_id
HAVING total > 500;

Rule of thumb: filter row attributes with WHERE, filter aggregate results with HAVING.

🏋️ Exercise 5.3

From sql_store, find customers from Virginia who have spent more than $100 total. (You'll need to join customers, orders, order_items.)

Solution

SELECT

  c.first_name,
  c.last_name,

5.4 The ROLLUP Operator

MySQL-specific. Adds a "subtotal" row per group, plus a grand total.

SELECT
  state,
  city,
  SUM(invoice_total) AS total
FROM invoices i
JOIN clients c USING (client_id)
GROUP BY state, city WITH ROLLUP;

You'll see rows like ('CA','San Francisco', X), ('CA',NULL, sum-for-CA), then at the end (NULL,NULL, grand_total).

PG Note: PostgreSQL uses GROUP BY ROLLUP (state, city) — slightly different syntax.

SECTION 6

Writing Complex Queries

6.1 Introduction to Subqueries

A subquery is a SELECT inside another query. Used in WHERE, SELECT, or FROM.

The pattern: "Find rows where X matches the result of this other query."

-- Products more expensive than the average
SELECT *
FROM products
WHERE unit_price > (SELECT AVG(unit_price) FROM products);

The inner query runs first, produces a value (or a set of values), then the outer query uses it.

6.2 Subqueries

-- Find products that have never been ordered
SELECT *
FROM products
WHERE product_id NOT IN (SELECT DISTINCT product_id FROM order_items);

The subquery returns a list of product IDs that appear in any order. The outer query keeps products NOT in that list.

🏋️ Exercise 6.2

In sql_invoicing, find clients with no invoices.

Solution 
USE sql_invoicing;

SELECT *
FROM clients
WHERE client_id NOT IN (SELECT DISTINCT client_id FROM invoices);

6.3 The IN Operator (with subqueries)

IN checks membership in a list — and a subquery just provides that list:

-- Customers who have ordered lettuce (product 3)
SELECT *
FROM customers
WHERE customer_id IN (
  SELECT customer_id
  FROM orders
  WHERE order_id IN (SELECT order_id FROM order_items WHERE product_id = 3)
);

Reads inside-out: which orders contain product 3 → which customers placed those orders → those customers.

6.4 Subqueries vs Joins

Often you can rewrite a subquery as a join (and vice versa). Use whichever is clearer.

-- Same result, with join:
SELECT DISTINCT c.*
FROM customers c
JOIN orders o      USING (customer_id)
JOIN order_items oi USING (order_id)
WHERE oi.product_id = 3;

General guideline: - Joins are usually clearer when you need columns from multiple tables. - Subqueries are clearer when checking existence/membership without needing those columns. - Performance differences exist but vary; trust the optimizer first.

6.5 The ALL Keyword

> ALL (...) = greater than every value in the set.

-- Invoices larger than EVERY invoice of client 3
SELECT *
FROM invoices
WHERE invoice_total > ALL (SELECT invoice_total FROM invoices WHERE client_id = 3);

Equivalent to > MAX(...), but more readable in some cases.

6.6 The ANY Keyword

= ANY (...) is identical to IN (...). > ANY (...) = greater than at least one (i.e., greater than the MIN).

-- Clients with at least 2 invoices
SELECT *
FROM clients
WHERE client_id = ANY (
  SELECT client_id FROM invoices GROUP BY client_id HAVING COUNT(*) >= 2
);

6.7 Correlated Subqueries

The inner query references the outer row. It runs once per outer row — slower, but expressive.

-- Employees whose salary is above their office's average
USE sql_hr;

SELECT *
FROM employees e
WHERE salary > (
  SELECT AVG(salary)
  FROM employees
  WHERE office_id = e.office_id   -- e from the outer query
);

Mental model: "for each employee e, compute the avg salary in that same office, and keep e only if their salary exceeds it."

6.8 The EXISTS Operator

EXISTS (subquery) returns TRUE if the subquery returns any row. Used to test existence without bringing back the data.

-- Clients who have at least one invoice
SELECT *
FROM clients c
WHERE EXISTS (SELECT 1 FROM invoices WHERE client_id = c.client_id);

-- Clients with no invoices
SELECT *
FROM clients c
WHERE NOT EXISTS (SELECT 1 FROM invoices WHERE client_id = c.client_id);

EXISTS vs IN: - IN materializes the full list first; EXISTS short-circuits as soon as it finds one match. - For large datasets, EXISTS is usually faster. - IN is simpler when the list is small.

6.9 Subqueries in the SELECT Clause

Compute a value per row using a subquery:

SELECT
  invoice_id,
  invoice_total,
  (SELECT AVG(invoice_total) FROM invoices) AS overall_avg,
  invoice_total - (SELECT AVG(invoice_total) FROM invoices) AS difference
FROM invoices;

6.10 Subqueries in the FROM Clause

Treat a query as if it were a table. Must alias it.

SELECT *
FROM (
  SELECT
    c.client_id,
    c.name,
    (SELECT SUM(invoice_total) FROM invoices WHERE client_id = c.client_id) AS total_sales,
    (SELECT AVG(invoice_total) FROM invoices) AS average,
    (SELECT total_sales - average) AS difference
  FROM clients c
) AS sales_summary
WHERE total_sales IS NOT NULL;

Use sparingly — when the inner query gets complex, prefer a view (Section 8) or a CTE (WITH ...).

Essential MySQL Functions

Built-in functions for transforming values inside queries.

7.1 Numeric Functions

SELECT ROUND(5.7);        -- 6
SELECT ROUND(5.7345, 2);  -- 5.73 (round to 2 decimals)
SELECT TRUNCATE(5.7345,2);-- 5.73 (chop, don't round)

SELECT FLOOR(5.7);        -- 5  (round down)
SELECT ABS(-5.7);         -- 5.7

7.2 String Functions

SELECT LENGTH('sky');             -- 3
SELECT UPPER('sky');              -- 'SKY'
SELECT LOWER('SKY');              -- 'sky'
SELECT LTRIM('   sky');           -- 'sky'
SELECT RTRIM('sky   ');           -- 'sky'
SELECT TRIM('   sky   ');         -- 'sky'
SELECT LEFT('Kindergarten', 4);   -- 'Kind'
SELECT RIGHT('Kindergarten', 6);  -- 'garten'
SELECT SUBSTRING('Kindergarten', 3, 5); -- 'nderg'  (start at pos 3, length 5)
SELECT LOCATE('garten', 'Kindergarten'); -- 7   (1-based; 0 if not found)
SELECT REPLACE('Kindergarten','garten','GARDEN'); -- 'KinderGARDEN'
SELECT CONCAT('Hello', ' ', 'World'); -- 'Hello World'

PG Note: PostgreSQL uses || for concat: 'Hello' || ' ' || 'World'. CONCAT also works.

SELECT CONCAT(first_name, ' ', last_name) AS full_name FROM customers;

7.3 Date Functions

SELECT NOW();          -- 2026-05-26 14:23:01
SELECT CURDATE();      -- 2026-05-26
SELECT CURTIME();      -- 14:23:01
SELECT YEAR(NOW());    -- 2026
SELECT MONTH(NOW());   -- 5
SELECT DAY(NOW());     -- 26
SELECT HOUR(NOW());
SELECT MINUTE(NOW());
SELECT DAYNAME(NOW()); -- 'Tuesday'
SELECT MONTHNAME(NOW());-- 'May'
SELECT EXTRACT(YEAR FROM NOW()); -- standard-SQL alternative

Use in queries:

SELECT * FROM orders WHERE YEAR(order_date) = YEAR(NOW());

7.4 Formatting Dates and Times

SELECT DATE_FORMAT(NOW(), '%M %d %Y');   -- 'May 26 2026'
SELECT DATE_FORMAT(NOW(), '%y-%m-%d');   -- '26-05-26'
SELECT TIME_FORMAT(NOW(), '%H:%i %p');   -- '14:23 PM'

Common specifiers: %Y 4-digit year, %y 2-digit, %M month name, %m month number, %D day with suffix (1st, 2nd), %d 2-digit day, %H hour 24h, %h hour 12h, %i minutes, %s seconds, %p AM/PM.

7.5 Calculating Dates and Times

SELECT DATE_ADD(NOW(), INTERVAL 1 DAY);
SELECT DATE_ADD(NOW(), INTERVAL -1 YEAR);
SELECT DATE_SUB(NOW(), INTERVAL 1 YEAR);

SELECT DATEDIFF('2026-12-31', '2026-01-01');   -- 364 (days)
SELECT TIME_TO_SEC('09:00') - TIME_TO_SEC('09:02'); -- -120

DATEDIFF returns days, not negative-aware semantics across types.

7.6 IFNULL and COALESCE

Handle NULLs in a query.

-- If shipper_id is NULL, show 'Not assigned'
SELECT
  order_id,
  IFNULL(shipper_id, 'Not assigned') AS shipper
FROM orders;

-- COALESCE: return the first non-NULL argument (any number of args)
SELECT
  order_id,
  COALESCE(shipper_id, comments, 'Unknown') AS info
FROM orders;

IFNULL takes exactly 2 args; COALESCE takes many. COALESCE is standard SQL — prefer it.

7.7 The IF Function

IF(condition, value_if_true, value_if_false):

SELECT
  order_id,
  order_date,
  IF(YEAR(order_date) = YEAR(NOW()), 'Active', 'Archived') AS category
FROM orders;

7.8 The CASE Operator

For multi-way conditions. Standard SQL — works everywhere.

SELECT
  order_id,
  CASE
    WHEN YEAR(order_date) = YEAR(NOW())     THEN 'Active'
    WHEN YEAR(order_date) = YEAR(NOW()) - 1 THEN 'Last year'
    WHEN YEAR(order_date) < YEAR(NOW()) - 1 THEN 'Archived'
    ELSE 'Future'
  END AS category
FROM orders;

This is the clean replacement for the UNION-based loyalty classification we did in 3.13:

SELECT
  customer_id,
  first_name,
  points,
  CASE
    WHEN points >= 3000 THEN 'Gold'
    WHEN points >= 2000 THEN 'Silver'
    ELSE                      'Bronze'
  END AS type
FROM customers;

Whenever you find yourself writing UNIONs of the same table with different labels, reach for CASE.

SECTION 8

Views

8.1 Creating Views

A view is a saved query. You query it like a table. Underneath, the DB just runs the underlying SELECT.

USE sql_invoicing;

CREATE VIEW sales_by_client AS
SELECT
  c.client_id,
  c.name,
  SUM(invoice_total) AS total_sales
FROM clients c
JOIN invoices i USING (client_id)
GROUP BY c.client_id, c.name;

Now use it as if it were a table:

SELECT * FROM sales_by_client;
SELECT * FROM sales_by_client WHERE total_sales > 500;

Why views matter: - Encapsulate complex queries. Your team queries sales_by_client instead of remembering the joins. - Security. Grant access to the view, not the underlying tables. - Stability. Change the schema; if you keep the view's output the same, calling code doesn't break.

8.2 Altering or Dropping Views

DROP VIEW sales_by_client;

-- Or replace in one statement (recommended workflow):
CREATE OR REPLACE VIEW sales_by_client AS
SELECT … ;

8.3 Updatable Views

A view is updatable (you can INSERT/UPDATE/DELETE through it) only if it doesn't use: - DISTINCT - Aggregate functions (SUM, COUNT, etc.) - GROUP BY or HAVING - UNION

If those rules are met, modifications flow to the base table.

CREATE OR REPLACE VIEW invoices_with_balance AS
SELECT
  invoice_id, number, client_id, invoice_total, payment_total,
  invoice_total - payment_total AS balance,
  invoice_date, due_date, payment_date
FROM invoices
WHERE (invoice_total - payment_total) > 0;

DELETE FROM invoices_with_balance WHERE invoice_id = 1;  -- works

8.4 The WITH CHECK OPTION Clause

Without it, an update through a view can move a row out of the view's WHERE clause — making it disappear from the view, which is surprising.

CREATE OR REPLACE VIEW invoices_with_balance AS
SELECT … FROM invoices WHERE (invoice_total - payment_total) > 0
WITH CHECK OPTION;

Now any UPDATE that would cause the row to no longer match WHERE is rejected.

8.5 Other Benefits of Views

  • Simpler queries — your business logic lives once, in the view definition.
  • Reduce impact of schema changes — rename a column? Update the view; consumers unchanged.
  • Restrict access — expose a view to a user who has no direct table access.
SECTION 9

Stored Procedures

9.1 What are Stored Procedures?

A stored procedure is a named, saved block of SQL code stored on the database server. Call it by name, pass parameters, and get results back. They're like functions for your database.

Benefits: - Store and version business logic in the DB. - Reduce network round-trips (one CALL vs many queries). - Easier permission control. - Reuse and consistency across applications.

Drawbacks: - Versioning is harder than for application code. - Logic in two places (app + DB) can confuse a team. - Performance is great until it isn't — and debugging is painful.

Modern practice: keep app logic in your application; use stored procs sparingly for performance-critical operations.

9.2 Creating a Stored Procedure

We need to change the statement delimiter temporarily so the procedure body can contain ;:

USE sql_invoicing;
DELIMITER $$

CREATE PROCEDURE get_clients()
BEGIN
  SELECT * FROM clients;
END $$

DELIMITER ;

Then call it:

CALL get_clients();

9.3 Creating Procedures Using MySQL Workbench

In Workbench, right-click "Stored Procedures" under the schema → "Create Stored Procedure" → it gives you a template with the DELIMITER already handled. Edit, save, done.

9.4 Dropping Stored Procedures

DROP PROCEDURE get_clients;
DROP PROCEDURE IF EXISTS get_clients;

9.5 Parameters

Procedures can take input parameters:

DELIMITER $$
CREATE PROCEDURE get_clients_by_state(state CHAR(2))
BEGIN
  SELECT * FROM clients c WHERE c.state = state;
END $$
DELIMITER ;

CALL get_clients_by_state('CA');

9.6 Parameters with Default Values

MySQL has no syntactic "default" for parameters. Use IFNULL:

DELIMITER $$
CREATE PROCEDURE get_clients_by_state(state CHAR(2))
BEGIN
  IF state IS NULL THEN
    SELECT * FROM clients;
  ELSE
    SELECT * FROM clients c WHERE c.state = state;
  END IF;
END $$
DELIMITER ;

CALL get_clients_by_state(NULL); -- returns all
CALL get_clients_by_state('CA');

Or shorter with IFNULL:

CREATE PROCEDURE get_clients_by_state(state CHAR(2))
BEGIN
  SELECT * FROM clients c
  WHERE c.state = IFNULL(state, c.state);
END;

9.7 Parameter Validation

Reject bad inputs by raising an error:

DELIMITER $$
CREATE PROCEDURE make_payment(invoice_id INT, payment_amount DECIMAL(9,2), payment_date DATE)
BEGIN
  IF payment_amount <= 0 THEN
    SIGNAL SQLSTATE '22003'
      SET MESSAGE_TEXT = 'Invalid payment amount';
  END IF;

  UPDATE invoices i
  SET i.payment_total = payment_amount,
      i.payment_date  = payment_date
  WHERE i.invoice_id = invoice_id;
END $$
DELIMITER ;

Rule of thumb: validate at the application layer too. The DB is a last line of defense, not your only one.

9.8 Output Parameters

For returning scalars. Use OUT and SET:

DELIMITER $$
CREATE PROCEDURE get_unpaid_invoices_for_client(
  client_id    INT,
  OUT invoices_count INT,
  OUT invoices_total DECIMAL(9,2))
BEGIN
  SELECT COUNT(*), SUM(invoice_total)
  INTO invoices_count, invoices_total
  FROM invoices i
  WHERE i.client_id = client_id AND payment_total = 0;
END $$
DELIMITER ;

CALL get_unpaid_invoices_for_client(3, @count, @total);
SELECT @count, @total;

Output parameters are powerful but discouraged in modern style — return a result set instead.

9.9 Variables

Two kinds:

User / session variables — prefixed with @, last as long as your session:

SET @invoices_count = 0;

Local variables — inside a procedure, declared with DECLARE:

DELIMITER $$
CREATE PROCEDURE get_risk_factor()
BEGIN
  DECLARE risk_factor DECIMAL(9,2) DEFAULT 0;
  DECLARE invoices_total DECIMAL(9,2);
  DECLARE invoices_count INT;

  SELECT COUNT(*), SUM(invoice_total) INTO invoices_count, invoices_total
  FROM invoices;

  SET risk_factor = invoices_total / invoices_count * 5;
  SELECT risk_factor;
END $$
DELIMITER ;

9.10 Functions

A function returns a single value (a procedure can return result sets or multiple OUT params). Functions can be used inside other queries.

DELIMITER $$
CREATE FUNCTION get_risk_factor_for_client(client_id INT)
RETURNS INTEGER
READS SQL DATA   -- describes side-effects: see below
BEGIN
  DECLARE risk_factor DECIMAL(9,2) DEFAULT 0;
  DECLARE invoices_total DECIMAL(9,2);
  DECLARE invoices_count INT;

  SELECT COUNT(*), SUM(invoice_total) INTO invoices_count, invoices_total
  FROM invoices i WHERE i.client_id = client_id;

  SET risk_factor = IFNULL(invoices_total,0) / NULLIF(invoices_count,0) * 5;
  RETURN IFNULL(risk_factor, 0);
END $$
DELIMITER ;

SELECT client_id, name, get_risk_factor_for_client(client_id) AS risk
FROM clients;

DROP FUNCTION IF EXISTS get_risk_factor_for_client;

The required clause is one of: DETERMINISTIC (same input ⇒ same output), READS SQL DATA (reads but doesn't modify), MODIFIES SQL DATA.

9.11 Other Conventions

  • Naming: start procedures with a verb (get_, make_, update_); functions describe what they return (risk_factor_for).
  • Comments: put a header comment on every proc explaining purpose and parameters.
  • Avoid silently failing: validate parameters and SIGNAL errors when needed.
  • Keep them small: one job per procedure.
SECTION 10

Triggers and Events

10.1 Triggers

A trigger is SQL that runs automatically in response to an INSERT, UPDATE, or DELETE on a table. Classic use cases: auditing, keeping derived columns in sync, enforcing complex rules.

USE sql_invoicing;
DELIMITER $$

CREATE TRIGGER payments_after_insert
  AFTER INSERT ON payments
  FOR EACH ROW
BEGIN
  UPDATE invoices
  SET payment_total = payment_total + NEW.amount
  WHERE invoice_id = NEW.invoice_id;
END $$

DELIMITER ;

Now whenever a row is inserted into payments, the corresponding invoice's payment_total updates automatically.

NEW refers to the row being inserted/updated. OLD refers to the row being updated/deleted.

Rule: A trigger on table payments cannot modify payments (the row that fired it). It can modify other tables.

10.2 Viewing Triggers

SHOW TRIGGERS;
SHOW TRIGGERS LIKE 'payments%';

10.3 Dropping Triggers

DROP TRIGGER IF EXISTS payments_after_insert;

10.4 Using Triggers for Auditing

A classic pattern: log every change to a sensitive table.

CREATE TABLE payments_audit (
  client_id      INT NOT NULL,
  date           DATE NOT NULL,
  amount         DECIMAL(9,2) NOT NULL,
  action_type    VARCHAR(50) NOT NULL,
  action_date    DATETIME NOT NULL
);

DELIMITER $$
CREATE TRIGGER payments_after_insert_audit
  AFTER INSERT ON payments
  FOR EACH ROW
BEGIN
  INSERT INTO payments_audit
  VALUES (NEW.client_id, NEW.date, NEW.amount, 'Insert', NOW());
END $$
DELIMITER ;

You'd add similar triggers for UPDATE and DELETE, each writing a row to payments_audit.

Trade-off: triggers are invisible to people reading the application code. Document them well or your teammates will lose hours debugging "magical" data changes.

10.5 Events

An event is a stored task that runs on a schedule — like cron, but inside MySQL.

First, ensure the event scheduler is on:

SHOW VARIABLES LIKE 'event_scheduler';
SET GLOBAL event_scheduler = ON;

Create a recurring event:

DELIMITER $$
CREATE EVENT yearly_delete_stale_audit_rows
ON SCHEDULE
  EVERY 1 YEAR STARTS '2026-01-01' ENDS '2036-01-01'
DO BEGIN
  DELETE FROM payments_audit WHERE action_date < NOW() - INTERVAL 1 YEAR;
END $$
DELIMITER ;

Or a one-off:

ON SCHEDULE AT '2026-12-31 23:59:00'

10.6 Viewing, Dropping, and Altering Events

SHOW EVENTS;
DROP EVENT IF EXISTS yearly_delete_stale_audit_rows;

ALTER EVENT yearly_delete_stale_audit_rows DISABLE;
ALTER EVENT yearly_delete_stale_audit_rows ENABLE;
SECTION 11

Transactions and Concurrency

11.1 Transactions

A transaction is a group of SQL statements that either all succeed or all fail together. The textbook example is a bank transfer: debit account A and credit account B — never just one.

The four properties (ACID): - Atomicity — all-or-nothing. - Consistency — DB is in a valid state before and after. - Isolation — concurrent transactions don't see each other's intermediate state. - Durability — once committed, changes survive crashes.

11.2 Creating Transactions

START TRANSACTION;

INSERT INTO orders (customer_id, order_date, status) VALUES (1, '2026-01-01', 1);
INSERT INTO order_items VALUES (LAST_INSERT_ID(), 1, 2, 2.99);

COMMIT;

If anything goes wrong, ROLLBACK undoes everything since START TRANSACTION:

START TRANSACTION;
INSERT INTO orders ...;
-- something fails or you realize a mistake
ROLLBACK;

MySQL autocommits by default — each statement is its own transaction. The START TRANSACTION … COMMIT block lets you batch.

11.3 Concurrency and Locking

When multiple users access the same data simultaneously, the DB uses locks to prevent corruption. While one transaction is updating row 5, others wait.

Several real problems arise from concurrency:

11.4 Concurrency Problems

  1. Lost updates — Two transactions read & write the same row; the second overwrites the first.
  2. Dirty reads — Transaction A reads data Transaction B wrote but hasn't committed; B rolls back; A used invalid data.
  3. Non-repeating reads — A reads a row twice in one transaction; B updated it in between; A sees different values.
  4. Phantom reads — A reads a set of rows twice; B inserted new rows matching the filter; A sees more rows the second time.

DBs offer isolation levels that trade off correctness for performance.

11.5 Transaction Isolation Levels

From weakest (fastest, least safe) to strongest:

Level Dirty reads Non-repeating Phantom
READ UNCOMMITTED ❌ possible ❌ possible ❌ possible
READ COMMITTED ✓ prevented ❌ possible ❌ possible
REPEATABLE READ ✓ prevented ✓ prevented ❌ possible (in standard); MySQL InnoDB prevents these too
SERIALIZABLE ✓ prevented ✓ prevented ✓ prevented

MySQL defaults to REPEATABLE READ. PostgreSQL defaults to READ COMMITTED.

Set:

SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;            -- next transaction only
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;    -- this session
SET GLOBAL  TRANSACTION ISOLATION LEVEL SERIALIZABLE;    -- everyone

11.6 READ UNCOMMITTED Isolation Level

Allows dirty reads. Almost never the right choice in production. Useful only for non-critical reporting where speed > correctness.

11.7 READ COMMITTED Isolation Level

Reads only see committed data. Prevents dirty reads. Most common default in many DBs (Postgres, Oracle, SQL Server).

11.8 REPEATABLE READ Isolation Level

A repeat of the same SELECT within one transaction returns the same rows with the same values, even if other transactions committed in between. MySQL InnoDB default.

11.9 SERIALIZABLE Isolation Level

Strictest — transactions are scheduled as if they ran one after the other. Eliminates all anomalies, at a real performance cost. Use only for accounting-critical workflows.

11.10 Deadlocks

Two transactions waiting for locks the other holds → frozen forever. The DB detects this and kills one of them with an error like "Deadlock found when trying to get lock; try restarting transaction".

Strategies to minimize: - Acquire locks in a consistent order across all code paths. - Keep transactions short. - Avoid mixing reads and writes on many tables in one transaction. - Catch the deadlock error in your application and retry.

SECTION 12

Data Types

A guide to picking the right type for each column. Choose narrowly: smaller types mean faster indexes, smaller backups, less RAM used.

12.1 Introduction

Categories: - String — for text. - Integer / decimal — for numbers. - Boolean — actually just a tiny int in MySQL. - Enum / Set — limited string choices. - Date / Time. - Blob — for binary data. - JSON — structured data.

12.2 String Types

Type Storage Use when
CHAR(n) exactly n chars (pads) fixed length: state codes ('CA'), country codes
VARCHAR(n) up to n chars variable length: names, addresses
MEDIUMTEXT up to 16 MB JSON blobs, long descriptions
LONGTEXT up to 4 GB books, large documents
TINYTEXT up to 255 bytes rarely used (VARCHAR(255) is usually better)
TEXT up to 64 KB medium prose

Rule of thumb: - IDs/codes with fixed length → CHAR. - Anything else → VARCHAR(n) with n slightly larger than what you expect. - Very large content → TEXT/MEDIUMTEXT.

12.3 Integer Types

Type Bytes Range (signed)
TINYINT 1 -128 .. 127
SMALLINT 2 -32 768 .. 32 767
MEDIUMINT 3 ~ -8M .. 8M
INT 4 ~ -2B .. 2B
BIGINT 8 ~ -9.2 × 10¹⁸ .. 9.2 × 10¹⁸

UNSIGNED doubles the positive range (e.g., TINYINT UNSIGNED → 0..255).

Choose the smallest that fits your data forever. Don't use BIGINT "just in case" — it costs you 4 bytes per row × every row × every index.

12.4 Fixed-point and Floating-point Types

DECIMAL(p, s)   -- p total digits, s digits after the decimal point
NUMERIC(p, s)   -- alias for DECIMAL
FLOAT           -- 4 bytes, approximate
DOUBLE          -- 8 bytes, approximate

Money: use DECIMAL. Floats lose precision (0.1 + 0.2 != 0.3). For scientific data where speed matters and tiny error is fine: float/double.

unit_price DECIMAL(9,2)   -- up to 9,999,999.99

12.5 Boolean Types

BOOLEAN / BOOL are aliases for TINYINT(1). There's no real boolean type in MySQL.

is_published BOOLEAN

Stored as 0 / 1. TRUE and FALSE are aliases.

12.6 Enum and Set Types

ENUM constrains a column to a list of strings:

size ENUM('small','medium','large')

Compact and indexed efficiently — but adding a value requires ALTER TABLE, which on large tables can be slow. For values that change rarely (status codes), fine. For anything that might grow, use a lookup table instead.

SET allows multiple values from a list. Almost never the right choice — use a junction table.

12.7 Date and Time Types

Type Format Range
DATE YYYY-MM-DD 1000-01-01 .. 9999-12-31
TIME HH:MM:SS
DATETIME YYYY-MM-DD HH:MM:SS 1000 .. 9999, 8 bytes
TIMESTAMP YYYY-MM-DD HH:MM:SS 1970 .. 2038, 4 bytes, time-zone aware
YEAR 4-digit year

DATETIME vs TIMESTAMP: TIMESTAMP is smaller and auto-converts to the session's time zone, but is limited to 2038. For records far in the past or future, use DATETIME.

12.8 Blob Types

For binary data: images, files, executables.

Type Max size
TINYBLOB 255 bytes
BLOB 64 KB
MEDIUMBLOB 16 MB
LONGBLOB 4 GB

Advice: don't store large blobs in your DB. Store files in object storage (S3, GCS) and keep just the URL/key in the DB. Your backups and queries will thank you.

12.9 JSON Type

Native JSON column. Validates JSON on insert and gives you query operators:

CREATE TABLE products_json (
  product_id INT PRIMARY KEY,
  properties JSON
);

INSERT INTO products_json VALUES (1, '{"weight":10,"dimensions":[1,2,3],"manufacturer":{"name":"Acme"}}');

-- Read with JSON_EXTRACT or the ->> shortcut
SELECT JSON_EXTRACT(properties, '$.weight')           FROM products_json;
SELECT properties -> '$.weight'                       FROM products_json;
SELECT properties ->> '$.manufacturer.name'           FROM products_json;  -- unquoted string

-- Update JSON
UPDATE products_json
SET properties = JSON_SET(properties, '$.weight', 20, '$.age', 10)
WHERE product_id = 1;

UPDATE products_json
SET properties = JSON_REMOVE(properties, '$.age')
WHERE product_id = 1;

When to use JSON columns: for flexible, semi-structured data where columns would proliferate. Don't use it as a substitute for proper schema design — if you'd want to join, filter, or aggregate on a field, make it a column.

SECTION 13

Designing Databases

This is arguably the most valuable section in the course. Querying you can pick up. Designing well is what separates competent from senior.

13.1 Introduction

A well-designed database makes everything else easier — queries become natural, performance is predictable, bugs are rare. A badly-designed database makes simple tasks hellish.

The design process has stages: 1. Conceptual model — what's the world we're modeling? (entities, relationships) 2. Logical model — what tables and columns? (no DBMS-specific decisions yet) 3. Physical model — actual MySQL DDL with types, indexes, constraints.

13.2 Data Modelling

Four steps for any new system: 1. Understand the business requirements — talk to stakeholders, read existing docs, sketch use cases. Don't model what you don't understand. 2. Build a conceptual model — entities and how they connect, in plain language. 3. Build a logical model — tables, columns, relationships, types abstractly. 4. Build a physical model — the actual DDL for your DBMS.

The first step costs nothing and saves everything.

13.3 Conceptual Models

A high-level sketch of the entities (things) and the relationships between them. No types, no PKs. Just nouns and lines.

Tools: pen and paper, draw.io, Lucidchart, dbdiagram.io.

Example — a blogging app:

[User] ---writes---> [Post] ---has---> [Comment]
                      |
                      tagged with
                      ↓
                    [Tag]

That's it. No technical details — just what exists and how it relates.

13.4 Logical Models

Now add structure: which entity has which attributes, and what type of relationship (one-to-many, many-to-many).

User  (id, email, name)
   ↓ 1 — ∞
Post  (id, user_id, title, body, published_at)
   ↓ 1 — ∞
Comment (id, post_id, user_id, body, created_at)

Post  ∞ — ∞  Tag       (via post_tags join table)
Tag   (id, name)

Still no DBMS-specific concerns.

13.5 Physical Models

The actual MySQL DDL: INT vs BIGINT, VARCHAR(50), indexes, default values, character set.

CREATE TABLE users (
  id     INT PRIMARY KEY AUTO_INCREMENT,
  email  VARCHAR(255) NOT NULL UNIQUE,
  name   VARCHAR(100) NOT NULL
);

13.6 Primary Keys

The column (or combination of columns) that uniquely identifies a row.

CREATE TABLE customers (
  customer_id INT PRIMARY KEY AUTO_INCREMENT,
  ...
);

Rules of thumb: - Always have a primary key. A table without one is hard to update reliably. - Prefer surrogate keys (an artificial id INT AUTO_INCREMENT or UUID) over natural keys (email, SSN). Natural keys can change; surrogates don't. - Keep it narrow (INT or BIGINT) — every index references the PK, so wide PKs slow everything down.

13.7 Foreign Keys

A column that points to a primary key in another table — the mechanism of relational design.

CREATE TABLE orders (
  order_id    INT PRIMARY KEY AUTO_INCREMENT,
  customer_id INT NOT NULL,
  ...,
  FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);

Now you can't insert an order with a non-existent customer_id. The DB enforces referential integrity.

13.8 Foreign Key Constraints

What happens when the parent row changes? Configure with ON DELETE / ON UPDATE:

  • RESTRICT (or NO ACTION) — block the operation if dependents exist.
  • CASCADE — propagate. Delete parent → delete children. Update PK → update FK.
  • SET NULL — set the FK to NULL.
  • SET DEFAULT — use the column's default.
FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
  ON UPDATE CASCADE
  ON DELETE RESTRICT

Common choices: - ON UPDATE CASCADE is usually right (and rarely matters with surrogate keys). - ON DELETE RESTRICT is safest by default — force the application to handle child rows explicitly.

13.9 Normalization

A set of formal rules for organizing tables to eliminate redundancy and avoid update anomalies. We'll cover 1NF–3NF; beyond is academic.

13.10 1NF — First Normal Form

Each cell holds a single value. No repeating groups.

-- BAD: tags packed into one column
posts(id, title, tags)
1,    "Hello", "sql,beginner,tutorial"

-- GOOD: separate row per tag
posts(id, title)
post_tags(post_id, tag_id)
tags(id, name)

If you ever find yourself splitting a string with LIKE '%tag%', your data isn't in 1NF.

13.11 Link Tables (Junction Tables)

For many-to-many: a third table holds the relationship.

CREATE TABLE post_tags (
  post_id INT NOT NULL,
  tag_id  INT NOT NULL,
  PRIMARY KEY (post_id, tag_id),       -- composite PK
  FOREIGN KEY (post_id) REFERENCES posts(id),
  FOREIGN KEY (tag_id)  REFERENCES tags(id)
);

You can also put extra columns on the junction table itself (e.g., created_at of the tag assignment).

13.12 2NF — Second Normal Form

No partial dependencies. Every non-key column must depend on the whole primary key, not just part of it. Only relevant when you have a composite PK.

-- BAD (PK = order_id + product_id, but customer_name depends only on order_id)
order_items(order_id, product_id, quantity, customer_name)

-- GOOD
orders(order_id, customer_id)
customers(customer_id, name)
order_items(order_id, product_id, quantity)

13.13 3NF — Third Normal Form

No transitive dependencies. Non-key columns must depend on the PK and nothing else.

-- BAD (zip depends on city, not directly on customer)
customers(id, name, city, zip)

-- GOOD
customers(id, name, city_id)
cities(id, name, zip)

In practice, 3NF is the standard target. Beyond that (BCNF, 4NF, 5NF) is rarely worth the cost.

13.14 My Pragmatic Advice (Mosh)

  • Don't obsess over normalization. Get to 3NF, then make pragmatic exceptions when measured performance demands it.
  • Talk to the business. Models are theory; usage is reality.
  • Iterate. Your first schema won't be perfect. Use migrations.

13.15 Don't Model the Universe

Tempting trap: you start modeling a video rental app and end up with a Country table, Currency table, Language table, TimeZone table. Stop. Only model what your app needs right now. You can always add later.

13.16 Forward Engineering a Model

In MySQL Workbench (Database → Forward Engineer), the tool generates the CREATE TABLE DDL from your diagram. Useful for documenting and re-deploying schemas.

13.17 Synchronizing a Model with a Database

Workbench can compare your model file with the live DB and generate migration scripts. Handy on a small team; weaker than dedicated migration tools (Flyway, Liquibase, Alembic, Rails migrations).

13.18 Reverse Engineering a Database

Point Workbench at an existing DB → it draws the diagram for you. Great way to learn an inherited codebase.

13.25 Creating and Dropping Databases

CREATE DATABASE sql_store2;
CREATE DATABASE IF NOT EXISTS sql_store2;
DROP DATABASE IF EXISTS sql_store2;

IF EXISTS / IF NOT EXISTS make scripts idempotent.

13.26 Creating Tables

USE sql_store2;

CREATE TABLE IF NOT EXISTS customers (
  id          INT          NOT NULL PRIMARY KEY AUTO_INCREMENT,
  first_name  VARCHAR(50)  NOT NULL,
  points      INT          NOT NULL DEFAULT 0,
  email       VARCHAR(255) NOT NULL UNIQUE
);

UNIQUE creates a unique constraint — duplicate emails are rejected.

13.27 Altering Tables

ALTER TABLE customers
  ADD    last_name  VARCHAR(50) NOT NULL AFTER first_name,
  ADD    city       VARCHAR(50) NOT NULL,
  MODIFY first_name VARCHAR(55) DEFAULT '',
  DROP   points;

Beware: ALTER TABLE on a large table can lock it for a long time. In production, use online schema-change tools (pt-online-schema-change, gh-ost).

CREATE TABLE orders (
  id          INT PRIMARY KEY AUTO_INCREMENT,
  customer_id INT NOT NULL,
  FOREIGN KEY fk_orders_customers (customer_id)
    REFERENCES customers(id)
    ON UPDATE CASCADE
    ON DELETE NO ACTION
);

13.29 Altering Primary and Foreign Key Constraints

ALTER TABLE orders
  DROP PRIMARY KEY,
  ADD PRIMARY KEY (id),
  DROP FOREIGN KEY fk_orders_customers,
  ADD FOREIGN KEY fk_orders_customers (customer_id)
    REFERENCES customers(id)
    ON UPDATE CASCADE
    ON DELETE NO ACTION;

13.30 Character Sets and Collations

A character set is the mapping from characters to bytes. A collation is the sort/comparison rule (e.g., is 'a' = 'A'?).

-- Recommended modern default:
CREATE DATABASE sql_store2
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;

utf8mb4 supports the full Unicode range including emoji 🎉. (MySQL's older utf8 is actually a 3-byte subset — avoid.) _ci = case-insensitive.

13.31 Storage Engines

MySQL supports pluggable storage engines. The two relevant ones: - InnoDB — the default. Supports transactions, foreign keys, row-level locking. Use this. - MyISAM — legacy. No transactions or FKs. Don't use for new tables.

ALTER TABLE customers ENGINE = InnoDB;

SECTION 14

Indexing for High Performance

14.1 Introduction

An index is a separate data structure (a B-tree) the database uses to find rows quickly — like the index at the back of a book.

Trade-offs: - Reads get faster — sometimes 1000×. - Writes get slower — every INSERT/UPDATE/DELETE has to update every index. - Disk usage grows — indexes can be larger than the table itself.

Rule: index for actual query patterns, not in advance.

14.2 Indexes

The PK is automatically indexed. Unique constraints create unique indexes. You add extra indexes on columns frequently used in WHERE, JOIN ON, and ORDER BY.

USE sql_store;

-- Suppose this is slow:
SELECT * FROM customers WHERE state = 'CA';

EXPLAIN SELECT * FROM customers WHERE state = 'CA';
-- type: ALL  → full table scan, bad

EXPLAIN shows the query plan. type: ALL is a full scan; type: ref or range usually means an index was used.

14.3 Creating Indexes

CREATE INDEX idx_state ON customers (state);

EXPLAIN SELECT * FROM customers WHERE state = 'CA';
-- type: ref, key: idx_state

14.4 Viewing Indexes

SHOW INDEXES IN customers;

Cardinality shows the index's selectivity — roughly how many distinct values it has. High cardinality = useful index. Low cardinality (e.g., 2 distinct values across 1M rows) = probably not worth it.

14.5 Prefix Indexes

For long string columns, index only the first N characters:

CREATE INDEX idx_lastname ON customers (last_name(20));

For CHAR/VARCHAR, MySQL requires prefix length on TEXT/BLOB. Even on VARCHAR, prefixing saves space without losing much selectivity. The right N is the smallest number that captures most of the uniqueness.

14.6 Full-text Indexes

For searching natural-language text. Designed for "find articles mentioning these words" type queries.

CREATE FULLTEXT INDEX idx_title_body ON posts (title, body);

SELECT *
FROM posts
WHERE MATCH(title, body) AGAINST ('react redux');

Boolean mode supports more operators:

SELECT *
FROM posts
WHERE MATCH(title, body) AGAINST ('react -redux +hooks' IN BOOLEAN MODE);

For serious full-text needs, dedicated engines (Elasticsearch, Meilisearch, Postgres tsvector) outperform MySQL.

14.7 Composite Indexes

An index on multiple columns. Often the answer when single-column indexes don't help.

CREATE INDEX idx_state_points ON customers (state, points);

-- This query uses the composite index for both predicates:
EXPLAIN SELECT * FROM customers WHERE state = 'CA' AND points > 1000;

MySQL practical limit is 16 columns per index, but real-world composite indexes are rarely more than 3–4.

14.8 Order of Columns in Composite Indexes

The order matters enormously. Rules: 1. Most-used columns first — columns used in many queries should come first. 2. Highest cardinality first — within equally-used columns, prefer the more selective one.

A composite index on (A, B, C) can be used for filters on: - A alone - A AND B - A AND B AND C

But not for B alone or B AND C — the index is a phone book sorted by last name then first name; you can't find someone by first name alone.

14.9 When Indexes Are Ignored

Common pitfalls — the optimizer skips your index when:

  • Functions on indexed columns: sql -- BAD - index ignored WHERE YEAR(order_date) = 2019 -- GOOD - index used WHERE order_date >= '2019-01-01' AND order_date < '2020-01-01'
  • Leading wildcards in LIKE: sql WHERE name LIKE '%john%' -- index can't help WHERE name LIKE 'john%' -- index can help
  • OR across different columns: add a covering index or rewrite as UNION.
  • Implicit type conversion: WHERE id = '5' on an INT column may skip the index.

14.10 Using Indexes for Sorting

If you ORDER BY a column with no index, MySQL has to sort the result — slow on big tables. With an index, results are already in order.

CREATE INDEX idx_points ON customers (points);

EXPLAIN SELECT * FROM customers ORDER BY points;
-- No "filesort" in Extra → using the index

Composite indexes also help sorting if the ORDER BY matches the leading columns.

14.11 Covering Indexes

If an index contains every column the query needs, MySQL can answer the query from the index alone — no table lookups. This is the holy grail of query performance.

CREATE INDEX idx_state_points ON customers (state, points);

EXPLAIN SELECT state, points FROM customers WHERE state = 'CA';
-- Extra: Using index   ← covering index in action

When EXPLAIN shows Using index, you've achieved a covering index.

14.12 Index Maintenance

  • Don't add indexes "just in case" — measure first.
  • Drop unused indexes — they cost on every write. sql -- In production, use the slow query log / performance_schema to find unused indexes
  • Watch for duplicates — an index on (A) is redundant if you also have (A, B).
  • Rebuild after big data churn — OPTIMIZE TABLE customers; (MySQL handles this lazily in InnoDB, so usually not needed).

14.13 Performance Best Practices

A short cheat sheet: 1. Smaller is faster. Smaller rows, smaller indexes, smaller backups. 2. Index columns used in WHERE, JOIN, ORDER BY. Not all columns. 3. Avoid SELECT *. Fetch only what you need — helps covering indexes. 4. Use EXPLAIN on slow queries. Look for type: ALL (bad), Using filesort (often bad). 5. Watch the slow query log. Set long_query_time = 1 and you'll learn what's actually slow. 6. Don't over-normalize for read-heavy workloads. A controlled denormalization (cached total_orders on customer) sometimes wins. 7. Add indexes incrementally, measuring impact each time.

SECTION 15

Securing Databases

15.1 Introduction

By default MySQL has one user: root. Don't use root in your applications. Create dedicated users with the minimum privileges they need.

The pattern is principle of least privilege: every user gets exactly what they need, and not one grant more.

15.2 Creating a User

-- Allow from any host
CREATE USER john IDENTIFIED BY 'StrongPassword!23';

-- Restrict by host
CREATE USER john@localhost     IDENTIFIED BY 'StrongPassword!23';
CREATE USER john@127.0.0.1     IDENTIFIED BY 'StrongPassword!23';
CREATE USER 'john'@'10.0.0.%'  IDENTIFIED BY 'StrongPassword!23';   -- IP range
CREATE USER 'john'@'%.example.com' IDENTIFIED BY 'StrongPassword!23'; -- domain

Restricting by host is the first line of defense. A user that can only connect from 10.0.0.0/24 is useless to attackers outside that range.

15.3 Viewing Users

SELECT user, host FROM mysql.user;

15.4 Dropping Users

DROP USER john@localhost;

15.5 Changing Passwords

SET PASSWORD FOR john@localhost = 'NewStrongerPassword!42';

-- Self-service
SET PASSWORD = 'MyOwnNewPassword!42';

15.6 Granting Privileges

For an application user that needs read/write on one DB:

GRANT SELECT, INSERT, UPDATE, DELETE, EXECUTE
ON sql_store.*
TO moshapp@localhost;

For an admin user across all DBs:

GRANT ALL ON *.* TO admin@localhost;

Granular grants:

GRANT SELECT ON sql_store.customers TO reportbot@localhost;
GRANT SELECT (first_name, last_name) ON sql_store.customers TO maskedbot@localhost;

Common privileges:

Privilege Allows
SELECT read
INSERT insert rows
UPDATE update rows
DELETE delete rows
EXECUTE call stored procedures
CREATE create tables / dbs
ALTER change schema
DROP delete tables / dbs
INDEX create / drop indexes
ALL every privilege

15.7 Viewing Privileges

SHOW GRANTS;                          -- for the current user
SHOW GRANTS FOR john@localhost;       -- for a specific user
SHOW GRANTS FOR root@localhost;

15.8 Revoking Privileges

If you grant too much, revoke specifically:

REVOKE CREATE VIEW ON sql_store.* FROM moshapp@localhost;

To start over:

REVOKE ALL PRIVILEGES, GRANT OPTION FROM moshapp@localhost;

Security Best Practices

  • Never connect as root from an application. Create a dedicated user per application.
  • Use strong passwords, store them in a secrets manager (Vault, AWS Secrets Manager, Doppler), never in code or .env committed to git.
  • Restrict by host whenever possible.
  • Grant least privilege. Most apps don't need DROP or CREATE. They probably don't need ALTER. They likely need SELECT, INSERT, UPDATE, DELETE on specific tables — that's it.
  • Don't expose MySQL to the public internet. Bind to localhost or to a private network only.
  • Always use parameterized queries in your application code to prevent SQL injection. Never concatenate user input into a SQL string.
  • Encrypt at rest and in transit. Use TLS for connections; use disk encryption on DB servers.
  • Back up regularly, test your restores. A backup you've never restored is a wish, not a backup.
SECTION 16

Window Functions & CTEs

This is the single biggest jump from intermediate to advanced SQL — and the topic interviewers reach for most. Everything up to now either returned raw rows or collapsed them with GROUP BY. Window functions let you do both at once: compute an aggregate and keep every individual row.

16.1 The Problem: Aggregates That Erase Your Rows

Say you want each invoice shown next to its client's total. GROUP BY can give you the total — but it throws away the individual invoices:

-- We want each order line AND the total across all lines.
-- GROUP BY collapses the rows — we lose the detail:
SELECT client_id, SUM(invoice_total) AS client_total
FROM invoices
GROUP BY client_id;

You get one row per client. The individual invoices are gone. To keep them, you'd normally need a self-join or a correlated subquery. Window functions do it in one clean line.

16.2 Your First Window Function: OVER()

Add OVER () to an aggregate and it becomes a window function. The aggregate is computed over a "window" of rows, but the original rows survive:

-- A window function keeps every row and adds the aggregate alongside it.
SELECT
  invoice_id,
  client_id,
  invoice_total,
  SUM(invoice_total) OVER () AS grand_total
FROM invoices;

Every invoice row now carries the grand_total beside it. Empty OVER () means "the window is the entire result set."

16.3 PARTITION BY

PARTITION BY is like GROUP BY for the window — it restarts the calculation for each group, without collapsing rows:

SELECT
  invoice_id,
  client_id,
  invoice_total,
  SUM(invoice_total) OVER (PARTITION BY client_id) AS client_total,
  invoice_total / SUM(invoice_total) OVER (PARTITION BY client_id) * 100 AS pct_of_client
FROM invoices;

Now each row shows its client's total and what percentage of that client's spend this one invoice represents. Try writing that without a window function — it's painful.

Mental model: PARTITION BY slices the rows into buckets; the function runs independently inside each bucket; the rows all come back out.

16.4 Ranking: ROW_NUMBER, RANK, DENSE_RANK

These need an ORDER BY inside the OVER() — they number rows by that order:

SELECT
  client_id,
  invoice_total,
  ROW_NUMBER() OVER (PARTITION BY client_id ORDER BY invoice_total DESC) AS rn,
  RANK()       OVER (PARTITION BY client_id ORDER BY invoice_total DESC) AS rnk,
  DENSE_RANK() OVER (PARTITION BY client_id ORDER BY invoice_total DESC) AS dense_rnk
FROM invoices;

The difference shows up on ties (equal values):

  • ROW_NUMBER — always distinct: 1, 2, 3, 4. Ties broken arbitrarily.
  • RANK — ties share a rank, then it skips: 1, 2, 2, 4.
  • DENSE_RANK — ties share a rank, no gaps: 1, 2, 2, 3.

The killer use case — "top N per group" — needs a subquery wrapper, because you can't use a window function directly in WHERE (it's computed after filtering):

-- "Top 3 invoices per client" — the classic interview question.
SELECT * FROM (
  SELECT
    invoice_id, client_id, invoice_total,
    ROW_NUMBER() OVER (PARTITION BY client_id ORDER BY invoice_total DESC) AS rn
  FROM invoices
) ranked
WHERE rn <= 3;

16.5 LAG and LEAD: Peeking at Neighbors

LAG reaches back to a previous row; LEAD reaches forward. Perfect for period-over-period comparisons:

-- Compare each invoice to the client's PREVIOUS one.
SELECT
  client_id,
  invoice_date,
  invoice_total,
  LAG(invoice_total)  OVER (PARTITION BY client_id ORDER BY invoice_date) AS prev_total,
  LEAD(invoice_total) OVER (PARTITION BY client_id ORDER BY invoice_date) AS next_total,
  invoice_total - LAG(invoice_total) OVER (PARTITION BY client_id ORDER BY invoice_date) AS change
FROM invoices;

No self-join, no correlated subquery — just "the row before this one, in this order."

16.6 Running Totals & Moving Averages (the frame clause)

When you add ORDER BY to an aggregate window, you can define a frame — which rows relative to the current one to include:

-- Running total (cumulative sum) ordered by date.
SELECT
  invoice_date,
  invoice_total,
  SUM(invoice_total) OVER (
    ORDER BY invoice_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS running_total
FROM invoices;

-- 7-row moving average (this row + 6 before it).
SELECT
  invoice_date,
  AVG(invoice_total) OVER (
    ORDER BY invoice_date
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS moving_avg_7
FROM invoices;

ROWS BETWEEN ... PRECEDING AND CURRENT ROW is the frame. This is how dashboards compute cumulative revenue and smoothed trend lines.

16.7 Common Table Expressions (WITH)

A CTE is a named, temporary result set that exists only for the query. It replaces ugly nested subqueries with something you read top-to-bottom, like a paragraph:

-- The same query rewritten with a CTE — read it top to bottom.
WITH client_totals AS (
  SELECT client_id, SUM(invoice_total) AS total
  FROM invoices
  GROUP BY client_id
)
SELECT c.name, ct.total
FROM client_totals ct
JOIN clients c USING (client_id)
WHERE ct.total > 500;

Compare that to burying the aggregation in a subquery inside the FROM. Same result; far more readable. CTEs don't (by themselves) make queries faster — their value is clarity and reuse.

16.8 Chaining CTEs

You can define several CTEs in one WITH, and each may reference the previous ones. This lets you build a pipeline of transformations:

-- Chain multiple CTEs — each can reference the ones above it.
WITH monthly AS (
  SELECT
    DATE_FORMAT(invoice_date, '%Y-%m-01') AS month,
    SUM(invoice_total) AS revenue
  FROM invoices
  GROUP BY month
),
with_growth AS (
  SELECT
    month,
    revenue,
    LAG(revenue) OVER (ORDER BY month) AS prev_revenue
  FROM monthly
)
SELECT
  month,
  revenue,
  ROUND((revenue - prev_revenue) / prev_revenue * 100, 1) AS growth_pct
FROM with_growth;

Read it as a data pipeline: raw invoices → monthly buckets → add previous month → compute growth. Each step is named and testable on its own.

16.9 Recursive CTEs (hierarchies & sequences)

A recursive CTE references itself. It has an anchor (the starting row) and a recursive step that keeps running until it produces no new rows. This is how you query trees — org charts, threaded comments, category trees, bill-of-materials:

-- Recursive CTE: walk an employee -> manager hierarchy.
WITH RECURSIVE org_chart AS (
  -- Anchor: the top boss (no manager)
  SELECT employee_id, first_name, reports_to, 1 AS depth
  FROM employees
  WHERE reports_to IS NULL

  UNION ALL

  -- Recursive step: everyone who reports to someone already in the tree
  SELECT e.employee_id, e.first_name, e.reports_to, oc.depth + 1
  FROM employees e
  JOIN org_chart oc ON e.reports_to = oc.employee_id
)
SELECT depth, first_name FROM org_chart ORDER BY depth, first_name;

The anchor picks the root. Each pass finds the direct reports of everyone found so far, incrementing depth. It stops when nobody new is added. They're also handy for generating rows:

-- Recursive CTEs also generate sequences (no source table needed).
WITH RECURSIVE numbers AS (
  SELECT 1 AS n
  UNION ALL
  SELECT n + 1 FROM numbers WHERE n < 10
)
SELECT n FROM numbers;   -- 1..10

Guardrail: a recursive CTE with no terminating condition runs forever. MySQL caps it at cte_max_recursion_depth (default 1000) and errors out — but write your WHERE stop condition deliberately, don't rely on the cap.

SECTION 17

Query Diagnostics & Deadlocks

Section 14 taught you to add indexes. This section teaches you what to do when the query is still slow despite them — and how to survive the concurrency problems that show up under real traffic. This is the difference between "I wrote an index" and "I can diagnose a production incident."

17.1 Why an Indexed Query Can Still Be Slow

An index only helps if the optimizer actually uses it. It often can't — because you wrapped the column in a function, used a leading wildcard, compared mismatched types, or the index simply doesn't cover the columns you filter on. You can't guess at this. You have to ask the database what it's doing. That tool is EXPLAIN.

17.2 EXPLAIN: Reading the Plan

Prefix any query with EXPLAIN and MySQL shows the plan it would use — without running it:

-- Put EXPLAIN in front of ANY SELECT/UPDATE/DELETE to see the plan.
EXPLAIN
SELECT * FROM invoices WHERE client_id = 3;

The columns that matter most:

  • type — the access method (see the ladder below). The most important column.
  • key — which index it chose. NULL here is a warning sign.
  • rows — estimated rows examined. Big numbers on a filtered query are suspicious.
  • Extra — where the bodies are buried: Using filesort, Using temporary, Using where.

17.3 The Access-Type Ladder

The type column, from best to worst — memorize this ordering:

typeMeaningVerdict
const / eq_refOne row via a unique key✅ Ideal
refRows matching a non-unique index value✅ Good
rangeIndex range scan (BETWEEN, >, IN)🟡 Usually fine
indexFull scan of the index🟠 Often slow
ALLFull table scan — reads every row🔴 Red flag on big tables

On MySQL 8, prefer the tree format — it reads like a nested plan and shows cost estimates:

-- Force the modern, readable tree format (MySQL 8.0+):
EXPLAIN FORMAT=TREE
SELECT i.invoice_id, c.name
FROM invoices i
JOIN clients c USING (client_id)
WHERE i.invoice_total > 100;

17.4 EXPLAIN ANALYZE: Estimated vs Actual

EXPLAIN shows the plan. EXPLAIN ANALYZE actually executes the query and reports real timings and real row counts next to the estimates:

-- EXPLAIN ANALYZE actually RUNS the query and reports real timings.
EXPLAIN ANALYZE
SELECT i.invoice_id, c.name
FROM invoices i
JOIN clients c USING (client_id)
WHERE i.invoice_total > 100;

The gold is in comparing estimated rows to actual rows. A huge gap means the optimizer's statistics are stale (ANALYZE TABLE refreshes them) and it may be choosing a bad plan based on bad guesses.

Careful: EXPLAIN ANALYZE really runs the statement. Never point it at an UPDATE/DELETE in production unless you want those changes to happen.

17.5 Three Red Flags in Extra

  • Using filesort — MySQL must sort rows separately because no index provides the order. Fine for small results; expensive when sorting millions. Fix: an index matching your ORDER BY.
  • Using temporary — it built a temp table (common with GROUP BY + ORDER BY on different columns, or DISTINCT). If it spills to disk, it's slow.
  • type: ALL with a big rows — the full table scan. The most common cause of a slow query.

The typical fix is a targeted index that turns ALL into ref:

-- The fix for a full table scan is usually the right index.
-- Before: type = ALL (scans every row)
EXPLAIN SELECT * FROM invoices WHERE client_id = 3;

CREATE INDEX idx_invoices_client ON invoices (client_id);

-- After: type = ref (jumps straight to matching rows via the index)
EXPLAIN SELECT * FROM invoices WHERE client_id = 3;

17.6 Deadlocks in Practice

You met isolation levels in Section 11. Here's the failure they cause. A deadlock is two transactions each holding a lock the other needs — a standoff neither can break:

-- Session A                          -- Session B
START TRANSACTION;                    START TRANSACTION;
UPDATE accounts                       UPDATE accounts
  SET balance = balance - 100           SET balance = balance - 50
  WHERE id = 1;                         WHERE id = 2;

-- now each waits for the other's row:
UPDATE accounts                       UPDATE accounts
  SET balance = balance + 100           SET balance = balance + 50
  WHERE id = 2;   -- blocks            WHERE id = 1;   -- DEADLOCK

Session A holds row 1 and wants row 2; Session B holds row 2 and wants row 1. Neither can proceed. InnoDB detects the cycle, picks a victim, and rolls it back with error 1213. Deadlocks are not bugs you eliminate — they're events you handle.

17.7 Reading the Deadlock Log

When a deadlock happens, InnoDB records exactly what went wrong. This is your incident report:

-- After a deadlock, InnoDB logs the details. Read the LATEST one:
SHOW ENGINE INNODB STATUS;
-- Look for the section headed:  LATEST DETECTED DEADLOCK
-- It names both transactions, the SQL each was running,
-- the locks each held, and which one InnoDB rolled back (the "victim").

The log tells you the two statements involved. The fix is almost always the same principle: always acquire locks in a consistent order. If every transaction touches rows in ascending id order, the cycle above can't form.

17.8 Pessimistic vs Optimistic Locking

Pessimistic — assume conflicts are likely, lock the row before you touch it with SELECT ... FOR UPDATE. Safe, but holds locks and reduces concurrency:

-- Pessimistic: lock the row up front, assume conflict is likely.
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;  -- locks the row
-- ... do work; nobody else can modify row 1 until you COMMIT ...
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;

Optimistic — assume conflicts are rare, take no lock, and detect collisions at write time using a version column. Higher throughput; you must handle the "someone beat me" case:

-- Optimistic: no lock; detect conflict at write time with a version column.
-- Read:
SELECT balance, version FROM accounts WHERE id = 1;   -- say version = 7

-- Write only if nobody else changed it in the meantime:
UPDATE accounts
SET balance = balance - 100, version = version + 1
WHERE id = 1 AND version = 7;

-- If 0 rows were affected, someone else won the race -> re-read and retry.

Rule of thumb: high contention on the same rows → pessimistic. Low contention, lots of readers → optimistic.

17.9 Retry Logic: The Missing Half

Because deadlocks and lock-wait timeouts will happen, correct application code wraps transactions in a retry with backoff. A transaction that isn't retryable isn't production-ready:

# Application-side retry loop — deadlocks are normal; retry them.
import time
from pymysql.err import OperationalError

def run_with_retry(fn, attempts=3):
    for i in range(attempts):
        try:
            return fn()                      # your transaction
        except OperationalError as e:
            # 1213 = deadlock, 1205 = lock wait timeout
            if e.args[0] in (1213, 1205) and i < attempts - 1:
                time.sleep(0.05 * (2 ** i))  # exponential backoff
                continue
            raise

SECTION 18

Application Integration

Almost nobody types SQL into a terminal all day. They write application code that talks to the database through a driver or an ORM. That layer hides the SQL — and hidden SQL is where the two most common real-world performance disasters live. Knowing the queries isn't enough; you have to know how your application issues them.

18.1 The Bridge: Drivers and ORMs

Your code reaches the database through a driver (a low-level client that speaks MySQL's protocol) usually wrapped by an ORM (Object-Relational Mapper) that turns objects into SQL for you. ORMs are wonderful for productivity and dangerous for performance, because a single line of innocent-looking code can fire hundreds of queries. The rest of this section is about seeing through the abstraction.

18.2 The N+1 Query Problem

The single most common performance bug in database-backed apps. You fetch a list, then loop over it fetching related data one item at a time:

# The N+1 problem, hiding in plain sight. Looks innocent:
authors = db.query("SELECT * FROM authors")          # 1 query
for author in authors:
    books = db.query(                                 # + N queries!
        "SELECT * FROM books WHERE author_id = %s",
        author.id
    )
    print(author.name, len(books))
# 1 author query + 1 book query PER author = N+1 round trips.
# 100 authors -> 101 queries. Each with network latency.

Each iteration is a separate round trip over the network. It works fine with 10 test rows and falls over with 10,000 production rows — the classic "it was fast on my machine" bug.

18.3 Fixing N+1

Three ways, in rough order of preference. First, batch the child lookups into one query with IN:

# Fix 1 — one extra query with IN(), stitch in memory.
authors = db.query("SELECT * FROM authors")
ids = [a.id for a in authors]
books = db.query(
    "SELECT * FROM books WHERE author_id IN %s", (ids,)
)                                                     # just 2 queries total
by_author = group_by(books, key=lambda b: b.author_id)
for author in authors:
    print(author.name, len(by_author.get(author.id, [])))

Better still, push the work to the database with a join — one round trip, and the engine does what it's good at:

-- Fix 2 — let the database do the join. One round trip.
SELECT a.author_id, a.name, COUNT(b.book_id) AS book_count
FROM authors a
LEFT JOIN books b ON b.author_id = a.author_id
GROUP BY a.author_id, a.name;

And if you're using an ORM, learn its eager-loading feature — it's the built-in cure:

# Fix 3 — most ORMs have "eager loading" to batch the child query for you.
# SQLAlchemy:
authors = session.query(Author).options(selectinload(Author.books)).all()
# Django:
authors = Author.objects.prefetch_related("books")
# Rails / ActiveRecord:
#   authors = Author.includes(:books)
# All collapse N+1 into 2 queries under the hood.

How to catch it: log your queries in development, or use your ORM's query counter. If one page or endpoint issues a number of queries that grows with the data, you have an N+1.

18.4 Connection Pooling

Opening a database connection is not free. Each one costs a TCP handshake, authentication (and TLS negotiation), and session setup. Doing that per request wastes time and quickly exhausts the server's connection limit. A connection pool keeps a handful of connections open and hands them out:

# Why connections are expensive: every new one pays for
#   TCP handshake  ->  auth / TLS negotiation  ->  session setup
# Opening one per request, under load, exhausts the server's
# max_connections and adds latency to every single call.

# A pool keeps a set of warm, reusable connections:
from sqlalchemy import create_engine

engine = create_engine(
    "mysql+pymysql://app:pw@db.internal/shop",
    pool_size=10,        # steady-state connections kept open
    max_overflow=5,      # extra burst connections when busy
    pool_timeout=30,     # wait this long for a free connection
    pool_recycle=1800,   # recycle conns older than 30 min
    pool_pre_ping=True,  # test a conn before handing it out
)
# Borrow -> use -> return to the pool (not closed):
with engine.connect() as conn:
    conn.execute(text("SELECT 1"))

Code borrows a warm connection, uses it, and returns it to the pool — it's never actually closed. This turns a per-request cost into a one-time startup cost.

18.5 Sizing the Pool

The database has a hard limit on concurrent connections, and every app instance's pool draws from that same shared budget:

-- The database has a hard ceiling. Every app instance's pool shares it.
SHOW VARIABLES LIKE 'max_connections';   -- e.g. 151
SHOW STATUS LIKE 'Threads_connected';    -- currently in use

The trap: multiply your pools out and you can blow past the ceiling, causing "Too many connections" errors that look like a database outage:

# Sizing rule of thumb:
#   total_app_connections = instances * pool_size (+ overflow)
#   MUST stay comfortably under the DB's max_connections.
#
# 8 app servers * pool_size 20 = 160 > max_connections 151  -> outages!
# Bigger pools are NOT better: past the DB's core/disk limit,
# more connections just means more contention and context-switching.
# Start small (10-20 per instance) and measure.

Counter-intuitive but true: a bigger pool is usually worse once you exceed the database's real capacity (CPU cores, disk throughput). Excess connections just queue and contend. Small pools + short transactions beat big pools almost every time.

18.6 Transactions From Application Code

Everything you learned about transactions in Section 11 applies here — but now your app owns the COMMIT and ROLLBACK. Most libraries give you a context manager that does the right thing:

# Transactions from app code: commit on success, roll back on error,
# always release the connection back to the pool.
with engine.begin() as conn:          # begins a txn; commits on exit,
    conn.execute(text(               # rolls back automatically on exception
        "UPDATE accounts SET balance = balance - :amt WHERE id = :id"),
        {"amt": 100, "id": 1})
    conn.execute(text(
        "UPDATE accounts SET balance = balance + :amt WHERE id = :id"),
        {"amt": 100, "id": 2})
# Keep transactions SHORT. A long-open txn holds locks and pins a
# pooled connection, starving everyone else.

Two rules that prevent most production incidents: keep transactions short (they hold locks and pin a pooled connection), and never do slow work — an HTTP call, a big file read — inside an open transaction.

SECTION 19

Migrations & Zero-Downtime Deployments

In production, nobody opens a GUI and clicks "add column," and nobody pastes CREATE TABLE into a console by hand. Schema changes are code — versioned, reviewed, and applied automatically. And on a table with millions of rows, the wrong change locks your database and takes the whole app offline. This section is how professionals evolve a live schema without an outage.

19.1 Schema Changes Are Code

Every schema change lives in a migration: a small, ordered, version-controlled file. A migration tool tracks which have already run and applies the rest, so your laptop, CI, staging, and production all reach the identical schema by replaying the same history:

# Migrations are version-controlled SQL files, applied in order.
# A typical Flyway-style project:
migrations/
  V1__create_clients.sql
  V2__create_invoices.sql
  V3__add_email_to_clients.sql
  V4__add_index_on_invoice_date.sql
# Each file runs exactly once; the tool records which have run
# in a schema-history table, so every environment converges to
# the same schema from the same ordered history.

The common tools: Flyway and Liquibase (JVM world, language-agnostic), Alembic (Python/SQLAlchemy), Prisma Migrate and framework tools like Rails and Django migrations. They differ in syntax, not in concept.

19.2 Anatomy of a Migration

At its simplest a migration is just a numbered SQL file that runs once:

-- V3__add_email_to_clients.sql
ALTER TABLE clients
  ADD COLUMN email VARCHAR(255) NULL;

Better tools pair every change with a way to reverse it — an up (apply) and a down (revert) — so a failed deploy can roll the schema back cleanly:

# Framework tools generate up/down (apply/revert) pairs.
# Alembic (Python):
def upgrade():
    op.add_column("clients", sa.Column("email", sa.String(255)))

def downgrade():
    op.drop_column("clients", "email")
# 'up' moves the schema forward; 'down' rolls it back if a deploy fails.

Rules of the road: migrations are immutable once merged (never edit a migration that has run somewhere — write a new one), and they run in strict order. Treat them like git history: append-only.

19.3 The Danger: Locking a Big Table

Here's the trap that causes real outages. Some ALTER statements rebuild the entire table or block writes for their whole duration. On a small table that's milliseconds. On 50 million rows it can be minutes of frozen application:

-- DANGER: on a large InnoDB table, some ALTERs lock or rebuild the table,
-- blocking writes (or reads) for the whole operation. On 50M rows that
-- can be minutes of downtime. NEVER assume an ALTER is instant.
ALTER TABLE invoices ADD COLUMN status VARCHAR(20) NOT NULL DEFAULT 'open';
-- ^ Adding a NOT NULL column with a default historically rewrote every row.

Modern MySQL 8 does many ALTERs as online DDL (allowing concurrent reads and writes), but not all — and you must never assume. The safe pattern is to design changes that avoid a blocking rebuild entirely.

19.4 The Expand-Contract Pattern

The core technique for zero-downtime schema change. Instead of one risky change, you make several small safe ones across multiple deploys, so the old and new schema coexist while the app transitions. Splitting a name column, or here, migrating to a new column shape:

Expand — add the new structure without removing the old. Adding a NULLable column is fast and non-blocking:

-- Step 1 (EXPAND): add the new column as NULLable. Fast, non-blocking.
ALTER TABLE users ADD COLUMN full_name VARCHAR(255) NULL;

-- Deploy app code that WRITES both old and new columns.

Backfill — populate the new column in small batches so you never lock the whole table at once:

-- Step 2 (BACKFILL): copy old data into the new column in small batches,
-- so you never lock millions of rows at once.
UPDATE users
SET full_name = CONCAT(first_name, ' ', last_name)
WHERE full_name IS NULL
LIMIT 5000;
-- Repeat until 0 rows affected. Sleep briefly between batches to
-- let replicas catch up and to avoid saturating the server.

Contract — only after everything reads and writes the new shape, drop the old columns in a later, separate deploy:

-- Step 3 (CONTRACT): once every row is backfilled and all app code
-- reads/writes only the new column, retire the old ones in a later deploy.
ALTER TABLE users DROP COLUMN first_name, DROP COLUMN last_name;

The magic is that at no single moment does the app depend on a change that hasn't finished. Each step is independently safe and reversible. This same expand → backfill → contract dance is how you rename columns, split tables, and change types without downtime.

19.5 Online Schema-Change Tools

When even the expand step would rebuild a giant table, reach for a purpose-built tool. They create a shadow copy of the table, backfill it in the background, keep it in sync with live writes, then swap it in atomically:

# For huge tables, use an online-schema-change tool instead of raw ALTER.
# They build a shadow copy, backfill it, sync live changes via triggers,
# then swap it in atomically -- with reads and writes staying online.

# pt-online-schema-change (Percona Toolkit):
pt-online-schema-change \
  --alter "ADD COLUMN status VARCHAR(20) NOT NULL DEFAULT 'open'" \
  D=shop,t=invoices --execute

# gh-ost (GitHub's tool) is the other popular choice; it avoids triggers
# by reading the binary log to keep the shadow table in sync.

19.6 A Safe-Migration Checklist

  • Add before you require. Add columns as NULLable first; add the NOT NULL constraint only after backfilling.
  • Backfill in batches, with a small pause between them so replicas keep up.
  • Deploy code and schema in a compatible order — new code must tolerate the old schema, and vice versa, during the rollout.
  • Drop things last, in a separate deploy, once nothing references them.
  • Test the migration on a production-sized copy before running it for real. "It was fast on the dev database" is how outages happen.

🏋️ Exercise 19

You must add a NOT NULL country column to a customers table with 40 million rows, defaulting existing customers to 'US'. Doing it in one ALTER risks locking the table. Outline the expand-contract sequence.

Solution 
-- 1. EXPAND: add it NULLable (fast).
ALTER TABLE customers ADD COLUMN country CHAR(2) NULL;

-- 2. App now writes `country` on every new/updated customer.

-- 3. BACKFILL existing rows in batches.
UPDATE customers SET country = 'US' WHERE country IS NULL LIMIT 10000;
-- repeat until 0 rows affected.

-- 4. CONTRACT: now that no NULLs remain, enforce the constraint.
ALTER TABLE customers MODIFY country CHAR(2) NOT NULL DEFAULT 'US';

Add NULLable → deploy code writing the column → backfill old rows in batches → once no NULLs remain, tighten to NOT NULL. Each step is fast and non-blocking; the risky single-shot rewrite never happens.

SECTION 20

Scaling Out

When your data and traffic outgrow a single server, a database engineer's real job begins. This final technical section is a map of the territory: replication, partitioning, sharding, and the fundamental split between transactional and analytical workloads. You won't master each in one read — but you'll know what the tools are, what they cost, and when to reach for them.

20.1 The Single-Server Ceiling

One MySQL server can go remarkably far on good hardware and good indexes — do not scale out before you need to. But eventually you hit a wall: reads overwhelm the box, or the working set no longer fits in RAM, or a single machine can't be made more available. Scaling has two directions: up (a bigger machine — simple, but finite and expensive) and out (more machines — powerful, but it introduces distributed-systems complexity). The rest of this section is about scaling out.

20.2 Replication: Primary and Replicas

The first and most common step. You keep one primary (accepts writes) and one or more replicas (copies that stream changes from the primary via its binary log). Replicas serve reads. Since most applications read far more than they write, this alone relieves enormous pressure.

You'll also see this called primary/replica or the older master/slave terminology. Same idea: one authoritative writer, many read copies.

20.3 Read/Write Splitting

To use replicas, the application routes queries by type — writes to the primary, reads to the replicas:

# Read/write splitting: writes go to the primary, reads to replicas.
def get_connection(query_kind):
    if query_kind == "write":
        return primary_pool.connect()      # single source of truth
    else:
        return random.choice(replica_pools).connect()  # spread reads

# Caveat: REPLICATION LAG. A replica may be milliseconds-to-seconds
# behind the primary. Read-after-write (a user editing then viewing
# their own data) can show stale results -> route those reads to the
# primary, or wait for the replica to catch up.

The catch you must respect is replication lag: replicas trail the primary slightly. If a user saves a change and immediately reloads, a lagging replica might show the old value. The standard fix is to route "read-your-own-write" reads back to the primary.

20.4 Failover & High Availability

Replication also buys availability. If the primary dies, a replica can be promoted to become the new primary — a failover. Done manually it means downtime; tools like Orchestrator, MySQL Group Replication, or managed services (Amazon RDS/Aurora, Cloud SQL) automate detection and promotion so an outage lasts seconds, not hours. "High availability" fundamentally means: no single machine's death takes you offline.

20.5 Partitioning

Replication scales reads but every server still holds all the data. Partitioning splits one big table into smaller physical pieces. Vertical partitioning separates columns (rarely-used big TEXT/BLOB columns into their own table). Horizontal partitioning splits rows by a rule — by date range, by hash — while still living on one server:

-- Horizontal partitioning inside ONE server: split a table by a rule.
-- MySQL prunes partitions it doesn't need, so a date-scoped query
-- only touches the relevant chunk.
CREATE TABLE events (
  id BIGINT NOT NULL,
  created_at DATE NOT NULL,
  payload JSON,
  PRIMARY KEY (id, created_at)
)
PARTITION BY RANGE (YEAR(created_at)) (
  PARTITION p2023 VALUES LESS THAN (2024),
  PARTITION p2024 VALUES LESS THAN (2025),
  PARTITION p2025 VALUES LESS THAN (2026),
  PARTITION pmax  VALUES LESS THAN MAXVALUE
);
-- Bonus: dropping old data is instant -> ALTER TABLE events DROP PARTITION p2023;

The engine can prune partitions, skipping the ones a query can't match, which keeps scans fast as the table grows. And retiring old data becomes a metadata operation instead of a giant DELETE.

20.6 Sharding

When even one server can't hold the data or absorb the writes, you shard: split rows across multiple independent servers by a shard key. Users 0–2M on server A, 2M–4M on server B, and so on. The application computes which shard to talk to before it queries:

# Sharding: split rows across SEPARATE servers by a shard key.
# The app picks the shard before it queries.
def shard_for(user_id, num_shards=4):
    return user_id % num_shards          # user 0->shard0, 1->shard1, ...

conn = shard_pools[shard_for(user.id)].connect()
conn.execute("SELECT * FROM orders WHERE user_id = %s", user.id)
# Each shard is a full, independent MySQL server holding a SUBSET of users.

20.7 The Real Cost of Sharding

Sharding is the heaviest hammer in the box, and it takes away things you've relied on this whole course. Be sure you need it:

  • Cross-shard joins and queries become hard — data for one query may live on different servers; the database can't join across them for you.
  • Transactions don't span shards — the neat ACID guarantees from Section 11 stop at one shard's boundary.
  • Choosing the shard key is fateful — a bad key creates "hot" shards that take all the load, and re-sharding later is painful.
  • Operational weight — backups, migrations, and monitoring now multiply across every shard.

Order of operations: optimize queries and indexes → scale up → add read replicas → partition → and only then shard. Most applications never need the last step.

20.8 OLTP vs OLAP: Knowing MySQL's Boundary

A senior engineer knows when a relational database is the wrong tool. There are two very different workload shapes:

OLTPOLAP
Stands forOnline Transaction ProcessingOnline Analytical Processing
Looks likeMany tiny reads/writesFew huge aggregations

MySQL is a superb OLTP engine — that's what everything in this course optimized for. Run heavy OLAP scans against it and you'll bog down the same server your customers need for checkout.

20.9 When to Reach for a Data Warehouse

For analytics at scale, move the data to a system built for it: a columnar store that reads only the columns a query needs and scans huge volumes fast. Snowflake, BigQuery, Redshift, ClickHouse, and DuckDB are the common destinations. A pipeline (ETL/ELT, often orchestrated with tools like dbt) copies data out of your OLTP database into the warehouse, so analysts can run monster queries without ever touching production. The rule to remember: keep transactions in MySQL, keep analytics in a warehouse.

🏋️ Exercise 20

Your OLTP MySQL primary is fine for the app, but the analytics team's dashboards run enormous multi-year aggregations that slow checkout to a crawl during business hours. You are not out of storage and writes are light. What's the first thing you'd try, and what's the durable fix?

Solution
